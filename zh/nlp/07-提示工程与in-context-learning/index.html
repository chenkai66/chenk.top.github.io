自然语言处理（七）：提示工程与In-Context Learning

九月 08, 2025 Updated 5月 06, 2026 — min read — words

从提示结构、思维链到 Self-Consistency 与 ReAct：一套关于 In-Context Learning 的工作原理、必须正面应对的方差问题，以及能扩展到生产系统的提示模式。

同一个模型，既能一针见血地分析，也能一本正经地胡说八道。差异几乎不在于权重，而在于提问的方式。一句"分析一下这段文本"多半换来泛泛的总结；同样的请求，加上一个角色设定、两条清晰的示例和一段严格的输出 schema，就能得到下游解析器直接可用的 JSON。提示工程的意义，在于将这种差距从"碰运气"转变为可重复、可度量的工程实践。

让这一切发生的机制叫 In-Context Learning（上下文学习，ICL）。当你把示例嵌入提示，模型并未重新训练，只是将这些示例作为前缀，改变一次前向计算的条件分布——本质上是从示例中推断任务。理解 ICL 的能力边界，决定了你是在和模型对抗，还是在驾驭它。

这是 NLP 系列的第 7 部分，默认你已大致了解 Transformer 解码器如何逐 token 生成（第 4 部分），以及自回归语言模型是什么（第 6 部分）。下文所有结论尽量基于已有研究，但需提前说明：提示工程领域的文献噪声较大，很多数字强烈依赖具体模型和数据集。图中的柱状图请视为"示意形状"，而非任何 benchmark 的精确数值。

¶你将学到什么

  • 提示结构：五个可组合块（system、指令、示例、查询、格式说明）构成完整提示，各部分用途明确。
  • 三种范式：零样本、少样本、思维链——适用场景与 token 成本一目了然。
  • ICL 的工作理论：无需训练也能学习，模型从示例中捕捉到的信号究竟是什么。
  • 方差问题：示例顺序和格式如何引发准确率波动，测量方法清晰明了。
  • Self-Consistency：通过采样将随机解码器转化为集成方法。
  • ReAct：推理与工具调用交替进行，奠定现代 Agent 的基础框架。
  • 小型提示系统：注册表、A/B 评估、版本治理——让一组提示在团队中稳定运行半年以上。

¶前置知识

  • 熟悉大语言模型——见第 6 部分：GPT 与生成式模型。
  • 掌握基础 Python，能读懂短代码。
  • 拥有任意一个 LLM API（OpenAI、Anthropic 或开源权重模型）。

¶1. 一个提示由什么组成

提示本质上是一段模型用来条件化的文本字符串。所谓 system / user 角色、function 描述、检索结果，不过是 API 在 tokenize 前拼接成的一段结构化文本。把提示看作"带命名块的纯文本"，是最简洁的心智模型。

以下五个块不必全部出现，但生产环境中的提示通常包含其中的一部分，顺序也大致如此：

  1. System / 角色：设定人格、拒答策略、语气、长度预算。请求间保持稳定，因此对 KV 缓存和 prompt cache 友好。
  2. 任务指令：用一句祈使句说明目标。
  3. 少样本示例：输入-输出示例对，是 ICL 的主要信号来源。
  4. 用户查询：实际需要处理的输入。
  5. 格式说明：固定输出形态的 schema（JSON、可正则匹配的标签、表格等）。

一个务实的提示构造器：

两个新手常忽略的细节：

  • 顺序很重要。将示例放在格式说明之后、查询之前，往往比放在最顶部效果更好——解码器有显著的"近因偏置"，越靠近查询的内容影响越大。
  • 稳定前缀，可变后缀。所有不变的内容（system、示例、格式）放在最前面，方便 KV 缓存或 prompt cache 复用；变化的输入放在最后。

¶四条经得起生产考验的原则

我在设计不少提示后，以下四条仍然是我会推荐给别人的：

  1. 清晰胜过聪明。把"分析这段文本"换成"将文本分类为 {正面, 负面, 中性} 之一，并以 JSON 返回"。你在和模型脑中所有可能的解读竞争，含糊就是把话语权交给它。
  2. 具体性买的是确定性。明确"做什么"和"不要做什么"，并规定不确定时如何输出（"如果文档里找不到答案，请返回 {\"answer\": null}"）。模型对负向约束的遵守比想象中好。
  3. 上下文要完整。如果回答需要某个模型不一定知道的定义，就直接带上。这比错答便宜得多。
  4. 角色设定按需使用。"你是一名资深安全审计员"在代码评审任务上确实能收紧输出分布，但它不是咒语——通用任务别套，徒增 token。

¶2. 零样本、少样本、思维链

这是其他所有技巧赖以构建的三种基线框架。

¶零样本

只描述任务，不提供示例。模型完全依赖预训练和指令微调期间学到的知识。

零样本适合模型本来就熟悉的任务（情感、翻译、短文本摘要），并且你在乎延迟和成本。它的弱点是输出格式不稳定——模型可能输出 "Positive sentiment"、"POSITIVE"，甚至是一段分析。用一句严格的格式约束钉死它（"只回复 {positive, negative, neutral} 中的一个单词"）。

¶少样本

少样本就是在查询前面放 kkk 条示例，是 ICL 的教科书设定。

示例其实在做三件事：

  • 任务识别：消除歧义。"翻译"可能指音译、改写或重写，两条示例就能说清你要的是哪种。
  • 格式对齐：每条示例的输出侧就是模板，模型会照抄。
  • 标签空间锚定：示例里出现过的标签集合事实上变成了模型的输出词表，即使你从没显式列举过。

一个反直觉的发现：示例里"标签是否正确"远没有想象中那么重要。Min 等人（2022）发现把少样本示例的金标随机打乱，在不少任务上几乎不掉点——真正起作用的是输入分布和标签空间。这并不是说"标签不重要"，而是说不要在标签正确性上过度打磨；要在覆盖度和格式上下功夫。

¶思维链（Chain-of-Thought）

对于多步任务，让模型先把推理过程写出来再给答案。

机制并不神秘。每一个生成出来的 token 都会改变下一个 token 的条件上下文。模型是自回归的，把中间状态（"累计已读：90"）显式写出来，后面所有 token——包括最终答案——就能直接用上这个状态。如果不写出来，模型就只能在一次定深网络的前向中算出全部中间量。CoT 实际上是用输出 token 的成本，买到了串行计算量。

用概率视角看会更干净：对问题 xxx 和答案 aaa，对潜在推理链 zzz 做边际化：

P(a∣x)  =  ∑zP(z∣x) P(a∣x,z).P(a \mid x) \;=\; \sum_z P(z \mid x)\, P(a \mid x, z). P(a∣x)=z∑​P(z∣x)P(a∣x,z).

贪心 CoT 就是只采一个 zzz 然后赌它对。Self-Consistency（第 5 节）是用采样去近似这个求和。

CoT 什么时候有用，什么时候反而拖后腿：

  • 有用：算术、多跳问答、代码推理，凡是有中间状态的任务。
  • 持平或更糟：简单分类、检索、单条事实查询。推理铺垫只是徒增 token 和延迟，还多了一次"被自己说服走错"的机会。
  • 现代有"思考"模式的推理模型（在 CoT 风格轨迹上专门训练过的长上下文推理模型）已经把这种行为内化了，显式 CoT 提示仍然有用，但提升幅度比 2022 年那一代 GPT-3 时代要小。

¶3. ICL 为什么会工作

往提示里塞几条示例，模型权重没变，行为却变了。为什么？目前业界没有统一答案，但三种互补的解释加在一起，最接近共识。

1. 隐式任务推断。预训练 LM 在训练数据中见过大量以文本形式表达的任务（问答对、代码-注释对、原文-译文对）。推理时，提示里的示例相当于一次贝叶斯后验更新：让 P(task∣prompt)P(\text{task} \mid \text{prompt})P(task∣prompt) 收敛到最匹配的"任务"。这是 Xie 等人（2022）的观点。

2. 注意力内的隐式梯度下降。一系列研究（Akyurek 等、von Oswald 等，2022-2023）证明，在某些玩具设置下，attention 层可以实现对提示中编码的线性回归任务做一步梯度下降。严格来说，机制结论仅在 toy setting 成立，但"注意力在做某种快速适应"这一直觉很有价值。

3. 模式匹配 + 复制。最朴素的解释是：induction head（Olsson 等，2022）会复制上下文中早些时候出现的模式。少样本提示给了模型一个可复制的模式。

无论更倾向于哪种解释，实际推论都相同：

  • 示例数量越多越好，但收益迅速衰减。大部分增益来自前 2-4 条。
  • 覆盖度比正确率重要。能覆盖输入空间的示例比单条巧妙的示例更有用。
  • 打平时近的赢。提示中最后一条示例对模型影响最大。
  • 位置偏置真实存在。多选题任务上，模型常系统性偏向 A 选项或最近一个选项，具体取决于模型。

¶4. 方差问题：必须正面应对的事

宣传材料很少提到，但这是事实：仅仅调整示例顺序、把 Q: 改成 Input:、答案加不加引号，准确率就能波动 10-30 个百分点。

Lu 等人（2022）将这种现象称为顺序敏感性。他们证明，在分类任务中，同一模型、同一组示例，不同排列的准确率可以从接近随机水平跨越到接近 SOTA。Sclar 等人（2024）进一步扩展了这项研究，提出格式敏感性——将 Q:/A: 替换为 Question:/Answer:，准确率可能相差两位数。

实际操作建议：

  • 至少在 50-100 条留出样本上评估。单个案例无法说明问题。
  • 比较两个提示时，使用多个种子或多种示例顺序。报告中位数，而非最佳结果。
  • 尽早固定格式，每次修改都应视为一次带版本号的事件，而不是"随手调整"。

一个最小评估框架：

如果 spread > 0.05，你的提示仍然不稳定，任何单次运行的结果都不过是噪音。

¶示例到底要给多少条

各类任务的趋势基本一致：前几条示例收益显著，当 k≈4k \approx 4k≈4 至 888 时收益趋于饱和，超过 k≈16k \approx 16k≈16 基本只是浪费 token。两条经验法则：

  • 类别数 ≤5\le 5≤5 的分类任务，k=2×类别数k = 2 \times \text{类别数}k=2×类别数 是不错的默认值。
  • 生成任务中，222-333 条高质量示例通常远胜 101010 条平庸示例。

挑选示例时记住两点：输入侧覆盖全面、输出侧清晰无歧义。模型会忠实复制你的格式，包括格式中的错误。

¶5. Self-Consistency：把解码器变成集成

一条 CoT 链可能在第二步就走偏，错误一路传到结尾。Self-Consistency（Wang 等，2022）的解法只有一句话：对同一个问题采样 kkk 条不同的推理链，然后对最终答案多数投票。

形式上，对问题 xxx 采 kkk 条链 z1,…,zkz_1, \dots, z_kz1​,…,zk​，每条链给出答案 aia_iai​：

a^  =  arg⁡max⁡a∑i=1k1[ai=a].\hat{a} \;=\; \arg\max_a \sum_{i=1}^{k} \mathbb{1}[a_i = a]. a^=argamax​i=1∑k​1[ai​=a].

这是对边际 ∑zP(z∣x) P(a∣x,z)\sum_z P(z \mid x)\, P(a \mid x, z)∑z​P(z∣x)P(a∣x,z) 的一次蒙特卡洛近似。它之所以有效，是因为错误倾向于多样、正确倾向于收敛——错的推理链往往各自走到不同的错答案上，对的推理链则会汇聚到同一个答案。

两点上线之后才会体会到的细节：

  • 温度很关键。建议 T∈[0.5,0.9]T \in [0.5, 0.9]T∈[0.5,0.9]。T=0T = 0T=0 时所有样本会塌成同一条链，集成退化为单点。
  • 得票率就是你的置信度信号。5/5 一致和 3/5 险胜的可信度差很多。把这个数字暴露给下游消费者，这是最便宜的可靠性指标之一。

Tree of Thoughts（Yao 等，2023）是它的推广：不再独立采线性链，而是搜索一棵推理树，对低分分支剪枝。能力更强但成本也更高，等 Self-Consistency 真饱和了再上。

¶6. ReAct：推理 + 行动

Self-Consistency 提升的是“模型用已有知识能做到什么”。ReAct（Yao 等，2022）解决的是更难的问题：模型需要外部信息或外部动作时怎么办？这个模式在输出里穿插三种块：

  • Thought（思考）：用自然语言对当前状态做推理。
  • Action（行动）：一次结构化工具调用，例如 search("..."); calc("..."); read_file("...")。
  • Observation（观察）：工具的输出，作为下一轮的上下文喂回去。

循环在模型输出 Answer: 而不是 Action: 时终止。现代 Agent 框架（LangChain Agents、OpenAI 的 function calling、Anthropic 的 tool use）本质上都是这个模板的变体——只不过把“结构化工具调用”从自由文本里搬到了一个 JSON 字段里。

一个抓住要害的最小实现：

生产里真正会咬人的三件事：

  • Stop token：在 \nObservation: 处停止生成，否则模型会自己幻觉出工具结果。
  • 工具异常处理：把工具调用包起来，把异常信息当成观察喂回去。模型经常能据此自我纠正。
  • 步数预算：必须给上限。没有预算的 Agent 会循环到把你账户烧穿。

¶7. 构建一个提示系统

几个写得好的提示是脚本；能撑过团队换人、模型升级、四个月 A/B 实验的，才叫系统。区别在三个习惯。

¶把提示当代码对待

加版本号、走代码评审、放进仓库，而不是堆在一份共享文档里。一个最小注册表：

每个调用点都打上 (prompt_name, version) 标签。改提示时升版本号，老代码继续用老提示直到你迁移它。

¶上线前必须有评估

每个进生产的提示，应该有三件配套：

  • 一个 golden set：50-200 条输入和期望输出（或评分细则）。
  • 一个自动评估器：精确匹配、JSON schema 校验，或是 LLM-as-judge（评判用的提示也要钉死版本）。
  • 一个 CI 回归门：跑当前提示和新提示，准确率掉了就拒绝合入。

¶有意识地组合技巧

最大的增益通常来自堆叠：

  • 角色 + 少样本 + 格式说明：任何结构化输出任务的标准三件套。
  • CoT + Self-Consistency：任何"答错代价高"的推理任务。
  • ReAct + 检索：任何依赖模型不知道的事实的任务。

但不要为了堆而堆。每个块都要花 token、加延迟，并多一个出错的地方。

¶常见问题

提示是不是越长越好？
不是。超过某个点，多余的上下文会分散注意力，还会增加延迟和成本。先从短提示开始，再根据评估集结果有针对性地扩展。

示例给多少条合适？
分类任务 2-5 条，生成任务 1-3 条。超过 8 条很少有额外收益，反而常因格式问题引入噪声。始终要在数据上验证，别靠直觉判断。

CoT 在所有任务上都有效吗？
不。它在多步推理任务（数学、逻辑、代码、多跳问答）中效果显著，但在简单分类或事实查询任务中往往只是增加噪声和 token 消耗。

温度怎么调？
确定性输出（分类、抽取）：T∈[0.0,0.2]T \in [0.0, 0.2]T∈[0.0,0.2]；平衡的生成：T∈[0.5,0.7]T \in [0.5, 0.7]T∈[0.5,0.7]；Self-Consistency 采样：T∈[0.7,0.9]T \in [0.7, 0.9]T∈[0.7,0.9]——目标是追求路径多样性。

提示工程能取代微调吗？
能取代相当一部分过去需要微调的场景，尤其是 instruction following 类任务。仍需微调的情况包括：(a) 大规模场景下要求行为高度一致，无法接受提示漂移；(b) 想用更小、更便宜的模型在特定领域追平大模型；(c) 改造基础模型抗拒的某种行为。

示例顺序到底有多大影响？
比预期的大很多。近因偏置确实存在——最后一条示例的影响最大。比较两个提示时，务必尝试多种顺序，取中位数结果。

要不要担心 Prompt Injection？
要。任何时候，只要提示中拼接了不受控制的文本（用户输入、检索文档、工具输出），就要将这些内容视为数据，绝不允许它们改写你的指令。这是个独立话题，后续关于 Agent 和安全的内容会专门讨论。
