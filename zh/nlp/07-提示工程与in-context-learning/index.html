自然语言处理（七）：提示工程与In-Context Learning

从提示结构、思维链到 Self-Consistency 与 ReAct：一套关于 In-Context Learning 的工作原理、必须正面应对的方差问题，以及能扩展到生产系统的提示模式。

#NLP#提示工程#LLM#In-Context Learning
Read in English →

同一个模型，可以给出一针见血的分析，也能一本正经地胡说八道。区别几乎从不在权重，而在你怎么问。一句"分析一下这段文本"换来的多半是泛泛的总结；同样的请求加上一个角色、两条干净的示例和一段严格的输出 schema，得到的就是下游解析器能直接消费的 JSON。提示工程的意义，是把这种差距从"碰运气"变成可重复、可度量的工程实践。

让这一切发生的机制叫 In-Context Learning（上下文学习，ICL）。当你把示例塞进提示，模型并没有重新训练，它只是把这几条示例作为前缀，让一次前向计算的条件分布发生改变——说白了就是从示例里推断出任务。理解 ICL 能做什么、不能做什么，决定了你是在和模型对抗，还是在驾驭它。

这一篇是 NLP 系列的第 7 部分，默认你已经大致清楚 Transformer 解码器是怎么逐 token 生成的（第 4 部分），以及自回归语言模型是什么（第 6 部分）。下文所有结论都尽量贴着已有研究，但要提前说明：提示工程领域的文献噪声相当大，很多数字都强烈依赖具体模型和数据集。图里的柱状图请当成"示意形状"看，而不是任何一个 benchmark 的精确数字。

你将学到什么

  • 提示结构：五个可组合的模块（system、指令、示例、查询、格式说明），每个模块的作用是什么。
  • 三种范式：零样本、少样本和思维链，分别适合哪些场景，token 消耗是多少。
  • ICL 的工作原理：为什么模型不需要训练也能从提示中“学习”，它到底捕捉了哪些信号。
  • 方差问题：仅靠格式和顺序的变化，准确率会有多大波动，如何测量。
  • Self-Consistency：通过采样多条推理路径，把随机解码器变成集成方法。
  • ReAct：推理和工具调用交替进行，这是现代 Agent 的核心基础。
  • 小型提示系统：提示注册表、A/B 测试框架、版本管理——让一组提示在团队中长期可用。

前置知识

  • 要对大语言模型有基本了解，可以参考第 6 部分：GPT 与生成式模型 。
  • 会一点 Python，能看懂简短的代码片段。
  • 有任意 LLM API 的使用权限，比如 OpenAI、Anthropic，或者开源权重模型。

1. 提示的组成部分

提示本质上是一段模型用来生成结果的文本字符串。其他内容——比如“系统”和“用户”角色、函数描述、检索结果——都只是 API 在分词之前拼接成的一个完整序列。把提示看作一个带有命名块的纯文本，是最简单直观的理解方式。

下面五个部分不是强制要求，但生产环境中的提示通常会包含其中的一部分，顺序也大致如下：

  1. 系统/角色：定义角色性格、拒绝策略、语气风格、长度限制。这部分在请求间保持稳定，因此能很好地利用缓存。
  2. 任务指令：用一句祈使句明确目标。
  3. 少样本示例：输入到输出的示例对，是 ICL 的核心信号。
  4. 用户查询：需要处理的实际输入。
  5. 格式说明：固定输出格式的 schema（JSON、正则匹配标签、表格等）。

from dataclasses import dataclass, field

@dataclass
class Prompt:
    system: str = ""
    instruction: str = ""
    examples: list[tuple[str, str]] = field(default_factory=list)
    query: str = ""
    format_spec: str = ""

    def render(self) -> str:
        parts: list[str] = []
        if self.system:
            parts.append(f"[SYSTEM]\n{self.system}")
        if self.instruction:
            parts.append(f"[TASK]\n{self.instruction}")
        if self.examples:
            shots = "\n\n".join(
                f"Input: {x}\nOutput: {y}" for x, y in self.examples
            )
            parts.append(f"[EXAMPLES]\n{shots}")
        if self.format_spec:
            parts.append(f"[FORMAT]\n{self.format_spec}")
        if self.query:
            parts.append(f"[INPUT]\n{self.query}\nOutput:")
        return "\n\n".join(parts)

新手容易忽略的两个细节：

  • 顺序很重要。示例放在格式说明之后、查询之前，效果往往比放在开头更好。解码器有“近因偏置”，越靠近查询的内容影响越大。
  • 前缀固定，后缀可变。所有不变的内容（系统、示例、格式）放在前面，方便 KV 缓存和提示缓存复用；变化的输入放在最后。

四条经得起实践检验的原则

我做了大量提示工程后，仍然会推荐以下四条原则：

  1. 清晰胜过聪明。把“分析这段文本”改成“将文本分类为 {正面, 负面, 中性} 之一，并以 JSON 返回”。你是在和模型可能的所有解读竞争，模糊只会让模型自己做主。
  2. 具体带来确定性。明确“不要做什么”，以及不确定时如何输出（“如果文档里找不到答案，请返回 {\"answer\": null}”）。模型对负向约束的遵守比想象中好。
  3. 上下文要完整。如果回答需要某个模型可能不知道的定义，就直接带上。这比错答便宜得多。
  4. 按需设定角色。“你是一名资深安全审计员”在代码评审任务上确实能收紧输出分布，但它不是万能咒语——通用任务别乱用，徒增 token。

2. 零样本、少样本、思维链

这三种方法是所有其他技术的基础。

零样本

直接描述任务，不给任何例子。模型完全依赖预训练和指令微调中学到的知识。

zero_shot = """将句子的情感分类为 positive、negative 或 neutral。

句子：这部电影剧情精彩、表演出色。
情感："""

零样本适合模型已经很熟悉的任务，比如情感分析、翻译或短文本摘要，尤其是当我关心延迟和成本时。它的缺点是输出格式不稳定，可能返回 “Positive sentiment”、“POSITIVE” 或一段分析文字。用一句严格的话固定格式，比如：“只返回 {positive, negative, neutral} 中的一个词。”

这些例子其实做了三件事：

  • 明确任务：消除歧义。比如 “翻译” 可能指音译、改写或重写，两个例子就能说清楚具体要什么。
  • 统一格式：每个例子的输出部分就是模板，模型会模仿。
  • 限定标签范围：例子中出现的标签集合会成为模型的实际输出词汇表，即使你没显式列出。

一个反直觉的发现是：标签本身没那么重要。Min 等人（2022）证明，随机打乱少样本示例中的正确标签，在很多任务上性能几乎不变——真正重要的是输入分布和标签空间。这并不是说标签完全不重要（在复杂任务上还是重要），而是不要过度纠结标签的准确性，而要关注覆盖范围和格式。

问题：一本书有 120 页。第一天读了 30 页，第二天读了第一天的两倍，
第三天读了剩下页数的一半。第三天读了多少页？

一步步算：
1. 第一天：30 页。
2. 第二天：2 × 30 = 60 页。
3. 累计已读：30 + 60 = 90 页。
4. 剩下：120 − 90 = 30 页。
5. 第三天：30 / 2 = 15 页。

答案：15 页。

原理很简单，没有神秘的地方。每个生成的 token 都会改变下一个 token 的上下文条件。模型是自回归的，把中间状态（如 “累计已读：90”）写出来后，后续 token 包括最终答案都能直接用上这个信息。如果不写出来，模型只能通过一次固定深度网络的前向计算得出所有中间结果。CoT 实际上是用输出 token 的代价换取了更多的串行计算能力。

$$P(a \mid x) \;=\; \sum_z P(z \mid x)\, P(a \mid x, z).$$

贪心 CoT 是选一个 $z$ 并希望它是对的。Self-Consistency（第 5 节）则是通过采样多个 $z$ 来近似求和。

CoT 在什么时候有用，什么时候反而拖后腿：

  • 有用：算术题、多跳问答、代码推理等涉及中间状态的任务。
  • 持平或更糟：简单分类、检索、单条事实查询。推理铺垫增加了 token 数量和延迟，还可能让模型自己绕进错误答案。
  • 现代长上下文推理模型（在 CoT 风格轨迹上专门训练的模型）已经内化了这种能力，显式 CoT 提示仍然有帮助，但提升幅度比 2022 年 GPT-3 那一代小得多。

3. ICL 的工作原理猜想

为什么在提示中加入示例会改变模型的行为？模型的权重明明是固定的。目前业界没有统一的答案，但三种互补的解释加在一起，最接近共识。

1. 隐式任务推断
预训练语言模型在训练时见过大量以文本形式表达的任务，比如问答对、代码注释对、翻译对等。推理时，提示中的示例相当于告诉模型：“接下来的任务属于哪一类”。这是贝叶斯视角下的后验更新：通过示例让 $P(\text{task} \mid \text{prompt})$ 更加明确。Xie 等人（2022）的研究支持了这一点。

2. 注意力机制中的隐式梯度下降
一些研究（Akyurek 等、von Oswald 等，2022-2023）表明，注意力层可以在某些简单场景下实现一步梯度下降，针对的是提示中编码的线性回归任务。虽然这种机制只在玩具场景中严格成立，但它提供了一个有用的直觉：注意力机制确实在快速适应新信息。

3. 模式匹配与复制
最简单的解释是：induction head（Olsson 等，2022）会从上下文中复制前面出现过的模式。少样本提示为模型提供了一个可以模仿的模板。

无论你更倾向于哪种解释，实际应用中的结论都是一样的：

  • 示例越多越好，但收益递减很快。前 2-4 条示例带来的提升最明显。
  • 分布比正确性更重要。覆盖输入空间的示例比单条精巧的示例更有价值。
  • 越靠后的示例影响越大。提示中最后一条示例对模型的影响最显著。
  • 选项位置偏置确实存在。在多选题任务中，模型往往会系统性地偏向 A 选项或最近的选项，具体取决于模型本身。

4. 方差问题：必须正视的现实

宣传材料里不会告诉你一个残酷的事实：仅仅调整示例顺序、把 Q: 改成 Input:、或者给答案加不加引号，准确率就能波动 10-30 个点。

Lu 等人（2022）把这个现象称为顺序敏感性。他们证明，在分类任务中，同一个模型用同一组示例，不同排列的准确率可以从接近随机水平飙升到接近 SOTA。Sclar 等人（2024）进一步扩展了这个研究，提出了格式敏感性——把 Q:/A: 换成 Question:/Answer:，准确率能波动两位数。

实际操作中要注意以下几点：

  • 评估提示时，至少用 50-100 条独立样本。单个例子说明不了任何问题。
  • 比较两个提示时，跑多个随机种子或示例顺序。报告中位数，而不是最高分。
  • 尽早固定提示格式，每次改动都要当成一次有版本号的更新，而不是随意调整。

import statistics, random
from typing import Callable

def evaluate(
    build_prompt: Callable[[str, list], str],
    model_call: Callable[[str], str],
    cases: list[dict],
    examples_pool: list[dict],
    k: int = 4,
    n_seeds: int = 5,
) -> dict:
    """跑 n_seeds 次不同的示例采样，返回准确率中位数与波动范围。"""
    accs = []
    for seed in range(n_seeds):
        rng = random.Random(seed)
        shots = rng.sample(examples_pool, k)
        correct = 0
        for case in cases:
            out = model_call(build_prompt(case["input"], shots))
            if out.strip() == case["expected"].strip():
                correct += 1
        accs.append(correct / len(cases))
    return {
        "median": statistics.median(accs),
        "min": min(accs),
        "max": max(accs),
        "spread": max(accs) - min(accs),
    }

如果 spread > 0.05，你的提示就不稳定，任何单次运行的结果都只是噪音。

示例数量到底多少合适？

无论什么任务，规律都很一致：前几条示例提升效果显著，收益在 $k \approx 4$ 到 $8$ 之间趋于饱和，超过 $k \approx 16$ 后基本只是浪费 token。我总结了两条经验：

  • 对于类别数 $\le 5$ 的分类任务，$k = 2 \times \text{类别数}$ 是不错的默认值。
  • 对于生成任务，$2$-$3$ 条高质量示例几乎总是优于 $10$ 条平庸示例。

挑选示例时记住两点：输入侧要覆盖广泛，输出侧要干净无歧义。模型会忠实复制你的格式，包括格式里的 bug。

5. Self-consistency：把解码器变成集成模型

一条 CoT 链在第二步就可能走偏，错误会一路传递到最后。Self-consistency（Wang 等，2022）用一个简单的办法解决了这个问题：对同一个问题采样 $k$ 条不同的推理链，然后通过多数投票选出最终答案。

$$\hat{a} \;=\; \arg\max_a \sum_{i=1}^{k} \mathbb{1}[a_i = a].$$

这是对边际分布 $\sum_z P(z \mid x)\, P(a \mid x, z)$ 的蒙特卡洛近似。它之所以有效，是因为错误往往是分散的，而正确答案通常是收敛的——错误的推理链往往指向不同的错误答案，而正确的推理链则会汇聚到同一个正确答案上。

from collections import Counter

def self_consistency(
    model_call,
    prompt: str,
    k: int = 8,
    temperature: float = 0.7,
    extract_answer=lambda s: s.strip().split()[-1],
) -> dict:
    """采样 k 条 CoT 推理链，对最终答案进行多数投票"""
    answers = []
    for _ in range(k):
        out = model_call(prompt, temperature=temperature)
        answers.append(extract_answer(out))
    counts = Counter(answers)
    top, n = counts.most_common(1)[0]
    return {"answer": top, "confidence": n / k, "votes": dict(counts)}

实际使用中有两点需要注意：

  • 温度参数很关键。建议设置 $T \in [0.5, 0.9]$。如果 $T = 0$，所有样本都会坍缩成同一条推理链，集成模型退化为单点。
  • 得票率是置信度的重要信号。5/5 的一致性和 3/5 的微弱多数可信度差别很大。把这个数字暴露给下游用户，这是最简单且有效的可靠性指标之一。

Tree of Thoughts（Yao 等，2023）进一步扩展了这个方法：不再独立采样线性推理链，而是搜索一棵推理树，并对低分分支进行剪枝。这种方法能力更强，但成本也更高，只有在 Self-consistency 达到瓶颈时才值得尝试。

6. ReAct：推理 + 行动

Self-Consistency 提升的是模型利用已有知识的能力。ReAct（Yao 等，2022）解决了一个更复杂的问题：当模型需要外部信息或执行外部操作时该怎么办？它的输出模式交替包含三个部分：

  • Thought（思考）：用自由形式对当前状态进行推理。
  • Action（行动）：调用结构化工具，比如 search("..."); calc("..."); read_file("...")。
  • Observation（观察）：工具的输出结果，反馈到下一轮输入中。

循环会在模型输出 Answer: 而不是 Action: 时结束。现代 Agent 框架（LangChain Agents、OpenAI 的 function calling、Anthropic 的 tool use）本质上都是这个模式的变体——通常把结构化工具调用移到 JSON 类型的 API 字段中，而不是从自由文本解析。

import re
from typing import Callable

class ReActAgent:
    def __init__(self, model_call: Callable[[str], str],
                 tools: dict[str, Callable[[str], str]]):
        self.model_call = model_call
        self.tools = tools

    def _system(self) -> str:
        names = ", ".join(self.tools)
        return (
            "可用工具：" + names + "。\n"
            "严格遵循以下格式：\n"
            "Thought: <推理>\n"
            "Action: <tool>(<参数>)\n"
            "Observation: <工具结果>\n"
            "...重复...\n"
            "Answer: <最终答案>"
        )

    def run(self, question: str, max_steps: int = 6) -> str:
        ctx = f"{self._system()}\n\nQuestion: {question}\n"
        for _ in range(max_steps):
            out = self.model_call(ctx, stop=["\nObservation:"])
            ctx += out
            if "Answer:" in out:
                return out.split("Answer:")[-1].strip()
            m = re.search(r"Action:\s*(\w+)\((.*?)\)", out)
            if not m:
                return "(无法解析 Action)"
            name, arg = m.group(1), m.group(2).strip()
            obs = self.tools[name](arg) if name in self.tools \
                  else f"未知工具：{name}"
            ctx += f"\nObservation: {obs}\n"
        return "(达到最大步数)"

在生产环境中，有三件事特别重要：

  • Stop token：在 \nObservation: 处停止生成，避免模型自己编造工具结果。
  • 工具异常处理：包装工具调用，把异常信息作为观察返回。模型通常能根据这些信息自我纠正。
  • 步数限制：必须设置上限。没有限制的 Agent 会一直循环，直到耗尽你的预算。

7. 构建提示系统

几个高质量的提示可以称为脚本，但能经得起团队人员流动、模型升级和四个月 A/B 测试考验的，才算是系统。两者之间的差距体现在三个习惯上。

把提示当作代码管理

from pathlib import Path

class PromptRegistry:
    def __init__(self, root: Path):
        self.root = root
        self._cache: dict[str, str] = {}

    def get(self, name: str, version: str = "latest") -> str:
        key = f"{name}@{version}"
        if key not in self._cache:
            path = self.root / name / f"{version}.txt"
            self._cache[key] = path.read_text()
        return self._cache[key]

每个调用点都标记 (prompt_name, version)。修改提示时记得升级版本号，老代码会继续使用旧提示，直到我手动迁移。

上线前必须评估

任何进入生产环境的提示，都需要准备三样东西：

  • 一个 golden set：50 到 200 条输入及其期望输出（或者评分标准）。
  • 一个自动评估器：可以是精确匹配、JSON schema 校验，或者用 LLM 当裁判（裁判提示也要固定版本）。
  • 一个回归测试的 CI 步骤：运行当前提示和新提示，如果准确率下降就拒绝合并。

def regression_check(old_prompt: str, new_prompt: str,
                     cases, model_call, judge) -> bool:
    old_acc = mean(judge(case, model_call(old_prompt + case.input))
                   for case in cases)
    new_acc = mean(judge(case, model_call(new_prompt + case.input))
                   for case in cases)
    return new_acc >= old_acc - 0.01   # 允许 1% 的准确率下降

有目的地组合技术

最大的收益通常来自技术叠加：

  • 角色 + 少样本 + 格式说明：适用于所有结构化输出任务。
  • CoT + 自洽性：适用于任何不能出错的推理任务。
  • ReAct + 检索：适用于需要模型不具备的事实的任务。

但不要为了叠加而叠加。每增加一个模块都会消耗 token、增加延迟，并多一个可能出错的地方。

def build_advanced(question: str, examples: list[dict],
                   *, role: str = "", use_cot: bool = True) -> str:
    parts: list[str] = []
    if role:
        parts.append(f"你是 {role}。")
    parts.append("请解决以下问题。")
    for ex in examples:
        parts.append(f"Q: {ex['question']}")
        if use_cot and "reasoning" in ex:
            parts.append(f"Reasoning: {ex['reasoning']}")
        parts.append(f"A: {ex['answer']}\n")
    if use_cot:
        parts.append("先一步步推理，再在以 'A:' 开头的一行里给出最终答案。")
    parts.append(f"Q: {question}\n")
    return "\n".join(parts)

import json, re
from collections import Counter

SYSTEM = (
    "你是一个严谨的文本分类器。只输出合法的 JSON，不要任何解释。"
)

EXAMPLES = [
    ("这部电影很棒，演技精湛、节奏紧凑。",
     {"sentiment": "positive", "confidence": 0.95}),
    ("电池只能撑三小时，App 还经常崩。",
     {"sentiment": "negative", "confidence": 0.92}),
    ("快递周二送到。",
     {"sentiment": "neutral", "confidence": 0.88}),
]

FORMAT_SPEC = (
    '严格输出：{"sentiment": "<positive|negative|neutral>", '
    '"confidence": <0.0-1.0>}'
)

def build_prompt(text: str) -> str:
    shots = "\n".join(
        f"Text: {x}\nOutput: {json.dumps(y, ensure_ascii=False)}"
        for x, y in EXAMPLES
    )
    return (
        f"[SYSTEM]\n{SYSTEM}\n\n"
        f"[FORMAT]\n{FORMAT_SPEC}\n\n"
        f"[EXAMPLES]\n{shots}\n\n"
        f"[INPUT]\nText: {text}\nOutput:"
    )

def parse(raw: str) -> dict | None:
    m = re.search(r"\{.*\}", raw, re.DOTALL)
    try:
        return json.loads(m.group(0)) if m else None
    except json.JSONDecodeError:
        return None

def classify(model_call, text: str, *, k: int = 5) -> dict:
    """用 Self-Consistency 提升情感分类器的稳定性。"""
    prompt = build_prompt(text)
    labels, confs = [], []
    for _ in range(k):
        parsed = parse(model_call(prompt, temperature=0.5))
        if parsed and "sentiment" in parsed:
            labels.append(parsed["sentiment"])
            confs.append(float(parsed.get("confidence", 0.5)))
    if not labels:
        return {"sentiment": "unknown", "confidence": 0.0, "votes": {}}
    counts = Counter(labels)
    top, n = counts.most_common(1)[0]
    return {"sentiment": top,
            "confidence": n / len(labels),
            "model_confidence": sum(confs) / len(confs),
            "votes": dict(counts)}

这段代码里包含了文章提到的所有关键点：一个角色（system）、格式规范、三条多样化的少样本示例、一个能处理模型噪声的解析器，以及通过 Self-Consistency 抑制标签翻转的逻辑。单看每个部分都不复杂，但组合起来，就能让流水线的准确率从 60% 提升到 90%。

常见问题

提示越长越好吗？

不是。超过某个点，多余的上下文反而会干扰模型，还会增加延迟和成本。先从简短开始，只添加对评估集有实际帮助的内容。

示例数量怎么定？

分类任务用 2 到 5 条，生成任务用 1 到 3 条。超过 8 条基本没用，还容易因为格式问题引入更多方差。一定要测试，别靠猜测。

CoT 对所有任务都有效吗？

不是。它在多步推理任务（数学、逻辑、代码、多跳问答）上表现突出，但在简单分类或事实查询中只会增加噪声和 token 消耗，没有收益。

温度参数怎么设置？

确定性输出（分类、抽取）：$T \in [0.0, 0.2]$；平衡生成：$T \in [0.5, 0.7]$；Self-Consistency 采样：$T \in [0.7, 0.9]$——这时我就是要路径多样性。

提示工程能代替微调吗？

它已经能替代很多过去需要微调的场景，尤其是指令跟随类任务。但有些情况还是得微调：(a) 大规模场景下需要行为高度一致，提示漂移不可接受；(b) 想用更小、更便宜的模型在特定领域追平大模型；(c) 基础模型抗拒的行为改造。

示例顺序重要吗？

比想象中重要得多。近因偏置确实存在——最后一条示例对模型影响最大。比较不同提示时，一定要跑多个顺序，取中位数结果。

要担心 Prompt Injection 吗？

要。只要提示里包含不受控制的文本（用户输入、检索文档、工具输出），就要把这些内容当作数据处理，绝不能让它们改写指令。这是个独立的话题，后面讲 Agent 和安全时会详细讨论。
