¶1. Agent 与工具调用

继 instruction tuning 之后，模型能力的最大跃迁是学会调用函数。普通 LLM 是一个静态的近似器：训练时未学到的知识，永远无法补足。而具备 Agent 能力的 LLM 则是一个控制器，能够向外部世界查询事实、运行代码、访问数据库，并继续生成内容。这种转变让系统从"聪明的自动补全"升级为"可编程的执行器"。

¶1.1 Function Calling — 协议本身

Function Calling 由 OpenAI 在 2023 年中推出，现已成为 Claude、Gemini、Llama-3、Qwen 的通用能力。它是叠加在 chat completion 上的一个类型化协议：应用通过 JSON Schema 声明工具，模型被训练为"要么直接回答，要么发出结构化的工具调用"。整个过程分为五个阶段，没有魔法：

  1. Schema 声明：应用为每个工具注册名称、说明和参数 JSON Schema。
  2. 路由：模型读取用户消息，决定是否回答或调用工具。
  3. 参数生成：约束解码生成符合 schema 的 JSON。
  4. 执行：函数由应用在沙箱中运行，而非模型直接执行。
  5. 结果回灌：执行结果作为 tool 消息追加到对话，模型据此组织最终答复。

常见错误有三个。第一，让模型自行决定是否调用（tool_choice="auto"）；强行调用可能产生奇怪的参数。第二，始终在沙箱中执行——模型完全可能生成 delete_all_users() 调用，只要 schema 允许。第三，工具描述本身就是 prompt：需要不断改写，直到模型每次都能选对工具。

¶1.2 ReAct — 推理与行动的循环

Function Calling 是单轮接口。ReAct（Yao et al., ICLR 2023, arXiv:2210.03629）将其推广为 Thought -> Action -> Observation 的迭代循环，使模型能够分解任务、分支逻辑、重新规划。这是 LangChain、AutoGPT、OpenAI Assistants、Claude tool-use 模式以及 2025 年大多数生产级 Agent 的核心架构。

什么时候用什么？Function Calling 是默认选择：结构化、prompt 简短、单轮往返成本低。ReAct 适合多步规划、基于中间观察分支决策、或从工具失败中恢复的场景——研究综述、数据分析、多跳问答、复杂预订都是典型例子。介于两者之间的任务，现代框架允许将 Function Calling 组织成一张带显式状态的图，这通常是维护性最高的折中方案。

¶2. 代码生成

代码是 LLM 从“有趣 demo”转变为“必备工具”的最显著领域。GitHub 公开数据显示，Copilot 用户接受约三分之一的建议，在基准任务上效率提升约 55%。技术路径清晰：在代码上预训练，在指令上微调，用运行结果评估，再加入一个自我修复回路。

¶2.1 流水线

现代代码-LLM 流水线分为五个环节：自然语言意图、增强后的上下文（已打开文件、仓库符号、测试桩、检索到的 API 文档）、在代码上训练的解码器、编译并运行单测的执行器、将失败信息反馈给模型的自我修复。修复回路是将 65% pass@1 提升到 85% pass@5 的关键。AlphaCode、Reflexion、Code Llama Instruct 均采用了某种形式的修复机制。

¶2.2 模型与基准

标准基准是 HumanEval（Chen et al., arXiv:2107.03374, 2021）：包含 164 道手写 Python 题目，通过执行打分。pass@k 表示 kkk 次采样中至少一次通过全部单测的概率；pass@1\mathrm{pass}@1pass@1 是最严格的单次成功率。两点提醒：HumanEval 属于单文件、纯 Python、题目极短的任务，会高估真实工程性能；此外，该基准已被严重数据污染，2024 年之后的分数需结合 MBPP、LiveCodeBench、SWE-bench Verified 或 CRUXEval 进行交叉验证。

2025 年场景预测：对于纯 Python 任务，DeepSeek-Coder-V2、Qwen2.5-Coder、GPT-4o 处于第一梯队；针对仓库级 PR 任务，SWE-bench Verified 是比 HumanEval 更可靠的代理指标，且榜单完全不同——Claude 3.5/3.7 Sonnet、GPT-4o 以及 Devin 类执行 Agent 占据领先地位，通常表现为较弱基模 + 强执行 Agent 的组合。

¶3. 长上下文建模

标准自注意力在时间和显存上的复杂度都是 O(n2)O(n^2)O(n2)。将上下文从 4K 扩展到 8K 几乎无感；但从 32K 扩展到 128K 就会遇到显存悬崖。解决这个问题有几条互补的路径，通常组合使用。

  • 滑动窗口（Longformer，Beltagy et al., 2020）— 每个 query 只关注最近 www 个 key，开销降到 O(nw)O(nw)O(nw)。局部结构靠多层堆叠逐层传播远端信息。
  • 稀疏 + 全局 token（BigBird、Longformer-global）— 步长稀疏注意力加几个所有 query 都能看到的"全局" token。问答场景中，问题 token 必须看到全文时尤其有用。
  • Infini-attention（Munkhdalai et al., 2024）— 使用一个小局部窗口保证精度，外加一个压缩记忆汇总更早的内容。显存有界，等效上下文无界。
  • 位置编码外推 — RoPE base scaling、NTK-aware 插值、YaRN、LongRoPE — 通过重参数化旋转频率，把在 4K 训练的模型扩展到 128K，几乎不需要微调。
  • 参数高效长上下文微调 — LongLoRA 把 shifted sparse attention 与 LoRA 结合，让 7B 模型在两张 A100 上扩展到 100K。

实际工程中，现代长上下文 LLM 通常这样组合：预训练阶段使用 RoPE 扩展，内核里采用滑动窗口或 GQA，再在长文档混合数据上继续训练。推理时用 FlashAttention-2/3 和 PagedAttention（vLLM）控制常数项。

¶4. 推理模型

2024–2025 年，NLP 的拐点在于测试时算力：不再单纯追求基模的规模，而是让模型在回答前思考得更久。OpenAI 的 o1（2024-09）和 DeepSeek-R1（arXiv:2501.12948, 2025-01）都选择了这条路：先内部生成一段 chain-of-thought，进行打分、必要时回滚，最终只将答案返回给用户。

驱动收益的核心有两个想法。第一，过程监督：训练一个奖励模型，为中间步骤而非最终答案打分（Lightman et al., Let's Verify Step by Step, 2023）。第二，可验证任务上的结果型 RL：在数学和代码这类成功可以自动判定的领域，用验证器作为奖励进行大规模强化学习。DeepSeek-R1 的 "R1-Zero" 配方进一步证明，在足够强的基模上，纯 RL（无需 SFT）就能涌现出推理能力，包括"啊哈时刻"和自我纠错。

代价显而易见。推理模型速度慢、token 费用高——一道 AIME 题可能消耗 1 万到 10 万推理 token。它们还隐藏 CoT（o1 对隐藏推理 token 也收费），带来了新的评估与信任问题。2025 年的正确设计模式是路由：简单 query 发给快模型（GPT-4o、Claude Haiku），复杂推理 query 发给 o1 / R1 / Claude Sonnet thinking 模式。代价与质量的差距常常达到 10–50 倍。

¶5. 安全、对齐与幻觉

有用但不安全的模型无法上线。安全但无用的模型甚至不如没有。对齐是一门工程化学科，目标是精准命中这条狭窄的平衡走廊。

¶5.1 幻觉的分类

幻觉并非单一的失败模式。一个实用的划分（Huang et al., Survey of Hallucination, 2023）如下：

  • 事实性：与可验证的事实不符（例如，“居里夫人获得过菲尔兹奖”）。
  • 忠实性：与给定上下文矛盾——这是 RAG 和摘要场景中的核心问题（例如，“合同写的是 30 天付款”，但实际合同写的是 60 天）。
  • 逻辑/算术：推理步骤无效，但表述流畅得像正确答案。
  • 自一致性：同一对话中前后矛盾。

缓解手段层层叠加：RAG 解决事实性问题（第 10 章）、约束解码 + JSON schema 保证结构忠实性、自一致性多采样 + 多数投票应对算术错误、过程监督 + 推理模型处理逻辑问题、强制引用 prompting 提升可核查性、弃权训练（学会回答“我不知道”）作为最后防线。

¶5.2 对齐 — RLHF、DPO、Constitutional AI

主流对齐方法仍是三步走：在示范数据上做 SFT，在偏好数据上训练奖励模型，再用 PPO 基于奖励进行强化学习。DPO（Rafailov et al., 2023）将后两步合并为一个有监督损失函数，简单且成本低，目前已成为开源生态中的事实标准。Constitutional AI（Bai et al., Anthropic, 2022）通过一份“宪法”规则让模型自我批判，大幅减少大规模安全微调中的人工标注，转而依赖模型生成内容，从而实现安全调优的规模化。

生产环境中，输入过滤器主要拦截越狱攻击和 prompt injection（特别是来自 RAG 上下文的注入，目前是头号攻击向量）；输出过滤器则负责拦截 PII、毒性内容和违规信息。这两个过滤器本身都是小型分类器（如 Llama-Guard-3、ShieldGemma、OpenAI Moderation），比让主模型自行监管更便宜，也更可控。

¶6. 评估

基准是最少欺骗自己的方式。三个维度都很重要，但多数团队在其中两个维度上投入不足。

轴 公开基准 实际衡量的内容
能力 MMLU、GPQA-Diamond、GSM8K、MATH、HumanEval、MBPP、SWE-bench、MT-Bench、Arena-Hard、C-Eval 知识 + 推理 + 生成，普遍存在数据污染
安全 TruthfulQA、ToxiGen、HarmBench、JailbreakBench、BBQ 拒答行为、偏见、越狱鲁棒性
效率 MLPerf-Inference、vLLM bench、MMLU-Pro tokens/answer 吞吐、p50/p95 延迟、每百万 token 成本

三个值得养成的习惯。建立私有评测集：从真实用户中抽取 100–500 条样本，这是与上线效果最相关的指标。采用成对比较（CoT-as-judge 加定期人工抽检）——绝对分数容易漂移，成对胜率则稳定可靠。逐版本回归测试（lm-evaluation-harness、evalplus、inspect_ai）——私有评测下降 2 分应阻塞发布，即使 MMLU 提升也不行。

¶7. 生产部署

这是漂亮 Notebook 撞上现实的地方。参考栈的轮廓是：前端 FastAPI、中段 vLLM、外层 Docker + K8s、全链路 Prometheus 监控。这套架构将是 2025 年大多数生产团队的选择，剩下的只是 Triton、TGI 和 TensorRT-LLM 的口味之争。

¶7.1 推理层

服务上线后，别再用 transformers.generate。原型开发可以，但用于生产就是灾难：没有连续批处理、没有 PagedAttention、KV-cache 管理粗糙。推荐使用 vLLM（开源默认）、TensorRT-LLM（NVIDIA，H100 上最快）或 TGI（Hugging Face，运维最简单）。三者均支持连续批处理、分页 KV-cache、前缀缓存与投机解码。

¶7.2 API 层

FastAPI 是阻力最小的路径：异步、pydantic 校验、自动 OpenAPI、SSE 流式开箱即用。不那么显然但重要的部分是请求卫生——严格限制输入、添加请求 ID、记录结构化日志，并在 GPU 池打满时提供兜底路径。

¶7.3 容器化

K8s 部署时，将 Pod 调度到带 nvidia.com/gpu: 1 的节点，收紧 CPU/内存请求。livenessProbe 探 /healthz，readinessProbe 在模型加载完成后才转 ready（最常见的部署 bug 是在 30–90 秒预热期间就接流量）。

¶7.4 可观测性与 SLO

度量不到的就修不好。最小可用埋点如下：

  • 指标（Prometheus + Grafana）：QPS、p50/p95/p99 延迟、GPU 利用率、显存、队列深度、缓存命中率、tokens/s。
  • 追踪（OpenTelemetry）：每个请求一条 span，子 span 覆盖检索、生成、后处理。
  • 日志（Loki / ELK）：结构化 JSON，每条带请求 ID。
  • 错误（Sentry）：按异常类聚合，速率突增报警。

7-8B 模型在单张 A100 80GB + vLLM + bf16 上的合理上线目标：聚合吞吐 ~1500-2500 tok/s/GPU、p95 首 token < 300 ms、p95 token 间 < 80 ms、~30 并发流。量化到 4-bit AWQ 通常翻倍吞吐，代价是大多数基准上 1–2 分；用私有评测确认是否可接受再切。

¶8. 常见问题

Function Calling、ReAct、图框架怎么选？ 单工具、确定性调用选 Function Calling。多步操作、需要分支重试与中间检查，用 ReAct 或 LangGraph 这类图框架。如果发现自己在 ReAct 的字符串上手写状态机，就该切到图框架——它让控制流显式且可测。

开源模型还是托管 API？ 托管 API 上线快、能力上限高、低请求量成本低。开源自托管在数据主权、微调自由度、高吞吐稳态成本上有优势（7-13B 模型在自有硬件上的盈亏平衡点大约是 5000 万–1 亿 tokens/天）。

真的需要推理模型吗？ 日常聊天、摘要、分类、短代码：不需要，快模型便宜 10-50 倍且够用。竞赛数学、复杂代码、多跳调研、任何"系统 2"任务：需要，差距是质的。按 query 路由即可。

我的 Agent 死循环了。 几乎一定是这三种之一：(1) 工具描述歧义，模型反复试错工具；(2) 忘了 Final Answer 哨兵；(3) 观察过长，把原始目标挤出了上下文。给迭代设上限、给每一步加日志、裁剪观察到只剩相关字段。

最便宜的扩上下文方式是什么？ 如果能微调：RoPE base scaling + 一次小 LongLoRA。如果不能：分块 + RAG（第 10 章）。真正的 100 万 token 上下文很少是正确答案——检索更便宜、更可调试，通常也更准确。

¶系列收官

从空白符分词起步，最终抵达能让推理 Agent 流式输出 token、隐藏在负载均衡器背后的服务。回望这张地图，可以清晰看到四道弧线：

  • 基础（第 1-3 章）——文本转化为向量，序列映射为隐状态。
  • Transformer（第 4-6 章）——注意力取代循环，预训练分化为 BERT 式理解与 GPT 式生成。
  • 适配（第 7-9 章）——prompting、PEFT 与 LLM 内部机制，将预训练模型塑造成可控工具。
  • 应用（第 10-12 章）——RAG 将模型锚定于知识，多模态突破纯文本边界，Agent + 生产工程闭环整条链路，形成可部署系统。

如果这整套系列只能留下三件事，我希望是这三点：架构不如数据和目标重要——Transformer 已经是底牌，差异化体现在预训练数据混合、指令数据与评估方法上。工程纪律优于模型规模——一个带 RAG、评测和护栏的精心调优 8B 部署，能稳定胜过一个粗糙搭建的 70B 服务。前沿变化迅速，但基本功恒久——embedding、注意力、微调、检索、对齐与服务，到 2027 年仍是核心词汇表，只是具体模型名称会迭代一轮。
