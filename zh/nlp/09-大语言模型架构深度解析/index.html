自然语言处理（九）：大语言模型架构深度解析

拆解现代 LLM 的内部结构：Pre-norm + RMSNorm + SwiGLU + RoPE + GQA、KV Cache 机制、FlashAttention 的 IO 调度、稀疏 MoE，以及 INT8/INT4 量化。

Chen Kai November 10, 2025 · 14 分钟 · 6958 字
#NLP#LLM#Transformer#RoPE#Flash Attention#MoE
Read in English →

2017 年的 Transformer 论文里画了一个 block。今天每一款生产级 LLM 还在沿用它的轮廓，但内部几乎每一个零件都换过：post-norm 换成了 pre-norm，LayerNorm 换成了 RMSNorm，GELU 换成了 SwiGLU，正弦位置换成了旋转位置（RoPE），多头注意力变成了分组查询注意力（GQA），稠密 FFN 在某些模型里被稀疏 MoE 替换。更重要的是，主导推理性能的那个数据结构——KV Cache——根本没出现在原论文里。

这篇文章按照"实现/训练/部署时真正起作用的顺序"来串这些改动：先讲现代 decoder block，再讲为长上下文买单的 KV Cache，然后讲位置怎么编（RoPE / ALiBi），讲让 cache 小下来的注意力布局（GQA / MQA），讲让注意力跑得快的 IO 友好内核（FlashAttention），最后讲怎么"长大但不增加每 token 计算量"（MoE）以及"压小但不掉精度"（量化）。

你将学到什么

  • 现代 block 布局：pre-norm、RMSNorm、SwiGLU、RoPE 和 GQA，我会解释每一项替换的原因。
  • KV Cache 机制：如何把注意力前缀从 $O(n^2)$ 的重复计算优化为 $O(n)$ 的摊销成本，以及它在内存上的开销。
  • 位置编码：sinusoidal、RoPE 和 ALiBi，三种回答“这个 token 在哪”的不同方式。
  • 注意力变种：MHA、MQA 和 GQA，用具体数字说明它们在 70B 模型上如何权衡头部多样性与缓存大小。
  • FlashAttention：一种精确的注意力算法，通过 IO 感知调度让数据块留在 SRAM 中，完全避免生成 $n \times n$ 的分数矩阵。
  • MoE：稀疏 top-$k$ 路由，增加总参数量但不增加每个 token 的 FLOPs。
  • 量化：从 FP16 到 INT8 再到 INT4，使用 GPTQ 和 AWQ 的方法，以及精度和显存的实际影响。

前置知识

  • 熟悉基础 Transformer 模块（详见 第四篇 — 注意力机制与 Transformer ）。
  • 了解预训练模型的基本概念（详见 第五篇 — BERT 与预训练模型 ）。
  • 能看懂 PyTorch 代码，掌握基本的线性代数。

三大家族：encoder-only、decoder-only、encoder-decoder

在聊现代架构之前，先想想为什么几乎所有通用 LLM 最后都选择了 decoder-only。其实这三种架构的区别只在注意力 mask 上：

家族Mask预训练目标擅长任务例子
Encoder-only双向掩码语言模型（MLM）语义理解BERT、RoBERTa、DeBERTa
Decoder-only因果（下三角）下一 token 预测（LM）文本生成、上下文学习GPT、LLaMA、Qwen、Mistral
Encoder-Decoder双向 encoder + 因果 decoder + cross-attn去噪 / span corruption序列到序列T5、BART、FLAN-T5

from transformers import AutoModel, AutoModelForCausalLM, AutoModelForSeq2SeqLM

enc  = AutoModel.from_pretrained("bert-base-uncased")            # encoder-only
dec  = AutoModelForCausalLM.from_pretrained("gpt2")               # decoder-only
ed   = AutoModelForSeq2SeqLM.from_pretrained("t5-small")          # encoder-decoder

Decoder-only 能在 scaling 竞赛中胜出有两个原因。首先，所有任务都能归结为“预测下一个 token”，单一目标和数据格式让它扩展起来非常干净利落。其次，因果 mask 让前缀缓存（KV Cache）变得特别高效——而 encoder-decoder 还得额外缓存 cross-attention，encoder-only 更是完全没法生成内容。今天提到“LLM”，基本默认就是指后面会提到改进的 decoder-only 模型。

现代 decoder block

LLaMA 风格的模块（比如 LLaMA、LLaMA-2、Mistral、Qwen、Yi、DeepSeek 等）相比 2017 年的经典 Transformer 块改动了五个地方。单独看每个改动都不算大，但合在一起让模型训练更稳定，不需要 warmup 调参，还能扩展到更长上下文，推理速度也更快。

1. Pre-norm 取代 post-norm。 原版是先残差加和再归一化（x + Sublayer(x) 然后 norm），现代版是先归一化再进子层（x + Sublayer(Norm(x))）。Pre-norm 让残差路径保持干净的恒等映射，深层堆叠时梯度尺度更稳定，也不用再做 Transformer 的学习率 warmup。

2. RMSNorm 取代 LayerNorm。 LayerNorm 是减均值除标准差，RMSNorm 只除以均方根，没有均值，也没有偏置项：

$$ \mathrm{RMSNorm}(x) = \frac{x}{\sqrt{\frac{1}{d}\sum_i x_i^2 + \varepsilon}} \cdot g. $$

每层少一次归约操作，少一组参数，效果没区别。

3. SwiGLU 取代 GELU。 标准 FFN 是 Linear → GELU → Linear，SwiGLU 加了一条门控线性分支：

$$ \mathrm{SwiGLU}(x) = \big(\mathrm{Swish}(W_1 x) \odot (W_3 x)\big) W_2. $$

逐元素乘法给 FFN 引入了乘性交互，同等参数量下困惑度通常能降低 1–2%。为了控制参数量，隐藏层维度缩小 $2/3$。

4. RoPE 取代可学习绝对位置编码。 不再把位置信息加到 embedding 上，而是在注意力计算时旋转 Q 和 K 向量，细节见下一节。

5. GQA / MQA 取代 MHA。 多个 query 头共享一个 KV 头，具体见注意力变体部分。

import torch
import torch.nn as nn
import torch.nn.functional as F

class RMSNorm(nn.Module):
    def __init__(self, d, eps=1e-6):
        super().__init__()
        self.g = nn.Parameter(torch.ones(d))
        self.eps = eps
    def forward(self, x):
        rms = x.pow(2).mean(-1, keepdim=True).add(self.eps).rsqrt()
        return x * rms * self.g

class SwiGLU(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff, bias=False)   # gate
        self.w3 = nn.Linear(d_model, d_ff, bias=False)   # value
        self.w2 = nn.Linear(d_ff, d_model, bias=False)   # down-proj
    def forward(self, x):
        return self.w2(F.silu(self.w1(x)) * self.w3(x))

class LlamaBlock(nn.Module):
    """Pre-norm + RMSNorm + SwiGLU；注意力模块从外部注入。"""
    def __init__(self, d_model, attn, d_ff):
        super().__init__()
        self.norm1 = RMSNorm(d_model)
        self.attn  = attn
        self.norm2 = RMSNorm(d_model)
        self.ffn   = SwiGLU(d_model, d_ff)
    def forward(self, x, **kw):
        x = x + self.attn(self.norm1(x), **kw)
        x = x + self.ffn(self.norm2(x))
        return x

KV Cache：为长上下文买单的数据结构

自回归解码每次生成一个新 token。如果直接计算注意力，新位置需要所有之前的 token 的 K 和 V。每步都重新投影一次，单步复杂度是 $O(n)$，总复杂度就是 $O(n^2)$。KV Cache 的核心思想很简单：这些投影是过去 token 的确定性函数，算一次存下来就够了。

它能成立有两个原因：

  • 模型是因果的，cache 中已经写入的内容不会因为新 token 到来而改变。
  • K 和 V 的投影是输入的线性变换，缓存投影后的张量和重新计算是等价的。

class KVCache:
    """每层的 KV cache，布局：[B, H_kv, T, D_h]。"""
    def __init__(self, B, H_kv, D_h, max_T, device, dtype=torch.float16):
        self.k = torch.empty(B, H_kv, max_T, D_h, device=device, dtype=dtype)
        self.v = torch.empty(B, H_kv, max_T, D_h, device=device, dtype=dtype)
        self.t = 0                                     # 当前长度

    def append(self, k_new, v_new):
        T_new = k_new.size(2)
        self.k[:, :, self.t : self.t + T_new] = k_new
        self.v[:, :, self.t : self.t + T_new] = v_new
        self.t += T_new
        return self.k[:, :, : self.t], self.v[:, :, : self.t]

def attention_step(q_new, cache: KVCache, k_new, v_new, scale):
    """单步解码：q_new 形状 [B, H_q, 1, D_h]。"""
    K, V = cache.append(k_new, v_new)                  # 完整前缀
    scores = (q_new @ K.transpose(-2, -1)) * scale     # [B, H, 1, T]
    attn   = scores.softmax(-1)
    return attn @ V                                    # [B, H, 1, D_h]

代价不小：cache 占用的显存随序列长度线性增长，是解码阶段的主要内存开销。以 LLaMA-2-70B 结构为例（80 层、64 个 KV 头、head_dim=128、fp16），朴素 MHA 的显存需求是：

$$ 2 \cdot 80 \cdot 64 \cdot 128 \cdot 2\text{ B} = 2.6\text{ MB / token}, $$

32K 的上下文光是 cache 就要占用 84 GB 显存——比模型权重还大。这正是 GQA、MQA 和 PagedAttention（vLLM）诞生的背景。

位置编码：sinusoidal、RoPE、ALiBi

自注意力机制本身没有顺序的概念：打乱输入，输出也会跟着打乱。目前主流的三种方法在不同地方注入位置信息。

Sinusoidal 绝对位置（Vaswani 等，2017）。 在输入层把固定的 $\sin/\cos$ 向量直接加到 token embedding 上。位置信息需要通过每一层的线性变换传递下去。这种方法在训练分布内表现不错，但超出训练长度后外推能力很差。

RoPE — 旋转位置编码（Su 等，2021）。 不是加到 embedding 上，而是在计算注意力时旋转 Q 和 K 向量。把 $d$ 维 head 分成 $d/2$ 个二维平面，第 $i$ 个平面的频率定义为 $\theta_i = 10000^{-2i/d}$，位置 $m$ 时将第 $i$ 个平面旋转角度 $m\theta_i$。核心公式如下：

$$ \langle R_m q,\; R_n k \rangle = \langle q,\; R_{n-m} k \rangle, $$

点积只依赖于相对偏移 $n - m$。这解释了为什么 RoPE 能处理比训练长度更长的上下文，也解释了为什么所有现代模型（LLaMA、Qwen、Mistral、Yi、DeepSeek、GPT-NeoX）都选择它。

def precompute_rope(d_head, max_T, base=10000.0, device="cpu"):
    half = d_head // 2
    freqs = 1.0 / (base ** (torch.arange(0, half, device=device).float() / half))
    t = torch.arange(max_T, device=device).float()
    angles = torch.outer(t, freqs)                     # [T, d_head/2]
    return torch.cos(angles), torch.sin(angles)

def apply_rope(x, cos, sin):
    """x: [B, H, T, D_h]。旋转维度对 (i, i+half)。"""
    half = x.size(-1) // 2
    x1, x2 = x[..., :half], x[..., half:]
    cos = cos[: x.size(-2)].to(x.dtype)
    sin = sin[: x.size(-2)].to(x.dtype)
    rot1 = x1 * cos - x2 * sin
    rot2 = x1 * sin + x2 * cos
    return torch.cat([rot1, rot2], dim=-1)

ALiBi — 带线性偏置的注意力（Press 等，2021）。 完全不用位置编码，而是直接在 softmax 前的分数上加一个按头不同的线性惩罚 $-m_h \cdot |i - j|$。$m_h$ 小的头关注全局，$m_h$ 大的头聚焦局部。原论文中 ALiBi 的外推能力比 RoPE 更强，但由于缺乏相对相位信息，在知识密集型长上下文任务上通常不如 RoPE。BLOOM 和 MPT 使用了这种方法。

实际工程中，RoPE 配合 NTK-aware scaling、YaRN、position interpolation 等技巧，可以让训练长度为 4K 的模型通过微调扩展到 32K–128K，已经成为当前的主流做法。

注意力变种：MHA → GQA → MQA

标准多头注意力（MHA）为每个 query 头单独维护一组 K 和 V 投影。KV Cache 的开销与 KV 头的数量成正比，这正是前面提到的 84 GB 内存占用的主要原因。为了优化，有两种折中方案：

  • MQA — 多查询注意力（Shazeer, 2019）： 所有 query 头共享一个 KV 头。这样，KV Cache 缩小了 $H$ 倍，但由于 K/V 投影失去了头多样性，在复杂任务上性能会明显下降。
  • GQA — 分组查询注意力（Ainslie 等, 2023）： 这是 LLaMA-2-70B、Mistral 等大多数新模型采用的中间方案。把 $H_q$ 个 query 头分成 $G$ 组，每组共享一个 KV 头。以 LLaMA-2-70B 为例，$H_q = 64$、$G = 8$，KV Cache 缩小 8 倍，但性能几乎和 MHA 持平。

class GroupedQueryAttention(nn.Module):
    def __init__(self, d_model, n_q_heads, n_kv_heads):
        super().__init__()
        assert n_q_heads % n_kv_heads == 0
        self.h_q  = n_q_heads
        self.h_kv = n_kv_heads
        self.rep  = n_q_heads // n_kv_heads
        self.d_h  = d_model // n_q_heads
        self.wq = nn.Linear(d_model, n_q_heads  * self.d_h, bias=False)
        self.wk = nn.Linear(d_model, n_kv_heads * self.d_h, bias=False)
        self.wv = nn.Linear(d_model, n_kv_heads * self.d_h, bias=False)
        self.wo = nn.Linear(n_q_heads * self.d_h, d_model, bias=False)

    def forward(self, x, cos, sin, cache: KVCache | None = None):
        B, T, _ = x.shape
        q = self.wq(x).view(B, T, self.h_q,  self.d_h).transpose(1, 2)
        k = self.wk(x).view(B, T, self.h_kv, self.d_h).transpose(1, 2)
        v = self.wv(x).view(B, T, self.h_kv, self.d_h).transpose(1, 2)
        q = apply_rope(q, cos, sin)
        k = apply_rope(k, cos, sin)
        if cache is not None:
            k, v = cache.append(k, v)
        # 把每个 KV 头广播到对应的 query 组
        k = k.repeat_interleave(self.rep, dim=1)
        v = v.repeat_interleave(self.rep, dim=1)
        out = F.scaled_dot_product_attention(q, k, v, is_causal=cache is None)
        out = out.transpose(1, 2).contiguous().view(B, T, -1)
        return self.wo(out)

LLaMA-2-70B 形状下的具体数字（fp16，每 token）：

  • MHA，64 个 KV 头 → 2.56 MB/token → 32K 上下文需要 80 GB；
  • GQA-8，8 个 KV 头 → 0.32 MB/token → 32K 上下文需要 10 GB；
  • MQA，1 个 KV 头 → 0.04 MB/token → 32K 上下文需要 1.25 GB。

GQA-8 几乎实现了 MQA 的全部内存节省，同时性能几乎没有损失。这就是为什么最近的开源权重模型都默认使用它。

FlashAttention：数学不变，调度优化

普通的注意力核函数会先计算 $S = QK^\top$，然后把完整的 $n \times n$ 分数矩阵存到 HBM（GPU 主显存）里，在 HBM 上运行 softmax，最后再乘以 $V$。当 $n=8192$、使用 fp16 精度时，单是 $S$ 矩阵每个头每层就占了 128 MB，而这些数据大部分都被反复读写，浪费严重。

FlashAttention（Dao 等，2022）用的是同样的数学公式，但调度方式不同：

  1. 把 $Q$、$K$、$V$ 切分成行或列的小块（tile），确保它们能放进 A100/H100 每个 SM 上约 192 KB 的 SRAM。
  2. 在 SRAM 中一次只计算一个 $S$ 小块。
  3. 使用在线 softmax 技巧：维护一个动态最大值 $m$ 和分母 $\ell$，每个小块直接更新部分输出，完全不需要加载整行数据。
  4. 只把最终结果 $O$ 写回 HBM，完整的 $S$ 始终不会出现。

最终效果是数学上完全一致的注意力（仅浮点运算顺序可能略有不同），但 HBM 流量从 $O(n^2)$ 降到 $O(n)$。当 $n \geq 2048$ 时，实际运行速度提升 2–4 倍，显存占用最多减少 8 倍。FlashAttention-2 进一步优化了 warp 级别的任务分配，性能达到 A100 FP16 峰值的约 70%。

混合专家：容量提升，计算量不增

稠密 Transformer 的大部分参数和计算量都集中在 FFN。MoE 把单个 FFN 替换为 $N$ 个“专家”FFN，再加一个小型路由器，每个 token 都会选出 top-$k$ 个专家来运行：

$$ y = \sum_{i \in \mathrm{TopK}(W_g x)} g_i(x)\, E_i(x),\qquad g(x) = \mathrm{softmax}(W_g x). $$

每 token 的计算量随 $k$ 增长（通常为 2），而不是随 $N$ 增长。所以，8 专家模型的 FFN 参数量大约是稠密模型的 8 倍，但计算量只多了 2 倍。Mixtral 8×7B 总参数量 47B，每 token 只激活约 13B，推理成本相当于 13B 模型，但质量接近 70B 模型。

不过，代价体现在其他地方：

  • 显存占用。 所有专家都需要常驻内存，即使每次只用到 $k$ 个。Mixtral 8×7B 在 fp16 下需要约 94 GB 显存，比稠密 70B 模型还大；用 INT4 压缩后才能降到 24 GB。
  • 负载均衡。 如果不对路由器干预，它会倾向于选择少数几个专家。实际训练中需要加入辅助的负载均衡损失（Shazeer 2017、Switch Transformer），并添加少量噪声鼓励探索。
  • 全互联通信。 多 GPU 训练时，每个 token 需要跑到目标专家上再返回，因此 MoE 对网络拓扑结构非常敏感。

class TopKRouter(nn.Module):
    def __init__(self, d_model, n_experts, k=2):
        super().__init__()
        self.gate = nn.Linear(d_model, n_experts, bias=False)
        self.k = k

    def forward(self, x):
        logits = self.gate(x)                                   # [*, N]
        topv, topi = logits.topk(self.k, dim=-1)
        weights = topv.softmax(-1)                              # 重新归一化
        return weights, topi                                    # [*, k], [*, k]

class SparseMoE(nn.Module):
    def __init__(self, d_model, d_ff, n_experts, k=2):
        super().__init__()
        self.experts = nn.ModuleList(
            [SwiGLU(d_model, d_ff) for _ in range(n_experts)]
        )
        self.router = TopKRouter(d_model, n_experts, k)

    def forward(self, x):
        B, T, D = x.shape
        flat = x.view(-1, D)
        w, idx = self.router(flat)                              # [BT, k]
        out = torch.zeros_like(flat)
        for slot in range(self.router.k):
            for e in range(len(self.experts)):
                mask = idx[:, slot] == e
                if mask.any():
                    out[mask] += w[mask, slot, None] * self.experts[e](flat[mask])
        return out.view(B, T, D)

（生产环境中的实现会通过 grouped GEMM 和容量限制的分桶机制并行调度 token 到专家，上面的循环只是为了清晰展示逻辑，而非追求性能。）

量化：每个权重少几比特

现代 LLM 的权重通常用 BF16 训练（每参数 2 字节）。一个 70B 的模型需要 140 GB 显存，单张 A100 80GB 或 H100 80GB 根本装不下。量化把每个权重换成低比特整数，再加上一个 per-block scale，显存占用大幅减少，精度损失却很小。

对称 INT8。 先选一个 per-tensor（或 per-channel）的 scale $s = \max|w| / 127$，然后存 $\hat w = \mathrm{round}(w / s) \in [-127, 127]$。计算时再恢复 $w \approx s \cdot \hat w$。显存直接减半，而且在 INT8 tensor core 上吞吐量还能翻倍。

def quantize_int8_per_channel(W):
    """W: [out, in]。返回 int8 权重 + 每行的 fp16 scale。"""
    scale = W.abs().amax(dim=1, keepdim=True) / 127.0          # [out, 1]
    Wq = (W / scale).round().clamp(-127, 127).to(torch.int8)
    return Wq, scale.to(torch.float16)

INT4 + GPTQ（Frantar 等，2022）。 如果直接把每个权重独立量化到 4 bit，精度会崩得很厉害。GPTQ 改为按列量化，每量化完一列后，就用一小批校准数据估计的 Hessian 来调整剩余未量化的列，补偿当前列的舍入误差。最终结果是，7B 以上的模型量化到 4 bit 后，perplexity 损失不到 1%。

INT4 + AWQ（Lin 等，2023）。 AWQ 发现，权重通道里只有约 1% 是“显著”的——这些通道由大激活值驱动。只要保护这部分通道（通过 per-channel scaling，而不是留成 fp16），就能恢复绝大部分精度。AWQ 比 GPTQ 快得多，已经成为许多量化开源 checkpoint 的默认选择。

激活值通常还是保留 FP16/BF16，因为它们的分布是重尾的（LLM.int8()，Dettmers 2022 提到，少量“离群特征”承载了大部分激活能量，必须留高精度）。仅对权重做 INT4 量化，是目前推理的最佳平衡点。

组合起来：高吞吐推理

这些组件可以灵活组合。现代的 vLLM 服务栈整合了以下技术：

  • LLaMA 风格的 decoder block（pre-norm + RMSNorm + SwiGLU + RoPE + GQA）。
  • 每层 KV 缓存，但以固定大小的页形式组织（PagedAttention），上下文动态增减时不会产生内存碎片。
  • 预填充和解码阶段都使用 FlashAttention 内核加速。
  • 权重采用 INT4 量化（AWQ / GPTQ），节省显存。
  • 连续批处理：某个请求完成后，空出的槽位立刻被新请求填上，无需等待批中最慢的序列。

常见问题

为什么选择 decoder-only 而不是 encoder-decoder？

decoder-only 模型只要换个提示词，就能变成分类器、翻译工具或者聊天机器人。而 encoder-decoder 模型需要额外的 cross-attention 参数，训练流程也更复杂，生成时还无法利用廉价的前缀缓存。此外，decoder-only 更容易扩展，因为语料中的每个 token 都能直接作为训练信号。

RoPE 和 ALiBi，哪个更好用？

实际工程中几乎都选 RoPE：它能为模型提供真正的相对相位信息，并且通过 NTK/YaRN 缩放可以轻松扩展上下文长度，从 4K 扩展到 32K–128K。ALiBi 的优势在于无需微调就能实现外推，但它的分布内质量会差一些。目前所有主流开源大模型都采用 RoPE。

FlashAttention 会影响模型输出吗？

不会。FlashAttention 是精确的注意力计算，只是浮点归约顺序略有不同。与朴素实现相比，数值差异远低于训练噪声，可以忽略不计。

MoE 是否一定比稠密模型更高效？

计算上更高效，但显存占用更高。如果显存是瓶颈（比如在单张 24GB 显卡上推理），量化的稠密模型通常表现更好。如果显存够用，但 decode 时受限于 FLOPs，MoE 则更有优势。

7B 模型用 INT4 量化会损失多少精度？

使用 GPTQ 或 AWQ，在 ~128 条样本的校准集上进行量化，perplexity 通常只增加不到 2%，大多数下游任务的质量几乎没有明显变化。对于 70B 模型，损失一般小于 1%。但如果模型规模小于 7B，量化带来的负面影响会更加明显。

为什么长上下文场景下 KV Cache 是瓶颈？

KV Cache 的大小公式是 $2 \cdot L \cdot H_{kv} \cdot d_h \cdot T \cdot \text{字节数}$。以 70B 模型为例，32K 上下文使用 MHA 时，KV Cache 占用约 80GB，比模型权重还大。GQA 将其降到约 10GB，PagedAttention 则将碎片率控制在 5% 以内，两者结合才让普通硬件也能支持长上下文推理。
