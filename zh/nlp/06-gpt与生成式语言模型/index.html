自然语言处理（六）：GPT与生成式语言模型

从GPT-1到GPT-4：理解自回归语言建模、解码策略（贪心、束搜索、top-k、top-p）、上下文学习，并用HuggingFace构建聊天机器人。

当你向 ChatGPT 提一个问题，看到一段流畅的多段回答一个 token 接一个 token 流式涌出时，你目睹的其实是一个朴素到惊人的循环：把"目前为止的所有内容"喂给一个 Transformer 解码器，看它输出的词表概率分布，挑一个 token，拼到末尾，再循环。这就是自回归语言模型干的全部事情。神奇的不是这个循环，而是当你把循环背后的网络放大到几千亿参数、用半个互联网训练时，会发生什么。

如果说 BERT（第 5 篇）是"理解之王"，GPT 就是"生成之王"。本文会把 GPT 的整条家谱讲清楚，把自回归解码的机制掰开来看，让你对"采样策略到底意味着什么"有切身感受，最后落到一个能在你笔记本上跑起来的聊天机器人。

你将学到什么

  • 一个仅用解码器的 Transformer 是如何把“预测下一个 token”变成通用人工智能的
  • 因果（掩码）自注意力——这是区分 GPT 和 BERT 的核心设计点
  • 四种经典解码方法：贪心搜索、束搜索、top-k 采样、top-p 采样，以及温度的作用
  • 为什么上下文学习（zero/few-shot）不需要更新参数就能生效
  • 经验上的缩放定律和涌现能力这一奇特现象
  • 如何评估生成文本的质量（BLEU、ROUGE、困惑度），以及这些指标各自的局限性
  • 怎么用 HuggingFace 在 GPT-2 上实现一个多轮对话的聊天机器人

前置知识：第 4 篇（Transformer 架构）、第 5 篇（预训练与 BERT）。

1. 仅解码器 Transformer

最初的 Transformer（Vaswani 等，2017）分为两部分：编码器负责读取源句子，解码器负责生成目标句子。BERT 只保留了编码器，而 GPT 则只保留了解码器，并且做了一个关键改动：每一层自注意力都使用因果掩码，让位置 $i$ 只能关注 $1, 2, \ldots, i$，永远看不到未来。

这个设计为什么如此重要？因为它让训练和推理完全一致。训练时，模型虽然能看到整个句子，但因果掩码会挡住未来的信息，迫使每个位置只能根据过去预测下一个词——这正是生成时的场景。没有训练和推理的差异，也不需要单独的“生成模式”。训练时计算损失的前向传播，直接就是推理时生成下一个 token 的过程。

前向传播流程

假设输入是 $x_1, \ldots, x_t$：

  1. 嵌入：$h^0_i = E_{\text{tok}}(x_i) + E_{\text{pos}}(i)$
  2. 重复 $L$ 次（一个 Transformer block）：
$$ \tilde h = h + \text{MaskedMHA}(\text{LN}(h)), \quad h \leftarrow \tilde h + \text{FFN}(\text{LN}(\tilde h)) $$
  1. 投影：计算 logits $z_i = W_o\,h^L_i$，然后通过 softmax 得到 $P(x_{i+1}\mid x_{\le i}) = \text{softmax}(z_i)$。

因果掩码的具体实现

带掩码的注意力机制和普通的缩放点积注意力几乎一样，只是在 softmax 之前给禁止的位置加上 $-\infty$：

$$ \text{Attention}(Q, K, V) = \text{softmax}\!\left(\tfrac{QK^\top}{\sqrt{d_k}} + M\right) V, \quad M_{ij} = \begin{cases} 0 & j \le i \\ -\infty & j > i \end{cases} $$

经过 softmax 后，$-\infty$ 变成 $0$，所以未来位置对当前结果没有任何贡献。右图展示了这一点：每一行是一个查询位置，每一列是一个被注意的位置，只有下三角区域（过去）是有效的。

训练目标

最大化语料库的对数似然，也就是把每个位置的交叉熵加起来：

$$ \mathcal{L} = -\sum_{i=1}^{n} \log P(x_i \mid x_1, \ldots, x_{i-1}) $$

用这一个损失函数，在 TB 级别的文本上训练，这就是 GPT 的全部故事。

2. 自回归生成，一步步来

推理时，模型每次只生成一个 token。每一步都做三件事：

  1. 前向传播当前序列，只取最后一个位置的 logits。
  2. 把 logits 转换成词表上的概率分布（可以加温度调节）。
  3. 挑一个 token（确定性选择或者采样），追加到序列末尾，重复这个过程，直到遇到 <eos> 或达到长度限制。

下图展示了 prompt "The cat sat on the" 后面的下一个 token 分布：mat 的概率最高，大约占 42%，但其他合理选项（比如 floor、couch、sofa 等）也分到了不少概率。模型是每次都输出同样的内容，还是时不时给你点惊喜，完全取决于你如何从这个分布里采样——这部分内容我会在第 4 节详细讲。

关于速度的一个实用提示。如果直接生成 $T$ 个 token，意味着要跑 $T$ 次前向传播，而每次的计算成本会随着序列长度平方增长——这根本没法用。实际实现中都会用 KV cache：过去 token 的 keys 和 values 只算一次并缓存起来，新 token 只需要和缓存的向量做注意力计算。这样一来，生成的总计算量从 $O(T^3)$ 降到了 $O(T^2)$，这也是真实系统能高效运行的关键原因。

3. GPT 家族：5 年，参数增长近万倍

GPT 系列的故事很简单：把一个简单的想法不断放大，看看会发生什么。

GPT-1（2018 年 6 月）——概念验证

  • 12 层解码器，1.17 亿参数，BooksCorpus 数据集（约 4.5 GB）。
  • 方法：先用语言建模目标预训练，再针对每个下游任务微调一个小任务头。
  • 收获：一个通用的 Transformer 解码器，用原始文本预训练后，能在各种基准测试中击败手工设计的任务专用架构。

GPT-2（2019 年 2 月）——零样本能力初现

  • 15 亿参数，WebText 数据集（约 40 GB，从 Reddit 出站链接抓取）。
  • 新思路：完全跳过微调。只要在 prompt 里描述任务，模型往往就能完成。

GPT-4（2023 年 3 月）——多模态与指令优化

  • 架构和参数量未公开（传闻是万亿参数级别的 MoE 模型）。
  • 同时支持文本和图像输入。
  • 使用 RLHF（基于人类反馈的强化学习）大幅优化了安全性、实用性和指令遵循能力。
  • 律考成绩达到前 10%，AP 考试拿满分 5 分，还能处理早期模型无法应对的长程推理任务。

4. 解码策略

调整解码策略，可以让生成结果从“机械重复、毫无生气”变成“出人意料、紧扣主题”，完全不用重新训练模型。下面我用第 2 节提到的同一个下一 token 分布，把四种经典策略的效果并排展示出来。

def greedy_decode(model, tokenizer, prompt, max_new=100):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    for _ in range(max_new):
        logits = model(ids).logits[:, -1, :]
        nxt = logits.argmax(dim=-1, keepdim=True)
        ids = torch.cat([ids, nxt], dim=1)
        if nxt.item() == tokenizer.eos_token_id:
            break
    return tokenizer.decode(ids[0], skip_special_tokens=True)

优点：确定性强、速度快。
缺点：容易陷入退化循环（比如 "the the the"），生成内容单调、可预测。通常只在需要可复现的调试场景下使用。

4.2 束搜索（Beam Search）

每一步维护 $k$ 条最优的部分序列，按累计对数概率排序（通常加一个长度惩罚，避免偏好短序列）。

def beam_search(model, tokenizer, prompt, beam=5, max_new=100, lp=0.6):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    beams = [(ids, 0.0)]
    for _ in range(max_new):
        cands = []
        for seq, score in beams:
            if seq[0, -1].item() == tokenizer.eos_token_id:
                cands.append((seq, score)); continue
            logp = torch.log_softmax(model(seq).logits[:, -1, :], dim=-1)
            top_lp, top_id = logp.topk(beam, dim=-1)
            for p, idx in zip(top_lp[0], top_id[0]):
                new_seq = torch.cat([seq, idx.view(1, 1)], dim=1)
                cands.append((new_seq, score + p.item()))
        cands.sort(key=lambda x: x[1] / (x[0].size(1) ** lp), reverse=True)
        beams = cands[:beam]
    return tokenizer.decode(beams[0][0][0], skip_special_tokens=True)

优点：似然值比贪心解码高，是机器翻译和文本摘要的核心方法。
缺点：在开放式生成中容易生成平淡、保守的内容——这就是所谓的“束搜索诅咒”（似然值最高的句子往往最无趣）。

def top_k_sample(model, tokenizer, prompt, k=50, T=1.0, max_new=100):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    for _ in range(max_new):
        logits = model(ids).logits[:, -1, :] / T
        top_logits, top_ids = logits.topk(k, dim=-1)
        probs = torch.softmax(top_logits, dim=-1)
        idx = torch.multinomial(probs, 1)
        nxt = top_ids.gather(-1, idx)
        ids = torch.cat([ids, nxt], dim=1)
        if nxt.item() == tokenizer.eos_token_id:
            break
    return tokenizer.decode(ids[0], skip_special_tokens=True)

问题：$k$ 是固定的。如果模型非常自信（某个 token 占了 95% 的概率），top-$k$ 还是会考虑 $k$ 个候选，可能采到不合理的 token；如果模型在几百个 token 上都不确定，$k=50$ 又显得太局限。

4.4 Top-$p$（核采样）

def top_p_sample(model, tokenizer, prompt, p=0.9, T=1.0, max_new=100):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    for _ in range(max_new):
        logits = model(ids).logits[:, -1, :] / T
        sorted_logits, sorted_ids = logits.sort(descending=True)
        probs = torch.softmax(sorted_logits, dim=-1)
        cum = probs.cumsum(dim=-1)
        keep = cum <= p
        keep[..., 0] = True               # 至少保留 top-1
        keep[..., 1:] = keep[..., :-1].clone() | (cum[..., :-1] <= p)
        sorted_logits[~keep] = -float("inf")
        nucleus_probs = torch.softmax(sorted_logits, dim=-1)
        idx = torch.multinomial(nucleus_probs, 1)
        nxt = sorted_ids.gather(-1, idx)
        ids = torch.cat([ids, nxt], dim=1)
        if nxt.item() == tokenizer.eos_token_id:
            break
    return tokenizer.decode(ids[0], skip_special_tokens=True)

上图的 (c) 子图中，核包含 5 个 token，因为前 5 个就覆盖了 85% 的概率。换一个分布，核可能只有 1 个 token（模型非常自信）或 200 个（分布很平）。这种自适应性让 top-$p$ 成为开放式生成的首选策略。

4.5 温度

温度 $T$ 在 softmax 之前对 logits 做缩放：

$$ P_T(w) = \text{softmax}(z / T) $$

上图的 (d) 子图展示了同一组 logits 在 $T = 0.5$ 和 $T = 1.5$ 下的变化。直观来说，$T$ 控制分布的“尖锐程度”：

  • $T \to 0$：分布塌缩到 argmax 对应的 one-hot（等价于贪心解码）。
  • $T = 1$：原始分布（不变）。
  • $T \to \infty$：分布趋于均匀。

经验法则：top-$p = 0.9$ 配合 $T = 0.7$–$0.9$ 是大多数对话和创意写作任务的默认配置。

5. 缩放定律：涌现前的可预测性

2020 年，Kaplan 等人发现了一个规律：测试损失会随着三个因素——模型参数 $N$、数据集大小 $D$、训练算力 $C$——呈干净的幂律下降，前提是这三个因素中没有一个是瓶颈。

$$ L(N) \approx \left(\frac{N_c}{N}\right)^{\alpha_N},\quad L(C) \approx \left(\frac{C_c}{C}\right)^{\alpha_C} $$

实验测得的指数非常小（$\alpha_N \approx 0.076$，$\alpha_C \approx 0.050$）。在对数-对数坐标下，这些关系表现为直线，所以上图两个面板都呈现线性。这带来两个重要结论：

  1. 可以预测大模型的表现。通过 $\le 1$ B 参数的小规模实验，我就能估算出 1750 亿参数模型的损失。这就是 GPT-3 投资背后的底气——团队在花几百万美元买 GPU 之前，就已经大致知道损失曲线会落在哪里。
  2. 损失存在一个不可逾越的下限（图中虚线），也就是数据本身的熵。无论怎么扩大规模，都无法突破这个极限。

2022 年的 Chinchilla 论文进一步完善了这一规律：在固定算力预算下，GPT-3 的训练量其实远远不足。最优的算力分配比例大约是每个参数对应 20 个 token。因此，用 1.4 T tokens 训练的 70 B 模型（Chinchilla），在相同算力下性能超过了用 300 B tokens 训练的 175 B 模型（GPT-3）。如今主流的开源模型（LLaMA、Mistral、Qwen）全都采用了 Chinchilla 式的缩放策略。

6. 涌现能力

缩放定律表明，整体损失会平稳下降。但很多具体任务的表现却不是这样——它们在很长一段时间内都停留在随机水平，直到某个临界点突然跃升到高准确率。

图中展示了这种现象的定性特征：情感分类（蓝色）从小规模开始就稳步提升；少样本上下文学习、三位数算术、思维链推理（紫色、橙色、绿色）则在达到某个关键模型规模之前几乎保持随机水平，随后迅速攀升。Wei 等人（2022）在 BIG-Bench 基准中整理了 137 个类似的任务。

关于涌现是否“真实存在”，还是仅仅是测量方式（比如离散任务的精确匹配准确率）导致的假象，学界仍有争论。Schaeffer 等人（2023）的研究表明，换成连续指标后，部分涌现曲线确实会变得平滑。但从实际操作的角度看，事实很清楚：小规模时模型做不了这些任务，而当参数量达到 $10^{10}$ 到 $10^{11}$ 之间时，它能完成了。规划技术路线时，这一点必须考虑进去。

7. 上下文学习

GPT 类大模型最让人惊讶的地方是：我只需要在 prompt 里给几个例子，就能教会它完成新任务。不需要反向传播，不需要优化器，也不用更新参数——模型完全靠“阅读”来适应。

为什么能奏效？

背后的机制仍然是研究热点，目前主流的解释有以下几种：

  • 大规模模式匹配。预训练阶段，模型见过互联网上的无数输入输出格式，prompt 激活了其中匹配的模板。
  • 隐式梯度下降。Garg 等（2022）的研究表明，在线性回归这类简单场景中，Transformer 在前向传播过程中可以通过上下文示例模拟一步梯度下降。模型越大，模拟的步数越多。
  • 贝叶斯隐任务推断。Xie 等（2022）将 ICL 解释为从示例中推断出一个隐藏的“任务变量”，然后基于这个变量生成结果。
  • 规模带来的相变效应。ICL 只有在参数量达到几十亿以上时才有效——这与涌现现象的描述是一致的。

def few_shot_prompt(task, examples, query):
    parts = [task, ""]
    for inp, out in examples:
        parts.append(f"Input: {inp}\nOutput: {out}\n")
    parts.append(f"Input: {query}\nOutput:")
    return "\n".join(parts)

以下是几条经过验证有效的建议：

  1. 说清楚任务。开头用一句话明确说明任务是什么。
  2. 选择代表性样例。覆盖测试时可能遇到的各种情况。
  3. 保持格式一致。分隔符、大小写和标签都要统一。
  4. 推理任务用思维链。让模型“一步步思考”，或者在少样本示例中直接展示推理过程。这种方法在数学和逻辑类任务上能显著提升准确率。

8. 评估生成文本

没有完美的指标，但搞清楚该用哪个近似指标、它有哪些局限，这一点很重要。

BLEU（翻译）

BLEU 通过对比生成文本和参考文本的 n-gram 精确率，并加上一个长度惩罚来计算得分：

$$ \text{BLEU} = \text{BP}\cdot\exp\!\left(\sum_{n=1}^{N} w_n \log p_n\right) $$

    return sentence_bleu([reference.split()], generated.split(),
                         smoothing_function=SmoothingFunction().method1)

在翻译任务中，BLEU 和人工评分的相关性很强；但在开放式生成任务中，相关性就很弱了。

def rouge(generated: str, reference: str) -> dict:
    scorer = rouge_scorer.RougeScorer(["rouge1", "rouge2", "rougeL"], use_stemmer=True)
    s = scorer.score(reference, generated)
    return {k: v.fmeasure for k, v in s.items()}

def perplexity(model, tokenizer, text: str) -> float:
    enc = tokenizer(text, return_tensors="pt")
    with torch.no_grad():
        loss = model(**enc, labels=enc.input_ids).loss
    return torch.exp(loss).item()

困惑度是训练过程中很有用的内在信号，但低困惑度并不代表聊天机器人就一定好。模型可能只是擅长预测常见模式，却仍然会胡编乱造。

from transformers import GPT2LMHeadModel, GPT2Tokenizer
import torch

class ChatBot:
    def __init__(self, name: str = "gpt2") -> None:
        self.tokenizer = GPT2Tokenizer.from_pretrained(name)
        self.model = GPT2LMHeadModel.from_pretrained(name).eval()
        if self.tokenizer.pad_token is None:
            self.tokenizer.pad_token = self.tokenizer.eos_token

    def respond(self, user: str, max_new: int = 100,
                temperature: float = 0.8, top_p: float = 0.9) -> str:
        prompt = f"User: {user}\nAssistant:"
        ids = self.tokenizer.encode(prompt, return_tensors="pt")
        with torch.no_grad():
            out = self.model.generate(
                ids,
                max_new_tokens=max_new,
                do_sample=True,
                temperature=temperature,
                top_p=top_p,
                no_repeat_ngram_size=3,
                pad_token_id=self.tokenizer.eos_token_id,
            )
        text = self.tokenizer.decode(out[0], skip_special_tokens=True)
        return text.split("Assistant:")[-1].strip()

print(ChatBot().respond("What is machine learning?"))

class MultiTurnBot(ChatBot):
    def __init__(self, name: str = "gpt2") -> None:
        super().__init__(name)
        self.history: list[str] = []

    def respond(self, user: str, max_turns: int = 5, **gen_kwargs) -> str:
        # 只保留最近几轮对话，确保不超过 1024 token 的上下文限制
        recent = self.history[-2 * max_turns:]
        prompt = "\n".join(recent + [f"User: {user}", "Assistant:"])
        ids = self.tokenizer.encode(prompt, return_tensors="pt")
        with torch.no_grad():
            out = self.model.generate(
                ids,
                max_new_tokens=gen_kwargs.get("max_new", 100),
                do_sample=True,
                temperature=gen_kwargs.get("temperature", 0.8),
                top_p=gen_kwargs.get("top_p", 0.9),
                no_repeat_ngram_size=3,
                pad_token_id=self.tokenizer.eos_token_id,
            )
        reply = self.tokenizer.decode(out[0], skip_special_tokens=True)
        reply = reply.split("Assistant:")[-1].split("User:")[0].strip()
        self.history += [f"User: {user}", f"Assistant: {reply}"]
        return reply

10. GPT 和 BERT 的对比，一张表搞定

维度BERTGPT
架构编码器，双向自注意力解码器，因果自注意力
预训练Masked LM + 下一句预测因果语言建模
上下文左右都看只看左边（过去）
擅长任务分类、NER、问答、检索文本生成、对话、代码、创意写作
适配方式每个任务加一个任务头微调Prompt（零样本/少样本）或微调
规模效应参数量超过约 1B 后收益递减收益持续增长，能力逐渐涌现

一个简单的心智模型：BERT 是搜索引擎，GPT 是写作者。如果需要把输入转成向量表示，用 BERT 系列；如果需要生成输出文本，用 GPT 系列。

11. 必须坦诚面对的局限性

  • 幻觉问题。模型对不知道的内容，默认会生成流畅但错误的文本。解决办法包括：检索增强（第 10 篇）、工具使用、以及经过校准的拒答训练。
  • 上下文长度。GPT-2 最大支持 1024 个 token，GPT-3 提升到 2048，GPT-4 先是扩展到 32K，后来又达到 128K，现在 1M 已经逐渐成为主流。但长上下文代价高昂，而且中间部分的内容容易被遗忘（“lost in the middle”）。
  • 算力开销。一个 1750 亿参数的模型，单次前向传播就需要几百 GB 的 GPU 显存。在模型的整个生命周期中，推理成本远远高于训练成本。
  • 训练数据偏差。模型会复现训练数据中的统计规律，包括各种刻板印象。
  • 可控性不足。如果要严格控制风格、格式或事实准确性，必须引入额外的技术手段，比如 system prompt、约束解码、RLHF 或后处理过滤器。

12. 核心要点

  • GPT 模型很简单，就是一个只有解码器的 Transformer，带因果掩码，专门用来预测下一个 token。简单就是它的亮点。
  • 规模是关键：参数、数据和算力三者结合，能让损失呈幂律下降，同时还会带来意想不到的能力飞跃。
  • 上下文学习让一个固定模型通过 prompt 就能完成上千种任务，大多数情况下根本不需要微调。
  • 解码策略是个灵活工具：对话和创意写作推荐用 top-$p$ + 温度 0.7–0.9；翻译和需要稳定输出的场景则用贪心搜索或束搜索。
  • BERT 负责理解，GPT 负责生成。两者联手覆盖了现代 NLP 的全领域。接下来第 7 到 12 篇会深入讲解如何高效使用 GPT 类模型（提示工程、微调、RAG、多模态等）。
