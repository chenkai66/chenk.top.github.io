自然语言处理（八）：模型微调与PEFT

深入参数高效微调：LoRA 为什么用低秩更新就够、QLoRA 把 7B 模型塞进 6GB 显存的内存账本、Adapter 与 Prefix-Tuning 的取舍，以及生产环境怎么选。

Read in English →

2020 年微调一个 70 亿参数的语言模型还得排预算：八张 A100、几天时间，再加一个会调梯度检查点的工程师。2024 年，研究生在笔记本上就能跑。中间这段距离，几乎完全由 Hu 等人 2022 年的 LoRA 论文，以及 Dettmers 等人 2023 年的 QLoRA 论文铺平。

但这不只是工程优化。参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）重新定义了"拥有一个模型"这件事。过去每个任务都得有一份完整的权重文件，现在只需要一个冻结的基座，加上一个目录里几十兆的小适配器。换任务变成了换文件，N 个领域的服务从 O(N) 变成 O(1) 个基座加 N 个 ε。

本文从第一性原理重建 PEFT。先回答全量微调能解决什么、不能解决什么，再推导 LoRA 的低秩假设，把 QLoRA 让 7B 模型塞进 6GB 显存的算账过程拆开，最后落到工程选择：用哪种方法、秩多少、改哪些模块。

你将学到

  • 为什么 在 LLM 时代全量微调是浪费资源——过参数化问题和内在秩假说
  • LoRA 的原理：分解公式 $\Delta W = BA$、为什么 $B$ 初始化为零、$\alpha/r$ 缩放如何调整有效学习率
  • QLoRA：NF4 量化、双重量化、分页优化器，以及精确的内存占用分析
  • Adapter 和 Prefix-Tuning：它们在 Transformer 模块中的位置，适用场景与局限性
  • 实际应用中的选择：如何选择秩、目标模块有哪些、多 LoRA 服务的实现、通过指令微调和 RLHF 进行对齐

前置知识

  • 了解 GPT 风格解码器（第 6 部分 ）
  • 掌握 PyTorch 基础，理解 GPU 内存的基本概念（优化器状态、激活值、梯度）

1. 为什么不直接全量微调？

显存账单

全量微调意味着所有参数都要更新，每个梯度都得存下来，AdamW 还要为每个参数额外维护两个 fp32 缓冲区。以一个 7B 模型为例，在混合精度训练下，单步显存占用是这样的：

项目每参数字节7B 模型
权重（fp16）214 GB
梯度（fp16）214 GB
AdamW 状态（fp32 m + v）856 GB
激活值（取决于序列长度、batch 大小）—8–20 GB
总计—约 95 GB

也就是说，还没写一行训练代码，至少需要两张 A100-80GB 显卡。这笔账清楚地说明了，PEFT 不是什么锦上添花的优化——对大多数从业者来说，它是能用上 7B 以上模型的唯一办法。

内在秩假设

还有更深层次的原因。Aghajanyan 等人（2020）和 LoRA 论文都提到：微调引入的权重变化具有非常低的内在秩。如果预训练模型需要的改动本来就局限在一个低维子空间里，那么训练完整的 $d \times k$ 矩阵不仅昂贵，还选错了搜索空间——应该直接在低秩流形上找解。

实际测试表明，对 175B 模型进行下游任务微调时，只需训练大约 200 个参数方向就能达到相同效果（Aghajanyan et al., 2020）。这就是 LoRA 方法的核心思想。

冻结微调——弱基线

在 PEFT 出现之前，最简单的省钱方法是冻结模型主体，只训练分类头，或者只解冻顶部几层：

2. LoRA：低秩适配

分解方式

对于一个冻结的权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$，LoRA 学习一个加性更新，这个更新被参数化为两个瘦矩阵的乘积：

$$W = W_0 + \Delta W, \qquad \Delta W = \frac{\alpha}{r}\, B A, \qquad B \in \mathbb{R}^{d \times r},\ A \in \mathbb{R}^{r \times k},\ r \ll \min(d, k).$$

前向计算变为 $h = x W_0^\top + \frac{\alpha}{r} (x A^\top) B^\top$。原始计算 $x W_0^\top$ 不变，LoRA 分支增加了一个秩为 $r$ 的修正。

参数数量

假设 $d = k = 4096$（LLaMA-7B 的隐藏层维度），$r = 8$：

  • 原始 $W_0$：$d \cdot k = 16{,}777{,}216$ 个参数
  • LoRA 的 $A, B$：$r \cdot (d + k) = 65{,}536$ 个参数
  • 减少：每个矩阵的可训练参数减少 256 倍

将 LoRA 应用到 32 层中的所有四个注意力投影上，总共训练约 800 万个参数——占 70 亿总参数的 0.12%。

两个关键设计

初始化不对称性。 $B$ 初始化为零，$A$ 初始化为小高斯分布。因此在训练的第一步，$BA = 0$，模型的行为与预训练版本完全一致。训练从一个已知的良好起点开始，不需要额外的 warmup 操作。

$\alpha/r$ 缩放因子。 这不是一个装饰性常数。如果固定 $\alpha$ 并增大 $r$，更新的幅度会随之增加（更多方向、更大的组合范数），迫使重新调整学习率。$\alpha/r$ 因子将两者解耦：只需一次设定 $\alpha$（通常取 $\alpha = 2r$），通过调整 $r$ 来控制容量，有效步长基本保持不变。

import torch
import torch.nn as nn
import torch.nn.functional as F

class LoRALinear(nn.Module):
    """为冻结的 nn.Linear 添加一个可训练的低秩更新。"""

    def __init__(self, base: nn.Linear, r: int = 8, alpha: int = 16,
                 dropout: float = 0.0):
        super().__init__()
        self.base = base
        for p in self.base.parameters():
            p.requires_grad = False

        in_f, out_f = base.in_features, base.out_features
        # A: (r, in_features) — 小高斯初始化
        self.lora_A = nn.Parameter(torch.empty(r, in_f))
        nn.init.kaiming_uniform_(self.lora_A, a=5 ** 0.5)
        # B: (out_features, r) — 零初始化，分支初始不激活
        self.lora_B = nn.Parameter(torch.zeros(out_f, r))

        self.scaling = alpha / r
        self.dropout = nn.Dropout(dropout) if dropout > 0 else nn.Identity()

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # 冻结路径
        out = self.base(x)
        # 低秩分支：x @ A^T @ B^T，缩放后相加
        lora_out = self.dropout(x) @ self.lora_A.T @ self.lora_B.T
        return out + self.scaling * lora_out

    @torch.no_grad()
    def merge(self) -> None:
        """将 LoRA 增量合并到基座权重中，推理时无额外开销。"""
        delta = self.scaling * (self.lora_B @ self.lora_A)
        self.base.weight.data += delta
        self.lora_B.zero_()

merge() 方法是 LoRA 在推理时零成本的关键：训练完成后，将 $\Delta W$ 合并到 $W_0$ 中，模型结构与原始模型完全一致——没有额外的矩阵乘法，也没有延迟。

3. QLoRA：把基座模型压缩到 4 比特

LoRA 解决了可训练参数的问题，但冻结的权重仍然以 fp16 存储，占用了大量显存。QLoRA 就是为了解决这个问题。

QLoRA 的三大创新点

1. NF4（NormalFloat 4 比特）量化。
普通的 4 比特整数量化效率低，因为大语言模型的权重分布并不均匀，而是接近高斯分布。NF4 根据标准正态分布选取了 16 个等概率的量化级别，理论上实现了最优的信息覆盖。每个块通常包含 64 个权重，并配有一个 fp16 的缩放因子。

2. 双重量化。
每个块都需要一个缩放常数。对于 7B 参数的模型，块大小为 64 时，大约需要 1.1 亿个常数，占用约 0.44 GB 内存。QLoRA 把这些常数本身再量化到 8 比特，每参数节省约 0.4 比特。虽然单个看不多，但在 7B 规模下效果显著。

3. 分页优化器。
长序列会导致激活内存激增。Paged AdamW 借助 NVIDIA Unified Memory 把优化器状态分页到 CPU 内存，需要时再加载回 GPU。这样牺牲了一些吞吐量，但能训练原本会爆显存的长序列。

代价与收益

相比 fp16 LoRA，QLoRA 在大多数基准测试中性能下降 1–2 个百分点，但基本恢复了全量微调的效果。而显存的节省非常显著：

70B 模型全量微调需要近 1 TB 显存，基本只有顶级实验室才能做到。QLoRA 把它压缩到了单张 80 GB 的 GPU 上。这直接决定了一个项目是“只能研究”还是“可以落地”。

4. Adapter：最早的 PEFT 方法

Houlsby 等人在 2019 年提出了 Adapter，比 LoRA 早了两年。核心思想很简单：在 Transformer 的每个子层后面加一个小瓶颈模块。

$$\text{Adapter}(x) = x + W_{\text{up}}\, \sigma(W_{\text{down}}\, x), \qquad W_{\text{down}} \in \mathbb{R}^{m \times d},\ W_{\text{up}} \in \mathbb{R}^{d \times m},\ m \ll d.$$

$W_{\text{up}}$ 初始化为零，这样 Adapter 刚开始就是一个恒等映射——这个技巧和 LoRA 的 $B$ 一样。

class Adapter(nn.Module):
    def __init__(self, d_model: int, bottleneck: int = 64):
        super().__init__()
        self.down = nn.Linear(d_model, bottleneck)
        self.up = nn.Linear(bottleneck, d_model)
        nn.init.zeros_(self.up.weight)
        nn.init.zeros_(self.up.bias)
        self.act = nn.GELU()

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return x + self.up(self.act(self.down(x)))

Adapter 和 LoRA 的对比

AdapterLoRA
位置残差路径内部权重矩阵旁边
推理开销每块多一次矩阵乘（约 5–10% 延迟）调用 merge() 后无开销
可组合性多个 Adapter 串联直接叠加 LoRA 增量
多任务服务每个任务一个 Adapter每个任务一个 LoRA，支持批量混合
典型大小基础模型的 0.5–3%基础模型的 0.1–1%
适合场景推理图稳定的场景对延迟要求严格的 LLM 服务

LoRA 凭借推理零开销的优势，抢走了 Adapter 的大部分市场。不过，Adapter 在需要显式模块化组合的场景中仍然很有吸引力。比如在多语言任务中，AdapterFusion（Pfeiffer 等，2021）可以在推理时动态融合多个语言专用的 Adapter。

5. 基于提示的 PEFT：调整输入，而非权重

一个全新的思路：模型完全冻结，只学习该给它喂什么数据。

Prompt-Tuning（Lester et al., 2021）

在输入序列前拼接 $n$ 个可训练的嵌入向量。方法就这么简单。用 11B 的 T5 模型时，Prompt-Tuning 在 SuperGLUE 上的表现能追平全量微调——但这只有在大模型上才成立。对于小于 1B 的模型，Prompt-Tuning 的效果差得远。

Prefix-Tuning（Li & Liang, 2021）

思路类似，但应用到每一层 Transformer 的注意力机制：学习键和值的前缀矩阵 $P_K, P_V$，在计算注意力时将它们拼接到前面：

$$\text{Attention}(Q,\ [P_K; K],\ [P_V; V]).$$

因为每层都有自己的前缀，模型可以比单一输入提示更深入地改写计算过程。

P-Tuning v2（Liu et al., 2022）

去掉了 Prefix-Tuning 中使用的重参数化 MLP（因为它在不同规模的模型上表现不稳定），改为均匀分布深层提示。这是第一个在所有规模（包括小模型）的 NLU 任务上都能追平全量微调的提示类方法。

什么时候用这些方法？

  1. 需要极致的参数效率——每个任务只需几 KB。
  2. 想在一个服务化的模型背后挂载多个提示。

不过，在生成类任务上，提示类方法通常不如 LoRA 表现好。

6. 方法怎么选

一个实用的决策树：

                    ┌─ 要顶级分数？     → 全量微调（如果显存够用）
                    │
显存够不够？  ──────┤
                    │  够 → LoRA r=16 改 q/k/v/o
                    └─ 不够 → QLoRA r=16（NF4 + 分页 AdamW）

一个基座服务多个任务？ → LoRA（能合并、能切换）

生成任务延迟要求高？   → LoRA 合并后推理
编码器 NLU 任务要省参数？→ P-Tuning v2 或 BitFit
模块化组合场景？       → Adapter（加 AdapterFusion）

秩的选择

先设 $r = 16$。从右图能看出，大多数任务的收益拐点在 $r = 8$ 到 $r = 32$ 之间：

  • 简单分类：$r = 8$ 就够了
  • 代码生成、推理：$r = 32$ 或 $r = 64$ 更好
  • 领域适配（医疗、法律）：通常 $r = 16$–$32$ 比较合适

拿不准就试 $r \in \{8, 16, 32\}$，挑能满足留出集目标指标的最小值。

7. 对齐：指令微调与 RLHF

PEFT 是工具，对齐才是我用它来实现的目标。现代大语言模型的后训练主要分为两个阶段：

有监督指令微调
用 (instruction, response) 数据对基础模型进行微调，让它学会理解并响应人类编写的提示。质量远比数量重要：1K 到 10K 条精心挑选的样本，往往能胜过 10 万条众包数据（Zhou 等人 2023 年的 LIMA 论文展示了 1K 条样本就能让 65B 参数模型表现得相当不错）。

class RewardModel(nn.Module):
    def __init__(self, base):
        super().__init__()
        self.base = base
        self.head = nn.Linear(base.config.hidden_size, 1)

    def forward(self, input_ids, attention_mask=None):
        out = self.base(input_ids=input_ids, attention_mask=attention_mask)
        last = out.last_hidden_state[:, -1, :]
        return self.head(last).squeeze(-1)

常见问题

什么时候适合全量微调？

算力充足、有 $\geq$10 万条高质量样本、并且每个小数点的性能提升都很关键时——比如基座模型厂商推出旗舰指令模型。其他情况下，用 LoRA 或 QLoRA 就够了。

LoRA 的秩怎么选？

从 16 开始试。如果欠拟合（训练损失降不下去），就提高到 32 或 64。如果过拟合且数据量少，就降到 4–8。对于分类任务，8 通常已经足够。

改哪些模块？

q_proj 和 v_proj 能带来 80% 的效果提升。再加 k_proj 和 o_proj 拿下剩余的部分。FFN 的三个模块（gate/up/down_proj）只在生成任务占比高且预算允许时添加，因为可训练参数会增加 3 倍。

LoRA 影响推理速度吗？

merge 后完全不影响。merge 前，每层多一次很小的矩阵乘操作——可以忽略，但确实存在。

QLoRA 质量下降多少？

在标准 benchmark 上，相比 fp16 LoRA 通常下降 1–2 个点，基本在误差范围内。对绝大多数人来说，显存节省的价值远超这点损失。

需要多少指令数据？

LIMA 的研究表明，1K 条精心挑选的样本就能让一个强基座模型变成连贯的助手。实际使用中，1K–10K 条高质量样本是底线；质量比数量重要得多。

PEFT 方法能组合吗？

可以。LoRA + Prompt-Tuning 是有文献支持的组合，QLoRA 本身就是一个栈（4 比特基座 + LoRA + 分页优化器）。Adapter 和 LoRA 同时用在一个模型里则很少见。

9. 一份具体的微调配方（7B 上的 LoRA）

这些超参是我实际用过并且上线验证过的，真正影响效果。

数据量。在 7B 模型上用 LoRA，1000 到 5000 条高质量的指令-响应对就能拿到大部分收益。如果少于 500 条，不如直接用 few-shot 提示。超过 20000 条，除非任务特别复杂，否则基本是花冤枉钱。

rank r。默认值 r=8。对于代码生成、多步推理这种复杂任务，可以提到 r=16 或 r=32。r 加倍会让参数量翻倍，但训练时间几乎不变。r 超过 64 很少有帮助，反而容易过拟合。

alpha。设置 alpha = 2 * r，比如 r=8 时 alpha=16。alpha/r 是 LoRA 路径上的等效学习率缩放系数。固定为 2 是一个可靠的默认值。

目标模块。如果显存够用，LoRA 应该同时加到 q_proj, k_proj, v_proj, o_proj 和 gate_proj, up_proj, down_proj 上。原始论文只加到 attention 模块，在指令微调中会丢掉不少分数。实测显示，attention 加 MLP 在 Open LLM Leaderboard 的任务上能多得 2 到 4 分。

学习率。LoRA 的初始学习率设为 2e-4 比较合适。全量微调一般用 2e-5，而 LoRA 可以高 10 倍，因为只有少量参数在更新。使用 cosine 学习率调度，并设置 3% 的 warmup。

batch size 和梯度累积。等效 batch size 控制在 32 到 64 条序列之间。单卡 A100 80GB，序列长度 2048 时，micro-batch 通常只能塞下 2 到 4 条，靠梯度累积达到等效 batch。

epoch 数。跑 1 到 3 轮就够了。盯着 eval loss，通常第 2 轮就会降到最低点。小指令数据集上跑到 5 轮以上，基本都会过拟合。

序列打包。把多个短样本打包成一条 2048-token 的序列，注意正确设置 attention mask。相比 padding，吞吐量能提升 2 到 3 倍。transformers 的 SFTTrainer 现在原生支持这个功能。

5000 条数据、7B 基座、单卡 A100，整轮训练大约需要 2 到 4 小时，租卡成本 $8 到 $12。

10. LoRA 悄悄翻车的地方

我见过三个让团队踩坑的失败模式，分享一下。

失败 1：稀有 token 长尾部分的灾难性遗忘
LoRA 对基座模型的保留效果比全量微调好，但还是会改变输出分布。比如，指令微调一个代码模型后，我发现像 Erlang、Haskell 这些稀有编程语言的困惑度（perplexity）飙升了 30% 以上，而英文指令跟随能力却提升了。解决办法很简单：在 SFT 数据中加入 5%-10% 的“基座风格”数据做回放。

失败 2：非英语输出质量下降
用英文为主的 SFT 数据集做 LoRA 和指令微调时，中文、韩文、日文的输出质量会明显下降，即使基座模型本身是多语言的也没用。问题出在 LoRA 学到了偏向英文的回答模式。解决方法是确保 SFT 数据中至少包含 20% 的目标语言样本。

失败 3：合并后的 LoRA 不等于推理时的 LoRA
当你把 LoRA 权重合并到基座模型中准备部署（model.merge_and_unload()），在 4-bit 量化场景下，合并后的模型性能可能会显著低于未合并的版本。这是因为合并扰动了基座权重，而量化重新取整放大了这些变化。解决办法有两个：要么推理时保持 LoRA 分离（稍微增加一点延迟），要么在合并后用一个预留的数据集做量化校准。
