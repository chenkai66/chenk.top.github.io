自然语言处理（八）：模型微调与PEFT

九月 12, 2025 Updated 5月 06, 2026 — min read — words

深入参数高效微调：LoRA 为什么用低秩更新就够、QLoRA 把 7B 模型塞进 6GB 显存的内存账本、Adapter 与 Prefix-Tuning 的取舍，以及生产环境怎么选。

2020 年，微调一个 70 亿参数的语言模型需要精打细算：八张 A100、几天时间，外加一位懂梯度检查点的工程师。到了 2024 年，研究生在笔记本上就能完成。这中间的进步，几乎完全归功于 Hu 等人 2022 年的 LoRA 论文，以及 Dettmers 等人 2023 年的 QLoRA 论文。

这不仅是工程优化。参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）重新定义了"拥有一个模型"的意义。过去每个任务都需要一份完整的权重文件，现在只需要一个冻结的基座，加上一个目录里几十兆的小适配器。切换任务变成了更换文件，N 个领域的服务从 O(N) 变成 O(1) 个基座加 N 个 ε。

本文从第一性原理出发重建 PEFT。先分析全量微调能解决什么、不能解决什么，再推导 LoRA 的低秩假设，拆解 QLoRA 如何让 7B 模型塞进 6GB 显存的计算逻辑，最后落到工程选择：用哪种方法、秩设多少、调整哪些模块。

¶1. 为什么不直接全量微调？

¶一笔账

全量微调需要为所有参数计算梯度，AdamW 还要为每个参数额外存储两个 fp32 状态。对于一个 7B 模型，在混合精度下，单步训练的显存占用如下：

项目 每参数字节 7B 模型
权重（fp16） 2 14 GB
梯度（fp16） 2 14 GB
AdamW 状态（fp32 m + v） 8 56 GB
激活（依赖序列长度、batch） — 8–20 GB
合计 — 约 95 GB

这意味着，还没写第一行训练代码，至少需要两张 A100-80G 显卡。这笔账清楚地说明，PEFT 不是“锦上添花的优化”——对绝大多数从业者来说，它是接触 7B 以上模型的唯一途径。

¶内在秩假说

更深一层的原因来自 Aghajanyan 等人（2020）和 LoRA 原论文的观察：微调引入的权重变化具有极低的内在秩。如果预训练模型所需的改动本身就位于一个低维子空间中，那么训练完整的 d×kd \times kd×k 矩阵不仅昂贵，还使用了错误的假设空间进行搜索——应该直接在低秩流形上寻找解。

实证研究表明，对 175B 模型在下游任务上的微调，仅需大约 200 个参数方向即可达到匹配效果（Aghajanyan et al., 2020）。这正是 LoRA 方法的核心思想。

¶冻结微调——一个弱基线

在 PEFT 出现之前，最朴素的省钱方法是冻结主体部分，只训练分类头，或者仅解冻顶层几层：

这种方法在 BERT 类编码器上做分类任务尚可，但生成任务表现不佳：被训练的层数太少，无法将上游表示重新引导到任务所需的方向。PEFT 方法则不同——它在网络各处插入可训练参数，但总数依然极少。

¶2. LoRA：低秩适配

¶分解形式

对于冻结的权重矩阵 W0∈Rd×kW_0 \in \mathbb{R}^{d \times k}W0​∈Rd×k，LoRA 学习一个加性更新，将其参数化为两个瘦矩阵的乘积：

W=W0+ΔW,ΔW=αr BA,B∈Rd×r, A∈Rr×k, r≪min⁡(d,k).W = W_0 + \Delta W, \qquad \Delta W = \frac{\alpha}{r}\, B A, \qquad B \in \mathbb{R}^{d \times r},\ A \in \mathbb{R}^{r \times k},\ r \ll \min(d, k). W=W0​+ΔW,ΔW=rα​BA,B∈Rd×r, A∈Rr×k, r≪min(d,k).

前向计算变为 h=xW0⊤+αr(xA⊤)B⊤h = x W_0^\top + \frac{\alpha}{r} (x A^\top) B^\toph=xW0⊤​+rα​(xA⊤)B⊤。原始部分 xW0⊤x W_0^\topxW0⊤​ 保持不变，LoRA 分支增加一个秩为 rrr 的修正。

¶数一下参数

假设 d=k=4096d = k = 4096d=k=4096（LLaMA-7B 的隐层维度），r=8r = 8r=8：

  • 原始 W0W_0W0​：d⋅k=16,777,216d \cdot k = 16{,}777{,}216d⋅k=16,777,216 个参数
  • LoRA 的 A,BA, BA,B：r⋅(d+k)=65,536r \cdot (d + k) = 65{,}536r⋅(d+k)=65,536 个参数
  • 缩减：每个矩阵的可训参数减少 256 倍

将 LoRA 应用到 32 层每层的 4 个注意力投影上，可训参数大约 800 万——占 7B 总量的 0.12%。

¶两个不能省的设计

初始化的不对称性。 BBB 初始化为零，AAA 使用小高斯初始化。训练第一步时 BA=0BA = 0BA=0，模型的前向输出与预训练版完全一致。训练从一个已知良好的工作点开始，无需额外 warmup。

α/r\alpha/rα/r 这个比例。 它不是装饰性常数。固定 α\alphaα、增大 rrr 时，更新的范数会随之变大（方向更多、合成模长更长），需要重新调整学习率。α/r\alpha/rα/r 缩放将这两件事解耦：α\alphaα 一次定下（典型取 α=2r\alpha = 2rα=2r），扫 rrr 找容量，有效步长基本不变。

¶一个最小实现

merge() 方法是 LoRA 在推理时不花钱的关键：训练完成后将 ΔW\Delta WΔW 折入 W0W_0W0​，模型在结构上与原始模型一模一样——没有多余的矩阵乘，也没有额外延迟。

¶改哪些模块

Hu 等人在原论文中发现，将 LoRA 加到查询和值投影（q_proj、v_proj）即可获得大部分收益。生产环境的默认范围更广——四个注意力投影都加，有时连 FFN 也一起：

再将 gate_proj、up_proj、down_proj（LLaMA 的 FFN 三件套）加入，通常在 benchmark 上多换不到一个点，但可训参数量是原来的 3 倍。代码生成任务值得，分类任务基本不值。

¶3. QLoRA：把基座压到 4 比特

LoRA 解决了可训练参数的问题。冻结的那部分权重仍然以 fp16 格式占用显存，是显存消耗的主要来源。QLoRA 的目标正是这块。

¶QLoRA 的三个创新

1. NF4（NormalFloat 4-bit）量化。 朴素的 4 比特整数量化浪费分辨率，因为 LLM 权重并非均匀分布，而是近似服从高斯分布。NF4 选取 16 个分位数，使每一档在标准正态下等概率，理论上达到最优编码。块大小通常为 64，每块配一个 fp16 缩放常数。

2. 双重量化。 每个块需要一个缩放常数。对于 7B 模型，块大小为 64 时，大约有 1.1 亿个常数 × 4 字节 = 0.44 GB 额外开销。QLoRA 将这些常数本身量化到 8 比特，每参数节省约 0.4 比特——单个看不多，但在 7B 规模上意义显著。

3. 分页优化器。 长序列会导致激活内存突然飙升。Paged AdamW 借助 NVIDIA Unified Memory 将优化器状态分页到 CPU 内存，需要时再换回。牺牲少量吞吐，换来训练原本会 OOM 的长序列的能力。

¶代价与收益

QLoRA 相较于 fp16 LoRA，在大部分 benchmark 上损失 1–2 个百分点，但能恢复全量微调的绝大部分性能。它带来的显存收益是颠覆性的：

70B 模型全量微调大约需要一 TB 显存（基本只有前沿实验室能做到）。QLoRA 将其压缩到一张 80 GB 卡内。这是“只能搞研究”和“你的团队能落地”之间的差距。

¶4. Adapter：最早的 PEFT

Houlsby 等人在 2019 年提出 Adapter，比 LoRA 早两年。核心思路是在 Transformer 每个子层后插入一个小瓶颈模块。

Adapter(x)=x+Wup σ(Wdown x),Wdown∈Rm×d, Wup∈Rd×m, m≪d.\text{Adapter}(x) = x + W_{\text{up}}\, \sigma(W_{\text{down}}\, x), \qquad W_{\text{down}} \in \mathbb{R}^{m \times d},\ W_{\text{up}} \in \mathbb{R}^{d \times m},\ m \ll d. Adapter(x)=x+Wup​σ(Wdown​x),Wdown​∈Rm×d, Wup​∈Rd×m, m≪d.

WupW_{\text{up}}Wup​ 初始化为零，让 Adapter 起步是恒等映射——这与 LoRA 的 BBB 使用了同样的技巧。

¶Adapter 还是 LoRA

Adapter LoRA
位置 在残差路径之内 在权重矩阵旁边
推理开销 每块多一次矩阵乘（约 5–10% 延迟） merge 后为零
可组合性 多 Adapter 串叠 LoRA 增量直接相加
多任务服务 一任务一个 Adapter 一任务一个 LoRA，可以 batch 混跑
典型大小 基座的 0.5–3% 0.1–1%
适合场景 推理图稳定的设置 对延迟敏感的 LLM 服务

LoRA 抢走了 Adapter 的大部分市场，主要原因是推理零开销这一特性。Adapter 目前仍有一席之地，例如在显式模块化组合的场景中——比如多语言任务中的 AdapterFusion（Pfeiffer et al., 2021），在推理时将多个语言专用 Adapter 融合。

¶5. 基于提示的 PEFT：不调权重，调输入

一个完全不同的思路：冻结整个模型，只学习如何喂给它数据。

¶Prompt-Tuning（Lester et al., 2021）

在输入序列前拼接 nnn 个可训练的嵌入向量，就这么简单。在 11B 的 T5 上，Prompt-Tuning 在 SuperGLUE 数据集上能媲美全量微调——但这个结论仅适用于超大模型。对于 1B 以下的小模型，Prompt-Tuning 的效果差得多。

¶Prefix-Tuning（Li & Liang, 2021）

同样的思想，但应用到每一层注意力机制：学习一组键和值前缀矩阵 PK,PVP_K, P_VPK​,PV​，在注意力计算前拼接到输入中：

Attention(Q, [PK;K], [PV;V]).\text{Attention}(Q,\ [P_K; K],\ [P_V; V]). Attention(Q, [PK​;K], [PV​;V]).

每层都有专属的前缀，相比单一输入提示，这种方法能够更深入地改写模型的计算流程。

¶P-Tuning v2（Liu et al., 2022）

去掉了 Prefix-Tuning 中使用的重参数化 MLP（在不同模型规模下表现不稳定），让深层提示均匀分布到所有层。这是第一个在所有模型规模上（包括小模型）都能在 NLU 任务上媲美全量微调的提示类方法。

何时选择提示类方法。（a）追求极致的参数效率——每个任务只需占用几 KB；（b）希望用一个服务化模型挂载多个提示。不过，在生成类任务上，提示类方法通常不如 LoRA。

¶7. 对齐：指令微调与 RLHF

PEFT 是杠杆，对齐往往是拉动这根杠杆的真正原因。现代 LLM 的后训练分为两个阶段：

有监督指令微调。 使用 （指令，回答） 对微调基座模型，使其学会遵循人类编写的提示。质量胜过数量：1K–10K 条精心挑选的样本，通常优于 10 万条众包数据（Zhou 等人 2023 年的 LIMA 论文表明，仅用 1K 条样本即可让 65B 参数模型具备相当不错的助手能力）。

RLHF（Ouyang 等人 2022 年的 InstructGPT 论文）。在 SFT 之后，基于人类偏好训练一个奖励模型，再通过 PPO 将策略推向该奖励目标。Bradley–Terry 偏好损失公式如下：

LRM=−log⁡σ(rθ(x,ychosen)−rθ(x,yrejected)).\mathcal{L}_{\text{RM}} = -\log \sigma\bigl(r_\theta(x, y_{\text{chosen}}) - r_\theta(x, y_{\text{rejected}})\bigr). LRM​=−logσ(rθ​(x,ychosen​)−rθ​(x,yrejected​)).

RL 相关内容详见 RL 第 12 部分：RLHF 与 LLM 应用。DPO（Rafailov et al., 2023）是一种当前流行的简化替代方案，无需显式的奖励模型。

¶常见问题

什么时候适合全量微调？ 算力充足、样本数量 10 万条且高质量、每个百分点的性能提升都关键时——例如基座厂商发布旗舰指令模型。其他场景推荐 LoRA 或 QLoRA。

如何选择 LoRA 秩？ 初始值设为 16。如果欠拟合（训练损失停滞在较高水平），可提升至 32 或 64。过拟合且数据量有限时，降至 4–8。分类任务通常 8 即可满足需求。

应该调整哪些模块？ 修改 q_proj 和 v_proj 可获得 80% 的收益。再加入 k_proj 和 o_proj 获取剩余部分。FFN 的三组件（gate/up/down_proj）仅在生成密集型任务且预算允许时添加，否则可训参数会增至 3 倍。

LoRA 对推理有影响吗？ merge 后无影响。merge 前，每个被包装的层会多一次极小的矩阵乘法操作——虽可忽略，但并非零成本。

QLoRA 会损失多少质量？ 在标准 benchmark 上，相比 fp16 LoRA 通常下降 1–2 个百分点，这往往在噪声范围内。对大多数从业者而言，节省的显存非常值得。

需要多少指令数据？ LIMA 表明，1K 条精心挑选的样本足以从一个强基座中训练出连贯的助手。实际操作中，底线是 1K–10K 条高质量样本；质量远比数量重要。

能否组合 PEFT 方法？ 可以。LoRA + Prompt-Tuning 是有论文支持的组合，QLoRA 本身即是一个栈（4 比特基座 + LoRA + 分页优化器）。Adapter + LoRA 在同一模型中则较少见。
