自然语言处理（十）：RAG与知识增强系统

九月 20, 2025 Updated 5月 06, 2026 — min read — words

从第一性原理构建生产级 RAG 系统：retrieve-then-generate 概率分解、向量索引（FAISS / Milvus / Chroma / Weaviate / Pinecone）、稠密+稀疏混合检索与 RRF 融合、Cross-encoder 重排、切块策略、查询改写、HyDE，以及 Self-RAG / Corrective-RAG。

冻结知识的语言模型是个自信的撒谎者：它看不到昨天的故障报告，读不到公司 wiki，也读不到今早刚合入的 patch notes。当你提问时，它会给出语法完美但事实错误的答案。检索增强生成（RAG） 通过分离"记忆"和"推理"来打破这一困局：保持 LLM 小而稳定，把易变的知识放到可随时更新的外部存储中，在生成前检索相关证据作为上下文。

理念只有一段，工程是这篇文章的剩余部分。一个真实的 RAG 系统大约有十几个旋钮——切块大小、Embedding 模型、索引类型、kkk、混合权重、重排深度、Prompt 模板、引用格式、拒答策略——它们大多互相耦合。下文逐一讲清楚：数学、权衡、可运行的代码。

¶1. RAG 的概率分解

RAG 系统通过对少量被检索文档 Dk\mathcal{D}_kDk​ 做边缘化来回答查询 qqq：

P(y∣q)  =  ∑d∈Dk  P(d∣q)⏟retriever  ⋅  P(y∣q,d)⏟generatorP(y \mid q) \;=\; \sum_{d \in \mathcal{D}_k}\; \underbrace{P(d \mid q)}_{\text{retriever}}\; \cdot \; \underbrace{P(y \mid q, d)}_{\text{generator}} P(y∣q)=d∈Dk​∑​retrieverP(d∣q)​​⋅generatorP(y∣q,d)​​

实践中用两种方式近似这个求和。一种是便宜且占主导的方式（在 LangChain 中叫 stuff），将 top-kkk 文档拼接到 LLM 上下文中，让 attention 隐式完成边缘化。另一种是原则上更优但代价更高的方式（Fusion-in-Decoder，Atlas 和原版 RAG 论文采用此方法），对每个 (q,d)(q,d)(q,d) 对单独编码，在 decoder 中做融合。对于大多数生产系统，stuff + k∈[3,8]k\in[3,8]k∈[3,8] + 一个好的重排器就是最佳选择。

这种分解的吸引力体现在以下几点：

  • 检索器 P(d∣q)P(d\mid q)P(d∣q) —— 更新成本低。几分钟即可重建文档集索引，推理时无需 GPU。
  • 生成器 P(y∣q,d)P(y\mid q,d)P(y∣q,d) —— 冻结状态。同一个 LLM 可服务任意领域。
  • 引用 —— 每条声明都能追溯到一个 chunk，可展示给用户、支持审计、过滤幻觉。

¶一个最小但诚实的流水线

三个细节值得特别关注。分隔符层级确保段落和句子不被切断；temperature=0 消除方差源，使"检索质量"成为唯一变量；拒答条款是最经济有效的幻觉防线。

¶2. Embedding 空间与 ANN 权衡

双塔编码器 Eθ:text→RdE_\theta:\text{text}\to\mathbb{R}^dEθ​:text→Rd 将查询和文档映射到同一空间，检索就是该空间内的余弦最近邻：

sim⁡(q,d)  =  Eθ(q)⋅Eθ(d)∥Eθ(q)∥ ∥Eθ(d)∥.\operatorname{sim}(q,d) \;=\; \frac{E_\theta(q)\cdot E_\theta(d)}{\lVert E_\theta(q)\rVert \, \lVert E_\theta(d)\rVert}. sim(q,d)=∥Eθ​(q)∥∥Eθ​(d)∥Eθ​(q)⋅Eθ​(d)​.

精确搜索每次查询代价为 O(Nd)O(N d)O(Nd) —— 10510^5105 规模尚可承受，10710^7107 已显吃力，10910^9109 则完全不可行。近似最近邻（ANN） 索引通过牺牲精确性换取数量级的延迟改善。生产中主要采用两类索引：

  • HNSW（分层可导航小世界图）—— 构建多层近邻图，从顶层入口贪心下降，经 O(log⁡N)O(\log N)O(logN) 次跳转收敛。可调参数包括：MMM（图度数，影响内存占用）、efSearch\textit{efSearch}efSearch（束宽，决定召回率与延迟的权衡）。适合绝大多数 RAG 工作负载。
  • IVF-PQ —— 基于粗 kkk-means 聚类（IVF 桶）和残差的 Product Quantization 实现。提供 8× 至 32× 的内存压缩，但召回率会损失几个百分点。当索引无法完全装入内存时优先选择。

图右侧展示了 1 M × 768-d 语料的 Pareto 前沿：HNSW 与 ScaNN 占据左上角；精确 Flat 在几十万规模仍可用；PQ-only 是内存极度受限时的兜底方案。

¶3. 混合检索：稠密 + 稀疏 + RRF

稠密检索擅长改述，但在三类查询上逊于 BM25：罕见命名实体（CVE-2024-3094）、精确标识符（订单 #482915）、短到无法填满 Embedding 模型有效感受野的查询。稀疏检索则相反：字面匹配强，同义词无感。

BM25 给查询 qqq 与文档 ddd 的打分公式如下：

BM25⁡(q,d)  =  ∑t∈qIDF⁡(t)⋅f(t,d) (k1+1)f(t,d)+k1 ⁣(1−b+b ∣d∣∣d∣‾)\operatorname{BM25}(q,d) \;=\; \sum_{t \in q} \operatorname{IDF}(t)\cdot \frac{f(t,d)\,(k_1+1)}{f(t,d) + k_1\!\left(1-b+b\,\frac{|d|}{\overline{|d|}}\right)} BM25(q,d)=t∈q∑​IDF(t)⋅f(t,d)+k1​(1−b+b∣d∣​∣d∣​)f(t,d)(k1​+1)​

其中 k1 ⁣≈ ⁣1.2k_1\!\approx\!1.2k1​≈1.2，b ⁣≈ ⁣0.75b\!\approx\!0.75b≈0.75。IDF 项奖励稀有词命中，长度归一化项避免长文档自动占优。

¶为什么 RRF 优于线性混合

朴素方法 α⋅sdense+(1−α)⋅ssparse\alpha\cdot s_\text{dense} + (1-\alpha)\cdot s_\text{sparse}α⋅sdense​+(1−α)⋅ssparse​ 要求两种分数处于可比尺度，但它们偏偏不在同一尺度 —— BM25 无界，余弦在 [−1,1][-1,1][−1,1]。Reciprocal Rank Fusion（Cormack 等，2009）直接绕过"刻度对齐"问题，仅用 排名：

RRF⁡(d)  =  ∑r∈R1k+rank⁡r(d),k ⁣= ⁣60.\operatorname{RRF}(d) \;=\; \sum_{r \in R} \frac{1}{k + \operatorname{rank}_r(d)}, \qquad k \!=\! 60. RRF(d)=r∈R∑​k+rankr​(d)1​,k=60.

常数 k=60k=60k=60 抑制单一检索器顶端结果的贡献，因此"在两个列表中均排第 1"的文档会胜过"仅在一个列表中排第 1"的。图中右下面板展示了异构查询集上的典型提升：BM25-only ≈ 54，dense-only ≈ 62，RRF ≈ 71，RRF + Cross-encoder rerank ≈ 78。

混合检索很少成为 RAG 系统的瓶颈，但它稳定地提供最便宜的 +5 到 +10 召回点。Dense 与 BM25 并行运行，RRF 融合后，将 50 个候选送入重排器。

¶4. 重排：精度大头来自这里

Bi-encoder 分别编码查询和文档，因此文档向量可以离线计算并建索引——速度优势明显；代价是模型从未同时看到 query 和 document 的 token。Cross-encoder 将两者拼接为 [CLS] q [SEP] d [SEP]，让整个 Transformer 跑过这一对，输出一个相关性 logit。联合 attention 能捕捉否定、问句改述、词级交互，这些是 bi-encoder 的盲区。代价是每次查询需要跑 O(k)O(k)O(k) 次联合前向，所以 cross-encoder 只适合小候选池。

两阶段流水线 是标准答案：bi-encoder（或混合）选出 top 50–100，cross-encoder 重排到最终 5。右图给出 MS MARCO dev 的典型曲线：单次重排提升 +12 nDCG，候选数翻倍再提升 +1.5，还能负担一次 listwise LLM rerank，再提升 +2.5。

实操要点。 选择训练分布贴近你领域的 reranker —— 多语言通用选 bge-reranker-v2-m3，英文 web 风格选 ms-marco-MiniLM-L-12-v2，有几千条领域标注就微调一个领域 reranker。max_length 需要封顶（更长的 passage 没收益只会拖慢速度）。GPU 上启用批处理。最重要的是测一下 —— reranker 的收益与召回率耦合，用真实流量跑 A/B 测试，再为其延迟成本辩护。

¶5. 切块：沉默的精度泄漏点

查询目标是"信息"，但索引只认得"chunk"。主流策略有三种：

  • 固定大小（NNN tokens，无 overlap）—— 简单且确定，但边界场景表现糟糕。问题的答案常被切断，横跨两个句子。
  • 递归 + overlap —— LangChain 的默认选择。按分隔符层级 ["\n\n", "\n", "。", ". ", " ", ""] 尝试，若仍超长则递归到更细一级。64–128 token 的 overlap 确保每个边界附近的句子出现在两个 chunk 中，避免因切边漏掉关键信息。
  • 语义切块 —— 对每个句子求 Embedding，计算相邻句子的余弦距离，在距离尖峰处切割。话题切换成为 chunk 边界，每块内部主题一致。建索虽慢，但在长文档上质量提升显著。

右下面板总结了经验权衡：检索 Hit@5 与回答 faithfulness 在 256–512 token 处达到峰值。更小 → 答案被切碎；更大 → chunk 向量被稀释（变成有噪的平均）。代码或结构化文档可扩展到 768–1024，因为有意义的最小单元更大。

¶父子切块：嵌入小、返回大

检索器用 400-token 的 child 嵌入提升精度（小窗口的余弦更锐利），返回 2000-token 的 parent 提供上下文（让模型看到能真正回答问题的相邻段落）。这是少数几乎零成本、却稳定提升检索与生成质量的 RAG 技巧之一。

¶6. 查询优化

用户输入的查询，几乎从来不是索引期望的查询。这里有三个值得掌握的技巧：

查询改写。 使用一次简短的 LLM 调用，将问题改写得更紧凑、更适合检索。例如，将 "昨天清掉的那个 OAuth 啥来着" 改为 "OAuth 2.0 access token 撤销事件，2024-04-23"。这种方法成本低、速度快，并且能显著提升对话式查询的召回率。

Multi-Query / 问题分解。 针对复杂问题，先生成 nnn 个改写或子问题，分别进行检索，取候选项的并集后交给重排器处理。RAG-Fusion 就是基于 multi-query 和 RRF 的方法。

HyDE —— 假设性文档嵌入。 让 LLM 先生成一个"看似合理的答案"，然后用这个伪答案计算嵌入并进行检索。这种方法看似反直觉，但效果出奇地好：幻觉生成的答案与真实答案往往处于同一邻域，因此它的 embedding 比问题本身的 embedding 更接近相关文档。代价是多一次 LLM 调用，但在处理"简短模糊的问题 × 技术语料"时收益最大。

¶7. Self-RAG 与 Corrective RAG

朴素 RAG 检索无条件触发，既浪费资源（闲聊无需检索）又容易出错（无关证据可能误导模型）。Self-RAG（Asai 等，2024）和 Corrective RAG（Yan 等，2024）将"是否检索"的决策权交还给模型，通过 reflection token 实现：

Token 决策
[Retrieve] 是否需要检索？
[ISREL] 每个召回的 chunk 是否相关？（逐文档判断）
[ISSUP] 生成的答案是否有证据支持？
[ISUSE] 答案对用户是否有价值？

控制流类似一个小图：先发出 [Retrieve]，按相关性打分分支；如果召回的 chunk 全部不相关，则进入 corrective 分支 —— 改写查询、调用 web 搜索、重新检索并评分。最终的 [ISSUP] 和 [ISUSE] 闸门确保答案基于保留的证据生成，而非单纯依赖模型的先验知识。实验表明，Self-RAG 在长文本 QA 基准上的表现优于同规模无反思 RAG，提升幅度达 +5 到 +9 点。

无需专门微调 Self-RAG 模型也能实现类似功能 —— 用普通 LLM 的多次结构化调用即可模拟相同控制流，代价是延迟增加：

代价显而易见 —— 调用 LLM 的次数比 vanilla RAG 多 3–4 倍。但在"答错的代价 > 答慢的代价"的场景中，这种取舍是合理的。

¶8. 向量数据库的实践选型

FAISS 是库，不是数据库。追求纯速度、无需持久化、不支持过滤、不处理并发 —— 适合嵌入其他服务或用于百万级离线实验。

Chroma 最容易上手：pip install 安装、支持落盘持久化、不干扰现有技术栈。单节点支持大约 10510^5105–10610^6106 向量。

Milvus 是开源重量级选手：分布式架构、多租户支持、兼容 HNSW / IVF-PQ / DiskANN 算法、支持标量过滤和原生混合检索（dense + sparse）。需要专业运维（副本管理、滚动升级、可观测性），适合 10710^7107+ 规模场景。

Weaviate 与 Milvus 在同一规模竞争，主打一流的混合检索能力和清晰的 schema 模型。运维稍简单，但原始吞吐略低。

Pinecone 是托管选项：零运维、API 设计优秀、价格较高。当工程时间是瓶颈而非基础设施成本时，选择它。

左侧雷达图为主观汇总，右侧条形图为 recall ≥ 0.95 下的单节点 QPS 量级 —— 这些只是起点，最终需结合你的语料、查询模式和过滤条件进行测试。

¶决策树

  • < 1 万向量、原型开发 → 内存版 FAISS 或 Chroma。
  • 10510^5105–10610^6106、单节点、简单过滤 → Chroma 持久化 或 Weaviate。
  • 10710^7107+、多租户、混合检索、私有化部署 → Milvus。
  • 任意规模、无运维资源、可接受云调用延迟 → Pinecone。
  • 仅关注静态语料的 top-kkk 检索 → 落盘版 FAISS。

¶9. 评估一个 RAG 系统

RAG 存在两种失败模式 —— 检索差和生成差，必须分开测试。

检索指标（需要 query→相关 chunk 的标注对）：

  • Hit@k —— top-kkk 中出现任意一个相关 chunk 即算命中。粗粒度但易于执行。
  • MRR —— 第一个相关 chunk 排名倒数的平均值。对位置敏感。
  • nDCG@k —— 折扣累计增益，归一化后使用。适合分级相关性场景。

生成指标（用 gpt-4o / claude-sonnet 充当裁判）：

  • Faithfulness —— 答案中的每条声明是否被检索到的上下文支撑？专门捕捉幻觉。
  • Answer Relevance —— 答案是否真正回答了问题？
  • Context Relevance —— 检索到的 chunk 是否确实有用？

标准工具是 RAGAS 和 TruLens，它们封装了上述这些 prompt，无需重复推导。每次调整 Prompt、更换 Embedding 或修改 chunker，都应跑一组 50–200 query 的 golden set。没有这个反馈回路，无法判断昨天的“改进”是否真的有效。

¶10. FAQ

RAG 太慢，从哪查起？ 先做 profile。Query 嵌入耗时个位数 ms，HNSW 检索也是个位数 ms，P90 延迟几乎都卡在 LLM 调用上。优先缓存命中相同的查询，压缩 Prompt（父子切块在这里很有用），接着考虑换更小的 reranker，最后才调整索引。

RAG 在幻觉。 按顺序尝试三层解决方案：(a) 收紧 Prompt，要求引用并允许拒答；(b) 加一个 Cross-encoder reranker，确保上下文真正相关；(c) 升级到 Self-RAG，加入 [ISSUP] 闸门。如果三层都加完仍有幻觉，说明 retriever 召回了垃圾内容 —— 回头检查切块和 Embedding 选择。

要不要微调 Embedding 模型？ 如果有 ≥ 5K 标注的 query-passage 对，且领域明确（如法律、生物医学或内部黑话）—— 需要。5–10 个百分点的召回提升会显著放大到下游效果。只有几十条样本时 —— 不需要，bge-large 在通用中英文场景已经足够强。

dense 还是 hybrid？ 永远优先选择 dense。当用户抱怨"明明文档里有，却搜不出来"这种精确匹配问题时，再加入 BM25 + RRF。边际工程成本仅需一个下午，但召回收益是永久的。

索引怎么保鲜？ 小语料采用 embed-on-write 策略，其他场景定期批量重建。删除是个坑 —— 多数 ANN 索引仅支持 tombstone，规划一个重建周期（多数团队每周一次足够），将其变成一件无聊但可靠的事。
