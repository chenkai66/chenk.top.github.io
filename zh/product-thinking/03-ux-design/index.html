用户体验与设计系统：Token、暗色与双语

从 CSS Token 到暗色模式到双语内容的设计系统构建——以及为什么一致性比创意更重要。

CK

#Product Thinking#Design Systems#CSS#Dark Mode#i18n
Read in English →

“好看就行"的问题#

每个写过 CSS 的工程师都经历过这种感觉：打开半年前写的样式表，发现里面有四十七种灰色、十二种毫无规律的字号、以及一个靠 filter: invert(1) 实现的暗色模式。代码能跑，页面能渲染。但每次加新功能都需要考古——翻遍层层叠叠的即兴决策，搞清楚"正确的"淡灰文字到底该用哪个色值。

我做了两个产品，被迫在不同尺度上面对这个问题。一个是 chenk.top，一个有 738 篇文章的中英双语技术博客，服务对排版和长文可读性有要求的读者。另一个是 AI4Marketing，一个支持九种语言的 SaaS 平台，几十条生成工作流，界面要在曼谷的手机和旧金山的宽屏显示器上同样好用。

两个项目教会我同一个道理：设计系统不是表达创意的工具，而是说"不"的工具——持续地、系统性地、坚定到让未来的自己（或队友）无法轻易推翻约束。

这篇文章追溯我如何通过五个设计问题抵达这个信念：Token 架构、暗色模式、双语内容管理、响应式布局、以及约束本身的哲学。

Token 驱动设计：唯一的真实来源#

问题#

chenk.top 最初的 CSS 是典型的混乱：颜色内联定义、字号凭感觉选、间距在我 14 寸 MacBook 上"看着对"但其他设备全崩。新增一个系列意味着从已有系列里复制色值然后手动调色相。暗色模式是后补的，用 media query 覆盖半个样式表。

根本问题不是审美——是架构。没有唯一的真实来源，每个设计决策都是局部的。局部决策积累成全局的不一致。

原则：语义化 Token 即 API#

我采用了借鉴自 Material Design 但去掉企业冗余的方式：一组扁平的 CSS 自定义属性写在 tokens.css 里，作为设计系统的公开 API。下游一切——组件、页面、暗色模式——只引用 Token，绝不使用原始值。

Token 架构分三层：

原语层——原始色板和尺度值：

1
2
3
4
5

--hue-0: #c0392b;   /* 暖红 */
--hue-1: #1f6feb;   /* 靛蓝 */
--hue-2: #1f8a70;   /* 深翠 */
--hue-3: #b45309;   /* 琥珀铜 */
--hue-4: #6d28d9;   /* 皇家紫 */

五种强调色。不是四种，不是七种——五种，因为这恰好足以区分每个内容系列又不至于记不住。每种色相映射到一个语义域：红色对应强化学习和系统、靛蓝对应 AI 平台和时间序列、翠绿对应推荐和云、琥珀对应基础设施和微分方程、紫色对应 NLP 和算法。映射不是随机的，它反映了我对内容的心智分类。

尺度层——基于大三度比例(1.250)的模块化字号：

1
2
3
4
5

--fs-base: 1.0625rem;
--fs-md: 1.1875rem;
--fs-lg: 1.4rem;
--fs-xl: 1.85rem;
--fs-2xl: 2.5rem;

没有像素值。没有魔法数字。每个尺寸都是尺度上的一级台阶。如果我需要 --fs-lg 和 --fs-xl 之间的某个值，答案是：我不需要。约束的存在就是为了防止"再大 2px"这种逐渐侵蚀视觉节奏的漂移。

语义层——按功能命名的颜色，不是按外观：

1
2
3
4
5
6
7

--paper: #fdfcf9;
--ink: #181821;
--ink-soft: #2c2c38;
--ink-mute: #5a5a68;
--ink-faint: #8a8a96;
--rule: #e8e4dc;
--tint: #f3efe7;

注意命名：--paper 和 --ink，不是 --white 和 --black。这不只是审美考量——这是一份契约。暗色模式切换时，--paper 变成 #0f1117，--ink 变成 #ecedef。每个引用 --paper 的组件自动适配。引用 --white 的组件则需要手动覆盖。

结果#

新增一个内容系列只需要做一个决策：分配五种色相中的哪一种。其他一切——渐变、卡片强调色、悬停态——自动派生。Token 文件 85 行。整个网站的视觉身份就活在这 85 行里。

AI4Marketing 同理但尺度不同。平台用 Tailwind CSS 加底层 CSS 变量，一个 ThemeProvider 在文档根节点设置 data-theme。主题根据时间自动切换（18:00 后暗色，06:00 后亮色），手动覆盖持久化到 localStorage。Token 词汇不同——--bg-primary、--text-primary、--accent、--border——但架构完全一致：语义命名、唯一来源、组件代码零原始值。

暗色模式做对#

问题#

暗色模式的天真做法是颜色反转：取亮色板、翻转明度通道、完事。这产出的暗色界面充满敌意——浅淡的强调色在或太深（纯黑导致视觉疲劳）或太浅（灰色看着像加载错误）的背景上显得苍白。在奶油背景上效果漂亮的渐变，切到暗色后变得浑浊。阴影要么看不见，要么产生怪异的光晕。

原则：暗色模式是新色板，不是变换#

我的暗色模式（dark.css）不做反转——它重新创作。每个 Token 都有经过深思的暗色对应值：

背景不是黑色。是 #0f1117——一种深蓝灰，读起来暗但没有纯黑的刺目。这是 Linear/Vercel 的路线：黑暗中的环境温度。

强调色有意偏移。亮色模式的 #c0392b（沉稳的暗红）在暗色模式变成 #f87171（更亮、更饱和的珊瑚红）。为什么？因为在暗底上低饱和色会消失。你需要更高的饱和度和更亮的值才能达到同等视觉重量。这不是公式——是每个色相的独立决策：

1
2
3
4
5
6

/* 亮色 */                     /* 暗色 */
--hue-0: #c0392b;    -->      --hue-0: #f87171;
--hue-1: #1f6feb;    -->      --hue-1: #60a5fa;
--hue-2: #1f8a70;    -->      --hue-2: #34d399;
--hue-3: #b45309;    -->      --hue-3: #fbbf24;
--hue-4: #6d28d9;    -->      --hue-4: #a78bfa;

渐变重新调校。亮色模式渐变从浓郁基色过渡到柔和色调。暗色模式渐变起点更深（保持与暗底的对比度），终点绽放得更亮：

1
2
3
4
5

/* 亮色：平静绽放 */
--grad-1: linear-gradient(135deg, #1f6feb 0%, #4f8ef7 55%, #82b6ff 100%);

/* 暗色：更深基底、更通透的绽放 */
--grad-1: linear-gradient(135deg, #1e40af 0%, #3b82f6 55%, #93c5fd 100%);

最有辨识度的决策是环境光照。暗色模式下 body 背景不是平面色——它带有模拟彩色环境光的微妙径向渐变：

    radial-gradient(1200px at 8% -10%, rgba(96,165,250,0.10), transparent 60%),

三个彩色光源——左上靛蓝、右上紫罗兰、底部中央翠绿——不透明度极低（6-10%）。效果几乎是潜意识的：页面感觉有生命、有纵深，而非扁平。悬停时，暗色模式的卡片发出与其系列色相匹配的彩色辉光：

结果#

暗色模式不是一个开关——它是同一内容的不同体验。夜间访问的读者（自动时间切换确保很多人如此）看到的是一个为弱光阅读精心设计的空间，不是仓促改编的亮色版本。

双语挑战#

问题#

chenk.top 有 738 篇文章，中英镜像。不是机器翻译——每个版本都为其受众原生撰写。英文读者得到直接的技术语言。中文读者得到假设共享文化语境、正确使用领域中文术语的版本（不是 MT 那种生硬输出）。

挑战是结构性的，不是语言学的。如何维护 738 对并行文档？一篇更新后如何保持同步？如何提供无缝的语言切换？如何正确处理两种语言变体的 SEO？

原则：translationKey 作为绑定契约#

Hugo 的多语言系统用内容目录（content/en/ 和 content/zh/），通过 frontmatter 中的 translationKey 绑定对应文章：

1
2
3
4
5

# en/some-article.md 中
translationKey: "reinforcement-learning-policy-gradient"

# zh/some-article.md 中
translationKey: "reinforcement-learning-policy-gradient"

Key 是语义标识符，不是文件名。这将 URL slug（为 SEO 可以不同语言各异）与绑定关系解耦。每种语言有自己的 slug，同时共享逻辑身份。

模板层面，<head> 中的 hreflang 标签向搜索引擎声明关系：

1
2
3
4

<link rel="alternate" hreflang="{{ .Language.Lang }}" href="{{ .Permalink }}">
{{ if .IsTranslated }}
<link rel="alternate" hreflang="x-default" href="{{ .Permalink }}">
{{ end }}

语言切换器的 UI 刻意简单：头部一个旗帜/文字切换，跳转到当前页面的译文版本，不是另一语言的首页。如果不存在译文，切换器隐藏——绝不提供一个无法兑现的承诺。

扩展到九种语言：AI4Marketing#

AI4Marketing 把双语推向更远——九种语言（zh-CN、en-US、ja-JP、ko-KR、es-ES、th-TH、vi-VN、pt-BR、ar-SA）。但理念不同：这不是人工撰写的并行内容，而是 AI 辅助的本地化。

本地化模块的核心洞察是翻译与本地化的区别：

“你的任务不是翻译——是本地化：让内容读起来像目标市场的母语营销人员写的。”

系统用 Qwen 按格式定制提示词来适配营销内容（文章、社交笔记、短视频脚本、分镜旁白）。每种格式有不同的本地化优先级：社交笔记需要适配标签惯例；视频脚本需要根据目标语言的音节密度调整节奏；文章需要本地化的 SEO 关键词。

这是一个有 UX 含义的设计决策：用户用一种语言生成内容，得到八个其他市场的版本——不仅是翻译而是文化适配。UI 将此呈现为单一的"本地化"操作，而非每种语言一个翻译按钮——将心智模型从"我需要翻译八次"简化为"我可以触达八个市场”。

结果#

chenk.top 上，每种语言 36 个内容目录，translationKey 确保每篇文章有其对应版本。审计流水线（scripts/audit/）捕获孤儿翻译和中文文章中的残留英文片段。AI4Marketing 上，本地化是一等公民管线阶段，不是事后补丁。

响应式但不陷入断点地狱#

问题#

传统响应式方法创建断点与组件状态的矩阵，增长是组合级的。一个卡片组件可能有手机、平板、桌面三种变体；乘以二十种组件类型就是六十种布局状态要维护。CSS media query 变成 bug 的温床：后面的断点以意料之外的方式覆盖前面的，特异性(specificity)战争升级，移动端测试变成打地鼠。

原则：独立文件中的增量式修复#

我的做法反转了通常的方法论。基础 CSS 为桌面可读性设计。移动端适配集中在单独文件——mobile-fixes.css——施加的是增量修复而非重写布局。

文件头注释一目了然：

1

/* ===== Mobile-friendly fixes (audit pass 2026-05-07) ===== */

它是一次审计遍历——系统性排查并修补具体的移动端痛点，而非从头重新构想布局。理念如下：

  1. 全局溢出防护先行：body 上 overflow-x: hidden，flex/grid 子元素加 min-width: 0。这单条规则预防了 80% 的移动端布局 bug（元素撑破视口）。

  2. 代码块是主要挑战：Hugo 语法高亮输出表格结构标记。修复方法将其转换为 flex 布局——行号固定、代码横向滚动：

1
2
3
4
5
6
7

.highlight .lntable tr { display: flex; max-width: 100%; min-width: 0; }
.highlight .lntable td.lntd:last-child {
  flex: 1 1 0;
  min-width: 0;
  overflow-x: auto;
  -webkit-overflow-scrolling: touch;
}

  1. 触摸目标强制执行：480px 以下屏幕，交互元素获得显式最小尺寸。抽屉导航链接设 padding: 12px 4px。这不是审美——是 Fitts 定律应用于拇指。

  2. 渐进缩小尺寸，不改布局：小手机上，hero 标题从基础的 clamp(2.4rem, 6.4vw, 4.6rem) 显式变为 1.7rem。布局不变；只是比例调整。

  3. 数学公式处理：KaTeX 在 iPhone SE（320px 逻辑宽度）上需要显式缩小以避免横向溢出。一个针对性的 @media (max-width: 380px) 处理这个边缘情况，不影响更大的手机。

文件共 438 行——相当可观，但只有移动修复。没有布局逻辑、没有组件定义、没有设计决策。其中每条规则都是对特定移动端渲染问题的补丁。这种分离意味着我可以通过阅读一个文件来审计移动端行为，而不是在二十个样式表中的分散 @media 块里搜寻。

结果#

站点在我测试过的所有设备（iPhone SE 到 27 寸 iMac）上用单一布局架构运行。移动修复不与桌面设计对抗——它们适配桌面设计。新增组件只需检查一个问题：“它在 320px 屏幕上会溢出吗？“如果是，在移动文件里加一条修复。如果不是，继续。

设计即约束#

按钮规则#

AI4Marketing 的 UI 中，一条原则统治每个操作行：每行只有一个填充主按钮，其余全部描边。这不是指南——是通过组件设计强制执行的硬规则。

原因是认知层面的：当用户看到多个填充按钮时，必须阅读每个标签才能判断哪个是主操作。当只有一个填充按钮时，主操作在前注意力阶段就能识别——在意识阅读之前。描边的次要操作可被发现但不争夺注意力。

这条规则级联为具体实现决策：

  • 主按钮使用主题色变量（var(--accent)），绝不硬编码颜色
  • 次要按钮用 border: 1px solid var(--border) 加透明背景
  • 暗色背景上主按钮必须仍用主题变量（不能用硬编码灰阶，暗色模式会坏）
  • 破坏性操作（删除、取消订阅）获得红色变体但仍遵循单主操作规则——只是将主色切换为 --hue-0

“无表情符号"哲学#

一个小决策但影响深远：chenk.top 不在标题、导航或正文中使用 emoji。这是对"友好型"技术写作潮流（到处撒火箭和灯泡）的有意抵抗。

理由是：emoji 有文化负载，跨平台渲染不一致（有时令人不适），破坏衬线标题的排版节奏，增加噪声而无信息。一个写着”## 性能优化"的标题和”## 性能优化"传达的内容完全相同——emoji 增加的只是读者必须过滤的视觉噪声。

这延伸到 AI 生成内容：当 Claude 或任何模型为博客产出内容时，指令明确——除非用户特别要求，否则不加 emoji。约束保护设计系统的声音。

字体栈决策#

1
2
3

--font-serif: "Source Serif Pro", "Lora", Georgia, "Songti SC", "STSong", serif;
--font-sans: "Inter", "IBM Plex Sans", -apple-system, ...;
--font-mono: "JetBrains Mono", "IBM Plex Mono", "SFMono-Regular", ...;

在技术博客用衬线字体（Source Serif Pro）做标题是刻意的逆潮流。大多数开发者博客全站用几何无衬线（Inter、Geist、Space Grotesk）。我选择衬线标题因为：

  1. 与正文的对比：正文用 --font-sans（Inter）。标题用衬线通过字体对比而非仅靠尺寸创造视觉层级。这意味着我需要更少的尺寸台阶来建立层级——同样大小的衬线 H2 比无衬线 H2 因为形式感的转换而读起来更重要。

  2. 杂志感：站点追求长文可读性，不是文档密度。衬线标题唤起编辑设计感——The Atlantic、Stripe 的文字——而非 API 文档。

  3. 中文回退：字体栈包含"宋体 SC"和"华文宋体"用于中文衬线渲染。这很重要，因为中文版应携带同样的编辑感，而宋体（衬线对应物）在 CJK 排版中实现这一点。

  4. 代码专用等宽：JetBrains Mono 有内置连字且为代码可读性设计。它仅用于代码块和键盘快捷键指示器——绝不用于正文、绝不用于 UI 标签。

渐进披露：少即是多#

搜索覆盖层#

搜索功能直到被调用才可见。没有持久的搜索栏占据头部空间。用户按 /（或 Cmd+K），一个全屏覆盖层带着聚焦的输入框、毛玻璃背景和即时结果出现：

1

5

.search-overlay {
  position: fixed;
  inset: 0;
  background: rgba(20, 20, 30, 0.5);
  backdrop-filter: blur(6px);
  display: none;
  padding-top: 12vh;

按 Escape 或点击外部关闭。结果随打字实时出现——无提交按钮、无单独结果页。交互模型借鉴 Spotlight/Alfred：呼出、输入、选择、关闭。三次击键找到任何文章。

抽屉导航#

移动端，导航从水平头部栏变为侧边滑入的抽屉。抽屉包含与桌面导航相同的链接加语言切换器，但以触摸优化的布局呈现：

1

.drawer nav a { font-size: 1.08rem; padding: 12px 4px; }

每个链接有充裕的垂直内边距（12px），创造 44px 以上的触摸目标——Apple 推荐的最小值。抽屉本身有 padding: 20px 22px，确保内容绝不触及视口边缘。

按键看起来是物理的——斜切边缘、底部阴影模拟厚度、悬停时甚至有微妙的按压动画。这个微交互奖赏发现它的用户：他们得到一张精美渲染的参考卡片，不是一个扁平文字列表。

移动端（720px 以下），键盘 FAB 完全隐藏——触屏用户没有键盘，因此这个功能暗示是无关的。不浪费视口空间。

教训：设计系统的本质是一致地说"不”#

我描述的每个设计决策本质上都是一种约束：五种色相，不是无限的；每行一个主按钮，不是两个；绝不用 emoji；衬线只在标题用，不是其他地方；移动修复集中一个文件，不是散布各处。

这些约束在当下感觉限制重重。“但这张卡片配第六种色相会很棒。““但这个对话框真的需要两个主操作。““但这个章节标题加个闪烁 emoji 会更亲切。”

答案永远是不。不是因为单个例外会丑——它大概看着还行。而是因为每个例外都在侵蚀系统。一种额外色相变成三种。一个额外的主按钮变成别人复制的模式。一个 emoji 变成一打，然后博客的视觉一致性沦为 Notion 模板的水平。

设计系统的力量不在于它能做什么——CSS 什么都能做。力量在于它阻止什么。当我打开一个新文件需要给组件写样式时，我不做决策。我引用 Token。决策只在一个地方做一次，然后传播到所有地方。这不是限制创意——是将创意引导到唯一重要的地方（Token 定义），在其他所有地方消灭它。

在生产中，这体现为速度。给新组件加暗色模式无需额外工作——它本来就能用，因为组件使用语义 Token。AI4Marketing 加一种新语言只需一条数据库记录和一个本地化提示词，无需重新设计。让页面响应式意味着检查一个文件看是否需要修复，而不是重构布局。

公式很简单：在约束层（Token、原则、规则）上重投入，然后在每个下游决策上花近乎零的精力。能用的设计系统用起来无聊。这就是重点。
