架构设计：从单体到自治 Agent

我的架构思维如何从一个 Next.js 应用演进到分布式自治 Agent 系统——以及沿途涌现的模式。

系统的形状#

每一种架构都是一场被冻结的争论。它记录了你在提交代码那一刻对问题的信念。回顾过去十八个月我构建的四个系统——一个营销内容平台、一个零依赖技能路由引擎、一个自主研究 Agent 和一个多模型编码编排器——我能清晰地追溯自己架构直觉的变迁。并非总是向前，有时是横向偏移。但有一条清晰的主线：从"把一切放在一个进程里"到"让 Agent 自己治理自己"。

这不是分布式系统教程。这是对我为何做出那些结构选择、什么东西崩了、以及什么原则在生产环境中存活下来的反思。

第一幕：运转良好的单体 (AI4Marketing)#

AI4Marketing 起初只是一个周末原型：一个调用 Qwen 生成营销文案的 Next.js 应用。十八个月后，它有 121 条 API 路由、21 张数据库表、一条视频生产管线、一个短剧生成引擎、一套 GEO 优化系统，以及接入四家支付提供商（支付宝、微信支付、Stripe、PayPal）的支付流程。它仍然是 PM2 fork 模式下的单个 Node.js 进程——甚至不是 cluster 模式。一个进程，一条 JavaScript 线程，一个事件循环。

我没有计划做单体。我计划快速交付。架构是那种速度的残余物。

为什么它一直是单体#

关键洞察在于 AI4Marketing 本质上是一个带有长时间副作用的请求-响应应用。用户提交内容生成请求；系统验证、扣减配额、调用 LLM、返回结果。视频管线需要数分钟，但它仍然由一个 HTTP 请求触发，并将进度写入同一个 PostgreSQL 数据库。没有服务间通信，因为根本没有服务——只有进程内的函数调用。

从来没有拆分的必要。“微服务会更清晰"的论点持续败给"我能在一个终端里 grep 整个代码库”。视频管线挂了，我查 lib/video-pipeline-v2.ts。配额逻辑有误，我看 lib/quota-checker.ts。支付 webhook 行为异常，handler 就在 app/api/webhook/alipay/route.ts。没有服务网格，没有消息队列，没有容器编排，没有 Kubernetes，没有 Docker Compose。就一个进程、一个数据库、一个部署目标，和 pm2 restart ai4m。

数据库 schema 是以 User 为中心的星型模型。Post、VideoProject、DramaProject、Order、Subscription 向外辐射。共 21 张表。每个关系都是直接外键——没有事件溯源，没有 CQRS，没有最终一致性。需要知道用户配额时，读一行。需要他们的视频项目时，join 两张表。这是无聊的技术，但它有效。

让单体可维护的模式#

让 121 条路由没有变成不可维护意面的，是 handler 层面的纪律。每一条路由都遵循完全相同的分层模式：

认证 -> 校验 (Zod) -> 限流 -> 配额检查 -> 业务逻辑 -> 失败退款

这不是框架提供的功能。这是我一遍遍读自己代码直到模式成为肌肉记忆后形成的约定。withMetrics 高阶函数包裹每个路由处理器，记录 http_request_duration_seconds 和 http_requests_total，无需 opt-in。路由标签显式指定——从不从动态路径参数自动推导——以防止指标存储中的基数爆炸。

配额系统值得特别关注，因为它解决了一个真实的并发问题。朴素方法——“读余额、检查是否充足、扣减”——有教科书级的 TOCTOU 竞态：两个并发请求都读到"余额 = 1"，都通过检查，都扣款。修复方案是一个条件更新，让检查和扣减原子化：

这个 WHERE 子句消除了竞态条件。两个并发请求不可能同时成功，因为在行级别只有一个能匹配条件。PostgreSQL 的行级锁保证了这一点。这是用 3 行代码修复一个曾让生产系统损失数百万美元的问题。

限流器使用九个预设：GENERATE（10/分钟）、VIDEO（5/小时）、LOGIN（5/分钟）、REGISTER（3/5分钟）、QUERY（60/分钟）、STATUS_POLL（120/分钟）、SEARCH（30/分钟）、ADMIN（30/分钟）、WEBHOOK（20/分钟）。这些是内存滑动窗口——不依赖 Redis。内存通过定期清理限界（每 5 分钟，上限 10,000 条）。对于单进程应用这足够了，因为内存永远不会迁移。

优雅关停处理器在 SIGTERM 时注册，设置 shuttingDown 标志拒绝新请求，然后每两秒轮询 activePipelines 集合。如果所有管线在 10 分钟内排空，干净退出。否则强制退出。PM2 遵守 10 分钟的 kill_timeout。这意味着我可以部署而不杀死进行中的视频渲染——它们完成后进程才回收。

API Key 管理器：单体中的分布式可靠性#

一处隐藏的复杂性是 API key 管理器（lib/api-key-manager.ts）。AI4Marketing 每次生成都调用 DashScope。16 个 API key 分布在 CN 和 INTL 地域，系统需要智能轮转：

  • 收到 429（限流）：立即切换到下一个 key，不做退避
  • 其他错误：指数退避重试
  • Per-key 熔断器：N 次连续非限流失败就搁置该 key
  • 全局熔断器：所有 key 同时失败时停止所有重试
  • 地域路由：CN key 走 dashscope.aliyuncs.com，INTL key 走 dashscope-intl.aliyuncs.com

这是分布式系统模式（熔断、key 轮转、跨地域容灾）被压缩进单进程库。它处理了最常见的生产故障模式——API 限流——不需要任何基础设施，只需一个追踪 key 状态的 JavaScript 对象。

单体做不到的事#

局限在我添加 Spark 短剧管线时暴露了——这是一个 agentic 短片制片系统，LLM 写剧本、导演场景、拿角色定妆照审看样片并迭代。这条管线本质上是多步骤、多模型、状态密集的。一个短剧项目通过 child_process.spawn 启动 Python sidecar，在编剧、导演和审片阶段编排多次 LLM 调用，将 NDJSON 状态行流式回传给 Node。

它跑得起来。它很丑。它是单体接缝变得可见的那个点——因为短剧管线需要自己的生命周期、自己的错误恢复、自己的资源预算，独立于共享同一事件循环的内容生成管线。当短剧 sidecar 内存泄漏时，它把整个应用拖下水。当我需要重启短剧处理时，我必须重启一切。

这是让我理解真正边界的时刻：不是代码规模，而是生命周期独立性。

第二幕：零依赖即架构 (DaaS)#

DaaS（Documentation-as-a-Skill）是对 AI4Marketing 意外复杂性的有意识反应。前提是：Python 标准库就够了。不用 FastAPI，不用 SQLAlchemy，不用 Redis，不用 Docker，生产环境不需要 pip install。

整个 HTTP 服务器就是 http.server 的 ThreadingHTTPServer。一个类 DaaSHandler 覆写 do_GET 和 do_POST，做模式匹配分发。96 种路由模式，通过 URL 路径的字符串操作匹配。前端是 22 个静态 HTML/CSS/JS 文件，用内存 gzip 缓存和 mtime 失效来服务。没有构建步骤，没有 webpack，没有 node_modules，运行时没有 CDN。

零依赖背后的哲学#

每个依赖都是有 TTL 的负债。它会在升级时崩溃，变得无人维护，或者引入供应链漏洞。left-pad 事件。colors.js 自毁。event-stream 后门。这些不是假设——它们是 npm 生态系统的历史事实。对于一个我打算以最小维护量运行多年的系统，我唯一信任的依赖是随 CPython 一起发行并由 CPython 核心团队维护的那些。

DaaS 跑在广州的一台 ECS 实例上，前面挡着 Cloudflare Tunnel。部署流程是 rsync -a . server:/var/www/DaaS/ && systemctl restart daas-api。如果 Python 版本从 3.10 升到 3.12，http.server 和 threading 和 json 和 os 的行为完全一致。需要部署到一台全新的机器上时，我复制目录然后跑 python3 -m src.api.server --host 0.0.0.0 --port 80。什么都不需要安装。

零依赖迫使你怎样设计#

没有 ORM，我用原子文件写入来持久化所有状态。每次变更先写到临时文件，再调 os.replace() 原子地换到位。这意味着写到一半崩溃永远不会损坏状态——你要么得到旧版本，要么得到新版本，绝不会是半成品。这个模式贯穿整个代码库：keys.json、usage.json、billing.json、geo_measurements.json——/var/lib/daas/ 中全部 14 个状态文件都用它。

没有消息队列，后台任务作为带超时守卫的守护线程运行。调度器启动七个任务（drift_check、graph_rebuild、recall_snapshot、data_gc、billing_cycle、edge_explore、geo_remeasure），每个在独立线程中，带硬超时（60-300 秒视任务而定）。如果任务挂起，超时杀掉它。调度器下个周期重试。没有死信队列，没有重试策略，没有可见性超时。就是一个线程、一个计时器、一次重启。

没有框架的中间件栈，认证就是在每个 handler 头部调用的一个函数，返回身份或抛 401。三层分类（公开、已认证、已付费）用注释文档化、用约定执行。支付 webhook 是公开的（支付宝/Stripe 需要触达）。路由查询是公开的（AI 爬虫需要触达）。Ingest 是付费的。Workspace 创建是付费的。这简单到可以装进脑子里。

缓存架构值得审视，因为它从零依赖约束中涌现而非自顶向下设计：

缓存范围TTL失效条件
静态文件内存字节 + gzip 变体mtime 驱动检测到文件修改
路由索引Per-skills-dir trigger 条目5s 过期检查SKILL.md mtime 变化
遥测分析Per (mtime, size, args)30s日志追加或 TTL

所有缓存使用 threading.Event 做 inflight 去重。当两个请求同时到达同一未缓存数据时，一个计算，另一个阻塞在 Event 上。这防止了惊群重算——真实生产系统中缓存未命中风暴导致的问题。

上帝文件问题及其解决#

最初的 server.py 有 4750 行。这是"没有框架"的自然终态——所有逻辑汇聚到 HTTP handler，因为没有路由器、没有中间件链、没有依赖注入容器来分离关注点。修复方法是机械式的：识别内聚的函数组，提取到模块（routing.py、analytics.py、geo.py、geo_loop.py、geo_routes.py、billing.py、payment.py、keys.py、auth.py、ingest.py、edge_explore.py、improve.py、gap_radar.py、skill_graph.py），让 server 文件只剩纯 HTTP 分发 + 编排。

结果是 17 个提取模块，每个职责清晰且没有循环导入。server.py 降到约 1200 行路由分发。教训：零依赖不等于零结构。结构必须是自我施加的而非框架施加的。而提取模块的纪律必须主动施行——在上帝文件变得太纠缠而无法安全分解之前。

第三幕：状态机作为通用控制器 (Research Agent)#

Research Agent 是我的架构做出最大概念跃迁的地方。这是一个完全自主的系统：从 arXiv 读论文，在 SQLite 中构建知识图谱（65,000+ 节点、197,000+ 边），生成研究想法，通过三轮对抗辩论筛选，设计实验，将执行派发到 worker 节点，运行统计分析，撰写论文。它在一台 4 vCPU、7.5 GB 主服务器上 7x24 无人干预运行，配备两台 128 GB 的计算 worker 节点。

让这一切成为可能的核心洞察：为每一个实体生命周期建立显式有限状态机，在数据层强制执行。

为什么用 FSM#

在引入 FSM 之前，协调器用一个 15 级优先的 if/elif 链来决定下一步做什么。逻辑检查标志：“如果有一个状态为 approved 且没有 protocol.json 的 idea，派发 experiment_designer。“它在三个状态时能用。到了十二个状态加上失败路径、重试和基础设施错误，它变成了组合爆炸。真实 bug 出现了：一个实验永远停在 executed 状态，因为"派发 statistician"的条件检查 analysis.json 不存在——但分析过程中的崩溃留下了一个满足存在检查但不含有效结果的残缺文件。

FSM（framework/pipeline_state.py）通过让转移显式且穷举来消除了整个这类 bug：

Idea 生命周期:
  proposed -> approved -> experiment_designed -> experiment_completed
      |          |              |                        |
      |          |              +-> negative_result      |
      |          |              +-> paper_written ------>+
      |          |                                      |
      |          +-> killed                             +-> paper_reviewed
      |          +-> failed
      +-> merged
      +-> subsumed
      +-> killed

Protocol 生命周期:
  designed -> running -> executed -> analyzed
      |          |          |           |
      |          |          +-> failed  +-> pending_failure_analysis
      +-> needs_redesign               +-> needs_redesign (回到 designed)
      +-> abandoned                    +-> needs_rerun (回到 running)
      +-> dataset_failed

每次 transition(entity, from_state, to_state) 调用验证转移是否合法，带时间戳记录到 .history.jsonl，非法移动时抛出 InvalidTransition。你不能把一个 idea 从 proposed 直接移到 paper_written——FSM 强制它必须经过设计、执行和分析。这是在数据层的编译时保证。

派发决策层#

Boss Agent 是一个 ReAct LLM 编排器，决定下一步做什么。但它不是从头决定——它评估来自 framework/dispatch_decision.py 的九个结构化谓词：

每个谓词（如 should_dispatch_experimenter(experiment_id)）读取 FSM 状态，检查必要前置条件（protocol 存在、没有活跃锁、slot 可用），返回结构化决策。Boss 读取所有九个谓词的输出，按优先级排序，派发最高的那个。

为什么这重要：当某事未能派发时，reason 字段告诉你确切原因。“Protocol 尚未设计"或"所有 experimenter slot 已占用"或"目标被 PID 12847 锁定”。没有谜团。没有"系统就是没做”。每一个非行为都有可审计的解释。

优先级创造了隐式的调度策略：statistician 200（因为分析解锁 writer + reviewer + formatter），failure_analyst 150（因为失败分析触发 redesign 从而解锁 experimenter），merger 85（辩论前检查）。这些数字在一个每 30 秒热重载的 JSON 文件中——我可以不重启任何进程就重新排列整个系统的行为优先级。

Fleet：通过 HTTP 轮询的分布式执行#

主服务器只有 7.5 GB 内存。运行 LLM 驱动的实验（可能需要加载大数据集并发起数百次 API 调用）会立即 OOM。fleet 架构用极简方案解决这个问题：

Worker 每 5 秒轮询主服务器：“有命令给我吗？“主服务器响应排队的工作或空。Worker 执行，然后 POST 结果回来。Worker 还每 60 秒 POST 心跳附带系统指标（负载、内存、磁盘、运行中 agent 数）。独立的同步循环每 60 秒推送生成的产物（论文、KG patch、实验数据）。

这就是 HTTP 轮询。它是最简单的分布式执行模型。没有 WebSocket，没有 gRPC 流，没有消息代理。Worker 可以离线再回来——它只是恢复轮询。主服务器通过心跳新鲜度追踪 worker 健康状态。如果心跳过期（>600s），worker 被标记为不可选，其已派发任务最终被孤儿派发回收器收割。

为什么不用 Celery/Temporal/Bull？因为那些系统本身就是复杂的分布式系统，需要监控、Redis/RabbitMQ 后端存储，以及我必须学习和调试的故障模式。我的 fleet 协议是四个 HTTP 端点。我可以用 curl 调试它。我可以通过重新 POST 命令 JSON 来重放任何派发。整个 fleet 服务器约 400 行 Python。

自愈：四十条自动恢复规则#

在一台 7.5 GB 内存的机器上跑六个 systemd 服务（coordinator、pipeline、dashboard、supervisor、kg-merger、dingtalk-listener），OOM kill 不是理论上的——它每周都发生。架构不试图防止 OOM。它假设 OOM 会发生并为恢复而设计：

  1. OOM 优先级分层：pipeline 得到 OOMScoreAdjust=+500（优先被牺牲）。coordinator 得到 -500（受保护）。当内存压力来袭，Linux 先杀 pipeline 再杀 coordinator。pipeline 的状态全在磁盘——它重启后从中断处恢复。

  2. 内部内存看门狗：两个进程都有后台线程每 2-3 分钟检查 RSS+Swap。coordinator 在 1500 MB swap 时触发激进 GC，2400 MB RSS 时强制退出。pipeline 在 1900 MB 时强制退出。强制退出是刻意的：从已知状态重启比带着退化的内存苟活要廉价。

  3. Supervisor 调和：supervisor_lib/heal_rules.py 包含 40+ 条规则，每分钟触发。规则 1：如果 analysis.json 存在但状态仍是 executed，强制转移到 analyzed。规则 2：如果 failure_analysis.json 存在并有决策，应用该决策。规则 3：如果 redesign 次数超过 5，杀掉 idea（防止无限循环的后备）。规则 4：如果 needs_redesign 卡住超过 24 小时且有 5+ 次派发尝试，设为 abandoned。

  4. 基础设施错误检测：如果 failure_analysis 发现根因包含"timeout"或"429"或"404”，将 protocol 重置为 designed 以重试，而非触发 redesign。瞬态基础设施故障不应浪费一次 redesign 周期。

  5. 陈旧锁收割：锁使用 fcntl.flock()，内部写入 PID + 时间戳。supervisor 检查 PID 是否存活。死 PID 的锁被移除。超过 4 小时 TTL 的锁无论如何被强制移除。

invariant checker（framework/invariant_checker.py）在每个 boss 周期后运行，强制全局一致性：状态/产物对齐、无重复派发、无孤立锁。违规以 ERROR 级别记录并持久化到 data/invariant_violations.json。这是最终安全网——即使 bug 引入了不一致，下一个 boss 周期也会检测并标记它。

第四幕：DAG 分解复杂任务 (Elevator)#

Elevator 是一个自主编码系统，通过结构化管线编排多个中国 LLM（Qwen、DeepSeek、Kimi、GLM、MiniMax）：目标分解为子目标 DAG，并行批次执行，跨模型验证，经验学习。所有模型通过 DashScope 的统一 OpenAI 兼容 API 访问，这意味着模型切换只是改一个字符串。

DAG 作为执行计划#

当用户提交一个目标——“开发一个微信小游戏仙侠放置 RPG”——planner（planner.py，474 行）将其转换为结构化 JSON DAG。planner 的上下文注入链至关重要：它在生成计划前读取全局 lessons（过去要避免的失败）、skills（成功的计划模板）、workspace 状态（当前文件）和 project memory（跨里程碑决策）。

一个真实项目的 DAG 看起来像：

m1:  Canvas 框架 + 游戏循环      (无依赖)
m2:  角色系统 + 属性            (无依赖)
m3:  配置驱动架构              (依赖: m1, m2)
m4:  战斗系统                  (依赖: m3)
m5:  背包 + 装备               (依赖: m3)
m6:  修炼进阶                  (依赖: m4)
m7:  NPC + 对话系统            (依赖: m3)
m8:  地图 + 探索               (依赖: m7)
m9:  UI 打磨                   (依赖: m5, m6)
m10: 美术资源                  (依赖: m9)
m11: 音效                      (依赖: m10)
m12: 存档/读档系统             (依赖: m8)
m13: 性能优化                  (依赖: m11, m12)
m14: 最终 QA + 打包            (依赖: m13)

runtime 通过拓扑批次执行处理这个 DAG：m1 和 m2 并行执行（无依赖），然后 m3 等待两者，然后 m4/m5/m7 并行执行（都只依赖 m3），依此类推。ThreadPoolExecutor 管理并行执行。挂钟时间等于关键路径，而非所有里程碑的时间之和。

DAG 验证并非平凡，因为 LLM 有时产出无效计划。planner 执行环检测、悬空引用移除（引用不存在的子目标 ID）和 ID 唯一性强制，然后才接受计划。可选的 critique 步骤（critique_plan()）将计划发送给跨家族模型审查——如果发现严重问题，revise_plan() 重写（受 MAX_CRITIQUE_ROUNDS 限制以防止无限修改）。

跨家族验证：反阿谀模式#

Elevator 最有后果的架构决策是没有任何模型评估自己的输出。执行管线在结构上强制这一点：

执行器 (qwen3.6-plus) 产出代码
    |
    v
审查者 (deepseek-v4-pro) 检查 bug、边界情况、测试覆盖缺口
    |
    +--> 发现问题: 注入反馈，回到执行器 (最多 2 轮)
    +--> 无问题: 进入验证
    |
    v
验证器 (qwen3.6-plus, 不同实例) 独立检查每个标准
    |
    +--> 低置信度: 升级到集成面板 (3 个模型, 多数投票)
    +--> 所有标准通过: git commit, 标记完成
    +--> 标准未通过: 升级模型层级, 重试

为什么跨家族？因为同一家族的模型共享训练数据、架构偏见和故障模式。当 Qwen 写出有 bug 的代码，另一个 Qwen 实例往往也抓不住 bug——它有相同的盲区。DeepSeek 的 MoE 架构处理信息的方式足够不同，能捕获 dense transformer 遗漏的错误。我通过经验发现了这一点：同家族审查对跨家族审查能捕获的 bug 有约 15% 的漏检率。

渐进式升级：伪装成可靠性的成本优化#

三层升级路径的设计使得 80% 的任务在最便宜的模型上完成：

Attempt 0: qwen3.6-plus        (4 CNY/M 输入, 12 CNY/M 输出) — 处理大部分任务
Attempt 1: qwen3.6-max-preview (20 CNY/M 输入, 60 CNY/M 输出) — 更难的问题
Attempt 2: deepseek-v4-pro     (4 CNY/M 输入, 16 CNY/M 输出) — 跨家族后备

attempt 0 和 attempt 1 之间的成本差异是 5 倍。洞察在于模型能力对任务难度遵循幂律分布：大多数编码任务是直截了当的，便宜模型就能搞定。只有复杂任务的尾部——架构决策、微妙的并发 bug、复杂算法实现——需要贵模型。而当同家族的贵模型也失败时，切换家族（attempt 2）往往成功，因为失败模式是家族特定的。

复杂度在执行前估算：“fix”、“typo”、“rename"等关键词分类为 light（最多 12 轮），而"architecture”、“refactor”、“full-stack"分类为 heavy（最多 35 轮）。这个预分类防止在琐碎任务上浪费贵模型预算。

绊线：行为异常检测#

执行器运行一个 agent 循环——LLM 反复选择和调用工具直到它宣布任务完成。这个循环可能以特征性的方式出错：无限读循环（只读文件不写）、重复工具调用（相同 bash 命令调用 10 次）、或过早提交（没跑测试就宣布完成）。

绊线系统（tripwires.py，309 行）实时监控这些模式：

  • shallow_no_test：写了代码但从没跑 pytest/bash？阻止提交。
  • repeat_loop：同一工具调用重复 N 次？警告，然后强制换个动作。
  • no_progress_streak：多轮没有文件写入？注入反思提示。
  • explore_overload：长时间阅读不产出代码？提示开始写。
  • consecutive_read_only：N+ 轮连续只读？强制行动。

升级路径是：warn -> force_action -> auto-submit（3 次 force 后）。这防止了最昂贵的故障模式：一个 agent 循环烧掉 35 轮贵模型调用却一事无成。

技能进化：积累的机构知识#

每次任务成功后，Elevator 提炼一个"技能”——从具体任务中抽象的计划模板。具体的文件路径、变量名和 API 端点被剥离；结构性模式被保留。技能遵循一个生命周期：

candidate (分数 0.5) -> 验证关 -> active (复用成功时分数增长)
    |                                  |
    | 被拒绝                            | fail_streak >= 3: needs_review
    v                                  | 分数 < 阈值: archived
  archived                             | 闲置 180+ 天: archived

检索使用双策略：语义（embedding 余弦相似度）和关键词（查询关键词与技能上下文关键词的重叠）。高分技能排在低分技能上面。只检索 active 技能——candidate 和 archived 技能对 planner 不可见。

当技能库超过阈值时，相似技能（关键词集 Jaccard 系数 >= 0.3）被聚类并合并为聚合元技能。这是无监督的知识压缩：系统保留一般性有效的，忘记只有效过一次的。

涌现的原则#

回顾这四个系统，我能提取出不论语言、框架或问题域都反复出现的模式：

1. 状态机不只适用于网络协议#

每一个管理多步骤工作流的系统都受益于显式 FSM。AI4Marketing 的视频管线追踪六个阶段。Research Agent 的 idea 生命周期有 12 种可能状态并带审计转移。Elevator 的 task/subgoal/milestone 层级是嵌套三级状态机。DaaS 通过管线阶段追踪 ingest job 状态。

替代方案——检查布尔标志和时间戳来推断状态——是我在所有四个系统中遇到的最大 bug 来源。“这个实验完成了吗？“应该由 protocol.status == 'analyzed' 回答，而不是从文件存在性和修改时间推断。

2. 单体在生命周期边界出现之前一直是对的#

121 条路由的 AI4Marketing 仍然是单体，对其问题域仍然是正确的选择。当我需要多步骤、多 Agent 编排并带有独立故障域和独立资源预算时（Research Agent），单体变得不可持续。拐点不是代码行数、团队规模或请求吞吐——是系统的不同部分是否需要独立的生命周期管理（启动、关停、崩溃恢复、资源分配、部署节奏）。

3. 零依赖是可靠性工程的一种形式#

DaaS 已经运行数月零依赖相关事故。没有 npm audit 告警，没有上游破坏性 API 变更，没有"库 X 的新版本改了返回类型”。代价是更多样板代码。收益是每一个故障模式都是我自己的代码——可调试、可修复、可预测。与其在别人的事件循环中追查神秘崩溃，我宁可处理我自己线程代码中的已知 bug。

4. 派发谓词胜过优先队列#

Research Agent 和 Elevator 都使用声明式谓词（“这件事该发生吗？为什么或为什么不？"）而非命令式调度（“下一步做这个”）。谓词方式是可组合的（添加新谓词无需修改现有的）、可测试的（单独单元测试每个谓词）、可调试的（某事未派发时读 reason 字段）。优先队列告诉你什么跑了。谓词告诉你什么为什么没跑。

5. 自愈对自主系统不是可选的#

Research Agent 每次无人干预连续运行数周。这只有因为它把故障视为正常运行条件才成为可能：内存看门狗、陈旧锁清理、状态/产物调和、每个周期后的不变量检查和 40+ 条自动修复规则。一个没有自愈能力的自主系统不过是一个静默崩溃并累积不可见状态腐化的系统。

6. 跨家族验证防止模型幻觉洗白#

如果同一个模型（或模型家族）既生产又评估输出，错误会复合而非抵消。模型有系统性盲区，它无法在自己的输出中检测到。Elevator 坚持跨家族验证捕获了同家族审查遗漏的真实 bug。成本是审查环节双倍的 API 支出。收益是在 bug 通过依赖子目标复合传播之前就捕获它们。

元模式#

如果我必须把以上一切压缩为一条架构原则，它会是：

让系统的状态显式、转移可审计、恢复自动化。其余一切都是优化。

AI4Marketing 通过原子条件更新让配额状态显式。DaaS 通过 mtime 失效缓存和原子文件持久化让路由状态显式。Research Agent 通过审计到历史文件的 FSM 转移让研究进展显式。Elevator 通过 DAG 批次和结构化验证检查点让任务执行显式。

给我带来最少运维痛苦的系统，是那些我能通过读一个文件或查询一个端点就回答"现在正在发生什么？“的系统。在凌晨三点把我叫醒的系统，是那些状态隐式的——散落在日志文件中、从时间戳推断、或最糟糕的，编码在其他状态的缺失中（“如果这个文件不存在，那么我们一定在第 3 阶段”）。

架构不是在单体和微服务之间、SQL 和 NoSQL 之间、同步和异步之间做选择。这些是跟随更根本选择的实现细节：**状态住在哪里、如何变化、以及谁被允许改变它？**用清晰和纪律回答这三个问题，系统的其余部分会自己设计自己。
