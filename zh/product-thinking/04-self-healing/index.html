自愈系统：教机器自己修自己

能自己发现、诊断和修复故障的系统的哲学与工程——从 watchdog 反模式到自主改进引擎。

自己修好的那个 Bug#

2026 年 5 月下旬的一个早晨，我醒来看到钉钉上有一条系统消息：“self_heal Rule 37 triggered: restarted research-pipeline after 3 consecutive OOM kills. Root cause: scanner thread retained full PDF buffers across iterations. Applied patch: explicit del after extraction. Validation: 45 minutes post-patch, RSS stable at 1.2GB (was 2.4GB pre-patch).”

这个补丁不是我写的。我甚至不知道有这个问题存在。系统自己注意到了一个模式——六小时内三次 OOM 被杀——通过关联 kill 时间点与 scanner 的活动日志诊断出了内存泄漏，提出了一个有针对性的修复方案，以 git commit 的形式应用了补丁，在 45 分钟的验证窗口中确认了效果，然后在确认成功后通知了我。

这就是自愈。不是吞错误。不是盲目重试。而是一种结构化的、经过验证的故障响应，让系统比故障前更强壮。

我不是一夜之间就拥有了这个能力。这花了几个月的时间：构建、搞坏东西、用惨痛代价学到反模式、逐步将这些经验提炼为系统可以自主消费的东西。这篇文章追溯那段旅程——从一个会杀掉健康进程的幼稚 watchdog，到一个能提议、守卫、应用、验证和学习的完整 kaizen 自动驾驶仪。

为什么自愈比可靠性工程更重要#

传统可靠性工程问的问题是：“我们怎样预防故障？“自愈问的是一个根本不同的问题：“既然故障不可避免，我们怎样让系统对故障的响应成为一种内在能力，而不是一种外部干预？”

这个区别不是学术性的。我在一台 4 核 CPU、7.5 GB 内存的服务器上运行一个研究智能体系统。它管理着 12 个并发 agent（scanner、reader、ideator、experimenter、designer、writer、reviewer、statistician 等等），协调跨多台机器的 fleet worker，维护一个拥有数千节点的知识图谱，并自主发表研究论文。系统 24/7 运行。我是一个人。我要睡觉。

如果每个故障模式都需要我干预，系统一半时间都是死的。数学很简单：一个有 40+ 已知故障模式的系统，即使每种每天发生概率很低，几乎每天都会碰上点什么。如果每个都需要 30 分钟的诊断加修复，我全部清醒时间都会花在救火上。

自愈改变了经济学。每种故障模式，一旦被人类遇到并修复，就被编码为一条下次自动触发的规则。几个月下来，系统积累了制度性知识——不是在没人看的文档里，而是在每分钟都执行的可运行逻辑中。

师傅原则#

我现在构建的每个系统都遵循我所说的"师傅原则”。这个比喻是刻意的：一个老师傅不只是修屋顶——他教徒弟如何识别屋顶什么时候需要修、如何诊断具体的故障、以及如何正确修复。

这个原则有三层：

第一层：即时修复。 东西坏了，你修好它。这是基本功。每个工程师都做得到。

第二层：经验提取。 修完 bug 之后，你提取模式：症状是什么？诊断路径是什么？修复技术是什么？你把这些写下来，不是作为一个模糊的"经验教训"文档，而是结构化数据：问题签名、诊断步骤、修复模板。

第三层：自动化扫描规则。 你把经验编码为系统可以执行的东西。一个主动扫描，在同类问题显现之前就能检测到它。一个自动修复规则，在模式重现时应用同样的修复。一个守卫，从根本上防止条件产生。

大多数工程师止步于第一层。好的工程师能到第二层。我构建的系统被设计为自动到达第三层——kaizen 自动驾驶仪读取过去干预的结果，通过 LLM 提炼教训，并基于观察到的模式提出新的扫描规则。

元原则是：你的代码应该是你的徒弟，而不仅仅是你的工具。

自愈系统的架构#

我的研究智能体的自愈在四个层次上运作，每个层次有不同的时间范围和自主程度：

Level 0：进程监控（秒级）#

supervisor 每分钟通过 cron 运行一次。它的工作很直接：核心进程还活着吗？如果不在了，重启它们。它检查六个 systemd 服务和一个 supervisor 生成的子进程。

research-coordinator    (systemd, on-failure 重启)
research-pipeline       (systemd, on-failure 重启)
research-dashboard      (systemd, always 重启)
research-supervisor     (systemd, always 重启)
research-kg-merger      (systemd, on-failure 重启)
research-dingtalk       (systemd, on-failure 重启)
research-fleet          (supervisor 子进程)

这是最粗暴的自愈形式——进程级重启。它处理崩溃、OOM 被杀和进程卡死。但它是必要的基础。没有这一层，更高层的愈合就没有稳定的基底。

supervisor 还处理"陈旧代码"问题：当 agents/、framework/ 或 lib/ 中的源文件被修改时，coordinator（在启动时 import 了那些模块）运行的是旧代码。supervisor 通过比较源文件的 mtime 和 coordinator 进程启动时间来检测这个情况，带 5 分钟防抖地发送 SIGTERM，如果优雅关闭在 60 秒内未完成则升级为 SIGKILL。

Level 1：健康监测（分钟级）#

健康模块（lib/health.py）提供系统状态的结构化快照：服务状态、知识图谱统计、错误率、fleet 连通性、磁盘使用率和内存消耗。这同时喂给仪表板和 kaizen 观察器。

关键设计决策是：健康不仅仅是"开或关”。它是一个多维信号空间：

  • publish_rate_24h（每天发表的文章数——终极输出指标）
  • alerts_24h（不变量违反次数）
  • ratelimit_24h（API 背压）
  • coordinator_rss_mb、pipeline_rss_mb（内存趋势）
  • worker_sync_lag_s（fleet 一致性）
  • self_heal_recoveries_24h（自愈系统自身的成功率）
  • self_heal_failures_24h（自动愈合失败的升级次数）

每个信号都有元数据：单位、方向（“上升"是好还是坏？）、告警阈值。kaizen 系统将这些信号同时作为观察输入和验证输出。

Level 2：自愈规则（分钟到小时）#

研究智能体在 supervisor 和 coordinator 中有 40+ 条自愈规则。每条规则遵循相同结构：

  1. 触发条件：对日志、指标或状态的模式匹配。
  2. 诊断步骤：确认问题是真实的（不是瞬态毛刺）。
  3. 修复动作：一个具体的、有边界的修复。
  4. 验证：确认修复生效。
  5. 升级路径：如果修复失败，通知人类。

规则用编号标识（Rule 1、Rule 2、…、Rule 41+），并以一致的前缀记录日志：self_heal Rule N: <action>。这使得它们可以被 grep、被计数、被审计。

一条规则的概念性示例：

Rule 12: 过期锁清扫
  触发: 锁文件存在时间超过预期进程生命周期的 2 倍
  诊断: 检查锁文件中的 PID 是否还活着
  动作: 如果 PID 已死，删除锁文件
  验证: 被阻塞的进程应在 60 秒内恢复运行
  升级: 如果锁移除后进程仍然卡住，重启它

Level 3：Kaizen 自动驾驶仪（小时到天）#

这是最有意思的部分。kaizen 系统是一个独立的守护进程，从外部观察研究智能体，提出改进方案，带安全护栏地应用它们，验证结果，并为未来的提议提炼教训。

循环是：观察 -> 提议 -> 守卫 -> 应用 -> 验证 -> 学习。

观察：每 5 分钟，从 host adapter 采样所有信号。构建时间序列。检测异常和趋势。

提议：每小时，一个 LLM（“提议者”）审查最近的观察、当前的教训和历史干预。它生成候选干预——带有明确假设和预测的具体动作。

守卫（预飞检查）：在任何干预被应用之前，它必须通过一系列关卡：

  • 不在被阻止的路径上（kaizen 不能修改自己、git 内部或密钥文件）
  • 不是被阻止的动作类型
  • 在每日干预预算之内（最多 5 次/天）
  • 在并发干预限制之内（最多 2 个同时进行）
  • 不是最近动作的重复（去重窗口）
  • 已定义有效的回滚方案
  • 中等以上风险的：必须已配置金丝雀基础设施
  • 该区域未被冻结（3 次连续回滚会冻结一个区域 24 小时）

应用：adapter 执行动作。对研究智能体来说，这意味着一个包含变更的 git commit。commit SHA 被记录下来用于回滚。

验证：在可配置窗口（最少 60 分钟）之后，检查干预的预测是否成立。目标信号是否按预测方向移动了预测幅度？

学习：每 24 小时，蒸馏器审查所有终结的干预（已验证、无效果、已回滚），提取教训。这些教训有范围（如 “supervisor_rules”、“worker_sync”、“dashboard”），可操作（“当 X 时，优先选择 Y，因为 Z”），且可证伪。它们反馈到下一个提议周期。

元审查：每周，一个更高层次的分析检视校准漂移（预测变得更准还是更不准？）、反复出现的模式、盲区和最高价值的下一步行动。

战争故事：教会我一切的自愈失败#

杀掉健康进程的 Watchdog#

系统早期，我给钉钉监听进程写了一个幼稚的 watchdog。它检查日志文件的修改时间是否够新。如果日志 5 分钟没被写入，watchdog 就假定进程已死并杀掉它。

问题在于：钉钉是消息驱动的。如果 5 分钟没有消息到达（这在夜间是正常的），进程不会往日志写任何东西。它完全健康——只是在等待。watchdog 每天晚上都忠实地杀掉它，引起重启循环，生成虚假的"进程已重启"告警，进而触发其他下游问题。

教训是根本性的：存活信号必须是进程的内在属性，而不是从其输出推断的。 一个没有消息要处理的消息驱动进程不是死了——它在等待。修复方案是给钉钉监听器添加显式心跳（一个周期性的 “still alive” 日志条目），并切换到 systemd 原生的 watchdog 协议（WatchdogSec=）。

但更深层的教训是关于这个反模式本身：用输出产物作为存活探针混淆了"正在工作"和"还活着”。这是正交的属性。系统可以活着但空闲，也可以在忙循环中实际上已死（没有处理任何有用的东西）。好的存活检查测试的是控制平面，不是数据平面。

这个教训被编码为 kaizen 扫描规则：检测任何使用文件 mtime 作为存活信号的监控并标记为需审查。

0% 有效的奖励守卫#

研究智能体有一个"奖励守卫"——一个质量门控，在实验结果被接纳进知识图谱前验证它们。守卫检查结果是否满足最低统计标准（效应量、样本量、复现尝试）。

有两天，守卫命中率是 0%。每一个实验都没被检查就过关了。系统在快乐地积累垃圾数据。

根因是一个 async 竞态条件。守卫被放在了"生产端"——它在 experimenter agent 完成后立即运行，检查磁盘上的结果文件。但 experimenter 异步写结果：agent 报告完成，coordinator 移动到下一步，而实际的文件写入通过异步 flush 在毫秒后才发生。

结果：守卫检查的是一个尚未落盘的文件。什么也没找到。它的逻辑是"如果没有产物存在，就没什么需要守卫的"（对于实验不产生输出的情况，这是合理的默认行为）。所以它通过了。每次都通过。

修复是把守卫移到"消费端"——在下一个消费者（statistician 或 knowledge engine）读取产物的那个时刻验证其有效性。在那个时间点，文件肯定存在（否则消费者会报错）。这就是在消费处守卫，而非在生产处守卫的原则。

在 2 天窗口期内被接纳的 14 篇论文被追溯重新验证（backfill）。其中几篇被标记为需要复现。

这教会了我：在异步管线中，守卫要放在消费边界，而不是生产边界。kaizen 系统现在有一条教训：“在异步流中守卫产物时，优先选择消费端验证而非生产端验证。”

丢了 57% 数据的实验检查点#

experimenter agent 通过 Claude Code CLI 运行 Python 实验。每个实验有多个条件（通常 4-8 个）。CLI 进程有超时。超时触发时，进程被杀。

几个月来，57% 的实验没有可用数据。原因：当超时杀掉 CLI 进程时，下一次运行从头开始。条件 1-3 可能已经成功完成，产生了有效数据——但这些都没被保留。重跑会重做条件 1-3，然后可能在条件 4 再次超时，进入无限浪费计算的循环。

修复有两部分：

  1. 检查点感知：将每个条件的原始计数注入 prompt。experimenter agent 能看到"条件 1：已收集 200 样本，条件 2：200 样本，条件 3：45 样本（部分），条件 4：未开始。“然后它会从条件 3 恢复。

  2. 部分数据保留：即使实验被杀，已经写入的数据也不被丢弃。statistician 被教会处理部分数据集（标记它们统计功效较低但仍有信息价值）。

结果：实验完成率从 43% 提高到 89%。剩余 11% 是真正困难的实验，需要的计算量超过超时允许的范围——而且它们跨运行累积数据，而不是每次都从零开始。

原则：永远不要设计一个崩溃意味着全部数据丢失的系统。 每个长时间运行的操作都应该是可检查点的。这在数据库设计中是显而易见的（预写日志），但很少被应用于 LLM agent 工作流，那里人们把每次调用当作无状态的。

永远不会断开的熔断器#

系统有一个 API 调用的熔断器（DashScope、Claude、外部服务）。当某个服务返回太多错误时，熔断器打开，将调用重定向到降级路径。

好几周，我注意到对某个服务的写操作以约 30% 的比率失败，但熔断器从未触发。读操作是成功的。熔断器显示一切正常。

bug 在 record_success() 里。当记录一次成功调用时，它清除了整个失败窗口——不是递减计数器，而是把计数归零。在读成功/写失败交替的模式下，每次成功的读都在写失败能累积到阈值之前把失败计数器重置为零。

修复区分了熔断器的状态：

  • CLOSED（正常）：失败按时间窗口跟踪；旧失败按时间过期，但成功不会清除它们。
  • HALF_OPEN（测试中）：单次探针调用成功 → 回到 CLOSED 并重置。
  • OPEN（已触发）：所有调用被短路到降级路径。

关键洞察：在 CLOSED 状态下，成功和失败是独立证据。一次成功不会撤销一次失败。它们可能打的是不同端点或代码路径。只有在 HALF_OPEN——我们显式地在测试"问题解决了吗？"——一次成功才意味着"清除失败记录”。

这是一个微妙的点，大多数熔断器实现都搞错了。流行的模式"M 秒内 N 次失败，任何成功都重置"对混合操作服务来说从根本上是有缺陷的。

自愈不是吞错误#

我必须把这一点讲清楚，因为对自愈系统最常见的误解是：它们只是精心包装的 try/except: pass。

自愈是吞错误的反面。吞错误隐藏问题。自愈把问题浮现出来、诊断它、修复它、验证修复、并记录教训。实现自愈后系统的可观测性更高了，不是更低。

每个自愈动作都记录了完整的决策链：

  • 什么触发了它（异常或模式匹配）
  • 它诊断了什么（根因假设）
  • 它做了什么（具体动作）
  • 它预期什么（预测）
  • 实际发生了什么（验证结果）
  • 它学到了什么（提炼的教训）

如果我运行 grep "self_heal Rule" /data/research-agent/logs/*.log | wc -l，我会得到每次自主干预的计数。我可以审计每一条。我可以不同意系统的选择。我可以添加约束。kaizen 系统的护栏（每天最多 5 次干预、强制回滚计划、修复失败后区域冻结）存在的目的正是防止系统陷入不可观察的自我修改。

设计原则：自愈必须比手动修复更透明，而不是更不透明。 当我手动修某个东西时，我可能忘了记录。当系统修某个东西时，它在结构上不可能不记录——记录就是机制本身。

进化路径：从被动到主动到自主#

回顾系统的演进，我能识别出三个不同的阶段：

第一阶段：被动（第 1-2 个月）#

一切都是手动的。supervisor 对崩溃的进程运行 systemctl restart。我收到钉钉的错误通知。我登录、诊断、修复。平均恢复时间：30-60 分钟（取决于我什么时候注意到）。

系统大概有 5 种已知故障模式，都通过"重启那个东西"来处理。当系统简单时（一个管线线程、没有 fleet、没有知识图谱），这就够了。

第二阶段：主动（第 3-4 个月）#

我开始编码模式。supervisor 不仅仅重启崩溃的进程，还能检测预崩溃条件：内存向 OOM 攀升、磁盘快满、API 速率限制接近。它可以采取预防动作：触发垃圾回收、轮转日志、降低 API 调用频率。

自愈规则从 5 条增长到 20+ 条。每条规则都有文档：当我添加 Rule 15（如果 scanner 线程的 RSS 超过 2GB 就重启 pipeline）时，我写下了为什么、什么条件触发了它、以及预期结果是什么。

已知故障模式的恢复时间降到了 1-5 分钟（大多数在我注意到之前就解决了）。但未知故障模式仍然需要我的干预。

第三阶段：自主（第 5 个月以后）#

kaizen 自动驾驶仪把系统从"固定响应库"带到了"提出新颖响应"。它不限于我写的那 40+ 条规则——它可以观察系统、对从未见过的问题提出假设、并提出修复方案。

三条硬性规则（每个干预都有回滚、中等以上风险需要金丝雀、被守卫信号回归时自动回滚）提供了安全网。系统可以在自己身上做实验，但不能永久性地搞坏自己。

最有意思的涌现行为：kaizen 系统开始提议改进自愈规则本身。Rule 22 触发得太激进了（重启一个实际上健康但速度较慢的进程）。Kaizen 注意到了模式（高重启次数 + 输出指标无实际改善），提议放宽阈值，应用了它，验证了重启次数下降且输出无退化，并提炼了一条教训：“Rule 22 的阈值是为较小的知识图谱校准的；随着图谱增长，查询时间自然增加。”

这是真正的元学习。系统不仅在修 bug——它在改进自己的 bug 修复机制。

自主修改的三条铁律#

任何修改自身的系统都需要硬性约束。没有它们，你得到的是一个把自己"修"成不可恢复状态的系统。我的三条规则，通过惨痛经验学到：

规则一：每个干预都有回滚方案#

没有回滚方案就不应用。这在守卫层强制执行——预飞检查会拒绝任何不包含具体回滚动作的干预。对研究智能体的 adapter 来说，回滚就是 git revert，这使它们确定性且可审计。

为什么这很重要：kaizen 早期曾提议一个"修复"来修改 SQLite schema。“回滚"是"从备份恢复”。但并没有做备份。如果 schema 变更导致了数据损坏，恢复将是不可能的。现在，adapter 要求任何修改 schema 的动作都必须在回滚方案中包含备份步骤。

规则二：中等以上风险需要金丝雀#

对于可能影响系统稳定性的变更（修改进程参数、改变 API 路由、修改管线逻辑），系统必须有金丝雀：一部分流量或影子模式，让变更可以在不完全提交的情况下被验证。

在实践中，这意味着某些干预因为那个区域不存在金丝雀测试基础设施而被拒绝。这是设计使然——它迫使我在系统能在某个区域自我修改之前先建设金丝雀能力。约束是生成性的。

规则三：三振出局就冻结#

如果一个区域连续收到三次回滚（应用了干预然后因守卫违规被回滚），该区域被冻结 24 小时。不再提议或应用进一步的干预。

这防止系统震荡：在一个问题是结构性的、需要人类洞察力的区域反复尝试同类修复。冻结是系统在说"我试了三次都失败了。我需要帮助。"

给任何构建自愈系统的人的教训#

运营一个自己修自己的系统几个月之后，如果从零开始，我会告诉别人这些：

先做监控，再做智能。 在你构建 LLM 驱动的提议者之前，确保你的进程崩溃时会重启。这处理了 60% 的生产问题。

先让故障可观测，再让它可修复。 你不能治愈看不见的东西。在投入自动修复逻辑之前，先投入结构化健康快照、错误聚合和信号时间序列。

在消费处守卫，而非生产处。 在异步系统中，在产物被读取的地方验证它，而不是在写入的地方。写入可能还没完成。产物可能在写入后被损坏了。消费者知道它需要什么。

区分存活和生产力。 进程可以活着但不产出（在等待输入）。进程可以在产出但实际上已死（无进展的无限重试循环）。存活探针应该测试控制平面。生产力探针应该测试数据平面。不要混淆它们。

编码"为什么"，不仅仅是"什么"。 当你写自愈规则时，包含为什么这个修复是合适的。kaizen 系统的教训结构是"当 X 时，优先选择 Y，因为 Z"。“因为 Z"这部分让系统知道教训何时不再适用。

接受自愈有成本。 每条自愈规则都是需要维护的代码。每次 kaizen 干预都是可能引入新 bug 的变更。系统的复杂性在增长。为此做预算：每天最大干预次数、强制验证窗口、区域冻结。增长必须有界。

目标不是零人工干预。 目标是人工干预只用于新颖问题，而不是反复出现的问题。每个需要人工干预的反复出现的问题都是自愈系统的失败。每个被升级给人类的新颖问题都是成功。

徒弟原则#

这篇文章以一个在我睡觉时修好内存泄漏的系统开始。我想以驱动上面一切的哲学框架结束。

传统软件工程把代码当工具。你写了它，它做你告诉它做的事，出问题时你来修。关系是单向的：人 -> 代码。

自愈系统反转了这个关系的一部分。代码观察自身、提出变更并执行它们。但——这是关键——它在一个由人设计的约束、验证和教训积累框架内这样做。系统不是"想做什么就做什么"意义上的自主。它是"处理日常问题以便人可以专注于困难问题"意义上的自主。

正确的比喻不是机器人取代人类。而是徒弟向师傅学习。师傅（我）修问题，并在修的过程中教徒弟（系统）如何识别和修复那一类问题。随着时间推移，徒弟处理越来越多的日常工作。师傅的角色从"做"转变为"教"再转变为"审计”。

这就是师傅原则在最深层次的含义。每次我修一个 bug，我不仅仅是在修——我是在教。而我教学的质量决定了我的徒弟会变得多能干。

理想状态是一个我可以消失一周、回来后发现它比我离开时更健康的系统——不是因为什么都没出错，而是因为每件出错的事都被处理了、验证了、记录了、并从中学习了。

我们还没到那里。但每一条自愈规则、每一条 kaizen 教训、每一次守卫验证都在把我们推近。而且系统比六个月前学得更快了，因为它在从自己的学习中学习。那个递归品质——改进自我改进的自我改进——我认为是今天系统工程中最有意思的前沿。

你的代码应该是你的徒弟，而不仅仅是你的工具。用心教它。
