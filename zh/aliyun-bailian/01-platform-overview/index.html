阿里云百炼实战（一）：平台总览与第一个请求

一个工程师视角的阿里云百炼（DashScope）导览——模型目录里真正能用的那几个、两种 endpoint 形态、异步任务模式，外加一个 hello-world 请求把后续文章的基础铺好。

Read in English →

只要你的产品要面对中文用户，迟早会调一次百炼。Qwen-Max 是拿到 GPT-4 级别中文理解能力最便宜稳妥的方式；万相视频是国内唯一能开人民币发票的生产级文生视频 API；Qwen-TTS-Flash 是唯一能把粤语和四川话讲得不像海关广播的 TTS。在某 AI 营销平台跑了一年生产之后，这个系列就是我希望第一天有人塞给我的那本书。

第一篇是地理课：百炼是什么、有哪些模型家族、两种 endpoint 怎么分，最后跑通一个请求，把后续不再重复的基础铺好。

百炼是什么，DashScope 又是什么

这两个名字确实容易混，因为阿里中途改过名。官方的 “DashScope” 文档从 API 角度讲，“百炼” 文档从控制台角度讲。同一个产品，两个名字。

文档里你会在同一段看到两个名字。当作"控制台 vs API"理解就好。说"部署百炼应用"指的是控制台，说"DashScope 报错"指的是 API 返回了非 200。

你真正在意的模型目录

百炼上挂着上百个模型。这一年生产里，我只为下面这些付过钱：

家族代表 model_id用途
Qwen LLM（文本）qwen-max, qwen-plus, qwen-turbo, qwen3-max, qwen3-coder-plus对话、推理、工具调用、代码
Qwen-Omni（多模态）qwen3-omni-flash, qwen3.5-omni-plus视频 / 音频 / 图像理解
Qwen-VL（视觉）qwen3-vl-plus只做图像理解（比 Omni 便宜）
万相（视频）wan2.5-t2v-plus, wan2.5-i2v-plus文生视频、图生视频
Qwen-TTSqwen3-tts-flash语音合成，40+ 音色
向量text-embedding-v3, text-embedding-v4向量检索

不在这张表里的，要么已经弃用，要么是薄变体，要么是研究预览。盯着这几个用，就不会上线 6 周后被某个模型 EOL 给坑了。

计费一段话讲完

LLM 按 token 计费（输入和输出分开定价，输出贵 2-4 倍），TTS 按音频秒数收，万相按视频秒数收，向量按调用次数收。每个模型都有免费额度——通常是 100 万 token 或者 100 次生成——新版本上线时还会重置，意思是你只要愿意切版本，几乎可以一直白嫖原型阶段。生产流量一定要用独立的 API key 配上预算告警；我吃过一次四位数的账单，原因是有人晚上忘了关一个 debug 循环。

API key：别提交进 git

去控制台左侧 API-KEY 拿。会有一个默认空间的 key，外加每个工作空间各自的 key；任何生产项目都用工作空间 key，将来轮换也不影响 dev。然后：

1

export DASHSCOPE_API_KEY="sk-xxxxxxxxxxxxxxxxxxxxxxxx"

本系列每段代码都直接读 os.environ['DASHSCOPE_API_KEY']。别硬编码，别把 .env 提交进仓库，生产环境请把它放到密钥管理器后面。DashScope 团队会撤销泄露的 key，但要等公开爬虫先扫到，那时候已经晚了。

两个 endpoint：OpenAI 兼容 vs DashScope 原生

这是整篇文章最重要的事实。按 Qwen API reference，百炼的每一个文本/多模态模型都有两个 HTTP 入口。

OpenAI-compatible endpoint

Base URL：https://dashscope.aliyuncs.com/compatible-mode/v1（中国大陆），或 https://dashscope-intl.aliyuncs.com/compatible-mode/v1（新加坡）。

走的是 OpenAI 协议。把官方 openai Python SDK 指过来，95% 的现有 OpenAI 代码原样能跑。这是我的默认选择。

client = OpenAI(

resp = client.chat.completions.create(

    ],
)
print(resp.choices[0].message.content)

DashScope 原生 endpoint

Base URL：https://dashscope.aliyuncs.com/api/v1/...——按模态分路径（/services/aigc/text-generation/generation、/services/aigc/multimodal-generation/generation、/services/aigc/text2video/...）。

这是阿里自家的协议——请求结构不一样，字段名也不一样（用的是 input.messages 不是 messages、还有单独的 parameters 块等等）。可以走 dashscope SDK，也可以直接 HTTP。

resp = dashscope.Generation.call(
    model="qwen-plus",
    messages=[
        {"role": "system", "content": "你是一个资深后端工程师。"},

    ],
    result_format="message",
)
print(resp.output.choices[0].message.content)

什么时候该用哪个

我用血泪经验提炼出来的规则：

  • OpenAI-compatible：默认走这条。普通对话、function calling、JSON mode、streaming 都行。一行代码就能 A/B 切到 GPT-4。
  • DashScope 原生：万相视频必走、Qwen-TTS 必走、Qwen-Omni 多模态推荐走（兼容层会丢掉一部分视频参数）、需要异步任务模式时必走、最新功能还没回灌到兼容层时必走。

最常见的坑：以为"OpenAI-compatible" 意味着所有模型都能这么用。错。wan2.5-t2v-plus 只能走原生。qwen3-tts-flash 只能走原生。第四第五篇会反复强调这件事。

经验： 收到 400 + parameter X is not supported 之类报错，第一反应去查是不是把"原生 only"的模型走到了兼容 endpoint。我修过的"百炼坏了"工单里，大约一半都是这个原因。

三个反复出现的概念

model_id

每次调用都以 qwen-plus 或 wan2.5-t2v-plus 这样的字符串作为 key。模型本身不带版本号——阿里在同一个 ID 下不断发新权重，并在 changelog 里告诉你。要锁版本就用模型卡里列出的日期别名（比如 qwen-plus-2025-09-11）。任何面向客户的场景请锁日期——没锁的别名我见过一夜之间就换了风格。

异步任务

任何超过 ~30 秒的事（视频生成、大批量 embedding、长文 TTS）都走异步。模式永远是这一套：

  1. POST 到 create endpoint，加上 header X-DashScope-Async: enable。
  2. 拿回 task_id。
  3. 轮询 GET /api/v1/tasks/{task_id} 直到 SUCCEEDED 或 FAILED。
  4. 在 24 小时内下载输出 URL——会过期。

第四篇有完整的退避轮询实现。

流式

LLM 和 Qwen-Omni 都支持 SSE 流式。Qwen3 配上 enable_thinking=True 时必须走流式——非流式会被拒绝。Qwen-Omni 则是流式强制（第三篇细讲）。早点习惯 stream=True，你用到它的频率会比你想象中多。

import os
from openai import OpenAI

def main():
    api_key = os.environ.get("DASHSCOPE_API_KEY")
    if not api_key:
        raise SystemExit("先 export DASHSCOPE_API_KEY")

    client = OpenAI(
        api_key=api_key,
        base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    )

    stream = client.chat.completions.create(
        model="qwen-plus",
        messages=[
            {"role": "user", "content": "用一段话解释百炼是什么。"},
        ],
        stream=True,
    )

    for chunk in stream:
        delta = chunk.choices[0].delta.content
        if delta:
            print(delta, end="", flush=True)
    print()

if __name__ == "__main__":
    main()

想确认原生 endpoint 也能用，把上面 dashscope 那段代码替换跑一遍。两条路都该成功，账单也一致。

控制台、SDK、OpenAI 兼容——分别什么时候用

实际上有三个地方可以驱动百炼，选错了影响比文档里说得更大。我花了两个月在同一个产品里混用三种方式，才把这件事搞明白。

控制台（bailian.console.aliyun.com）只干两件事：开通（建工作空间、API key、RAM 授权、应用）和诊断（看每次请求日志，定位为什么模型返回了垃圾）。它不该承担运行时流量。控制台里的"Playground"对调 prompt 很方便，但它会忽略你 SDK 调用里要传的部分参数，所以 Playground 里 work 的 prompt 上线后还会出问题。我把 Playground 当成"模型可达"的探活，不是"prompt 是对的"的证明。

DashScope 原生 SDK（pip install dashscope）适合所有阿里特有的场景：异步任务、视频 / TTS / 万相、最新模型参数、跨工作空间计费的 workspace-id header、调试用的 X-DashScope-DataInspection header。原生 SDK 暴露的 parameters 块里，有兼容层会丢掉的字段（incremental_output、图像生成的 seed、联网模型的 enable_search）。批量推理、RemoteService 部署也只能走原生这一条路。

OpenAI 兼容 endpoint 适合：你的代码本来就写在 OpenAI SDK 上，想用一行代码切换百炼和 OpenAI 做 A/B。兼容层是个薄翻译器，能讲 chat.completions.create、embeddings.create、支持 tools / tool_choice / response_format / stream，并且接受 extra_body 传那几个 Qwen 特有的旋钮（enable_thinking、enable_search）。它讲不了万相、TTS 或任何异步模式。要那些就降到原生 SDK 那一支调用——同一个 app 里两个客户端同时存在没问题。

我跑了一年沉淀下来的规则：

  • 新项目，主要用 LLM，想保持 provider 无关 → OpenAI SDK 指到兼容 endpoint。
  • 多模态混合（LLM + Omni + 视频 + TTS）→ 原生 SDK 全用，只把 chat completion 那一支留在兼容 endpoint 上，让消息处理代码可移植。
  • 生产排错 → 全切原生，因为错误信封更全（code + message + request_id，兼容层只给你 OpenAI 形状的 error）。

Region 路由和 RAM 授权：必须知道的事

百炼真正在用的 region 有两个：北京 和 杭州（外加新加坡的"国际"endpoint，功能上算第三个 region）。默认的 dashscope.aliyuncs.com 解析到你账号最初开通时所在的 region。这是"昨天还好好的"工单数最多的单一原因。

具体陷阱是有些模型 region 锁定。wan2.5-t2v-plus 在北京池里。qwen3-omni-flash 多 region。text-embedding-v4 杭州先上、北京 6 周后才上。如果你账号路由到的 region 还没上这个模型，会拿到一个完全合法的 404，报 Model not exist——你会先花一小时去调 prompt，然后才意识到是路由问题。

# 直接打 region URL，强制走某 region
client = OpenAI(
    api_key=key,

import dashscope
dashscope.api_key = key
dashscope.base_http_api_url = "https://dashscope.aliyuncs.com/api/v1"

RAM 这边的坑是：API key 绑工作空间，工作空间挂主账号下面。RAM 子账号不会自动继承主账号下百炼工作空间的访问权。要么图省事直接发 AliyunBailianFullAccess，要么写一个 scope 到 workspace ARN 的自定义策略。Resource 写法是 acs:bailian:*:*:workspace/<workspace_id>——workspace id 在控制台 URL 栏里，不在任何明显的"详情"页里。RAM 用户能看控制台、API 调用却报 Forbidden.RAM，几乎都是少了这条策略。

生产环境我每个环境（dev / staging / prod）开一个独立 RAM 子账号，每个子账号一个工作空间，每个工作空间一把秘钥管理器托管的轮换 API key。跨环境泄漏就成了显式策略动作，不会意外发生。

没人提前告诉你的配额真相

营销页面写着"高并发、低延迟"。实际数字非常具体，且对免费 tier 账号非常低。三个旋钮：

  • RPM（每分钟请求数）——任意 60 秒窗口内能发的请求数。
  • TPM（每分钟 token）——in + out 总 token 计量。
  • TPD（每天 token）——按北京时间 00:00 重置的日上限。

我自己在新工作空间（没申请配额）测出的默认值：

模型RPMTPM异步并发
qwen-turbo500500K不适用
qwen-plus200200K不适用
qwen-max60100K不适用
qwen3-omni-flash60100K不适用
wan2.5-t2v-plus不适用不适用5
qwen3-tts-flash10050K不适用

这些不是合同数字，阿里会悄悄改。真正要记住的是：免费默认值够开发、够 100 DAU 的原型用，一条爆款推文带来真流量的那一刻就被限流。报错是 Throttling.RateQuota + HTTP 429——你的重试包装器要把 429 当成"退避重试"，不是"致命"。

提配额走控制台 API-KEY → 限流配置，写理由提交。合理诉求（当前的 10×）一般 24 小时内批。直接申请 100× qwen-max 又拿不出营收数据会被拒——准备好你的预测 QPS、平均 prompt 长度、商业理由。这条流程我走过 4 次，只被拒过 1 次。

当前 Qwen 全家定价对照

定价页每季度都重排版，下面具体数字会漂，但比例稳得能拿来做规划。截至我上次核对（2026-04），单位是元/百万 token：

模型输入输出缓存输入备注
qwen-turbo0.30.60.15分类用的便宜大头
qwen-plus0.82.00.4日常主力
qwen-max206010推理、代码评审
qwen3-max249612最新，比 qwen-max 贵
qwen3-coder-plus4162代码专精，中档价位
qwen3-omni-flash2.5（文本）/ 12（视觉）5不适用视觉 token 单独计
text-embedding-v40.7（每百万 token）不适用不适用按调用计

两件事必须刻到肌肉里：

  • 输出比输入贵 2-4 倍。 一个会循环 3 次的 function-calling agent，账单大头永远在 assistant 的输出 token 上，不在用户 prompt 上。要砍 prompt 先砍输出，再砍 system。
  • 缓存输入半价。 百炼 2025 年底上线了隐式 prompt 缓存——同一前缀 5 分钟内再次发送命中缓存。你不用主动开，省钱也只在你 log 了 usage.cached_tokens 时才看得到。固定 system prompt 的 RAG 应用，缓存命中率一般 60-80%，账单按比例下降。

万相（按视频秒数）和 Qwen-TTS（按音频秒数）的合理单位是"按完成资产计"，不是按 token 计。720p 5 秒大约 1.5 元；60 秒 TTS 旁白大约 0.6 元。都便宜到瓶颈是人工审核，不是 API 花销。
