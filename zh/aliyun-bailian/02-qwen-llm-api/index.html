整个系列里，这一篇带来的生产收益最大。其它模型都很有趣，但真正让我每个上线的百炼产品分分钟都在调的，还是 LLM。官方 Qwen API reference 详尽但稠密，本文从中挑了一条可读的路径走一遍。

按场景挑 Qwen 变体

Qwen 家族很大。多数团队默认上 qwen-max，钱花得冤；少数团队默认 qwen-turbo，省了质量。正确答案永远是"按任务挑变体"。

我自己的生产经验：

  • qwen-turbo — 用来做分类、意图识别、短摘要，以及那种单次用户请求里要调 10 次以上的场景。Qwen 里最便宜的那一档，抽取场景下表现出乎意料地好。
  • qwen-plus — 日常主力。对话、RAG 综合、多步推理。是性价比的拐点。
  • qwen-max / qwen3-max — 用在代码评审、复杂推理这类场景，也就是答错的代价比答慢更大的事。
  • qwen3-coder-plus — 所有代码任务都用它。同等参数规模下，写代码明显比通用的 qwen-plus 强。
  • qwen3-vl-plus / qwen3-omni-flash — 用在图像/视频/音频输入上。第三篇专门讲。

经验： 一个常见错误是用 qwen-max 去做 embedding 类的分类任务。别这么干。qwen-turbo 配一个紧凑的 system prompt，能省 10 倍成本，质量基本不掉。

一次请求实际发了什么

不论你走 OpenAI 兼容还是原生，一次 chat 的本质都一样：模型 ID、messages 数组、参数块。

最常碰的几个字段：

  • messages — {role, content} 数组。role 是 system / user / assistant / tool。文档里说，多模态模型的 content 还可以是带类型的部分数组（text、image_url、input_audio、video_url）——具体见第三篇。
  • temperature — 0.0-2.0。我做抽取/分类时设 0.0，默认对话用 0.2-0.4，做创作 0.7 起步。文档默认大约 0.7，对多数 agent 场景来说偏高。
  • top_p — 不知道为什么要改就别改。temperature 和 top_p 一起调是混乱的配方。
  • max_tokens（兼容）/ parameters.max_tokens（原生）— 这是输出上限，不是总上限。一定要设，否则一次失控就要算钱。
  • stream — 切到 SSE 流式。
  • response_format={"type": "json_object"} — JSON mode。比单纯在 prompt 里写"请返回 JSON"要靠谱得多。
  • tools / tool_choice — function calling。

import json, os
from openai import OpenAI

client = OpenAI(
    api_key=os.environ['DASHSCOPE_API_KEY'],
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
)

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询某城市当前天气",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
        },
    },
}]

def call_weather(city):
    # 真实场景：调你自己的 API。这里 stub。
    return {"city": city, "temp_c": 22, "conditions": "晴"}

messages = [{"role": "user", "content": "去上海要带伞吗？"}]
resp = client.chat.completions.create(
    model="qwen-plus", messages=messages, tools=tools,
)
msg = resp.choices[0].message

if msg.tool_calls:
    for call in msg.tool_calls:
        args = json.loads(call.function.arguments)
        result = call_weather(**args)
        messages.append(msg)
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": json.dumps(result),
        })
    final = client.chat.completions.create(model="qwen-plus", messages=messages)
    print(final.choices[0].message.content)
else:
    print(msg.content)

三个容易踩的坑：

  • 第一次响应和工具结果之间，必须 messages.append(msg)。 模型需要在历史里看到自己之前的 tool_call 消息，否则第二次调用会 400，报"orphan tool result"。
  • tool_choice="auto" 是默认值。 必要时强制指定工具：tool_choice={"type": "function", "function": {"name": "..."}}——工作流首次调用时常这么干。
  • parallel_tool_calls=True 是支持的。工具之间互相独立时就用——模型会一次返回多个 tool_calls。

resp = client.chat.completions.create(
    model="qwen-plus",
    messages=[
        {"role": "system", "content": '返回 JSON：{"sentiment": "positive|negative|neutral"}'},
        {"role": "user", "content": "这个产品我超喜欢。"},
    ],
    response_format={"type": "json_object"},
)
data = json.loads(resp.choices[0].message.content)

生产里有两点要注意：

  • 模型有时候还是会用 ```json 把结果包起来。在 json.loads 之前先把围栏剥掉更稳。
  • 如果你要严格的结构化（比如配 Pydantic schema），改用 function calling。约束更严，调试也更友好。

enable_thinking 和流式陷阱

Qwen3 系列模型支持 enable_thinking=True——让模型先产推理链，再产最终答案。质量提升明显，尤其是重推理任务。但必须开流式。 非流式直接 400。

实操模式——把推理链丢去日志，把答案流给 UI：

stream = client.chat.completions.create(
    model="qwen3-max",
    messages=[{"role": "user", "content": "如果一只表每天慢 5 分钟，10 天后差多少？"}],
    extra_body={"enable_thinking": True},
    stream=True,
)

reasoning, answer = [], []
for chunk in stream:
    delta = chunk.choices[0].delta
    rc = getattr(delta, "reasoning_content", None)
    if rc:  reasoning.append(rc)
    if delta.content: answer.append(delta.content)

print("ANSWER:", "".join(answer))
print("(推理链已隐藏，", sum(len(r) for r in reasoning), "字符)")

我会把 reasoning 转给日志系统，永远不给用户看。三个原因：(1) 客户看到了会泄露 chain-of-thought 的 IP；(2) 非技术读者看了会困惑；(3) 会让可见响应长度直接翻倍。

异步 chat（少见但有用）

特别长的对话（比如 30k token 的 RAG 综合），可以加上 X-DashScope-Async: enable 异步提交，再轮询结果，跟万相一样的玩法。Qwen API reference 里"异步调用"章节有写。我用它来跑那种不需要立即响应的定时批量摘要任务。

真正能省钱的成本控制

  • 永远要设 max_tokens。 默认上限是"模型最大"，失控循环要命。
  • 每个环境一个工作空间 key。 在控制台里给生产 key 设硬性日预算。
  • 打印 token 数。 每个响应都有 usage.prompt_tokens 和 usage.completion_tokens。每周聚合一次，能发现某个 prompt 不知不觉膨胀了 3 倍。
  • 同前缀的请求自己缓存。 DashScope 现在没像 Anthropic 那样把 prompt caching 暴露出来——高 QPS 同前缀场景请自己在边缘缓存。

Token 计数：DashScope 和 tiktoken 的差异，CJK 膨胀

从 OpenAI 生态过来的团队最大的意外是：tiktoken 算 Qwen token 数会撒谎。 Qwen tokenizer 跟 cl100k_base 或 o200k_base 不是同一个 BPE。如果你用 tiktoken.encoding_for_model("gpt-4o") 来估算上下文预算，中文会差 20-40%，英文差 5-10%。我有一个周五晚上就栽在这上面：一个 RAG 流水线 tiktoken 算下来"明确小于 32k 上下文"，Qwen 这边算出来 41k。

正确做法是本地用 Qwen 自家 tokenizer：

# pip install transformers tiktoken
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B", trust_remote_code=True)

def count_qwen_tokens(text: str) -> int:
    return len(tok.encode(text))

print(count_qwen_tokens("阿里云百炼 DashScope SDK"))   # 9
# 同一字符串 tiktoken o200k_base 算出来 11

任意一个 Qwen 模型的 HuggingFace 仓库里都带 tokenizer——Qwen2.5-7B、Qwen3-7B 等共享一套 tokenizer 家族，加载任意一个，你拿到的计数和 DashScope 实际计费数差距 ±1 token 之内。事后请求响应里的 usage.prompt_tokens 和 usage.completion_tokens 才是权威，能拿到就以它为准。

CJK 膨胀是真问题，得在预算里算进去。一句典型中文大约 1.5 token / 字，英文大约 0.25 token / 字（约 4 字 / token）。1000 字中文 RAG 上下文要花 1500 token；同样长度英文只要 250 token。规划上下文窗口时按 token 来，不按字数来，并用本地 tokenizer 估。我亲眼见过一个"用 100k 字上下文"的方案最后变成"我们需要 150k 上下文模型"——一次就够了。

流式 + 背压：drain 模式、partial JSON 解析

最朴素的流式代码就像前面那段——迭代 stream、append、结束。CLI demo 没问题。生产 HTTP 服务里你多两个问题：背压（下游比模型产出慢）和部分解析（用户要的是结构化输出，但 buffer 卡在 token 中段）。

背压：把 stream chunk 转发给慢客户端时（4G 上的手机浏览器），chunk 在你的进程内存里堆积，要么 OOM，要么上游连接超时。修法是把上游 drain 到一个有界队列，再把压力反推给客户端连接：

import asyncio
from openai import AsyncOpenAI

client = AsyncOpenAI(api_key=key, base_url="https://dashscope.aliyuncs.com/compatible-mode/v1")

async def relay(send_to_client):
    stream = await client.chat.completions.create(
        model="qwen-plus", messages=msgs, stream=True,
    )
    queue: asyncio.Queue[str | None] = asyncio.Queue(maxsize=32)

    async def producer():
        async for chunk in stream:
            d = chunk.choices[0].delta.content
            if d:
                await queue.put(d)        # 队列满了就阻塞 → 反向背压
        await queue.put(None)

    async def consumer():
        while True:
            d = await queue.get()
            if d is None: break
            await send_to_client(d)        # 客户端慢，队列填满，producer 就阻塞了

    await asyncio.gather(producer(), consumer())

队列大小（这里 32）就是你允许的飞行 buffer 深度。小一点背压更敏感，但下发会略微卡顿。SSE 转 WebSocket 跨公网时，32 是我跑下来的合适值。

部分 JSON：输出是 JSON、且想边到边渲染（比如表单边接边填）时，不能等流结束才 json.loads。诀窍是用流式 JSON 解析器，比如 json-stream 或 partial_json_parser：

# pip install partial-json-parser
from partial_json_parser import loads as partial_loads

buffer = ""
last_render = None
for chunk in stream:
    buffer += chunk.choices[0].delta.content or ""
    try:
        partial = partial_loads(buffer)   # 能解析多少解析多少，缺的填 None
        if partial != last_render:
            render_to_ui(partial)
            last_render = partial
    except Exception:
        continue   # 还不够一个有效前缀，跳过

这就解锁了"表单在用户眼前一格一格填出来"的体验，不用等模型出完。我们营销工具里那几个结构化抽取接口都是这么做的——感知延迟从 4 秒降到 500ms 以内，墙钟时间其实没变。

def run_agent(user_msg, tools, dispatch, max_rounds=8):
    messages = [{"role": "user", "content": user_msg}]
    for round_n in range(max_rounds):
        resp = client.chat.completions.create(
            model="qwen-plus", messages=messages, tools=tools,
            parallel_tool_calls=True,
        )
        msg = resp.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:
            return msg.content
        # 并行：一次性派发所有 tool call
        for call in msg.tool_calls:
            args = json.loads(call.function.arguments)
            result = dispatch(call.function.name, args)
            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": json.dumps(result, ensure_ascii=False),
            })
    raise RuntimeError("agent 没收敛")

文档没说清楚的三件事：

tool_choice="auto" 死循环陷阱。 设 auto 时，模型自己决定要不要调工具。Qwen-Plus 上我反复看到它对同一个城市连调 get_weather 4 轮，每轮都觉得上一轮答得不够。修法三选一：(a) 加一个硬 max_rounds 上限（我必加）；(b) 第 3 轮之后强制 tool_choice="none"，逼模型直接答；(c) 检测重复 tool call 短路掉。我生产里用 (b)——给 agent 3 轮调工具机会，剩下一轮只能直接答。

并行 tool call 在一条 assistant 消息里返回。 开 parallel_tool_calls=True 时，msg.tool_calls 是多个 call 的列表。你必须先 append 那一条 包含全部 call 的 assistant 消息，再每个 call append 一条 tool 消息，最后才发下一次请求。如果不夹 assistant 消息直接 append tool 消息，立刻 400 报 orphan tool result。

Tool 参数 schema 要扁平，要 required。 Qwen 的 function calling 在嵌套深的参数上明显比 GPT-4 不稳。一个 {type: object, properties: {filter: {type: object, properties: {date: {...}, region: {...}}}}} 的 tool，date 一般出得对，region 大约 15% 概率漏掉。把参数拍平到顶层、标 required，可靠性能干到 99% 以上。这条我是花了一周"模型今天怎么这么蠢"的吐槽才学会的——其实是 schema 太深。

enable_thinking 的细节：什么时候帮、什么时候坏、要付什么代价

enable_thinking=True 的宣传是"免费质量提升"。它既不免费，也不总是提升。生产里跑了半年，我的分类如下：

会帮的场景：

  • 多步推理（数学应用题、逻辑题、代码执行 trace）。
  • “用户是否同时要求 X 和 Y 但不要 Z” 这类多约束分类。
  • 代码评审，多文件上下文。
  • 任何你自己也想先打草稿再下结论的事。

没帮甚至添乱：

  • 纯抽取（“从这段文字提 5 个字段”）。推理链只是把输入复述一遍。
  • 短事实查询。一字答案先想 2 秒。
  • 高温创作。推理会把风格往中性方向拉。
  • Tool-calling agent。推理内容会以微妙方式干扰 tool 决策——开 thinking 时我见过 call 直接漏掉 tool_choice 信号。

延迟代价： TTFT（首字节时间）从 ~400ms 涨到 ~1.5-3 秒，因为推理链得先出完答案才开始流。总 token 大约翻倍——推理内容的 token 你也照付，虽然不给用户看。聊天 UI 里那段思考停顿"系统是不是挂了"的体验很糟，我要么加"思考中…“spinner，要么把推理内容流到一个可折叠侧栏。

怎么决策： 这个任务我自己会想先打草稿再答的，开 thinking。会脱口而出的，不开。Agent 多轮里我每轮都关，只在最后综合那一轮开。

长上下文：缓存命中率与截断策略

Qwen-Plus 上下文窗口 128k。Qwen-Max 默认 32k，有 1M 长上下文变体。窗口大不代表你就该把它填满。

第一篇提到的隐式 prompt 缓存有一个关键性质：按精确前缀匹配。 如果 system prompt 每个用户都一样、只有 user 消息变，system 部分就被缓存。如果你把动态数据放到 prompt 中间（比如 “今天是 {date}"），每个变体都打不中缓存。修法是把所有动态内容放到 messages 数组的末尾——动态日期 / user ID / 时间戳放 user 消息里，别放 system。

我每个 endpoint 都打缓存命中率：

1
2
3
4

total = sum(usage.prompt_tokens for usage in window)
cached = sum(getattr(usage, "cached_tokens", 0) for usage in window)
hit_rate = cached / total
print(f"近 {len(window)} 次缓存命中率: {hit_rate:.1%}")

system prompt 稳定的良构 RAG endpoint 命中率 70-80%。把请求级 metadata 拼到 system prompt 里的 endpoint 命中率 0%。账单上输入 token 这一项差 2 倍。

超出窗口要截断时，安全模式是"保留 system + 最近一对 user/assistant，中间滑动窗口”。我会保留首条 system 和最后 6 条原文，中间内容跨阈值就用便宜的 qwen-turbo 总结掉。摘要作为合成 system 消息塞回 messages 数组（"role": "system", "content": "本对话此前: ..."）。聊天类工作负载质量损失小，代码上下文类损失大——文件内容不能 lossy 压缩，那种就别截断，直接换长上下文模型。
