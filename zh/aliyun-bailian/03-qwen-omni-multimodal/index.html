阿里云百炼实战（三）：Qwen-Omni 多模态——视频、音频、图像理解

Qwen-Omni 生产实践：四种输入、文档没强调的流式必填，加上一个真实可跑的视频理解示例和合理的像素预算。

把我从最多产品坑里拽出来的，就是 Qwen-Omni。“能告诉我这条 2 分钟广告片在讲什么吗"以前是个三周项目——抽帧、逐帧 caption、再 stitch。现在 Qwen-Omni 一次 HTTP 请求就搞定。可惜文档对踩坑警告稀疏，有一条（流式必填）让不止一个团队损失了半天时间。咱们别成为下一个。

Qwen-Omni 接受什么

按 Qwen API reference 的多模态部分，单条 user 消息的 content 数组里可以混合 text、image、audio、video 几种类型。这个能力的核心不是"支持图像”，而是"任意组合"：

每种类型的结构（来自 API reference）：

类型字段备注
texttext: "..."纯字符串。
image_urlimage_url: {url}URL 或 base64 data URI。min_pixels / max_pixels 控缩放。
input_audiodata, formatmp3、wav 等。URL 或本地 base64。
video_urlvideo_url: {url}URL 或 data URI。或者改用 video 数组传一组帧图。

from openai import OpenAI
import os

client = OpenAI(
    api_key=os.environ["DASHSCOPE_API_KEY"],
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",

stream = client.chat.completions.create(
    model="qwen3-omni-flash",
    messages=[{
        "role": "user",
        "content": [
            {"type": "text", "text": "用两句话描述这条视频在讲什么。"},
            {"type": "video_url",
             "video_url": {"url": "https://your-bucket.oss-cn-shanghai.aliyuncs.com/clips/promo.mp4"}},
        ],
    }],
    stream=True,           # <- 必须
)

for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)

流式不是可选——这是个陷阱

文档把流式当成功能特性写，但对 Qwen-Omni 是必填。设 stream=False 直接 400。

理由想想就明白：模型在处理几 MB 的视频，要产出长响应。协议假设是增量送达。攒成一坨发，客户端就要阻塞几十秒，全程毫无进度信号。

下游想拿完整字符串，自己 buffer 一下就好：

def call_omni_buffered(messages):
    stream = client.chat.completions.create(
        model="qwen3-omni-flash", messages=messages, stream=True,
    )
    return "".join(c.choices[0].delta.content or "" for c in stream)

写一个函数，以后到处用。

像素和帧率预算——花钱的地方

文档对成本旋钮讲得含糊：图像有 min_pixels / max_pixels，视频有 fps / resize。Qwen-Omni 默认按原始分辨率和默认帧率处理。一段 2 分钟 1080p 等价的 token 很多，账单也跟着膨胀。

我自己生产里的做法：

  • 理解任务的图像 — 设 max_pixels: 1280*720。“图里有什么"几乎不掉质量，但能省下不少钱。再加 min_pixels: 640*480 防止小图被放大。
  • 描述任务的视频 — 上传前先 resize 到 720p，静态内容（人讲话）downsample 到 4 fps，动态内容（运动、快剪）8 fps。8 fps 以上多半是在为冗余帧付钱。
  • 长视频 — 切。模型有上下文上限。3 分钟以上的切成 90 秒一段，每段 summarize，再用 qwen-plus 做 summary-of-summary。和长文档 RAG 一个套路。

把本地视频发出去

两条路，文档都讲了。

路径 1（推荐）：上传 OSS，发签名 URL 过去。

1
2
3
4
5

import oss2, os
auth = oss2.Auth(os.environ["OSS_AK"], os.environ["OSS_SK"])
bucket = oss2.Bucket(auth, "https://oss-cn-shanghai.aliyuncs.com", "your-bucket")
bucket.put_object_from_file("clips/promo.mp4", "/tmp/promo.mp4")
signed = bucket.sign_url("GET", "clips/promo.mp4", 600)  # 10 分钟过期

把 signed 当 url 字段传过去。30 秒以上的视频都走这条——base64 会把 payload 撑大 33%。

路径 2：base64 内联。 短片、想省掉 OSS 一来一回时用。

import base64
with open("/tmp/short.mp4", "rb") as f:
    b64 = base64.b64encode(f.read()).decode()
content = [{
    "type": "video_url",
    "video_url": {"url": f"data:video/mp4;base64,{b64}"},
}]

真实经验： Qwen-Omni 报 400 时，先确认 URL 在公网能直接拿到。模型服务进不了你的 VPC。签名 URL 可以；不签名的私有 OSS 不行。

一个真实产品场景

我在 AI 营销里反复用的一个模式：创意团队上传 60 秒产品片；要一段结构化字幕（scene_description、product_features、target_audience、music_style）。一次 Qwen-Omni 调用，开 JSON mode（多模态也支持），720p 输入 4 秒之内出结果。

       "target_audience、music_style。")
stream = client.chat.completions.create(
    model="qwen3.5-omni-plus",
    messages=[
        {"role": "system", "content": sys},
        {"role": "user", "content": [
            {"type": "video_url", "video_url": {"url": signed_url}},
        ]},
    ],
    response_format={"type": "json_object"},
    stream=True,
)
text = "".join(c.choices[0].delta.content or "" for c in stream)
import json; result = json.loads(text)
