阿里云百炼实战（三）：Qwen-Omni 多模态——视频、音频、图像理解

把我从最多产品坑里拽出来的，就是 Qwen-Omni。“能告诉我这条 2 分钟广告片在讲什么吗"以前是个三周项目——抽帧、逐帧 caption、再 stitch。现在 Qwen-Omni 一次 HTTP 请求就搞定。可惜文档对踩坑警告稀疏，有一条（流式必填）让不止一个团队损失了半天时间。咱们别成为下一个。

Qwen-Omni 接受什么

按 Qwen API reference 的多模态部分，单条 user 消息的 content 数组里可以混合 text、image、audio、video 几种类型。这个能力的核心不是"支持图像”，而是"任意组合"：

每种类型的结构（来自 API reference）：

类型字段备注
texttext: "..."纯字符串。
image_urlimage_url: {url}URL 或 base64 data URI。min_pixels / max_pixels 控缩放。
input_audiodata, formatmp3、wav 等。URL 或本地 base64。
video_urlvideo_url: {url}URL 或 data URI。或者改用 video 数组传一组帧图。

流式不是可选——这是个陷阱

文档把流式当成功能特性写，但对 Qwen-Omni 是必填。设 stream=False 直接 400。

理由想想就明白：模型在处理几 MB 的视频，要产出长响应。协议假设是增量送达。攒成一坨发，客户端就要阻塞几十秒，全程毫无进度信号。

像素和帧率预算——花钱的地方

文档对成本旋钮讲得含糊：图像有 min_pixels / max_pixels，视频有 fps / resize。Qwen-Omni 默认按原始分辨率和默认帧率处理。一段 2 分钟 1080p 等价的 token 很多，账单也跟着膨胀。

我自己生产里的做法：

  • 理解任务的图像 — 设 max_pixels: 1280*720。“图里有什么"几乎不掉质量，但能省下不少钱。再加 min_pixels: 640*480 防止小图被放大。
  • 描述任务的视频 — 上传前先 resize 到 720p，静态内容（人讲话）downsample 到 4 fps，动态内容（运动、快剪）8 fps。8 fps 以上多半是在为冗余帧付钱。
  • 长视频 — 切。模型有上下文上限。3 分钟以上的切成 90 秒一段，每段 summarize，再用 qwen-plus 做 summary-of-summary。和长文档 RAG 一个套路。

音频输入编码：采样率、格式、最大时长

Qwen-Omni 的音频路径看着简单——传一个 input_audio 部分，带 data 和 format——但编码要求严格，错了不会给你友好提示。报错就一句"unsupported audio format”，没下文。

生产里真正稳的配置：

  • 采样率：16 kHz 是最佳点。模型接受 8 / 16 / 22.05 / 24 / 44.1 / 48 kHz，但非 16 kHz 都会服务端 resample，多花延迟。VOIP 录音（8 kHz 原生）我会客户端先 librosa.resample 到 16 kHz——质量一样，延迟更可控。
  • 格式：wav（PCM 16-bit）、mp3、m4a、flac、ogg。短话术（< 30 秒）用 WAV 不在乎大小，长内容用 MP3。MP4 容器里的 AAC 偶尔能用，看 codec profile——直接转 MP3 比 debug 它简单。
  • 声道：建议单声道。立体声会被 downmix；如果两声道是不同说话人（采访类），downmix 之后说话人就分不开了。自己 downmix 留你想要那一路，或者拆两次请求。
  • 位深：16-bit PCM 是安全默认。24-bit 和 32-bit float 能用但偶尔被拒——同一个 WAV 文件，同一周里我见过 16-bit 通过、32-bit 被拒。
  • 最大时长：qwen3-omni-flash 单次 3 分钟。再长就要切。报错是 Audio duration exceeds limit，message 里写了具体上限。
  • 最大文件大小：10 MB。16 kHz / 16-bit mono PCM 折合大约 5 分钟 WAV——但 64 kbps MP3 在同样信封里能塞 20+ 分钟。长内容必须用 MP3。

import subprocess, os

def normalize_audio(src: str, dst: str) -> None:
    """转成 Qwen-Omni 处理最稳的编码。"""
    subprocess.run([
        "ffmpeg", "-y", "-i", src,
        "-ac", "1",            # mono
        "-ar", "16000",        # 16 kHz
        "-c:a", "libmp3lame",  # MP3 codec
        "-b:a", "64k",         # 64 kbps，对语音绰绰有余
        dst,
    ], check=True, capture_output=True)

每次上传前都跑一遍。200ms 转码成本相对 API 延迟可以忽略，整个"格式不支持"类的 400 都消失了。

视频帧采样：人头说话 1 fps，动作内容 8 fps

Qwen-Omni 内部处理视频的路径是按一个模型内部速率抽帧、每帧编码成视觉 token 块。Token 成本随帧数线性涨，所以帧率是视频成本最大的旋钮。

文档说参数是视频 URL 部分的 fps，但没告诉你不同内容该怎么挑值。生产里跑过大约 2000 次视频调用之后，我的规则：

  • 人头说话内容（采访、产品演示、讲座）：1 fps 够用。有意义的信号在音频 + 稀疏视觉线索（翻页、手势）上。更高 fps 是在为长得一样的帧浪费 token。我在同一段 2 分钟采访片上测了 1 / 4 / 8 / 16 fps，描述质量在 1 fps 就拉平了。有时降到 0.5 fps 也能拿到合理摘要。
  • 动作内容（体育、带剪辑的广告、运动）：8 fps。低于这个会漏掉运动定义的事件。高于这个模型在处理冗余中间帧。三家广告科技客户上验证过，这个值就是甜点。
  • 混合内容（UGC、来源未知的用户上传）：默认 4 fps。两端都不算最优，但都不会灾难性失败。
  • 幻灯片 / 静态加叠加内容：0.5 fps。模型要看清每张幻灯片，不是去处理过场。

import cv2
def extract_frames(path: str, fps: float) -> list[str]:
    cap = cv2.VideoCapture(path)
    src_fps = cap.get(cv2.CAP_PROP_FPS)
    step = int(src_fps / fps)
    frames = []
    i = 0
    while True:
        ok, frame = cap.read()
        if not ok: break
        if i % step == 0:
            _, buf = cv2.imencode(".jpg", frame, [cv2.IMWRITE_JPEG_QUALITY, 85])
            frames.append(f"data:image/jpeg;base64,{base64.b64encode(buf).decode()}")
        i += 1
    return frames

video_part = {"type": "video", "video": extract_frames("/tmp/clip.mp4", fps=4)}

预抽帧给你精确控制模型看到哪些帧——自动采样跳过关键时刻这种事我遇到过不止一次，这就是兜底。

视频 token 成本算账

Omni 的成本意外不在单价，在每秒视频对应的 token 数。从原理算：

1280×720 一帧，过 Qwen-Omni 内部视觉编码器后，大约 256 视觉 token / 帧。（具体数因模型而异；qwen3.5-omni-plus 更密，qwen3-omni-flash 更轻，但 256 是规划用的合理数。）

那么 30 秒片子 4 fps 就是 30 * 4 * 256 = 30,720 视觉 token。qwen3-omni-flash 视觉 12 元/百万 token，光输入就是 30720 / 1e6 * 12 = 0.37 元，描述输出还没算。

放大看：

内容时长fps视觉 token输入成本（元）
人头说话60s115,3600.18
人头说话60s461,4400.74
广告位15s830,7200.37
长演示180s4184,3202.21
讲座纯幻灯600s0.576,8000.92

两个结论：

  • fps 选择能让成本差 4-8 倍。慎选。
  • 10 分钟讲座 0.5 fps 比 1 分钟人头 4 fps 还便宜。长内容稀疏采样很便宜。

对比一下，纯音频按大约 50 token/秒 算（内部转写成文本 token 等价）。10 分钟音频是 600 * 50 = 30,000 音频 token——比 1 分钟密集视频还便宜。人头说话类内容意义都在音频上，只发音频，理解质量等价，成本约 5 倍便宜。

延迟剖面：视频理解 TTFT、批处理模式

Qwen-Omni 端到端延迟拆成三段比较有用：

  1. 上传 + URL 签名：200ms-2s，看文件大小和 OSS endpoint。调 prompt 阶段同一个签名 URL 复用。
  2. 视觉编码（服务端）：30 秒 720p 大约 1-4 秒。这是用户感知"什么都没发生"的那段——token 还没开始流。大致随帧数线性。
  3. 流式生成：视觉编码后 TTFT 一般 1.5-3 秒，之后 30-60 token/秒。

30 秒视频 → 60 字描述，端到端生产里大约 6-8 秒墙钟。比人快，比纯文本 LLM 慢。UX 要求亚秒级就别用 Omni——只能在入库时预处理、缓存描述。

批量场景（一晚处理 1000 条视频），Omni 没有 OpenAI 那种 native batch endpoint。我用的模式：

  • 异步队列（Celery / SQS / 你自己的）配 10 个 worker。
  • 每个 worker 拿一个 OpenAI 客户端，串行调 Omni，单任务 10s 软超时（长尾到 30s）。
  • 每 key 并发上限 10（匹配工作空间默认配额）。
  • 429 退避重试，最长 3 分钟。
  • 生产吞吐：每工作空间约 600 条/小时，瓶颈是 60 RPM 的 qwen3-omni-flash 配额。

要更高吞吐就提配额（第一篇有流程），再跨多工作空间扇出。百炼没有 native 并行 batch 原语，这块得自己造。
