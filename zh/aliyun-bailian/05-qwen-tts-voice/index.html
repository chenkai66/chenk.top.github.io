阿里云百炼实战（五）：Qwen-TTS 多语言语音合成

Qwen-TTS-Flash 实战：只走原生 API、40+ 音色目录（含粤语 / 川语）、流式合成，以及文档一笔带过的 SSML 行为细节。

Read in English →

我做的每个中文产品最后都会调 Qwen-TTS-Flash。原因不是便宜——便宜的 TTS 多得是。是它是唯一一个能在同一个 SDK 里同时讲清楚中国大陆方言（粤语、川语、吴语）和英文，而且音色还不像 2019 年海关广播那种 TTS。在某营销视频配音流水线里跑了大约半年，下面是我希望第一天就有人告诉我的事。

音色目录

按模型卡，Qwen-TTS-Flash 有 40+ 音色。我用得最多的几个：

普通话产品旁白我默认用 Cherry（温暖，30 多女声）做营销内容，用 Ethan（沉稳，40 多男声）做教程。粤语广告位 Sunny 是稳妥选择。音色名是稳定的，但新音色会陆续加——上线前到模型卡确认你要的还在。

Qwen-TTS 不走 OpenAI 兼容层。要用 DashScope 原生 SDK：

import os, dashscope, requests
from dashscope.audio.qwen_tts import SpeechSynthesizer

dashscope.api_key = os.environ["DASHSCOPE_API_KEY"]

resp = SpeechSynthesizer.call(
    model="qwen3-tts-flash",
    text="欢迎来到杭州，全城最安静的咖啡馆就在西湖边上。",
    voice="Cherry",
    format="mp3",
)

audio_url = resp.output.audio["url"]
with open("/tmp/out.mp3", "wb") as f:
    f.write(requests.get(audio_url, timeout=30).content)

两点要强调：

  • 输出默认是 URL，不是字节。和万相一样，24 小时内一定要下载（我的做法是立刻下载、立刻 re-upload 到自己的 OSS）。
  • format 默认 mp3，也支持 WAV。下游要做拼接的话 WAV 更方便，每段不带 MP3 头开销。

from dashscope.audio.qwen_tts import SpeechSynthesizer

with open("/tmp/streamed.mp3", "wb") as f:
    for resp in SpeechSynthesizer.call(
        model="qwen3-tts-flash",
        text="这段语音是从模型那边一段一段流式返回的。",
        voice="Cherry",
        stream=True,
    ):
        if resp.output and resp.output.audio:
            f.write(resp.output.audio["data"])

上海区流式首字节通常 < 400ms，用户感知是即时的。非流式合成 30 秒话术大约 4-6 秒墙钟时间——批量旁白可以用，对话场景就偏慢了。

多语言和方言细节

Qwen-TTS 会从文本里检测语种，但混排时建议显式设 language。我的生产规则：

  • 纯普通话 → language="zh"（默认）。
  • 纯英文 → language="en"。Eric 这类音色出彩。
  • 粤语繁体 → language="zh-yue"，音色用 Sunny 或 Lily。
  • CJK + 英文混排（科技旁白常见）→ 不设语种，让模型自然处理 code-switch。

经验： 方言上线前务必找母语者 A/B 一遍。Qwen-TTS 粤语好但音调不完美——粤语单字音调错了，意思会变。

SSML——什么管用、什么不管用

文档列了 SSML 支持，但对哪些标签真正生效没说清。经验：

  • <break time="500ms"/> — 管用。营销文案句间停顿就用它。
  • <emphasis level="strong"> — 管用。
  • <prosody rate="slow"> — 管用。slow / medium / fast 或者百分比都行。
  • <prosody pitch="..."> — 相对调整管用（+10%）。
  • <say-as interpret-as="digits"> — 电话、code、日期管用。
  • <phoneme> — 部分管用。中文场景下，带声调拼音比 IPA 稳。
  • <voice> — 不管用。一句话中间不能换音色。要换就分次合成再拼接。

长旁白拼接

营销脚本通常很长。我做 60 秒旁白的模式是这样：

def synthesize_long(script: str, voice: str = "Cherry") -> str:
    sentences = split_sentences(script)  # 自家分句器，简单 regex 也行
    parts = []
    for s in sentences:
        resp = SpeechSynthesizer.call(model="qwen3-tts-flash", text=s,
                                       voice=voice, format="wav")
        parts.append(download(resp.output.audio["url"]))
    return concat_wavs(parts, output="/tmp/full.wav")  # ffmpeg concat

为什么要分句？两个原因：(1) 单句调用延迟低很多，并发合成更快；(2) 单句出问题只重做那一句，不用整段重录。我们广告位就是这么干的——句间用 <break> 处理停顿，并发合成下来，60 秒片子大约 4 秒能出。

成本

按输出音频秒数计费。流式和非流式一个价。60 秒广告位几块钱——比配音演员小时费便宜得多，也快得让营销团队下午能试几十个版本。

音色克隆、预置音色、SSML 控制：选对旋钮

百炼给了三层音色控制，选哪一层比一般人想得更重要。成本、复杂度、控制力的顺序和"高级感"的顺序刚好相反：

预置音色。 目录里那 40+ 个。除非有特定理由，否则就用它。覆盖普通话男女声各年龄段和语域，外加粤语、川语、吴语、英文（美/英），还有几个旁白用的"人物"音色。每秒成本是标准价；延迟最低；质量录音棚级。我生产里 90% 流量都走预置音色。

预置音色 + SSML 控制。 同音色，但你能微调韵律、停顿、强调、读音。成本和预置音色一样。什么时候用：

  • 长篇旁白需要刻意节奏（技术教程、有声书）。
  • 模型默认漏掉的某句话需要强调。
  • 模型读错的专有名词或术语（CJK + 英文混排尤其常见：“DSL” 经常被读成 “D-S-L” 而不是 “deh-es-el”）。

我营销流水线里一个真实例子：

text = """
<speak>
  欢迎使用 <say-as interpret-as="characters">DSL</say-as> 翻译系统。
  <break time="300ms"/>
  本次更新支持<emphasis level="strong">多语种</emphasis>批量处理。
  <prosody rate="95%">详细文档请参见官网。</prosody>
</speak>
"""
resp = SpeechSynthesizer.call(model="qwen3-tts-flash", text=text, voice="Cherry", format="wav")

<say-as> 让 “DSL” 一个字母一个字母读（缩写正确读法）。<break> 在听者需要消化的地方加刻意停顿。<prosody rate="95%"> 把 URL 推荐稍微放慢，方便听者记下来。这些用纯文本输入都做不到。

音色克隆（音色迁移）。 百炼有独立的克隆 API：上传 10-30 秒参考音频，拿到一个 voice id 给后续调用用。这是真能用，但要带两个警告：

  • 质量看素材。 单说话人、清晰、录音棚的参考音频克隆出来很棒。手机录音、有底噪、多说话人的参考音频，克隆出来 uncanny。生产用前必须人工筛一遍。
  • 法律 / 伦理面。 未经同意克隆真人声音在很多司法辖区越来越违法，百炼 ToS 也明确禁止。只克隆你拥有或拿到书面授权的声音。平台对疑似公众人物冒充会撤销 voice id。

我营销流水线里克隆过创始人声音（征得同意），用在某条产品线，他想做几十个版本的 30 秒 pitch。一次克隆，20 条样本验证质量，然后这条线上一直用那个 voice id。克隆完之后成本和预置音色一样。

决策树：

  • 预置目录里没有需要的声音？→ 克隆，要授权。
  • 预置音色行，但发音 / 节奏不对？→ SSML。
  • 预置音色直接行？→ 直接用，别想多。

延迟预算：流式分块、口型同步窗口

交互式语音产品（语音机器人、实时数字人）边际上延迟比质量更重要。流式 TTS 延迟拆解：

  1. 请求建立（TLS 握手、首请求包）：境内 30-80ms，跨区域 200-400ms。
  2. TTFT（首块音频到达）：上海区 fresh request 一般 200-400ms。
  3. 每块节奏：每 100-200ms 到一块，每块约 200-400ms 音频。
  4. 尾巴：最后一块在文本对应说话时长 + ~100ms 后到。

“用户输文本 → 音频开播” 这种 UX，端到端首音感知延迟落在 250-500ms。这是"感觉即时"的临界——< 500ms 读作即时，> 1s 读作卡。

和数字人对口型 既要音频块又要每音素时间戳。Qwen-TTS 基础 API 不返回每音素时间戳。两个变通：

  • 输出跑强制对齐器。 whisper-timestamped 或 mfa（Montreal Forced Aligner）拿合成音频和源文本，给出每词或每音素时间戳。给整体 pipeline 加 200-500ms。我非实时数字人就这么干。
  • 从文本长度估时间戳。 实时场景跑不起对齐：估算——中文每字默认语速约 150ms，英文每词约 250ms。误差累积，但前 5-10 秒一般够准，口型粗看对得上。

import time
def stream_with_phoneme_sync(text: str, on_chunk, on_phoneme_estimate):
    start = time.time()
    elapsed_audio_ms = 0
    for resp in SpeechSynthesizer.call(model="qwen3-tts-flash", text=text,
                                        voice="Cherry", stream=True):
        if not resp.output or not resp.output.audio: continue
        chunk = resp.output.audio["data"]
        on_chunk(chunk)
        # 粗估：16kHz mono PCM = 32 KB / 秒
        chunk_duration_ms = len(chunk) / 32 * 1000
        elapsed_audio_ms += chunk_duration_ms
        on_phoneme_estimate(elapsed_audio_ms)

营销数字人够用了。高保真配音不够——那种就渲染成文件再跑强制对齐。

音频后处理：响度归一化、静音裁剪

原始 TTS 输出基本不是上线 ready 状态。任何送听者的流水线里这两步是必做：

响度归一化。 不同音色、不同文本、甚至同音色不同次出，听感响度都不一样。不归一化，听者上一秒被一句吼到，下一秒一句没听清。标准是 EBU R128，长内容 / 播客目标 -16 LUFS，流媒体平台目标 -14 LUFS。

1

ffmpeg -y -i in.wav -af loudnorm=I=-16:TP=-1.5:LRA=11 out.wav

一行 ffmpeg 把每条 TTS 输出对到同一感知响度。每条都跑，无例外。CPU 成本微秒级。

静音裁剪。 Qwen-TTS 经常前导 100-300ms 静音，尾部类似。拼接旁白时累加起来——20 句脚本最后多 4-6 秒死气。裁剪用：

1

ffmpeg -y -i in.wav -af "silenceremove=start_periods=1:start_duration=0.1:start_threshold=-50dB:detection=peak,areverse,silenceremove=start_periods=1:start_duration=0.1:start_threshold=-50dB:detection=peak,areverse" out.wav

拼接前裁，再用显式静音 padding 加回受控的句间停顿。出来的旁白听感是有意为之，不是节奏尴尬。

def post_process(parts: list[str], output: str):
    """归一化、裁剪、拼接。"""
    cleaned = []
    for i, p in enumerate(parts):
        out = f"/tmp/clean_{i}.wav"
        subprocess.run(["ffmpeg", "-y", "-i", p,
                        "-af", "loudnorm=I=-16:TP=-1.5:LRA=11,"
                               "silenceremove=start_periods=1:start_duration=0.1:start_threshold=-50dB:detection=peak,"
                               "areverse,silenceremove=start_periods=1:start_duration=0.1:start_threshold=-50dB:detection=peak,areverse",
                        out], check=True, capture_output=True)
        cleaned.append(out)
    # 句间加 250ms 受控静音
    with open("/tmp/concat.txt", "w") as f:
        for i, c in enumerate(cleaned):
            f.write(f"file '{c}'\n")
            if i < len(cleaned) - 1:
                f.write("file '/tmp/silence_250ms.wav'\n")
    subprocess.run(["ffmpeg", "-y", "-f", "concat", "-safe", "0",
                    "-i", "/tmp/concat.txt", "-c", "copy", output], check=True)

句间 250ms 静音 padding 是让背靠背 TTS 听上去"一镜到底"而不是"拼贴怪"的关键。

每分钟生成音频成本

官方按秒定价是正确单位，下面是按分钟的版本，含合理 overhead：

模式每分钟基础加 ffmpeg 后处理（可忽略）加克隆增量
预置音色，非流式~0.6 元/分钟+0不适用
预置音色，流式~0.6 元/分钟+0不适用
克隆音色~0.6 元/分钟+0一次性克隆 ~5 元
长 SSML 标记~0.6 元/分钟+0不适用

60 分钟有声书按 0.6 元/分钟算 36 元。30 秒广告位 0.3 元。对比配音演员的项目费（中端中文配音演员：单项目 2000-5000 元），API 每条产出便宜大约 100-1000 倍——代价是每条只占几分之一秒 CPU 时间，零档期摩擦。“该请配音演员还是用 Qwen-TTS” 的盈亏平衡点就一种：这条是品牌 hero spot，需要观众已经熟悉的某个具体人声。其它情况都倒向 API。

隐藏成本是后处理工程时间：响度 / 裁剪 / 拼接流水线 1 天搭好，之后摊到未来几年每条片子上。跳过后处理，模型再好你的音频也听着像业余作品。
