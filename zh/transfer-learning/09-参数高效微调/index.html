从低秩适配的数学原理出发，系统讲解 LoRA、Adapter、Prefix-Tuning、Prompt-Tuning、BitFit、QLoRA 等参数高效微调方法，附 LoRA 从零实现与方法选型指南。

单张 GPU 怎么微调 1750 亿参数的模型？只更新 0.1% 的参数就够了。参数高效微调（Parameter-Efficient Fine-Tuning，PEFT）让这成为可能——而且在大多数基准测试中，效果几乎和全量微调持平。这篇文章会从数学原理出发，推导 LoRA、Adapter、Prefix-Tuning、Prompt-Tuning、BitFit 和 QLoRA 的设计逻辑，并用一张图帮你选对方法。

全量微调的难题

全量微调会更新所有参数 $\boldsymbol{\theta}$：

$$\boldsymbol{\theta}^* = \arg\min_{\boldsymbol{\theta}} \mathcal{L}(\boldsymbol{\theta})$$

代价很高：

  • 显存占用巨大：GPT-3（175B 参数）FP32 权重约 700 GB，再加上梯度和优化器状态，显存需求轻松达到数 TB。
  • 存储成本高昂：每个任务需要保存一份完整的 checkpoint。如果有 100 个任务，就得存 100 份 700 GB 的模型，总共 70 TB。
  • 灾难性遗忘：所有参数都在更新，预训练学到的知识很容易被破坏。
  • 服务复杂：上线时无法用一个基座模型加路由的方式同时支持多个任务。
$$\boldsymbol{\theta}^* = \boldsymbol{\theta}_0 + \Delta\boldsymbol{\theta}, \qquad |\Delta\boldsymbol{\theta}| \ll |\boldsymbol{\theta}_0|.$$

冻结预训练权重 $\boldsymbol{\theta}_0$，只训练并存储一个小得多的任务特定增量 $\Delta\boldsymbol{\theta}$。预训练权重成为共享主干，而适配部分变成轻量级的增量，可以按需存储、版本管理和动态路由。

LoRA：低秩适配

核心思想一句话概括

$$\mathbf{W}' = \mathbf{W}_0 + \frac{\alpha}{r}\, \mathbf{B}\mathbf{A}, \qquad \mathbf{A} \in \mathbb{R}^{r \times d_{\text{in}}},\; \mathbf{B} \in \mathbb{R}^{d_{\text{out}} \times r},\; r \ll \min(d_{\text{in}}, d_{\text{out}}).$$

冻结的 $\mathbf{W}_0$ 包含 $d_{\text{in}} d_{\text{out}}$ 个参数，而可训练部分只有 $r(d_{\text{in}} + d_{\text{out}})$ 个。比如当 $d=4096, r=8$ 时，新增参数仅占原矩阵的 0.39%。

为什么低秩假设成立？

有两个实验结果支持这个假设：

  1. 本征维度（Aghajanyan 等，2020）：微调过程实际上只在权重空间的一个低维子空间中进行。几百个参数就足以在许多任务上达到全量微调的效果。
  2. 预训练矩阵的奇异值分布：注意力投影矩阵的奇异值衰减很快，少数方向承载了大部分信息。而针对新任务的更新方向更加集中。

直观来说，预训练模型已经掌握了大部分知识，下游任务只需要稍微调整视角，不需要太多自由度。

初始化、缩放与前向计算

LoRA 的实际应用中有几个关键细节：

  • 初始化：$\mathbf{A} \sim \mathcal{N}(0, \sigma^2)$（Kaiming 风格），$\mathbf{B} = \mathbf{0}$。这样保证第 0 步时 $\Delta\mathbf{W} = \mathbf{0}$，模型从预训练行为开始，不会一开始就偏离。
  • 前向计算：直接计算 $\mathbf{h} = \mathbf{W}_0 \mathbf{x} + \tfrac{\alpha}{r}\,\mathbf{B}(\mathbf{A}\mathbf{x})$。不要显式构造 $\mathbf{B}\mathbf{A}$——它是一个 $d_{\text{out}}\times d_{\text{in}}$ 的稠密矩阵，正是我们要避免的。
  • 缩放因子 $\alpha/r$：将学习率和秩解耦。即使秩翻倍，等效学习率也不会变化，因此 $\alpha$ 可以在不同秩之间复用。
  • 权重合并：推理时执行 $\mathbf{W}_{\text{merged}} = \mathbf{W}_0 + (\alpha/r)\,\mathbf{B}\mathbf{A}$，将增量合并到原权重中。推理延迟零额外开销——最终交付的仍然是单个权重矩阵。
  • 适用位置：实测表明，将 LoRA 应用于 Attention 的 query 和 value 投影效果最佳；单独应用于其中一个不够好，而扩展到所有线性层则性价比下降。

参数量差距有多大？

以 GPT-3 175B 为基座，全量微调与 LoRA $r=8$ 的可训练参数差距大约是 五个数量级。但稍后我们会看到，在 GLUE 上得分几乎一样。

Adapter：Block 中的瓶颈模块

$$\text{Adapter}(\mathbf{h}) = \mathbf{h} + \mathbf{W}_{\text{up}}\, \sigma(\mathbf{W}_{\text{down}}\, \mathbf{h}), \qquad \mathbf{W}_{\text{down}} \in \mathbb{R}^{m \times d},\; \mathbf{W}_{\text{up}} \in \mathbb{R}^{d \times m},\; m \ll d.$$

“先降维再升维"的操作和 LoRA 类似，但 Adapter 是作为额外的一层加入，而不是对现有层进行调整。把 $\mathbf{W}_{\text{up}}$ 初始化为接近零，可以让 Adapter 初始行为接近恒等映射，避免破坏预训练模型的表现。

Adapter 和 LoRA 的对比：

AdapterLoRA
修改内容新增模块重新参数化已有权重
推理延迟增加（多一层串行计算）无（合并到 $W_0$ 中）
单任务存储每层约 $2md$ 参数每层约 $r(d_\text{in}+d_\text{out})$ 参数
适用场景编码器（BERT 系列）生成式模型（GPT 系列）

后续有两个改进版本值得关注。Pfeiffer Adapter 每个 Block 只保留一个 Adapter，放在 FFN 后面，参数量减半但性能几乎不变。并行 Adapter（He 等，2021）将瓶颈计算与 FFN 并行处理，避免了 GPU 上的串行依赖，大幅缩小了与 LoRA 的推理延迟差距。

Prefix-Tuning：前置虚拟 Token

$$\bigl[\mathbf{P}_1, \ldots, \mathbf{P}_m,\; \mathbf{x}_1, \ldots, \mathbf{x}_n\bigr] \to \text{Transformer}.$$

训练时只更新这些前缀矩阵，参数总量是 $m \times d \times L \times 2$（key + value，$L$ 层）。实际操作中有两点需要注意：

  • 直接优化前缀向量容易不稳定。原论文通过一个小 MLP 来训练前缀，推理时丢掉 MLP，只保留生成的前缀向量。
  • 前缀的信息容量有限，比如 20 个 Token 的前缀在复杂任务上会比 LoRA 差 0.5-1 分。

从概念上看，这些前缀从第一层开始就引导注意力机制，影响模型对每个真实 Token 的关注点。它和软提示类似，但作用于注意力机制内部，而不是嵌入表。

Prompt-Tuning 与 P-Tuning v2

Prompt-Tuning（Lester 等，2021）是一种极简方法：只在输入层添加 $m$ 个可学习的 prompt 向量。参数量仅为 $m \times d$，通常只有几千个。

但它的局限性也很明显：只适合超大模型。Lester 等人的实验表明，随着模型规模增大，Prompt-Tuning 和全量微调之间的差距会逐渐缩小。1B 参数以下时效果较差，但到了 10B 参数以上，差距几乎可以忽略。原因很简单：百亿参数的大模型有足够的能力“解读”任意连续的 prompt，而小模型则需要更直接、更局部的调整。

P-Tuning v2（Liu 等，2022）结合了 Prompt-Tuning 和 Prefix-Tuning 的优点：在每一层都加入可学习的 prompt，而不仅限于输入层。这种方法在各种任务和模型大小上都能媲美全量微调，同时可训练参数依然保持在约 0.1%。

BitFit：只调 Bias

最简单粗暴的方法来了：只微调偏置项（Zaken 等，2021）。BERT 里偏置项只占总参数的 0.08%，但在小数据 GLUE 任务上，效果能和全量微调掰手腕。为什么有效？偏置调整了每个非线性层的输入分布，这就足够重新引导激活路径，还不用动线性映射本身。当基线方法用，几乎零成本；认真比拼的话，性能略逊 LoRA 一点。

QLoRA：量化 + LoRA

QLoRA（Dettmers 等，2023）是一项工程突破，它让单张 48 GB 显卡微调 65B 参数模型成为现实。它的核心是三个关键技术：

  1. NF4（4-bit NormalFloat）量化：对冻结的基座权重进行量化。假设预训练权重近似服从零均值高斯分布，那么 NF4 码本在信息论上是最优的——而这个假设基本成立。
  2. 双重量化：对每个 block 的量化常数再次量化，每个参数再节省约 0.4 bit。
  3. 分页优化器：通过统一内存将 Adam 的状态在 CPU 和 GPU 之间分页调度，避免长序列处理时显存不足。

最关键的是，梯度仍然可以通过量化矩阵传递到上方的 bf16 LoRA 适配器。冻结的基座不需要 FP16 权重，只有少量的 LoRA 参数需要。右图清楚地展示了效果：全量微调 65B 模型需要约 700 GB 显存，而 QLoRA 只需约 50 GB。

QLoRA 在 Vicuna 基准测试中性能仅下降不到 2%，但硬件成本却降低了 1 到 2 个数量级。

import numpy as np
import torch
import torch.nn as nn
import torch.nn.functional as F

class LoRALayer(nn.Module):
    """W' = W_0 + (alpha / r) * B @ A.

    基础权重冻结，只有 A 和 B 参与梯度更新。
    """

    def __init__(self, in_features: int, out_features: int,
                 rank: int = 8, alpha: float = 16.0,
                 dropout: float = 0.0):
        super().__init__()
        self.in_features, self.out_features = in_features, out_features
        self.rank, self.alpha = rank, alpha
        self.scaling = alpha / rank

        # 冻结的预训练权重（稍后加载）
        self.weight = nn.Parameter(
            torch.empty(out_features, in_features), requires_grad=False)
        self.bias = nn.Parameter(
            torch.zeros(out_features), requires_grad=False)

        # 可训练的低秩因子
        self.lora_A = nn.Parameter(torch.empty(rank, in_features))
        self.lora_B = nn.Parameter(torch.zeros(out_features, rank))
        nn.init.kaiming_uniform_(self.lora_A, a=np.sqrt(5))

        self.dropout = nn.Dropout(dropout) if dropout > 0 else nn.Identity()
        self._merged = False

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        base = F.linear(x, self.weight, self.bias)
        if self._merged:
            return base
        # 关键点：绝不显式计算 B @ A
        delta = (self.dropout(x) @ self.lora_A.T) @ self.lora_B.T
        return base + self.scaling * delta

    @torch.no_grad()
    def merge_weights(self) -> None:
        """将 LoRA 更新合并到基础权重中，推理时无额外开销。"""
        if self._merged:
            return
        self.weight.add_(self.scaling * (self.lora_B @ self.lora_A))
        self.lora_A.zero_()
        self.lora_B.zero_()
        self._merged = True

def apply_lora(module: nn.Module, rank: int = 8, alpha: float = 16.0,
               targets: tuple[str, ...] = ("query", "value")) -> None:
    """递归替换指定的 nn.Linear 层为 LoRALayer。"""
    for name, child in module.named_children():
        if isinstance(child, nn.Linear) and any(t in name for t in targets):
            lora = LoRALayer(child.in_features, child.out_features, rank, alpha)
            lora.weight.data.copy_(child.weight.data)
            if child.bias is not None:
                lora.bias.data.copy_(child.bias.data)
            setattr(module, name, lora)
        else:
            apply_lora(child, rank, alpha, targets)

有三个细节值得注意：

  1. 前向传播永远不会显式构造 $\mathbf{B}\mathbf{A}$——先左乘 $\mathbf{A}$，生成宽度为 $r$ 的中间结果，再乘 $\mathbf{B}$。
  2. merge_weights 是一次性原地操作。合并后，该层的行为与普通的 nn.Linear 完全一致，推理延迟不受影响。
  3. apply_lora 遍历模型树，只替换名称包含 "query" 或 "value" 的层——这是 LoRA 论文中的标准建议。

方法对比：一张图看懂 GLUE 上的取舍

用气泡图总结 RoBERTa-base 在 GLUE 上的表现，整个设计空间一目了然：

  • LoRA $r=8$：只训练 0.24% 的参数，效果和全量微调几乎持平，差距不到 0.1 分。
  • Adapter：准确率和 LoRA 差不多，但参数量是它的 4 倍，推理延迟也明显更高。
  • Prefix-Tuning：稍微牺牲一点准确率，换来极低的参数量，适合需要存储几百个任务适配器的场景。
  • Prompt-Tuning：在 RoBERTa-base 这种规模的模型上表现一般，比其他方法差 3-4 分；底座越大越有优势。
  • BitFit：实现简单，效果却出人意料地好，是个值得关注的基线方法。

如何选择 LoRA 的秩 $r$：从 $r=8$ 开始。如果验证集性能落后超过 0.5 分，就提高到 16；显存不足时降到 4。大模型反而更倾向于较小的 $r$——任务适配的本征维度不会随模型增大而线性增长。

DoRA、OFT 与后 LoRA 时代的浪潮

LoRA 在 2022 到 2024 年间独领风骚，原因很简单：方法易用，效果稳定。2024 到 2025 年的论文从三个正交方向对这一思路进行了优化，其中几篇已经具备生产价值。

DoRA：分解权重为幅度和方向，再对方向应用 LoRA

$$W' = (m + \Delta m) \cdot \frac{W + \Delta V}{\|W + \Delta V\|}.$$

在 Llama-2-7B 推理基准测试中，DoRA 在相同参数量下缩小了 LoRA 和全量微调之间约一半的差距。代价是每个适配矩阵增加一个 Linear(out_dim, 1) 操作，并在前向传播中加入一次归一化——推理时可以将 $\Delta V$ 融合回 $W$，因此线上开销几乎可以忽略。

OFT：保持几何特性的正交微调

Orthogonal Fine-Tuning（Qiu et al., 2023）摒弃了加法更新，改为正交旋转：$W' = R \cdot W$，其中 $R$ 是通过 Cayley 变换生成的小型反对称矩阵。由于 $R$ 是正交矩阵，权重 $W$ 的奇异值得以完全保留。这一点在底模校准敏感时尤为重要——文生图微调是典型场景，LoRA 容易漂移底模的风格特性，而 OFT 能完整保留。缺点是，在相同 rank 下，OFT 每步计算成本比 LoRA 更高。

VeRA、IA³ 与极致压缩

VeRA（Kopiczko et al., 2024）让所有层共享同一组随机低秩矩阵 $(A, B)$，每层仅学习一个缩放向量。在 GLUE 基准上，VeRA 将参数量压缩到 LoRA 的十分之一，精度却几乎持平——当你需要存储数千个客户专属适配器且磁盘空间受限时，这非常有用。IA³ 更进一步，每个 Transformer 层只学习三个缩放向量（分别对应 key、value 和 FFN），适合适配器必须小于 100 KB 的场景。

实话实说：2025 年通用 LLM 微调，LoRA 仍是首选默认方案。当低 rank 下精度损失过大时，选择 DoRA；图像生成任务选 OFT；只有在适配器存储成为瓶颈时，才考虑 VeRA。

大规模多 LoRA 在线服务

LoRA 的架构优势在于，适配器是小矩阵，可以加到冻结的基础模型上。但这种优势只有在实际利用时才能转化为线上服务的优势。我见过三种有效的生产模式。

热插拔适配器缓存

基础模型常驻 GPU 显存，每个请求动态加载对应的 LoRA 矩阵。对于 7B 参数的模型，rank 为 16 时，每个适配器大约 16 MB。CPU 内存可以轻松存放几百个适配器，通过 PCIe 传输到 GPU 只需不到 5 毫秒。这种模式非常适合多租户 SaaS 场景，每个客户有自己的微调模型。目前主流框架如 vLLM 的 --enable-lora、S-LoRA 和 Punica 都支持这种模式，只是显存布局略有不同。

异构请求批处理

真正的难点在于，同一个批次中包含针对不同 LoRA 的请求。简单做法是串行处理，但这会完全浪费批处理的优势。S-LoRA（Sheng et al., 2023）提出了一种解决方案：将适配器矩阵分页加载到统一缓冲区，并通过定制内核按请求动态获取对应的 $A$ 和 $B$。在 A100 上，“单 LoRA” 和 “单批次 100 LoRA” 的吞吐差距从 8 倍缩小到不到 1.5 倍。如果你需要同时服务多个不同的适配器，这种模式值得参考。

何时将适配器合并回底模

如果某个适配器的流量占比超过 95%，正确的选择是将其合并到底模权重中（公式：$W' = W + BA$），然后只部署一个合并后的模型。这样虽然失去了多租户的优势，但换来了单模型服务的简洁性，还能略微降低延迟（省去适配器前向计算）。我在一个面向客户的助手中做过这样的权衡：单一企业角色占了 98% 的 QPS，合并后 p99 延迟降低了 12%，且精度完全没有损失。

决策逻辑如下：如果各适配器的 QPS 差不多，分开存储并使用 S-LoRA 风格的批处理；如果某个适配器占主导地位，直接合并；如果有几百个低 QPS 的适配器，则采用热插拔模式，配合 CPU 缓存动态加载。

Q&A

PEFT 和全量微调相比，性能损失大吗？

模型参数超过 10B 的时候，差距通常不到 1 分，基本在噪声范围内。参数低于 1B 的情况下，差距会明显一些，通常是 2-5 分，尤其是数据集比较小的时候。

LoRA 的学习率怎么设置？

比全量微调高 1 到 2 个数量级。LoRA 的可训练参数初始值接近零，能承受更大的步长而不会发散。一般用 $\text{lr} = 1\text{e-}4$ 到 $5\text{e-}4$，搭配 cosine 衰减。

PEFT 能和量化一起用吗？

当然可以，QLoRA 就是这么干的。NF4 + 双重量化 + bf16 LoRA，可以在单张 48 GB 显卡上微调 65B 模型，性能下降不到 2%。

LoRA 需要加到所有线性层上吗？

没必要。原论文实验表明，query 和 value 已经能带来大部分收益；加上 key 和 output 收益很小，但参数量会翻倍。对于纯解码器 LLM，有时再加 MLP 的 up/down 投影对长上下文任务有帮助。

Adapter 和 LoRA 在推理时哪个更好？

LoRA 合并后没有额外开销，推理性能和全量微调完全一致。Adapter 因为多了一层串行子模块，每个模块会增加 1-3% 的延迟；如果用并行 Adapter，延迟可以降到 1% 以下。

小结

PEFT 的核心思想其实很简单：预训练模型已经几乎无所不能，下游任务只需要稍微调整一下就行。

  • LoRA 用低秩分解来实现这种调整，没有延迟，合并也方便，已经成为当前的默认选择。
  • Adapter 在模型内部插入一个瓶颈模块来做调整，存储成本低，但会增加推理时间。
  • Prefix-Tuning、Prompt-Tuning 和 P-Tuning v2 把调整部分设计成"软 token”，参数量非常少。
  • BitFit 更是简单到只调整 bias 参数，但作为基线方法却出人意料地有效。
  • QLoRA 在 LoRA 的基础上加入了 4-bit 量化，把单卡微调的模型规模上限推到了 65B。
