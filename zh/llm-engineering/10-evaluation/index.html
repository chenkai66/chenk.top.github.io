大模型工程（十）：评估

为什么 MMLU 坏了、污染问题、LLM-as-judge 偏置、位置偏置缓解，以及生产里真正能在客户之前抓到回归的 A/B 测试模式。

Read in English →

评估是 LLM 栈里人人有意见、没人有把握的一段。Leaderboard 被刷分，公开 benchmark 被污染，我加入过的多数团队没有 eval set。这一篇讲评估实际告诉你什么、benchmark 在隐藏什么、生产里在客户之前抓到回归的模式。

为什么公开 benchmark 撒谎

标准 benchmark 套件（MMLU、GSM8K、HumanEval、ARC、HellaSwag 等）的问题清单：

污染。 多数公开 benchmark 通过网络爬虫可访问。在 CommonCrawl 上训的模型见过题、常常见过答案。Phi-3 被发现训练数据里有原文 MMLU 题；很难相信其他实验室更干净。

饱和。 MMLU 设计是给 2020 年模型为难的。2026 年顶级模型 88-92 %——在噪声带里竞争。MMLU 上 1.5 个点的提升可能是真提升、也可能是边界 case 上 50 题抽样的方差。这个 benchmark 几年前就不再区分能力了。

多选题偏置。 MMLU 是 4 选 1。永远选 “B” 的模型 25 %。在多选数据上重训过的模型有不能转移到自由格式输出的优势。

只有英文。 MMLU、GSM8K、HumanEval 都是英文。中文或法文出色的模型在权威 leaderboard 上分数平庸，但赢自己实际的部署。

格式耦合。 GSM8K 答案用正则抓 “the answer is X”。不按这个确切格式的模型即便对了也算错。

2024-2026 一代 benchmark（MMLU-Pro、GPQA、BIG-Bench Hard、RULER、SWE-bench、LiveBench、ArenaHard）想解决这些，但每个发布几个月内都有自己的刷分问题。

MMLU 实际还在量什么

虽然挨这么多批，MMLU 不是没用。它量某些东西——大致 “模型预训练时见过并保留多少大学水平知识”。这跟通用能力松散相关。MMLU 50 分的模型确实比 80 分的差；88 到 92 之间的差距多数是噪声。

把 MMLU 当粗筛子用（“这个模型在不在对的邮编”）。别用来比 85+ 范围内的两个模型。

LiveBench、ArenaHard 和动态 benchmark

2024-2025 对污染的回应是 动态 benchmark：每月新生成或新策展的题、测试前从未发布。

  • LiveBench（2024 年底发布，每月刷新）：数学、编程、推理、语言、指令遵循题，源自 cutoff 之后的论文和竞赛。难污染。
  • ArenaHard / Chatbot Arena（lmsys）：真实 chat 流量的人类成对偏好。更新慢但抗刷分，因为人在选。
  • SimpleQA（OpenAI，2024）：有具体短答案的事实题。设计来揭幻觉——胡编的模型输。

2026 年这些比 MMLU 信号更好但仍有问题。ArenaHard 重权 “格式好”、“温暖” 这种用户喜爱的东西，跟正确性可能脱钩。LiveBench 出题流程本身在你知道策展标准时可刷分。

防污染：留出评测

生产工作不应该在公开 benchmark 上评。建你自己的 eval set：

  1. 从真实（或合成-但-像真实）生产流量采 100-500 条代表性输入。
  2. 手写或手策展黄金输出。
  3. 永远不公开这个集合。永远不要把它放进任何打外部 API 的 prompt（厂商可能从你流量学）。
  4. 每次模型变更对它重评。

投入是真的（200 题手标 1-2 人天），但第一次抓到公开 benchmark 没抓到的回归就回本。

LLM-as-judge

自由格式输出（多数生产任务）精确匹配不行。你需要给 “这个答案好吗” 打分的办法。主流模式是 LLM-as-judge：强模型给被测模型的输出打分。

JUDGE_PROMPT = """你在评估 AI 助手的回答。

问题：{question}

参考答案（仅作打分参考；助手没看到）：
{reference}

助手的回答：
{candidate}

按 1-5 评分助手回答：
1 = 事实错或跑题
2 = 部分对但缺关键信息
3 = 对但解释差
4 = 对且解释好
5 = 对、解释好、且加了相关有用信息

只输出整数分数。"""

def judge(question, reference, candidate, model="claude-4-5-sonnet-20250901"):
    score = client.messages.create(
        model=model, max_tokens=10,
        messages=[{"role": "user", "content": JUDGE_PROMPT.format(...)}]
    )
    return int(score.content[0].text.strip())

LLM-as-judge 跟人类判断相关 70-85 %，看任务。比人类评估便宜得多，能扩到日均上千例。但有偏置：

  • 位置偏置：比两个答案时 judge 偏好先出现的（某些模型偏好后出现的）。缓解：两种顺序都呈现，平均。
  • 长度偏置：更长的答案分更高，即使更短更好。缓解：明确告诉 judge “除非有用，否则不奖励长度”。
  • 自我偏好：GPT-4 judge 偏好 GPT-4 输出；Claude 偏好 Claude。缓解：用跟被测模型不同家族的 judge。
  • 格式偏置：结构化（Markdown、标题）输出比同等好的散文得分高。缓解：打分前归一化格式。
  • 谄媚偏置：judge 有时偏向更自信的答案即便错。缓解：可能时带参考答案。

def pairwise_judge(question, answer_a, answer_b, judge_model):
    # 第一种顺序：A 先
    score_ab = judge(question, answer_a, answer_b, judge_model)
    # 反序：B 先
    score_ba = judge(question, answer_b, answer_a, judge_model)
    if score_ab == "A" and score_ba == "B": return "A"
    if score_ab == "B" and score_ba == "A": return "B"
    return "tie"

2024 年一篇论文（Zheng 等，“Judging LLM-as-a-Judge”）显示 swap-consistent 判断跟人类相关约 85 %；非 swap-consistent 的本质是噪声。

代码和数学：基于程序的评估

可验证领域跳过 judge：

  • 代码：候选跑 test suite。HumanEval、MBPP、SWE-bench 都这么做。Pass@k 量 $k$ 个采样里通过的比例。
  • 数学：抽最终数值答案，跟 ground truth 比。
  • JSON 输出：对 schema 验证、检查 key 在不在。
  • Tool call：模拟工具，检查调用效果。

基于程序的 eval 无偏、计算免费、即时。能用就用。前沿推理模型（o 系列、Qwen3-Reasoning）RLVR（第 4 篇）的主信号都是基于程序的 eval。

生产 A/B 测试

Eval set 在部署前抓回归。A/B 测试抓 eval set 漏的。

生产 A/B：5-10 % 流量路由到新模型变体，比结果指标：

  • 互动：用户继续对话了吗？
  • 解决：用户标对话已解决了吗？
  • 延迟：p50/p95 变差了吗？
  • 成本：每轮 token 数、每个解决请求的 $。
  • 满意度：thumbs up/down 率。
  • 升级：用户要求转人工了吗？

“互动” 这种表面指标会误导——以有趣方式跑题的模型驱动更多轮但解决更少问题。轮次数配解决率一起看。

统计显著性：1 万流量足以在 95 % 置信下检测二元指标 2 % 偏移。实验至少跑 7 天洗掉星期效应。流量异质（免费 vs 付费、移动 vs 桌面、语种）就分层。

Eval set 维护

你今天建的 eval set 6 个月后就过时。生产流量漂移、客户用例演进、你模型的失败模式不是下一季度模型的失败模式。

能跑的维护例程：

  • 每周：采 50 条生产调用，标异常，把有意思的失败加进 eval set。
  • 每月：所有生产 prompt 对 eval set 重跑，画质量趋势。
  • 每季：退役所有当前模型 100 % 通过的 eval 项（不再有信息量）。从近期失败模式加新项。
  • 每次模型变更：完整重评，无例外。eval 太慢就自动化、并行化。

Calibration：人们跳过的指标

模型 校准 是指其声明的置信度跟实际准确率匹配。说 “我 90 % 确信” 的模型应该 90 % 时间对。多数 LLM 系统性过自信——它们说 “我确定” 然后 30 % 时间错。

高风险部署（医疗、法律、金融），calibration 跟准确率一样重要。用 Expected Calibration Error 量：

$$ \text{ECE} = \sum_{m=1}^{M} \frac{|B_m|}{n} |\text{acc}(B_m) - \text{conf}(B_m)| $$

按置信度把预测分到 $M$ 个 bucket，每个 bucket 算 |准确率 - 平均置信度|，按 bucket 大小加权。

2026 年前沿模型 ECE 仍在 5-15 % 范围——有意义的过自信。RLHF 倾向 恶化 calibration（模型学到自信声调答案得高奖励）。模型需要知道自己不知道的部署里，可能需要在 “我不知道” 例子上 SFT 撤销一些 post-training 的自信度。

小结与下一篇

公开 benchmark 80 分以上多数是噪声。在你真实流量上手策展 eval set，把它当承重代码对待。LLM-as-judge 配偏置纠正能用（swap order、归一化长度、用不同 judge 家族）。任务可验证就用基于程序的 eval。线下 eval 漏的用生产 A/B 抓。Eval set 持续维护。别忘 calibration——很多部署里置信度跟准确率一样重要。
