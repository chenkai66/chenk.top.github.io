大模型工程（十）：评估

为什么 MMLU 坏了、污染问题、LLM-as-judge 偏置、位置偏置缓解、校准、生产里真正能在客户之前抓到回归的 A/B 测试模式。

评估是 LLM 技术栈中一个让人头疼的部分：人人都有想法，但没人真正有信心。排行榜被各种技巧刷分，公开的基准测试数据集早已被污染，我加入过的大多数团队甚至根本没有评估集。这一章我会聊聊评估到底能告诉你什么，基准测试隐藏了哪些问题，为什么没人解决 LLM 作为评判者的偏差，为什么大多数团队会跳过校准指标，以及如何在生产环境中通过模式设计赶在用户之前发现模型退化。

这一章的风格和系列中的其他内容略有不同。评估的问题大多不是技术性的，而是认识论层面的。问“模型 A 是否比模型 B 更好”其实是一个假设检验问题，但整个领域在设计干净实验方面的表现一直很糟糕。我下面引用的文献不是什么排行榜，而是一些关于失败模式的论文，这些内容应该让每个从业者都更加谨慎。

为什么公开 benchmark 撒谎

标准基准测试套件（MMLU、GSM8K、HumanEval、ARC、HellaSwag 等）存在以下问题：

数据污染。 大多数公开基准测试可以通过网络爬虫获取。用 CommonCrawl 训练的模型很可能已经见过题目，甚至答案。Phi-3 被发现训练数据中直接包含了 MMLU 的原题；很难相信其他实验室的数据会更干净。Zhou 等人在 2023 年的论文《Don’t Make Your LLM an Evaluation Benchmark Cheater》中系统测量了这一效应——即使少量测试集泄露，也能带来 10-30 个百分点的提升，这种提升与“合法”能力改进几乎无法区分。论文中最令人不安的发现是：随机抽取 CommonCrawl 数据时，已经有 5-10% 的流行基准测试题目混入其中。要避免污染，必须主动过滤，但大多数实验室只是嘴上说说，真正记录下来的很少。

另一篇相关论文，Sainz 等人 2023 年的《NLP Evaluation in trouble》，调查了 250 多个近期发布的基准测试，发现约 40% 的测试在至少一个主流基础模型的训练语料中存在可检测到的泄露。泄露率随着预训练语料规模的增长而上升。

饱和效应。 MMLU 最初设计是为了难倒 2020 年的模型。但到了 2026 年，顶级模型的得分已经达到 88-92%——它们已经在噪声范围内竞争了。MMLU 上 1.5 个百分点的提升，可能是真实的能力进步，也可能是从边界案例中抽取 50 道题时产生的采样方差。这个基准测试早在几年前就失去了区分能力。简单算一笔账：14K 道题，模型准确率为 90%，分数的标准误差大约是 $\sqrt{0.9 \cdot 0.1 / 14000} \approx 0.25$ 个百分点；如果某个声称的 1 点提升没有超过 ~0.7 点，那基本可以认为是在噪声范围内。

多选题偏差。 MMLU 是四选一的多选题。一个永远选 “B” 的模型能得 25% 的分数。那些在多选题数据上大量训练的模型，具备一种无法转移到自由形式输出的优势。更糟糕的是，多选题提示中的位置偏差确实存在——Wang 等人在 2023 年的论文《Large Language Models are not Fair Evaluators》中证明，打乱答案选项的位置会让大多数模型的准确率波动 2-8 个百分点。

仅限英文。 MMLU、GSM8K、HumanEval 全部是英文测试。中文或法语表现出色的模型，在权威排行榜上可能得分平平，但在实际部署中却表现优异。

格式依赖。 GSM8K 的答案提取依赖正则表达式匹配 “the answer is X”。如果模型没有严格按照这个格式输出，即使答案正确也会被判错。HuggingFace 评估团队在 2024 年的一项审计中估计，GSM8K 中 5-15% 的“错误”答案实际上是内容正确但格式不符合规范的结果。

2024-2026 年的新一代基准测试（MMLU-Pro、GPQA、BIG-Bench Hard、RULER、SWE-bench、LiveBench、ArenaHard、IFEval）试图解决这些问题，但每个测试发布几个月内都会出现新的刷分漏洞。

MMLU 实际还在量什么

虽然批评不少，但 MMLU 并非毫无价值。它测的是一个简单问题：模型在预训练阶段是否接触并记住了大量大学水平的知识？这和通用能力有一定程度的关联。MMLU 得分 50 的模型确实不如得分 80 的；至于 88 和 92 之间的差距，基本可以看作是噪声。

可以把 MMLU 当成一个粗略的筛选工具，用来判断“这个模型是不是在正确的范围内”。但别用它来比较 85 分以上的模型。

Benchmark 动物园：什么时候用什么

一份实用指南，告诉你每个 benchmark 能回答什么问题：

  • 通用知识 / 粗略能力：MMLU（只用来筛选）、MMLU-Pro（在前沿领域仍然有意义）。
  • 不确定性推理：GPQA Diamond（Rein 等，2023）——研究生级别的科学问题，专门设计来防止 LLM 使用取巧方法。2026 年顶级模型得分 60-75%，相关领域的博士生得分约 65%。这个 benchmark 非常难。
  • 数学，简单：GSM8K（Cobbe 等，2021，Training Verifiers to Solve Math Word Problems）——8.5K 道小学水平的应用题。前沿模型已经接近满分（>95%），但对小模型来说仍然有用。
  • 数学，困难：MATH（Hendrycks 等，2021，Measuring Mathematical Problem Solving）——12.5K 道来自 AMC/AIME 级别的竞赛题。在前沿领域仍有区分度（顶级模型 85-92%）。2025 年被 AIME-2025 和 Putnam-2025 取代，以体现更严格的竞赛标准。
  • 代码，封闭场景：HumanEval（Chen 等，2021，Evaluating Large Language Models Trained on Code）——164 道手写的 Python 编程题。已经被前沿模型饱和。MBPP（Austin 等，2021）类似，但范围稍广。
  • 代码，真实场景：SWE-bench（Jimenez 等，2023）——来自 12 个流行 Python 仓库的真实 GitHub 问题，模型需要通过修改代码库来解决问题。涉及多文件操作，要求理解现有代码。2026 年顶级模型在 SWE-bench Verified 上的得分为 50-60%。
  • 代码，广泛场景：BigCodeBench（Zhuo 等，2024）——涵盖 723 个函数和 139 个库的 1140 个任务。有效避免了 HumanEval 克隆版中常见的 GitHub 数据污染问题。
  • 指令遵循：IFEval（Zhou 等，2023）——验证模型是否满足明确的约束条件（例如“恰好用 3 段回答，每段以问题开头”）。采用程序化验证，歧义性低。
  • 长上下文：RULER（Hsieh 等，2024）——多难度的“大海捞针”测试，最长支持到 128K token。许多号称支持“1M 上下文”的模型，在这里测到 32K 时就撑不住了。
  • 幻觉检测：SimpleQA（OpenAI，2024）——4326 道简短的事实性问题，专门设计用来揭露模型的胡编乱造。前沿模型得分 30-55%，低端模型得分仅约 10%。
  • 成对人类偏好：Chatbot Arena / ArenaHard（lmsys）——基于真实聊天交互的百万级众包 A vs B 投票。

在实际生产中，很少有人关心“哪个模型在 MMLU 上表现最好”，几乎所有人都想知道的是“哪个模型在我的工作负载上表现更好”。Benchmark 动物园的主要作用是帮我在花钱做定制评估之前，快速淘汰明显不合格的候选模型。

LiveBench、ArenaHard 和动态 benchmark

2024 到 2025 年，针对数据污染问题，业界提出了 动态基准 的解决方案：每个月生成或筛选一批全新的题目，测试之前从未公开过。

  • LiveBench（2024 年底发布，每月更新）：涵盖数学、编程、推理、语言理解和指令跟随等类型的题目，全部来自 cutoff 之后的论文和竞赛。由于题目来源实时更新，很难被污染。
  • ArenaHard / Chatbot Arena（lmsys）：基于真实聊天流量中的人类两两偏好数据。虽然更新速度慢，但因为是人类选择，抗刷分能力较强。不过，Arena 的公开排行榜存在一个问题——很多用户更看重 风格偏好 而非 正确性偏好，导致排名更倾向于奖励那些语气友好、格式工整但略显啰嗦的答案，而不是简洁准确的回答。
  • SimpleQA（OpenAI，2024）：以简短明确的事实性问题为主，专门设计用来检测模型的幻觉问题——如果模型胡编乱造，就会直接暴露。

这些基准在 2026 年的表现确实比 MMLU 更有参考价值，但它们也有各自的缺陷。ArenaHard 过于偏向用户喜欢的“格式美观”和“语气友好”，这可能与答案的正确性脱节。而 LiveBench 的题目筛选流程，一旦你知道了策展标准，也有可能被针对性优化。一句话总结：没有任何公开基准能在持续的优化压力下长期存活。

防污染：留出评测

在生产环境中，不要用公开的 benchmark 做评估。自己构建一个 eval set：

  1. 从真实的生产流量中（或者合成但接近真实的场景）抽取 100 到 500 条代表性输入。
  2. 手动编写或精心整理出标准答案。
  3. 这个集合绝对不能公开，也不能出现在任何会调用外部 API 的 prompt 中，避免厂商通过你的数据学习。
  4. 每次模型更新后，都用这个集合重新评估。

投入是实实在在的——标注 200 道题需要 1 到 2 人天，但只要它第一次帮你发现公开 benchmark 没有捕捉到的性能回退，就值回票价。

一个关键点：如何划分 eval set 和它的内容同样重要。常见的错误是从生产流量中均匀采样 200 道题，而生产环境中的流量往往是 80% 简单、20% 困难。结果就是，简单问题占了大头，每个模型在这些题目上都能拿到 95% 的分数，eval 几乎失去意义。正确的做法是分层设计：让简单、中等和困难问题各占 30%-40%，其中“困难”部分直接从错误日志中提取。困难问题才是能真正区分模型能力的关键。

LLM-as-judge：主导模式和它的失败模式

自由格式输出（大部分生产任务）无法用精确匹配来评估。我需要一种方法来判断“这个答案好不好”。目前的主流方法是 LLM-as-judge（Zheng 等，2023，Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena）：用一个强大的模型为被测模型的输出打分。

JUDGE_PROMPT = """你在评估 AI 助手的回答。

问题：{question}

参考答案（仅用于评分参考；助手未见过）：
{reference}

助手的回答：
{candidate}

请按 1-5 分为助手的回答评分：
1 = 事实错误或跑题
2 = 部分正确但缺少关键信息
3 = 正确但解释不清
4 = 正确且解释清晰
5 = 正确、解释清晰，并补充了相关有用信息

只输出整数分数。"""

def judge(question, reference, candidate, model="claude-4-5-sonnet-20250901"):
    score = client.messages.create(
        model=model, max_tokens=10,
        messages=[{"role": "user", "content": JUDGE_PROMPT.format(...)}]
    )
    return int(score.content[0].text.strip())

LLM-as-judge 的评分与人类判断的相关性在 70%-85% 之间，具体取决于任务类型。这种方法比人工评估便宜得多，每天可以处理上千个样本。但它也有一些偏见，Zheng 的论文列举了这些问题，后续研究进一步细化了它们：

  • 位置偏见：比较两个答案时，模型倾向于选择先出现的答案（某些模型则偏好后出现的答案）。Zheng 测试发现，在 GPT-4-as-judge 上，交换顺序后有 30%-65% 的成对判断结果会翻转——远高于随机概率。解决办法：同时呈现两种顺序，取平均值。
  • 长度偏见：更长的答案往往得分更高，即使更短的答案可能更好。Dubois 等（2024，Length-Controlled AlpacaEval）测试发现，仅将答案长度加倍而不改变质量，就能提升 3-7 分。解决办法：明确告诉模型“除非内容有用，否则不要奖励长度”，或者使用长度控制评分（减去长度项，或对长度匹配的基线进行归一化）。
  • 自我偏好：GPT-4 偏好 GPT-4 的输出，Claude 偏好 Claude 的输出。Panickssery 等（2024，LLM Evaluators Recognize and Favor Their Own Generations）证明这是一种自我识别现象——模型能识别自己的输出并给予更高评分。解决办法：使用与被测模型不同家族的判别模型，或者组建多模型评审团，采用多数投票。
  • 格式偏见：结构化输出（如 Markdown、标题）比同等质量的散文得分更高。解决办法：在评分前统一格式。
  • 谄媚效应 / 冗长即严谨：模型有时会偏向更自信或更详细的答案，即使这些答案是错的。解决办法：尽可能提供参考答案，并告诉模型优先考虑正确性而非表达形式。

def pairwise_judge(question, answer_a, answer_b, judge_model):
    # 第一种顺序：A 在前
    score_ab = judge(question, answer_a, answer_b, judge_model)
    # 反序：B 在前
    score_ba = judge(question, answer_b, answer_a, judge_model)
    if score_ab == "A" and score_ba == "B": return "A"
    if score_ab == "B" and score_ba == "A": return "B"
    return "tie"

Zheng 等人指出，顺序一致的判断与人类评分的相关性约为 85%；而顺序不一致的判断基本上是噪声。直接丢弃这些噪声是最简单的修正方法。

Pointwise 和 Pairwise。这两种评分范式有不同的问题。Pointwise（独立为每个答案打 1-5 分）成本低，但容易受到批次间评分漂移和模型隐式锚定最近样本的影响。Pairwise（A 和 B 谁更好？）每次比较更可靠，但复杂度随模型数量呈平方增长。实际生产中的折中方案是：用 Pointwise 进行初步筛选（过滤掉明显差的候选），用 Pairwise 对最后 2-3 个重要候选进行最终评估。

代码和数学：基于程序的评估

在可验证领域，直接跳过人工评判：

  • 代码：用测试套件跑候选代码。HumanEval、MBPP、SWE-bench 都是这么做的。Pass@k 衡量的是 $k$ 个样本中通过测试的比例。
  • 数学：提取最终的数值答案，与标准答案对比。
  • JSON 输出：校验是否符合 schema，检查关键字段是否存在。
  • 工具调用：模拟工具行为，验证调用结果是否正确。

基于程序的评估无偏见、计算成本低、响应速度快。能用就尽量用。前沿推理模型（o 系列、Qwen3-Reasoning、DeepSeek-R1）在 RLVR（第 4 章）中都把基于程序的评估作为主要信号。

一个有用的视角是：任何能够用程序验证的领域，都可以转化为 RLVR 中的 训练信号。这意味着模型在该领域的能力提升速度大致取决于验证器的质量。数学和代码的进步比自然语言快，因为它们有更可靠的验证方法。如果你为某个可验证任务构建内部基准测试集，你不仅是在测量性能，还在创造潜在的训练信号，而开源社区迟早会因此回馈你。

生产 A/B 测试

Eval 集合能在部署前发现模型退化问题。A/B 测试则能捕捉 Eval 集合漏掉的问题。

生产环境的 A/B 测试方法是：将 5%-10% 的流量分配给新模型变体，然后对比结果指标：

  • 用户互动：用户是否继续对话？
  • 问题解决：用户是否标记对话已解决？
  • 响应延迟：p50 或 p95 延迟是否变差？
  • 成本：每轮对话的 token 数量、每次解决问题的花费（美元）。
  • 满意度：用户点赞或点踩的比例。
  • 升级需求：用户是否要求转接人工客服？

像“用户互动”这样的表面指标可能会误导人。一个跑题但有趣的新模型可能增加对话轮次，却解决不了多少问题。因此，我会把对话轮次和问题解决率结合起来看。

关于统计显著性：如果要在 95% 置信水平下检测二元指标 2% 的变化，1 万流量就够了。相关的样本量计算公式如下：

$$ n \approx \frac{16 \cdot p (1-p)}{\delta^2} $$

其中，$p$ 是基线比例，$\delta$ 是最小可检测效应。当 $p = 0.5$、$\delta = 0.02$ 时，每组需要约 1 万样本；如果 $\delta = 0.005$（检测 0.5% 的变化），则需要约 16 万样本。实验至少要运行 7 天，以消除星期周期的影响。如果流量来源复杂（比如免费 vs 付费、移动端 vs 桌面端、不同语言），我会按流量分层分析。

以下是两个值得了解的生产实践模式：

  • 延迟敏感的质量指标。如果新模型虽然准确率提高了 1%，但响应时间增加了 200 毫秒，最终效果可能是负面的，因为用户流失率会随着延迟增加而上升。在判定胜者之前，我会把质量和延迟整合成一个综合目标，比如 quality - $\alpha$ · latency_seconds，其中 $\alpha$ 是根据历史流失曲线校准的参数。
  • 带早停机制的序列 A/B 测试。传统的频率学派 t 检验假设样本量是固定的，如果中途偷看数据并提前停止实验，假阳性率会升高。为了既快速决策又控制假阳性率，我会使用序列检验方法，比如 mSPRT，或者简单地对偷看次数进行 Bonferroni 校正。

Eval set 维护

你今天构建的 eval set，半年后就会过时。生产环境的流量会变化，客户的使用场景会演进，当前模型的失败模式也不再是下个季度模型的失败模式。

以下是一个有效的维护流程：

  • 每周：抽取 50 条生产调用记录，标记异常情况，把有意思的失败案例加入 eval set。
  • 每月：用所有生产环境的 prompt 重新跑一遍 eval set，绘制质量趋势图。
  • 每季度：移除那些所有当前模型都能 100% 通过的 eval 项（因为它们已经没有区分度）。从最近的失败模式中补充新的测试项。
  • 每次模型更新：必须完整重新评估，没有任何例外。如果评估太慢，就自动化流程并并行处理。

一个有用的判断标准是：eval set 上模型准确率的 四分位距 不应该降为零。如果每个模型在每道题上都达到 99% 的准确率，说明你的 eval set 已经过于简单，失去了作用。这时需要删掉容易的题目，补充更有挑战性的新问题。

Calibration：人们跳过的指标

模型 校准 的意思是，它的置信度要和实际准确率一致。如果模型说“我有 90% 的把握”，那它就应该在 90% 的情况下是对的。但大多数 LLM 都过于自信——它们嘴上说“我很确定”，结果 30% 的时候是错的。

在医疗、法律、金融这些高风险场景中，校准的重要性不亚于准确率。衡量校准程度可以用 Expected Calibration Error（Guo 等，2017，《On Calibration of Modern Neural Networks》）：

$$ \text{ECE} = \sum_{m=1}^{M} \frac{|B_m|}{n} |\text{acc}(B_m) - \text{conf}(B_m)| $$

把预测按置信度分成 $M$ 个区间，每个区间计算 |准确率 - 平均置信度|，再按区间大小加权求和。可靠性图把准确率画在 y 轴，置信度画在 x 轴；完美校准的模型会落在对角线上。

Guo 的论文指出，提升准确率往往会牺牲校准效果。那些在分类任务中提高准确率的技术（比如深度网络、label smoothing、temperature scaling），如果不特别调整，通常会让校准变差。这个结论同样适用于 LLM：RLHF 和 DPO 这些后训练方法会让校准变得更糟，因为奖励模型更倾向于奖励听起来自信的答案，而不是模棱两可的回答。

到了 2026 年，前沿模型的 ECE 仍然在 5%-15% 之间——这说明它们还是过于自信。RLHF 甚至会让校准变得更差（模型学会了用自信的回答换取更高的奖励）。如果部署场景要求模型知道自己不知道的东西，可以试试以下方法：

  1. 通过 SFT 撤回一些后训练带来的过度自信。用那些基模型不确定但后训练模型变得自信的案例，训练模型回答“我不知道”。
  2. 推理时对 logprobs 使用 temperature scaling。在验证集上调校温度参数，这种方法成本低，通常能把 ECE 减少一半。
  3. 采样多个补全结果，用答案的熵作为置信度的代理。比如对一个问题采样 $N=10$ 次，如果 9 次答案一致，那就认为置信度高；如果答案分布是 5/10 或 0/0 这种分散的情况，那就认为置信度低。
  4. 拟合一个校准模型。基于（model_logprob、prompt_features）→ 实际正确性，在标注数据上训练。

对于 RAG 系统来说，检索置信度（比如 top-k 相似度得分、reranker 得分）比 LLM 自己生成的 token 更能反映真实的置信度。把这个信号接入管道里就好。

长格式评估：benchmark 漏的

大多数公开的基准测试关注的是短答案。但在实际生产任务中，长格式生成（如总结、草稿、500 行以上的代码块）更为常见。针对这一领域的基准测试包括：

  • AlpacaEval / AlpacaEval 2（Li 等，2023）：包含 805 条指令；使用 LLM-as-judge 方法与参考模型（最初是 text-davinci-003，后来改为 GPT-4）进行对比。Length-controlled 版本（Dubois 等，2024）解决了长度偏差的问题。
  • MT-Bench（Zheng 等，2023）：涵盖 8 个类别的 80 组多轮对话；LLM-as-judge 在评分前会提供 chain-of-thought 的理由。
  • Arena Hard（lmsys，2024）：从真实的 Chatbot Arena 流量中提炼出的 500 道难题；LLM-as-judge 的结果与人类在 Arena 中的排名高度相关。

在内部进行长格式评估时，我的实践经验是这样的：

  1. 收集 50 到 200 个能覆盖实际工作负载的提示词。
  2. 设计一个包含 5 到 10 个维度的评分标准，比如准确性、完整性、格式合规性、语气、幻觉率等。
  3. 使用一个强大的评判模型对每个维度单独打分，并附上一句简短的理由。
  4. 将各维度得分汇总成综合分数，但在决定是否上线时，一定要查看每个维度的详细表现。

按维度拆解的结果非常重要。如果一个模型在完整性上提高了 5 分，但在幻觉问题上下降了 3 分，这并不一定算是改进。最终是否上线，取决于哪个维度对你的产品更重要。

Eval pipeline as code

一个成熟的评估系统，运维规范应该和 CI 流水线一样清晰：

[eval_set.jsonl 放在版本控制中]
     ↓
[runner：对 N 个候选模型 × M 道题目并行处理]
     ↓
[grader：程序打分 + LLM-as-judge + 可选人工抽查]
     ↓
[结果存储：每次运行的分数，按题目和维度细分]
     ↓
[diff 视图：对比候选模型和基线，突出回归问题]
     ↓
[gate：如果回归超过阈值，阻止部署]

2026 年能用的工具包括：Promptfoo（简单易用，YAML 驱动，适合本地迭代）、Inspect AI（英国 AISI 的框架，用于安全评估）、OpenAI evals、DeepEval，或者如果你喜欢折腾，也可以用 pytest 自己搭建。有没有框架不重要，重要的是得有一个；最糟糕的做法是写个 notebook，等有人想起来才跑一次评估。

小结与下一篇

公开 benchmark 超过 80 分的大部分是噪声，而且很多已经被污染。我建议基于真实流量手工构建一个评估集，按难度分层，并像对待核心代码一样维护它。使用 LLM-as-judge 时，可以通过调整顺序、控制长度、更换 judge 家族或采用多模型投票来校正偏差。如果任务可以验证，优先选择基于程序的评估方法。线下评估遗漏的部分，通过生产环境中的 A/B 测试来补充；如果需要提前查看结果，记得使用序列检验。持续更新评估集，移除那些已经过时或过于简单的问题。别忘了校准——在很多实际部署中，置信度和准确率同样重要，而 post-training 往往会降低置信度。对于长文本生成任务，先按维度评分，再综合打分。
