大模型工程（十）：评估

为什么 MMLU 坏了、污染问题、LLM-as-judge 偏置、位置偏置缓解、校准、生产里真正能在客户之前抓到回归的 A/B 测试模式。

评估是 LLM 栈里人人有意见、没人有把握的一段。Leaderboard 被刷分，公开 benchmark 被污染，我加入过的多数团队没有 eval set。这一篇讲评估实际告诉你什么、benchmark 在隐藏什么、没人修的 LLM-as-judge 偏置、多数团队跳过的校准指标、生产里在客户之前抓到回归的模式。

这一章味道跟系列其他略不同。多数评估问题不是技术——是 认识论的。“模型 A 比模型 B 好吗” 是假设检验问题，这领域跑干净实验的集体记录很差。我下面引的文献不是 leaderboard；是失败模式论文集合，应该让任何从业者更谨慎。

为什么公开 benchmark 撒谎

标准 benchmark 套件（MMLU、GSM8K、HumanEval、ARC、HellaSwag 等）的问题清单：

污染。 多数公开 benchmark 通过网络爬虫可访问。在 CommonCrawl 上训的模型见过题、常常见过答案。Phi-3 被发现训练数据里有原文 MMLU 题；很难相信其他实验室更干净。Zhou 等（2023，Don’t Make Your LLM an Evaluation Benchmark Cheater）系统地量了效应——即便少量 test-set 泄露也产生 10-30 个百分点增益，跟"合法"能力提升不可区分。论文最不舒服的发现：CommonCrawl 随机子采样已包含 5-10 % 流行 benchmark 题。避污染要主动过滤，多数实验室声称、少数文档化。

相关论文 Sainz 等（2023，NLP Evaluation in trouble）调查 250+ 近期 benchmark 发布，发现 ~40 % 在至少一个主流基模型训练语料里有可检测泄露。泄露率大致跟预训练语料规模一起涨。

饱和。 MMLU 设计是给 2020 年模型为难的。2026 年顶级模型 88-92 %——在噪声带里竞争。MMLU 上 1.5 个点的提升可能是真提升、也可能是边界 case 上 50 题抽样的方差。这个 benchmark 几年前就不再区分能力了。简单统计论证：14K 题、模型 90 % 准确率，分数标准误约 $\sqrt{0.9 \cdot 0.1 / 14000} \approx 0.25$ 个百分点；不超过 ~0.7 点的 1 点声明在噪声内。

多选题偏置。 MMLU 是 4 选 1。永远选 “B” 的模型 25 %。在多选数据上重训过的模型有不能转移到自由格式输出的优势。更糟，多选 prompt 内位置偏置真实——Wang 等（2023，Large Language Models are not Fair Evaluators）证明洗答案位置在多数模型上让准确率变 2-8 点。

只有英文。 MMLU、GSM8K、HumanEval 都是英文。中文或法文出色的模型在权威 leaderboard 上分数平庸，但赢自己实际的部署。

格式耦合。 GSM8K 答案用正则抓 “the answer is X”。不按这个确切格式的模型即便对了也算错。HuggingFace 评估团队 2024 年审计估 GSM8K 上 5-15 % 的"错"答案是非规范格式的正确内容。

2024-2026 一代 benchmark（MMLU-Pro、GPQA、BIG-Bench Hard、RULER、SWE-bench、LiveBench、ArenaHard、IFEval）想解决这些，但每个发布几个月内都有自己的刷分问题。

MMLU 实际还在量什么

虽然挨这么多批，MMLU 不是没用。它量某些东西——大致 “模型预训练时见过并保留多少大学水平知识”。这跟通用能力松散相关。MMLU 50 分的模型确实比 80 分的差；88 到 92 之间的差距多数是噪声。

把 MMLU 当粗筛子用（“这个模型在不在对的邮编”）。别用来比 85+ 范围内的两个模型。

Benchmark 动物园：什么时候用什么

实用指南，哪个 benchmark 答哪个问题：

  • 通用知识 / 粗能力：MMLU（仅当筛子用）、MMLU-Pro（前沿仍有意义）。
  • 不确定下推理：GPQA Diamond（Rein 等，2023）——研究生水平科学题，设计来抗 LLM 友好 trick。2026 顶级模型 60-75 %；相关领域博士人类 ~65 %。这 benchmark 真的难。
  • 数学，较易：GSM8K（Cobbe 等，2021，Training Verifiers to Solve Math Word Problems）——8.5K 小学水平应用题。前沿模型饱和（>95 %）但小模型仍有用。
  • 数学，较难：MATH（Hendrycks 等，2021，Measuring Mathematical Problem Solving）——12.5K 来自 AMC/AIME 级竞赛题。前沿仍区分（顶模型 85-92 %）。2025 年被 AIME-2025 和 Putnam-2025 取代真竞赛严谨性。
  • 代码，封闭：HumanEval（Chen 等，2021，Evaluating Large Language Models Trained on Code）——164 道手写 Python 题。饱和。MBPP（Austin 等，2021）类似但稍广。
  • 代码，真实：SWE-bench（Jimenez 等，2023）——12 个流行 Python repo 的真实 GitHub issue，模型必须改 repo 解决。多文件、要懂现有代码。2026 顶级模型在 SWE-bench Verified 上 50-60 %。
  • 代码，广：BigCodeBench（Zhuo 等，2024）——723 个函数 139 个库的 1140 任务。抗 HumanEval 克隆受的 GitHub 污染。
  • 指令遵循：IFEval（Zhou 等，2023）——验证模型是否满足显式约束（“恰好 3 段，每段以问题开头”）。程序验证、低歧义。
  • 长上下文：RULER（Hsieh 等，2024）——多难度 needle-in-haystack 到 128K token。多数 “1M context” 声明在这里测到 32K 时就垮。
  • 幻觉：SimpleQA（OpenAI，2024）——4326 道短答事实题，设计揭穿胡编。前沿模型 30-55 %；低端约 10 %。
  • 成对人偏好：Chatbot Arena / ArenaHard（lmsys）——真实 chat 互动上百万众包 A vs B 投票。

生产工作里相关问题很少 “哪个模型赢 MMLU”，几乎总是 “哪个模型赢我负载”。Benchmark 动物园主要用来 淘汰明显差的候选，再花钱定制 eval。

LiveBench、ArenaHard 和动态 benchmark

2024-2025 对污染的回应是 动态 benchmark：每月新生成或新策展的题、测试前从未发布。

  • LiveBench（2024 年底发布，每月刷新）：数学、编程、推理、语言、指令遵循题，源自 cutoff 之后的论文和竞赛。难污染。
  • ArenaHard / Chatbot Arena（lmsys）：真实 chat 流量的人类成对偏好。更新慢但抗刷分，因为人在选。Arena 公开 leaderboard 受不同 bug——多数用户 风格偏好 主导 正确性偏好，所以排名奖励温暖、格式好、稍啰嗦的答案胜过正确简洁的。
  • SimpleQA（OpenAI，2024）：有具体短答案的事实题。设计来揭幻觉——胡编的模型输。

2026 年这些比 MMLU 信号更好但仍有问题。ArenaHard 重权 “格式好”、“温暖” 这种用户喜爱的东西，跟正确性可能脱钩。LiveBench 出题流程本身在你知道策展标准时可刷分。没有公开 benchmark 能在持续优化压力下幸存。

防污染：留出评测

生产工作不应该在公开 benchmark 上评。建你自己的 eval set：

  1. 从真实（或合成-但-像真实）生产流量采 100-500 条代表性输入。
  2. 手写或手策展黄金输出。
  3. 永远不公开这个集合。永远不要把它放进任何打外部 API 的 prompt（厂商可能从你流量学）。
  4. 每次模型变更对它重评。

投入是真的（200 题手标 1-2 人天），但第一次抓到公开 benchmark 没抓到的回归就回本。

微妙点：你怎么切 eval set 跟里面有什么一样重要。常见错误是从生产流量均匀采 200 题，而生产是 80 % 易、20 % 难。你的 eval 被每个模型都 95 % 的易题主导。分层：30-40 % 易、30-40 % 中、30-40 % 难，“难” 来自你错误日志。难那片是区分模型的部分。

LLM-as-judge：主导模式和它的失败模式

自由格式输出（多数生产任务）精确匹配不行。你需要给 “这个答案好吗” 打分的办法。主流模式是 LLM-as-judge（Zheng 等，2023，Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena）：强模型给被测模型的输出打分。

JUDGE_PROMPT = """你在评估 AI 助手的回答。

问题：{question}

参考答案（仅作打分参考；助手没看到）：
{reference}

助手的回答：
{candidate}

按 1-5 评分助手回答：
1 = 事实错或跑题
2 = 部分对但缺关键信息
3 = 对但解释差
4 = 对且解释好
5 = 对、解释好、且加了相关有用信息

只输出整数分数。"""

def judge(question, reference, candidate, model="claude-4-5-sonnet-20250901"):
    score = client.messages.create(
        model=model, max_tokens=10,
        messages=[{"role": "user", "content": JUDGE_PROMPT.format(...)}]
    )
    return int(score.content[0].text.strip())

LLM-as-judge 跟人类判断相关 70-85 %，看任务。比人类评估便宜得多，能扩到日均上千例。但有 Zheng 论文枚举、后续工作精炼的偏置：

  • 位置偏置：比两个答案时 judge 偏好先出现的（某些模型偏好后出现的）。Zheng 量在 GPT-4-as-judge 上换序时 30-65 % 的成对判断翻转——远高于随机。缓解：两种顺序都呈现，平均。
  • 长度偏置：更长的答案分更高，即使更短更好。Dubois 等（2024，Length-Controlled AlpacaEval）量了仅把答案长度翻倍质量不变就有 +3-7 点提升。缓解：明确告诉 judge “除非有用，否则不奖励长度”，或用长度控制评分（减一个长度项，或对长度匹配的基线归一化）。
  • 自我偏好：GPT-4 judge 偏好 GPT-4 输出；Claude 偏好 Claude。Panickssery 等（2024，LLM Evaluators Recognize and Favor Their Own Generations）证明这是自识别现象——judge 模型能识别自己输出并打更高分。缓解：用跟被测模型不同家族的 judge，或用不同家族 judge 面板多数票。
  • 格式偏置：结构化（Markdown、标题）输出比同等好的散文得分高。缓解：打分前归一化格式。
  • 谄媚 / 啰嗦当严谨：judge 有时偏向更自信或看起来更彻底的答案即便错。缓解：可能时带参考答案，告诉 judge 把正确性权重高于呈现。

def pairwise_judge(question, answer_a, answer_b, judge_model):
    # 第一种顺序：A 先
    score_ab = judge(question, answer_a, answer_b, judge_model)
    # 反序：B 先
    score_ba = judge(question, answer_b, answer_a, judge_model)
    if score_ab == "A" and score_ba == "B": return "A"
    if score_ab == "B" and score_ba == "A": return "B"
    return "tie"

Zheng 等显示 swap-consistent 判断跟人类相关约 85 %；非 swap-consistent 的本质是噪声。扔掉它们是最简单的修正。

Pointwise vs pairwise。两种打分范式有不同失败模式。Pointwise（独立给每个答案 1-5 评分）便宜但受跨 batch 评分漂移和 judge 隐式锚定最近例子困扰。Pairwise（A 还是 B 更好？）每比较更可靠但模型数二次。标准生产折中：pointwise 做分诊（过滤明显差候选），pairwise 给最后 2-3 个重要候选。

代码和数学：基于程序的评估

可验证领域跳过 judge：

  • 代码：候选跑 test suite。HumanEval、MBPP、SWE-bench 都这么做。Pass@k 量 $k$ 个采样里通过的比例。
  • 数学：抽最终数值答案，跟 ground truth 比。
  • JSON 输出：对 schema 验证、检查 key 在不在。
  • Tool call：模拟工具，检查调用效果。

基于程序的 eval 无偏、计算免费、即时。能用就用。前沿推理模型（o 系列、Qwen3-Reasoning、DeepSeek-R1）RLVR（第 4 篇）的主信号都是基于程序的 eval。

有用的框架：每个允许程序验证的领域都成了 RLVR 里的 训练信号，意味着模型在该领域能力大致随验证器质量增长。数学和代码长得比散文快因为它们有更好的验证器。如果你为可验证任务建内部 benchmark，你不只是在量——你在产生潜在训练信号，开源社区最终会回报你。

生产 A/B 测试

Eval set 在部署前抓回归。A/B 测试抓 eval set 漏的。

生产 A/B：5-10 % 流量路由到新模型变体，比结果指标：

  • 互动：用户继续对话了吗？
  • 解决：用户标对话已解决了吗？
  • 延迟：p50/p95 变差了吗？
  • 成本：每轮 token 数、每个解决请求的 $。
  • 满意度：thumbs up/down 率。
  • 升级：用户要求转人工了吗？

“互动” 这种表面指标会误导——以有趣方式跑题的模型驱动更多轮但解决更少问题。轮次数配解决率一起看。

统计显著性：1 万流量足以在 95 % 置信下检测二元指标 2 % 偏移。相关 power 计算：

$$ n \approx \frac{16 \cdot p (1-p)}{\delta^2} $$

其中 $p$ 是基线率、$\delta$ 是最小可检测效应。$p = 0.5$、$\delta = 0.02$，则每臂 $n \approx 10{,}000$；$\delta = 0.005$（抓半个百分点偏移）则 $n \approx 160{,}000$。实验至少跑 7 天洗掉星期效应。流量异质（免费 vs 付费、移动 vs 桌面、语种）就分层。

值得知道的两个生产模式：

  • 延迟感知质量指标。准确率高 1 % 但慢 200 ms 的新模型可能净负，因为用户流失随延迟扩展。声明赢家前把质量和延迟组合成单一目标（如 quality - $\alpha$ · latency_seconds，$\alpha$ 按历史流失曲线校准）。
  • 带早停的序列 A/B。频率派 t 检验假设固定样本量；偷看早停假阳性率会膨胀。用序列检验（mSPRT，或简单 Bonferroni 修正你偷看次数）保假阳性率诚实同时仍快决策。

Eval set 维护

你今天建的 eval set 6 个月后就过时。生产流量漂移、客户用例演进、你模型的失败模式不是下一季度模型的失败模式。

能跑的维护例程：

  • 每周：采 50 条生产调用，标异常，把有意思的失败加进 eval set。
  • 每月：所有生产 prompt 对 eval set 重跑，画质量趋势。
  • 每季：退役所有当前模型 100 % 通过的 eval 项（不再有信息量）。从近期失败模式加新项。
  • 每次模型变更：完整重评，无例外。eval 太慢就自动化、并行化。

有用的不变量：你 eval set 上模型准确率的 四分位距 不该塌缩到零。每个模型每题 99 % 你过饱和了，eval 不再做工。剔易题，找新难题。

Calibration：人们跳过的指标

模型 校准 是指其声明的置信度跟实际准确率匹配。说 “我 90 % 确信” 的模型应该 90 % 时间对。多数 LLM 系统性过自信——它们说 “我确定” 然后 30 % 时间错。

高风险部署（医疗、法律、金融），calibration 跟准确率一样重要。用 Expected Calibration Error 量（Guo 等，2017，On Calibration of Modern Neural Networks）：

$$ \text{ECE} = \sum_{m=1}^{M} \frac{|B_m|}{n} |\text{acc}(B_m) - \text{conf}(B_m)| $$

按置信度把预测分到 $M$ 个 bucket，每个 bucket 算 |准确率 - 平均置信度|，按 bucket 大小加权。Reliability diagram 把准确率画 y 轴对置信度画 x 轴；完美校准模型在对角线。

Guo 论文证明 准确率提升常以校准为代价——分类 benchmark 上提升准确率的技术（深网、label smoothing、temperature scaling）倾向伤校准除非显式纠正。同教训迁移到 LLM：post-training（RLHF、DPO）可靠地降级校准，因为奖励模型奖励自信声调答案多于 hedged。

2026 年前沿模型 ECE 仍在 5-15 % 范围——有意义的过自信。RLHF 倾向 恶化 calibration（模型学到自信声调答案得高奖励）。模型需要知道自己不知道的部署里，可能需要：

  1. 撤销一些 post-training 自信度 通过在 “我不知道” 例子上 SFT，例子源自基模型不确定但 post-train 模型变自信的 case。
  2. 推理时对 logprob 用 temperature scaling。在留出集上校准 temperature；这便宜常常把 ECE 折半。
  3. 采多个补全用答案熵作置信度代理。问题上 $N=10$ self-consistency——9/10 同意则高置信；5/10/0/0 等分散则低置信。
  4. 校准模型 在（model_logprob、prompt_features）→ empirical_correctness 标注样本上拟合。

RAG 系统专门：检索置信度（top-k 相似分、reranker 分）比 LLM 自家生成 token 是更诚实的置信信号。把它管道化。

长格式评估：benchmark 漏的

多数公开 benchmark 评短答。生产任务常涉及长格式生成（总结、草稿、500+ 行代码块）。瞄这块的 benchmark：

  • AlpacaEval / AlpacaEval 2（Li 等，2023）：805 个指令；LLM-as-judge 对参考（原 text-davinci-003，后 GPT-4）。Length-controlled 版（Dubois 等，2024）纠上面长度偏置问题。
  • MT-Bench（Zheng 等，2023）：8 类别 80 个多轮对话；LLM-as-judge 在评分前带 chain-of-thought 理由。
  • Arena Hard（lmsys，2024）：从真实 Chatbot Arena 流量蒸馏的 500 道难题；LLM-as-judge 跟人类 Arena 排名强相关。

内部长格式 eval 实战模式：

  1. 策展 50-200 个覆盖你真实负载的 prompt。
  2. 定义 5-10 维 rubric（准确率、完整性、格式合规、语气、幻觉率等）。
  3. 让强 judge 模型分维度打分，每维一行理由。
  4. 聚合到复合分，但决定是否上线时总看分维度拆解。

分维度视图关键。完整性涨 5 点幻觉跌 3 点的模型不严格是改进；上线与否看哪维对你产品更重要。

Eval pipeline as code

成熟评估系统跟 CI pipeline 一样运维卫生：

[版本控制里的 eval_set.jsonl]
     ↓
[runner：跨 N 个候选 × M 题并行]
     ↓
[grader：基于程序 + LLM-as-judge + 可选人工抽检]
     ↓
[结果存储：每次跑分，按题和维度拆]
     ↓
[diff 视图：候选 vs 基线，标回归]
     ↓
[gate：回归超阈值阻止部署]

2026 年能用的工具：Promptfoo（简单 YAML 驱动，适合本地迭代）、Inspect AI（英国 AISI 框架，安全 eval 用）、OpenAI evals、DeepEval、有这口味就用 pytest 自卷。框架不重要，有一个就行；最差模式是某人记起来时跑一次的 notebook。

小结与下一篇

公开 benchmark 80 分以上多数是噪声、很多被污染。在你真实流量上手策展 eval set，按难度分层，把它当承重代码对待。LLM-as-judge 配偏置纠正能用（swap order、长度控制、用不同 judge 家族、面板投票）。任务可验证就用基于程序的 eval。线下 eval 漏的用生产 A/B 抓，偷看时用序列检验。Eval set 持续维护、剔饱和题。别忘 calibration——很多部署里置信度跟准确率一样重要、post-training 可靠地降级它。长格式分维度评再组合。

下一篇：安全与对齐。拒绝行为、RLHF 目标、谄媚、红队方法学、幻觉指标、constitutional AI。
