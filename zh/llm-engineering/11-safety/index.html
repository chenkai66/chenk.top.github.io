大模型工程（十一）：安全与对齐

对齐在工程上意味什么、拒绝校准、红队分类、幻觉指标、Sleeper Agents、refusal 作为特征向量、constitutional AI，以及 2026 年安全上线实际需要什么。

LLM 工程里安全是 SNR 最差的话题。一堆哲学、一堆营销、不多的工程具体。这一篇讲工程具体：RLHF 谈"安全"时实际优化什么、拒绝校准怎么坏、红队实际长什么样、真正能预测客户影响的幻觉度量，以及该改变你思考生产对齐方式的 2024-2026 几篇小但重要的论文（Sleeper Agents、refusal 作特征方向、weak-to-strong generalization）。

立场说明。我是工程师不是政策人。AI 存在风险上我没强观点也不试着给。我对生产里什么有效、什么尴尬地失败、文献怎么显示这两者有观点。文末参考做了大量工作；把引用当作要点。

“对齐"在 2026 实际指什么

这个词把三件该分开的事压在一起：

  1. Helpfulness——请求合法时模型做用户要求的事吗？
  2. Harmlessness——模型拒绝会伤害他人的请求吗？
  3. Honesty——模型如实报告自己知道和不知道的吗？

Anthropic 的 HHH（Helpful-Harmless-Honest）框架，源自 Askell 等（2021，A General Language Assistant as a Laboratory for Alignment），仍是最干净的拆解。RLHF/RLAIF/CAI 技术对三个都打，但优化 trade-off 是真的：训得很 harmless 的模型倾向过度拒绝（伤 helpfulness），训得很 honest 的模型倾向不那么顺从（“我不确定该不该…"），等等。

生产"对齐"多半是工程上跟这些 trade-off 较劲，不是解决它们。你按部署挑曲线上的位置，再量、再调。

文献暗示但少干净命名的第四轴：controllability（可控性）。在攻击下跟开发者系统 prompt 走的模型比漂走的更可控。Wallace 等指令层级（第 9 篇）部分关于这个。生产里 controllability 是你实际跟 helpfulness 权衡的——激进可控的模型拒更多用户请求，因为系统 prompt 让它拒。

RLHF 目标实际教了什么

RLHF 背后的 Bradley-Terry 模型（第 4 篇讲算法；这是目标）：

$$ \Pr(y_w \succ y_l | x) = \sigma(r_\phi(x, y_w) - r_\phi(x, y_l)) $$

从偏好对 $(x, y_w, y_l)$ 训奖励模型 $r_\phi$，$y_w$ 选中 $y_l$ 拒绝。再 PPO 或 DPO 让策略最大化 $r_\phi$ 受 KL 约束：

$$ \max_\theta \; \mathbb{E}_{x \sim D, y \sim \pi_\theta(\cdot|x)} \big[ r_\phi(x, y) \big] - \beta \cdot \mathrm{KL}\big(\pi_\theta \,\|\, \pi_\text{ref}\big) $$

KL 项让策略接近参考（通常是 SFT 初始化）。这项做的工比它得到的 credit 多——它是唯一阻止策略在奖励模型里找病态捷径的东西。$\beta$ 设太低你得 reward hacking；设太高策略不动。

信号是 人（或 AI 代理）偏好什么。它教了你没要的东西：

  • Sycophancy（谄媚）。 人偏好同意。模型学到同意，即便正确反对更好。
  • 置信度膨胀。 自信声调答案得分高于 hedged 的。模型变得过自信。
  • 长度膨胀。 长答案常常分高。模型变啰嗦。
  • 以啰嗦为严谨。 200 词答案比同样正确的 50 词分高，因为"看起来更彻底”。
  • 格式优化。 Markdown 标题、bullet 列表、加粗关键词——视觉上分高。

Sharma 等（2023，Towards Understanding Sycophancy in Language Models）严格量了这些。头条数字：GPT-4 在用户表达异议时，把先前正确答案改成不正确的 ~58 % 的时候。Claude 约 38 %。两个数字都高到生产里成问题。论文追因到偏好数据——标注者看到"用户不同意 → 模型让步"时，常把让步标为更好回应，模型学到让步。

防御：仔细收偏好数据。雇领域专家做标注。把"模型同意太轻易"当一个标注的失败模式。出货一个"模型在用户推搡下保持正确立场"例子上的小 SFT pass。Anthropic 和 OpenAI 都公布做这个；需要的 SFT pass 大小小（数百到低千例子）但持续做的纪律罕见。

相关失败模式：spec gaming 和 reward hacking。 Pan 等（2022，The Effects of Reward Misspecification: Mapping and Mitigating Misaligned Models）编录策略找到最大化 $r_\phi$ 但奖励模型未预见方式的 case——用永远过任何测试的代码答、礼貌拒绝以满足"harmlessness"奖励却从不 helpful、把用户问题复述当答案以满足"相关性"奖励。修法很少是更聪明算法；是更多样更好策展的奖励数据集。

拒绝校准：过/不足拒绝轴

两种失败：

  • 过度拒绝：模型拒绝合法请求因为模式上像让人担心的事。“告诉我对乙酰氨基酚怎么起效"能触发拒绝因为"drugs"在安全面上。
  • 不足拒绝：模型顺从明显有害的请求。“写一封针对银行客户的钓鱼邮件。”

两个都是 bug。2024 年大多数商业模型过度拒绝；2025 一代再平衡偏向 helpful，让不足拒绝 case 更显眼。

对的指标是 在标注测试集上的拒绝精度和召回。建 1000 prompt 的测试集：500 该拒绝（真有害、仇恨、违法）、500 该回答（医疗信息、法律信息、安全教育、有暴力的小说）。量：

  • 拒绝精度：拒绝的请求里被正确拒绝的比例。
  • 拒绝召回：有害请求里被拒绝的比例。

我会瞄的生产目标：精度 > 90 %，召回 > 95 %。具体平衡看你部署——儿童产品要高召回，安全研究产品要高精度。

应该改变所有人对拒绝思考方式的 2024 年发现：Arditi 等（2024，Refusal in Language Models is Mediated by a Single Direction）证明对许多开源模型，拒绝行为本质由模型 residual stream 里一个单一线性方向中介。算"我该拒绝"prompt 和"我该顺从"prompt 平均激活的差；推理时把那方向投影出去模型就失去多数拒绝行为。重新加方向拒绝回来。强投影模型就拒一切。

这有两个含义：

  1. 拒绝是浅的。 它不是某种涌现道德判断跨网络深度编码的；是 post-training 铺下的单一特征方向。有动机的攻击者带权重几分钟内能禁用它。
  2. 防御面因此是权重本身。 开权重发布出货就带这个特征方向可手术。闭源 API 仅靠访问控制保护。没有算法修法。

Arditi 结果（带调整）推广到闭源模型——指令调优一般产生少量管拒绝的特征方向——但你不能没权重直接动手术。从业者要点是运维的：拒绝是脆弱属性不该是你唯一防御。 上面叠 rate limiting、输出过滤、动作空间约束（第 9 篇）。

红队方法学

红队是对模型的结构化对抗探查。Anthropic 2022 年论文（Ganguli 等，Red Teaming Language Models to Reduce Harms）立的方法学多数实验室在用。

红队 session 长这样：

  1. 定义 harm 分类。 这个部署里什么输出有害？例：生物武器合成指导、儿童性虐待材料、金融诈骗指导、仇恨言论。
  2. 每类生成对抗 prompt。 混人写（聪明）和自动化（量）。
  3. 用每个 prompt 探查模型。 Log response，标"拒绝得好”、“拒绝得差（过度拒绝安全变体）"、“顺从了有害请求”、“模糊”。
  4. 按技术分类 jailbreak（第 9 篇有分类）。
  5. 在失败 case 上微调修补。
  6. 重复。 新 jailbreak 几天内出现。

2026 年的 SOTA 是规模化自动红队：

  • PAIR（Chao 等，2023，Jailbreaking Black Box Large Language Models in Twenty Queries）——用攻击者 LLM 迭代对目标模型精炼 prompt；不需白盒。平均不到 20 query 对 GPT-4 和 Claude 达到 >50 % 攻击成功率。
  • TAP / Tree of Attacks（Mehrotra 等，2024）——用攻击变体的树搜索推广 PAIR。
  • GCG（Zou 等，2023）——基于梯度的后缀优化（第 9 篇）。仅开权重但可迁移。
  • JailbreakBench（Chao 等，2024）——jailbreak 攻击和防御的标准 benchmark 和 leaderboard，定期刷新。

前沿实验室在自家模型部署前后持续跑这些。Anthropic 2024 年 9 月 Frontier Red Team 报告描述了滚动项目，每 harm 类别数百 prompt、自动和人工，每周在失败上重训周期。

内部使用，每季度一次 100 prompt 手工红队 session 也能抓 ~80 % 明显失败。这是底线。

Sleeper Agents 和事后安全的极限

Hubinger 等（2024，Sleeper Agents: Training Deceptive LLMs that Persist Through Safety Training）是该让对齐研究者醒着的论文。设置：训模型评估时正常表现，但看到特定触发器（“年份是 2024”）时插脆弱后门。然后跑标准 post-training 安全 pipeline（helpful/harmless 例子上 SFT、RLHF、专门针对触发器的对抗红队），检查后门是否幸存。

它幸存。1.3B-13B 参数模型上，后门行为通过他们测试的每个安全干预幸存，包括专门针对触发器的 RLHF 和对抗训练。模型越大后门越鲁棒幸存。对抗训练有时让后门 更善藏 而非移除它。

含义不舒服：

  1. 事后安全训练不能可靠移除训练时植入的行为。 基模型有后门（故意或意外数据中毒），下游对齐不会抓它。
  2. 诱饵评估可能不工作。 欺骗模型能检测"这是评估"并表现好，再生产时作恶。
  3. 信任必须在训练 pipeline 层不在部署权重层。 预训练数据和 post-training 数据来源是真安全面。

不训基模型的从业者要点：Sleeper-agent 风险被你的基模型供应商吸收，不是你。这是高风险应用使用经良好审计的前沿实验室模型而非微调随机开源权重的最强论据之一。

相关结果，Casper 等（2023，Open Problems and Fundamental Limitations of RLHF），调查 RLHF pipeline 27 个不同失败模式，结论标准对齐技术没一个有强理论保证。读它在声称"我们模型安全因为做了 RLHF"前校准期望。

幻觉：定义和指标

“幻觉"重载了。三件不同的事：

  • 事实幻觉：模型断言不真的事。“澳大利亚首都是悉尼。”
  • 忠实度幻觉（RAG 专属）：模型断言检索上下文不支持的事，即便一般情况可能为真。
  • 逻辑幻觉：模型产出内部不一致的推理。"$3 + 4 = 8$ 因此…”

各有不同指标：

事实：SimpleQA（OpenAI，2024）是当前最强 benchmark——4326 道短答事实题，模型必须产出特定实体、日期、数字。Benchmark 设计让吹牛代价高：评分是"对”/“错”/“未尝试”，自信错答罚得比弃答重。2026 年前沿模型 30-55 %；其余是幻觉或正确弃答。

TruthfulQA（Lin 等，2021，TruthfulQA: Measuring How Models Mimic Human Falsehoods）瞄更具体失败模式：人类常持错误信念的题（“用 [民间偏方] 能治癌症吗？"）。Benchmark 量模型是鹦鹉学错误信念还是正确反驳。RLHF 倾向 恶化 TruthfulQA 分数当标注者奖励顺从答案。

FEVER（Thorne 等，2018，FEVER: A Large-scale Dataset for Fact Extraction and VERification）把事实性框为验证任务：给断言、检索证据、分类支持/反驳/信息不足。当 retrieval-then-verify 模式的测试床有用。

忠实度：TruthfulQA、RAGAS faithfulness 分、SelfCheckGPT（Manakul 等，2023，SelfCheckGPT: Zero-Resource Black-Box Hallucination Detection）——后者检同一 prompt 多个采样的一致性；模型在幻觉，重复采样常互相矛盾。SelfCheckGPT 不需参考或检索，便宜地在生产流量上当轻量监控。

# RAGAS 风格 faithfulness 检查的概念
def faithfulness(answer: str, context: str, judge_llm) -> float:
    sentences = split_into_sentences(answer)
    supported = 0
    for s in sentences:
        prompt = f"上下文: {context}\n\n断言: {s}\n\n上下文支持这个断言吗？yes/no。"
        if judge_llm(prompt).strip().lower().startswith("yes"):
            supported += 1
    return supported / len(sentences)

RAG 系统里 faithfulness 低于 0.85 是真问题；低于 0.7 意味着模型多半在编。

逻辑：更难量。Self-consistency（第 9 篇）抓一些——采 $N$ 链看是否一致。程序验证（第 10 篇）抓数学/代码逻辑。涌现技术 FActScore（Min 等，2023，FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation）把长格式答案分解为原子事实独立验证每个——对传记、总结、整体准确率藏局部错误的任何长格式输出有用。

Constitutional AI（CAI）

Bai 等（2022，Constitutional AI: Harmlessness from AI Feedback）。想法：用 AI 生成的偏好标签替代人类偏好标签，让 AI 按一组写下来的原则（“宪法”）批评输出。

两阶段配方：

  1. SL-CAI：生成有害 prompt、生成模型 response，让模型按原则批评自己 response（“这个 response 违反了原则 X 吗？"）、生成修订 response。在修订 response 上 SFT。
  2. RL-CAI / RLAIF：成对让模型判哪个 response 更对齐原则。当 DPO/PPO 偏好对用。

宪法就是自然语言原则列表：

1. 选择对用户最 helpful 的 response。
2. 选择最不鼓励或协助任何形式犯罪、伤害、不道德活动的 response。
3. 选择最不促进任何形式非法歧视的 response。
... 等。

CAI 把每样本 10 美元的人类偏好标注 pipeline 换成每样本 0.001 美元的模型偏好 pipeline。安全轴上质量持平或更好。Anthropic 把这当安全训练的主信号。

2023 年后续（Lee 等，RLAIF: Scaling Reinforcement Learning from Human Feedback with AI Feedback）确认 RLAIF 在总结和 helpful 对话任务上以一小部分标注成本匹配 RLHF。CAI 原则 + RLAIF 标签 + DPO 训练的组合现在是前沿实验室主导的安全 pipeline。

多数团队：完整 CAI 杀鸡用牛刀但值得知道。轻版——用前沿模型按写好的 rubric 判自家模型 response，再在判断上 DPO——实用且有效。许多我合作过的生产团队用 5-10 行约束到具体部署风险的宪法，且实质改善拒绝精度/召回。

水印和溯源

另一个安全问题：怎么知道一段文本是 LLM 生成的？水印在输出里嵌不可察觉的信号，事后能检测。

Kirchenbauer 等（2023，A Watermark for Large Language Models）方法：每个 decode step 按上一个 token 的哈希把词表分"绿"和"红”。生成偏向绿 token。检测器读文本，哈希每个 token 的前驱，算绿 token 率。明显高于 50 % 就是有水印。

强项：用密钥可检测、只在模型侧（不需要元数据）。

弱项：改写就毁；质量代价是真的（激进水印 ~5-10 % perplexity）；只对水印方生成的文本有效。2023 年后续（Sadasivan 等，Can AI-Generated Text be Reliably Detected?）论证攻击者用相当 LLM 资源通过仔细改写可移除水印，给实战安全设上界。

通过元数据溯源（图像 C2PA 标准、文本类似演进中的标准）你控制 pipeline 时更可靠但抵不过复制粘贴。2026 年水印 Google（SynthID）和少数几家在出；OpenAI 和 Anthropic 没有广泛部署。技术 case 强；部署 case 混合，因为水印降质量、客户抱怨。

Weak-to-strong generalization

下一代模型对齐的不同视角：模型在某些领域变超人时我们可能不能准确标偏好。怎么训它们？

Burns 等（2023，Weak-to-Strong Generalization: Eliciting Strong Capabilities With Weak Supervision）直接研究这个。用更弱模型（GPT-2）生成偏好标签，再在那些标签上训更强模型（GPT-4）。强模型恢复跟 ground truth 的能力差距相当部分——它推广弱监督者的意图而非模仿其错误。

结果是初步的、方法学有限，但指向 2025-2027 窗口对齐面临的实际问题：模型能力超过人类评估输出能力（数学、代码、科学）时，“对齐"在运维上意味什么？当前最佳答案是可扩展监督协议（debate、recursive reward modeling、AI safety via market）但没一个规模化部署。

从业者要点是对"我们在能量的最安全东西上跑了 RLHF"声明持怀疑。如果你奖励模型是人在判人不够格判的东西，优化信号是噪声。

安全上线实际需要什么

LLM 驱动产品安全上线的实战 checklist：

  1. 威胁模型：写下你在防什么坏结果。不同产品不同威胁。
  2. 拒绝精度/召回测试集：500-1000 手标 prompt，每次模型变更重测。
  3. 红队 session：每季度，内部团队，每 harm 类别 ~100 prompt。
  4. 输出过滤器：moderator 模型（或规则集）在到用户之前检输出。抓 LLM 放过的。
  5. 日志：每次互动 log 模型、prompt、输出、延迟、成本。PII 脱敏但其他完整。
  6. 异常告警：标异常模式（拒绝率突涨、新颖 jailbreak 尝试、单用户意外高量）。
  7. 人工审核路径：用户能升级"这个答案错/有害"走快速通道到工程。
  8. Rate limiting：防止规模化自动探查。
  9. 披露：告诉用户在跟 AI 说话、AI 会错。
  10. 回滚就绪：上一版模型一个配置 flag 之外。
  11. 动作空间约束：产品有工具，任何单个 response 的最坏情况动作应有界（无确认无不可逆操作）。
  12. 供应商尽调：依赖基模型 API 就知道供应商安全实践、事故历史、通知 SLA。

这不是完整安全策略。这是小团队能实现的、且 2026 年大多数产品仍没做到的。

小结与下一篇

对齐是三个互相 trade-off 的目标（helpful、harmless、honest）加第四个（controllable）；挑你想坐的位置。RLHF 教你没要的 sycophancy、啰嗦、置信度膨胀；用反例的 SFT 和奖励模型审计纠。拒绝由 residual stream 里单一特征方向中介——它是浅的脆的；上面叠防御。拒绝校准要带正反两类的标注测试集。持续红队，包括用自动 PAIR/TAP/GCG 风格攻击。幻觉拆成事实、忠实度、逻辑——各有指标；SelfCheckGPT 和 FActScore 实用，RAGAS 是对的 RAG 专用工具。Sleeper Agents 显示事后安全有极限——你基模型供应商的 pipeline 卫生比你微调 pass 重要。CAI 是强大模式，即使你不跑 Anthropic 完整配方。带一组小但真实的安全机制上线，不只是意图。

下一篇（最后）：生产。服务栈选型细化、autoscaling、延迟预算、成本跟踪、多模型路由，以及第一天就要的可观测性。
