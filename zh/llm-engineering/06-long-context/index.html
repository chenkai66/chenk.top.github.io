大模型工程（六）：长上下文 —— RoPE、YaRN、Attention Sinks

RoPE 怎么编码位置、为什么朴素扩展会崩、NTK-aware 和 YaRN 缩放、ALiBi vs RoPE、流式生成的 attention sinks，以及 1M 上下文承诺为什么常在检索测试上崩盘。

“1M token 上下文” 是 LLM 里最常被夸大的数字之一。模型能够注意到 1M token，这是架构层面的说法。而模型能否利用第 80 万位置的信息来回答问题，则是行为层面的陈述，难度更大。这一章主要讨论位置编码的数学原理、如何通过工程技巧将上下文扩展到训练长度之外，以及为什么大多数长上下文的声明在实际测试中站不住脚。

长上下文 LLM 的发展历程可以分为三个阶段。第一阶段（2017-2021 年）：模型通常在 512 到 2048 token 的范围内训练，因为注意力机制的时间复杂度是 $O(n^2)$，这个范围内的数据量是可以处理的。第二阶段（2022-2023 年）：高效的注意力内核（如 FlashAttention，Dao 2022 ↗ ）使得更长的训练成为可能，事后扩展技术（如 Position Interpolation、NTK-aware scaling 和 YaRN）让从业者能够将预训练的检查点从 4K 推广到 32K 甚至更多。第三阶段（2024-2026 年）：原生长上下文训练（例如 Llama 3.1 的 128K、Gemini 的 1-2M 和 Claude 的 200K）成为标准，但 可注意 的上下文和 可用 的上下文之间的差距依然存在——这也是这一章的重点。

位置不是免费的

自注意力机制对排列顺序不敏感。没有位置信号，模型无法区分“猫坐在垫子上”和“垫子坐在猫上”。以下是三种注入位置信息的方法：

  1. 正弦绝对位置（原始 Transformer）：在第一层之前，将位置的 $\sin/\cos$ 函数加到 token 嵌入中。
  2. 可学习绝对位置：为每个绝对索引学习一个位置嵌入，直到最大长度。GPT-2 和 BERT 使用这种方法。
  3. 旋转位置编码（RoPE）：在每个注意力层内部，按位置成比例的角度旋转 Q 和 K 向量。LLaMA、Qwen、Mistral 和 DeepSeek 都使用 RoPE。

RoPE 成为了主流选择。我知道的所有可靠的 2026 年 LLM 都采用 RoPE。原因有两个：每层都注入位置信息（信号更强），并且相对位置自然地从点积中得出——这正是注意力机制所需要的。

第四种方法是 ALiBi（线性偏置注意力），它在 2022 年前后曾认真竞争过但最终落败；本章后面会详细介绍，作为最有意思的替代方案。第五种方法是 xPos（Sun 等，2022），它是 RoPE 的改进版本，通过添加长度依赖的衰减来提高外推稳定性；DeepSeek 和其他一些现代模型采用了这种方法，但核心思想仍然是 RoPE 加上工程优化。

RoPE 的数学

对于位置 $m$ 的查询向量 $q$ 和位置 $n$ 的键向量 $k$，RoPE 会分别乘以旋转矩阵 $R(m\theta)$ 和 $R(n\theta)$，其中 $\theta$ 与维度有关（Su 等，2021 ↗ ）：

$$ \theta_i = b^{-2i/d}, \quad i = 0, 1, ..., d/2 - 1 $$

这里的 $b$ 是 rope base（默认值为 10000）。每对维度 $(2i, 2i+1)$ 以频率 $\theta_i$ 旋转。低索引的维度旋转快（携带细粒度位置信息），高索引的维度旋转慢（携带粗略位置信息）。

关键在于：旋转后的点积 $q \cdot k$ 只依赖于相对位置 $m - n$：

$$ \langle R(m\theta) q, R(n\theta) k \rangle = \langle q, R((n-m)\theta) k \rangle. $$

这就是 RoPE 能够外推的原因：模型看到的不是绝对位置 5 万，而是相对于当前 token 的偏移 -3——这种偏移在训练时见过十亿次了。

def rotate_half(x):
    x1, x2 = x[..., :x.shape[-1]//2], x[..., x.shape[-1]//2:]
    return torch.cat((-x2, x1), dim=-1)

def apply_rope(q, k, cos, sin):
    return q * cos + rotate_half(q) * sin, k * cos + rotate_half(k) * sin

def rope_cos_sin(seq_len, dim, base=10000.0, device="cuda"):
    inv_freq = 1.0 / (base ** (torch.arange(0, dim, 2, device=device).float() / dim))
    t = torch.arange(seq_len, device=device).float()
    freqs = torch.einsum("i,j->ij", t, inv_freq)
    emb = torch.cat([freqs, freqs], dim=-1)
    return emb.cos()[None, None, :, :], emb.sin()[None, None, :, :]

几乎所有现代注意力内核都直接处理 RoPE；这是概念上的过程。

波长与每个维度携带的信息

每个频率 $\theta_i$ 对应一个波长 $\lambda_i = 2\pi / \theta_i$——这对维度完成一整圈旋转所需的 token 位置数。假设 base $b=10000$、head_dim $d=128$：

  • $i=0$：$\lambda_0 = 2\pi \approx 6.3$ token。这对维度编码非常局部的位置。
  • $i=32$：$\lambda_{32} = 2\pi \cdot 10000^{32/64} \approx 628$ token。中程距离。
  • $i=63$：$\lambda_{63} = 2\pi \cdot 10000 \approx 62832$ token。最长波长为 62K token；如果训练上下文是 4K，这个维度在训练过程中连一圈都没转完。

这正是扩展问题的根源。波长远超训练上下文长度的维度，在训练时只见过它们旋转范围的一小部分。当要求模型关注远超训练长度的位置时，这些维度进入了模型从未见过的旋转区域。高频（短波长）维度编码局部位置，没问题，因为它们在每个训练上下文中循环多次，模型已经熟悉它们的全范围。

这一洞见推动了现代扩展方法的发展：不要均匀缩放所有维度，而是按波长缩放。

为什么朴素扩展会崩

用 rope_base=10000 和 max_position=4096 训练的模型，直接在位置 32768 使用会怎样？

最低频维度每 token 旋转 $10000^{-1} \approx 10^{-4}$ 弧度。到位置 32768 时，它们已经旋转了约 3.3 弧度——超过了 $\pi$，进入了训练时没见过的范围。点积几何进入模型从未学习过的领域，质量崩溃。

两种有效的扩展策略：

位置插值（PI）（Chen 等，2023 ↗ ）：将位置按比例 $s = L_{\text{new}} / L_{\text{train}}$ 缩小。位置 32768 在 $s=8$ 下变成有效位置 4096，这是模型见过的位置。继续微调几百步可以修正分布漂移。PI 能用，但短上下文质量会下降一些——每个位置现在对应的是模型从未训练过的小数旋转。原论文中，LLaMA 从 2K 扩展到 32K，只需 1000 步微调，质量退化极小；这使得事后上下文扩展成为可行的工程选项。

NTK-aware 缩放（2023 年中通过 r/LocalLLaMA 社区 ↗ 引入）：不是统一缩放位置，而是调整 rope base，让低频维度保持在训练范围内，而高频维度（携带局部信息）几乎不受影响。新的 base 是 $b' = b \cdot s^{d/(d-2)}$。这个直觉与上面的波长分析一致：训练时已经多次循环的维度不需要调整；几乎没有旋转的维度才需要调整。

YaRN（Peng 等，2024 ↗ ）：目前最好的方法。结合 NTK-aware 缩放和基于维度波长的插值强度（根据维度波长是否比训练上下文长或短），再加上温度校正（通过 $\sqrt{\log s}$ 重新缩放注意力 logit，补偿固定查询的 token 数增加）。YaRN 将 LLaMA-2-7B 从 4K 扩展到 128K，几乎不需要额外训练（几百步），短上下文质量几乎不下降（困惑度与基础模型相差不到 0.1）。

# YaRN 风格的 rope_base 上下文扩展
import math
def yarn_base(orig_base, orig_ctx, target_ctx, alpha=1, beta=32):
    ratio = target_ctx / orig_ctx
    return orig_base * (ratio ** (alpha / (alpha - beta)))
print(yarn_base(10000, 4096, 32768))  # ~1.6e8

实际部署开源模型（如 Qwen3、LLaMA-3、Mistral）时，它们通常已经完成了上下文扩展。一般不需要重新扩展；读技术报告了解它们使用的方法和实际限制即可。

LongRoPE 与基于搜索的缩放

YaRN 效果不错，但假设一个频段内所有维度使用统一的缩放公式。LongRoPE（Ding 等，2024 ↗ ）更进一步：它将每个维度的缩放因子视为一个搜索问题。在几千个候选缩放向量（每个长度为 $d/2$，每个 RoPE 频率对一个因子）上进行进化搜索，然后在小校准集（长上下文文本上的困惑度）上评估每个向量，保留表现最好的。LongRoPE 将 LLaMA-2-7B 扩展到 2M token，RULER 分数与原生长上下文模型相当，且无需显著的继续预训练。

这说明正确的缩放形状不是固定的——它取决于模型、数据分布和目标长度。逐维度搜索再微调是目前最有效的扩展策略。

实际应用中，如果看到模型卡上写着“上下文扩展到 1M”，要问清楚用的是哪种方法。PI、NTK-aware 和 YaRN 基本上是确定性的。而 LongRoPE 涉及搜索过程；发布的检查点编码了发现的缩放因子。部署成本相同（只是 cos/sin 表初始化方式不同），但在高扩展比下，质量有明显差异。

ALiBi：更简单的替代方案

ALiBi（Press 等，2022 ↗ ）完全省略了位置旋转，直接在注意力分数上加上线性偏置：

$$ \text{attn}_{ij} = \text{softmax}\left(\frac{q_i k_j^T}{\sqrt{d}} - m_h \cdot |i - j|\right) $$

其中 $m_h$ 是每个头的斜率，按头索引几何变化（通常情况下，对于 $H$ 个头，$m_h = 2^{-8h/H}$）。相邻的 token 获得更高的注意力；偏置在训练时固定，不需要旋转。

优点：可以在比训练时更长的上下文中进行扩展，无需微调。ALiBi 论文展示了在训练长度为 1024、测试长度为 2048 的情况下，质量没有下降。缺点：多数实验表明，在需要精确长程检索的任务中，ALiBi 不如 RoPE——线性衰减偏置导致非常远的 token 无论是否相关，其注意力都会指数级减少。

ALiBi 在 BLOOM、MPT 和一些研究模型中有所应用。RoPE 在实际生产中胜出。例外情况是 Mamba-注意力混合模型，ALiBi 在这里可能更经济，以及一些 “attention sink” 实现中，将 ALiBi 风格的衰减项加入 RoPE 以提高流式稳定性。

一个微妙的观察是：ALiBi 之所以有效，是因为语言中的大多数 token 确实存在大致对数的距离-相关性关系，log-attention 空间中的线性偏置近似了这种关系。它在检索任务上输给 RoPE，是因为检索任务特别需要 反 单调注意力——相关 token 可能是上下文中 最远 的，而不是最近的。RoPE 可以学习任意的注意力模式；而 ALiBi 则内置了距离先验。

Attention sinks：流式处理的技巧

Xiao 等人（2024, “StreamingLLM” ↗ ）发现了一个奇怪的现象：如果使用 窗口注意力（每个 token 只关注前 $w$ 个 token），并在超过窗口后继续解码，质量会在位置 $w+1$ 立刻下降。不是逐渐衰退，而是断崖式下跌。

原因在于：即使没有 key 匹配 query，注意力 softmax 也必须把质量分配到某个地方。训练好的模型会把这些质量分配给 序列的前几个 token——它们成了“attention sink”。去掉这些 sink 后，softmax 分布就会变得不连贯。

解决方法非常简单：在每个注意力窗口中始终保留前 4 到 8 个 token。结合滑动窗口，这样可以让模型解码任意长度的序列而不出现质量崩溃。

def streaming_attention(q, k_cache, v_cache, sink_size=4, window=4096):
    # 保留前 sink_size 个 + 最后 window 个
    sink_k, sink_v = k_cache[:, :, :sink_size, :], v_cache[:, :, :sink_size, :]
    win_k,  win_v  = k_cache[:, :, -window:, :], v_cache[:, :, -window:, :]
    k = torch.cat([sink_k, win_k], dim=2)
    v = torch.cat([sink_v, win_v], dim=2)
    return F.scaled_dot_product_attention(q, k, v)

从 Mistral-7B-v0.2 开始、Qwen3 以及大多数生产环境中的长上下文模型都采用了这种 sinks 和滑动窗口的方法。这决定了“1M 上下文能用”和“1M 上下文胡说”的区别。

为什么 softmax 需要 sink

数学上的原因是结构性的。Softmax 输出的是 key 上的概率分布，概率总和为 1。如果没有 key 与 query 有实际相关性（例如，模型正在处理一个“填充”token），softmax 仍然会产生一个分布。那么这些质量去了哪里？训练好的模型学会了将这些质量倾倒在几个特定的位置上——通常是 BOS 附近的前几个 token，因为每个训练样本中都有这些 token。这些 token 充当了“no-op”目标。

这对流式处理的影响非常严重。如果你把注意力窗口滑过这些 BOS 附近的 token，就移除了学到的 no-op 目标。此时，softmax 必须将质量分配给看起来相关的 key，导致注意力模式扭曲，质量急剧下降。永远保留这些 sink 几乎没有额外成本（只需 4 到 8 个 KV 条目），并且可以保持学到的动力学特性。

另一篇相关但不同的论文《Massive Activations》（Sun 等，2024）表明，sink 行为是更广泛现象的一部分——少量特征激活在训练好的 transformer 中具有不成比例的重要性。剪掉这些激活会毁掉模型。Sinks 是这一现象在注意力机制中的表现。

生产中的滑动窗口注意力

Mistral 7B（2023 年发布）是首个采用 滑动窗口注意力（SWA） 的流行模型：每个 token 只关注前 $w=4096$ 个 token。内存和计算量随 $w$ 线性增长，而不是随 $n$ 二次增长，这让 32K 上下文变得经济。

感受野仍然在增长：在第 $\ell$ 层，token 通过 $\ell$ 次跳跃，每次跳跃大小为 $w$，因此有效感受野是 $\ell \cdot w$。对于 Mistral 7B，它有 32 层且 $w=4096$，第 32 层的感受野达到 131K token，远超名义上的上下文窗口。远处的 token 信息确实会通过中间层间接传播。

实际应用中，SWA 结合 attention sink 能实现大多数生产环境下的长上下文行为。超过 32K 的纯密集注意力很少见；成本-质量曲线表明，对于长尾需求，SWA + sinks + 扩展技术（如 YaRN/LongRoPE）更优。

Needle in a haystack：唯一诚实的基准测试

架构允许你注意 N 个 token，但是否真的利用了第 N 个位置的信息是另一回事。标准测试（由 Greg Kamradt 在 2023 年提出）：在长上下文的位置 $p$ 插入一句 “magic number 是 7392”，然后问 “magic number 是几”。

“针” 提供了一个可分享的可视化结果（位置与上下文长度的红绿热图），但这只是一个简单的测试。单事实检索对现代模型来说太容易了。RULER（Hsieh 等，2024 ↗ ）是更现代的替代方案：13 个任务类别在控制上下文长度下进行测试，包括多事实检索、隐藏事实的多跳推理、变量跟踪和聚合任务。RULER 分数能告诉你模型在标称长度上是否真正有用，而不仅仅是能否复述字符串。

更严格的测试是 多事实检索。隐藏 $k$ 个事实，让模型全部找出来。大多数模型在 $k=5$ 时表现急剧下降，即使是在 32K 上下文中也是如此。基于多个事实的推理（将不同位置的 3 个事实链接起来）更难——前沿模型在 128K 上保持约 80% 的准确率，但在 256K 上则降至 50% 以下。

Lost in the Middle

另一个有详细记录的长上下文失败模式是 Lost in the Middle（Liu 等，2023 ↗ ）。即使模型能够注意到所有位置，它也会不成比例地关注开头和结尾的 token。长上下文中间 60% 的部分系统性地被低估。

Liu 等人的关键实验：在多文档 QA 提示中，将单个相关文档放在 19 个干扰文档的不同位置。位置 1（开头）和位置 20（结尾）的准确率在 75-80%；而位置 10（中间）的准确率下降到 50-55%。U 形曲线在每个测试模型（GPT-3.5、Claude-1、MPT 等）中都出现，甚至在 2026 年的模型中依然存在，只是程度较轻。

原因在于：训练时的分布使注意力偏向开头（大多数文档顶部有重要框架），而自回归解码中的近因偏置使注意力偏向结尾。中间部分因此被挤压。

实际解决方法：

  • 把问题放在长上下文的末尾，而不是开头。Lost-in-the-middle 注意力偏好末尾，这样模型在读完文档后会“思考”问题。经验上，这能提升 5-15% 的准确率。
  • 将最重要的上下文放在文档列表的开头或结尾，而不是中间，当你能控制顺序时。
  • 重新排序很重要。 即使检索返回了正确的文档，将其放在第 5 位（共 10 个位置）的效果也不如放在第 1 位。RAG 流程（第 8 篇）应按相关性从高到低排序检索片段。

原生长上下文 vs 事后扩展

到 2026 年，长上下文领域已经分化。有些模型在预训练时就将长上下文作为首要考虑；而另一些则是在事后进行扩展。

原生长上下文模型（如 Llama 3.1 的 128K、Gemini 2.5/3 的 1-2M 和 Claude 4.5 的 200K）在整个预训练过程中穿插长文档训练数据。这样，模型能自然地学习位置依赖行为，RULER 分数在整个上下文范围内都保持较高水平。

事后扩展的模型（例如社区版 LLaMA-2 扩展到 32K 用 YaRN，Qwen2.5 扩展到 128K 用扩展方法）则从短上下文的基础模型开始，通过应用 YaRN/LongRoPE 并加上少量长上下文数据继续预训练。RULER 分数在中等扩展（2-4 倍）时表现不错，但在激进扩展（16 倍以上）时会下降。

成本差异显著。从头开始预训练 128K 模型的成本大约是 4K 预训练的 2-4 倍（长序列下注意力计算占主导，即使使用 FlashAttention；KV 缓存内存限制了 batch 大小）。相比之下，事后扩展几乎免费（只需几千步微调）。如果我是一个基础实验室，愿意为一次训练投入 1 亿美元，那么原生长上下文的 2-4 倍溢价是合理的。但如果我是一个发布开源权重模型的小实验室，事后扩展则是唯一可行的选择。

RAG vs 长上下文：真正的权衡

最常见的应用问题是：“应该把文档塞进长上下文，还是用 RAG？”2026 年的答案比两年前更复杂。

适合塞进上下文的情况：

  • 文档能放进模型的 128K token 内。
  • 检索质量难以保证（查询多变，没有好的分块方法）。
  • 延迟预算可以接受 5 到 30 秒的预填充时间。
  • 成本预算可以接受每次查询约 0.30 美元（200K token 的提示词，每百万 token 1.50 美元）。

适合用 RAG（第 8 章）的情况：

  • 文档库远大于上下文窗口。
  • 延迟必须小于 1 秒。
  • 每次查询的成本必须是几分钱。
  • 需要来源归因。

长上下文在需要整体阅读的任务中表现出色，比如审查整个代码仓库、总结长时间会议、多文档问答且无法预测正确片段。其他情况下，RAG 在成本上能降低 1 到 2 个数量级。

混合方法（RAG 找候选，长上下文综合）是大多数非平凡工作负载的最佳选择。Anthropic 的 “Contextual Retrieval”（2024）和微软的 GraphRAG（2024）都是这种混合方法的变体；第 8 章会详细介绍。

生产小贴士

  • 部署前务必在实际负载上做 needle 测试。厂商提供的数字通常是最佳情况。
  • 重复服务时，给长输入启用 prompt 缓存。100K token 的系统 prompt 预填充一次要 1.50 美元，缓存后只需 0.05 美元（vLLM enable_prefix_caching、OpenAI/Anthropic 的 prompt 缓存 API）。
  • 把问题放在长上下文的末尾，而不是开头。Lost-in-the-middle（Liu 等，2023）研究表明，注意力在长上下文中更偏向开头和结尾。问题放在末尾能提高 5-15% 的准确率。
  • 位置 5 万之后的算术运算不可信。即使是 Claude-4.5，在输入数据较远时多步数学计算也会出错更多。如果可以，尽量将计算上下文移到问题附近。
  • 自己构建长上下文模型时，使用滑窗 + sinks。这种成本-质量取舍非常划算。
  • 关注 prefill 延迟曲线。TTFT 在单 GPU 上随 prompt 长度大致线性增长，但在 TP 模式下增长更快（all-reduce 占主导），跨节点设置下更是如此。TP=2 H100 上，200K token 的 prompt 预填充需要 6-8 秒；用户会注意到这个延迟。
  • 在自己的领域进行基准测试。一个在 RULER 上得 90 分的模型，在你的医疗记录任务上可能只有 70 分，因为医疗文本的分布与 RULER 合成数据不同。领域特定的评估才是最可靠的信号。

小结与下一篇

RoPE 让长上下文变得可行；YaRN 把它扩展到训练长度之外；sinks 使流式处理更稳定；但实际工作中的上下文总是小于标称的上下文。建议在自己的负载上测试。对于大多数生产任务，RAG 在成本上更有优势。长上下文则在整体阅读和短时交互工作流中表现更好。
