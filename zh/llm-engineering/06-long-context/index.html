大模型工程（六）：长上下文 —— RoPE、YaRN、Attention Sinks

RoPE 怎么编码位置、为什么朴素扩展会崩、NTK-aware 和 YaRN 缩放、ALiBi vs RoPE、流式生成的 attention sinks，以及 1M 上下文承诺为什么常在检索测试上崩盘。

“1M token 上下文” 是 LLM 里最虚标的数字之一。模型能注意到 1M token 是架构陈述。模型能 用 第 80 万位置的信息回答问题是行为陈述，难得多。这一篇讲位置编码的数学、把上下文延伸到训练长度之外的工程技巧、以及为什么大多数长上下文承诺过不了 needle-in-a-haystack 测试。

长上下文 LLM 的历史分三幕。第一幕（2017-2021）：模型在 512-2048 token 上训，因为注意力是 $O(n^2)$ 那是放得下的。第二幕（2022-2023）：高效注意力 kernel（FlashAttention，Dao 2022 ↗ ）让更长训练可行，事后扩展技术（Position Interpolation、NTK-aware、YaRN）让从业者把预训 checkpoint 从 4K 推到 32K 以上。第三幕（2024-2026）：原生长上下文训练（Llama 3.1 的 128K、Gemini 的 1-2M、Claude 的 200K）成标准，但 可注意 上下文和 可用 上下文之间的鸿沟仍在——这就是这一篇主要讲的。

位置不是免费的

自注意力对排列不变。没有位置信号，模型分不出 “猫坐在垫子上” 和 “垫子坐在猫上”。三种注入位置的办法：

  1. 正弦绝对位置（原始 Transformer）：在第 1 层之前给 token embedding 加位置的 $\sin/\cos$。
  2. 可学习绝对位置：每个绝对索引学一个位置 embedding，封顶在 max length。GPT-2、BERT 用。
  3. 旋转位置编码（RoPE）：在 每个 注意力层内部，按位置成比例的角度旋转 Q 和 K。LLaMA、Qwen、Mistral、DeepSeek 用。

RoPE 赢了。我知道的所有靠谱的 2026 LLM 都是 RoPE。两个原因：每层都注入位置（信号更强）、相对位置自然从点积里冒出来——这正是注意力真正要的。

第四个答案，ALiBi（线性偏置注意力），2022 年前后认真竞争过然后输了；本篇后面讲，作为最有意思的替代路径。第五个，xPos（Sun 等，2022），是 RoPE 的精修，加了长度依赖的衰减让外推更稳；DeepSeek 和几个其他现代模型在用，但核心思路还是 RoPE 加工程打磨。

RoPE 的数学

位置 $m$ 的 query $q$ 和位置 $n$ 的 key $k$，RoPE 各乘旋转矩阵 $R(m\theta)$ 和 $R(n\theta)$，$\theta$ 跟维度有关（Su 等，2021 ↗ ）：

$$ \theta_i = b^{-2i/d}, \quad i = 0, 1, ..., d/2 - 1 $$

$b$ 是 rope base（默认 10000）。每对维度 $(2i, 2i+1)$ 以频率 $\theta_i$ 旋转。低索引旋转快（携带细粒度位置），高索引旋转慢（携带粗位置）。

关键性质：旋转后的点积 $q \cdot k$ 只依赖 相对 位置 $m - n$：

$$ \langle R(m\theta) q, R(n\theta) k \rangle = \langle q, R((n-m)\theta) k \rangle. $$

这是 RoPE 能外推的原因：模型看到的不是绝对位置 5 万，而是相对当前 token 偏移 -3——训练时见过十亿次了。

def rotate_half(x):
    x1, x2 = x[..., :x.shape[-1]//2], x[..., x.shape[-1]//2:]
    return torch.cat((-x2, x1), dim=-1)

def apply_rope(q, k, cos, sin):
    return q * cos + rotate_half(q) * sin, k * cos + rotate_half(k) * sin

def rope_cos_sin(seq_len, dim, base=10000.0, device="cuda"):
    inv_freq = 1.0 / (base ** (torch.arange(0, dim, 2, device=device).float() / dim))
    t = torch.arange(seq_len, device=device).float()
    freqs = torch.einsum("i,j->ij", t, inv_freq)
    emb = torch.cat([freqs, freqs], dim=-1)
    return emb.cos()[None, None, :, :], emb.sin()[None, None, :, :]

几乎所有现代注意力 kernel 都把 RoPE 内联了；这是概念上发生的事。

波长与每个维度携带什么

每个频率 $\theta_i$ 对应波长 $\lambda_i = 2\pi / \theta_i$——那对维度完成一整圈旋转所需的 token 位置数。base $b=10000$、head_dim $d=128$：

  • $i=0$：$\lambda_0 = 2\pi \approx 6.3$ token。这对编码非常局部的位置。
  • $i=32$：$\lambda_{32} = 2\pi \cdot 10000^{32/64} \approx 628$ token。中程。
  • $i=63$：$\lambda_{63} = 2\pi \cdot 10000 \approx 62832$ token。最长波长 62K token；如果训练上下文是 4K，这个维度训练时连一整圈都没转完。

这是扩展问题的种子。波长 远长于 训练上下文的维度，训练时只见过它们旋转范围的极小一段。当我们让模型在远超训练长度的位置注意时，那些维度进入了模型从未见过的旋转领域。高频（短波长）维度——编码局部位置——没问题，每个训练上下文里循环很多次，模型对它们的全范围都熟。

这个洞见促成了现代扩展方法：别均匀缩所有维度，按波长缩。

为什么朴素扩展会崩

rope_base=10000、max_position=4096 训出来的模型，硬塞到位置 32768 用，会发生什么？

最低频维度每 token 转 $10000^{-1} \approx 10^{-4}$ 弧度。到位置 32768 转了约 3.3 弧度——超过 $\pi$，wrap 到了训练时没见过的地方。点积几何到了模型从未学习如何解释的领域。质量崩盘。

两种能跑的扩展策略：

Position interpolation（PI）（Chen 等，2023 ↗ ）：位置统一缩放 $s = L_{\text{new}} / L_{\text{train}}$。位置 32768 在 $s=8$ 下变成有效位置 4096，模型见过。继续微调几百步修小的分布漂移。PI 能跑，但短上下文质量损一点——每个位置现在对应模型从未训过的小数旋转。原 PI 论文把 LLaMA 从 2K 扩到 32K，1000 步微调、质量退化极小；这是让事后上下文扩展成为真工程选项的突破。

NTK-aware 缩放（2023 年中通过 r/LocalLLaMA 社区 ↗ 引入）：不是统一缩放位置，而是缩放 rope base，让低频维度留在训练范围、高频维度（携带局部信息）几乎不动。新 base $b' = b \cdot s^{d/(d-2)}$。直觉跟上面波长分析吻合：训练时已经循环很多次的维度不需要缩；几乎没转的维度才需要。

YaRN（Peng 等，2024 ↗ ）：当前最好。NTK-aware 缩放 + 按维度的插值强度（看维度波长比训练上下文短还是长）+ temperature 修正（注意力 logit 按 $\sqrt{\log s}$ 重缩，补偿对固定 query 注意的 token 数变多）。YaRN 把 LLaMA-2-7B 从 4K 扩到 128K，几乎不需要继续训（几百步），短上下文几乎不退化（perplexity 跟基础模型差 0.1 内）。

# YaRN 风格的 rope_base 上下文扩展
import math
def yarn_base(orig_base, orig_ctx, target_ctx, alpha=1, beta=32):
    ratio = target_ctx / orig_ctx
    return orig_base * (ratio ** (alpha / (alpha - beta)))
print(yarn_base(10000, 4096, 32768))  # ~1.6e8

实战中你会部署的开源模型（Qwen3、LLaMA-3、Mistral）都已经做完了上下文扩展。一般不需要重做；读它们技术报告知道用了哪种方法、实战上限在哪里。

LongRoPE 与基于搜索的缩放

YaRN 不错但假设一个频段内所有维度的统一缩放公式。LongRoPE（Ding 等，2024 ↗ ）走得更远：把 per-dimension 缩放因子当搜索问题。在几千个候选缩放向量（每个长度 $d/2$，每个 RoPE 频率对一个因子）上跑进化搜索，每个在小校准集（长上下文文本上的 perplexity）上评估，留下赢家。LongRoPE 把 LLaMA-2-7B 扩到 2M token，RULER 分数跟原生长上下文模型可比，没显著继续预训。

教训是对的缩放形状不是固定的——它依赖模型、数据分布、目标长度。Per-dimension 搜索再微调是当前已知最具表达力的扩展策略。

实战意义：模型卡看到 “上下文扩到 1M”，问哪种方法。PI、NTK-aware、YaRN 本质确定。LongRoPE 涉及搜索；发布的 checkpoint 编码了发现的缩放因子。部署成本一样（cos/sin 表初始化方式不同而已），但高扩展比下质量有意义地不同。

ALiBi：更简单的替代

ALiBi（Press 等，2022 ↗ ）完全跳过位置旋转，给注意力分数加线性偏置：

$$ \text{attn}_{ij} = \text{softmax}\left(\frac{q_i k_j^T}{\sqrt{d}} - m_h \cdot |i - j|\right) $$

$m_h$ 是 per-head slope，按头索引几何变化（典型 $H$ 头时 $m_h = 2^{-8h/H}$）。靠近的 token 注意力高；偏置训练时定死，不需旋转。

优点：训练长度之外的扩展无需微调。ALiBi 论文显示训 1024 / 测 2048 无质量损。缺点：多数实验显示 ALiBi 在需要精确长程检索的任务上不如 RoPE——线性衰减偏置意味着远 token 不管是否相关都指数级减少注意力。

ALiBi 在 BLOOM、MPT、几个研究模型里。RoPE 赢了生产之战。例外是 Mamba-注意力混合模型，那里 ALiBi 可能更便宜，以及几个 “attention sink” 实现里 ALiBi 风格的衰减项被加进 RoPE 来稳流式。

一个微妙观察：ALiBi 能跑是因为语言对多数 token 确实有大致对数的距离-相关性关系，log-attention 空间里的线性偏置近似了它。它在检索任务上输给 RoPE，是因为检索任务特别要求 反 单调注意力——相关 token 可能是上下文里 最远 的，不是最近的。RoPE 能学任意注意力模式；ALiBi 内置距离先验。

Attention sinks：流式 hack

Xiao 等（2024，“StreamingLLM” ↗ ）发现一个怪现象：如果做 窗口注意力（每个 token 只看前 $w$ 个）、解码超过窗口，质量在位置 $w+1$ 立刻崩盘。不是缓慢衰退——悬崖。

原因：注意力 softmax 必须把质量放某处，即使没有 key 匹配 query。训练好的模型把这部分质量路由到 序列的前几个 token——它们成了 “attention sink”。截掉这几个 sink，softmax 分布就不连贯了。

修法可笑地简单：每个注意力窗口里始终保留前 4-8 个 token。配合滑动窗口，模型就能解码任意长序列不崩。

def streaming_attention(q, k_cache, v_cache, sink_size=4, window=4096):
    # 保留前 sink_size 个 + 最后 window 个
    sink_k, sink_v = k_cache[:, :, :sink_size, :], v_cache[:, :, :sink_size, :]
    win_k,  win_v  = k_cache[:, :, -window:, :], v_cache[:, :, -window:, :]
    k = torch.cat([sink_k, win_k], dim=2)
    v = torch.cat([sink_v, win_v], dim=2)
    return F.scaled_dot_product_attention(q, k, v)

Mistral-7B-v0.2 之后、Qwen3，以及大多数生产长上下文模型内部都用 sinks + 滑动窗口。这就是 “1M 上下文能用” 和 “1M 上下文胡说” 的差别。

为什么 softmax 需要 sink

数学原因是结构性的。Softmax 输出 key 上的概率分布；按构造概率和为 1。如果没有 key 跟 query 有有意义的相关性（如模型在处理 “filler” token），softmax 仍产生一个分布。质量去哪？训练好的模型学会把这部分质量倾倒在几个特定位置上——通常是 BOS 附近的前几个 token，每个训练样本里都有。这些 token 充当 “no-op” 目标。

对流式的影响很严重。如果你把注意力窗口滑过这些 BOS 附近 token，你就移走了学到的 no-op 目标。Softmax 现在必须把质量放看起来真相关的 key 上，注意力模式扭曲、质量悬崖式跌。永远保留 sink 几乎没成本（4-8 个 KV 条目），且保住学到的动力学。

一篇相关但不同的论文，Massive Activations（Sun 等，2024），显示 sink 行为是更宽模式的一部分——少量特征激活在训练好的 transformer 里携带不成比例的重要性。剪掉这些激活毁掉模型。Sinks 是这个现象在注意力侧的呈现。

生产里的滑动窗口注意力

Mistral 7B（2023 发布）是首个流行的带 滑动窗口注意力（SWA） 出货的模型：每个 token 只注意前 $w=4096$ 个。内存和计算随 $w$ 线性而不是 $n$ 二次，让 32K 上下文便宜。

感受野仍增长：在层 $\ell$ 上，token 通过 $\ell$ 跳大小 $w$ 注意，所以有效感受野是 $\ell \cdot w$。Mistral 7B 32 层 $w=4096$，第 32 层感受野 131K token，远超名义上下文窗。远 token 信息确实通过中间层间接传播。

实战中 SWA 配 attention sink 给出多数生产长上下文行为。32K 之外的纯密集注意力少见；成本-质量曲线对长尾偏好 SWA + sinks + 扩展技术（YaRN/LongRoPE）。

Needle in a haystack：唯一诚实的 benchmark

架构让你能注意 N 个 token。是否真的用了第 N 个位置的信息是另一回事。标准测试（2023 年 Greg Kamradt 发起）：在长上下文位置 $p$ 插一句 “magic number 是 7392”，问 “magic number 是几”。

“针” 给了我们一个能分享的可视化（位置 vs 上下文长度的红绿热图）但是弱测试。单事实检索对现代模型太简单。RULER（Hsieh 等，2024 ↗ ）是现代替代：13 个任务类目在控制上下文长度下，包括多 needle 检索、隐藏事实多跳推理、变量追踪、聚合任务。RULER 分数告诉你模型在标称长度上 实际 是否有用，不只是能否回声字符串。

规律：标称上下文一般是工作上下文的 2-4 倍。预训练就是长上下文的架构（Gemini、Claude）比事后扩展的稳。生产里假设工作上下文是标称的一半，除非厂商公布 RULER 结果。

更痛的测试：多 needle。藏 $k$ 个事实，让模型全部取回。多数模型在 $k=5$ 处掉崖，即使 32K 上下文。针之间的推理（在不同位置的 3 个事实之间链接）更难——前沿模型在 128K 上保持约 80 %，256K 跌到 50 % 以下。

Lost in the Middle

第二个有充分文档的长上下文失败模式：Lost in the Middle（Liu 等，2023 ↗ ）。即使模型 能 注意到所有位置，它对开头和结尾的 token 不成比例地加权。长上下文中间 60 % 系统性被低用。

Liu 等的关键实验：在多文档 QA prompt 里把单个相关文档放在 19 个干扰里的不同位置。位置 1（开头）和位置 20（结尾）准确率 75-80 %；位置 10（中间）跌到 50-55 %。U 形曲线在测试的每个模型（GPT-3.5、Claude-1、MPT 等）都出现，2026 模型也仍存在，只是没那么严重。

原因：注意力被训练时分布偏向开头（多数文档头部有重要框架），被自回归解码的近因偏置偏向结尾。中间被挤压。

实战修法：

  • 问题放在长上下文的末尾，不是开头。Lost-in-the-middle 注意力偏好末尾，所以模型读完文档后 “想着” 问题。经验上：5-15 % 准确率提升。
  • 最重要的上下文放在文档列表开头或结尾，不是中间，当你能控顺序时。
  • Reranking 重要。 即使检索返回了对的文档，把它放第 5 个（共 10 个）位置不如放第 1。RAG pipeline（第 8 篇）应该按相关性最高在前排序检索 chunk。

原生长上下文 vs 事后扩展

到 2026 年长上下文景观分裂。一些模型把长上下文作为头等关注预训；其他事后扩展。

原生长上下文模型（Llama 3.1 的 128K、Gemini 2.5/3 的 1-2M、Claude 4.5 的 200K）在预训练里贯穿穿插长文档训练数据。模型原生学位置依赖行为；RULER 分数在全上下文范围一致高。

事后扩展模型（社区 LLaMA-2 → 32K 用 YaRN、Qwen2.5 → 128K 用扩展）从短上下文基础开始，应用 YaRN/LongRoPE 加少量长上下文数据继续预训。RULER 分数在中等扩展（2-4 倍）下不错，激进的（16 倍+）退化。

成本差是真的。从头预训 128K 比 4K 预训每 token 大致贵 2-4 倍（长序列下注意力主导计算，即使 FlashAttention；KV cache 内存限制 batch 大小）。事后扩展本质免费（几千微调步）。如果你是基础实验室在一次训练上花 1 亿美元，原生长上下文的 2-4 倍溢价说得通。如果你是发布开源权重模型的小实验室，事后扩展是唯一可行路径。

RAG vs 长上下文：真正的取舍

最常见的应用问题：“文档塞长上下文还是 RAG？” 2026 年的答案比两年前更细。

塞上下文如果：

  • 文档 < 128K token、放得进你的模型。
  • 检索质量难做（query 流动、没有好的切分）。
  • 延迟预算容忍 5-30s 的 prefill。
  • 成本预算容忍每查询约 0.30 美元（200K token prompt @ $1.50/Mtok）。

走 RAG（第 8 篇）如果：

  • 文档库远大于上下文窗口。
  • 延迟必须 <1s。
  • 单查询成本必须是分级。
  • 需要溯源。

长上下文在需要整体阅读的任务上闪光——一个仓库的代码 review、长会议总结、对的 chunk 不可预测的多文档 QA。其他场景 RAG 能在成本上甩出 1-2 个数量级。

混合方法（RAG 找候选、长上下文综合）是多数非平凡负载的生产甜点。Anthropic 的 “Contextual Retrieval”（2024）和微软的 GraphRAG（2024）都是这个混合主题的变种；第 8 篇细讲。

生产小贴士

  • 永远在自己负载上做 needle 测试 再部署。厂商数字是最好情况。
  • 重复服务时给长输入开 prompt cache。100K token 系统 prompt prefill 一次 1.50 美元，cache 后 0.05 美元（vLLM enable_prefix_caching、OpenAI/Anthropic 的 prompt caching API）。
  • 问题放在长上下文的末尾，不是开头。Lost-in-the-middle（Liu 等，2023）显示注意力在长上下文上不成比例地偏向开头和结尾。问题在末尾能高 5-15 % 准确率。
  • 位置 5 万之后的算术不可信。即便 Claude-4.5，输入数据在远处时多步数学错得更多。能挪就把计算上下文挪到问题附近。
  • 自己造长上下文模型就用滑窗 + sinks。成本-质量取舍无敌。
  • 盯着 prefill 延迟曲线。 TTFT 在单 GPU 上随 prompt 长度大致线性，但带 TP 更陡（all-reduce 主导）、跨节点更陡。TP=2 H100 上 200K token prompt prefill 6-8 秒；用户能感觉到。
  • 在自己领域 benchmark。 RULER 上 90 分的模型可能在你的医疗记录任务上 70 分，因为医疗文本分布跟 RULER 合成 needle 不同。领域专项 eval 是唯一诚实信号。

小结与下一篇

RoPE 让长上下文变得可行；YaRN 把它扩到训练长度之外；sinks 让流式稳住；但工作上下文永远小于标称上下文。在自己的负载上测。多数生产任务上 RAG 在成本上赢。长上下文赢在整体阅读和短时长交互工作流。
