大模型工程（三）：规模化预训练

数据混合、去重、benchmark 污染、μP，FSDP / ZeRO-3 / Pipeline 并行，实战意义上的 200B token 悬崖，以及 1000 卡以上才会出现的失败模式。

预训练是 LLM 大部分能力的来源，也是 leaderboard 跟现实差距最大的环节。绝大多数公开的训练 run 是工程奇观多于科学结果。这一篇讲的不是 OpenAI 那种规模上你才在乎的事，而是你不是 OpenAI 时真正必须搞对的几件事：数据、并行选型、以及只有集群大到一次糟糕的 NCCL all-reduce 能干掉一次 30 天 run 时才出现的失败模式。

数据混合比架构更重要

过去三年，所有靠谱的 scaling 研究都得出了一个共识：在相同算力下，两种 LLaMA 风格架构之间的差异很小（约 5% perplexity），但两种数据混合之间的差异却非常大（超过 30%）。Chinchilla 论文提出的 compute-optimal scaling law 假设了固定的数据分布。一旦允许数据分布变化，数据就成了决定性因素。

现代预训练数据混合大致如下（FineWeb-Edu [Penedo et al., 2024]、RedPajama-V2 [Together AI, 2024]、Dolma [Soldaini et al., 2024] 这些公开数据集之间的差异都在几个百分点内）：

来源占比备注
过滤后的网页（CommonCrawl）50-65%数据量最大
代码（GitHub、StackExchange）8-15%不仅提升代码能力，还增强推理能力
书籍5-10%提供长上下文连贯性
Wikipedia2-5%每个 token 的价值远高于其他来源
学术 / arXiv3-5%包含数学公式和引用
数学（证明、教材）2-4%专为推理任务优化
多语言网页5-15%各语言质量差异较大

一个关键数字很少有人提及：去重率。CommonCrawl 2024 在文档级别去重后，只保留了约 25% 的原始字节；如果按行级别去重（更激进的方式），则只剩下 12%。Lee 等人在 2022 年的论文中指出，尽管激进去重会移除 75% 的数据，但它反而能降低 perplexity。重复数据对语言模型来说是毒药——它会让模型学会死记硬背，而不是泛化。

DeepSeek 的预训练报告（DeepSeek-V3 技术报告 [DeepSeek-AI, 2024]，2024 年 12 月发布）是我见过最坦诚的。他们明确列出了数据比例：14.8T token 中，87% 是代码、数学和网页数据，13% 是“高质量书籍和合成数据”。合成数据的作用被大大低估了，实际上它的贡献远超人们的认知。

DataComp-LM：数据质量大于数量，证据确凿

目前公开研究中，对数据质量最严谨的分析来自 DataComp for Language Models 基准测试 [Li et al., 2024]。这项研究固定了模型架构和训练算力，然后通过 416 组对照实验，仅调整数据组合。以下是几个关键发现：

  • 质量过滤是核心。 使用 FastText 训练的质量分类器（用于判断文本是否接近“高质量”参考数据）在相同算力下，比基线模型 MMLU 提高了 6.6 个百分点，Core（一个包含 22 项任务的基准测试）提高了 3.5 个百分点。
  • 激进过滤效果更好。 只保留质量分类器评分最高的前 10% 文档，效果优于保留前 50%，尽管前者丢弃了 5 倍的 token。
  • 去重与过滤相辅相成。 使用 MinHash-LSH 在 0.7 Jaccard 阈值下去重，MMLU 提升了 2.1；结合质量过滤后，提升幅度达到 8.7。两者的效果是超叠加的——去重移除了低质量内容的近似副本，而这些副本可能被质量过滤漏掉。
  • 最佳数据组合因任务而异。 针对代码密集型下游任务训练的模型，偏好不同的数据组合，而通用聊天模型则不然。不存在普适的最优解。

在 DCLM-Baseline 数据集（3T token）上训练的 7B 模型，性能超越了 Llama 2 7B（使用了 2T token 的未明确说明且过滤不那么严格的语料），MMLU 提高了 11.5 分，而且每参数算力还少了 33%。这就是数据质量带来的巨大影响。

合成数据：不能说的秘密

2023 年之前，大家都认为“合成数据就是污染，千万别用”。但 Phi-1 [Gunasekar et al., 2023]（Microsoft，2023）改变了这个看法。它证明了一个 1.3B 参数的模型，如果完全用 GPT-4 生成的教科书风格合成数据训练，性能可以媲美更大的代码模型。Phi 系列继续沿着这条路走，其他公司也悄悄跟上了。

到了 2026 年，顶级开源模型全都依赖合成数据：

  • Qwen3 技术报告：合成指令数据、合成代码、“多样化”的合成问答。
  • Llama 3.1 [Dubey et al., 2024]：代码、数学、多语言任务都用合成数据。
  • DeepSeek-V3：合成“高质量书籍”（比直接爬取数据更合法）。

主要风险是 mode collapse——如果只用一个 teacher 模型生成数据，分布会变得过于集中。解决办法是多 teacher 模型生成，并且进行严格的过滤。另一个风险是 评测集污染。比如 Phi-3 就被发现训练数据里包含了 MMLU 的题目。标准应对方法是精确匹配去重（去掉与评测集有 13-gram 重叠的数据），同时保留一些全新的评测基准。

[Shumailov et al., 2024] 提出了一个更隐蔽的风险：model collapse。如果一直用前一代模型生成的数据训练，数据分布会逐渐变窄，最终导致模型质量下降。他们的实验表明，即使使用高质量的模型输出递归训练，5 到 10 代之内也会收敛到退化的分布。实际生产中的缓解措施是，始终将合成数据和大量人类编写的数据混合使用（通常 50%-70% 是人类数据，30%-50% 是合成数据），并且从多个独立的 teacher 模型生成数据，而不是依赖单一的强模型。

经济成本也很关键。用 GPT-4o 生成 10 亿个高质量合成 token 大约需要 $30K。但如果用更小的、专门为合成任务微调的模型（比如在自托管集群上运行的 Qwen3-32B-Instruct），成本可以降到几百美元。2026 年，大多数生产环境中的合成数据 pipeline 都采用混合策略：先用强大的前沿模型（如 Claude 或 GPT-4o）生成多样化的“种子”数据，再用自托管的中型模型进行规模化扩展。

Scaling law：Chinchilla，然后是 200B

Chinchilla law [Hoffmann et al., 2022] 提出，计算最优的训练大约需要 每参数 20 个 token。7B 的模型需要 140B token，70B 的模型则需要 1.4T。

更早的 [Kaplan et al., 2020] 提供了不同的比例（约每参数 1.7 个 token），因为他们调整算力时更多改变的是模型大小，而不是数据量。Hoffmann 等人通过同时调整数据量和模型大小修正了 Kaplan 的方法。Chinchilla law 成为了 2022-2023 年前沿实验的默认规划参考。

但实际应用中，Chinchilla 并不完全正确。准确地说，它适用于 训练算力最优 的场景，但在 推理算力最优 的情况下——也就是模型会被反复调用几十亿次的场景——应该大幅过度训练小模型。[Sardana et al., 2023] 将这一点形式化为 “Beyond Chinchilla-Optimal: Accounting for Inference in Language Model Scaling Laws”。LLaMA-3 8B 训练用了 15T token（约每参数 1900 个 token），是 Chinchilla 推荐值的 95 倍。这个模型不仅远超 Chinchilla 最优的 8B 模型，部署成本也比 Chinchilla 最优的 70B 模型低得多。

一个关键问题在于数据天花板：大多数开源数据源去重后，高质量 token 总量在 200B 左右就到顶了。再往后，要么重复使用相同数据（多轮训练），要么生成合成数据。多轮训练在高质量数据上最多撑到 4 轮，再多就会导致过拟合 [Muennighoff et al., 2023]。之后，合成数据成了唯一的出路——这也是为什么 2024 年后的前沿模型都开始自己生成数据。

目前对自然语言数据天花板的最佳公开估计来自 [Villalobos et al., 2024]：公开互联网上大约有 300T 高质量英文文本。前沿实验室已经对这些数据进行了激进的处理和过滤。超过 50T 高质量 token 后（Llama 3 和 DeepSeek-V3 大概在这个范围），合成数据和私有数据源成了进一步扩展的唯一选择。这也是 “scaling 预训练” 这一策略在 2026 年逐渐趋于平缓的原因之一——我们正在逼近数据墙。

Curriculum：从低质量到高质量的退火策略

2024-2025 年的一项重要发现表明，训练效率更高时，数据不能简单地均匀打乱，而是需要设计一个合理的呈现顺序。DeepSeek-V3 和其他几个近期的训练都采用了类似的基本模式：

  1. 主体阶段（前约 80% 的 token）：数据广泛混合，网页内容占大头，质量门槛相对较低。
  2. 退火阶段（后约 20% 的 token）：高质量数据混合——包括书籍、论文、数学内容、精选代码和指令调优数据。
  3. 学习率在两个阶段中按照余弦曲线衰减。

背后的逻辑很简单：早期训练重点是建立广泛的语言能力，后期则通过高质量数据将模型调整到目标分布。在低学习率下使用高质量数据进行退火，比全程均匀混入高质量数据，在提升下游基准测试表现时更加高效。

Llama 3 [Dubey et al., 2024] 提到，他们在训练末尾单独增加了一个“退火阶段”，用经过严格过滤的 40B 高质量 token 数据集，以峰值学习率的 1/10 继续训练。虽然这个阶段只占总训练 token 的一小部分，却为 MMLU 指标贡献了好几个点的提升。

长上下文预训练：分阶段序列长度方案

从零开始训练一个 128K 上下文的模型非常浪费。早期预训练的主要收益来自短序列数据，而长序列注意力的计算成本是平方级增长的。现代长上下文模型采用分阶段调整序列长度的方法：

  • 第一阶段（0-90% 的 token）：4K 上下文。主要用来构建通用能力。
  • 第二阶段（90-95%）：32K 上下文。引入长程依赖模式。
  • 第三阶段（95-100%）：128K 上下文，使用精心挑选的长文档数据（如书籍、代码仓库、长篇论文）。

Llama 3 使用了四阶段课程逐步扩展到 128K。NVIDIA 的 Nemotron-340B 也采用了类似的分阶段方法。Qwen3 则通过分阶段预训练达到 128K，并在最后用 YaRN 扩展到 256K。

核心思路是，模型的通用语言建模能力在短上下文阶段就已经建立好了，长上下文阶段只需要教会模型如何利用扩展的注意力机制。这比全程使用长序列训练要便宜得多。短序列阶段可以通过“序列打包”——将多个短文档拼接成一个长序列并配合注意力掩码——几乎免费地提高计算资源利用率。

μP：小规模调参，大规模通用

大规模训练有个让人头疼的问题：在 1B 参数上调好的超参数，搬到 70B 模型上就不好使了。尤其是学习率。标准参数化下，激活值的幅值会随着模型宽度变化，导致规模一变就崩。

μP（Maximal Update Parameterization） 解决了这个问题。这是 [Yang et al., 2022] 提出的方法。用 μP，初始化方差和学习率会根据层宽度调整，确保激活值和梯度的幅值在不同模型宽度下保持一致。最终效果是：在 100M 参数上调好的学习率，可以直接用到 100B 参数的模型上。

# 标准做法：单层学习率不依赖宽度
# μP：隐藏 Linear 层学习率按 1 / fan_in_ratio 缩放
def mup_lr(base_lr, fan_in, base_fan_in=256):
    return base_lr * (base_fan_in / fan_in)

Cerebras 和 Microsoft Research 都做过实验，用 μP 调参的 7B、70B 和 700B 模型，在相同训练进度下，loss 曲线几乎完全对齐，误差在 1% 以内。如果没有 μP，每个规模都得花几百万美元去找合适的学习率。

完整的 μP 方法需要对每层宽度调整四样东西：

  1. 初始化标准差：$\sigma \propto 1/\sqrt{\text{fan}_{\text{in}}}$
  2. 隐藏层学习率：$\eta \propto 1/\text{fan}_{\text{in}}$
  3. 输出层乘子：$\text{logits} = (1/d) \cdot W_{\text{out}} \cdot h$，而不是 $W_{\text{out}} \cdot h$
  4. 嵌入层学习率：保持不变（不缩放）

到了 2026 年，大多数生产环境至少会用 μP 来迁移学习率。有些人更进一步，用 μTransfer 处理 batch size 和 weight decay。[Wortsman et al., 2024] 还发现，μP 让训练对噪声更鲁棒，也更容易从不稳定中恢复。

并行：FSDP、ZeRO、Pipeline、Tensor

模型超出单卡内存后，有四个独立的并行方向：

  1. 数据并行（DP）——每张 GPU 上复制一份模型，把 batch 分开处理。
  2. 分片数据并行（FSDP / ZeRO-3） [Rajbhandari et al., 2020]——参数分散到不同的 DP 节点，在每层前向计算之前按需聚合。
  3. 张量并行（TP） [Shoeybi et al., 2019]——将矩阵乘法拆分到多张 GPU 上（类似 Megatron 的做法）。
  4. 流水线并行（PP） [Huang et al., 2019]——把层分布到不同 GPU 上，用 micro-batch 流水执行。

对于一个 70B 参数的模型，使用 64 张 H100 的典型配置是：

TP=8 节点内（NVLink）
PP=2 跨节点（200 Gbps Infiniband）
DP=4 剩余维度用 FSDP

总共 64 张 GPU（8 × 2 × 4）。通信量最大的张量并行（TP）限制在节点内的 NVLink 上，而流水线并行（PP）的空闲时间被下一个 micro-batch 覆盖。到 2026 年，ZeRO-3（FSDP 的一种实现）会成为默认选择，因为它无需修改代码就能支持 70B 模型：

mixed_precision 部分非常重要。梯度规约一定要用 fp32——bf16 在跨 DP 节点求和上千个小梯度时会悄无声息地溢出。我曾经调试过 loss 发散的问题，最后发现就是漏写了 reduce_dtype=fp32。

对于超大模型（比如 DeepSeek-V3 671B 和 Qwen3-Max），即使 FSDP 也不够用了，必须用四维并行加自定义代码。目前主流的生产框架有四个：Megatron-LM、NVIDIA NeMo、ColossalAI 和 DeepSpeed。大多数实验室都会基于其中一个框架拉出自己的分支来开发。

实战案例：70B 模型在 64 张 H100 上的并行策略选择

为什么 TP=8、PP=2、DP=4 是这个配置的最佳选择？每张 H100 显卡有 80 GB HBM，总共是 64 × 80 = 5120 GB。

每张显卡的内存分配。 每个 rank 需要：

  • 权重：70B × 2 字节（BF16）÷ TP ÷ PP = 70e9 × 2 ÷ 16 = 8.75 GB
  • 梯度：和权重一样，也是 8.75 GB
  • Adam 优化器状态：4 字节/参数 × 2（m、v）÷ TP ÷ PP ÷ DP_FSDP = 70e9 × 8 ÷ 64 = 8.75 GB（如果 FSDP 切分优化器）
  • 激活值（带梯度 checkpointing）：大小随 batch × seqlen × model dim 变化，70B 模型在 seqlen 8K 下通常需要 30-50 GB
  • KV/scratch 缓冲区：大约 5 GB

总计：约 62 GB，占 80 GB 的大部分，剩下约 18 GB 可以留给 FlashAttention 的临时缓冲区、NCCL 缓冲区以及激活峰值。虽然紧张，但能跑起来。

为什么不直接用纯 FSDP=64？ 如果在 64 张显卡上完全使用 FSDP，每次前向传播之前都需要 all-gather 70B 参数。70B × 2 字节 = 140 GB 数据，分摊到 64 个 rank 上，每个 rank 每次 gather 需要处理 2.2 GB 数据。模型有 80 层，这意味着每次前向传播每个 rank 要处理约 175 GB 的 gather 流量。即使 NVLink 带宽达到 900 GB/s，这也会带来约 200 毫秒的纯通信时间——几乎和计算时间持平，严重拖累吞吐量。

为什么不直接用纯 TP=64？ TP 在每次矩阵乘法后都需要通信（行并行层上的 all-reduce）。每 token 的 all-reduce 数据量是 $O(d_{\text{model}} \times \text{TP})$。当 TP=64 跨节点运行时（通过 200 Gbps Infiniband），通信开销会成为瓶颈。TP 只有在单个 NVLink 互联节点内（当前最多支持 8 张显卡）才能高效运行。

为什么选择 TP=8、PP=2、DP=4？ TP=8 在每个节点内部利用高速 NVLink 处理 all-reduce 密集的通信流量。PP=2 将模型切分到两个节点，PP 通信频率较低（每个 micro-batch 一次）。DP=4 配合 FSDP 将优化器状态切分到 4 个 DP rank，每个 rank 存储 1/4 的权重和 1/(TP×PP) 的层数。PP 的流水线气泡（占步长时间的 10-20%）可以接受；其他并行策略的开销更大。

这只是其中一个配置。不同模型规模和集群拓扑会有不同的最优解。通用原则是：尽量减少跨节点通信，将 TP 限制在最快的互联范围内，用 PP 提供下一层粒度，剩下的并行性用 FSDP 或 DP 填补。

1000 卡以上才出现的失败模式

8 卡上无所谓，但到了 8000 卡就能搞垮整个训练任务：

静默数据损坏。 某张卡因为宇宙射线或者电压波动返回错误的计算结果，产生一个 NaN，随后被 all-reduce 扩散出去。解决办法是定期对梯度做校验和检查——如果 rank 0 的梯度范数和 rank 100 的差距超过 $10^{-3}$，就立即停止并重新分片。

Loss 突增。 一个坏 batch（比如 Python 解析器输出的乱码、HTML 垃圾数据等）会导致梯度突增，把优化器的动量推偏，再也回不来。应对方法是：激进的梯度裁剪（默认值 1.0，超大模型有时降到 0.3），再加上一个“跳过这个 batch”的触发器，当梯度范数超过跑动中位数的 5 倍时触发。Llama 3 的论文提到，“checkpoint 回滚”是标准操作流程：如果 loss 突增在 50 步内无法恢复，就回滚到上一个正常的 checkpoint，学习率降低 30%，然后继续训练。

NCCL 挂起。 某个节点的网卡固件 bug 会让该 rank 的 all-reduce 操作永远不返回，其他 999 个 rank 也会因此无限阻塞。防御措施是设置 NCCL_TIMEOUT_S=300，并加一个看门狗来重新调度任务。

Checkpoint 损坏。 一个 2.7 TB 的 FSDP checkpoint（比如 Qwen3-32B 的优化器状态）写入需要 20 多分钟。如果某个节点在写入中途 OOM，不仅会丢掉当前的 step，还会覆盖掉上一个正常的 checkpoint。解决办法是：先写到 step_xxx.tmp/ 目录，fsync 后再用 mv 命令移动到目标位置。

异构硬件。 即使是“同型号”的 H100，不同批次的时钟速度也可能有 2-3% 的差异。慢的 rank 会拖累整体性能。解决办法是：在 warmup 阶段按每个 rank 的吞吐量排序，把慢的 rank 放到 TP=1 的位置。

嵌入坍缩。 如果嵌入矩阵的梯度流在早期 epoch 被裁剪或者被 NaN 损坏，嵌入可能会坍缩成几乎相同的向量。症状是：训练 loss 卡在 log(V)，困惑度停留在 V。诊断方法是监控嵌入矩阵的奇异值；如果最大奇异值比次大的高出 10 倍以上，说明嵌入已经坍缩。修复方法是：回退到坍缩之前的状态，把嵌入的学习率提高 2 倍，然后重启训练。

梯度范数发散。 这和 loss 突增不同。即使 loss 稳定，梯度范数也可能在几千步内缓慢上升，这通常预示着未来会发生崩溃。监控时要注意每个参数组的 ||g||_2，而不仅仅是全局梯度范数。具体来说，LM head 或者最后一个 transformer 层往往是罪魁祸首。

Meta OPT-175B 的运行日志 [Zhang et al., 2022] 至今仍然是描述真实大规模训练的最佳公开文档。他们在 33 天的训练过程中，平均每 1.7 天就会遇到一次节点故障。

一次运行中到底发生了什么

一个现代的 70B 参数预训练任务（LLaMA-3 70B，15T tokens，约 1900 张 H100，耗时 60 天）大致消耗如下资源：

  • Tokens：输入 15T tokens，batch size 约为 16M tokens
  • Steps：总步数 15T / 16M ≈ 940K
  • 每步耗时：约 5.5 秒（使用 sequence packing 技术）
  • 总算力：约 $2.5 \times 10^{25}$ FLOPs，相当于 6000 张 H100 运行一年

学习率（LR）调度采用余弦衰减（cosine decay），从短暂的 warmup 开始，最终衰减到峰值的 10% 左右。对于 70B 模型，峰值学习率约为 1.5e-4（更大模型的学习率更低，如果严格按照 μP 缩放规则会更严谨）。优化器选用 AdamW，参数设置为 weight decay 0.1，β2=0.95。

评估按 checkpoint 进行：每 2000 步，在 5 到 10 个小型基准测试集上跑一遍，包括 Hellaswag、ARC-easy、OpenBookQA、GSM8K-100 和 HumanEval-50。跟踪 loss 成本较低，但下游任务的评估结果才是真正的信号。

生产真相：前沿实验室真正在做的事

论文讲的是架构，但生产系统更多是后勤保障。每个前沿预训练团队都会做三件事，这些事不会写进模型卡里：

多规模并行训练。 在跑 70B 参数的目标模型之前，先用 1/100 和 1/10 的数据量跑 1B 和 7B 的“侦察”模型，架构、优化器和数据混合方式完全一致。侦察模型用来验证训练配方，调整那些 μP 无法迁移的超参数（主要是学习率调度等细节）。侦察模型跑得顺利，70B 模型就可以放心启动。Llama 3 的论文提到，他们同时跑了 405B 和 70B 模型，持续验证扩展行为。

全面监控。 包括损失曲线、梯度范数、单个参数统计、注意力模式直方图、专家利用率（MoE）、每层激活值大小、评估分数等等。监控仪表盘不是可选的基础设施，而是提前发现问题的唯一手段。Anthropic 在模型卡中开源了部分监控工具（比如裁剪比和激活统计）；OpenAI 则在 GPT-4 论文中详细描述了他们的做法。

数据持续优化。 训练数据不是一成不变的。团队会不断运行新的质量分类器，移除新发现的污染数据，加入新的合成数据源。Llama 3 明确提到，他们在训练中途用重新训练的质量分类器对整个数据集进行过滤，并根据结果重新加权后续批次。学术界常说“训练前数据固定”，但在实际生产中，团队会不断迭代。

实时分词 vs 预分词。 预分词 15T 的 token 需要几周时间和超过 100TB 的存储空间。而训练时流式分词要求分词器跟上数据加载的速度，这意味着需要高度优化的 Rust 分词器。大多数生产环境会选择预先计算 token ID，但从分布式文件系统中流式读取，而不是直接加载原始文本。

常见踩坑

我亲眼见过 5 个搞砸预训练任务的错误。

1. 忘了设置 NCCL collectives 的 bf16/fp32 不匹配问题。 默认情况下，NCCL all-reduce 在 fp16 模式下对上千个小梯度值求和时会悄悄下溢。解决办法很简单：用 NCCL_PROTO=Simple 确保稳定性，同时为 FSDP 设置 reduce_dtype=fp32。Llama 3 的论文提到，他们在 step 500K 附近遇到不稳定问题，后来通过这个方法才得以恢复。

2. RoPE 基础频率与上下文长度不匹配。 RoPE 使用的基础频率是 $\theta = 10000$。在 32K 上下文长度下，频率会发生混叠，模型无法学习长程依赖关系。解决方法有两种：一是按照 [bloc97 NTK-aware] 的建议，将基础频率调整为 $10000 \times (\text{seqlen}/2048)^{d/(d-2)}$；二是直接使用 YaRN [Peng et al., 2024]。我们曾经花了一周时间排查一个 32K 上下文质量下降的问题，最后才发现 RoPE 基础频率还停留在 4K 预训练阶段的设置。

3. 打包序列时未正确处理 loss masking。 Sequence packing 把多个文档拼接成一个训练序列。如果不屏蔽跨文档的注意力，模型会学到虚假的跨文档关联，导致 MMLU 指标下降 1-3%。正确的做法是生成 position_ids 和 attention_mask，并在文档边界处重置它们。

4. 激活 checkpointing 中使用了非确定性操作。 在 checkpoint-recompute 阶段重新运行前向传播时，必须确保内核是确定性的。一些 FlashAttention 和 GeLU 的实现中存在非确定性的数值缩减操作。这会导致计算出的梯度与保存的前向值不一致，从而引发微妙的质量退化，这种问题通常要几千步后才会显现。解决方法是，在 checkpoint 区域设置 torch.use_deterministic_algorithms(True)，或者手动验证内核是否确定性。

5. Optimizer 状态未分片存储到磁盘。 对于 70B 参数的模型，AdamW 优化器的状态占用 70B × 8 字节 = 560 GB 的空间。如果上千个节点同时写入单个共享文件系统，很容易发生死锁。推荐使用 FSDP 的分片 checkpointing 功能（每个 rank 只写自己的分片），并启用并行写入。我们曾经因为不小心串行化存储，导致一个 checkpoint 花了整整 4 小时才保存完成。

研究前沿 2024-2026

当前 Llama 风格的 “过滤 + 去重 + 训练” 方法之后，接下来会有什么新突破？

大规模数据效率。 [Marion et al., 2023]（When Less is More）证明，用一个精心挑选的 200B token 子集训练，效果可以媲美使用 1T token 未过滤数据集。这其实就是 “数据的提前停止” —— 找到最有价值的样本，跳过其他部分。

Curriculum 驱动的数据混合。 [Albalak et al., 2024] 发现，动态调整数据源比例，根据模型在训练中的改进情况实时优化，比固定比例的方法高出 1-2 MMLU。Doremi [Xie et al., 2023] 提出了一种 min-max 优化方法，自动寻找最优权重。

超越 Chinchilla 的算力最优解。 Mixture-of-Experts 改变了 scaling law。[DeepSeek-AI, 2024] 报告了基于 V3 实验得出的 MoE scaling law：最优激活/总参数比、最优专家数量、最优专家规模，都以复杂的方式依赖于训练算力。社区仍在探索 MoE 特有的 scaling law。

持续预训练。 大多数生产模型现在采用更新的方式，而不是从头重新训练。比如，Llama 3.1 就是在 Llama 3 的基础上，通过额外数据进行持续预训练，再结合 post-training 得到的。持续预训练成本低得多，但需要仔细设计学习率调度，避免灾难性遗忘。

小结与下一篇

预训练，七成是数据工程，三成是分布式系统工程。架构选型反而是最次要的。先把数据混合调好，彻底去重，训练时长要超过 Chinchilla，这是为了降低推理成本。用 μP 确保超参数能迁移，根据硬件拓扑选择 FSDP 加上 TP 和 PP，还要为大规模场景下特有的故障模式写好防御代码。

下一篇的主题是 Post-training。SFT、DPO、RLHF、RLAIF——我会讲清楚每种方法到底优化了什么，奖励模型会在哪些情况下失效（它们经常出问题），LoRA 和全量微调的争论，以及如何通过生产流程把基础模型变成客户真正能用的产品。
