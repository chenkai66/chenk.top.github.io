大模型工程（三）：规模化预训练

数据混合、去重、benchmark 污染、μP，FSDP / ZeRO-3 / Pipeline 并行，实战意义上的 200B token 悬崖，以及 1000 卡以上才会出现的失败模式。

预训练是 LLM 大部分能力的来源，也是 leaderboard 跟现实差距最大的环节。绝大多数公开的训练 run 是工程奇观多于科学结果。这一篇讲的不是 OpenAI 那种规模上你才在乎的事，而是你不是 OpenAI 时真正必须搞对的几件事：数据、并行选型、以及只有集群大到一次糟糕的 NCCL all-reduce 能干掉一次 30 天 run 时才出现的失败模式。

数据混合比架构重要

过去三年所有靠谱的 scaling 研究结论一致：同样算力下，两个 LLaMA 风格架构之间的差距很小（约 5 % perplexity），两个数据混合之间的差距很大（>30 %）。Chinchilla 论文的 compute-optimal scaling law 假设了固定数据分布；一旦让数据也变，数据完全主导。

现代预训练混合大致这样（FineWeb-Edu、RedPajama-V2、Dolma 之间相差几个点）：

来源占比备注
过滤后的网页（CommonCrawl）50-65 %体量主力
代码（GitHub、StackExchange）8-15 %不只提升代码，也帮推理
书籍5-10 %长上下文连贯性
Wikipedia2-5 %单 token 价值高得不成比例
学术 / arXiv3-5 %数学、引用
数学（证明、教材）2-4 %推理专用
多语言网页5-15 %各语言质量参差

最重要的、没人提的数字是 去重率。CommonCrawl 2024 在文档级去重后剩约 25 % 字节，行级（更激进）约 12 %。Lee 等 2022 年的论文显示，激进去重 降低 perplexity，即使去掉 75 % 的数据也是。重复对 LM 是毒——它会教模型背而不是泛化。

DeepSeek 的预训练 writeup（DeepSeek-V3 技术报告，2024 年 12 月）是我见过最坦诚的：14.8T token，87 % 代码+数学+网页，13 % “高质量书籍和合成数据”。这个合成数据比例做了比大家承认的多得多的工作。

合成数据：脏秘密

2023 年之前的共识是 “合成数据=污染，永远别用”。Phi-1（Microsoft，2023）证明 1.3B 模型在纯 GPT-4 生成的教科书风格合成数据上能追上大得多的代码模型，这事变了。Phi 系列一路这么走，其他人悄悄跟上。

2026 年顶级开源模型全用合成数据：

  • Qwen3 技术报告：合成 instruction、合成代码、“多样化” 合成 Q&A。
  • Llama 3.1：代码、数学、多语言用合成数据。
  • DeepSeek-V3：合成 “高质量书籍”（比爬要在法律上干净）。

风险是 mode collapse——从单个 teacher 合成会让分布坍缩。防御是多 teacher 合成 + 激进过滤。另一个风险是 评测污染。Phi-3 被发现训练集里有 MMLU 题。标准对策是精确匹配去污（任何 13-gram 与评测集重叠就丢）+ 留一手新评测。

Scaling law：Chinchilla，然后是 200B

Chinchilla law（Hoffmann 等，2022）说 compute-optimal 训练大约 每参数 20 token。7B 想要 140B token，70B 想要 1.4T。

实战中 Chinchilla 不对。或者说，它对的是 训练算力最优，但 推理算力最优——模型会被用几十亿次的场景——你应当 大幅度过训 小模型。LLaMA-3 8B 训了 15T token（每参数约 1900 token），是 Chinchilla 的 95 倍。它远比 Chinchilla 最优的 8B 强，部署起来又远比 Chinchilla 最优的 70B 便宜。

那道悬崖：大多数开源数据源去重后在 200B 唯一 token 附近见顶。再往后要么重训同样数据（多 epoch），要么生成合成数据。多 epoch 训练在高质量数据上撑到 4 epoch 还行，再多就开始过拟合。再之后，合成数据是唯一前进方向——这就是 2024 年后所有前沿模型都在自己造数据的原因。

μP：让你能在小规模调超参

大规模训练的实战头疼事：1B 上调好的超参不能搬到 70B。学习率尤其是。标准参数化下激活幅值随宽度变化，规模一变就崩。

μP（Maximal Update Parameterization），Yang 等 2022，修这个问题。μP 下，按层宽度缩放初始化方差和学习率，让激活和梯度的幅值在不同模型宽度下保持一致。结果：在 100M 参数上调好 LR，搬到 100B 参数照样用。

# 标准做法：单层 LR 不依赖宽度
# μP：隐藏 Linear 层 LR 按 1 / fan_in_ratio 缩放
def mup_lr(base_lr, fan_in, base_fan_in=256):
    return base_lr * (base_fan_in / fan_in)

Cerebras 和 Microsoft Research 都做过 μP 调过的 7B → 70B → 700B 模型在同样训练比例下 loss 曲线对齐到 1 % 以内的实验。没有 μP，你要在每个规模上烧几百万美元找 LR。

2026 年大多数生产 run 至少用 μP 做 LR 迁移。有人走得更远，用 μTransfer 处理 batch size 和 weight decay。

并行：FSDP、ZeRO、Pipeline、Tensor

模型超出单卡内存之后，有四个正交并行轴：

  1. 数据并行（DP）——模型在每张卡上复制，batch 切分。
  2. 分片数据并行（FSDP / ZeRO-3）——参数切到 DP rank 上，每层 forward 之前 just-in-time gather。
  3. 张量并行（TP）——单个 matmul 切到多卡（Megatron 风格）。
  4. 流水线并行（PP）——层切到多卡，micro-batch 流过。

70B 模型在 64 张 H100 上的典型配方：

TP=8 节点内（NVLink）
PP=2 跨节点（200 Gbps Infiniband）
DP=4 用 FSDP 在剩下那一维

总共 64 卡（8 × 2 × 4），通信最重的并行（TP）压在节点内 NVLink 上，PP 的 bubble 跟下一个 micro-batch 重叠。ZeRO-3（FSDP 变体）2026 年是默认，因为它能让你 70B 一行代码不改：

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy

model = TransformerModel(...)
model = FSDP(model,
             sharding_strategy=ShardingStrategy.FULL_SHARD,
             mixed_precision=MixedPrecision(
                 param_dtype=torch.bfloat16,
                 reduce_dtype=torch.float32,
             ),
             auto_wrap_policy=transformer_auto_wrap_policy(
                 transformer_layer_cls={LlamaDecoderLayer},
             ))

mixed_precision 块是关键。reduce 永远走 fp32——bf16 在跨 DP rank 把上千个微小梯度求和时会悄悄下溢。我们 debug 过 loss 发散的 run，最后发现就是漏了一个 reduce_dtype=fp32。

巨型模型（DeepSeek-V3 671B、Qwen3-Max）连 FSDP 都不够，要四维并行加自己的代码。Megatron-LM、NVIDIA NeMo、ColossalAI、DeepSpeed 是四个生产框架，大多数实验室都在某个上面拉了自己的 fork。

1000 卡以上才出现的失败模式

8 卡上无所谓、8000 卡上能干掉整个 run 的事：

静默数据损坏。 单卡返回错的数学（宇宙射线、电压抖动）造成 NaN 被 all-reduce 出去。对策：定期对梯度做 checksum——如果 rank 0 的梯度范数和 rank 100 差超过 $10^{-3}$，停下来重新分片。

Loss spike。 一个坏 batch（Python parser dump、HTML soup 之类）产生梯度尖刺，把 optimizer 动量推偏，永不恢复。防御：激进梯度裁剪（默认 1.0，超大模型有时 0.3）+ 当梯度范数 > 跑动中位数 5 倍时 “跳过这个 batch”。

NCCL hang。 单节点的网卡固件 bug 让该 rank 的 all-reduce 永不返回。其他 999 个 rank 永远阻塞。防御：NCCL_TIMEOUT_S=300 + 看门狗重新调度。

Checkpoint 损坏。 2.7 TB 的 FSDP checkpoint（Qwen3-32B 优化器状态）写要 20 多分钟。中途某节点 OOM，你不只丢一个 step，要是原地覆盖还把上一个好 checkpoint 也丢了。永远写 step_xxx.tmp/、fsync，再 mv。

异构硬件。 即便 “同型号” 的 H100，时钟也会抖 2-3 %。慢 rank 拖后腿。防御：warmup 时按 rank throughput 排序，把慢 rank 放到 TP=1 的位置。

Meta OPT-175B 的 logbook（2022）至今是最好的公开 “真实大 run 长什么样” 文档。33 天的 run 里平均 1.7 天就有一次节点故障。

一次 run 里到底发生什么

70B 现代预训练 run（LLaMA-3 70B，15T token，约 1900 张 H100，60 天）大致烧：

  • Token：15T 输入，batch size 约 16M token
  • Step：15T / 16M ≈ 940K
  • 每 step 墙钟：约 5.5 秒（带 sequence packing）
  • 总算力：约 $2.5 \times 10^{25}$ FLOPs ≈ 6000 H100-年

LR schedule 是 cosine decay，从短 warmup 衰到峰值的 10 %。70B 的峰值 LR 约 1.5e-4（更大模型更低，按 μP 缩放如果你严谨的话）。Weight decay 0.1，β2=0.95，AdamW。

评估按 checkpoint 走：每 2000 step 在 5-10 个小评测（Hellaswag、ARC-easy、OpenBookQA、GSM8K-100、HumanEval-50）上跑一遍。Loss 便宜跟，downstream eval 才是真信号。

小结与下一篇

预训练 70 % 是数据工程，30 % 是分布式系统工程。架构选型是这三件里最小的一件。把数据混合做对，狠去重，按推理成本动机过 Chinchilla 训，用 μP 让超参可迁移，按硬件拓扑挑 FSDP+TP+PP，给只在大规模上出现的失败模式写防御代码。

下一篇：Post-training。SFT、DPO、RLHF、RLAIF——每个具体优化的是什么、奖励模型在哪些地方失败（它们一直在失败）、LoRA vs 全量微调那场争论、以及把 base model 变成客户能用的东西的生产配方。
