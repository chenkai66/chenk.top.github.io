大模型工程（三）：规模化预训练

数据混合、去重、benchmark 污染、μP，FSDP / ZeRO-3 / Pipeline 并行，实战意义上的 200B token 悬崖，以及 1000 卡以上才会出现的失败模式。

预训练是 LLM 大部分能力的来源，也是 leaderboard 跟现实差距最大的环节。绝大多数公开的训练 run 是工程奇观多于科学结果。这一篇讲的不是 OpenAI 那种规模上你才在乎的事，而是你不是 OpenAI 时真正必须搞对的几件事：数据、并行选型、以及只有集群大到一次糟糕的 NCCL all-reduce 能干掉一次 30 天 run 时才出现的失败模式。

数据混合比架构重要

过去三年所有靠谱的 scaling 研究结论一致：同样算力下，两个 LLaMA 风格架构之间的差距很小（约 5 % perplexity），两个数据混合之间的差距很大（>30 %）。Chinchilla 论文的 compute-optimal scaling law 假设了固定数据分布；一旦让数据也变，数据完全主导。

现代预训练混合大致这样（FineWeb-Edu [Penedo et al., 2024]、RedPajama-V2 [Together AI, 2024]、Dolma [Soldaini et al., 2024] 之间相差几个点）：

来源占比备注
过滤后的网页（CommonCrawl）50-65 %体量主力
代码（GitHub、StackExchange）8-15 %不只提升代码，也帮推理
书籍5-10 %长上下文连贯性
Wikipedia2-5 %单 token 价值高得不成比例
学术 / arXiv3-5 %数学、引用
数学（证明、教材）2-4 %推理专用
多语言网页5-15 %各语言质量参差

最重要的、没人提的数字是 去重率。CommonCrawl 2024 在文档级去重后剩约 25 % 字节，行级（更激进）约 12 %。Lee 等 2022 年的论文显示，激进去重 降低 perplexity，即使去掉 75 % 的数据也是。重复对 LM 是毒——它会教模型背而不是泛化。

DeepSeek 的预训练 writeup（DeepSeek-V3 技术报告 [DeepSeek-AI, 2024]，2024 年 12 月）是我见过最坦诚的：14.8T token，87 % 代码+数学+网页，13 % “高质量书籍和合成数据”。这个合成数据比例做了比大家承认的多得多的工作。

DataComp-LM：数据质量 > 数量，带数据

公开里关于数据质量最严谨的研究是 DataComp for Language Models benchmark [Li et al., 2024]。固定架构和训练算力，跑了 416 个对照实验，只变数据混合。值得记的发现：

  • 质量过滤主导。 模型化质量分类器（用 FastText 训练成预测文本是否像 “高质量” 参考数据）和 baseline 同算力下，MMLU 高 6.6 个百分点，Core（22 任务 benchmark）高 3.5 个。
  • 激进过滤赢。 按质量分类器分数只留前 10 % 文档，比留前 50 % 强，尽管丢了 5 倍 token。
  • 去重和过滤交互。 MinHash-LSH 去重 0.7 Jaccard 阈值给 +2.1 MMLU；配质量过滤给 +8.7。两个干预是超加法的——dedup 移掉低质量内容的近似副本，质量过滤可能漏掉。
  • 最优混合是任务特定的。 给代码重下游用的模型偏好的混合跟通用聊天的不同。通用最优不存在。

在 DCLM-Baseline 数据（3T token）上训的 7B 模型超过 Llama 2 7B（用了 2T token 的未指定、过滤不那么激进的数据）11.5 MMLU 点，且每参数算力少 33 %。这就是数据质量能造成的效果量级。

合成数据：脏秘密

2023 年之前的共识是 “合成数据=污染，永远别用”。Phi-1 [Gunasekar et al., 2023]（Microsoft，2023）证明 1.3B 模型在纯 GPT-4 生成的教科书风格合成数据上能追上大得多的代码模型，这事变了。Phi 系列一路这么走，其他人悄悄跟上。

2026 年顶级开源模型全用合成数据：

  • Qwen3 技术报告：合成 instruction、合成代码、“多样化” 合成 Q&A。
  • Llama 3.1 [Dubey et al., 2024]：代码、数学、多语言用合成数据。
  • DeepSeek-V3：合成 “高质量书籍”（比爬要在法律上干净）。

风险是 mode collapse——从单个 teacher 合成会让分布坍缩。防御是多 teacher 合成 + 激进过滤。另一个风险是 评测污染。Phi-3 被发现训练集里有 MMLU 题。标准对策是精确匹配去污（任何 13-gram 与评测集重叠就丢）+ 留一手新评测。

[Shumailov et al., 2024] 识别的更微妙风险是 model collapse：在前一代模型采样的数据上训练，会逐步窄化数据分布、最终降质。他们的实验显示在模型输出（即便高质量）上递归训练，5-10 代之内收敛到退化分布。生产里的缓解是合成数据始终跟可观比例的人类原创数据混（一般 50-70 % 人类、30-50 % 合成），且合成数据用多个独立 teacher 而不是单个强模型。

经济账重要。GPT-4o 生成 10 亿高质量合成 token 大约要 $30K。用更小的、为合成专门微调的模型（如自托管的 Qwen3-32B-Instruct 集群）降到几百块。2026 年大多数生产合成数据 pipeline 用混合：强前沿模型（Claude、GPT-4o）做 “种子” 多样化生成，自托管中型模型做规模化。

Scaling law：Chinchilla，然后是 200B

Chinchilla law [Hoffmann et al., 2022] 说 compute-optimal 训练大约 每参数 20 token。7B 想要 140B token，70B 想要 1.4T。

更早的 [Kaplan et al., 2020] 的 scaling law 给出不同比例（每参数约 1.7 token），因为他们改算力时变模型大小多于变数据。Hoffmann 等通过同时扫数据和模型大小修了 Kaplan 的方法。Chinchilla law 成了 2022-2023 年前沿 run 的默认规划启发。

实战中 Chinchilla 不对。或者说，它对的是 训练算力最优，但 推理算力最优——模型会被用几十亿次的场景——你应当 大幅度过训 小模型。[Sardana et al., 2023] 形式化了这个为 “Beyond Chinchilla-Optimal: Accounting for Inference in Language Model Scaling Laws”。LLaMA-3 8B 训了 15T token（每参数约 1900 token），是 Chinchilla 的 95 倍。它远比 Chinchilla 最优的 8B 强，部署起来又远比 Chinchilla 最优的 70B 便宜。

那道悬崖：大多数开源数据源去重后在 200B 唯一 token 附近见顶。再往后要么重训同样数据（多 epoch），要么生成合成数据。多 epoch 训练在高质量数据上撑到 4 epoch 还行，再多就开始过拟合 [Muennighoff et al., 2023]。再之后，合成数据是唯一前进方向——这就是 2024 年后所有前沿模型都在自己造数据的原因。

公开里对自然语言数据天花板最好的估计来自 [Villalobos et al., 2024]：公开互联网上大约有 300T token 的 “高质量” 英文文本。前沿实验室已经激进地处理过滤过这一池。50T 高质量 token 之上（Llama 3 和 DeepSeek-V3 大约这位置），合成数据和私有数据源是规模化的唯一路。这就是 “scaling 预训练” 这个策略 2026 年开始走平的原因之一——我们正接近数据墙。

Curriculum：从低质退火到高质

2024-2025 年一个稳健发现：训练效率高的做法不是均匀洗所有数据，而是结构化呈现顺序。基本模式，DeepSeek-V3 和最近几个 run 都用：

  1. 主体阶段（前约 80 % token）：广泛混合、网页重、质量门槛低些。
  2. 退火阶段（后约 20 % token）：高质量混合——书籍、论文、数学、精修代码、指令调教数据。
  3. 学习率 cosine schedule 贯穿两阶段衰减。

直觉是：早期训练建立广泛语言能力，晚期训练把模型塑形到目标分布。低 LR 下在高质量数据上退火，比把高质量数据均匀混在全程，下游 benchmark 提升的样本效率高得多。

Llama 3 [Dubey et al., 2024] 报告末尾单独跑了一个 “annealing run”，在重过滤的 40B token 高质量混合上以峰值 LR 1/10 继续训。这一阶段尽管只占总训练 token 的零头，贡献了好几个 MMLU 点。

长上下文预训练：分阶段序列长度配方

从零开始训一个 128K 上下文模型很浪费——大部分早期预训练受益于短序列数据，长序列 attention 的 FLOPs 代价是平方的。当代长上下文模型分阶段切序列长度：

  • 阶段 1（0-90 % token）：4K 上下文。建通用能力。
  • 阶段 2（90-95 %）：32K 上下文。引入长程模式。
  • 阶段 3（95-100 %）：128K 上下文，配精修长文档数据（书籍、代码仓、长论文）。

Llama 3 用 4 阶段课程到 128K。NVIDIA Nemotron-340B 用类似分阶段方案。Qwen3 分阶段预训练到 128K，末尾用 YaRN 扩展到 256K。

关键洞察：模型的通用语言建模能力在短上下文建立，长上下文阶段只需要教模型怎么用扩展的 attention。这比全程在长序列上训便宜得多。Sequence packing——把多个短文档拼成单个长序列配 attention mask——在短序列阶段提供几乎免费的算力利用率。

μP：让你能在小规模调超参

大规模训练的实战头疼事：1B 上调好的超参不能搬到 70B。学习率尤其是。标准参数化下激活幅值随宽度变化，规模一变就崩。

μP（Maximal Update Parameterization），[Yang et al., 2022] 修这个问题。μP 下，按层宽度缩放初始化方差和学习率，让激活和梯度的幅值在不同模型宽度下保持一致。结果：在 100M 参数上调好 LR，搬到 100B 参数照样用。

# 标准做法：单层 LR 不依赖宽度
# μP：隐藏 Linear 层 LR 按 1 / fan_in_ratio 缩放
def mup_lr(base_lr, fan_in, base_fan_in=256):
    return base_lr * (base_fan_in / fan_in)

Cerebras 和 Microsoft Research 都做过 μP 调过的 7B → 70B → 700B 模型在同样训练比例下 loss 曲线对齐到 1 % 以内的实验。没有 μP，你要在每个规模上烧几百万美元找 LR。

完整 μP 配方在每个层宽度上不同地缩放四样东西：

  1. 初始化标准差：$\sigma \propto 1/\sqrt{\text{fan}_{\text{in}}}$
  2. 隐藏层学习率：$\eta \propto 1/\text{fan}_{\text{in}}$
  3. 输出层乘子：$\text{logits} = (1/d) \cdot W_{\text{out}} \cdot h$ 而不是 $W_{\text{out}} \cdot h$
  4. 嵌入层 LR：保持常数（不缩放）

2026 年大多数生产 run 至少用 μP 做 LR 迁移。有人走得更远，用 μTransfer 处理 batch size 和 weight decay。[Wortsman et al., 2024] 显示 μP 也让训练对噪声更鲁棒、更容易从不稳定中恢复。

并行：FSDP、ZeRO、Pipeline、Tensor

模型超出单卡内存之后，有四个正交并行轴：

  1. 数据并行（DP）——模型在每张卡上复制，batch 切分。
  2. 分片数据并行（FSDP / ZeRO-3） [Rajbhandari et al., 2020]——参数切到 DP rank 上，每层 forward 之前 just-in-time gather。
  3. 张量并行（TP） [Shoeybi et al., 2019]——单个 matmul 切到多卡（Megatron 风格）。
  4. 流水线并行（PP） [Huang et al., 2019]——层切到多卡，micro-batch 流过。

70B 模型在 64 张 H100 上的典型配方：

TP=8 节点内（NVLink）
PP=2 跨节点（200 Gbps Infiniband）
DP=4 用 FSDP 在剩下那一维

总共 64 卡（8 × 2 × 4），通信最重的并行（TP）压在节点内 NVLink 上，PP 的 bubble 跟下一个 micro-batch 重叠。ZeRO-3（FSDP 变体）2026 年是默认，因为它能让你 70B 一行代码不改：

from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy

model = TransformerModel(...)
model = FSDP(model,
             sharding_strategy=ShardingStrategy.FULL_SHARD,
             mixed_precision=MixedPrecision(
                 param_dtype=torch.bfloat16,
                 reduce_dtype=torch.float32,
             ),
             auto_wrap_policy=transformer_auto_wrap_policy(
                 transformer_layer_cls={LlamaDecoderLayer},
             ))

mixed_precision 块是关键。reduce 永远走 fp32——bf16 在跨 DP rank 把上千个微小梯度求和时会悄悄下溢。我们 debug 过 loss 发散的 run，最后发现就是漏了一个 reduce_dtype=fp32。

巨型模型（DeepSeek-V3 671B、Qwen3-Max）连 FSDP 都不够，要四维并行加自己的代码。Megatron-LM、NVIDIA NeMo、ColossalAI、DeepSpeed 是四个生产框架，大多数实验室都在某个上面拉了自己的 fork。

实算：70B 在 64 张 H100 上的并行选择

为什么 TP=8、PP=2、DP=4 是这个配置的对答案。每张 H100 有 80 GB HBM。总 HBM：64 × 80 = 5120 GB。

每卡内存预算。 每 rank 需要：

  • 权重：70B × 2 字节（BF16）/ TP / PP = 70e9 × 2 / 16 = 8.75 GB
  • 梯度：跟权重一样 = 8.75 GB
  • Adam 优化器状态：4 字节/参数 × 2（m、v）/ TP / PP / DP_FSDP = 70e9 × 8 / 64 = 8.75 GB（FSDP 切优化器的话）
  • 激活（带梯度 checkpointing）：随 batch × seqlen × model dim 变，70B 在 seqlen 8K 下一般 30-50 GB
  • KV/scratch buffer：约 5 GB

合计：约 62 GB / 80 GB，留约 18 GB 余地给 FlashAttention scratch、NCCL buffer、激活峰值。紧但能跑。

为什么不纯 FSDP=64？ 64 卡纯 FSDP 在每层 forward 之前要 all-gather 70B 参数。70B × 2 字节 = 140 GB 数据每次 gather，64 rank 分摊 = 每 rank 每 gather 2.2 GB，按层级。80 层就是每 forward 每 rank 约 175 GB gather 流量。NVLink 900 GB/s 下这是约 200 ms 纯通信每次 forward——跟算力时间打成平手，吞吐被杀。

为什么不纯 TP=64？ TP 在每个 matmul 后都要通信（行并行层上的 all-reduce）。每 token all-reduce 量 $O(d_{\text{model}} \times \text{TP})$。TP=64 走跨节点链路（200 Gbps Infiniband）就主导了。TP 只在单个 NVLink 互联节点内（当代最多 8 卡）跑得好。

为什么 TP=8、PP=2、DP=4？ TP=8 在每节点内用快速 NVLink 跑 all-reduce 重的 TP 流量。PP=2 把模型切到两个节点，PP 通信相对稀疏（每 micro-batch 一次）。DP=4 配 FSDP 把优化器状态切到 4 个 DP rank，每 rank 持有 1/4 权重 × 1/(TP×PP) 层。PP 流水线 bubble（步时 10-20 %）可接受；其他并行配置开销更糟。

这是一个配置。不同模型大小和集群拓扑有不同最优。一般原则：最小化跨节点通信，TP 装进最快互联，PP 做下一层粒度，剩下的并行用 FSDP/DP 填。

1000 卡以上才出现的失败模式

8 卡上无所谓、8000 卡上能干掉整个 run 的事：

静默数据损坏。 单卡返回错的数学（宇宙射线、电压抖动）造成 NaN 被 all-reduce 出去。对策：定期对梯度做 checksum——如果 rank 0 的梯度范数和 rank 100 差超过 $10^{-3}$，停下来重新分片。

Loss spike。 一个坏 batch（Python parser dump、HTML soup 之类）产生梯度尖刺，把 optimizer 动量推偏，永不恢复。防御：激进梯度裁剪（默认 1.0，超大模型有时 0.3）+ 当梯度范数 > 跑动中位数 5 倍时 “跳过这个 batch”。Llama 3 论文提到 “checkpoint rollback” 是标准 SOP：loss spike 50 步内不可恢复时回滚到上一个好 checkpoint，LR 降 30 %，继续。

NCCL hang。 单节点的网卡固件 bug 让该 rank 的 all-reduce 永不返回。其他 999 个 rank 永远阻塞。防御：NCCL_TIMEOUT_S=300 + 看门狗重新调度。

Checkpoint 损坏。 2.7 TB 的 FSDP checkpoint（Qwen3-32B 优化器状态）写要 20 多分钟。中途某节点 OOM，你不只丢一个 step，要是原地覆盖还把上一个好 checkpoint 也丢了。永远写 step_xxx.tmp/、fsync，再 mv。

异构硬件。 即便 “同型号” 的 H100，时钟也会抖 2-3 %。慢 rank 拖后腿。防御：warmup 时按 rank throughput 排序，把慢 rank 放到 TP=1 的位置。

嵌入坍缩。 嵌入矩阵的梯度流在早期 epoch 被裁或 NaN 损坏，嵌入会坍缩到接近相同的向量。症状：训练 loss 卡在 log(V)，perplexity 在 V。诊断：监控嵌入矩阵奇异值；最大奇异值比次大主导 10 倍以上，嵌入坍缩了。修法：倒回坍缩前，嵌入 LR 升 2 倍，重启。

梯度范数发散。 跟 loss spike 不同。loss 稳定但梯度范数几千 step 缓慢上升趋势。这一般预示未来坍缩。按参数组监控 ||g||_2，不只全局。具体哪种层（一般是 LM head 或最后 transformer 层）通常是元凶。

Meta OPT-175B 的 logbook [Zhang et al., 2022] 至今是最好的公开 “真实大 run 长什么样” 文档。33 天的 run 里平均 1.7 天就有一次节点故障。

一次 run 里到底发生什么

70B 现代预训练 run（LLaMA-3 70B，15T token，约 1900 张 H100，60 天）大致烧：

  • Token：15T 输入，batch size 约 16M token
  • Step：15T / 16M ≈ 940K
  • 每 step 墙钟：约 5.5 秒（带 sequence packing）
  • 总算力：约 $2.5 \times 10^{25}$ FLOPs ≈ 6000 H100-年

LR schedule 是 cosine decay，从短 warmup 衰到峰值的 10 %。70B 的峰值 LR 约 1.5e-4（更大模型更低，按 μP 缩放如果你严谨的话）。Weight decay 0.1，β2=0.95，AdamW。

评估按 checkpoint 走：每 2000 step 在 5-10 个小评测（Hellaswag、ARC-easy、OpenBookQA、GSM8K-100、HumanEval-50）上跑一遍。Loss 便宜跟，downstream eval 才是真信号。

生产真相：前沿实验室真正在做的事

论文描述架构；生产系统大多是后勤。每个前沿预训练团队都在做的、模型卡里找不到的三件事：

多 scale 并行 run。 70B 目标 run 之前先跑 1B 和 7B 的 “scout” run，1/100 和 1/10 数据，同样架构、optimizer、数据混合。Scout run 验证配方，让你调 μP 也搬不动的超参（多是 schedule 细节）。Scout run 看着好，70B 高信心 kick off 配方靠谱。Llama 3 论文提到跟 70B 平行跑 405B 持续验证 scaling 行为。

激进监控。 Loss 曲线、梯度范数、单参数统计、attention pattern 直方图、专家利用率（MoE）、单层激活幅值、eval 分。监控仪表盘不是可选基础设施——是发现问题在它们累积之前的唯一办法。Anthropic 在模型卡里开源了部分监控（裁剪比、激活统计）；OpenAI 在 GPT-4 论文里描述过。

持续数据精修。 训练数据不静态。团队持续跑新质量分类器、移掉新发现的污染、加新合成数据源。Llama 3 明确提到 run 中途用重训过的质量分类器跑全数据并重新加权后续 batch。“训练前数据定死” 模型是学术的；生产团队迭代。

在线 tokenize vs 预 tokenize。 预 tokenize 15T token 要几周和 100+ TB 存储。训练时流式 tokenize 要 tokenizer 跟上数据 loader，意味着高度优化的 Rust tokenizer。多数生产配置预算 token ID 但从分布式文件系统流式而不是加载原始文本。

常见踩坑

我个人见过 5 件搞砸预训练 run 的事。

1. 忘了设 NCCL collectives 走 bf16/fp32 mismatch。 默认 NCCL all-reduce 在 fp16 模式下，把上千个小梯度求和时会悄悄下溢。永远：NCCL_PROTO=Simple 求稳定 + FSDP reduce_dtype=fp32。Llama 3 论文提到要加这个才能从 step 500K 附近的不稳定恢复。

2. RoPE base 上下文长度配错。 RoPE 用基频 $\theta = 10000$。32K 上下文下，频率 alias 了，模型学不到长程模式。修法：按 [bloc97 NTK-aware] 把 base 缩到 $10000 \times (\text{seqlen}/2048)^{d/(d-2)}$ 或用 YaRN [Peng et al., 2024]。我们追过一个 32K 上下文降质 bug 一周，才意识到 RoPE base 还是 4K 预训练阶段的设定。

3. 打包序列上的 loss masking。 Sequence packing 把多文档拼成一个训练序列。不 mask 跨文档 attention，模型学到伪跨文档关联，质量降 1-3 % MMLU。永远发出在文档边界 reset 的 position_ids 和 attention_mask。

4. 激活 checkpointing 配非确定性 op。 Checkpoint-recompute 阶段重跑 forward 要确定性 kernel。某些 FlashAttention 和 GeLU 实现有非确定性数值 reduction。结果：梯度对的 forward 值跟保存的不同。微妙的质量回退，几千 step 后才显现。永远给 checkpoint 区域设 torch.use_deterministic_algorithms(True) 或验证 kernel 是确定性的。

5. Optimizer 状态没切到磁盘。 70B 模型的 AdamW 优化器状态是 70B × 8 字节 = 560 GB。上千节点同时存到单个共享文件系统会死锁。用 FSDP 的 sharded checkpointing（每 rank 写自己的分片）配并行写。我们有过一个 checkpoint 因不慎串行化要存 4 小时。

研究前沿 2024-2026

当前 Llama 风格 “过滤 + 去重 + 训” 配方之后还会发生什么：

规模上的数据效率。 [Marion et al., 2023]（When Less is More）证明在精修的 200B token 子集上训能持平 1T token 未过滤集。本质是 “数据的 early stopping”——找最有信息的样本，跳过其余。

Curriculum 驱动的混合。 [Albalak et al., 2024] 显示动态数据混合——按训练中模型怎么改进调整每个数据源的比例——比固定混合赢 1-2 MMLU。Doremi [Xie et al., 2023] 提了一种 min-max 优化自动找最优权重。

Beyond Chinchilla 的算力最优。 Mixture-of-Experts 改 scaling law。[DeepSeek-AI, 2024] 报告了从 V3 实验导出的 MoE scaling law：最优激活/总参数比、最优专家数、最优专家大小都以非 trivial 方式依赖训练算力。社区还在校准 MoE 专属的 scaling law。

持续预训练。 多数生产模型现在是更新而不是从零重训。Llama 3.1 是 Llama 3 + 在额外数据上继续预训练 + post-training。继续预训练便宜得多，但要小心 LR schedule 避免灾难性遗忘。

小结与下一篇

预训练 70 % 是数据工程，30 % 是分布式系统工程。架构选型是这三件里最小的一件。把数据混合做对，狠去重，按推理成本动机过 Chinchilla 训，用 μP 让超参可迁移，按硬件拓扑挑 FSDP+TP+PP，给只在大规模上出现的失败模式写防御代码。

下一篇：Post-training。SFT、DPO、RLHF、RLAIF——每个具体优化的是什么、奖励模型在哪些地方失败（它们一直在失败）、LoRA vs 全量微调那场争论、以及把 base model 变成客户能用的东西的生产配方。
