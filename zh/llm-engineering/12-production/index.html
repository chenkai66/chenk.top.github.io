大模型工程（十二）：生产 —— 部署、监控、成本

服务栈选型细化、给 LLM 做 autoscaling、延迟预算、prompt+completion 成本跟踪、多模型路由、FrugalGPT 级联、第一天就要的可观测性，以及能用的 on-call 模式。

这是最后一章。前面讲的都是如何构建模型、设计 prompt、实现检索和评估。这一章的重点是如何让系统持续运行，同时不把预算烧光。生产环境下的 LLM 服务更像是在运行一个高流量的 Web 服务，而不是传统的 ML 推理服务——唯一的区别是，每个 Web 请求可能花费几美元，还可能卡上两分钟。

这一章我会更注重数字分析。原因很简单：在生产环境中，一个功能是盈利还是烧钱，往往取决于 2 到 5 倍的成本差异，而这个差异通常没人关注。最有价值的技能是学会快速估算 LLM 工作负载的成本。以下数据截至 2025 年底到 2026 年初，请在实际使用前核对最新的价格信息。

端到端服务栈

一个生产级 LLM 应用的服务栈通常包含以下几层：

[CDN / WAF]
   ↓
[API 网关]    ← 限流、认证、请求标准化
   ↓
[应用服务器]  ← 提示组装、RAG 检索、工具执行、智能体循环
   ↓
[LLM 网关]    ← 模型路由、回退机制、成本核算、提示缓存
   ↓
[LLM 服务]    ← vLLM/SGLang/托管 API
   ↓
[可观测性]    ← 日志、指标、追踪、评估运行

应用服务器和 LLM 网关是工程实现的核心部分。应用服务器负责业务逻辑，而 LLM 网关的作用是让多个模型像一个统一的服务那样工作。

从一开始就要把 LLM 网关设计成独立的服务。它的功能包括：

  • 多模型路由：根据分类器将部分请求发送到小型快速模型，其他请求发送到大型慢速模型。
  • 回退机制：当主供应商返回 5xx 错误时，切换到备用供应商重试。
  • 成本跟踪：记录每个请求的提示词 token 数、生成 token 数、使用的模型以及成本。
  • 提示缓存封装：即使供应商支持缓存，应用代码也不需要关心缓存键的设计。
  • A/B 测试：将一定比例的流量路由到新模型变体。
  • 配额管理 / 熔断机制：当单个用户消耗过多资源时，终止其请求。

从零开始构建这个网关需要几周时间。不过，开源社区已经提供了不少成熟的选项：

  • LiteLLM：纯 Python 实现的代理，支持 100 多个供应商集成，提供即插即用的 OpenAI 兼容接口，内置了不错的成本跟踪功能。这是最快搭建可用网关的方式。
  • OpenRouter：托管式网关，通过单一 API 跨越多个供应商，并内置了模型市场。虽然每 token 成本比直接对接供应商略高，但能自动处理故障转移和价格套利。
  • Cloudflare AI Gateway：部署在 CDN 边缘的托管代理，支持缓存、限流和分析功能。成本低且运维简单，但牺牲了一些灵活性。
  • BentoML / Bento Cloud：更重型的框架，适合需要在同一网关后托管自定义训练模型的场景。
  • Portkey、Langfuse Gateway：新兴玩家，观测能力非常强大。

尽早选择一个方案，同时做好后期替换的准备。网关是少数几个容易导致供应商锁定的地方（因为我会写大量代码与网关的接口绑定），所以尽量保持接口简洁。

自托管 vs 托管 API

一个永恒的选择。2026 年的坦诚答案如下：

什么时候用托管 API？

  • 我需要前沿模型的质量（比如 GPT-5、Claude-4.5、Gemini-3），但没有足够的 GPU 来自托管。
  • 每月处理的 token 量低于 10 亿——在这个规模下，托管服务通常更划算。
  • 延迟要求不高，TTFT 超过 500 毫秒也能接受。
  • 我无法安排工程师专门负责 GPU 运维。

什么时候选择自托管？

  • 每月 token 量超过 10 亿，并且开源模型能满足我的质量要求（比如 Qwen3-32B+、LLaMA-3.3-70B+）。
  • 数据驻留有严格要求。
  • 我需要稳定实现 TTFT 小于 100 毫秒。
  • 我有特定的微调需求。

对于 Qwen3-32B 级别的工作负载，盈亏平衡点大约是每月 10 亿 token。低于这个量，算上工程成本，托管 API 更便宜。高于这个量，使用专用 GPU（无论是租还是买）自托管的成本优势会达到 3 到 10 倍。

一个常见错误是 GPU 利用率太低。如果一台 4xH100 的设备只跑在 30% 的利用率，单 token 成本会比 OpenAI API 还贵。目标是让持续吞吐率达到 70% 以上。

2025 到 2026 年，一种更流行的混合模式是：用小规模开源模型自托管大部分低成本流量，把最难的 5%-10% 流量路由到托管的前沿 API。 这样既能享受自托管的成本优势，又能在关键场景保留前沿模型的质量。路由决策由一个小分类器完成（详见后面的多模型路由章节）。

多模型路由和 FrugalGPT

路由再级联的模式由 Chen 等人在 2023 年的论文《FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance》中正式提出。核心发现是：大多数 LLM 请求都很简单，小模型或低成本模型就能正确处理；只有少数请求真正需要前沿的大模型。如果能以很低的成本判断出哪些请求属于哪一类，就可以在几乎不影响质量的情况下将成本降低 5 到 10 倍。

FrugalGPT 提出了三种方法：

  1. Prompt 优化——压缩提示词，去掉冗余上下文，合并相关请求。
  2. LLM 级联——先用最便宜的模型尝试，获取它的置信度信号，如果置信度低就切换到更贵的模型。
  3. 模型路由——通过训练一个分类器，直接选择合适的模型。

在实际应用中，级联是最常用的方案。伪代码如下：

    return answer  # 返回最后一个模型的输出

难点在于如何获取置信度信号。以下是几种常见方法：

  • 自报告置信度——让模型自己打分，比如“请为你的答案信心打 1 到 5 分”。这种方法和正确性有一定相关性，但噪声较大。
  • 基于 logprob——使用答案片段的 logprob 值。这种方法比较合理，但只适用于你能控制采样的场景。
  • 采样一致性——对便宜模型采样 3 到 5 次，如果结果一致就接受；如果不一致，就交给更贵的模型处理。
  • 任务专用验证器——针对代码任务，可以检查“是否通过公开测试”；对于数学问题，可以判断“答案是否在合理范围内”。

基于 embedding 的路由方法（Hu 等，2024，《RouteLLM: Learning to Route LLMs with Preference Data》）通过离线偏好数据中的 (query, best-model) 对训练分类器，从而提前预测应该使用哪个模型。RouteLLM 训练的路由器在 MT-Bench 上以 26%-44% 的成本达到了 GPT-4 95% 的质量。

在生产环境中，路由决策通常归结为两个问题：“这个请求简单吗？”（交给便宜模型）和“这个请求是已知的复杂模式吗？”（交给昂贵模型）。中间情况则走默认路径。一个包含 50 条规则的启发式分类器可以捕捉 80% 的节省潜力，而训练好的路由器则能捕捉约 90%。

缓存层

有三种重要的缓存，它们可以叠加使用：

  1. 供应商侧的 prompt 缓存（第 9 章）——重复 prompt 前缀的 KV 缓存复用。由 Anthropic、OpenAI、DeepSeek 和 Google 提供支持。缓存部分的成本能降低 90%。
  2. 应用侧的语义缓存——当两个用户查询在语义上相似时，直接返回缓存的答案。常用工具包括 GPTCache（基于 FAISS 的原始版本）、Redis Semantic Cache、MemCache 加上自定义的 embedding 查找。对于 FAQ 类型的工作负载，命中率可达 30%-70%，而对于开放式对话，命中率几乎为 0。
  3. 确定性函数的结果缓存——如果 LLM 的输出被送入下游流水线（比如分类或信息抽取任务），且输入完全相同，就可以将输出以输入哈希值为键存储起来。这样，相同的输入永远不会触发第二次 LLM 调用。

GPTCache 值得深入研究一下。它的模式是这样的：用一个小的 embedding 模型对每个用户查询进行编码，然后与缓存中的内容进行对比（通过余弦相似度）。如果相似度超过阈值（通常为 0.95），就直接返回缓存的答案。这个阈值是一个关键参数——调高阈值会减少误判，但也会降低命中率。比如，一个客服机器人回答常见问题时，把阈值设为 0.92 可以节省 50% 的成本；但对于代码生成助手来说，每个查询都独一无二，语义缓存基本没什么用。

缓存的组合效果很重要：prompt 缓存降低了每次调用的成本，而语义缓存减少了调用的次数。两者结合能让优化效果成倍提升。

给 LLM 做 autoscaling

给 LLM 做自动扩缩容比给无状态的 Web 服务难得多。主要有三个原因：

  1. 模型加载耗时长。一个 70B 参数的模型，新启动的 vLLM 实例需要 2 到 4 分钟加载权重并预热。如果流量高峰只持续 10 分钟，响应式扩容根本来不及。
  2. GPU 按小时计费。为一个 5 分钟的流量高峰启动新的 GPU 实例，会白白浪费 55 分钟的 GPU 资源。
  3. 连续批处理的负载非线性。vLLM 服务器在 50% 和 90% 负载下的延迟表现差异很大，安全运行点远低于传统 CPU 服务的建议值。

实际操作中可以参考以下方法：

  • 定时预热：如果流量高峰通常出现在早上 9 点，可以在 8:50 提前扩容。
  • 保守扩容，积极缩容：当利用率到达 60% 时扩容，降到 30% 时缩容。（这和典型的 Web 服务策略相反。）
  • 备用回退方案：当自托管的服务达到容量上限时，将溢出的请求转到托管 API。虽然单价更高，但能有效应对流量高峰，避免资源浪费。
  • 多模型保底：即使在低流量时段，也要确保每个模型至少有一个副本在线。冷启动的成本太高，不能忽视。
  • 带背压的请求队列：当服务器过载时，用公平调度器按用户排队处理请求，并返回带有 Retry-After 头的 429 状态码，而不是让所有用户的延迟都崩溃。

vLLM 提供了两个重要参数：--max-num-seqs（最大并发请求数）和 --max-num-batched-tokens（每步最大 token 吞吐量）。调整这些参数时，优先考虑满足延迟目标，而不是一味追求吞吐量最大化。

延迟预算拆解

一个面向用户的聊天产品，延迟预算大致如下：

组件预算（ms）备注
网络输入50地理位置影响
API 网关 / 认证10要求快速响应
应用服务器逻辑50包括 RAG embedding、工具调度
RAG 检索100向量数据库 + 重排序
LLM 网关开销5仅路由操作
LLM TTFT（队列 + 预填充）300模型本身的处理时间
网络输出50和输入一致
首 token 总计~565 ms控制在 1 秒目标内
LLM ITL（解码）25每个 token，支持 40 tok/s
网络缓冲20平滑输出

两个最关键的延迟指标是：TTFT（Time To First Token）和 ITL（Inter-Token Latency）。TTFT 是用户感知“有没有反应”的关键；ITL 则决定了流式输出时的持续速度。2026 年的行业经验如下：

  • 聊天：TTFT < 800 ms 算“即时”，< 2 秒算“可接受”。
  • 语音：TTFT < 300 ms 才能实现自然对话。
  • 代码补全：TTFT > 200 ms 用户会放弃使用。
  • 批处理 / 智能体任务：TTFT 可以放宽到 5-30 秒，用户通常能容忍。

首 token 输出后，用户看到流式结果，5-10 秒内可以接受约 40-60 tok/s 的速度。总的响应时间取决于输出长度：比如 200 个 token，按 40 tok/s 的速度计算，需要 5 秒，加上 600 ms 的 TTFT，端到端总共是 5.6 秒。

延迟预算应该花在哪里？

  • 重排序（Reranking）：50-100 ms 是值得的（见第 8 篇）。
  • 推测解码（Speculative decoding）：能减少 50-100 ms 的 ITL（见第 5 篇），是个不错的投资。
  • 工具调用（Tool calls）：一个耗时 200 ms 的工具会阻塞模型。尽量并行化处理，或者积极缓存结果。
  • 推理 / 思考（Reasoning / Thinking）：思考型模型会在输出首个可见 token 前增加 1-10 秒。只在必要时使用。

对于延迟敏感的路径，实际测量时间花在哪。拆解结果往往出乎意料。我见过一个“慢”的功能，LLM 本身只用了 200 ms，但 RAG 的重排序花了 1.2 秒；另一个案例中，LLM 表现正常，但下游的 JSON 序列化却拖了 800 ms。优化之前，先做好追踪分析。

第一天就开始成本跟踪

每请求的成本核算，这是必须的。没有它，我无法做到以下几点：

  • 发现某个用户或功能导致的成本激增。
  • 对成本与质量的调整进行 A/B 测试。
  • 在账单到来之前预测支出。
  • 及时发现失控的 agent 循环，避免花掉 1000 美元。

按用户、功能和模型进行聚合。当出现以下情况时触发告警：

  • 单个用户在 1 小时内的花费超过中位数的 10 倍。
  • 某功能的单次调用成本周环比翻倍。
  • 新模型在质量相当的情况下，每次对话解决成本高于旧模型。

一个有用的算术练习。假设我的产品每天调用 LLM 1 万次，平均输入 4K token，输出 500 token，使用的是 Claude-4.5 Sonnet：

  • 每次调用：$4 \cdot 4 + 15 \cdot 0.5 = 16 + 7.5 = 23.5$ 千 tok 分 = $0.0235。
  • 每天：$235。
  • 每月：$7050。

加入 prompt 缓存（4K 输入中有 3K 被缓存）：

  • 每次调用：$4 \cdot 1 + 0.30 \cdot 3 + 15 \cdot 0.5 = 4 + 0.9 + 7.5 = 12.4$ 千 tok 分 = $0.0124。
  • 每月：$3720，节省 $3330（47 %）。

再加入 30 % 的语义缓存命中率：

  • 每月：$3720 \cdot 0.7 = $2604，总共节省 $4446（63 %）。

最后加入级联处理，其中 60 % 的查询由 Qwen3-32B 处理（自托管，输入 $0.10 + 输出 $0.30 每 Mtok）：

  • 低成本路径每次调用：$0.10 \cdot 4 + 0.30 \cdot 0.5 = 0.55$ 千 tok 分 = $0.00055。
  • 混合每日成本：$0.6 \cdot 7000 \cdot 0.00055 + 0.4 \cdot 7000 \cdot 0.0124 = $2.31 + $34.72 = $37/天。乘以 30 天 = $1110/月，比基线便宜 84 %。

这些节省是叠加的。如果没有显式成本跟踪，根本看不到这些优化效果。

成本之外的可观测

标准可观测性（trace、log、metric）加上 LLM 专用指标：

  • 每个模型的延迟和错误率，按模型分开统计。
  • 每个副本的 token 吞吐量（自托管）。如果这里出现下降，通常说明模型负载有问题。
  • 质量漂移：随机抽取 1% 的请求，通过评估集的评分流程，绘制随时间变化的通过率曲线。
  • Prompt 注入检测：统计每个用户的注入尝试频率。
  • 拒绝率：模型拒绝的比例。突然的变化可能意味着攻击或者模型退化。
  • 缓存命中率：包括 prompt 缓存和语义缓存。命中率下降可能是因为 prompt 模板改动破坏了缓存逻辑，或者流量模式发生了变化。

到 2026 年依然好用的工具：

  • Langfuse（开源）——最适合 LLM 的 tracing 工具，免费版功能强大，容易自托管。
  • Helicone——托管式的 LLM 可观测性工具，带成本跟踪功能，开箱即用。
  • Phoenix（Arize）——开源的 LLM 可观测性工具，专注于检索和 RAG 质量指标。
  • OpenLLMetry——基于 OpenTelemetry，可以无缝接入现有的可观测性平台（如 Datadog、Grafana、Honeycomb）。
  • Datadog + 自定义仪表盘——如果你已经在用 Datadog，并且希望统一在一个界面查看数据，这是不错的选择。

最省钱的方案：对每个请求进行结构化日志记录，发送到 ClickHouse 或 Snowflake，然后搭建一个仪表盘。不需要花钱买托管的可观测性产品，也能获得 80% 的价值，但前提是必须坚持记录正确的字段。

能用的 on-call 模式

凌晨 3 点把我叫醒的问题，以及对应的处理步骤：

LLM 供应商 5xx 错误激增。 先看供应商的状态页面。如果有备用供应商，立刻切换过去。如果主供应商在 5 分钟内恢复正常，就不用管；超过 5 分钟，联系供应商的客户经理。记得检查切换到备用供应商后有没有导致成本飙升（有时候备用供应商的价格是主供应商的 2 倍）。

延迟 p99 > 30 秒。 一般是某个工具调用失控，或者请求卡住了。先看看是不是某个用户或某个功能占用了大部分资源。通常杀掉卡住的请求，再收紧超时限制就能解决问题。

成本突然暴涨。 每小时花费突然涨了 10 倍。查一下用户表——几乎可以肯定是某个用户的 agent 出问题了。先对这个用户限流，然后调查根本原因。最常见的原因是 agent 没有设置递归步数限制，导致无限循环调用自己。

拒绝率突然上升。 模型一夜之间多拒绝了 30% 的请求。要么是模型更新出了问题，要么是有人在搞协同攻击。先回滚模型，再深入调查。

RAG 质量下降。 faithfulness 分数掉了 10%。通常是 embedding 或 index 版本不匹配，或者是新内容格式变了。检查数据摄入管道。

自托管 GPU 内存不足（OOM）。 KV 缓存耗尽了。要么增加容量，要么降低每个请求的 max_tokens 参数。别急着重启服务，不然下一批请求还是会 OOM。

供应商弃用通知。 我们依赖的某个模型版本将在 90 天内被移除。立刻安排迁移测试，针对推荐的新版本做至少 2 周的 A/B 测试，并记录下差异。

好的 runbook 能把平均恢复时间从 30 分钟缩短到 3 分钟。一定要在问题发生之前就把 runbook 写好。

值得命名的常见失败模式

以下是我见过或读到的一些生产事故，列出一个不完全清单：

  • 供应商区域故障。 整个可用区（AZ）瘫痪。应对方法：在网关层实现多区域故障切换；对于托管 API，准备备用供应商随时接管。
  • 限流级联效应。 供应商对某个客户限流，我的应用疯狂重试，结果耗尽了全局配额。应对方法：在网关前设置每用户限流；重试时采用带抖动的指数退避策略；为每个供应商配置熔断机制。
  • Prompt 模板问题。 系统 Prompt 的“无害”修改导致缓存失效（第 9 章），账单一夜之间涨了三倍。应对方法：在 CI 中加入缓存感知的差异检查；监控缓存命中率下降并触发告警。
  • 模型弃用意外。 OpenAI 宣布 gpt-4-turbo-0613 将在 60 天后退役，但我从未用替代模型跑过评估集。应对方法：订阅供应商的弃用通知；每季度进行迁移演练。
  • 静默质量下降。 托管模型收到未记录的更新，导致在我的特定场景中性能下降。应对方法：每周用生产流量样本运行评估集；与历史基线对比；如果质量下降超过 5%，立即告警。
  • Agent 循环失控。 Agent 调用工具时出错，重试后又遇到不同错误，模型开始自我争辩并不断重试，最终花费 $500。应对方法：限制每次对话的步骤数和成本上限；如果单次会话费用超过 $X，触发告警。

迁移 playbook

在生产环境中，模型迁移是一项被严重低估的技能。以下是经过验证的有效方法：

  1. 影子流量测试 3-7 天。 新模型接收所有请求的副本，但只记录输出结果，不返回给用户。离线对比质量。
  2. 灰度发布 1%，持续 3-7 天。 让新模型处理 1% 的真实流量，同时监控延迟、错误率、成本、拒绝率和用户满意度。如果没有问题，继续推进。
  3. 逐步提升流量：5% → 25% → 50% → 100%，耗时 1-2 周。 每一步都必须通过 SLA 监控的验证。
  4. 切换后保持旧模型热备 30 天。 回滚只需修改一个配置项。
  5. 确保 30 天无异常后，再彻底下线旧模型。

跳过影子流量测试和灰度发布，是导致“上线新模型后客户投诉激增”的主要原因。这不是技术问题，而是流程纪律问题。

输出成本通常是输入的 3 到 5 倍，因为输出是顺序解码，受内存限制。如果应用需要生成长内容（比如总结、报告或代码），换成输出成本更低的模型，往往比优化模型本身更划算。

Batch API 的折扣（24 小时 SLA 工作负载半价）是一个被严重低估的成本优化手段。如果你的任务可以接受次日完成（比如隔夜报告、批量内容审核、评估集生成或数据标注），使用批处理能直接将花费减半。迁移也很简单，通常只需改一行 API 调用代码。

性价比生产 LLM 产品的最终配方

如果我在 2026 年中启动一个生产级 LLM 产品，我会这么做：

  1. 前 6 到 12 个月直接用托管 API。根据延迟、质量和语言需求选择合适的。
  2. 第一天就搭建 LLM 网关，哪怕只对接一个供应商。迟早会用到。
  3. 精确记录每次请求的成本，保留到小数点后两位（$0.0001 级别）。每周汇总一次。
  4. 先准备一个 200 道题的评估集，再开始优化提示词。
  5. 如果有外部知识库需要依赖，加上检索和重排序（reranker）功能。
  6. 尽可能多地缓存：在供应商端缓存提示词，对应用内重复问题做语义缓存，对确定性流程的结果做缓存。
  7. 单次提示无法解决真实用户请求时，引入工具使用（tool use）。
  8. 简单查询交给便宜模型处理，覆盖 60%-80% 的流量；复杂查询交给高性能模型。
  9. 月均 token 超过 10 亿且流量稳定时，切换到自托管。
  10. H100 上用 FP8 量化，A100 和 L40 上用 AWQ-INT4 量化。
  11. 对于能接受 24 小时 SLA 的任务，直接用批量 API（batch API）。
  12. 提前搭建安全栈，别等到出问题才动手。

这套方案足够支撑产品上线和增长，同时避免了拖垮大多数 LLM 产品的技术债务。
