大模型工程（十二）：生产 —— 部署、监控、成本

服务栈选型细化、给 LLM 做 autoscaling、延迟预算、prompt+completion 成本跟踪、第一天就要的可观测性，以及能用的 on-call 模式。

Read in English →

最后一篇。前面所有都在讲建模型、prompt、检索、评估。这一篇讲怎么让它一直在线、不破产。生产 LLM 服务跟跑高流量 web 服务的共性比跟传统 ML 服务多——只不过 web 请求每条要花钱、还能挂两分钟。

端到端服务栈

生产 LLM 应用栈通常这样几层：

[CDN / WAF]
   ↓
[API 网关]    ← 限流、auth、请求归一化
   ↓
[App 服务器]  ← prompt 装配、RAG 检索、tool 执行、agent 循环
   ↓
[LLM 网关]    ← 模型路由、回退、成本核算、prompt caching
   ↓
[LLM 服务]    ← vLLM/SGLang/托管 API
   ↓
[可观测]      ← logs、metrics、traces、eval run

App 服务器和 LLM 网关是工程主要栖身处。App 服务器管业务逻辑；LLM 网关让一组模型像单个服务那样表现。

第一天就把 LLM 网关建成单独服务。你会需要它做：

  • 多模型路由：按 classifier 把一些请求送小快模型、其他送大慢的。
  • 回退：主供应商返 5xx 就换备用。
  • 成本跟踪：每个请求 log prompt token、completion token、模型、成本。
  • prompt caching wrapper：即使供应商支持 caching，你 app 代码也不该想 cache key。
  • A/B 测试：可配比例的流量到新模型变体。
  • 配额 / 熔断：单用户消耗不成比例成本时杀请求。

从零建几周工程。开源选项（LiteLLM、OpenRouter、BentoML）覆盖 70 % 用例。

自托管 vs 托管 API

永恒决定。2026 年诚实答案：

用托管 API 当：

  • 你需要前沿模型质量（GPT-5、Claude-4.5、Gemini-3）且没有 GPU 自托管。
  • 体量低于约 10 亿 token/月——这个规模托管通常更便宜。
  • 延迟要求宽松（>500 ms TTFT 可接受）。
  • 不能投人做 GPU 运维。

自托管当：

  • 体量超过 10 亿 token/月且开源模型符合质量线（Qwen3-32B+、LLaMA-3.3-70B+）。
  • 严格数据驻留要求。
  • 持续需要 <100 ms TTFT。
  • 有特定微调需求。

Qwen3-32B 级负载盈亏点大致 10 亿 token/月。低于这个，算上工程时间托管 API 在成本上赢。高于这个，专用 GPU（租或买）自托管赢 3-10 倍。

常见错误：自托管 GPU 利用率不足。30 % 利用率的 4xH100 部署单 token 比 OpenAI API 还贵。持续吞吐 >70 % 是目标。

给 LLM 做 autoscaling

给 LLM autoscaling 比给无状态 web 服务难。两个原因：

  1. 加载模型要几分钟。70B 模型新 vLLM replica 要 2-4 分钟加载权重和 warm up。流量峰值响应式扩容太慢，如果峰值只持续 10 分钟。
  2. GPU 按小时付费，不按请求。为 5 分钟峰值起新 GPU 实例浪费 55 分钟 GPU。

实战模式：

  • 按计划预热：流量可预测在 9 点峰值就 8:50 预扩。
  • 保守扩、急切缩：60 % 利用率扩，30 % 缩。（跟典型 web 服务反过来。）
  • 缓冲回退：自托管满了溢出到托管 API。单价高但吸收峰值不浪费配置。
  • 多模型底：低流量也保每个模型至少一个 replica 在线。冷启动代价太高。

vLLM 支持 --max-num-seqs（最大 in-flight 请求）和 --max-num-batched-tokens（每 step 最大 token 吞吐）。按延迟目标调而不是最大化吞吐。

延迟预算

面向用户的 chat 产品大致这个延迟预算：

组件预算（ms）备注
网络入50地理
API 网关 / auth10应该快
App 服务器逻辑50RAG embedding、tool dispatch
RAG 检索100向量库 + reranker
LLM 网关开销5仅路由
LLM TTFT300模型本身
网络出50跟入一样
首 token 总计~565 ms1s 目标内
LLM ITL25单 token，撑 40 tok/s
网络缓冲20平滑

首 token 出来后用户看流式，5-10 秒内能容忍约 40-60 tok/s。可接受总响应时间看长度：200 token @ 40 tok/s = 5s 加 600 ms TTFT = 5.6 s 端到端。

延迟预算花在哪：

  • Reranking：50-100 ms 花得值（第 8 篇）。
  • Speculative decoding：-50 到 -100 ms ITL（第 5 篇），好投资。
  • Tool 调用：200 ms 的 tool 阻塞模型。能并行就并行，狠 cache。
  • 推理 / 思考：thinking 模型在用户可见首 token 之前加 1-10 s。留给真要的任务。

第一天就开始成本跟踪

每请求成本核算不可妥协。没有它你不能：

  • 找单用户或单功能的成本膨胀。
  • A/B 测成本-质量改动。
  • 在账单到之前预测花费。
  • 在 agent 循环失控花掉 1000 美元之前抓住。

按用户、功能、模型聚合。告警当：

  • 单用户在 1 小时内超过中位花费 10 倍。
  • 单功能每调用成本周环比翻倍。
  • 新模型每解决会话成本超过老模型，尽管质量持平。

成本之外的可观测

标准可观测（trace、log、metric）加 LLM 专属：

  • 每模型延迟和错误率，按模型拆。
  • 每 replica token 吞吐（自托管）。这里回归常预示模型加载问题。
  • 质量漂移：采 1 % 请求过 eval set 评分流程，画 pass rate 时间线。
  • prompt-injection 检测：每用户检测到的注入尝试率。
  • 拒绝率：模型拒绝比例。突变预示攻击或模型回归。

2026 年能用的工具：

  • Langfuse（开源）——LLM 专属 tracing 最好。
  • Helicone——托管 LLM 可观测带成本跟踪。
  • Datadog + 自定义 dashboard——已有 Datadog 就行。
  • 用 OpenTelemetry 自卷——有运维带宽就行。

能用的 on-call 模式

凌晨 3 点 page 我的事和各自的 runbook：

LLM 供应商 5xx 暴涨。 看供应商状态页。有备用就 failover。主在 5 分钟内恢复就不动；更长就 page 供应商客户经理。

延迟 p99 > 30s。 通常是失控 tool call 或卡住的请求。看是否单用户/单功能主导。常常通过杀卡住请求和加紧超时解决。

成本暴涨：每小时 $ 突涨 10 倍。看用户表——几乎总是某个用户的 agent 失控。限流，查根因。

拒绝率跳升：模型一夜之间多拒 30 %。要么模型 rollout 回归要么有协调攻击。回滚模型，调查。

RAG 质量下降：faithfulness 分降 10 %。通常 embedding/index 版本不匹配或新内容格式不同。查摄入 pipeline。

自托管 GPU OOM：KV cache 耗尽。要么加容量要么降每请求 max_tokens。别盲目重启；下一 batch 同样 OOM。

好 runbook 把 mean-time-to-recovery 从 30 分钟降到 3 分钟。需要之前 先写 runbook。

配置$/Mtok 输入$/Mtok 输出备注

输出持续比输入贵 3-5 倍，因为输出是顺序 decode（内存受限）。生成长输出（总结、报告、代码）的应用，换到输出经济更好的模型常常打败优化模型本身。

性价比生产 LLM 产品的最终配方

2026 年中我要起一个生产 LLM 产品：

  1. 前 6-12 个月默认走托管 API。按延迟 / 质量 / 语言需求挑。
  2. 第一天就建 LLM 网关，即使只包一个供应商。你会需要。
  3. 每请求成本跟踪到两位小数（$0.0001 级）。每周聚合。
  4. 建 200 题 eval set 再迭代 prompt。
  5. 检索 + reranker 分层 如果有外部知识可依附。
  6. 狠 cache：供应商侧 prompt caching、应用里语义 cache 重复问题。
  7. 加 tool use 当单 shot prompt 解不了真实用户请求。
  8. 跨过 10 亿 token/月持续流量就自托管。
  9. H100 上 FP8 量化、A100/L40 上 AWQ-INT4。
  10. 必须之前先加安全栈。

够上线和增长，不背杀掉多数 LLM 产品的技术债。
