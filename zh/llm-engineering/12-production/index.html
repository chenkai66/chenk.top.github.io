大模型工程（十二）：生产 —— 部署、监控、成本

服务栈选型细化、给 LLM 做 autoscaling、延迟预算、prompt+completion 成本跟踪、多模型路由、FrugalGPT 级联、第一天就要的可观测性，以及能用的 on-call 模式。

最后一篇。前面所有都在讲建模型、prompt、检索、评估。这一篇讲怎么让它一直在线、不破产。生产 LLM 服务跟跑高流量 web 服务的共性比跟传统 ML 服务多——只不过 web 请求每条要花钱、还能挂两分钟。

我会比前几章更倚重数字。原因简单：生产里盈利功能和烧钱坑的差别通常是没人在跟踪的 2-5x 成本因子。最有用要发展的技能是 LLM 负载的信封背面成本算术。下面数字 2025 末/2026 初准确；承诺前对当前定价核实。

端到端服务栈

生产 LLM 应用栈通常这样几层：

[CDN / WAF]
   ↓
[API 网关]    ← 限流、auth、请求归一化
   ↓
[App 服务器]  ← prompt 装配、RAG 检索、tool 执行、agent 循环
   ↓
[LLM 网关]    ← 模型路由、回退、成本核算、prompt caching
   ↓
[LLM 服务]    ← vLLM/SGLang/托管 API
   ↓
[可观测]      ← logs、metrics、traces、eval run

App 服务器和 LLM 网关是工程主要栖身处。App 服务器管业务逻辑；LLM 网关让一组模型像单个服务那样表现。

第一天就把 LLM 网关建成单独服务。你会需要它做：

  • 多模型路由：按 classifier 把一些请求送小快模型、其他送大慢的。
  • 回退：主供应商返 5xx 就换备用。
  • 成本跟踪：每个请求 log prompt token、completion token、模型、成本。
  • prompt caching wrapper：即使供应商支持 caching，你 app 代码也不该想 cache key。
  • A/B 测试：可配比例的流量到新模型变体。
  • 配额 / 熔断：单用户消耗不成比例成本时杀请求。

从零建几周工程。开源选项覆盖大部分空间：

  • LiteLLM——纯 Python proxy，100+ 供应商集成、即插即用 OpenAI 兼容端点、像样的内置成本跟踪。最快得到能用网关的方式。
  • OpenRouter——跨供应商单 API 的托管网关、内置模型市场。每 token 成本比直走供应商高，但自动处理 failover 和定价套利。
  • Cloudflare AI Gateway——CDN 边缘的托管 proxy，带缓存、限流、分析。便宜运维平凡；放弃一些灵活性。
  • BentoML / Bento Cloud——更重框架，需要在同一网关后托管自训模型时有用。
  • Portkey、Langfuse Gateway——观测故事强的新进入者。

早选一个、计划晚换。网关是少数几个 vendor lock-in 真实的地方（你写很多代码对网关合约）；保合约小。

自托管 vs 托管 API

永恒决定。2026 年诚实答案：

用托管 API 当：

  • 你需要前沿模型质量（GPT-5、Claude-4.5、Gemini-3）且没有 GPU 自托管。
  • 体量低于约 10 亿 token/月——这个规模托管通常更便宜。
  • 延迟要求宽松（>500 ms TTFT 可接受）。
  • 不能投人做 GPU 运维。

自托管当：

  • 体量超过 10 亿 token/月且开源模型符合质量线（Qwen3-32B+、LLaMA-3.3-70B+）。
  • 严格数据驻留要求。
  • 持续需要 <100 ms TTFT。
  • 有特定微调需求。

Qwen3-32B 级负载盈亏点大致 10 亿 token/月。低于这个，算上工程时间托管 API 在成本上赢。高于这个，专用 GPU（租或买）自托管赢 3-10 倍。

常见错误：自托管 GPU 利用率不足。30 % 利用率的 4xH100 部署单 token 比 OpenAI API 还贵。持续吞吐 >70 % 是目标。

2025-2026 更流行的实战混合模式：自托管小开源模型扛主体便宜流量、把难的 5-10 % 路到托管前沿 API。 这捕获自托管多数成本节省同时保留前沿质量在重要的地方。路由决策由小分类器做（下面多模型路由章节）。

多模型路由和 FrugalGPT

路由再级联模式由 Chen 等（2023，FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance）形式化。核心观察：多数 LLM query 是简单的小/便宜模型能正确处理；只有小部分真需要前沿模型。能廉价决定哪是哪，质量损失最小情况下能砍 5-10x 花费。

FrugalGPT 提三个模式：

  1. Prompt 适配——压缩 prompt、丢冗余上下文、相关请求 batch。
  2. LLM 级联——先试最便宜模型，问它置信信号，置信低就回退到更贵模型。
  3. 模型路由——学到的分类器先选对的模型。

级联是实战最常部署的。伪代码：

难的部分是置信信号。选项：

  • 自报置信——问模型"评 1-5 置信"。跟正确性轻相关但噪声大。
  • 基于 logprob——用答案 span 的 logprob。合理但只在你控采样时工作。
  • 采样一致——便宜模型采 3-5 次；都同意就接受。不同意就升级。
  • 任务专用验证器——代码 “过公开测试吗？"；数学 “答案在合理范围吗？”

基于 embedding 的路由（Hu 等，2024，RouteLLM: Learning to Route LLMs with Preference Data）在线下偏好数据的 (query, best-model) 对上训分类器，先预测用哪个模型。RouteLLM 训练的路由器在 MT-Bench 上以 26-44 % 成本达到 GPT-4 95 % 质量。

生产里路由决策常塌缩到两个问题：“这 query 简单吗？"（路便宜）和 “这 query 是已知难模式吗？"（路贵）。中间走默认。50 题启发式分类器捕获 80 % 节省；学到的路由器捕获约 90 %。

缓存层

三个缓存重要、且叠加：

  1. 供应商侧 prompt cache（第 9 篇）——重复 prompt 前缀的 KV-cache 复用。Anthropic / OpenAI / DeepSeek / Google 跑。缓存部分 90 % 成本降。
  2. 应用侧语义 cache——两个用户 query 语义相似 时服务缓存答案。工具：GPTCache（原版、FAISS 基）、Redis Semantic Cache、MemCache + 自定义 embedding 查询。FAQ 风格负载 30-70 % 命中率，开放式 chat 接近 0 %。
  3. 确定性函数的结果 cache——LLM 输出喂下游 pipeline（如分类、抽取）且输入精确匹配，就按输入哈希存输出。同输入 LLM 永不被叫两次。

GPTCache 值得细看。模式：用小 embedding 模型嵌每个用户 query、对 cache 比（cosine 相似度）、相似度 > 阈值（典型 0.95）就服务缓存答案。阈值是旋钮——更高意味更少假阳性但更低命中率。客服 bot 答常见问题，0.92 阈值能交付 50 % 成本降；每 query 唯一的代码生成助手，语义 cache 约无用。

可组合性重要：prompt cache 降 每次 调用成本；语义 cache 降调用 次数。它们相乘。

给 LLM 做 autoscaling

给 LLM autoscaling 比给无状态 web 服务难。三个原因：

  1. 加载模型要几分钟。70B 模型新 vLLM replica 要 2-4 分钟加载权重和 warm up。流量峰值响应式扩容太慢，如果峰值只持续 10 分钟。
  2. GPU 按小时付费，不按请求。为 5 分钟峰值起新 GPU 实例浪费 55 分钟 GPU。
  3. 持续 batching 在负载下非线性。50 % 和 90 % 负载的 vLLM 服务器延迟 profile 很不同；安全工作点比 CPU 服务暗示的低得多。

实战模式：

  • 按计划预热：流量可预测在 9 点峰值就 8:50 预扩。
  • 保守扩、急切缩：60 % 利用率扩，30 % 缩。（跟典型 web 服务反过来。）
  • 缓冲回退：自托管满了溢出到托管 API。单价高但吸收峰值不浪费配置。
  • 多模型底：低流量也保每个模型至少一个 replica 在线。冷启动代价太高。
  • 带背压的队列：服务器过载时用按用户公平调度器排队请求，返带 Retry-After 头的 429 而不是把所有人延迟撞坏。

vLLM 支持 --max-num-seqs（最大 in-flight 请求）和 --max-num-batched-tokens（每 step 最大 token 吞吐）。按延迟目标调而不是最大化吞吐。

延迟预算拆解

面向用户的 chat 产品大致这个延迟预算：

组件预算（ms）备注
网络入50地理
API 网关 / auth10应该快
App 服务器逻辑50RAG embedding、tool dispatch
RAG 检索100向量库 + reranker
LLM 网关开销5仅路由
LLM TTFT（队列+prefill）300模型本身
网络出50跟入一样
首 token 总计~565 ms1s 目标内
LLM ITL（decode）25单 token，撑 40 tok/s
网络缓冲20平滑

两个延迟指标最重要：TTFT（Time To First Token）和 ITL（Inter-Token Latency）。TTFT 是用户感觉"有事发生吗”；ITL 决定 token 流后持续读速。2026 年行业经验法则：

  • Chat：TTFT < 800 ms 是"瞬时”，< 2 s 是"可接受”
  • 语音：TTFT < 300 ms 自然对话必需
  • 代码补全：TTFT < 200 ms 否则用户放弃
  • Batch / agentic：TTFT 能 5-30 s；用户容忍

首 token 出来后用户看流式，5-10 秒内能容忍约 40-60 tok/s。可接受总响应时间看长度：200 token @ 40 tok/s = 5s 加 600 ms TTFT = 5.6 s 端到端。

延迟预算花在哪：

  • Reranking：50-100 ms 花得值（第 8 篇）。
  • Speculative decoding：-50 到 -100 ms ITL（第 5 篇），好投资。
  • Tool 调用：200 ms 的 tool 阻塞模型。能并行就并行，狠 cache。
  • 推理 / 思考：thinking 模型在用户可见首 token 之前加 1-10 s。留给真要的任务。

延迟关键路径，量你实际花时间在哪。拆解很少是你猜的。我见过"慢"功能 LLM 200 ms 但 RAG reranker 1.2 s；另一个 LLM 没事但下游 JSON 序列化 800 ms。优化前 trace。

第一天就开始成本跟踪

每请求成本核算不可妥协。没有它你不能：

  • 找单用户或单功能的成本膨胀。
  • A/B 测成本-质量改动。
  • 在账单到之前预测花费。
  • 在 agent 循环失控花掉 1000 美元之前抓住。

按用户、功能、模型聚合。告警当：

  • 单用户在 1 小时内超过中位花费 10 倍。
  • 单功能每调用成本周环比翻倍。
  • 新模型每解决会话成本超过老模型，尽管质量持平。

有用的算术练习。假设你产品做 1 万 LLM 调用/天、平均 4K 输入 + 500 输出 token、Claude-4.5 Sonnet：

  • 每调用：$4 \cdot 4 + 15 \cdot 0.5 = 16 + 7.5 = 23.5$ 千 tok 分 = $0.0235。
  • 每天：$235。
  • 每月：$7050。

加 prompt caching（4K 输入里 3K 缓存）：

  • 每调用：$4 \cdot 1 + 0.30 \cdot 3 + 15 \cdot 0.5 = 4 + 0.9 + 7.5 = 12.4$ 千 tok 分 = $0.0124。
  • 每月：$3720，省 $3330（47 %）。

加 30 % 语义缓存命中率：

  • 每月：$3720 \cdot 0.7 = $2604，总省 $4446（63 %）。

加级联 60 % query 由 Qwen3-32B 处理（自托管，$0.10 输入 + $0.30 输出每 Mtok）：

  • 便宜路径每调用：$0.10 \cdot 4 + 0.30 \cdot 0.5 = 0.55$ 千 tok 分 = $0.00055。
  • 混合每天：$0.6 \cdot 7000 \cdot 0.00055 + 0.4 \cdot 7000 \cdot 0.0124 = $2.31 + $34.72 = $37/天。乘 30 天 = $1110/月，比基线便宜 84 %。

这些节省复利。没显式成本跟踪，没一个可见。

成本之外的可观测

标准可观测（trace、log、metric）加 LLM 专属：

  • 每模型延迟和错误率，按模型拆。
  • 每 replica token 吞吐（自托管）。这里回归常预示模型加载问题。
  • 质量漂移：采 1 % 请求过 eval set 评分流程，画 pass rate 时间线。
  • prompt-injection 检测：每用户检测到的注入尝试率。
  • 拒绝率：模型拒绝比例。突变预示攻击或模型回归。
  • 缓存命中率：prompt cache、语义 cache。命中率掉预示 prompt 模板改动毁缓存或流量漂移。

2026 年能用的工具：

  • Langfuse（开源）——LLM 专属 tracing 最好、慷慨免费层、易自托管。
  • Helicone——托管 LLM 可观测带成本跟踪；开箱强。
  • Phoenix（Arize）——开源 LLM 可观测，专注检索/RAG 质量指标。
  • OpenLLMetry——基于 OpenTelemetry，插入现有可观测栈（Datadog、Grafana、Honeycomb）。
  • Datadog + 自定义 dashboard——已有 Datadog 想单一玻璃面板就行。

能用的最便宜模式：结构化 log 每个请求、运到 ClickHouse 或 Snowflake、建 dashboard。不需托管可观测产品就能拿 80 % 价值，但需要 log 对字段的纪律。

能用的 on-call 模式

凌晨 3 点 page 我的事和各自的 runbook：

LLM 供应商 5xx 暴涨。 看供应商状态页。有备用就 failover。主在 5 分钟内恢复就不动；更长就 page 供应商客户经理。检查 failover 没在备用上引发成本暴涨（有时备用贵 2x）。

延迟 p99 > 30s。 通常是失控 tool call 或卡住的请求。看是否单用户/单功能主导。常常通过杀卡住请求和加紧超时解决。

成本暴涨：每小时 $ 突涨 10 倍。看用户表——几乎总是某个用户的 agent 失控。限流，查根因。经典 case 是 agent 没步限自递归。

拒绝率跳升：模型一夜之间多拒 30 %。要么模型 rollout 回归要么有协调攻击。回滚模型，调查。

RAG 质量下降：faithfulness 分降 10 %。通常 embedding/index 版本不匹配或新内容格式不同。查摄入 pipeline。

自托管 GPU OOM：KV cache 耗尽。要么加容量要么降每请求 max_tokens。别盲目重启；下一 batch 同样 OOM。

供应商弃用通知：依赖的模型版本将在 90 天内移除。立即排迁移测对推荐继任者、A/B 至少 2 周、文档化 diff。

好 runbook 把 mean-time-to-recovery 从 30 分钟降到 3 分钟。需要之前 先写 runbook。

值得命名的常见失败模式

我见过或读过的生产事件非完全清单：

  • 供应商区域宕机。 整个 AZ 倒。缓解：网关层多区 failover；托管 API 备用供应商待命。
  • Rate-limit 级联。 供应商对你某客户限流；你 app 激进重试吃掉全局配额。缓解：网关前每用户限流；重试指数退避带 jitter；每供应商熔断。
  • Prompt 模板回归。 系统 prompt 的"无害"改动让 prompt cache 失效（第 9 篇），账单一夜翻三倍。缓解：CI 里缓存感知 diff；缓存命中率掉告警。
  • 模型弃用惊讶。 OpenAI 宣布 gpt-4-turbo-0613 60 天后退役；你 eval set 从未对替代品跑过。缓解：订阅供应商弃用 feed；季度迁移演练。
  • 静默质量漂移。 托管模型受未文档化更新对你具体用例回归。缓解：每周对生产流量样本跑 eval set；对历史基线比；>5 % 降告警。
  • Agent 循环失控。 Agent 调工具收错、重试、收不同错、模型现在跟自己争论着重试，累积 $500。缓解：每会话步限；每会话成本限；会话 > $X 告警。

迁移 playbook

模型间迁移是被低估的生产技能。能用的模式：

  1. 影子流量 3-7 天。 新模型收每个请求副本但输出 log 不返。线下比质量。
  2. Canary 1 % 3-7 天。 新模型服务 1 % 真实流量；监控延迟、错误率、成本、拒绝率、满意度。无回归 → 继续。
  3. Ramp 5 → 25 → 50 → 100 % 1-2 周。每步监控 SLA gating。
  4. 切换后保老模型暖 30 天。 回滚 flag 一个配置改动外。
  5. 30 天干净运营后才下线。

跳过影子+canary 是"上了新模型客户投诉暴涨"最常因。是流程纪律，不是技术。

输出持续比输入贵 3-5 倍，因为输出是顺序 decode（内存受限）。生成长输出（总结、报告、代码）的应用，换到输出经济更好的模型常常打败优化模型本身。

性价比生产 LLM 产品的最终配方

2026 年中我要起一个生产 LLM 产品：

  1. 前 6-12 个月默认走托管 API。按延迟 / 质量 / 语言需求挑。
  2. 第一天就建 LLM 网关，即使只包一个供应商。你会需要。
  3. 每请求成本跟踪到两位小数（$0.0001 级）。每周聚合。
  4. 建 200 题 eval set 再迭代 prompt。
  5. 检索 + reranker 分层 如果有外部知识可依附。
  6. 狠 cache：供应商侧 prompt caching、应用里语义 cache 重复问题、确定性流的结果 cache。
  7. 加 tool use 当单 shot prompt 解不了真实用户请求。
  8. 级联到便宜模型 简单 60-80 % query，难 query 路前沿。
  9. 跨过 10 亿 token/月持续流量就自托管。
  10. H100 上 FP8 量化、A100/L40 上 AWQ-INT4。
  11. 24 小时 SLA 容忍的负载用 batch API。
  12. 必须之前先加安全栈。

够上线和增长，不背杀掉多数 LLM 产品的技术债。

系列收尾

12 篇讲端到端建一个现代 LLM 产品需要什么：

  1. 架构
  2. Tokenization
  3. 预训练
  4. Post-training
  5. 推理优化
  6. 长上下文
  7. Function calling
  8. RAG
  9. Prompting
  10. 评估
  11. 安全
  12. 生产

只记三件事的话：数据质量主导一切（第 3、4 篇）、tokenization 和 KV cache 付账单（第 2、5 篇）、没有 eval set 和成本跟踪不能上线（第 10、12 篇）。其余是执行。
