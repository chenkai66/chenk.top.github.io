大模型工程（五）：推理优化

KV cache 力学、paged attention、continuous batching、speculative decoding、INT8/INT4/AWQ/GPTQ 量化，以及 vLLM、SGLang、TensorRT-LLM 的取舍。

Read in English →

钱主要花在推理上。一个 70B 级别的模型，如果同时服务 1000 个用户，每个用户每秒处理 50 个 token，那么在大约 3 个月内就会耗尽训练该模型所需的 GPU 预算。本章所有内容都围绕两个关键指标：首 token 生成时间（TTFT）和 token 间延迟（ITL）。还有一个重要比率：每百万输出 token 所需的 GPU 秒数。

训练是一次性的资本支出——成本可以在数百万次推理中摊销。而推理是持续的运营支出，与训练不同，它不会自行摊薄。tokens-per-GPU-second 提升 0.5 倍的效果会在产品整个生命周期内日复一日地累积。这也是为什么所有认真的 LLM 团队至少会有一名全职工程师专注于推理优化，以及为什么开源社区在过去五年里推出了四代不同的推理引擎（FasterTransformer → DeepSpeed-Inference → vLLM → SGLang/TensorRT-LLM/llama.cpp）。

两个特性完全不同的阶段

每次 LLM 推理都分为两个阶段：

  1. Prefill（处理提示）：输入 token 并行通过模型，填充 KV 缓存。计算密集型。70B 模型上 4K token 的提示大约需要 280 TFLOP——一张 H100 大约 70 毫秒就能打满。
  2. Decode（生成）：每次生成一个 token，对缓存的 K 和 V 进行注意力操作。内存带宽密集型。每个解码步骤读取整个 KV 缓存（数 GB），生成一个 token。

不对称性是关键。Prefill 在不同用户间可以很好地批量处理（同一个内核，不同序列）。Decode 在朴素批处理下效果很差，因为每个用户在不同的序列位置。主流推理引擎都是基于这种不对称性构建的。

常用的经验法则是：TTFT 主要由 prefill 决定，ITL 则主要由 decode 的内存带宽决定。要降低 TTFT，增加 FLOPs（更多 SM、张量并行）。要降低 ITL，增加内存带宽（HBM3 优于 GDDR，通过量化减少参数）。

算术强度使这一点更加明确。对于 70B 模型的 4K 提示 prefill：模型权重（BF16 下 140 GB）加载一次，处理 4096 个 token。算术强度约为每字节参数 4096 FLOP。同一模型单 token decode：权重加载一次，处理 1 个 token。算术强度约为每字节 1 FLOP。H100 在 BF16 下 989 TFLOPS、HBM 3.35 TB/s，ridge point 约为 295 FLOP/字节。Prefill 的 4096 FLOP/字节远高于 ridge（计算受限）。Decode 的 1 FLOP/字节低两个数量级（内存受限）。这两个阶段需要不同的硬件特性，不分开处理它们的服务栈会严重损失性能。

KV cache：支撑长上下文的数据结构

KV cache 存储每一层每个历史 token 投影后的 K 和 V 向量。以 70B GQA-8 模型在 32K 上下文为例（数据来自第 1 篇）：

$$ \text{KV} = 2 \cdot 80 \cdot 2 \cdot 8 \cdot 128 \cdot 32{,}768 \cdot 2 \text{ 字节} = 8.6 \text{ GB} $$

每次请求都需要这么多内存。如果有 50 个并发请求，那就需要 430 GB 的 KV cache——远超模型权重。可见，KV cache 才是瓶颈，而不是权重。

最简单的实现方法是为每个请求分配一块连续的张量，大小为 max_context。这种方法有两个问题：

  1. 内部碎片。即使请求只用了 1K token，也得预留 32K 内存。
  2. 无法扩展。一旦超出 max_context，解码过程中就会出现 OOM。

第三个问题在负载下尤为明显：外部碎片。请求到达和离开的时间不同，导致空闲内存分散在不连续的区域，没有一块足够大来容纳新请求。服务器在变化的负载下运行数小时后，仅外部碎片就占用了 20-40% 的可寻址 KV 内存。这个问题与操作系统在 1960 年代通过分页解决的问题相同，解决方案也一样。

Paged attention

vLLM 的杀手级特性是 2023 年论文 Kwon 等 ↗ 中提出的 paged attention。KV 缓存按固定大小的 块 分配（通常每块 16 个 token），每个请求的 块表 将逻辑位置映射到物理块。这类似于操作系统的虚拟内存。

请求 A：47 个 token 的 KV → 3 块（16+16+15）
请求 B：200 个 token 的 KV → 13 块
A 的块表：[0x47, 0x12, 0x3a]
B 的块表：[0x05, 0x09, 0x21, ...]

好处：

  • 几乎零浪费：每个请求只有最后一块可能有内部碎片，最多 15 个 token。
  • 内存共享：前缀相同的请求（如系统提示、few-shot 示例、工具定义）可以共用同一物理块。这就是 prompt 缓存的来源。
  • 易于抢占：内存紧张时，可以把请求的块换出到 CPU；有空闲资源再拉回来。

vLLM 在 LLaMA-13B 上的实测吞吐量比朴素的 Hugging Face Transformers 循环提高了 14 到 24 倍（基于原论文的基准测试）。这个数字至今仍然成立——不过实际生产中的差距取决于负载组合。对于单流服务，提升接近 2 倍；对于高并发混合长度流量，通常为 5 到 10 倍；而 14 到 24 倍的提升是在与完全没有批处理的 真朴素 HF 代码对比时得出的。关键不在于具体倍数，而是 paged attention 让 KV 内存从硬约束变成了可管理的资源。

enable_prefix_caching=True 自动启用重复 prompt 前缀的块共享。如果一个系统提示在 1000 个请求中复用，这些请求的 prefill 速度会提高 5 到 50 倍。

Block manager：分页如何工作

Block manager 是每个 paged-attention 引擎的核心。它管理一个物理 KV 块池（通常每块 16 个 token × 2(KV) × 层数 × 头数 × head_dim 字节）、维护空闲列表，并原子地为新请求分配块。当请求需要扩展（下一个解码步骤）时，manager 会弹出一个空闲块，将其物理地址追加到请求的块表中，然后返回。在 copy-on-write 语义下，前缀共享的块使用引用计数：fork 序列（如并行采样、beam search 或推测回滚）会增加引用计数；释放时减少引用计数；只有当引用计数归零时，才会真正释放物理内存。

“换出到 CPU” 路径在生产环境中非常重要。当 GPU 没有空闲块但仍有新请求排队时，调度器会选择一个受害请求（通常是 FIFO 或运行时间最长的），通过 PCIe 将其块拷贝到 pinned host memory，然后释放 GPU 块。当内存可用时，再把请求拉回来。换出开销确实存在（PCIe Gen5 ×16 速率为 64 GB/s，意味着换出和重新加载一个 8 GB 请求需要 100 毫秒），但它让系统能够在不直接拒绝请求的情况下满足延迟 SLO。

Continuous batching

另一场革命。静态批处理要等同一批次中最慢的序列完成后，才会开始下一批次。由于输出长度不一致（这总是会发生），GPU 会有很多空闲时间。

连续批处理（Yu 等，Orca，OSDI 2022 ↗ ，vLLM 推广）：每个解码步骤中，移除已完成的序列，并加入等待中的序列。只要任务排队，GPU 始终运行不少于 max_batch_size 条序列。

对于 1000 个提示，输出长度从 50 到 2000 不等：

  • 静态批处理：总耗时约等于 2000 个 token 输出的时间（最坏情况主导）。
  • 连续批处理：总耗时约等于所有输出的平均值。

结合分页注意力机制，吞吐量改进更加显著——中途加入新序列只需分配新的 KV 块，无需重新分配整个张量。

Orca 的迭代级调度

Orca 论文引入了两个机制，到 2026 年仍在广泛应用。迭代级调度将批处理边界与请求边界解耦：每个 transformer 迭代独立调度，因此在第 500 步完成的请求不会影响需要 1500 步的请求。选择性批处理解决了预填充和解码不匹配的问题——让注意力算子按每个序列的形状操作，而线性层（不关心位置）则一起批处理。现代引擎进一步融合：vLLM 和 SGLang 都采用“分块预填充”，一个请求的预填充与其他请求的解码在同一前向传递中交错进行，即使解码队列稀疏，也能保持 GPU 利用率在 90% 以上。

调度决策很关键：什么时候在批次中途接受新请求？简单来说，立即接受。但将 32K token 的预填充与正在进行的解码一起处理，会显著增加每步迭代的延迟（32K 预填充的 FLOPS 是单个解码步骤的约 100 倍）。生产调度器设定了每步迭代的“预算”上限，并将长预填充切分成多个块，与解码步骤交错执行。这是决定负载下 TTFT p99 是 500 毫秒还是 5 秒的关键旋钮。

Speculative decoding

这个聪明的想法现在已经成为标准。解码过程受内存限制：每一步都要读取整个模型权重来生成一个 token。如果能一次验证 N 个 token 呢？

Speculative decoding（Leviathan 等，2023 ↗ ）使用 draft 模型（小而快，比如 1B 参数的模型）来提议 $k$ 个 token。目标模型（你真正要使用的 70B 参数模型）在一次前向传递中验证这些 token：它看到提议的前缀，并行计算 $k+1$ 个位置的概率。与目标模型 argmax 匹配的 token 保留；第一个不匹配的 token 及其后的 token 被丢弃，然后从那里重新采样。

数学上，如果 draft 模型的正确率为 $\alpha$，且解码过程受内存限制（即 $k+1$ 个 token 的前向传递成本与 1 个 token 的前向传递差不多），那么预期加速比为 $(1 - \alpha^{k+1})/(1 - \alpha)$。当 $\alpha=0.7$、$k=4$ 时，加速比为 2.5 倍。

实际应用中，draft 模型通常是从目标模型蒸馏出来的。目前有三种主要方法：

  • Draft-target 双模型对，这是 Leviathan 最初提出的形式：一个小的蒸馏模型提议，目标模型验证。质量很高，但维护两个同步的检查点是个运营难题。
  • Medusa（Cai 等，2024 ↗ ）：在目标模型本身增加 4-5 个额外的预测头。每个头从相同的 backbone 激活预测位置 $t+1, t+2, t+3, ...$。不需要独立的 draft 模型。加速比为 2-3 倍，且没有质量损失；在基础模型上只需几个 GPU 天就能训练出来。
  • EAGLE / EAGLE-2（Li 等，2024 ↗ ）：在倒数第二层隐藏状态加上前一个 token 上的自回归头。EAGLE-2 引入了动态树验证，每一步探索多个 draft 分支。大多数负载下可以达到 3-4 倍的加速，是当前自回归推测的 SOTA 方法。

需要注意的是，只有当目标 GPU 受内存限制时，推测解码才有帮助，如果是 FLOPs 受限则无效。大 batch 是 FLOPs 受限的（权重加载一次后摊到 batch 中的所有序列上），推测解码帮不上忙。单流低延迟服务是内存受限的，受益巨大。

关于正确性的一个微妙之处：验证步骤对目标分布进行 拒绝采样。如果 draft 模型对 token $t$ 的概率是 $p_d(t)$，目标模型的概率是 $p_t(t)$，则以 $\min(1, p_t(t)/p_d(t))$ 的概率接受；拒绝时从修正分布 $\propto \max(0, p_t(t) - p_d(t))$ 重新采样。这保证了输出分布 精确 是目标模型的分布，与 draft 模型的质量无关。推测解码是无损的：更差的 draft 模型只会导致更低的接受率，不会影响输出质量。这也是为什么团队上线推测解码时不需要质量回归测试——因为没有质量可退化。

量化：INT8、INT4、FP8

推理时，将 bf16 转换为 INT8/INT4/FP8 可以减少内存和带宽占用。通常只量化权重，有时也量化激活。

INT8 仅权重量化

最简单的方法是按通道量化每个权重矩阵：$w_q = \text{round}(w / s)$，其中 $s$ 是每个输出通道的缩放因子。推理时，将权重实时反量化回 bf16 再进行矩阵乘法。

内存：权重从 16 位降到 8 位，节省一半。质量：通常损失小于 0.5% 的困惑度，无需校准。这是个免费的优化。

INT4：GPTQ vs AWQ

INT4 难得多。简单的四舍五入会导致 2-5% 的困惑度损失。目前有两种有效的算法：

GPTQ（Frantar 等，2022 ↗ ）：基于 OBQ（Optimal Brain Quantization），适用于十亿参数量级模型。逐列量化权重矩阵；每列后用 Cholesky 分解的 Hessian 逆来更新剩余未量化的列，以补偿引入的误差。数学上，每列是一个有结构的最小二乘求解。计算成本：单 A100 上处理 70B 参数需要几小时，峰值内存大约等于一层激活（典型 128 个样本 × 2048 个 token 的小校准集）。

AWQ（Lin 等，2023 ↗ ）：发现 1% 的权重是“显著”的（穿过它们的激活值较大）。通过每个通道的缩放因子 $s$ 保护这些显著权重不受量化噪声影响——相当于在量化前将输入激活乘以 $s$，权重除以 $s$。这在全精度下是恒等变换，但重塑了量化地形。然后均匀量化。AWQ 比 GPTQ 快（不需要 Hessian 求解，只需搜索缩放因子），多数模型的质量略好，对 MoE 更友好。

SmoothQuant（Xiao 等，2022 ↗ ）：与 GPTQ 和 AWQ 正交，处理激活量化。难点在于激活中有异常通道（少数通道的幅值是其他通道的 100 倍），会破坏 INT8 激活量化。SmoothQuant 通过每个通道的缩放因子将激活幅值迁移到权重中，使两边都能进行 INT8 量化。作为 INT8 W8A8 量化前的预处理步骤使用。

对于 LLaMA 类模型，GPTQ 和 AWQ 都能在 INT4 仅权重量化时保持小于 1% 的困惑度损失。MoE 中，路由复杂化了校准——不同专家看到的激活分布差异很大，统一的缩放因子无法保护少用的专家。当前最佳实践是每个专家单独使用 AWQ。

FP8（以及 H100/H200 硬件路径）

H100 及更新的 GPU 支持 FP8 tensor core，吞吐量是 BF16 的两倍。FP8 推理是现代路径：权重存储为 FP8，激活在计算时转换为 FP8，累积使用 FP32。质量损失可忽略不计（<0.1%），因为 FP8 的动态范围比 INT8 大。

FP8 有两种格式：E4M3（4 位指数、3 位尾数）用于激活和权重——范围小、精度高；E5M2（5 位指数、2 位尾数）用于梯度和 KV 缓存——范围大、精度低。硬件将反量化缩放融合到矩阵乘法中，因此没有“INT4 反量化开销”；FP8 是 H100/H200 上的速度上限路径。

尽管困惑度损失极小，校准对 FP8 仍然重要。标准方法是在 128-516 个校准样本上收集激活统计，计算每个张量或每个 token 的缩放因子，并将其存储在检查点中。运行时从实际批次计算每个 token 的激活缩放因子，可以在较小的额外延迟开销下获得最佳质量。NVIDIA TransformerEngine 和 vLLM --quantization fp8 都实现了这一点。

需要注意的是，FP8 需要硬件支持。INT4/INT8 在任何 GPU 上都能运行，而 FP8 仅限于 H100/H200/B200。如果你在 A100 上部署，只能选择 INT8/INT4。

KV 缓存量化

KV 缓存压缩是内存账单的另一半。将 KV 缓存压缩到 INT8（节省一半内存），或 INT4（1-2% 的质量损失，需仔细的每个 token 校准）。vLLM 和 SGLang 都支持 FP8 KV 缓存；INT4 KV 缓存还在研究阶段，FlashInfer 提供了相关内核。

预计 2026 年，FP8 将成为主力。它将内存减半（同 GPU 上并发请求翻倍），质量损失小于 0.1%。每个 token 的缩放因子几乎免费，因为它们是在线计算的。INT4 KV 缓存需要更多注意——AWQ 用于权重的每个通道显著性结构不直接适用于 KV（KV 是数据而不是参数）。当前最佳实践是每个 token 每个头的缩放因子加上异常值保留：保留约 1% 的通道在高精度，其余部分进行激进量化。

SGLang 与 RadixAttention

vLLM 的前缀缓存技术在前缀完全相同的请求间共享块。SGLang（Zheng 等，2024 ↗ ）通过 RadixAttention 将这一概念扩展：所有活跃的 KV 块构成一个 radix 树，从根到叶的每条路径代表一个 token 序列。新请求会在树中查找最长匹配前缀，共享这些块，并只计算后缀部分。

这比 vLLM 的精确前缀缓存更有意义，因为 agent 负载会分支。例如，ReAct agent 可能发出 5 个工具调用子查询，每个查询都有相同的系统提示、工具和对话历史，但后缀不同。线性前缀缓存可以让每个子查询与共享根匹配，但无法让子查询之间共享状态。RadixAttention 则可以实现这一点。

根据 SGLang 论文报告，在 agent 和结构化输出负载上，RadixAttention 的吞吐量是 vLLM 等价前缀缓存的 5 倍，而在简单聊天任务上约为 1.5 倍。对于不共享前缀的流量，这种优势会减小。

具体实现方法是：维护一棵以 token ID 为键的 radix 树，节点保存 KV 块引用。当有新请求时，尽可能深地匹配输入前缀；增加所有访问过的节点的引用计数；为未匹配的后缀分配新的块；完成后减少引用计数；内存压力增大时，使用 LRU 算法驱逐子树。数据结构的开销相比节省下来的资源来说微不足道。

TensorRT-LLM 细节

NVIDIA 的 TensorRT-LLM 是第三大引擎。它的特点如下：

  • 编译内核：每个模型都会针对特定 GPU、批处理大小和序列长度编译成 TensorRT 引擎。编译时间 10 到 30 分钟；在支持的工作负载上，生成的引擎比 vLLM 运行时编译的版本快 1.1 到 1.3 倍。
  • 插件模型：attention（FMHA、paged FMHA、FlashAttention-3 路径）、MoE 路由和量化的自定义内核以插件形式提供。虽然不如 vLLM 灵活，但允许更激进的融合。
  • 飞行批处理：类似于连续批处理，使用 NVIDIA 优化的调度器。
  • 紧密 FP8 集成：TransformerEngine FP8 是首选。在 H100 上使用 FP8 时，TensorRT-LLM 通常吞吐量领先 5% 到 15%。
  • 转换麻烦：每种新模型架构都需要一个自定义的 Python 构建脚本。新模型的支持比 vLLM 滞后几周到几个月。

如果你在 NVIDIA 硬件上大规模运行一组固定的成熟模型，并且有足够的工程资源，可以选择 TensorRT-LLM。否则，vLLM 的操作简便性几乎总是更有优势。

2026 年的服务框架选型

三个主要选项：

  • vLLM——事实上的开源标准。社区最活跃，新模型和功能支持最快。Paged attention、连续批处理、推测解码、前缀缓存等功能开箱即用。默认配置就很好。除非有特殊原因，否则就选它。
  • SGLang——更新（2024），在结构化生成（约束 JSON、正则表达式）、前端缓存（RadixAttention 用于 prompt 树共享）以及高扇出 agent 场景中表现更好。共享前缀负载的 TTFT 更低。
  • TensorRT-LLM——NVIDIA 的框架。在 H100 上支持的模型吞吐量最高（编译内核、融合 FlashAttention 路径），但转换过程繁琐，新增模型支持较慢。如果有 NVIDIA 支持，并且需要榨干每一 token/s，就选这个。

vLLM 0.6+ 和 SGLang 都支持大部分功能：推测解码、FP8、AWQ/GPTQ、MoE、multi-LoRA、函数调用约束解码。差距正在缩小。默认选择 vLLM，如果工作负载是结构化输出或大量使用 agent，可以换成 SGLang。

Qwen3-32B 在 FP8 单 H100 上的纯吞吐数字（我自己的基准测试，2025 年底）：

引擎吞吐（tok/s）TTFT p50 (ms)ITL p50 (ms)
vLLM 0.674009513
SGLang 0.478007214
TensorRT-LLM89008811

对大多数负载来说，这些差异可以忽略不计。按开发体验来选。

服务并行模式

70B 模型在 BF16 精度下需要超过 140 GB 的内存。单个 H100（80 GB）无法容纳。解决方案分为三个正交轴——张量并行（TP）、流水线并行（PP）和序列并行（SP）。这些方法的组合方式与训练时不同（第 4 章）。

张量并行（TP）

将每个权重矩阵切分到多个 GPU 上。经典的 Megatron-LM 分解方法（Shoeybi 等，2019 ↗ ）是：QKV 投影按列切分（每块 GPU 负责一部分注意力头），注意力输出投影按行切分（之后进行 all-reduce），然后 FFN 的两个矩阵乘法分别按列和按行切分。每个 transformer 层需要两次 all-reduce。

对于推理任务，节点内的 TP（通过 NVLink 连接，双向带宽约 600-900 GB/s）是可行的；跨节点的 TP（通过 InfiniBand 连接，每条链路约 50 GB/s）会导致 all-reduce 延迟过高。通常 TP 不超过 8（一个节点）。

  • 使用 2 个 H100（同一节点，NVLink 连接）。吞吐量大约是单卡的 1.7 到 1.9 倍（不是 2 倍，因为 TP 会引入 all-reduce 通信）。
  • 如果需要长上下文，可以使用 TP=4（KV 缓存也会按 TP 切分）。
  • INT4 量化，单个 H100。虽然吞吐量低于 TP=2，但成本更低。

vllm serve Qwen/Qwen3-72B --tensor-parallel-size 2 一行命令搞定。

流水线并行（PP）

将模型层分配到多个 GPU 上。例如，GPU 0 负责层 0-19，GPU 1 负责层 20-39，依此类推。激活值在流水线中流动。经典问题是 流水线气泡：流水线末端的 GPU 在等待前端完成时会空闲，反之亦然。对于推理任务，PP 会引入与流水线深度 × 每层时间成正比的 TTFT 延迟（GPipe / Megatron-LM 分析）；现代引擎通过 微批处理（将批次分成更小的微批次，连续流经流水线）来摊销这种延迟。

PP 是推理任务中的跨节点并行方案。节点内使用 TP，跨节点使用 PP 是 200B+ 模型的标准部署。对于延迟敏感的任务，多数团队选择不超过一个节点——PP 气泡确实存在。

序列并行（SP）

对于非常长的序列，激活值本身可能无法装入单个 GPU。序列并行沿 token 维度切分，并使用环形或全对全通信进行注意力 all-reduce。在推理中，当上下文长度达到 512K 以上且每层激活缓冲区在 TP 后仍超过 GPU 内存时，才会用到 SP。大多数生产环境不需要这种并行方式。

数学：每种并行模式何时有效？

粗略估计，TP 每层的成本为 $2 \cdot \text{params}/\text{TP} / \text{HBM-BW} + 2 \cdot \text{batch} \cdot \text{seq} \cdot \text{hidden} / \text{NVLink-BW}$。第一项随 TP 线性减少；第二项增加。通常 H100 上 TP=2 到 TP=4 是最佳选择。超过 TP=8 时，all-reduce 成为主导因素，性能下降。对于 70B 模型，TP=2 时吞吐量约为 1.85 倍；TP=4 时约为 3.2 倍；TP=8 时约为 5.5 倍。边际递减效应确实存在。

PP 对于单批次查询的延迟成本约为 $\text{深度} \cdot \text{每层时间}$。在微批处理和稳态队列的情况下，吞吐量接近 $\text{batch} / (\text{每阶段时间})$。PP 适合提高吞吐量，但会增加延迟。

我会真正部署什么

7B 级模型：单张 L40S 或 4090，FP8，vLLM，16K 上下文，80% 利用率下每百万 token 0.10-0.15 美元。

32B 级模型：单 H100 加 AWQ INT4 或 TP=2 的 L40S 加 FP8。两种方案都可行。H100 单 token 处理更快；L40S 每小时成本更低。根据 $/Mtok 来决定。

70B 级模型：TP=2 的 H100 加 FP8。INT4 能省 30% 成本但损失约 1% 质量，生产环境中通常难以接受。

200B+ MoE：这是另一类优化问题（第 12 章）。

可观测性：实际要测量什么

在生产环境中，重要的指标不是 “tok/s”——这只是单批次的数字。真正关键的是负载下的百分位延迟。最基本的看板包括：

  • TTFT p50 / p95 / p99（ms）：从收到请求到发出第一个 token 的时间。p99 是用户实际感受到的；p50 是用户会夸耀的。
  • ITL p50 / p95 / p99（ms）：稳态下的 token 间延迟。这里的尖峰意味着抢占或内存压力。
  • 吞吐量（tok/s） 在不同并发水平下：1、8、32、128、512。画成曲线；拐点是你达到 FLOPs 或内存带宽饱和的地方。
  • 队列深度 和 队列等待时间：请求堆积是因为引擎处理不过来吗？
  • KV 缓存利用率（%）：快要开始抢占了吗？
  • Prefix 缓存命中率（%）：低命中率说明你的 prompt 工程团队生成了不可共享的 prompt。
  • GPU SM 利用率 和 HBM 带宽利用率：通过 dcgm-exporter 或 nvidia-smi dmon 获取。如果两者都低于 70%，你还有余地；如果 HBM 达到 95% 而 SM 只有 40%，说明你受内存限制，量化是解决办法。

明确设置 SLO。典型的生产目标是：TTFT p95 < 500 ms、ITL p95 < 50 ms、错误率 < 0.1%。超过这些标准就是紧急情况。

成本算术

关键数字是 每百万输出 token 的美元成本（$/Mtok-out）。以自托管 vLLM 在单 H100 上运行 Qwen3-32B FP8 为例，H100 的价格为：按需 $2.50/小时，1 年预留 $1.20/小时，3 年预留或 savings plan $0.80/小时。

  • 并发 32 时的吞吐量：约 7400 tok/s
  • 预留小时成本：$1.20/小时 = $0.00033/秒
  • $/Mtok-out = $0.00033 / 7400 \cdot 10^6 ≈ $0.045

对比 2026 年中期的 API 定价：

  • Claude-4.5-Sonnet：$15/Mtok 输出
  • GPT-5：$12/Mtok 输出
  • DeepSeek-V3.2：$1.10/Mtok 输出
  • Qwen3-Max API：$0.80/Mtok 输出

自托管开源权重和前沿 API 之间的成本差异可达 30 到 300 倍，这是团队选择自托管的主要原因。需要注意的是，每月处理少于 1 亿 token 时，工程成本会超过节省下来的费用。而每月处理超过 10 亿 token 时，节省下来的费用明显更多，自托管就显得非常合理了。

还有一个容易忽视的问题：API 上输入 token 通常比输出 token 便宜 5 到 10 倍，因为输入 token 可以更好地批量处理且不需要自回归。自托管情况下，输入 token 仍然更便宜，但差距没那么大（3 到 5 倍），因为两个阶段都在你的硬件上运行。如果成本模型假设输入和输出 token 价格相同，就会误导你。

小结与下一篇

推理分为两个不对称阶段：prefill 和 decode。现代服务栈包括 paged attention、continuous batching、speculation、量化和 FP8 硬件，这些技术旨在让这两个阶段的最糟糕情况变得可以接受。vLLM 是默认的好选择。量化（INT8 始终开启，预算紧张时用 INT4，H100+ 用 FP8）几乎不增加成本。Spec decoding 在低 batch 低延迟场景下能带来 2-3 倍的性能提升，但在高吞吐量场景中则效果不佳。到 2026 年，大规模自托管开源权重的成本优势将非常明显。
