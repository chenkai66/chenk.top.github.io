大模型工程（五）：推理优化

KV cache 力学、paged attention、continuous batching、speculative decoding、INT8/INT4/AWQ/GPTQ 量化，以及 vLLM、SGLang、TensorRT-LLM 的取舍。

#llm#inference#vllm#quantization#paged-attention#speculative-decoding
Read in English →

钱花在推理上。一个 70B 级模型给 1000 个并发用户跑 50 tok/s，能在 3 个月内吃掉训这个模型的 GPU 预算。这一篇里所有东西都为两个数字服务：time-to-first-token（TTFT）和 inter-token latency（ITL）。还有一个比值：每百万输出 token 的 GPU 秒。

训练是一次性资本支出——可以在数百万次推理里摊销。推理是经常性运营支出，跟训练不同，它不会自己摊薄。tokens-per-GPU-second 上 0.5 倍的改进会在产品的整个生命周期里日复一日复利。这就是为什么所有正经的 LLM 团队都至少有一个全职推理工程师，也是为什么开源社区在五年里出了四代推理引擎（FasterTransformer → DeepSpeed-Inference → vLLM → SGLang/TensorRT-LLM/llama.cpp）。

两个特性完全不同的阶段

每次 LLM 推理都是两个阶段：

  1. Prefill（处理 prompt）：把输入 token 并行过一遍模型，填 KV cache。算力受限。70B 上 4K token 的 prompt 大约 280 TFLOP——一张 H100 大约 70 ms 打满。
  2. Decode（生成）：每次出一个 token，对缓存的 K、V 做注意力。内存带宽受限。每个 decode step 把整个 KV cache（数 GB）读一遍，出一个 token。

不对称是关键。Prefill 跨用户 batch 得很好（同一个 kernel，不同序列）。Decode 在朴素 batch 下 batch 得 很差——每个用户在不同的序列位置。主流推理引擎全是围绕这个不对称建的。

常用经验：TTFT 由 prefill 主导，ITL 由 decode 内存带宽主导。要降 TTFT，往里堆 FLOPs（更多 SM、张量并行）。要降 ITL，往里堆内存带宽（HBM3 优于 GDDR、量化减参数）。

算术强度（arithmetic intensity）让这件事更严密。70B 模型 4K prompt prefill：模型权重（BF16 下 140 GB）加载一次，对 4096 个 token 操作。算术强度 ≈ 4096 FLOP / 每读一字节参数。同一模型单 token decode：权重加载一次，对 1 token 操作。算术强度 ≈ 1 FLOP/字节。H100 在 BF16 下 989 TFLOPS、HBM 3.35 TB/s，ridge point 约 295 FLOP/字节。Prefill 4096 FLOP/字节远在 ridge 之上（计算受限）。Decode 1 FLOP/字节低两个数量级（内存受限）。两个阶段想要不同的硬件特性，不分开处理它们的服务栈是把性能撒地上。

KV cache：撑起长上下文的数据结构

KV cache 存每一层每个历史 token 投影后的 K、V 向量。70B GQA-8 在 32K 上下文（数字来自第 1 篇）：

$$ \text{KV} = 2 \cdot 80 \cdot 2 \cdot 8 \cdot 128 \cdot 32{,}768 \cdot 2 \text{ 字节} = 8.6 \text{ GB} $$

每个请求。50 个并发请求就是 430 GB——远超模型权重。KV cache 是瓶颈，不是权重。

朴素实现按 max_context 给每个请求分配一块连续张量。两个失败：

  1. 内部碎片。 用 1K token 的请求仍占 32K 内存。
  2. 不能扩。 超出 max_context 就 decode 半路 OOM。

第三个失败在负载下出现：外部碎片。请求在不同时间到达和离开。空闲内存存在但散落在不连续的区域，没有一块够大放新请求。在变化负载下跑数小时的服务器光外部碎片就吃 20-40 % 的可寻址 KV 内存。这跟操作系统在 1960 年代用分页解决的问题完全一样，解法也一样。

Paged attention

vLLM 的杀手特性，2023 年论文 Kwon 等 ↗ ，paged attention。KV cache 按固定大小 block 分配（通常 16 个 token 一块），每个请求的 block table 把逻辑位置映射到物理 block。像操作系统的虚拟内存。

请求 A：47 token 的 KV → 3 块（16+16+15）
请求 B：200 token 的 KV → 13 块
A 的 block table：[0x47, 0x12, 0x3a]
B 的 block table：[0x05, 0x09, 0x21, ...]

好处：

  • 几乎零浪费：每个请求只有最后一块有内部碎片，最多 15 token。
  • 内存共享：前缀共享的请求（系统 prompt、few-shot 例子、tool 定义）能共用同一批物理 block。这就是 prompt caching 的来源。
  • 容易抢占：内存压力下把请求的 block 换到 CPU，有空间再拉回来。

vLLM 实测在 LLaMA-13B 上比朴素 HuggingFace Transformers 快 14-24 倍（原论文 benchmark）。这个数字至今站得住——不过实际生产差距严重看负载组合。Batch-1 单流服务赢面接近 2 倍；高并发混长度流量典型 5-10 倍；14-24 倍是跟 真朴素 的 HF 代码（完全不 batch）比出来的。重点不是具体倍数，是 paged attention 让 KV 内存从硬约束变成可支配的资源。

Block manager：分页实际怎么工作

Block manager 是每个 paged-attention 引擎的核心。它管一个物理 KV block 池子（通常每块 16 token × 2(KV) × 层数 × 头数 × head_dim 字节）、维护空闲列表、原子地为新请求分配 block。请求要扩（下一个 decode step），manager 弹出一个空闲 block、把物理地址追加到请求的 block table、返回。在 copy-on-write 语义下，前缀共享的 block 走引用计数：fork 序列（如并行采样、beam search、speculation 回滚）增 refcount；释放就减；refcount 归零才真正释放物理内存。

“换出 CPU” 路径在生产里很关键。GPU 没空闲 block 但有新请求排队时，scheduler 选一个受害请求（通常 FIFO 或最长运行的），把它的 block 通过 PCIe 拷到 pinned host memory、释放 GPU block。内存可用了再把请求拉回来。换出开销是真实的（PCIe Gen5 ×16 64 GB/s 意味着换出再加载一个 8 GB 请求要 100 ms），但它让系统不至于直接拒掉请求、能守住延迟 SLO。

Continuous batching

另一场革命。Static batching 等 batch 里最慢那条序列结束才开下一 batch。输出长度不齐（永远不齐），GPU 一堆空闲时间。

Continuous batching（Yu 等，Orca，OSDI 2022 ↗ ，vLLM 推广）：每个 decode step，赶走完成的序列，放进等待的。只要有活排队，GPU 永远在跑不少于 max_batch_size 条序列。

1000 个 prompt 输出长度从 50 到 2000 不等：

  • Static batching：总墙钟约等于 2000 token 输出的时间（最坏情况主导）。
  • Continuous batching：总墙钟约等于所有输出的平均。

跟 paged attention 复合，吞吐改进继续放大——半路放新序列只是分配新 KV block，整张张量不需要重新分配。

Orca 的 iteration-level 调度

Orca 论文引入了两个机制，到 2026 还在所有引擎里。Iteration-level scheduling 把 batching 边界跟请求边界解耦：每个 transformer iteration 独立调度，所以一个第 500 步结束的请求不卡住要 1500 步的请求。Selective batching 处理 prefill/decode 不匹配的问题——让 attention 算子按 per-sequence 形状跑，而 linear 层（不在乎位置）一起 batch。现代引擎进一步融合：vLLM 和 SGLang 都跑 “chunked prefill”，一个请求的 prefill 跟其他请求的 decode 在同一前向里交错，让 GPU 利用率即使 decode 队列稀疏也保持在 90 % 以上。

调度决策很关键：什么时候 batch 中途接新请求？朴素地：立刻。但接一个 32K token prompt 的 prefill 跟正在跑的 decode 一起，会把每 iteration 的延迟显著拉高（32K prefill 是单 decode step 的约 100 倍 FLOPS）。生产 scheduler 设每 iteration “预算” 上限、把长 prefill 切成 chunk、跟 decode step 交错。这是隐藏的取舍旋钮，决定你负载下 TTFT p99 是 500 ms 还是 5 秒。

Speculative decoding

聪明想法，现在已是标配。Decode 是内存受限：每 step 把整个权重读一遍出一个 token。如果能一次验证 N 个 token 呢？

Speculative decoding（Leviathan 等，2023 ↗ ）用 draft 模型（小、快——比如 1B）提议 $k$ 个 token。Target 模型（你真要服务的 70B）一次前向验证：它看提议的前缀，并行算 $k+1$ 个位置的概率。匹配 target argmax 的留下；第一个不匹配的及之后丢掉，从那里重采样。

数学：draft 命中率 $\alpha$、decode 内存受限（$k+1$ token 前向跟 1 token 前向开销差不多），期望加速 $(1 - \alpha^{k+1})/(1 - \alpha)$。$\alpha=0.7$、$k=4$：2.5 倍。

实际中 draft 通常从 target 蒸出来。三个家族在竞争：

  • Draft-target 双模型对，原始 Leviathan 形式：小蒸馏模型提议、target 验证。质量高但维护两个同步 checkpoint 是运营痛点。
  • Medusa（Cai 等，2024 ↗ ）：给 target 模型自己加 4-5 个额外预测头。每个头从同一 backbone 激活预测位置 $t+1, t+2, t+3, ...$。无需独立 draft 模型。加速 2-3 倍且无质量损失；在基础模型上几个 GPU-day 就能训出来。
  • EAGLE / EAGLE-2（Li 等，2024 ↗ ）：在倒数第二层 hidden state + 上一个 token 之上的自回归头。EAGLE-2 引入动态树验证，每 step 探索多个 draft 分支。多数负载 3-4 倍；当前自回归 speculation 的 SOTA。

注意：speculation 只有 target GPU 是内存受限时才有用，FLOPs 受限时不灵。大 batch 是 FLOPs 受限的（权重一次加载摊到 batch 里所有序列上），spec decoding 帮不上。单流低延迟服务是内存受限的，受益巨大。

正确性上一个微妙点：验证步对 target 分布做 拒绝采样。draft 对 token $t$ 的概率 $p_d(t)$、target 概率 $p_t(t)$，以 $\min(1, p_t(t)/p_d(t))$ 接受；拒绝时从修正分布 $\propto \max(0, p_t(t) - p_d(t))$ 重采。这保证输出分布 精确 是 target 的，跟 draft 质量无关。Spec decoding 是无损的：更差的 draft 只是接受率低、不是输出更差。这就是为什么团队上线 spec decoding 不需要质量回归测试——没有质量可回归。

量化：INT8、INT4、FP8

推理量化是 bf16 → INT8 / INT4 / FP8，缩内存和带宽。通常量化权重，有时也量化激活。

INT8 weight-only 量化

最简单。每个权重矩阵按 channel 量化：$w_q = \text{round}(w / s)$，$s$ 是 per-output-channel scale。推理时 on-the-fly 反量化回 bf16 再 matmul。

内存：权重缩 2 倍（16 bit → 8 bit）。质量：典型 <0.5 % perplexity 损失，无需校准。免费的赢面。

INT4：GPTQ vs AWQ

INT4 难得多。朴素 round-to-nearest 损 2-5 % perplexity。两个能跑的算法：

GPTQ（Frantar 等，2022 ↗ ）：OBQ（Optimal Brain Quantization）的扩展，对十亿参数量级模型可处理。一列一列量化权重矩阵；每列后用 Cholesky 分解的 Hessian 逆来更新剩余未量化列以补已引入的误差。数学上每列本质是一个有结构的最小二乘求解。开销：单 A100 上 70B 几小时，峰值内存约等于一层激活（典型 128 样本 × 2048 token 的小校准集）。

AWQ（Lin 等，2023 ↗ ）：观察到 1 % 的权重 “salient”（穿过它们的激活大）。给 channel 加 scale $s$ 保护这些 salient 权重不受量化噪声影响——等价于把输入激活乘 $s$、权重除 $s$ 再量化。在全精度下这是恒等变换、但重塑了量化地形。然后均匀量化。比 GPTQ 快（无 Hessian 求解，只是搜 scaling 因子）、多数模型质量略好、对 MoE 友好得多。

SmoothQuant（Xiao 等，2022 ↗ ）：跟 GPTQ/AWQ 正交——它处理 激活 量化。难点是激活有 outlier channel（少数 channel 幅值是其他的 100 倍）会毁了 INT8 激活量化。SmoothQuant 通过 per-channel scaling 把激活幅值迁到权重里，让两边都能 INT8 量化。作为 INT8 W8A8 量化前的预处理用。

LLaMA 级模型上 GPTQ 和 AWQ 都能做到 INT4 weight-only <1 % perplexity 损。MoE 上路由让校准复杂——不同 expert 看到的激活分布很不一样，统一 scaling 因子保护不了少用的 expert。Per-expert AWQ 是当前最佳实践。

FP8（以及 H100/H200 硬件路径）

H100 及更新的 GPU 有 FP8 tensor core，吞吐是 BF16 的 2 倍。FP8 推理是现代路径：权重存 FP8，计算时激活 cast 到 FP8，accumulation 走 FP32。质量损失可忽略（<0.1 %），FP8 比 INT8 动态范围更大。

FP8 有两种格式：E4M3（4 指数位、3 尾数）给激活和权重——范围小、精度高；E5M2（5 指、2 尾）给梯度和 KV cache——范围大、精度低。硬件把反量化 scaling 融进 matmul，所以没有 “INT4 反量化开销”；FP8 是 H100/H200 上的速度上限路径。

校准对 FP8 仍重要，虽然 perplexity 损极小。标准方法：在 128-512 校准样本上收激活统计、算 per-tensor 或 per-token scale、存入 checkpoint。Per-token 激活 scale（运行时从实际 batch 算）质量最好、额外延迟开销小。NVIDIA TransformerEngine 和 vLLM --quantization fp8 都实现了这个。

注意：FP8 要硬件支持。INT4/INT8 任何 GPU 都能跑。FP8 只 H100/H200/B200。如果你在 A100 上出货，INT8/INT4 是选项。

KV cache 量化

内存账单的另一半。压 KV cache 到 INT8（免费 2 倍），或 INT4（1-2 % 质量损，需仔细 per-token 校准）。vLLM 和 SGLang 都支持 FP8 KV cache；INT4 KV 是研究级，FlashInfer 有 kernel。

KV cache 走 FP8 是 2026 的主力。它砍内存一半（同 GPU 上 2 倍并发），<0.1 % 质量损。Per-token scaling 几乎免费，因为 scale 是 inline 算的。INT4 KV 需要更多注意——AWQ 给权重利用的 per-channel salience 结构不直接适用 KV（KV 是数据不是参数）。当前最佳实践是 per-token-per-head scaling 加 outlier 保留：约 1 % channel 留高精度，其余狠量化。

SGLang 与 RadixAttention

vLLM 的 prefix caching 在前缀完全相同的请求间共享 block。SGLang（Zheng 等，2024 ↗ ）用 RadixAttention 把这件事推广：所有活 KV block 的 radix 树，每条根到叶的路径代表一个 token 序列。新请求在树里查最长匹配前缀、共享那些 block、只算后缀。

为什么这比 vLLM 的精确前缀缓存重要：agent 负载会分支。一个 ReAct agent 可能发 5 个工具调用子 query，每个有相同的系统 prompt + tools + 对话历史但不同的短后缀。线性前缀缓存能让每个子 query 跟共享根命中，但不能让子 query 之间互相共享状态。RadixAttention 能。

SGLang 论文里报告的收益：agent 和结构化输出负载上比 vLLM 等价 prefix caching 快 5 倍吞吐，简单 chat 上约 1.5 倍。对不共享前缀的流量，赢面缩小。

实现大致是：维护以 token ID 为 key 的 radix 树，节点持 KV block 引用。新请求来时，匹配输入前缀尽可能深地走树；touched 节点的 refcount 加；为未匹配后缀分配新 block；完成时 refcount 减；内存压力上来 LRU 驱逐子树。数据结构开销跟节省比微不足道。

TensorRT-LLM 细节

NVIDIA 的 TensorRT-LLM 是第三大引擎。它的差异化：

  • 编译 kernel：每个模型针对具体 GPU、batch size 画像、序列长度编译成 TensorRT engine。编译 10-30 分钟；产物在它支持的负载上比 vLLM 运行时编译的等价物快 1.1-1.3 倍。
  • Plugin 模型：attention（FMHA、paged FMHA、FlashAttention-3 路径）、MoE 路由、量化的自定义 kernel 作为插件暴露。比 vLLM 的灵活性脆，但允许激进融合。
  • In-flight batching：TensorRT-LLM 版的 continuous batching，NVIDIA 调过的 scheduler。
  • 紧密 FP8 集成：TransformerEngine FP8 一等公民。H100 上 FP8，TensorRT-LLM 通常吞吐领先 5-15 %。
  • 痛苦的转换：每个新模型架构需要自定义 Python builder 脚本。新模型支持比 vLLM 滞后数周到数月。

如果你在 NVIDIA 硬件上以高规模服务一组固定的成熟模型且有工程带宽，选 TensorRT-LLM。否则 vLLM 的运营简单几乎总是胜出。

2026 年的服务框架选型

三个主选项：

  • vLLM——事实上的开源标准。社区最大、新模型新功能支持最快。Paged attention、continuous batching、speculative decoding、prefix caching 全部开箱可用。默认就是好的。没特殊原因就用它。
  • SGLang——更新（2024），结构化生成（约束 JSON、正则）、前端缓存（RadixAttention 做 prompt 树共享）、高扇出 agent 这类场景上更强。共享前缀负载的 TTFT 更低。
  • TensorRT-LLM——NVIDIA 自家。在它支持的模型上 H100 raw 吞吐最高（编译 kernel、融合 FlashAttention 路径），但转换痛苦、新模型支持慢。有 NVIDIA 支持、需要榨干每一 token/s 时选它。

vLLM 0.6+ 和 SGLang 都支持 speculative decoding、FP8、AWQ/GPTQ、MoE、multi-LoRA、function-call 约束解码。差距在收窄。默认 vLLM，结构化输出或 agent 重的场景换 SGLang。

服务并行模式

70B 模型 BF16 满质量需要 >140 GB。单 H100（80 GB）放不下。选项分到三个正交轴——张量并行（TP）、流水线并行（PP）、序列并行（SP）——它们的组合方式跟训练（第 4 篇）不一样。

张量并行（TP）

把每个权重矩阵切到多 GPU。经典 Megatron-LM 分解（Shoeybi 等，2019 ↗ ）：QKV 投影按列切（每 GPU 拿一部分注意力头）、注意力输出投影按行切（之后 all-reduce），然后 FFN 两个 matmul 按列、按行切。每个 transformer 层两次 all-reduce。

服务时，节点内 TP（NVLink，约 600-900 GB/s 双向）没问题；跨节点 TP（InfiniBand，单链约 50 GB/s）all-reduce 延迟受不了。标准上限 TP ≤ 8（一个节点）。

  • TP=2 H100（同节点 NVLink）。约 1.7-1.9 倍单卡吞吐（不是 2 倍——TP 引入 all-reduce 通信）。
  • TP=4 如果要长上下文（KV cache 也按 TP 切）。
  • INT4 量化，单 H100。比 TP=2 吞吐低，但便宜。

vllm serve Qwen/Qwen3-72B --tensor-parallel-size 2 一行搞定。

流水线并行（PP）

把层切到多 GPU。GPU 0 持层 0-19、GPU 1 持 20-39，等等。激活在流水线里流。经典问题是 流水线 bubble：流水线尾端的 GPU 等头端时空闲，反之亦然。服务时 PP 引入的 TTFT 延迟正比于 流水线深度 × 每层时间（GPipe / Megatron-LM 分析）；现代引擎用 micro-batching（把 batch 切成多个 micro-batch 在流水线里背靠背流）摊销。

PP 是推理的跨节点并行。节点内 TP、跨节点 PP 是 200B+ 标准部署。延迟敏感的负载多数团队封顶在单节点——PP bubble 是真的。

序列并行（SP）

非常长的序列上，激活本身一张 GPU 都放不下。序列并行按 token 维切，用 ring 或 all-to-all 通信做注意力 all-reduce。推理里 SP 在 512K+ 上下文出现，那时 per-layer 激活缓冲在 TP 之后还超过 GPU 内存。多数生产服务不需要。

数学：每种并行模式什么时候赢？

粗略地，TP 每层成本是 $2 \cdot \text{params}/\text{TP} / \text{HBM-BW} + 2 \cdot \text{batch} \cdot \text{seq} \cdot \text{hidden} / \text{NVLink-BW}$。第一项随 TP 线性缩；第二项增长。有甜点，通常 H100 上 TP=2 到 TP=4。超过 TP=8，all-reduce 主导你就输。70B 模型 TP=2 约 1.85 倍吞吐；TP=4 约 3.2 倍；TP=8 约 5.5 倍。边际递减是真的。

PP 单 batch 单 query 延迟成本约 $\text{深度} \cdot \text{每层时间}$。带 micro-batching 的稳态队列下，吞吐接近 $\text{batch} / (\text{每 stage 时间})$。PP 友好吞吐、敌对延迟。

我会真正部署什么

7B 级模型：单张 L40S 或 4090，FP8，vLLM，16K 上下文，80 % 利用率下 0.10-0.15 美元每百万 token。

32B 级模型：单 H100 + AWQ INT4 或 TP=2 L40S + FP8。两套都行。H100 单 token 更快；L40S 单时更便宜。按 $/Mtok 决策。

70B 级模型：TP=2 H100 + FP8。INT4 省 30 % $ 但损约 1 % 质量，生产里多数嫌多。

200B+ MoE：另一类优化问题（第 12 篇）。

可观测性：实际要量什么

生产里重要的指标不是 “tok/s”——那是单 batch 数字。真正的指标是负载下的 per-percentile 延迟。最简看板：

  • TTFT p50 / p95 / p99（ms）：从收到请求到吐第一个 token。p99 是用户感觉到的；p50 是用户拿来吹的。
  • ITL p50 / p95 / p99（ms）：稳态间 token 延迟。这里有尖峰意味抢占或内存压力。
  • 吞吐（tok/s） 在不同并发：1、8、32、128、512。画曲线；拐点是你饱和 FLOPs 或内存带宽的地方。
  • 队列深度 和 队列等待时间：请求堆积是因为引擎跟不上吗？
  • KV cache 利用率（%）：要开始抢占了吗？
  • Prefix cache 命中率（%）：低命中率意味你们 prompt 工程团队在生成不可共享的 prompt。
  • GPU SM 利用率 和 HBM 带宽利用率：从 dcgm-exporter 或 nvidia-smi dmon。两个都 <70 % 你有 headroom；HBM 95 %、SM 40 % 你内存受限，量化是答案。

明确设 SLO。典型生产目标：TTFT p95 < 500 ms、ITL p95 < 50 ms、错误率 < 0.1 %。超过这些就是火警。

成本算术

关键数字是 每百万输出 token 美元（$/Mtok-out）。自托管 vLLM 在单 H100（$2.50/小时按需、$1.20/小时 1 年预留、$0.80/小时 3 年预留或 savings plan）跑 Qwen3-32B FP8：

  • 并发 32 吞吐：约 7400 tok/s
  • 预留小时成本：$1.20/小时 = $0.00033/秒
  • $/Mtok-out = $0.00033 / 7400 \cdot 10^6 ≈ $0.045

对比 2026 年中期的 API 定价：

  • Claude-4.5-Sonnet：$15/Mtok 输出
  • GPT-5：$12/Mtok 输出
  • DeepSeek-V3.2：$1.10/Mtok 输出
  • Qwen3-Max API：$0.80/Mtok 输出

自托管开源权重和前沿 API 之间 30-300 倍的边际就是团队自托管的全部理由。陷阱是：每月 <100M token，工程成本主导节省。每月 >1B token，节省主导，自托管显然。

一个微妙坑：API 上输入 token 通常比输出 token 便宜 5-10 倍（因为它们 batch 得好、不自回归）。自托管下还是便宜，但没那么戏剧化（3-5 倍），因为两个阶段都跑你的硬件。假设输入输出价格平衡的成本模型会朝两个方向误导你。

小结与下一篇

推理是两个不对称阶段（prefill、decode）。现代服务栈——paged attention + continuous batching + speculation + 量化 + FP8 硬件——存在的目的就是让两个阶段的最坏情况都能忍。vLLM 是对的默认。量化（INT8 总开、预算紧 INT4、H100+ 走 FP8）几乎免费。Spec decoding 在低 batch 低延迟下 2-3 倍赢面，高吞吐场景是噪声。2026 年的成本算术强烈倾向于在规模上自托管开源权重。
