大模型工程（八）：检索增强生成 RAG

切分策略、dense vs sparse vs 混合检索、reranker 选型、2026 年长上下文 vs RAG 的取舍，以及 10 万文档以上才会冒头的失败模式。

RAG 是 LLM 应用中部署最多、工程最少的模式。2024 年的演示循环——用 text-embedding-3-large 嵌入所有内容，存进 pgvector，取 top-5 余弦相似度——适用于 1000 个文档和宽容的演示。但面对 10 万个真实文档和会注意到答案错误的客户时，这种方法就不管用了。我希望更多团队在构建第二代 RAG 之前了解这些。

原始 RAG 论文（Lewis 等，2020 ↗ ）将检索增强生成定义为混合模型：dense retriever（DPR）与 generator（BART）联合训练，以优化端任务准确率。到了 2026 年，生产环境中的 RAG 与 Lewis 的 RAG 已经大不相同——现代系统使用冻结的预训练 embedder、独立的 reranker 和不对 retriever 进行训练的 decoder-only 生成器。不过，核心思想（将知识与推理分开参数化）保留了下来，并成为主导范式。Gao 等（2023）RAG 综述 ↗ 对 2020 年后从 “Naive RAG → Advanced RAG → Modular RAG” 的演化过程进行了全面概述。

RAG 实际是什么

检索增强生成（RAG）：在查询时，从外部语料库中获取相关文本，将其放入 LLM 的上下文中，生成有依据的答案。“增强” 部分指的是 prompt 模板：

你是一个助手。仅根据下面的上下文回答用户问题。
如果上下文里没有答案，回答 "我不知道"。

上下文：
{retrieved_chunks}

问题：{user_query}

真正有趣的工程在于构建一个能够找到正确片段的检索器。这涉及三个子系统：切分、嵌入和排序。

切分是隐形杀手

文档的切分方式决定了检索器能找到什么。常见的 chunk 大小有 256、512 和 1024 个 token。常见的策略包括：

  • 固定大小：每 $N$ 个 token 切分。简单，但会破坏语义单元。
  • 句子：按句子边界切分。效果更好，但通常太短。
  • 递归字符：先按 \n\n 切分，再按 \n，最后按 . （LangChain 默认）。这是不错的基线。
  • 语义：使用滑动窗口嵌入，相似度下降时切分。效果更好，但成本更高。
  • late chunking（Günther 等，2024 ↗ ）：用长上下文 embedder 嵌入整个文档，然后在 embedding 序列 上切分——每个 chunk 的 embedding 包含周围文档的上下文。适合长文档，需要长上下文 embedder（如 Jina 或 BGE-M3）。

选择哪种方法取决于你的语料库。代码文件按函数或类切分。法律文件按条款切分。Markdown 文件按标题切分。PDF 文件中，表格和图片要单独解析，不要打断文本流。我调试过的失败案例大多是因为 “文档在表格中间被切开” 或 “答案跨越两个 chunk，但单个 chunk 无法理解”。

检查切分大小是否合理：挑选 20 个代表性问题，手动在语料库中找到答案并计算 token 数。如果大多数答案能在 512 个 token 内表达，就用 512。如果多数答案需要 1500 个 token 的上下文（例如法律合同），就用 1500 并设置 200 个 token 的重叠。

重叠很重要。没有重叠，从 chunk A 第 799 个字符开始、在 chunk B 第 821 个字符结束的句子会被截断。有了 100 个 token 的重叠，两个 chunk 都能包含完整的句子。

Late chunking：2024 年的突破

传统的切分方法独立嵌入每个 chunk。例如，“Q3 营收增长 12 %” 的 chunk 不携带公司或年份的信息。Late chunking 改变了顺序：将整篇文档（最多 8K-32K 个 token）输入长上下文 embedder，获取每个 token 的上下文化 embedding，然后在 chunk 边界内池化这些 token 的 embedding 得到最终的 chunk embedding。这样，“Q3 营收增长 12 %” 的 chunk embedding 就能反映 “Apple Q1 2024 财报” 的上下文。

报告表明，在长文档问答任务中，与相同 chunk 边界的传统切分方法相比，late chunking 可以提高 5-15% 的 NDCG，且不增加额外存储成本，索引延迟略有增加。Jina 的 jina-embeddings-v3 和 BGE-M3 都原生支持 late chunking。

什么时候用 late chunking？当文档足够长，以至于单个 chunk 内的局部上下文丢失了重要信息时，比如研究论文、法律合同、代码库或多部分的技术文档。对于短文档语料（如 FAQ 条目或产品描述），收益较小。

嵌入模型选型

嵌入模型把文本转换成向量，方便相似度搜索找到语义相关的片段。选择哪个模型直接影响质量、成本和延迟。

两个实战要点：

多语言很重要。 2022-2023 年的嵌入模型大多用英文训练，在中文、日文、阿拉伯语文本上检索效果差。如果语料库是多语言的，建议选择多语言模型——BGE-M3 和 Qwen3-Embedding 是开源中的佼佼者。

领域特定模型往往优于通用模型。 小型嵌入模型在法律领域的微调版本通常在法律语料上表现优于最先进的通用嵌入模型。如果你的语料库有明确的领域（如医疗、法律、代码或科学），可以考虑评估领域特定的嵌入模型或自己进行微调。

自托管嵌入模型越来越经济。单个 L4 GPU 上运行 BGE-M3 可以达到约 3000 个片段/秒。对于大多数小于 1000 万片段的语料库，不需要使用托管服务。

关于 维度：高维嵌入（3K-4K）通常检索效果更好，但存储成本更高，索引查找速度更慢。Matryoshka 表示学习方法（用于 text-embedding-3-large 和 Nomic 嵌入）训练嵌入模型，使不同长度的前缀（256、512、1024）都能有效检索，让你在部署时灵活调整维度而不需重新训练。对于 100K 文档的语料库，768-1024 维是一个实用的选择。

向量索引：HNSW、IVF 与取舍

有了向量表示后，需要一个向量索引来快速找到最近邻。目前主要有两种算法家族：

HNSW（Hierarchical Navigable Small World，Malkov & Yashunin, 2018 ↗ ）：构建多层图，每个节点连接大约 $M$ 个最近邻。查询从顶层入口开始，逐层贪婪下降，每一步都精化候选集。通过 ef_search 参数可以调整召回率（更多候选 = 更高召回率、更慢查询）。索引构建过程较慢（O(N log N) 图构造），内存消耗较大（约 1.5 倍的向量存储），但百万级数据查询速度在亚毫秒级别。

HNSW 是 pgvector、Milvus、Qdrant、Weaviate 和大多数现代向量数据库的默认选择。对于 1M 到 100M 向量的数据集，推荐参数为：M=16-32、ef_construction=200-400、ef_search=50-100。这些参数能在普通硬件上实现超过 95% 的召回率，并且查询延迟低于 5 毫秒。

IVF（Inverted File Index）：使用 k-means 将向量聚类到 $K$ 个中心点；查询时找到最近的几个中心点，只搜索它们的倒排表。这种方法内存效率高（没有图结构开销），但在召回率和速度之间的平衡不如 HNSW。FAISS 中使用了 IVF，并在此基础上发展出更复杂的变体，如 IVF-PQ（在 IVF 上进行产品量化以进一步压缩）。

FAISS（Johnson 等, 2017 ↗ ）是主要的实现库。它提供了多种原始索引方法（Flat、IVF、HNSW、IVF-PQ、IVF-SQ），并允许用户组合使用。生产环境中，团队通常用 FAISS 进行离线批量检索（如重排实验、训练数据构建），而在线服务则使用更高层次的向量数据库（如 Qdrant、pgvector）。

对于 100K 到 1M 向量的部署，选择哪种方法影响不大——pgvector 中的 HNSW 就足够了。当数据量达到 100M 以上时，选择取决于内存预算和分片数量。到了 10B 级别，就需要自定义基础设施和 Google/Meta 级别的工程支持了。

Dense vs sparse vs 混合

Dense 检索（基于神经 embedding 的余弦相似度）：擅长语义相似性，但对精确匹配（如缩写、ID 和稀有词）较弱。

Sparse 检索（如 BM25 或其现代变体 SPLADE）：擅长精确匹配和稀有词，但在同义词或改写方面表现不佳。BM25（Robertson 等，1995 ↗ ）是一种概率相关性模型，通过词频 × 逆文档频率并进行长度归一化来为文档打分。公式如下：

$$ \text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t,d) \cdot (k_1+1)}{f(t,d) + k_1 \cdot (1 - b + b \cdot |d|/\text{avgdl})} $$

标准参数是 $k_1 \approx 1.5$ 和 $b \approx 0.75$。BM25 在过去的 30 年里一直是主流的词法检索算法；尽管现代 dense embedder 出现了，但它在依赖特定 token 的查询（如产品 SKU、错误码和命名实体）上仍然是最佳选择。

混合检索 将两者结合后合并。到 2026 年，几乎所有生产中的 RAG 系统都会采用混合方式——相比纯 dense 检索，混合方式在大多数基准测试中 NDCG@10 提高了 10-30%，而成本却很低（BM25 很便宜，而且你已经有了这些片段）。

合并公式很重要。简单的加权和（score = 0.5 * dense + 0.5 * bm25）需要分数归一化，并且很挑剔。主流方法是 倒数排序融合（RRF，Cormack 等，2009 ↗ ）：

def rrf(rankings: list[list[str]], k: int = 60) -> list[str]:
    """rankings：来自不同检索器的排序后的 doc-id 列表。"""
    scores = defaultdict(float)
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            scores[doc_id] += 1.0 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

RRF 在分数维度上没有参数；常数 $k=60$ 经验上非常稳健。先分别独立运行 dense 和 sparse 检索，各取 top-50，然后用 RRF 合并，再取 top-20 进行重新排序。

RRF 之所以有效，是因为它基于排名而不是分数操作，因此不在乎 BM25 分数是无界正数，而 cosine 分数在 [-1, 1] 之间。任何检索器的排名第一对最终得分的贡献相同，无论底层检索器如何调整其置信度。这是组合具有不同分数分布的检索器时的正确不变量。

Reranking 是被忽略的英雄

Embedding 检索速度快但不够精确（毕竟这是有损压缩）。第二阶段的 cross-encoder reranker 会直接对每个候选和查询进行评分：联合编码 [query, candidate]，输出相关性分数。

Cross-encoder 的成本比 embedding cosine 高 100 倍，不适合在数百万个 chunk 上做检索。但在第一阶段检索出的前 20 个候选上使用 cross-encoder 是可行的，质量通常能提升 5-15%。

我会用以下模型：

  • BAAI/bge-reranker-v2-m3（开源，多语言，约 600M 参数，在 L4 上运行快）
  • cohere-rerank-v3（闭源，英文性能强，$2/1K queries）
  • jina-reranker-v2（开源，多语言，非常快）

from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")

def rerank(query, candidates, top_k=5):
    pairs = [[query, c.text] for c in candidates]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:top_k]]

整个流程是这样的：先通过 embed/sparse 检索出 top-50，再用 reranker 筛选出 top-5，最后把这些结果塞进 LLM 的上下文中。Reranker 是值得投入延迟预算的地方，它对质量的提升比调整 LLM 提示更明显。

Late interaction：ColBERT

Bi-encoder 的速度和 cross-encoder 的质量之间，有一种方法叫 late interaction（ColBERT，Khattab & Zaharia, 2020 ↗ ）。查询和文档各自独立编码成 per-token 向量（不进行池化），然后逐个 token 计算相似度：

$$ \text{score}(q, d) = \sum_{i} \max_j \langle q_i, d_j \rangle $$

ColBERT 保留了 token 级别的匹配，对稀有词特别有用，同时还能保持并行处理。ColBERTv2（Santhanam 等，2022）和 PLAID（Santhanam 等，2022）通过残差压缩和近似检索技术，使这种方法在百万文档规模下变得可行。BGE-M3 包含了一个 ColBERT 风格的组件，可以直接使用。

在 2025 到 2026 年的高精度检索生产栈中，late interaction 越来越受欢迎。当重新排序 50 个候选结果不够，但全量交叉编码又太昂贵时，这种技术就派上用场了。2024 年更新的 ColPali 将 ColBERT 的 late-interaction 原则扩展到了视觉-语言模型，用于文档图像检索。结果显示，token 级别匹配相比 OCR-then-embed 流程，在 PDF 和扫描件检索方面有显著提升。

Anthropic Contextual Retrieval 与 GraphRAG

2024 年，两项进展进一步提升了生产规模语料的 RAG 质量。

Anthropic Contextual Retrieval（2024）：在嵌入每个片段前，先生成一段 50 到 100 个 token 的上下文，说明该片段在文档中的位置（例如：“这个片段来自 Acme Corp 2023 年第三季度财报，在讨论供应链中断的部分”）。然后用密集嵌入器和 BM25 对这些带有上下文的片段进行嵌入，通过 RRF 混合检索并重新排序。Anthropic 在内部基准测试中报告，这种方法使检索失败率降低了 49%，结合重新排序后，降低到 67%。关键在于，上下文生成是一次性成本，在索引时完成，而不是每次查询时都做，这显著提高了嵌入的质量，使其更适合检索。

GraphRAG（微软研究院，2024 ↗ ）：不再将文档视为扁平的片段，而是在索引时提取实体和关系，构建知识图谱。然后在多个粒度上将图谱聚类成社区。查询时，对于跨多个文档的查询，返回社区总结，而不是原始片段。GraphRAG 在处理“全局”问题（如“这个语料库的主要主题是什么？”——单个片段无法回答的问题）时表现优于传统的 RAG。代价是更高的成本：图谱提取需要在索引时调用 LLM（每 1000 份文档约需 $1-10）。对于那些查询自然会跨文档聚合的语料库，GraphRAG 是当前的前沿技术；而对于事实检索查询，混合加重新排序的方法在成本和质量上仍然更优。

什么时候改用长上下文

第 6 章介绍过这个话题；这里是生产决策矩阵：

属性RAG 更优长上下文更优
语料规模大（>1M token）小（<200K token）
查询延迟预算<2秒5-30秒可接受
单查询成本$0.001-0.05$0.10-2.00
来源溯源重要不需要
问题局部性是（找一个事实）否（跨整篇综合）
更新频繁是否（每次更改需重新预填充）

2024 年流行的 “长上下文杀死 RAG” 的说法在成本上站不住脚。Claude-4.5-Sonnet 定价（$3/Mtok 输入）下，100K token 的 prompt 每次查询仅输入就需 $0.30。而 RAG 包括嵌入和 LLM 成本通常只需 $0.001-0.01。对于每天处理 100 万次查询的产品，RAG 的成本是 $300/天，而长上下文的成本则高达 $3000-10000/天。

正确的做法往往是 两者结合：先用 RAG 找出候选片段，再用长上下文处理这些候选集并生成综合提示。这才是大多数 “2026 生产 RAG” 系统的实际样子。

10 万文档以上的失败模式

Embedding 漂移。 三个月后，语料格式变了（新模板、不同词汇）。新文档检索效果差，因为它们的向量邻域不同。每月在留出测试集上重新评估检索质量。如果 MTEB 风格分数漂移超过 5%，就重新嵌入语料。

Chunk 边界切割。 单个连贯答案被切分到两个 chunk 中；单独看都不完整。症状：明确存在于语料中的问题检索召回率低。解决方法：使用重叠（100-200 个 token），结构化切分时尊重文档边界（如章节、段落），并抽样检查 chunk 是否独立有意义。Late chunking 通过让每个 chunk 的 embedding 携带上下文信息来部分解决这个问题。

查询分布变化。 用户开始问一些我没想到的问题。reranker 在不同类型的查询上训练过，表现下降。解决方法：记录查询，每周采样，人工标注，每季度重新训练 reranker。

热门 chunk 主导。 几个 chunk（例如退货政策 FAQ）在各种查询中都被频繁检索，不管相关性如何。这些 chunk 通常靠近簇心。解决方法：惩罚过度检索的 chunk（BM25 IDF 自然有这个效果；密集检索需要显式多样性项，如 MMR）。

只含元数据的 chunk。 一个只有 “Section 4.2.1” 的 chunk 可能在语义上匹配很多查询，但没有实际答案。解决方法：过滤掉内容密度低于阈值的 chunk（例如，拒绝少于 100 个字母数字字符或少于 5 个唯一非停用词的 chunk）。

摄入 bug 导致的重复 chunk。 我曾经遇到过一次，30% 的 chunk 是近似重复的，因为摄入脚本在一个子集上运行了两次。Top-K 检索返回同一 chunk 的多个版本；LLM 误以为有多个来源确认事实，从而自信地断言。解决方法：在摄入时去重（可以使用 MinHash、SimHash 或简单的归一化+哈希）。

Embedding 与文档更新滞后。 文档在源数据库中更新了，但 embedding 还没重新计算。检索返回的是旧版本的 chunk。解决方法：为每个文档跟踪 content_hash，hash 值变化时重新索引，并将索引与源的新鲜度作为 SLO 监控。

Reranker 过度自信。 Cross-encoder reranker 在标记为 “相关” 或 “不相关” 的对上训练——它输出的分数不是校准过的概率。在跨域情况下，0.9 分的 reranker 不一定比 0.6 分更准确。解决方法：不要直接对原始 reranker 分数设置绝对阈值；而是使用查询内的相对排名。

Eval：不量就什么都没意义

见过不少团队在没有 eval set 的情况下就开始用 RAG。然后几周里不断调整 prompt 和 chunk 大小，却没有任何迹象表明效果是否提升。先建一个 eval set，哪怕只有 50 道题。手动找到黄金 chunk 和黄金答案。然后跟踪以下指标：

  • 检索 recall@k：retriever 是否返回了包含答案的 chunk？
  • Rerank 后 precision@k：rerank 后 top-k 中有多少包含答案？
  • 答案忠实度：答案是否基于检索到的 chunk 推出（而不是 LLM 的先验知识）？
  • 答案正确性：与黄金答案匹配吗？

后两个指标用 LLM-as-judge 跑得不错；第 10 章会详细介绍。前两个是纯检索指标，容易计算，对调试最有帮助。

对于更大的 eval 集（超过 500 题），可以使用 RAGAS（RAG Assessment，2023 年开源）和 TruLens 这类工具，它们默认用 LLM-as-judge 自动化四指标流程。这些工具是不错的起点，但可以通过调整 prompt 来满足 judge，从而操纵它们计算的指标——黄金标准的人工标注子集仍然是唯一完全可信的信号。

生产架构推荐

2026 年典型的 100K-1M 文档 RAG 部署建议如下：

  • 存储：使用 Postgres 和 pgvector 存储向量和元数据。添加 pg_trgm 或 Elasticsearch 来实现 BM25。单区域写入，读副本用于查询。
  • Embedder：用 L4 GPU 自托管 BGE-M3 进行索引（一次性操作较多），Cohere 或 OpenAI API 用于查询嵌入（低延迟，无需基础设施）。
  • 检索：结合 dense 和 BM25 检索，使用 RRF 合并，选择 top-50 候选。
  • Reranker：在同一个 L4 GPU 上自托管 BGE-reranker-v2-m3（或使用 Cohere Rerank API 简化），将 top-50 缩减到 top-5。
  • 生成器：通用场景下使用 Claude-4.5-Sonnet 或 Qwen3-Max；成本敏感场景下自托管 Qwen3-32B。
  • 评估：准备 100-200 个手工标注的问题，每次部署时运行。如果 recall@10 或 faithfulness 下降超过 5%，就报警。
  • 监控：记录每个查询、检索结果、生成内容、延迟和成本。抽取 1% 的数据进行人工审核。

这条流水线在大多数负载下可以实现 <1s p95 延迟和 <$0.01 每次查询的成本，单个 Postgres 实例可扩展到约 1000 万文档，并且在高负载下能优雅地退化。

小结与下一篇

切分的重要性常被忽视，选择与答案实际位置相匹配的 chunk 大小至关重要。如果语料库是多语言的，就用多语言 embedder；如果有特定领域的 embedder，也请使用。始终采用混合方法（dense + sparse + RRF），并始终进行重排。长上下文在小规模语料和综合任务中表现更好，而 RAG 在成本和时效性上更胜一筹。在构建第二代系统前，先建立评估集。2024 年的新进展（Contextual Retrieval、GraphRAG、late chunking）将推动技术前沿，但基础仍然重要。
