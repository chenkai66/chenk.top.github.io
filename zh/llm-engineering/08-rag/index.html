大模型工程（八）：检索增强生成 RAG

切分策略、dense vs sparse vs 混合检索、reranker 选型、2026 年长上下文 vs RAG 的取舍，以及 10 万文档以上才会冒头的失败模式。

Read in English →

RAG 是 LLM 应用里被部署得最多、被工程得最少的模式。2024 年那套 demo 循环——text-embedding-3-large 全量嵌入、塞 pgvector、top-5 cosine——能撑 1000 个文档和一个宽容的 demo。它撑不住 10 万真实文档加一个会注意答案错误的客户。这一篇是我希望更多团队建第二代 RAG 之前就知道的事。

RAG 实际是什么

检索增强生成：查询时从外部语料拉相关文本、塞到 LLM 上下文里、生成有依据的答案。“增强” 的部分就是 prompt 模板：

你是一个助手。仅根据下面的上下文回答用户问题。
如果上下文里没有答案，回答 "我不知道"。

上下文：
{retrieved_chunks}

问题：{user_query}

有意思的工程在 “增强” 左边——构建一个能找到对的 chunk 的 retriever。三个子系统：切分、嵌入、排序。

切分是隐形杀手

文档怎么切决定了 retriever 能找到什么。常见 chunk 大小：256、512、1024 token。常见策略：

  • 固定大小：每 $N$ token 切。简单，破坏语义单元。
  • 句子：按句边界切。更好，常常太小。
  • 递归字符：先按 \n\n 切、再按 \n、再按 . （LangChain 默认）。好基线。
  • 语义：滑窗嵌入，相似度跌的地方切。更好，更贵。
  • late chunking：先嵌整篇文档，再在 embedding 序列上切。长文档最佳，需要长上下文 embedder。

对的答案看你的语料。代码：按 function/class 切。法律：按条款切。Markdown：按标题切。PDF：表格图单独解析，别让它们打断散文流。我 debug 的失败多数能追到 “文档切在表格中间” 或 “答案跨两个 chunk 但单独哪一个都说不通”。

切分大小自检：挑 20 个代表性问题，手工在语料里找答案，数 token。多数答案放得下 512 token 就用 512。多数需要 1500 token 上下文（如法律合同）就用 1500 加 200 token overlap。

# 一个合理的默认 chunker
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100,
    separators=["\n## ", "\n### ", "\n\n", "\n", ". ", " ", ""],
)
chunks = splitter.split_text(document)

Overlap 很关键。没 overlap，从 chunk A 第 799 字符开始、在 chunk B 第 821 字符结束的句子就被砸断了。100 token overlap 让两个 chunk 都包含完整句子。

嵌入模型选型

Embedding 把文本变成向量，相似度搜索能找到语义相关的 chunk。挑哪个模型决定质量、成本、延迟。

2026 年的 SOTA：

模型维度MTEB 平均备注
text-embedding-3-large（OpenAI）3072~64.6闭源，$0.13/Mtok
Cohere embed-v41024~67.1多语言强
BGE-M31024~69.4开源，多语言，dense+sparse+colbert
Qwen3-Embedding-8B4096~74.0开源，MTEB 顶
voyage-3-large1024~70.5闭源，premium
jina-embeddings-v31024~67.0开源，多语言

两个实战要点：

多语言很关键。 2022-2023 那一代 embedder 多数是英文训的，在中、日、阿语料上检索很差。多语言语料就挑多语言模型——BGE-M3 和 Qwen3-Embedding 是开源领头。

领域专项常常打败通用。 法律领域微调过的小 embedder 在法律语料上常常打败 SOTA 通用 embedder。语料有强领域（医、法、代码、科学），评估领域专项 embedder 或自己微调。

自托管 embedder 越来越便宜。BGE-M3 单 L4 GPU 约 3000 chunk/s。多数 <1000 万 chunk 的语料，不需要托管服务。

Dense vs sparse vs 混合

Dense 检索（神经 embedding 上的 cosine）：语义相似强，精确匹配（缩写、ID、稀有词）弱。

Sparse 检索（BM25 或现代变体如 SPLADE）：精确匹配和稀有词强，同义/改写弱。

混合检索 两者结合再合并。2026 年几乎所有生产 RAG 都是混合——比纯 dense 的赢面大（多数 benchmark 上 NDCG@10 高 10-30 %），代价小（BM25 便宜，chunk 已经在了）。

合并公式很关键。简单加权和（score = 0.5 * dense + 0.5 * bm25）需要分数归一化，挑剔。主流模式是 倒数排序融合（RRF）：

def rrf(rankings: list[list[str]], k: int = 60) -> list[str]:
    """rankings：来自不同 retriever 的排序后 doc-id 列表。"""
    scores = defaultdict(float)
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            scores[doc_id] += 1.0 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

RRF 在分数维度上无参数；常数 $k=60$ 经验稳健。dense 和 sparse 分别独立检索取各自 top-50，RRF 合并取 top-20，再 rerank。

Reranking 是被忽略的英雄

Embedding 检索快但不精确（这是有损压缩）。第二阶段 cross-encoder reranker 直接对 query 和每个候选打分：联合编码 [query, candidate]，输出相关性分数。

Cross-encoder 单对比 embedding cosine 贵 100 倍，所以不能拿来在百万 chunk 上做检索。但拿来在第一阶段检索的 top-20 候选上跑负担得起，质量持续提 5-15 %。

会用的模型：

  • BAAI/bge-reranker-v2-m3（开源，多语言，约 600M 参数，L4 上快）
  • cohere-rerank-v3（闭源，英文强，$2/1K queries）
  • jina-reranker-v2（开源，多语言，非常快）

from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")

def rerank(query, candidates, top_k=5):
    pairs = [[query, c.text] for c in candidates]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
    return [c for c, _ in ranked[:top_k]]

整条 pipeline：embed/sparse 检索 top-50 → rerank 到 top-5 → 塞 LLM 上下文。Reranker 是值得花延迟预算的地方；它持续比调 LLM prompt 更动质量针。

Late interaction：ColBERT

Bi-encoder 的速度和 cross-encoder 的质量之间是 late interaction（ColBERT）。query 和文档独立编码到 per-token 向量（不池化）。相似度按 token 算：

$$ \text{score}(q, d) = \sum_{i} \max_j \langle q_i, d_j \rangle $$

ColBERT 保留 token 级匹配（对稀有词好），同时仍可并行。ColBERTv2 和 PLAID 让它在百万文档规模可行。BGE-M3 自带 ColBERT 风格组件可以白嫖。

late interaction 在 2025-2026 高精度检索的生产栈里越来越多——50 个候选 rerank 不够、全量 cross-encode 又太贵的场景。

什么时候改用长上下文

第 6 篇引出过；这里是生产决策矩阵：

属性RAG 赢长上下文赢
语料规模大（>1M token）小（<200K token）
查询延迟预算<2s5-30s 可接受
单查询成本$0.001-0.05$0.10-2.00
来源溯源重要不需要
问题局部性是（找一个事实）否（跨整篇综合）
更新频繁是否（每改一次重 prefill）

2024 年时髦的说法 “长上下文杀死 RAG” 被证伪，光成本一项就站不住。Claude-4.5-Sonnet 价位（$3/Mtok 输入）的 100K token prompt 单查询光输入就 $0.30。RAG 包嵌入和 LLM 通常 $0.001-0.01。每天 100 万查询的产品，是 $300/天 vs $3000-10000/天。

正确答案常常是 两个都用：RAG 找候选 chunk，再用长上下文吞候选集 + 综合 prompt。这才是多数 “2026 生产 RAG” 实际长的样子。

10 万文档以上的失败模式

Embedding 漂移。 三个月后语料格式变了（新模板、不同词汇）。新文档检索不到，因为它们的向量邻域不一样了。防御：每月在留出测试集上重评检索质量。MTEB 风格分数漂移 >5 % 就重嵌语料。

查询分布漂移。 用户开始问你没预料的问题。reranker 在不同 query 类型上训过，表现下降。防御：log query、每周采样、人工标、每季度重训 reranker。

热门 chunk 主导。 几个 chunk（如退货政策 FAQ）跨各种 query 都被取出来，跟相关性无关。它们在簇心附近。防御：惩罚被过度检索的 chunk（BM25 IDF 自然就有这效果；dense 检索需要显式多样性项如 MMR）。

只含元数据的 chunk。 一个就 “Section 4.2.1” 的 chunk 在语义上匹配很多 query 但没答案。防御：滤掉低于最小内容密度的 chunk（如 <100 字母数字字符或 <5 个唯一非 stopword 拒掉）。

摄入 bug 造的重复 chunk。 我有过一次语料里 30 % 是近重复，因为摄入脚本在子集上跑了两遍。Top-K 检索返回同一 chunk 的 5 个版本；LLM 以为它有 5 个来源确认一个事实然后自信地断言。防御：摄入时去重（MinHash、SimHash 或简单的归一化+哈希）。

Eval：不量就什么都没意义

我见过太多团队没 eval set 就上 RAG。然后几周调 prompt 调 chunk 大小，没有任何信号告诉他们在不在变好。先建 eval set，哪怕 50 道题。手工找黄金 chunk 和黄金答案。然后跟踪：

  • 检索 recall@k：retriever 返回了包含答案的 chunk 吗？
  • Rerank 后 precision@k：rerank 后 top-k 里多少含答案？
  • 答案忠实度：答案是否能从检索到的 chunk 推出（不是 LLM 的先验）？
  • 答案正确性：跟黄金答案匹配吗？

后两个 LLM-as-judge 跑得不错；第 10 篇细讲。前两个是纯检索指标，好算，对 debug 最 actionable。

小结与下一篇

切分是被低估的旋钮，挑跟答案实际位置匹配的 chunk 大小。多语言语料用多语言 embedder，有领域用领域 embedder。永远走混合（dense + sparse + RRF）、永远 rerank。长上下文赢小语料和综合任务，其余 RAG 在成本和时效上赢。建第二代之前先建 eval set。
