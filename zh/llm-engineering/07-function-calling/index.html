大模型工程（七）：Function Calling 与工具使用

JSON 模式 vs function 模式 vs 自由格式、并行工具调用、用文法保证结构化输出、错误恢复模式，以及在真实负载里活下来的 agent loop。

Read in English →

Function calling 是 LLM 跟权重之外的世界之间的桥。它也是 chat template 细节（第 2 篇）、结构化输出 kernel（第 5 篇）、prompt engineering（第 9 篇）撞在一起的地方。这一篇讲底层到底在做什么、有什么保证可以靠、以及真实负载里能活的 agent loop 模式。

知识谱系很重要。LLM 工具使用追溯到 2022 年两篇近乎同时的论文：MRKL Systems（Karpas 等，AI21）提出在神经-符号模块间专家路由；ReAct（Yao 等，2022 ↗ ）把 chain-of-thought 推理跟工具动作交错。Toolformer（Schick 等，2023 ↗ ）展示了自监督地教工具使用——让模型在已有文本里插入 tool-call 标记来生成训练数据。到 2024 年每个前沿模型 post-training 数据都围绕工具使用格式构造，工具调用从 “研究 demo” 进入 “API 特性”。

“Function calling” 实际指的是什么

API 标 “function calling”，底下可能在做几件不同的事：

  1. 训出来的行为 + chat template 标记。模型 post-training 时被训成在合适时机吐 tool call，包在特殊 token 里。例：Qwen3 用 tool_call 标签、Mistral 用 [TOOL_CALLS]。
  2. JSON-mode 约束解码。输出被文法约束解码强制为合法 JSON。模型不一定为此训过；解码器 在执行约束。
  3. schema 引导的结构化输出。JSON-mode 加上 JSON 必须匹配特定 schema（function 名、参数类型）。
  4. 自由格式 prompt。系统 prompt 写 “请输出 JSON，键为 X、Y” 然后碰运气。能力强的模型还行，但没有保证。

生产系统四种混着用。OpenAI/Anthropic 的 API 用 1+3。vLLM 和 SGLang 给任何模型实现 2+3。自由格式（4）是其他都没有时的兜底。

一个微妙但重要的区分：JSON vs XML 工具格式。OpenAI 默认 JSON 工具调用；Anthropic Claude 内部按 XML 类结构化输出训练，API 上以 JSON 暴露。Anthropic 团队 2024 年公开的论点是 XML 标签更容易让模型学——尖括号结构跟训练数据怎么标特殊区域对齐，部分 XML 比部分 JSON 更容易在流中解析。经验上两种格式都能跑；选择主要影响解析工具链。模型内部两者看起来都是带学到的文法的 token 序列。

工具定义最佳实践

工具定义是伪装的 prompt。模型每次调用都把它们作为系统 prompt 的一部分读，定义质量直接影响模型是否选对工具、是否正确调用、是否能从错误恢复。一致能跑的模式：

Description 是你 prompt 里被读得最多的文本。 好的描述包含：(a) 工具一句话做什么，(b) 何时用（以及何时不用），(c) 返回什么。坏的描述只是把函数名重述。“搜索数据库” 是坏的；“在客户数据库里搜索匹配给定条件的订单。当用户问具体订单或订单历史时用。返回最多 10 个最近匹配。” 是好的。

参数描述比参数名更重要。 模型能从名字 location 推断要个地方，但不知道格式是 “Tokyo”、“Tokyo, Japan” 还是 “JP/Tokyo”——除非有描述。永远包含格式示例。

能用 enum 就用。 unit 参数带 enum: ["celsius", "fahrenheit"] 比自由字符串 unit 难误用得多。Schema 约束阻止模型造 “Kelvin” 或 “C°"。

棘手工具在描述里给示例调用。 “Example: transfer_money({from_account: 'A123', to_account: 'B456', amount: 100, currency: 'USD'})” 比三段散文更有用。

记录错误格式。 “账户不存在返回 404。调用方无权限返回 403。” 这让模型能正确解释错误响应、决定重试还是上报。

别过度定义工具。 我见过 80 个工具的注册表，模型经常选错。把相关工具组成更少的多态工具（如一个 query_orders 带可选 filter，而不是 query_orders_by_id、query_orders_by_date、query_orders_by_customer 等等）。模型填参数比从长菜单选好。

并行工具调用

2026 年所有前沿模型都支持并行 tool call——一轮里发多个 tool-use block。用户问 “东京和纽约的天气”，模型同时发两个 tool call，你并行执行，把两个结果一起回去。

为什么重要：串行 tool call 让延迟累加。5 个工具的 agent 串行调，每个 200 ms，纯工具延迟 1 秒。并行砍到 200 ms。对 agent（OpenClaw 的 7-12 章那种、任何多数据源的）来说，这是 “顺手” 和 “烦人” 的差别。

注意：并行调用只有在工具之间无依赖时才有意义。查两个城市的天气并行安全。“搜航班再订最便宜的” 不行——第二个调用依赖第一个的结果。模型应该从训练里知道，但不总知道。生产代码应该在并行执行前验证调用真的独立。

依赖分析可能微妙。两个工具都写同一外部资源（如对同一行的两个 update_database 调用），并行跑即使谁都不依赖对方 返回值 也引入 race condition。更安全的模式是为工具声明副作用类（read-only、write-isolated、write-shared），只在兼容类内并行。Anthropic 的 Claude 到 2025 年底比 GPT 类模型在并行调用上更保守，正是因为这个原因——不确定依赖时偏向串行。

用文法做结构化输出

Function-calling API 保证 JSON 良构。但你想要 JSON 每次都匹配一个特定 schema、零例外呢？

vLLM 和 SGLang 都实现了 文法约束解码。每个 decode step，把输出分布 mask 到只剩文法下能继续合法字符串的 token。实现可追溯到 Outlines（Willard & Louf，2023 ↗ ），它把 regex 和 JSON-schema 约束编译成有限状态机，每步对 logit 做 mask。更快的后继包括 XGrammar（SGLang 用）、llama.cpp 的 GBNF、微软的 Guidance 库。

# vLLM JSON schema 约束
from vllm.sampling_params import GuidedDecodingParams

schema = {
    "type": "object",
    "properties": {
        "city": {"type": "string"},
        "temp_c": {"type": "number"},
        "conditions": {"type": "string", "enum": ["sunny", "cloudy", "rainy"]},
    },
    "required": ["city", "temp_c", "conditions"],
}

params = SamplingParams(max_tokens=200,
                       guided_decoding=GuidedDecodingParams(json=schema))
out = llm.generate(prompts, params)
# out 保证可解析 JSON 且匹配 schema

这是最强的输出保证：不是 “通常是 JSON”，不是 “JSON 带正确键”，而是 “合法 JSON 严格符合 schema”。延迟代价小（XGrammar 约 3-5 % overhead）。

Token 级 masking：实际怎么工作

内部上，文法约束解码维护一个 状态机 跟踪当前生成在文法里的位置。每个 decode step：

  1. 模型在全词表（约 100K-150K token）上产 logit。
  2. 状态机算出从当前状态合法的延续 token 是哪些。对 JSON schema 来说可能是：“开括号和键名后，下一个 token 必须是 : 或空白。” 多数 token 都不合法。
  3. 不合法 token 的 logit 在 softmax 采样前置 -inf。
  4. 采到的 token 把状态机推进到新状态。

挑战是性能。朴素实现每步重算合法 token mask，是 $O(\text{vocab\_size})$——单 GPU 线程上每步约 0.5 ms，对小模型跟实际模型前向相当。Outlines 的洞见是给每个文法状态预算 valid-token 的位图（用 regex-to-FSM 编译）。XGrammar 进一步用字节码风格状态表示和增量 mask 更新。

现代实现 decode-step 开销 <2 %。编译开销（把 JSON schema 变成状态机）合理 schema 下通常 <100 ms，所以请求时可忽略。

一个微妙限制：文法约束影响 结构，不影响 内容。如果 schema 里没给 conditions 写 enum，模型可以写 "conditions": "elephant" 通过 schema 验证。Schema 约束不让输出 为真，只让它 可解析。

  • 把 JSON 包在 markdown fence 里。
  • 加前言（“好的，这是 JSON：…"）。
  • 多一个尾逗号。

import json, re

def parse_robust(text: str) -> dict:
    # 1. 直接 parse
    try:
        return json.loads(text.strip())
    except json.JSONDecodeError:
        pass
    # 2. 剥 markdown fence
    m = re.search(r"```(?:json)?\s*(.+?)\s*```", text, re.DOTALL)
    if m:
        try:
            return json.loads(m.group(1))
        except json.JSONDecodeError:
            pass
    # 3. 找第一个 { 到最后一个 }
    s, e = text.find("{"), text.rfind("}")
    if s != -1 and e != -1 and e > s:
        try:
            return json.loads(text[s:e+1])
        except json.JSONDecodeError:
            pass
    raise ValueError(f"Could not parse: {text[:200]}")

我每次自由格式结构化输出都跑这个 parser。我们生产 pipeline 失败率从约 3 % 降到 <0.1 %。

错误恢复模式

工具会失败。API 挂了、数据库返回了意外的 schema、function 超时、参数不合法。Agent 必须不打转地处理。

模式 1：tool error 当 tool result。 把错误信息 作为 tool result content 返回，而不是抛异常：

def execute_tool(name, args):
    try:
        return TOOLS[name](**args)
    except Exception as e:
        return f"Error: {type(e).__name__}: {str(e)}"

模型在上下文里看到错误，能决定重试、换工具、或优雅放弃。抛异常的话你就把模型踢出循环了。

模式 2：在工具里做业务校验，不在 schema 里。 Schema 管类型，业务逻辑管语义。transfer_money(from_account, to_account, amount) 工具应该用清晰的错误信息拒绝 amount=-100——schema 不知道你的业务规则。

模式 3：有界重试。 模型一直调同一个坏工具就终止循环。设 max_tool_calls=10 上限和 max_consecutive_errors=3 上限。我 debug 的 agent runaway 多数是模型在 “工具失败 → 同样方式重试” 循环里。

模式 4：不要展示 stack trace。 Python stack trace 几百 token，会把小模型搞糊涂。返回一句话错误描述，full trace 单独 log 留 debug。

模式 5：带理由重试。 模型重试工具时，前置一段简短推理解释什么变了。“上次调用失败因为日期格式错了；这次用 ISO 8601” 比只是用新参数重提更可靠。一些 agent 框架（LangGraph、CrewAI）通过在错误后注入反思 prompt 把这事内置了。

模式 6：求助。 不可恢复的错误，对的行为常常不是无尽重试而是上报给用户。“我尝试给那个地址发邮件但 SMTP 服务器返回 ‘invalid recipient’。能帮我再确认下地址吗？” 比 5 次静默重试后通用失败更好的 UX。

模式 7：优雅放弃。 Agent 用尽合理工具选项，返回带显式 caveat 的部分答案（“我没能拉到最新数据，所以这答案基于昨天的缓存信息”）比编造看起来完整的答案更好。

def run_agent(initial_message, tools, max_steps=20):
    messages = [{"role": "user", "content": initial_message}]
    for step in range(max_steps):
        response = client.messages.create(
            model=MODEL, tools=tools, messages=messages, max_tokens=4096,
        )
        messages.append({"role": "assistant", "content": response.content})

        tool_uses = [b for b in response.content if b.type == "tool_use"]
        if not tool_uses:
            return response  # final answer

        tool_results = []
        for tu in tool_uses:
            result = execute_tool(tu.name, tu.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tu.id,
                "content": str(result),
            })
        messages.append({"role": "user", "content": tool_results})
    raise RuntimeError("Agent exceeded max_steps")

够用 80 % 的 agent 用例。剩下 20 % 需要：

  • 工具输出截断：read_file 返回 10 万 token 会爆上下文。截到约 1 万带 “[truncated]” 标记。
  • 记忆压缩：第 15 步对话 5 万 token 了；继续之前把更早的 step 总结成单条消息。
  • 子 agent：复杂子任务（“研究 X”）拆给独立 agent 跑，只回最终总结。
  • Tool-call 流式：tool call 一边生成一边吐而不是等完整 response。降并行工具执行的 TTFT。

ReAct、Voyager 与 agent loop 的谱系

上面的最简循环是 ReAct 模式（Yao 等，2022 ↗ ）的精修。ReAct 交错三步：Thought（模型推理下一步行动）、Action（工具调用）、Observation（工具结果）。“thought” 步关键——给模型一个明确的位置来计划和自我修正。现代 agent loop 仍实现 ReAct，只是 thought 隐式发生在模型 tool-call 理由内部，而不是单独文本块。

Voyager（Wang 等，2023 ↗ ）给长程主体性给 ReAct 加了三件事：自动课程（agent 基于已知挑下一个子任务）、技能库（成功的 tool 使用模式存起来复用）、带环境反馈的迭代 prompting 循环。Voyager 在 Minecraft 里展示，但它的架构成了生产代码 agent（Cursor、Cline、Claude Code）和研究 agent 的模板。

Generative Agents（Park 等，2023 ↗ ）探索相关方向：带持久记忆流和反思式记忆 consolidation 的 agent。Park 的 Smallville 模拟显示 25 个 agent 加简单反应行为加记忆 + 反思就产生可信的涌现社交行为。记忆架构（embedding 检索 + 调度反思总结）现在是 SWE-bench 解和个人 AI 助手等长跑 agent 系统的标准。

2024-2026 演化：agent 拿到了自身返回丰富结构化数据的工具、递归子 agent 调用、跟执行步分开的显式任务规划步。OpenClaw “Memory-Planning-Tool-Reflection” 架构（OpenClaw 系列 7-12 章）是这个谱系的具体实例。

MCP：协议层

到 2024 每个框架有自己的 tool-spec 格式（LangChain tools、OpenAI function specs、Anthropic tool blocks 等）。每个都得为每个框架重新实现。Anthropic 2024 年底发布 Model Context Protocol（MCP） 作为连接 LLM 跟 tool 服务器的标准化 JSON-RPC 接口。

MCP 架构：client（LLM 应用如 Claude Desktop、Cursor、自定义 agent）跟 server（tool 提供者——文件系统、数据库、API 等）讲 JSON-RPC。Server 暴露三种 primitive：resource（如文件内容的只读上下文）、tool（可调函数）、prompt（可复用 prompt 模板）。协议处理发现、schema 校验、流式响应、认证。

MCP 重要因为它把工具开发跟 agent 框架选型解耦。你给内部 API 写一个 MCP server，它在任何 MCP 兼容客户端上工作，无需重写。到 2025 年中，MCP server 生态包含 GitHub、GitLab、Slack、Postgres、Sentry、Linear，以及数百个其他。协议在工具集成上成了 OpenAI chat-completions schema 之于推理 API 那样的——大家都对着的事实标准。

实战观察：MCP server 可以嵌入现有 app（你在代码库旁写个小 Python server），或独立服务跑。本地模式让 Claude Desktop 的文件系统访问安全——server 在用户机器上以用户权限跑、LLM 没有直接磁盘访问。这个本地性模型把 MCP 跟更老的 tool-server 协议区分开。

生产里出错的事

幻觉 tool 名。 模型调一个不存在的工具。修：验 tu.name in TOOLS 返回 tool-not-found 错误。一些模型（尤其 7B 以下小模型）幻觉的工具 几乎 跟真名匹配——get_weather_info vs get_weather。错误里建议最近匹配很有用。

幻觉 tool 参数。 模型造个 schema 里没有的 force=True。Schema 验证抓得到；返回清晰的 “参数 X 不支持” 错误。

自信地错的 tool result。 搜索工具返 “无结果”，模型还是幻觉答案。症状：模型用了 tool result 但它的断言跟结果矛盾。防御：系统 prompt 里明确提醒（“工具返回无结果就说无结果”）。高风险用例，用单独的模型对答案和工具脚本做后验证。

在 tool error 上打转。 上面模式 3。永远封顶。

Tool 延迟级联。 一个 30 秒的 tool 调用把用户面前的请求开着。每个 tool 调用都设超时。“这个工具慢，要不要不带它再试” 作为兜底 UX。

Schema 漂移。 你工具实际返回形状变了（字段重命名、新增 required 字段）但你给模型的 schema 定义没更新。模型按旧 schema 格式化请求，工具不透明地失败。修：在 dispatcher 里对 schema 校验工具输出，把不匹配作为版本偏差错误暴露。更好：从工具实现生成 schema 定义（如 Pydantic 模型反射到 JSON schema）让它们不能漂移。

工具脚本里的 token 预算耗尽。 20 步 agent 跑带工具调用和结果轻易到 50-100K token。模型撞上下文上限要么截断、要么开始丢更早上下文。修：实现记忆压缩（更早工具脚本老化前总结）、长分支用子 agent、监控每步 token 用量。

模糊工具选择。 描述重叠的两个工具（“搜索文档” vs “查找文档”）让模型振荡。修：写明确互斥的描述、或合并成一个带消歧参数的工具。

小结与下一篇

Function calling 是训出来的行为加 chat template 加可选文法强制。用最强的可用保证（schema 约束解码 > JSON mode > prompt 出 JSON）。模型支持就并行工具调用；有数据依赖就串行。把工具错误 作为 tool result 露出，让模型恢复。永远封顶循环；永远给工具超时；永远截断大输出。知识谱系跑 ReAct → Toolformer → Voyager → 现代生产 agent；协议谱系收敛到 MCP。

下一篇：Retrieval-augmented generation。切分策略、embedding 模型选型、混合检索（dense+sparse）、reranking，以及实战中长上下文 vs RAG 的决策。
