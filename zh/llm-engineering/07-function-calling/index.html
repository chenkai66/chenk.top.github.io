大模型工程（七）：Function Calling 与工具使用

JSON 模式 vs function 模式 vs 自由格式、并行工具调用、用文法保证结构化输出、错误恢复模式，以及在真实负载里活下来的 agent loop。

函数调用是 LLM 与外部世界之间的桥梁。它也是聊天模板细节（第 2 篇）、结构化输出内核（第 5 篇）和提示工程（第 9 篇）交汇的地方。本篇探讨底层机制、可以依赖的保证，以及在实际负载中有效的 agent 循环模式。

知识谱系很重要。LLM 的工具使用能力可以追溯到 2022 年两篇几乎同时发表的论文：MRKL Systems（Karpas 等，AI21）提出了神经-符号模块间的专家路由；ReAct（Yao 等，2022 ↗ ）将思维链推理与工具操作交错进行。Toolformer（Schick 等，2023 ↗ ）展示了自监督的工具使用教学方法——通过让模型在现有文本中插入工具调用标记来生成训练数据。到了 2024 年，每个前沿模型的后训练数据都围绕工具使用格式构建，工具调用也从“研究演示”变成了“API 特性”。

“Function calling” 实际指的是什么

API 提供 “function calling” 时，背后可能涉及几种不同的机制：

  1. 训练行为 + 聊天模板标记。模型经过后训练，在适当情况下生成工具调用，并用特殊标记包裹。例如，Qwen3 使用 tool_call 标签，Mistral 使用 [TOOL_CALLS]。
  2. JSON 模式约束解码。通过语法约束解码，强制输出为合法 JSON。模型不一定为此专门训练过；解码器负责执行这一约束。
  3. Schema 引导的结构化输出。在 JSON 模式基础上，进一步要求 JSON 符合特定 schema（函数名、参数类型）。
  4. 自由格式提示。在系统提示中写 “请以 JSON 格式响应，键为 X、Y” 并寄希望于模型。对于能力强的模型来说，这种方法有时也能奏效，但没有保证。

实际生产系统通常会混合使用这四种方法。OpenAI 和 Anthropic 的 API 结合了 1 和 3。vLLM 和 SGLang 则为任何模型实现了 2 和 3。自由格式（4）是其他方法不可用时的备选方案。

还有一个微妙但重要的区别：JSON 与 XML 工具格式。OpenAI 默认使用 JSON 工具调用；Anthropic Claude 内部则按 XML 类似的结构化输出进行训练，但在 API 中以 JSON 形式暴露。Anthropic 团队在 2024 年公开表示，XML 标签更容易让模型学习——尖括号结构与训练数据中标记特殊区域的方式一致，部分 XML 在流中也比部分 JSON 更容易解析。实际上两种格式都能工作；选择主要影响解析工具链。在模型内部，两者看起来都是带有学到的文法的 token 序列。

工具定义最佳实践

工具定义实际上就是提示词。每次调用时，模型都会将其作为系统提示的一部分读取，定义的质量直接影响模型是否能选对工具、正确调用以及从错误中恢复。以下是一些有效的模式：

描述是提示中最常被读取的部分。 一个好的描述应包括：(a) 一句话说明工具的功能，(b) 使用场景（以及不适用的场景），(c) 返回结果。差的描述只是重复函数名。“搜索数据库”就是一个差的例子；“在客户数据库中搜索符合给定条件的订单。当用户询问特定订单或订单历史时使用。返回最多 10 条最近的匹配记录。”则是一个好例子。

参数描述比参数名称更重要。 模型可以从 location 这个名字推断出需要一个地点，但没有描述的话，它不知道格式应该是 “Tokyo”、“Tokyo, Japan” 还是 “JP/Tokyo”。务必包含格式示例。

尽可能使用枚举。 带有 enum: ["celsius", "fahrenheit"] 的 unit 参数比自由字符串 unit 更难误用。Schema 约束可以防止模型发明 “Kelvin” 或 “C°"。

对于复杂的工具，在描述中提供示例调用。 “Example: transfer_money({from_account: 'A123', to_account: 'B456', amount: 100, currency: 'USD'})” 比三段文字说明更实用。

记录错误格式。 “账户不存在时返回 404。调用者无权限时返回 403。”这样可以让模型正确解释错误响应，并决定是否重试或上报。

不要过度定义工具。 我见过有 80 个工具的注册表，模型经常选错。可以把相关工具合并成少量多态工具（例如，一个带可选过滤器的 query_orders，而不是 query_orders_by_id、query_orders_by_date、query_orders_by_customer 等）。模型填充参数的能力比从长菜单中选择要强。

并行工具调用

到 2026 年，所有前沿模型都将支持并行工具调用——即在一轮中发出多个工具使用块。如果用户问“东京和纽约的天气怎么样？”，模型会同时发出两个工具调用，我并行执行这两个调用，并将结果一起返回。

这很重要：串行工具调用会累积延迟。一个需要调用 5 个工具的代理，每个调用耗时 200 毫秒，总延迟就是 1 秒。而并行调用可以将这个时间缩短到 200 毫秒。对于代理（比如 OpenClaw 的第 7 到 12 章提到的任何多数据源的应用），这是“响应迅速”和“令人沮丧”的区别。

需要注意的是，并行调用只适用于工具之间没有依赖关系的情况。查询两个城市的天气是安全的并行操作。“搜索航班，然后预订最便宜的那个”则不行，因为第二个调用依赖于第一个的结果。模型应该从训练中学到这一点，但并不总是如此。生产代码应在并行执行前验证这些调用是否真的独立。

依赖分析可能很微妙。如果两个工具都写入同一个外部资源（例如对同一行的两个 update_database 调用），即使它们不依赖对方的返回值，并行运行也会引入竞态条件。更安全的做法是为工具声明副作用类别（如只读、写隔离、写共享），并在兼容的类别内进行并行化。Anthropic 的 Claude 在 2025 年底比 GPT 类模型在并行调用上更为保守，正是出于这个原因——当不确定依赖关系时，它倾向于采用串行方式。

用文法做结构化输出

Function-calling API 能保证生成的 JSON 格式正确。但如果我希望每次生成的 JSON 都严格符合某个特定的 schema，该怎么办？

vLLM 和 SGLang 都实现了 文法约束解码。在每个解码步骤中，只保留那些能继续生成合法字符串的 token。这个实现源自 Outlines（Willard & Louf, 2023 ↗ ），它将正则表达式和 JSON-schema 约束编译成有限状态机，在每一步对 logit 进行掩码处理。后续更快的实现包括 XGrammar（SGLang 使用）、llama.cpp 的 GBNF 和微软的 Guidance 库。

# vLLM JSON schema 约束
from vllm.sampling_params import GuidedDecodingParams

schema = {
    "type": "object",
    "properties": {
        "city": {"type": "string"},
        "temp_c": {"type": "number"},
        "conditions": {"type": "string", "enum": ["sunny", "cloudy", "rainy"]},
    },
    "required": ["city", "temp_c", "conditions"],
}

params = SamplingParams(max_tokens=200,
                       guided_decoding=GuidedDecodingParams(json=schema))
out = llm.generate(prompts, params)
# out 保证可解析且严格符合 schema

这是最强的输出保证：不是“通常是 JSON”，也不是“JSON 带有正确的键”，而是“完全符合 schema 的有效 JSON”。延迟开销很小（使用 XGrammar 时约为 3-5%）。

Token 级掩码：实际工作原理

内部实现上，文法约束解码维护一个 状态机，跟踪当前生成的位置。每个解码步骤：

  1. 模型在全词表（约 100K-150K token）上生成 logits。
  2. 状态机计算从当前状态出发哪些 token 是合法的延续。对于 JSON schema 来说，这可能意味着：“在开括号和键名之后，下一个 token 必须是 : 或空白。” 大多数 token 都不合法。
  3. 不合法 token 的 logits 在 softmax 采样前被设为 -inf。
  4. 采样的 token 将状态机推进到新状态。

性能是个挑战。朴素实现每步重新计算合法 token 的掩码，复杂度为 $O(\text{vocab\_size})$——单个 GPU 线程上每步大约需要 0.5 ms，与小模型的实际前向传递时间相当。Outlines 的创新之处在于预先计算每个文法状态下的合法 token 位图（通过正则表达式到有限状态机的编译）。XGrammar 则进一步采用字节码风格的状态表示和增量掩码更新。

现代实现的解码步骤开销小于 2%。编译开销（将 JSON schema 转换为状态机）通常在合理 schema 下小于 100 ms，因此在请求时可以忽略不计。

一个小限制：文法约束影响的是 结构，而不是 内容。如果 schema 中没有为 conditions 定义 enum，模型可能会生成 "conditions": "elephant" 并通过 schema 验证。Schema 约束只能确保输出 可解析，不能保证其 真实性。

import json, re

def parse_robust(text: str) -> dict:
    # 1. 直接解析
    try:
        return json.loads(text.strip())
    except json.JSONDecodeError:
        pass
    # 2. 去掉 markdown 代码块
    m = re.search(r"```(?:json)?\s*(.+?)\s*```", text, re.DOTALL)
    if m:
        try:
            return json.loads(m.group(1))
        except json.JSONDecodeError:
            pass
    # 3. 找到第一个 { 和最后一个 }
    s, e = text.find("{"), text.rfind("}")
    if s != -1 and e != -1 and e > s:
        try:
            return json.loads(text[s:e+1])
        except json.JSONDecodeError:
            pass
    raise ValueError(f"无法解析: {text[:200]}")

每次自由格式结构化输出我都用这个解析器。生产流程中的失败率从约 3% 降到 <0.1%。

错误恢复模式

工具会出问题。API 挂了、数据库返回了意外的 schema、函数超时、参数不合法。Agent 必须妥善处理这些问题。

模式 1：将工具错误作为工具结果。 把错误信息 作为工具结果内容 返回，而不是抛出异常：

def execute_tool(name, args):
    try:
        return TOOLS[name](**args)
    except Exception as e:
        return f"Error: {type(e).__name__}: {str(e)}"

模型看到上下文中的错误后，可以决定重试、换工具或优雅地放弃。如果抛出异常，模型就无法继续参与循环了。

模式 2：在工具中进行业务验证，而不是在 schema 中。 Schema 负责类型检查，业务逻辑负责语义验证。transfer_money(from_account, to_account, amount) 工具应该用清晰的错误信息拒绝 amount=-100——schema 不知道你的业务规则。

模式 3：有界重试。 如果模型不断调用同一个有问题的工具，应终止循环。设置 max_tool_calls=10 和 max_consecutive_errors=3 的上限。我调试过的大多数 agent runaway 都是因为模型陷入了 “工具失败 → 同样方式重试” 的循环。

模式 4：不要显示堆栈跟踪。 Python 堆栈跟踪包含几百个 token，会让小模型困惑。返回一句简短的错误描述，完整的堆栈跟踪单独记录用于调试。

模式 5：带理由重试。 模型重试工具时，前置一段简短推理解释变化的原因。“上次调用失败因为日期格式错了；这次用 ISO 8601” 比单纯用新参数重试更可靠。一些 agent 框架（如 LangGraph、CrewAI）通过在错误后注入反思提示来实现这一点。

模式 6：请求帮助。 对于不可恢复的错误，正确的做法通常是上报给用户，而不是无尽重试。“我尝试给那个地址发邮件但 SMTP 服务器返回 ‘invalid recipient’。能帮我再确认下地址吗？” 比 5 次静默重试后通用失败更好的用户体验。

模式 7：优雅放弃。 如果 Agent 已经用尽了所有合理的工具选项，返回一个带有明确说明的部分答案（“我没能获取到最新数据，所以这个答案基于昨天的缓存信息”）比编造一个看似完整的答案更好。

def run_agent(initial_message, tools, max_steps=20):
    messages = [{"role": "user", "content": initial_message}]
    for step in range(max_steps):
        response = client.messages.create(
            model=MODEL, tools=tools, messages=messages, max_tokens=4096,
        )
        messages.append({"role": "assistant", "content": response.content})

        tool_uses = [b for b in response.content if b.type == "tool_use"]
        if not tool_uses:
            return response  # 最终答案

        tool_results = []
        for tu in tool_uses:
            result = execute_tool(tu.name, tu.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tu.id,
                "content": str(result),
            })
        messages.append({"role": "user", "content": tool_results})
    raise RuntimeError("Agent 超过最大步数")

这个循环能满足 80% 的 agent 用例。剩下的 20% 需要：

  • 工具输出截断：read_file 返回 10 万 token 会超出上下文限制。截到约 1 万 token，并加上 “[truncated]” 标记。
  • 记忆压缩：第 15 步时对话已达 5 万 token；继续之前，把更早的步骤总结成一条消息。
  • 子 agent：复杂子任务（如“研究 X”）交给独立的 agent 处理，只返回最终总结。
  • Tool-call 流式处理：一边生成一边发送 tool call，而不是等待完整响应。这样可以降低并行工具执行的 TTFT。

ReAct、Voyager 与 agent 循环的谱系

上面提到的最简循环是对 ReAct 模式（Yao 等，2022 ↗ ）的改进。ReAct 包含三个步骤：思考（模型推理下一步行动）、行动（工具调用）、观察（工具结果）。思考这一步很关键，它让模型有一个明确的地方来规划和自我修正。现代 agent 循环仍然采用 ReAct，只是思考过程隐含在模型的工具调用理由中，而不是单独的文本块。

Voyager（Wang 等，2023 ↗ ）为长程主体性扩展了 ReAct，增加了三项功能：自动课程（agent 根据已知选择下一个子任务）、技能库（存储并复用成功的工具使用模式）、带环境反馈的迭代提示循环。Voyager 在 Minecraft 中进行了展示，其架构成为生产代码 agent（如 Cursor、Cline、Claude Code）和研究 agent 的模板。

Generative Agents（Park 等，2023 ↗ ）探索了相关方向：具有持久记忆流和基于反思的记忆整合的 agent。Park 的 Smallville 模拟显示，25 个具有简单反应行为和记忆加反思的 agent 能够产生可信的涌现社交行为。这种记忆架构（基于嵌入的检索加上定期反思总结）现在是 SWE-bench 解决方案和个人 AI 助手等长期运行 agent 系统的标准。

2024-2026 年的发展趋势是：agent 使用能够返回丰富结构化数据的工具、递归子 agent 调用，以及将显式的任务规划步骤与执行步骤分开。OpenClaw “Memory-Planning-Tool-Reflection” 架构（OpenClaw 系列第 7-12 章）就是这个谱系的一个具体实例。

MCP：协议层

到 2024 年，每个框架都有自己的 tool-spec 格式（如 LangChain tools、OpenAI function specs、Anthropic tool blocks 等）。每种格式都需要为每个框架重新实现。Anthropic 在 2024 年底发布了 Model Context Protocol (MCP)，作为连接 LLM 和工具服务器的标准化 JSON-RPC 接口。

MCP 架构中，客户端（如 Claude Desktop、Cursor、自定义代理等 LLM 应用）通过 JSON-RPC 与 服务器（工具提供者——文件系统、数据库、API 等）通信。服务器暴露三种基本组件：资源（只读上下文，如文件内容）、工具（可调用函数）和 提示（可复用的提示模板）。该协议处理发现、模式验证、流式响应和认证。

MCP 很重要，因为它将工具开发与代理框架的选择解耦。如果你为内部 API 编写一个 MCP 服务器，它可以在任何兼容 MCP 的客户端上运行，无需重写代码。到 2025 年年中，MCP 服务器生态系统已经包括 GitHub、GitLab、Slack、Postgres、Sentry、Linear 以及数百个其他服务。这个协议在工具集成方面已经成为事实上的标准，就像 OpenAI 的 chat-completions 模式对于推理 API 一样。

实际应用中，MCP 服务器可以嵌入现有应用程序（你可以在代码库旁边编写一个小的 Python 服务器），也可以作为独立服务运行。本地模式让 Claude Desktop 的文件系统访问变得安全——服务器在用户的机器上以用户权限运行，LLM 没有直接磁盘访问权限。这种本地性模型使 MCP 与更老的工具服务器协议区分开来。

生产中常见的问题

虚构工具名称。 模型调用了不存在的工具。解决方法：验证 tu.name in TOOLS 并返回工具未找到错误。一些模型（尤其是 7B 以下的小模型）会生成几乎与真实名称相似的工具名，比如 get_weather_info 而不是 get_weather。在错误信息中建议最接近的匹配项很有帮助。

虚构工具参数。 模型发明了一个 schema 中不存在的 force=True 参数。Schema 验证可以捕捉到这个问题；返回清晰的 “参数 X 不支持” 错误。

自信地给出错误结果。 搜索工具返回 “无结果”，但模型仍然凭空生成答案。症状：模型使用了工具结果，但其结论与结果矛盾。防御措施：在系统提示中明确提醒（“如果工具返回无结果，请直接说明”）。对于高风险用例，用另一个模型对答案和工具日志进行后验证。

陷入工具错误循环。 参见上面的模式 3。始终设置上限。

工具延迟级联。 一个耗时 30 秒的工具调用让用户的请求长时间等待。每个工具调用都应设置超时。作为备用用户体验，可以提示用户 “这个工具很慢，要不要不带它再试一次？”

Schema 漂移。 工具的实际返回结构发生变化（如字段重命名或新增必填字段），但给模型的 schema 定义没有更新。模型按旧 schema 格式化请求，导致工具不透明地失败。解决方法：在调度器中校验工具输出与 schema 的一致性，并将不匹配情况作为版本偏差错误显示。更好的做法是从工具实现自动生成 schema 定义（例如，将 Pydantic 模型反射为 JSON schema），防止它们漂移。

工具脚本中的 token 预算耗尽。 包含 20 步 agent 运行、工具调用和结果的流程很容易达到 50-100K token。模型达到上下文限制后，要么被截断，要么开始丢失早期上下文。解决方法：实现记忆压缩（在早期工具脚本老化前进行总结）、长分支使用子 agent，并监控每一步的 token 使用量。

工具选择模糊。 描述重叠的两个工具（如 “搜索文档” 和 “查找文档”）会让模型摇摆不定。解决方法：编写明确且互斥的描述，或将它们合并成一个带有消歧参数的工具。

小结与下一篇

函数调用是训练行为加上聊天模板，再加上可选的语法强制。选择最强的保障方式（模式约束解码 > JSON 模式 > 提示生成 JSON）。如果模型支持，并行调用工具；如果有数据依赖，则串行调用。将工具错误作为工具结果展示，以便模型能够恢复。始终限制循环次数；始终为工具设置超时；始终截断大输出。知识谱系从 ReAct 到 Toolformer 再到 Voyager，最终发展到现代生产代理；协议谱系则逐渐收敛到 MCP。

下一篇：检索增强生成。内容切分策略、嵌入模型选择、混合检索（密集型 + 稀疏型）、重新排序，以及在实际应用中长上下文与 RAG 之间的决策。
