大模型工程（七）：Function Calling 与工具使用

JSON 模式 vs function 模式 vs 自由格式、并行工具调用、用文法保证结构化输出、错误恢复模式，以及在真实负载里活下来的 agent loop。

Read in English →

Function calling 是 LLM 跟权重之外的世界之间的桥。它也是 chat template 细节（第 2 篇）、结构化输出 kernel（第 5 篇）、prompt engineering（第 9 篇）撞在一起的地方。这一篇讲底层到底在做什么、有什么保证可以靠、以及真实负载里能活的 agent loop 模式。

“Function calling” 实际指的是什么

API 标 “function calling”，底下可能在做几件不同的事：

  1. 训出来的行为 + chat template 标记。模型 post-training 时被训成在合适时机吐 tool call，包在特殊 token 里。例：Qwen3（<tool_call>...</tool_call>）、Mistral（[TOOL_CALLS]）。
  2. JSON-mode 约束解码。输出被文法约束解码强制为合法 JSON。模型不一定为此训过；解码器 在执行约束。
  3. schema 引导的结构化输出。JSON-mode 加上 JSON 必须匹配特定 schema（function 名、参数类型）。
  4. 自由格式 prompt。系统 prompt 写 “请输出 JSON，键为 X、Y” 然后碰运气。能力强的模型还行，但没有保证。

生产系统四种混着用。OpenAI/Anthropic 的 API 用 1+3。vLLM 和 SGLang 给任何模型实现 2+3。自由格式（4）是其他都没有时的兜底。

并行工具调用

2026 年所有前沿模型都支持并行 tool call——一轮里发多个 ToolUseBlock。用户问 “东京和纽约的天气”，模型同时发两个 tool call，你并行执行，把两个结果一起回去。

为什么重要：串行 tool call 让延迟累加。5 个工具的 agent 串行调，每个 200 ms，纯工具延迟 1 秒。并行砍到 200 ms。对 agent（OpenClaw 的 7-12 章那种、任何多数据源的）来说，这是 “顺手” 和 “烦人” 的差别。

注意：并行调用只有在工具之间无依赖时才有意义。查两个城市的天气并行安全。“搜航班再订最便宜的” 不行——第二个调用依赖第一个的结果。模型应该从训练里知道，但不总知道。生产代码应该在并行执行前验证调用真的独立。

用文法做结构化输出

Function-calling API 保证 JSON 良构。但你想要 JSON 每次都匹配一个特定 schema、零例外呢？

vLLM 和 SGLang 都实现了 文法约束解码。每个 decode step，把输出分布 mask 到只剩文法下能继续合法字符串的 token。实现：Outlines（Python 正则编译器）、XGrammar（更快，SGLang 用）、llama.cpp 的 GBNF。

# vLLM JSON schema 约束
from vllm.sampling_params import GuidedDecodingParams

schema = {
    "type": "object",
    "properties": {
        "city": {"type": "string"},
        "temp_c": {"type": "number"},
        "conditions": {"type": "string", "enum": ["sunny", "cloudy", "rainy"]},
    },
    "required": ["city", "temp_c", "conditions"],
}

params = SamplingParams(max_tokens=200,
                       guided_decoding=GuidedDecodingParams(json=schema))
out = llm.generate(prompts, params)
# out 保证可解析 JSON 且匹配 schema

这是最强的输出保证：不是 “通常是 JSON”，不是 “JSON 带正确键”，而是 “合法 JSON 严格符合 schema”。延迟代价小（XGrammar 约 3-5 % overhead）。

一个微妙限制：文法约束影响 结构，不影响 内容。模型可能输出 {"city": "Tokyo", "temp_c": "warm", "conditions": "sunny"}——temp_c 是字符串不是数。等等，schema 角度这就不合法了；XGrammar 会把那个位置的字符串引号 token mask 掉、强制出数。但如果 schema 里没给 conditions 写 enum，模型可以写 "conditions": "elephant" 通过 schema 验证。Schema 约束不让输出 为真，只让它 可解析。

自由格式：没文法的时候

很多 API（多数非 OpenAI/Anthropic 提供商、设备端推理、内部服务）不暴露文法约束解码。兜底是 prompt 工程出结构化输出：

prompt = """输出 JSON 对象，包含以下键，不要 markdown，不要散文：
- "city"（string）
- "temp_c"（number）
- "conditions"（"sunny"、"cloudy"、"rainy" 之一）

查询：东京天气如何？
JSON:"""

能力强的模型（LLaMA-3.3-70B+、Qwen3-32B+）95-99 % 的时候照做。1-5 % 的失败模式通常是其中之一：

  • 把 JSON 包在 markdown fence 里（json ... ）。
  • 加前言（“好的，这是 JSON：…"）。
  • 多一个尾逗号。
  • JSON 跨多个段落。

import json, re

def parse_robust(text: str) -> dict:
    # 1. 直接 parse
    try:
        return json.loads(text.strip())
    except json.JSONDecodeError:
        pass
    # 2. 剥 markdown fence
    m = re.search(r"```(?:json)?\s*(.+?)\s*```", text, re.DOTALL)
    if m:
        try:
            return json.loads(m.group(1))
        except json.JSONDecodeError:
            pass
    # 3. 找第一个 { 到最后一个 }
    s, e = text.find("{"), text.rfind("}")
    if s != -1 and e != -1 and e > s:
        try:
            return json.loads(text[s:e+1])
        except json.JSONDecodeError:
            pass
    raise ValueError(f"Could not parse: {text[:200]}")

我每次自由格式结构化输出都跑这个 parser。我们生产 pipeline 失败率从约 3 % 降到 <0.1 %。

错误恢复模式

工具会失败。API 挂了、数据库返回了意外的 schema、function 超时、参数不合法。Agent 必须不打转地处理。

模式 1：tool error 当 tool result。 把错误信息 作为 tool result content 返回，而不是抛异常：

1
2
3
4
5

def execute_tool(name, args):
    try:
        return TOOLS[name](**args)
    except Exception as e:
        return f"Error: {type(e).__name__}: {str(e)}"

模型在上下文里看到错误，能决定重试、换工具、或优雅放弃。抛异常的话你就把模型踢出循环了。

模式 2：在工具里做业务校验，不在 schema 里。 Schema 管类型，业务逻辑管语义。transfer_money(from_account, to_account, amount) 工具应该用清晰的错误信息拒绝 amount=-100——schema 不知道你的业务规则。

模式 3：有界重试。 模型一直调同一个坏工具就终止循环。设 max_tool_calls=10 上限和 max_consecutive_errors=3 上限。我 debug 的 agent runaway 多数是模型在 “工具失败 → 同样方式重试” 循环里。

模式 4：不要展示 stack trace。 Python stack trace 几百 token，会把小模型搞糊涂。返回一句话错误描述，full trace 单独 log 留 debug。

def run_agent(initial_message, tools, max_steps=20):
    messages = [{"role": "user", "content": initial_message}]
    for step in range(max_steps):
        response = client.messages.create(
            model=MODEL, tools=tools, messages=messages, max_tokens=4096,
        )
        messages.append({"role": "assistant", "content": response.content})

        tool_uses = [b for b in response.content if b.type == "tool_use"]
        if not tool_uses:
            return response  # final answer

        tool_results = []
        for tu in tool_uses:
            result = execute_tool(tu.name, tu.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tu.id,
                "content": str(result),
            })
        messages.append({"role": "user", "content": tool_results})
    raise RuntimeError("Agent exceeded max_steps")

够用 80 % 的 agent 用例。剩下 20 % 需要：

  • 工具输出截断：read_file 返回 10 万 token 会爆上下文。截到约 1 万带 “[truncated]” 标记。
  • 记忆压缩：第 15 步对话 5 万 token 了；继续之前把更早的 step 总结成单条消息。
  • 子 agent：复杂子任务（“研究 X”）拆给独立 agent 跑，只回最终总结。
  • Tool-call 流式：tool call 一边生成一边吐而不是等完整 response。降并行工具执行的 TTFT。

生产里出错的事

幻觉 tool 名。 模型调一个不存在的工具。修：验 tu.name in TOOLS 返回 tool-not-found 错误。一些模型（尤其 7B 以下小模型）幻觉的工具 几乎 跟真名匹配——get_weather_info vs get_weather。错误里建议最近匹配很有用。

幻觉 tool 参数。 模型造个 schema 里没有的 force=True。Schema 验证抓得到；返回清晰的 “参数 X 不支持” 错误。

自信地错的 tool result。 搜索工具返 “无结果”，模型还是幻觉答案。症状：模型用了 tool result 但它的断言跟结果矛盾。防御：系统 prompt 里明确提醒（“工具返回无结果就说无结果”）。高风险用例，用单独的模型对答案和工具脚本做后验证。

在 tool error 上打转。 上面模式 3。永远封顶。

Tool 延迟级联。 一个 30 秒的 tool 调用把用户面前的请求开着。每个 tool 调用都设超时。“这个工具慢，要不要不带它再试” 作为兜底 UX。

小结与下一篇

Function calling 是训出来的行为加 chat template 加可选文法强制。用最强的可用保证（schema 约束解码 > JSON mode > prompt 出 JSON）。模型支持就并行工具调用；有数据依赖就串行。把工具错误 作为 tool result 露出，让模型恢复。永远封顶循环；永远给工具超时；永远截断大输出。
