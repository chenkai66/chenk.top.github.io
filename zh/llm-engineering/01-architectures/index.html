大模型工程（一）：从 Transformer 到 MoE 的架构演化

MHA、GQA、MQA 的取舍，Mixtral 与 Qwen3-MoE 的稀疏路由，滑动窗口注意力，以及 Mamba、RWKV 这条非注意力路径——每条路的代价和适用场景。

2017 年那张 Transformer 的方框图，到 2026 年还是所有生产级 LLM 的剪影，但里面几乎每一块都被换过、稀疏化过、特化过。这个系列从架构、训练、推理、检索、评估、安全到上线，一路走完现代 LLM 工程栈。第一篇讲方框本身：2026 年的注意力长什么样、MoE 怎么把参数量和 FLOPs 解耦、Mamba 和 RWKV 这类非注意力模型在哪里真正赢过 Transformer。

我假设你知道原始 Transformer。如果不熟，NLP 系列第 4 篇 已经讲过。这里只讲 变了什么。

相对于 2017 年的 “Attention Is All You Need” [Vaswani et al., 2017]，有 5 处替换：

  1. Pre-norm 替代 post-norm——梯度走干净的残差恒等通路，不用 warmup。原始 post-norm Transformer 必须配几千步小心翼翼的 LR warmup，不然前几步梯度就把 norm-then-residual 的路径炸掉。Pre-norm 在 GPT-2 里被推广，[Xiong et al., 2020] 给出了严格分析：从 step 0 就稳定，不需要 warmup。2020 年之后所有生产 LLM 都是 pre-norm。
  2. RMSNorm 替代 LayerNorm——不减均值，只除均方根。每层省一次 reduction。[Zhang & Sennrich, 2019] 证明 RMSNorm 在 Transformer FFN 上质量持平 LayerNorm，墙钟快 7-64 %。T5 和整个 LLaMA 系都采用了。2026 年只有少数遗留架构还在做均值中心化。
  3. SwiGLU 替代 GELU——带门控的 FFN，约 2-3 % perplexity 收益，FFN FLOPs 涨 50 %，划得来。[Shazeer, 2020] 系统扫了一遍 GLU 变体，SwiGLU（Swish gating）几乎在每个 benchmark 上都赢。约定：FFN inner dim 缩到 2/3，因为 GLU 把投影数从 2 个变成 3 个，缩 inner 才能保持总参数不变。
  4. RoPE 替代正弦位置编码——[Su et al., 2021] 提出旋转位置编码，把相对位置编码成 Q、K 在二维子空间里的旋转。2026 年所有长上下文模型走 128K-1M token，都是 RoPE 加上下文扩展技巧（NTK scaling、YaRN）。第 6 篇详细讲。
  5. GQA / MQA 替代 MHA——KV cache 小了一个数量级，质量几乎无损。长上下文场景下决定生死。

dense FFN 也越来越多被稀疏 MoE 替换，这是过去三年最大的架构变化，本篇一半篇幅讲它。

值得记一笔的历史：上面这些技术（pre-norm、RMSNorm、SwiGLU、RoPE、GQA）的名字看起来像一串干净的演进，但每一个在首篇论文之后都被争论过至少一年。Pre-norm 与 post-norm 的争论从 2018 持续到 2020。RoPE、ALiBi、NoPE [Press et al., 2022; Kazemnejad et al., 2023] 的论争 2022-2024 年还在跑——某些极端长度外推研究里 ALiBi 至今领先。2026 年的"已成定论"架构成定论，是因为它能上线，不是因为数学上盖棺。

注意力数学，认真讲

[Vaswani et al., 2017] 的 scaled dot-product attention：

$$ \text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{Q K^\top}{\sqrt{d_k}}\right) V $$

这条公式里有三处值得讲透。

为什么除 $\sqrt{d_k}$。 $Q$、$K$ 是输入经过线性层投影出来的，初始化让每列方差为 1。点积 $q \cdot k = \sum_{i=1}^{d_k} q_i k_i$ 是 $d_k$ 个单位方差项相加，所以方差是 $d_k$，标准差 $\sqrt{d_k}$。不除的话，$d_k$ 一大，点积幅度也变大，softmax 饱和到一行的某一列，softmax 的雅可比矩阵在 one-hot 处为零，梯度消失。除 $\sqrt{d_k}$ 让 pre-softmax logits 不管头维度多少都保持单位方差。这是整篇 Transformer 论文最重要的一个细节——不除模型根本训不起来。

为什么是 softmax、不是别的。 Softmax 可微、归一到概率单纯形、某一项主导时给出尖锐聚焦。它也是瓶颈。Linear attention [Katharopoulos et al., 2020] 把 $\text{softmax}(QK^\top)V$ 换成 $(\phi(Q)\phi(K)^\top)V$，借助矩阵乘结合律先算 $\phi(K)^\top V$，把代价从 $O(n^2 d)$ 降到 $O(n d^2)$。听上去很美，量起来打脸——稳定丢 2-5 个 perplexity 点，因为隐含的 kernel 跟语言对不上。最干净的研究是 [Schlag et al., 2021]，他们证明 linear attention 行为上像固定容量的关联记忆，远早于 scaled-dot-product 就饱和了。

为什么 FlashAttention 不只是个优化。 [Dao et al., 2022] 指出 vanilla attention 的 $O(n^2)$ 内存不在 FLOPs 上，而在物化 $n \times n$ 分数矩阵这一步上。FlashAttention 把 $Q$、$K$、$V$ tile 成能塞进 SRAM 的小块，每块在线做数值稳定的 softmax 归一，分数矩阵从不写回 HBM。结果：训练墙钟快 2-4 倍，内存从 $O(n^2)$ 降到 $O(n)$。FlashAttention-2 [Dao, 2023] 重排工作让 matmul 和 reduction 更好地重叠；FlashAttention-3 [Shah et al., 2024] 给 H100 加了异步 warp-specialized 调度。2026 年所有生产 LLM 训练栈（PyTorch SDPA、JAX TPU attention、vLLM、SGLang）调的都是 FlashAttention 派生 kernel。

FlashAttention 核心的 online softmax 技巧值得理解。标准 softmax 要扫两遍：一遍找 $\max$，一遍算 $\sum e^{x - \max}$。FlashAttention 一遍搞定，维护一对 $(m_t, \ell_t)$——当前的 max 和当前的 sum-of-exps，新块到来时增量更新。看到新最大值 $m_{t+1} > m_t$ 就把已有的和重新缩放：$\ell_{t+1} = e^{m_t - m_{t+1}} \ell_t + e^{x - m_{t+1}}$。这就是数值稳定流式统计里的标准技巧，搬到 attention 里的 softmax 上。输出在 fp32 累加精度内跟非 tile 的参考实现 bit-exact 等价。

GQA、MQA、MHA：KV cache 真正的代价

MHA 把每个 token 投影成 $h$ 组独立的 Q、K、V，每组维度 $d_{\text{head}}$。MQA 留 $h$ 组 Q，但 K、V 只有 一份，所有头共享。GQA 介于中间：$g$ 个分组，每组共享一份 K/V，跨 $h/g$ 个头。

长上下文下，KV cache 的内存才是关键。LLaMA-3-70B 的配置是 $h=64$、$d_{\text{head}}=128$、$L=80$ 层、FP16：

$$ \text{每 token KV 字节} = 2 \cdot L \cdot 2 \cdot h_{\text{kv}} \cdot d_{\text{head}} $$

import torch
import torch.nn.functional as F

class GQA(torch.nn.Module):
    def __init__(self, d_model=8192, n_heads=64, n_kv_heads=8, head_dim=128):
        super().__init__()
        self.n_heads, self.n_kv = n_heads, n_kv_heads
        self.head_dim = head_dim
        self.q_proj = torch.nn.Linear(d_model, n_heads * head_dim, bias=False)
        self.k_proj = torch.nn.Linear(d_model, n_kv_heads * head_dim, bias=False)
        self.v_proj = torch.nn.Linear(d_model, n_kv_heads * head_dim, bias=False)
        self.o_proj = torch.nn.Linear(n_heads * head_dim, d_model, bias=False)

    def forward(self, x):
        B, T, _ = x.shape
        q = self.q_proj(x).view(B, T, self.n_heads, self.head_dim).transpose(1, 2)
        k = self.k_proj(x).view(B, T, self.n_kv,    self.head_dim).transpose(1, 2)
        v = self.v_proj(x).view(B, T, self.n_kv,    self.head_dim).transpose(1, 2)
        # 把 K/V 复制到 Q 的头数（每组 n_heads/n_kv 个头共享一份 KV）
        k = k.repeat_interleave(self.n_heads // self.n_kv, dim=1)
        v = v.repeat_interleave(self.n_heads // self.n_kv, dim=1)
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        return self.o_proj(out.transpose(1, 2).reshape(B, T, -1))

代码里那个 repeat_interleave 只是逻辑示意。真正的 kernel（FlashAttention-2、FlexAttention）会在 SRAM tile 内部直接 broadcast，不会把 K/V 物化出来。

为什么偏偏是 GQA-8。 分组数是经验值，不是理论值。Ainslie 等扫了 $g \in \{1, 2, 4, 8, 16, 64\}$ 在 T5-XXL 上，发现 $g=8$ 到 $g=4$ 之间质量明显掉档，$g=2$ 到 $g=1$（MQA）再掉一档。$g=8$ 时验证 perplexity 跟 MHA 差 0.05——run-to-run 噪声以内。$g=1$ 时差 0.4——下游任务能感觉到。LLaMA-2-70B 直接照搬这篇论文挑了 $g=8$，LLaMA-3 沿用，Qwen3 沿用。8 不神奇，是经验拐点。

Multi-query latent attention（MLA）。 DeepSeek-V2 [DeepSeek-AI, 2024] 提了不同的 KV 节省思路。不在头之间复制 K/V，而是把 token 投影到一个小的 latent 维度 $d_c \ll h \cdot d_{\text{head}}$（一般 $d_c = 512$），cache 只存这个 latent 向量，attention 计算时再投影回每头的 K、V。每 token 每层 KV cache 降到大约 $d_c$ 字节——比 MQA 还小——而注意力质量持平 MHA。代价是 attention 时多算一些，但内存瓶颈的长上下文推理这个 trade 划算。DeepSeek-V3 生产用 MLA。2026 年开放前沿任何新出的 100B+ 长上下文模型，MLA 是默认架构选择。

KV cache 量化压缩。 跟架构选型正交，可以把 KV cache 用 INT8 或 INT4 存。KIVI [Liu et al., 2024] 和 FP8-KV 显示 INT4 KV 配 per-channel 非对称量化在大多数 benchmark 上几乎无损，节省 4 倍内存。叠加 GQA-8 总共相比 MHA-FP16 省 32 倍 KV。vLLM、SGLang、TensorRT-LLM 在 2026 年都把 INT8/FP8 KV cache 作为可选项发布。

滑动窗口注意力

Mistral-7B 引入 滑动窗口注意力（SWA） [Jiang et al., 2023]，窗口 4096。每个 token 只看前 $w$ 个。深度叠加之后感受野仍然线性增长——第 $L$ 层第 $n$ 位置的 token 能看到回 $L \cdot w$ 个 token——所以 32 层 $w=4096$ 的模型有效感受野 13 万 token。

SWA 把 KV cache 锁死在 $w$ 个 token 上，不管上下文多长。代价是模型必须学会用层叠的感受野，这意味着 SWA 模型在需要精确长程检索（“大海捞针”）的任务上常常不如全注意力模型。2026 年大多数长上下文模型把 SWA 和 attention sinks [Xiao et al., 2024] 一起用（Mistral-Large、Qwen3）——每层始终保留前 4-8 个 token 在窗口内，长上下文行为会稳很多。第 6 篇讲。

Attention sink 这个现象是现代 LLM 里最怪的实验观察之一。Xiao 等发现，流式推理（固定窗口前移、丢掉滑出窗口的 token）几千 token 之后会灾难性发散——perplexity 飙到几千。修法离谱地简单：永远不丢前 4 个 token。保留 sink 后，几百万 token 长度上 perplexity 仍然平稳。机理解释是 softmax 必须求和到 1，训练时所有"多余的"注意力质量没有有意义的目标，就被倾倒在前几个 token 上。把这些 token 丢掉就丢掉了倾倒处，注意力分布炸开。当代模型预训练时显式保留 sink，规避这个脆弱性。

Mixture of Experts：参数翻倍，FLOPs 不变

MoE 的核心戏法：把 dense FFN 换成 $E$ 个 FFN（“专家”）加一个小路由器，每个 token 选 $k$ 个。总参数随 $E$ 涨，单 token FLOPs 只随 $k$ 涨。

MoE 在深度学习里的血统比当代浪潮早得多。[Jacobs et al., 1991] 把 “mixture of experts” 提为集成想法。[Shazeer et al., 2017] 把它拉到 137B 参数语言模型规模，配稀疏 top-$k$ 门控——这篇是当代每个稀疏 MoE 设计的直系祖先。GShard [Lepikhin et al., 2020] 和 Switch Transformer [Fedus et al., 2022] 在 TPU 上把 expert parallelism 产品化。2024-2026 这一波（Mixtral、DeepSeek、Qwen3）是第三代：top-$k$ 路由 + auxiliary-loss-free 均衡 + 细粒度专家（更多更小）+ shared expert 设计。

Mixtral-8x7B 是教科书例子。每层 8 个专家，top-2 路由，32 层。总参数 46.7B（不是 56B，注意力是共享的）。激活参数 12.9B。也就是说你付 12.9B FLOPs 的算力换 46.7B 模型的建模能力——参数 / 算力比 3.6 倍。

Qwen3-235B-A22B（2025 年底发布）把这个比例拉到 235B / 22B，10.7 倍。DeepSeek-V3 [DeepSeek-AI, 2024] 是 671B / 37B，18 倍——目前生产级模型的最高比。

class MoEBlock(torch.nn.Module):
    def __init__(self, d_model, d_ffn, n_experts, top_k):
        super().__init__()
        self.gate = torch.nn.Linear(d_model, n_experts, bias=False)
        self.experts = torch.nn.ModuleList([
            SwiGLU(d_model, d_ffn) for _ in range(n_experts)
        ])
        self.top_k = top_k

    def forward(self, x):                        # x: [B, T, d]
        scores = self.gate(x)                    # [B, T, E]
        topk_w, topk_i = scores.topk(self.top_k, dim=-1)
        topk_w = F.softmax(topk_w, dim=-1)
        out = torch.zeros_like(x)
        for k in range(self.top_k):
            idx = topk_i[..., k]
            w   = topk_w[..., k:k+1]
            for e in range(len(self.experts)):
                mask = (idx == e)
                if mask.any():
                    out[mask] += w[mask] * self.experts[e](x[mask])
        return out

这段简化代码隐藏了两件生产里要命的事。

负载均衡。 路由器要是总挑同一个专家，你就只是花了多余参数堆出一个 dense 模型。常用做法是加辅助损失，惩罚不均衡——Switch Transformer 的 $\ell_{\text{aux}} = \alpha \cdot E \cdot \sum_e f_e \cdot p_e$，$f_e$ 是路由到专家 $e$ 的 token 占比，$p_e$ 是路由概率均值。DeepSeek-V3 干脆放弃辅助损失，改用 auxiliary-loss-free 方案 [Wang et al., 2024]：每个专家加一个偏置 $b_e$ 加到 gate logits 上，按梯度更新让 $f_e$ 自然均衡。更干净，质量不掉。

专家并行。 256 个专家（DeepSeek-V3）一张卡装不下。专家被切到多卡上，token 经 all-to-all 路由过去。all-to-all 的延迟就是瓶颈——DeepSeek 自家 DeepEP ↗ 在 NVLink 上能跑到 156 GB/s，已经接近硬件天花板。

MoE 的硬伤：总 VRAM 跟总参数走，激活的只有 $k$ 个不影响这件事。DeepSeek-V3 一张 80 GB H100 跑不动——大约要 700 GB 权重内存。激活参数等价说的是 算力，不是内存。

MoE 数学：路由、容量、均衡的细节

路由器是线性映射 $g(x) = W_g x \in \mathbb{R}^E$，后面接 top-$k$ 子集上的 softmax。专门 top-$k$，不是"挑超过阈值的所有专家"——单 token FLOPs 必须确定，kernel 才能跑。Mixtral 用 $k=2$，DeepSeek-V3 用 $k=8$（256 专家 + 1 个常驻共享专家），Qwen3-MoE 用 $k=8$（128 专家）。

Switch Transformer 的辅助损失分解成两个因子。设 $T$ 是 batch 的 token 集，$E$ 是专家数。定义 $f_e = \frac{1}{|T|} \sum_{t \in T} \mathbb{1}[\arg\max_e g(x_t) = e]$（top-1 落到专家 $e$ 的 token 占比），$p_e = \frac{1}{|T|} \sum_{t \in T} \text{softmax}(g(x_t))_e$（专家 $e$ 的平均路由概率）。损失是

$$ \ell_{\text{aux}} = \alpha \cdot E \cdot \sum_{e=1}^{E} f_e \cdot p_e. $$

$f_e = 1/E$ 且 $p_e = 1/E$ 时取最小。两个量乘起来这个设计很聪明：$f_e$ 单独用有不可微的 arg-max，$p_e$ 单独用可微但不惩罚硬不均衡。乘起来让路由器在软概率和硬分配两边都学会均衡。

专家容量 是单批里任一专家能接的 token 上限。容量 $c \cdot |T| \cdot k / E$（容量因子 $c$），路由到满员专家的 token 被 丢弃——绕过 FFN，零贡献。Switch Transformer 用 $c=1.25$。$c$ 小省内存和 all-to-all 带宽，但丢更多 token；$c$ 大避免丢弃但浪费容量。当代训练（DeepSeek-V3、Qwen3）训练时常用 $c=1.0$ 配 aux-loss-free 均衡，推理时 $c=\infty$（推理 batch 小，几乎碰不到容量）。

[Wang et al., 2024] 提的 auxiliary-loss-free 方案，DeepSeek-V3 采纳，把 aux loss 替换成路由 logits 上每专家加一个偏置再做 top-$k$：

$$ g'_e(x) = g_e(x) + b_e,\qquad b_e \leftarrow b_e - \eta \cdot \text{sign}(f_e - 1/E) $$

每批之后偏置更新，把 token 推向利用不足的专家。最终 softmax 权重（凸组合用的）仍然来自原始的 $g_e(x)$——只有 分配 用 $g'_e$。这把"哪个专家"和"权重多少"解耦，避免了 aux loss 跟模型自然路由偏好打架时的质量惩罚。

趋势很清楚：更多更小的专家、相对总数更少的 top-$k$ 激活。Mixtral 的"8 个大专家挑 2"是 OG 稀疏设计。DeepSeek-V3 的"256 个小专家挑 8 + 1 个常驻"是当下前沿。直觉是：细粒度专家允许更专的特化；常驻共享专家吞下不需要路由的通用模式（把路由专家的容量留给真正特化的工作）。

DeepSeek-V2 提的 “shared expert” 概念值得一段话。没有它的话，高频枯燥模式（英文功能词、常见代码套路）跟稀有特化模式抢路由位，路由器不得不为每种都学一份浪费的映射。有它之后，共享专家吞下普适模式，路由专家专心特化。经验上路由熵砍半（路由更果断），下游质量平均涨 1-2 %。

Mixtral 的"8 专家 top-2"部分是为了单 8 卡节点的推理效率——一卡一专家，top-2 意味着每个 token 在 all-to-all 里激活 2 张卡。DeepSeek 的"256 专家 top-8"需要更复杂的 expert parallelism，但负载分布更均匀，能拿到更高稀疏比。架构里编码了不同的推理部署假设。

混合架构：Jamba、Zamba、Samba

纯注意力 $O(n^2)$。纯状态空间（Mamba）$O(n)$ 但 copy 类任务掉分。自然答案是混。

Jamba [Lieber et al., 2024] 是第一个广泛部署的混合。Jamba 块交替：7 层 Mamba 接 1 层 attention，循环。Jamba-1.5-Large 32 层里 4 层 attention、28 层 Mamba。再叠 MoE——FFN 是稀疏的，16 个专家 top-2 路由。结果：398B 总参数、94B 激活、256K 上下文、长上下文推理比同档 dense Transformer 快约 5 倍。

Zamba [Glorioso et al., 2024] 在多层 Mamba 里穿插单个 共享 attention 块。共享 attention 摊销参数代价——不是 N 层 attention，而是 N 个引用指向 1 个 attention。Zamba-7B-v2 用 Mamba-2 层加 1 个跨网络多深度引用的共享 attention。这种设计省 30 % 参数，代价是稍多算力（共享块跑 N 次）。

Samba [Ren et al., 2024] 是最激进的混合：Mamba 和滑窗 attention 1:1 交替。3.8B 的 Samba 模型在多数 benchmark 上跟 Phi-3-3.8B 持平，且能干净外推到 100 万 token 上下文——纯 Transformer 即使配 RoPE 扩展也很难。

三个混合架构的经验教训：只要有少量（10-50 %）attention 层，就能补回纯 Mamba 在 copy/lookup 任务上的损失，同时保留大部分线性时间优势。具体比例尚有争议。Copy 重的任务（in-context learning 新模式）多 attention 帮助大。一般语言建模少 attention 也行。

状态空间模型：Mamba 与线性时间路线

注意力是 $O(n^2)$。线性时间替代品反复出现：linear attention [Katharopoulos et al., 2020]、Performer [Choromanski et al., 2021]、Linformer、Reformer。一个都没站住——规模一上去全都不如普通注意力。

然后 Mamba 出来了。Mamba [Gu & Dao, 2023] 是 selective state-space model：每层维护固定大小的隐状态 $h_t \in \mathbb{R}^N$，按递推更新：

$$ h_t = \bar{A}_t \, h_{t-1} + \bar{B}_t \, x_t, \quad y_t = C_t \, h_t. $$

“selective” 的关键：$A$、$B$、$C$ 都跟输入有关，由 $x_t$ 算出来。这是之前 SSM（S4 [Gu et al., 2022]）缺的拼图——S4 的动力学跟时间无关，做不了基于内容的记忆。Mamba 能做。

Mamba-2 [Dao & Gu, 2024] 把 $\bar{A}_t$ 简化为标量乘单位矩阵，递推就能写成结构化矩阵乘法（State Space Duality，SSD）在 GPU 上高效跑。2.7B 参数下 perplexity 跟 Pythia-2.8B 持平，推理快 5 倍，每 token 内存常数——没有 KV cache。

Mamba 不是 Transformer 杀手。Jamba 论文 [Lieber et al., 2024] 和后续多个混合模型（Zamba、Samba、Falcon-Mamba）都发现 纯 Mamba 在 in-context learning 和 copy 任务上吃亏——具体说，凡是要从前文精确取出某个 token 的任务它都做不好。补救办法是在多层 Mamba 里穿插几层注意力。Jamba 大约 7 层 Mamba 配 1 层注意力，Samba 1:1。

Mamba 在 copy 上的机理瓶颈是隐状态维度 $N$ 固定（一般 64-128）。要 copy 5000 步前的 token，模型必须把相关 token 压缩进隐状态、传 5000 步还不被覆写。Attention 直接绕过这个：每步重新对所有过去 token 算相似度。[Jelassi et al., 2024] 形式化证明了：Mamba 在序列长度超过隐状态大小线性比例后无法解关联召回问题，attention 可以。

2026 年长上下文领域，混合架构是实战上的最优解。通用 LLM 市场（Qwen3、GPT-4o、Claude-4.5）仍然是纯 Transformer 主导，但 256K 以上的上下文窗口里，Mamba-注意力混合（Jamba-1.5-Large、Falcon3-Mamba）以 5-10 倍更低的推理成本竞争。

RWKV：第三条路

RWKV [Peng et al., 2023] 是设计成训练时能像 Transformer 一样并行的循环网络。它用 time-mixing 块（带指数衰减的线性注意力）和 channel-mixing 块（门控 FFN）。RWKV-7（2025）引入 “Goose”——可学习的动态状态演化，把跟注意力的质量差距补得差不多了。

我把 RWKV 列进来是为了完整。但实战上，过去 12 个月我看到的非 Transformer LLM 出货团队，全都选的 Mamba-2 混合。RWKV 社区更小，工具链更弱。生产默认走注意力；长上下文窄场景考虑 Mamba 混合；RWKV 当研究项目看。

实算：70B 模型 32K 上下文的 KV cache 与 FLOPs

把单步解码时 attention 块在做什么具体数算出来。LLaMA-3-70B，GQA-8，$h=64$，$d_{\text{head}}=128$，$L=80$，$d_{\text{model}}=8192$，词表 $V=128256$，FFN inner $d_{\text{ffn}}=28672$（SwiGLU 三投影等价于约 57K 标准 FFN）。

单层参数。

  • Attention QKVO 投影：$d_{\text{model}} \cdot (n_{\text{heads}} \cdot d_{\text{head}} + 2 \cdot n_{\text{kv}} \cdot d_{\text{head}} + d_{\text{model}}) = 8192 \cdot (8192 + 2048 + 8192) = 152 \text{M}$
  • FFN SwiGLU（gate + up + down）：$3 \cdot d_{\text{model}} \cdot d_{\text{ffn}} = 3 \cdot 8192 \cdot 28672 = 705 \text{M}$
  • RMSNorm × 2：可忽略（约 32K）
  • 单层合计：约 857M

乘 80 层 = 68.6B。加 embeddings（$V \cdot d_{\text{model}} = 1.05 \text{B}$，tied）、加输出 norm，总 70B。✓

32K 单 token KV cache。 套上面公式：$2 \cdot 80 \cdot 2 \cdot 8 \cdot 128 = 327{,}680$ 字节每 token at FP16，即 4 KB。32K token 共 128 MB。✓

单解码 token 的 FLOPs。

  • Attention QKV 投影：$2 \cdot d_{\text{model}} \cdot (n_{\text{heads}} + 2 n_{\text{kv}}) \cdot d_{\text{head}} = 2 \cdot 8192 \cdot 80 \cdot 128 \approx 168 \text{M}$
  • Attention 计算（一个新 query 对 32K cached K/V）：$4 \cdot n_{\text{heads}} \cdot d_{\text{head}} \cdot 32{,}768 \approx 1.07 \text{G}$
  • O 投影：$2 \cdot d_{\text{model}}^2 \approx 134 \text{M}$
  • FFN：$2 \cdot 3 \cdot d_{\text{model}} \cdot d_{\text{ffn}} \approx 1.41 \text{G}$
  • 单层：约 2.78 GFLOPs
  • 80 层：约 222 GFLOPs 单 token

H100 上有效 FP16 吞吐约 2 TFLOPs/W，纯算 110 ms 一个 token——但解码是 内存瓶颈，不是算力瓶颈。真正的瓶颈是把 70B 参数从 HBM 以 3.35 TB/s 读出来，硬底约 21 ms 一个 token（一次解码必须摸过每个权重一次）。生产 serving 用 batching、KV 量化、speculative decoding 跟这个瓶颈打架（第 5 篇）。

生产真相：前沿实验室真正发布的样子

架构论文描述上线模型公开的 1 %，剩下 99 % 是水管。前沿实验室都在做的、模型卡里找不到的三件事：

Attention + FFN 融合的自定义 CUDA kernel。 vLLM 和 SGLang 都为 LLaMA 系手调了把 RMSNorm + QKV 投影 + RoPE + attention + 输出投影融成少数几个 kernel launch 的 kernel。朴素 PyTorch 图每层每 token 启 20 个左右 kernel；融合 kernel 只启 2-3 个。80 层 64-batch 解码下，kernel-launch 开销本身对小模型就能成主导。NVIDIA TensorRT-LLM 走得更远，从 graph IR JIT 编译每个架构专门的 kernel。

FP8 attention 计算。 H100 和 B200 有原生 FP8（E4M3 / E5M2）tensor core。FlashAttention-3 [Shah et al., 2024] 把 QK matmul 跑 FP8 + FP32 累加，吞吐比 FP16 attention 翻倍。精度损失在标准 benchmark 上 < 0.1 % perplexity；生产系统（GPT-4o、Claude-4.5）prefill 和 decode 都用 FP8。FP8 训练（NVIDIA Transformer Engine、[Micikevicius et al., 2022]）逐渐成为新预训练 run 的标准。

层级 LR 缩放和权重 tied。 LLaMA-3 论文 [Dubey et al., 2024] 透露 attention 输出投影和 FFN down 投影初始化方差比其他层小，避免训练早期不稳。多数生产训练脚本也把 embedding 和 LM head 矩阵 tied，省 1-2 % 总参数。这些调整单个看小，叠起来是肉眼可见的训练稳定性和质量差。

“Mixtral 8x7B 是一个模型一个架构"是部分虚构。线上 Mistral API 跑 Mixtral 时叠了几层推理时技巧（paged attention、走小 draft 模型的 speculative decoding、FP8 KV cache），不属于架构本身但确实影响质量和延迟。读 benchmark 数字时，看的是部署，不只是架构。

常见踩坑

这些架构上我见过的 5 件出错：

1. 写死 $h_{\text{kv}} = 1$ 假设 MQA。 一个 7B 模型自定义训练脚本，从教程拷代码时假设了 MQA。质量比基线差 3-5 个 perplexity 点。修法是切到 GQA-8。从零开始的话，GQA-8 是安全默认。

2. 忘了在 cache 里跨头共享 K/V。 我见过 FlashAttention 包装把 GQA 模型的 KV cache 也分配成 $[B, T, n_{\text{heads}}, d_{\text{head}}]$，运行时再对 cached K/V 做 repeat_interleave。GQA-8 模型上浪费 8 倍内存。Cache 应该是 $[B, T, n_{\text{kv}}, d_{\text{head}}]$，broadcast 在 attention kernel 内部做。

3. RMSNorm $\epsilon$ 太小。 PyTorch 默认 $\epsilon = 10^{-6}$。FP16 训练能下溢。FP16 用 $10^{-5}$，BF16 用 $10^{-6}$。我们追过一个 12000 步发散 bug，两天，最后就是这个。

4. MoE 路由梯度被静默清零。 自己实现 MoE 常见错：只有被选中的专家输出回流梯度到路由器。算了 router gate 又对 top-$k$ 重新 softmax 时，梯度路径里必须用 原始 logits，不是 top-$k$ 子集。否则路由器永远学不会 不 选某个坏专家。Mixtral 和 DeepSeek-V3 的参考实现都做对了；从零写的常错。

5. SwiGLU FFN inner dim 没调。 标准 Transformer FFN inner dim 是 $4 \cdot d_{\text{model}}$。朴素切到 SwiGLU 把投影数翻三倍，参数膨胀 3 倍。约定是 inner dim 缩到 2/3（约 $2.67 \cdot d_{\text{model}}$），保持 FFN 总参数大致不变。几个 LLaMA 开源 fork 没照这条规则做，参数量跟预期对不上。

研究前沿 2024-2026

“GQA + MoE Transformer” 共识之后还会发生什么：

Differential attention [Ye et al., 2024] 把两份不同参数的 attention map 相减。经验上抑制 attention 噪声、改进长上下文检索。已经在 2025-2026 部分模型发布里出现。

注意力原生稀疏。 Native Sparse Attention（NSA）这类工作训练模型原生地只对一个稀疏子集做 attention，而不是事后改造。[Yuan et al., 2025] 证明 native sparse attention 在长上下文 benchmark 上能在低得多的算力和 KV 代价下持平 dense attention。

Linear attention 卷土重来。 2024-2025 多篇工作（Gated Linear Attention、RetNet [Sun et al., 2023]、TransNormerLLM）通过加门控和衰减把跟 full attention 的质量差距收窄。生产规模上是否真能取代 softmax attention 还未定，但差距是 5 年来最小的。

Diffusion 语言模型。 [Lou et al., 2024]（SEDD）和 [Sahoo et al., 2024]（MDLM）证明离散 diffusion 在文本上能持平自回归 perplexity。2025 年发布的 Mercury Coder 号称 1000-token 生成不到 50 ms，比任何自回归模型在该长度上都快。前沿规模上质量是否撑得住未知，但是数年来非自回归路线最有说服力的挑战者。

Test-time 算力 scaling。 [Snell et al., 2024] 证明加推理时算力（chain-of-thought、self-consistency、MCTS）能在同等质量下替代预训练算力。o1 / DeepSeek-R1 / Claude-thinking 这条线认真这么做：32B 参数 thinking 模型能在硬推理任务上以 10 倍推理算力超越 70B 非 thinking 模型。这不是架构变化但改写了架构-成本的 trade。

什么场景用什么

场景架构理由
通用对话、代码Dense Transformer (LLaMA-3, Qwen3-Dense)单参数质量最高，工具链成熟
算力敏感的部署MoE (Mixtral, DeepSeek-V3, Qwen3-MoE)3-10 倍参数 / 算力比
256K+ 上下文，低延迟Mamba-注意力混合 (Jamba)每 token 常数内存
边缘侧 < 4 GB量化的小 dense 模型 (Qwen3-1.7B INT4, Phi-4-mini)内存瓶颈，MoE 帮不上忙
推理重（数学、代码）Dense Transformer + thinking RL质量随推理算力 scale

选型很少是 “选最好的架构”，多半是 “选约束跟我部署约束匹配的架构”。MoE 赢在多卡、算力瓶颈的场景。Dense 赢在单卡、VRAM 是上限的场景。混合赢在上下文是瓶颈的场景。

小结与下一篇

现代 LLM 还是 Transformer，但每一块都为了稳定性（pre-norm、RMSNorm）、质量（SwiGLU、RoPE）、推理代价（GQA、滑动窗口）、参数效率（MoE）被重新工程过。纯非注意力模型（Mamba、RWKV）通用场景吃亏，长上下文场景做混合后能赢。

下一篇深一层：Tokenization。为什么 CJK token 比英文贵 2-3 倍、BPE 在字节流上到底干了什么、chat template 的 token 怎么被烘进模型的行为里。这一层人人跳过，事后人人后悔。
