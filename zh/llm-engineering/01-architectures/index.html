大模型工程（一）：从 Transformer 到 MoE 的架构演化

MHA、GQA、MQA 的取舍，Mixtral 与 Qwen3-MoE 的稀疏路由，滑动窗口注意力，以及 Mamba、RWKV 这条非注意力路径——每条路的代价和适用场景。

2017 年那张 Transformer 的方框图，到 2026 年还是所有生产级 LLM 的剪影，但里面几乎每一块都被换过、稀疏化过、特化过。这个系列从架构、训练、推理、检索、评估、安全到上线，一路走完现代 LLM 工程栈。第一篇讲方框本身：2026 年的注意力长什么样、MoE 怎么把参数量和 FLOPs 解耦、Mamba 和 RWKV 这类非注意力模型在哪里真正赢过 Transformer。

我假设你知道原始 Transformer。如果不熟，NLP 系列第 4 篇 已经讲过。这里只讲 变了什么。

改了哪些、为什么改

LLaMA-3、Qwen3、Mistral、DeepSeek-V3、Yi——一个现代解码器层大致是这样：

# 单层伪代码
def layer(x, kv_cache):
    h = x + attention(rms_norm(x), kv_cache)   # pre-norm + RMSNorm
    h = h + ffn_or_moe(rms_norm(h))             # SwiGLU FFN，可能是路由的
    return h

相对于 2017 年的 “Attention Is All You Need”，有 5 处替换：

  1. Pre-norm 替代 post-norm——梯度走干净的残差恒等通路，不用 warmup。
  2. RMSNorm 替代 LayerNorm——不减均值，只除均方根。每层省一次 reduction。
  3. SwiGLU 替代 GELU——带门控的 FFN，约 2-3 % perplexity 收益，FFN FLOPs 涨 50 %，划得来。
  4. RoPE 替代正弦位置编码——第 6 篇会讲。
  5. GQA / MQA 替代 MHA——KV cache 小了一个数量级，质量几乎无损。长上下文场景下决定生死。

dense FFN 也越来越多被稀疏 MoE 替换，这是过去三年最大的架构变化，本篇一半篇幅讲它。

GQA、MQA、MHA：KV cache 真正的代价

MHA 把每个 token 投影成 $h$ 组独立的 Q、K、V，每组维度 $d_{\text{head}}$。MQA 留 $h$ 组 Q，但 K、V 只有 一份，所有头共享。GQA 介于中间：$g$ 个分组，每组共享一份 K/V，跨 $h/g$ 个头。

长上下文下，KV cache 的内存才是关键。LLaMA-3-70B 的配置是 $h=64$、$d_{\text{head}}=128$、$L=80$ 层、FP16：

$$ \text{每 token KV 字节} = 2 \cdot L \cdot 2 \cdot h_{\text{kv}} \cdot d_{\text{head}} $$

32K 上下文下：

变体$h_{\text{kv}}$KV / tokenKV / 32K
MHA6432 KB1.0 GB
GQA-884 KB128 MB
MQA10.5 KB16 MB

2026 年 GQA-8 是主流选择。GQA 论文 ↗ 报告大多数任务降幅不到 0.5 %，KV cache 缩 8 倍。MQA 早期模型（PaLM、Falcon-7B）试过，但规模上去之后质量损失太大，长上下文尤其依赖头的多样性。

import torch
import torch.nn.functional as F

class GQA(torch.nn.Module):
    def __init__(self, d_model=8192, n_heads=64, n_kv_heads=8, head_dim=128):
        super().__init__()
        self.n_heads, self.n_kv = n_heads, n_kv_heads
        self.head_dim = head_dim
        self.q_proj = torch.nn.Linear(d_model, n_heads * head_dim, bias=False)
        self.k_proj = torch.nn.Linear(d_model, n_kv_heads * head_dim, bias=False)
        self.v_proj = torch.nn.Linear(d_model, n_kv_heads * head_dim, bias=False)
        self.o_proj = torch.nn.Linear(n_heads * head_dim, d_model, bias=False)

    def forward(self, x):
        B, T, _ = x.shape
        q = self.q_proj(x).view(B, T, self.n_heads, self.head_dim).transpose(1, 2)
        k = self.k_proj(x).view(B, T, self.n_kv,    self.head_dim).transpose(1, 2)
        v = self.v_proj(x).view(B, T, self.n_kv,    self.head_dim).transpose(1, 2)
        # 把 K/V 复制到 Q 的头数（每组 n_heads/n_kv 个头共享一份 KV）
        k = k.repeat_interleave(self.n_heads // self.n_kv, dim=1)
        v = v.repeat_interleave(self.n_heads // self.n_kv, dim=1)
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        return self.o_proj(out.transpose(1, 2).reshape(B, T, -1))

代码里那个 repeat_interleave 只是逻辑示意。真正的 kernel（FlashAttention-2、FlexAttention）会在 SRAM tile 内部直接 broadcast，不会把 K/V 物化出来。

滑动窗口注意力

Mistral-7B 引入 滑动窗口注意力（SWA），窗口 4096。每个 token 只看前 $w$ 个。深度叠加之后感受野仍然线性增长——第 $L$ 层第 $n$ 位置的 token 能看到回 $L \cdot w$ 个 token——所以 32 层 $w=4096$ 的模型有效感受野 13 万 token。

SWA 把 KV cache 锁死在 $w$ 个 token 上，不管上下文多长。代价是模型必须学会用层叠的感受野，这意味着 SWA 模型在需要精确长程检索（“大海捞针”）的任务上常常不如全注意力模型。2026 年大多数长上下文模型把 SWA 和 attention sinks 一起用（Mistral-Large、Qwen3）——每层始终保留前 4-8 个 token 在窗口内，长上下文行为会稳很多。第 6 篇讲。

Mixture of Experts：参数翻倍，FLOPs 不变

MoE 的核心戏法：把 dense FFN 换成 $E$ 个 FFN（“专家”）加一个小路由器，每个 token 选 $k$ 个。总参数随 $E$ 涨，单 token FLOPs 只随 $k$ 涨。

Mixtral-8x7B 是教科书例子。每层 8 个专家，top-2 路由，32 层。总参数 46.7B（不是 56B，注意力是共享的）。激活参数 12.9B。也就是说你付 12.9B FLOPs 的算力换 46.7B 模型的建模能力——参数 / 算力比 3.6 倍。

Qwen3-235B-A22B（2025 年底发布）把这个比例拉到 235B / 22B，10.7 倍。DeepSeek-V3 是 671B / 37B，18 倍——目前生产级模型的最高比。

class MoEBlock(torch.nn.Module):
    def __init__(self, d_model, d_ffn, n_experts, top_k):
        super().__init__()
        self.gate = torch.nn.Linear(d_model, n_experts, bias=False)
        self.experts = torch.nn.ModuleList([
            SwiGLU(d_model, d_ffn) for _ in range(n_experts)
        ])
        self.top_k = top_k

    def forward(self, x):                        # x: [B, T, d]
        scores = self.gate(x)                    # [B, T, E]
        topk_w, topk_i = scores.topk(self.top_k, dim=-1)
        topk_w = F.softmax(topk_w, dim=-1)
        out = torch.zeros_like(x)
        for k in range(self.top_k):
            idx = topk_i[..., k]
            w   = topk_w[..., k:k+1]
            for e in range(len(self.experts)):
                mask = (idx == e)
                if mask.any():
                    out[mask] += w[mask] * self.experts[e](x[mask])
        return out

这段简化代码隐藏了两件生产里要命的事。

负载均衡。 路由器要是总挑同一个专家，你就只是花了多余参数堆出一个 dense 模型。常用做法是加辅助损失，惩罚不均衡——Switch Transformer 的 $\ell_{\text{aux}} = \alpha \cdot E \cdot \sum_e f_e \cdot p_e$，$f_e$ 是路由到专家 $e$ 的 token 占比，$p_e$ 是路由概率均值。DeepSeek-V3 干脆放弃辅助损失，改用 auxiliary-loss-free 方案：每个专家加一个偏置 $b_e$ 加到 gate logits 上，按梯度更新让 $f_e$ 自然均衡。更干净，质量不掉。

专家并行。 256 个专家（DeepSeek-V3）一张卡装不下。专家被切到多卡上，token 经 all-to-all 路由过去。all-to-all 的延迟就是瓶颈——DeepSeek 自家 DeepEP ↗ 在 NVLink 上能跑到 156 GB/s，已经接近硬件天花板。

MoE 的硬伤：总 VRAM 跟总参数走，激活的只有 $k$ 个不影响这件事。DeepSeek-V3 一张 80 GB H100 跑不动——大约要 700 GB 权重内存。激活参数等价说的是 算力，不是内存。

状态空间模型：Mamba 与线性时间路线

注意力是 $O(n^2)$。线性时间替代品反复出现：linear attention、Performer、Linformer、Reformer。一个都没站住——规模一上去全都不如普通注意力。

然后 Mamba 出来了。Mamba（Gu & Dao, 2023）是 selective state-space model：每层维护固定大小的隐状态 $h_t \in \mathbb{R}^N$，按递推更新：

$$ h_t = \bar{A}_t \, h_{t-1} + \bar{B}_t \, x_t, \quad y_t = C_t \, h_t. $$

“selective” 的关键：$A$、$B$、$C$ 都跟输入有关，由 $x_t$ 算出来。这是之前 SSM（S4）缺的拼图——S4 的动力学跟时间无关，做不了基于内容的记忆。Mamba 能做。

Mamba-2（2024）把 $\bar{A}_t$ 简化为标量乘单位矩阵，递推就能写成结构化矩阵乘法在 GPU 上高效跑。2.7B 参数下 perplexity 跟 Pythia-2.8B 持平，推理快 5 倍，每 token 内存常数——没有 KV cache。

Mamba 不是 Transformer 杀手。Jamba 论文（AI21, 2024）和后续多个混合模型（Zamba、Samba、Falcon-Mamba）都发现 纯 Mamba 在 in-context learning 和 copy 任务上吃亏——具体说，凡是要从前文精确取出某个 token 的任务它都做不好。补救办法是在多层 Mamba 里穿插几层注意力。Jamba 大约 7 层 Mamba 配 1 层注意力，Samba 1:1。

2026 年长上下文领域，混合架构是实战上的最优解。通用 LLM 市场（Qwen3、GPT-4o、Claude-4.5）仍然是纯 Transformer 主导，但 256K 以上的上下文窗口里，Mamba-注意力混合（Jamba-1.5-Large、Falcon3-Mamba）以 5-10 倍更低的推理成本竞争。

RWKV：第三条路

RWKV 是设计成训练时能像 Transformer 一样并行的循环网络。它用 time-mixing 块（带指数衰减的线性注意力）和 channel-mixing 块（门控 FFN）。RWKV-7（2025）引入 “Goose”——可学习的动态状态演化，把跟注意力的质量差距补得差不多了。

我把 RWKV 列进来是为了完整。但实战上，过去 12 个月我看到的非 Transformer LLM 出货团队，全都选的 Mamba-2 混合。RWKV 社区更小，工具链更弱。生产默认走注意力；长上下文窄场景考虑 Mamba 混合；RWKV 当研究项目看。

什么场景用什么

场景架构理由
通用对话、代码Dense Transformer (LLaMA-3, Qwen3-Dense)单参数质量最高，工具链成熟
算力敏感的部署MoE (Mixtral, DeepSeek-V3, Qwen3-MoE)3-10 倍参数 / 算力比
256K+ 上下文，低延迟Mamba-注意力混合 (Jamba)每 token 常数内存
边缘侧 < 4 GB量化的小 dense 模型 (Qwen3-1.7B INT4, Phi-4-mini)内存瓶颈，MoE 帮不上忙

选型很少是 “选最好的架构”，多半是 “选约束跟我部署约束匹配的架构”。MoE 赢在多卡、算力瓶颈的场景。Dense 赢在单卡、VRAM 是上限的场景。混合赢在上下文是瓶颈的场景。

小结与下一篇

现代 LLM 还是 Transformer，但每一块都为了稳定性（pre-norm、RMSNorm）、质量（SwiGLU、RoPE）、推理代价（GQA、滑动窗口）、参数效率（MoE）被重新工程过。纯非注意力模型（Mamba、RWKV）通用场景吃亏，长上下文场景做混合后能赢。

下一篇深一层：Tokenization。为什么 CJK token 比英文贵 2-3 倍、BPE 在字节流上到底干了什么、chat template 的 token 怎么被烘进模型的行为里。这一层人人跳过，事后人人后悔。
