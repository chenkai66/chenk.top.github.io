Tokenization 是人人跳过的一层，也是我在生产里 debug 最多 bug 的一层——悄无声息的质量回退、莫名其妙的成本暴涨、模型不听指令只因为有人 chat template 拼错了。这一篇是我希望自己上线多语言产品之前就内化的所有东西。

Tokenizer 到底在做什么

Tokenizer 的作用是把字符串转换成整数 ID 列表，反过来也能从 ID 列表还原成字符串。这两个方向的操作都是确定性的，但通常不是一一对应的。比如，tokenizer.decode(tokenizer.encode(s)) 走一圈后，可能会丢失空格、规范化 Unicode 或合并重复标点，具体取决于算法。

目前主要有三种主流方法：

  • WordPiece [Schuster & Nakajima, 2012; Wu et al., 2016]——BERT 使用的就是它。从前向后贪心匹配最长的子串，子词续接部分用 ## 标记。
  • BPE（Byte Pair Encoding） [Sennrich et al., 2016]——GPT-2/3/4、LLaMA、Qwen 和 DeepSeek 都用它。核心思想是不断合并最高频的相邻字符对，直到达到词表大小。最初的 BPE 算法来自 Philip Gage 在 1994 年发表的一篇关于数据压缩的论文，后来 Sennrich 等人将其改造用于神经机器翻译（NMT）。
  • Unigram（SentencePiece） [Kudo, 2018; Kudo & Richardson, 2018]——T5、mBART 和 XLM-R 使用这种方法。它先生成一个超大的候选词表，然后通过 EM 算法根据 unigram 语言模型的最大似然原则进行剪枝。

到 2026 年，几乎所有大型生成式 LLM 都会采用 byte-level BPE [Radford et al., 2019] 或 SentencePiece 的 BPE 变体。“byte-level” 是让 tokenizer 支持 Unicode 的关键：它的字母表只有 256 个字节，而不是约 15 万个 Unicode 码位。这样一来，任何编码下的字符串都能被成功分词，完全不需要 <UNK>。

这里有一个细微但重要的区别：BPE-on-characters（Sennrich 原始版本）的字母表是训练数据中出现的所有不同 Unicode 字符，通常是几万个，尾部低频字符会被丢弃并标记为 <UNK>。而 BPE-on-bytes（GPT-2 版本）的字母表固定为 256 个字节，就这么简单。任何 UTF-8 字符串都可以分解成字节并成功分词，不会失败。不过代价是，单个未合并的汉字会占用 3 个字节（3 个 token），而不是 1 个字符（1 个 token）。这就是后面要分析的 CJK token 膨胀问题的根本原因。

from collections import Counter

def get_pairs(word):
    return [(word[i], word[i+1]) for i in range(len(word) - 1)]

def merge(words, pair):
    new = []
    for w in words:
        out, i = [], 0
        while i < len(w):
            if i < len(w) - 1 and (w[i], w[i+1]) == pair:
                out.append(w[i] + w[i+1])
                i += 2
            else:
                out.append(w[i])
                i += 1
        new.append(out)
    return new

corpus = ["lower", "lowest", "newer", "wider"]
words  = [list(w) + ["</w>"] for w in corpus]
merges = []
for _ in range(8):
    pair_counts = Counter()
    for w in words:
        pair_counts.update(get_pairs(w))
    best = pair_counts.most_common(1)[0][0]
    merges.append(best)
    words = merge(words, best)
print(merges)
# [('e', 'r'), ('er', '</w>'), ('l', 'o'), ('lo', 'w'), ...]

实际应用中，BPE 不会直接用 list(w)，而是将字符串转为字节序列 w.encode("utf-8")。语料库规模通常达到万亿 token，但算法核心仍然是这个循环，只不过需要运行几十万次。

byte-level 的设计非常关键：GPT-2 将 256 个字节重新映射到可打印的 Unicode 子集，方便人类查看，但编码表仍然是 256 个符号。这样一来，任何输入都能成功分词，完全避免了 OOV（词汇表外）问题。

Tiktoken：OpenAI 的实际做法

OpenAI 的 tiktoken 库是 byte-level BPE 的生产级参考实现。它的设计选择和朴素的 BPE 方法有很大不同，这些差异在大规模场景下尤为重要。

预分词正则表达式
在进行 BPE 合并之前，tiktoken 会用一个正则表达式将文本切分成小块。正则表达式如下：

(?i:'s|'t|'re|'ve|'m|'ll|'d)|[^\r\n\p{L}\p{N}]?\p{L}+|\p{N}{1,3}| ?[^\s\p{L}\p{N}]+[\r\n]*|\s*[\r\n]+|\s+(?!\S)|\s+

这个正则按空格、数字（限制最多 3 位，避免类似 “9999999” 被当成单个 token 的奇怪情况）、缩写、以及连续的非字母数字字符来切分文本。BPE 合并只会在这些小块内部进行。这样可以防止跨单词边界的合并，否则会产生无意义的多词 token，影响模型的泛化能力。

Rust 实现与 BPE 缓存
性能关键路径是按预分词编码：先用正则表达式切分输入字符串，然后对每个小块使用缓存的合并表进行 BPE 合并。tiktoken 的 Rust 核心会对预分词内容进行哈希处理，并缓存合并结果——对于高频词（比如 “the”、“and”），编码速度可以达到 O(1)。单核处理英文文本的速度约为 3 MB/s，处理 CJK 文本的速度约为 1.5 MB/s。

词表文件格式
tiktoken 将词表存储为 base64 编码的字节串，以 token ID 为键。如果想查看 cl100k_base 中 token 12345 具体是什么，只需对 base64 解码，就能看到原始字节。很多 token 是不可打印的（比如连续的空白字符或部分 CJK 字符），因为词表是基于字节的，而不是基于字符的。

cl100k_base tokenizer（用于 GPT-3.5/4）包含 100,256 个 token。o200k_base tokenizer（用于 GPT-4o、GPT-4o-mini、o1）则包含 199,997 个 token。从 10 万扩展到 20 万的主要驱动力是多语言支持——o200k 合并了更多的 CJK 字符序列，并将英文的 token-per-word 比例从 1.3 降低到了 0.93。

词表大小：成本与质量的调节旋钮

词表大小 $V$ 是预训练阶段确定的一个超参数，用来平衡 嵌入表的计算量和内存占用 与 每篇文档的序列长度。

以一个 7B 级模型为例，假设 $d_{\text{model}} = 4096$，具体数据如下：

GPT-250,257tied206 M1.3
LLaMA-232,000untied262 M1.4
Qwen2.5151,936tied622 M0.95
GPT-4o200,000tied819 M0.93
Qwen3152,064tied623 M0.95

词表越大，每篇文档需要的 token 越少，按 token 计费时推理成本更低。但同时，嵌入层和 softmax 的规模也会变大。现代模型的词表大小通常在 100K 到 200K 之间。Qwen 和 GPT-4o 的词表范围在多语言支持和嵌入表占比之间找到了一个不错的平衡点。

词表增大时会发生什么？合并操作的次数会增加，更长的常见子词（比如整个低频词或常用短语）会被分配到单个 token。对于英语，词表超过 50K 后边际收益下降很快；而对于 CJK 语言，即使到 200K 仍然能看到明显收益。

关于词表大小和模型质量的关系，已经有了一些研究。[Tao et al., 2024] 发现，在固定训练预算的情况下，词表大小存在一个最优值，这个值会随着模型规模的增大而增大：小模型更适合小词表（< 32K），而大模型则更适合大词表（> 100K）。背后的逻辑是，大模型能够分摊嵌入参数的成本，并且从序列长度压缩中获益更多。他们推导出了一个 scaling law：$V_{\text{opt}} \propto N^{0.65}$，其中 $N$ 是非嵌入参数的数量。对于 70B 的模型，最优词表大小约为 200K；对于 1B 的模型，则约为 32K。

Llama 3 选择 128K 词表的原因

LLaMA-2 使用的是 32K 的 SentencePiece BPE tokenizer，主要基于英文训练。到了 LLaMA-3，直接跳到 128K（具体是 128,256），以 tiktoken 的 cl100k_base 为基础，在 Meta 的预训练数据混合集上重新训练。Llama 3 的论文 [Dubey et al., 2024] 提供了几个关键理由：

  • 多语言覆盖范围比 LLaMA-2 的 tokenizer 扩大了 4 倍
  • 预训练语料中序列长度平均缩短了 15%
  • 在 7B 参数以上的模型中，序列变短节省的计算量超过了嵌入表增加的成本

在 70B 参数的模型中，嵌入表（128K × 8192 = 1.05B 参数，tied）只占总参数的 1.5%——几乎可以忽略不计。但在 1B 参数的模型中，同样的嵌入表会占到总参数的 100%。因此，小型开源模型（比如 Phi-4-mini、Qwen3-0.5B）通常会选择更小的词表（50-64K），以保持嵌入参数和非嵌入参数的比例合理。

此外，LLaMA-3 还在词表末尾添加了 28 个保留的特殊 token（<|reserved_special_token_0|> 到 <|reserved_special_token_27|>）。这些是为未来微调预留的占位符。Meta 的 Llama-3-Instruct 就用其中几个作为聊天模板标记，完全不需要重新训练嵌入表。这种做法值得借鉴：在构建词表时预留几百个 token，方便后续的特定任务使用。

CJK Token 膨胀问题

这个问题让我在中文产品上吃了大亏。GPT-3.5 的 tokenizer（cl100k_base，10 万词表）把英文句子 “Hello, how are you?” 分成 6 个 token，而对应的中文 “你好，你今天怎么样？” 却变成了 17 个。

原因有三个：

  1. UTF-8 编码开销。 一个普通汉字占 3 字节 UTF-8，而 ASCII 只占 1 字节。还没开始合并时，字节级别就已经有 3 倍差距。
  2. 合并频率低。 BPE 算法会优先合并高频字符对。但中文在 OpenAI 的训练语料中占比一直很小，字符级别的合并自然就少。
  3. 没有词边界。 英文 BPE 得益于前导空格的约定（比如 Ġhello），而中文完全没有这种标记。

同一句话 “你好，请帮我用 Python 写一个快速排序”，不同 tokenizer 的分词结果如下：

TokenizerTokens
GPT-247
cl100k_base (GPT-3.5/4)24
o200k_base (GPT-4o)18
LLaMA-2 (32K)39
LLaMA-3 (128K)21
Qwen2.5 (152K)14

如果按 token 计费（所有 API 都这么算），同样的中文任务，Qwen2.5 的 token 数量比 GPT-4o 少 1.7 倍，还不算单价差异。Qwen3 更是降到 13。如果你的产品以中文为主，一定要选在中文数据上训练过的 tokenizer，而不是盲目跟风热门模型。

实际案例：一个中文产品的成本计算

看一个典型的中文客服机器人工作负载：用户消息平均 80 个汉字，模型回复平均 200 字，系统提示词平均 600 字。每天处理 100 万次对话。

Tokenizer单次对话 token 数每日 token 总数每月成本 @ $1/1M
cl100k_base(600 + 80 + 200) × 24/14 ≈ 15091.51 B$45,300
o200k_base(600 + 80 + 200) × 18/14 ≈ 11311.13 B$33,930
Qwen3880 × 13/14 ≈ 8170.82 B$24,510

（换算因子基于前面测得的每个 CJK 字符对应的 token 比例。）同样的工作负载，同样的单 token 价格，仅因 tokenizer 的选择不同，每月成本就能从 $45K 降到 $24K。如果规模更大，一年能省下七位数的成本。

反过来也成立：一个完全面向英文用户的产品，LLaMA-3 的 128K 词表比 LLaMA-2 的 32K 词表生成的 token 更多，因为更大的词表合并了更多多词短语（比如 “in order to” 从 4 个 token 变成 1 个）。对英文来说，提升幅度较小（5%-15%），远不如中文（2-3 倍），但仍然是实实在在的优化。

代码专用 tokenizer

代码是一种独特的“方言”，对分词有特殊需求。缩进（4 个空格、8 个空格或 Tab）、括号密度、标识符长度分布，这些都和普通文本不一样。通用的 tokenizer 通常在网页文本上训练，但它对代码中常见的模式合并得不够好。

StarCoder [Li et al., 2023] 使用了一个专门针对代码训练的 tokenizer，词汇表大小为 49,152。它基于 The Stack（一个 6 TB 的代码语料库）进行训练。以下是对一个典型 Python 文件（1 KB）的分词结果对比：

TokenizerTokens
GPT-2384
cl100k_base287
StarCoder218
DeepSeek-Coder215

StarCoder 比 cl100k_base 少用了 24% 的 token，这得益于它对常见代码模式的合并，比如 def 、 return、import 和 if __name__ == "__main__":。DeepSeek-Coder 的 tokenizer 更进一步，为缩进引入了显式的 token——4 个空格是一个 token，8 个空格是另一个。几乎所有代码预训练模型都会使用这种专用 tokenizer，因为节省的开销会不断累积：代码占预训练数据的 8%-15%，而代码任务在推理时对 token 的消耗非常大。

扩展词表：通常不建议

很多人会想：“我要在 LLaMA-3 上微调中文，它的 tokenizer 对中文支持太差，我加 5 万个中文 token 就好了。” 这种想法几乎总是错的。

给预训练模型增加 token 意味着三件事：

  1. embedding 表要新增行——要么随机初始化，要么用子词嵌入的平均值。
  2. LM head 也要新增行（如果 untied）——同样随机。
  3. 模型需要通过微调学会使用这些新 token。

新加入的嵌入要达到原有 token 的水平，需要数量级更多的训练数据。几篇论文（[Cui et al., 2023] Chinese-LLaMA、[Faysse et al., 2024] CroissantLLM）指出，扩展词表后需要 500 亿到 2000 亿 token 的持续预训练才能完全融合。如果数据量不够，结果就是模型虽然能高效分词，但生成的中文质量反而不如原版。

更实际的做法是直接选择一个 tokenizer 从一开始就针对目标语言训练过的基础模型。比如中文用 Qwen3，东南亚语言用 Sea-LION 或 Sailor，非洲语言用 BLOOM [Scao et al., 2022]。只有在没有合适模型时才考虑扩展词表。

如果确实无法避免扩词表（比如给一个完全没有重叠领域的基础模型添加医学术语），可以试试以下方法：

用子词分解的均值初始化新嵌入。 比如新 token “tachycardia”，可以用现有 BPE 分片（“tachy”、“card”、“ia”）的嵌入均值来初始化。这样能提供一个合理的起点。[Hewitt, 2021] 的研究表明，这种方法比随机初始化好 3-5 个困惑度点，而且只需少量训练。

warmup 阶段冻结模型其他部分。 先只训练新嵌入几百步，再解冻整个模型。这样可以避免模型其他部分为了补偿随机的新嵌入而破坏已有知识。

嵌入层也用 LoRA。 PEFT 支持对 embed_tokens 应用 LoRA。结合 warmup 技巧，这种方法能在极低的数据成本下获得大部分扩展词表的好处。

不用分词器的方法：ByT5、MEGABYTE、MAMBA-byte

两条值得关注但还没火起来的技术路线。

ByT5 [Xue et al., 2022] 直接在 UTF-8 字节上训练 T5，完全不用分词器。它的词表只有 256 个字节加上 3 个特殊标记。虽然序列长度比 mT5 长得多，ByT5-large 在多语言基准测试中表现和 mT5-large 差不多。不过，代价是序列变长了。比如，一段中文如果用 BPE 分成 100 个 token，换成字节就是 600 个。训练时每步速度慢一些，但每个字节的样本效率更高；推理时生成回复也更慢。ByT5 在多语言 NER 和噪声文本任务中找到了自己的位置，这些任务里分词器不匹配对质量影响很大。

MEGABYTE [Yu et al., 2023] 用分层方法处理长字节序列。一个叫 “patch” 的模型先把字节分成大小为 8 到 16 的组，每组跑一个小 Transformer，然后再用一个 “global” Transformer 处理这些组的表示。这种方法把计算复杂度从单个字节级别降到 patch 级别，接近 O(n²)，让字节级处理在长度达到 1M 的序列时也能用。

MAMBA-byte [Wang et al., 2024] 把 Mamba 状态空间架构直接用在字节流上。线性时间递推让字节级训练在大规模下变得可行。据报道，MAMBA-byte 350M 在 ARC-Easy 和 PIQA 上的表现和用 BPE 分词的基线模型相当，但在需要精确到词级推理的任务（比如 MMLU 和 HumanEval）上表现会差一些。

到 2026 年，byte-level BPE 还是主流选择。有意思的研究方向是更好的合并算法（贪心算法 vs 边际似然）以及按文档动态选择词表。

Chat template 是 tokenizer 的状态

Chat template 是一段 Jinja 模板，用来把消息列表转换成一个可以被分词的字符串。以 LLaMA-3 为例：

{% for m in messages %}
<|start_header_id|>{{ m.role }}<|end_header_id|>

{{ m.content }}<|eot_id|>
{% endfor %}

其中 <|start_header_id|>、<|end_header_id|>、<|eot_id|>，再加上 <|begin_of_text|>，每个都是单独的特殊 token，编码后只占一个 ID。它们记录在 tokenizer 的 added_tokens.json 文件中，也是模型在后续训练中学会识别的对话边界标记。

我在生产环境中遇到过两种常见的错误。

错误一：直接拼接普通字符串作为 prompt。 有人这样写代码：

1
2

prompt = f"User: {user_msg}\nAssistant:"
out = model.generate(tokenizer.encode(prompt))

这段代码能跑，但效果很差。模型在后续训练时从未见过这种格式。虽然它还能生成内容，但指令遵循和安全性表现都会大打折扣，因为控制这些行为的对话标记缺失了。一定要用 tokenizer.apply_chat_template(messages) 来构建 prompt。

错误二：渲染模板后又设置 add_special_tokens=True。 这样会导致 BOS 标记重复添加。模型会看到 <|begin_of_text|><|begin_of_text|>，然后行为变得奇怪。正确的做法是使用 tokenizer.apply_chat_template(messages, tokenize=True, add_generation_prompt=True)，这个方法会自动判断是否需要添加特殊标记。

常见踩坑

我在 tokenization 上浪费时间最多的 5 个坑，分享给大家：

1. 训练和推理用不同的 tokenizer。 训练时用 HuggingFace 的 tokenizer.json，部署时为了“更快”换成了 tiktoken。即使 BPE 合并规则相同，特殊 token 的处理方式也可能不一样。结果就是：模型生成了特殊 token，但推理层没去掉；或者推理层请求了模型没见过的 token。记住，训练和推理一定要用 完全相同的 tokenizer 文件。

2. 输入的空格规范化问题。 一些基于 SentencePiece 的 tokenizer 会把前导空格替换成 ▁（U+2581）。如果你用 tokenizer.decode(tokenizer.encode(s)) 转一圈，结果可能会多出空格或少掉空格，具体取决于 tokenizer 的实现。这个问题在链式 pipeline 中尤其明显：模型 A 的输出传给模型 B 时，空格偏移会导致质量下降。

3. 把 token 当成字符。 这是个常见错误：设置 max_tokens=200 时以为是限制“200 个字符”。实际上，Qwen3 处理中文时，200 token ≈ 200-260 字；处理英文时，200 token ≈ 800-1200 字。长度预算必须按模型的 token 来算，而不是按字符或字节。

4. 在特殊 token 中间截断。 如果你把一个 100K token 的 prompt 截短到 32K，而截断逻辑是按字节操作的，就可能把 <|im_end|>

生产环境中的现实：serving 栈里的 tokenizer

在生产环境中，serving 框架把 tokenizer 当作每个请求的起点和终点。整个流程的关键路径是：

  1. 接收请求 → 对 prompt 进行 tokenize → 将 token ID 传递给调度器
  2. 生成下一个 token ID
  3. 将 token ID 转换为文本 → 流式返回给客户端

实际实现受到三个生产环境约束的影响：

Tokenization 吞吐量。 vLLM 和 SGLang 都采用了 Rust 编写的 tokenizer（HuggingFace 的 tokenizers 库），相比 Python 的 transformers AutoTokenizer，性能提升了约 10 倍。在每秒 1000 次请求（RPS）且 prompt 包含 500 个 token 的情况下，Python 的 tokenize 操作会占满一个 CPU 核心；而 Rust 只需用到一小部分资源。

流式 detokenize。 每次生成一个 token 并进行 detokenize 时，需要引入缓冲机制。这是因为 BPE token 可能是不完整的 UTF-8 序列，尤其是对于 CJK 字符。vLLM 的 detokenizer 会维护一个近期 token 的缓冲区，尝试解码后，只输出到上一个完整的 UTF-8 边界为止。这会给流式输出增加 1 到 2 个 token 的延迟。

Tokenizer 缓存。 很多 serving 系统会对固定不变的系统 prompt 的 tokenize 结果进行缓存。Anthropic 的 prompt caching 功能就部分基于此——每个请求的前约 1000 个 token 会被预先 tokenize，并在具有相同前缀的请求间复用。结合 KV-cache 前缀共享技术，聊天应用中带有长系统 prompt 的场景，首 token 的生成时间可以减少 50% 到 70%。

字符级和视觉 tokenizer 呢？

最近有两个值得关注的方向，虽然还没成为主流。

字符级 / 纯字节级：比如 ByT5、MEGABYTE 和 MAMBA-byte。这些方法完全跳过 BPE 合并步骤，直接在原始字节流上训练。好处是简单，没有分词错误，也不会因为 CJK（中日韩）字符导致膨胀问题。但代价是序列长度变长：一段用 100 个 BPE token 表示的中文段落，换成字节级会变成 600 字节。即使使用亚二次复杂度的注意力机制，计算开销依然很大。MEGABYTE [Yu et al., 2023] 通过分层处理字节块解决了这个问题。这个方向很有潜力，但在通用场景下还无法与现有方法竞争。

视觉 tokenizer：把文本渲染成图像，然后按图像进行分词。代表工作有 Donut [Kim et al., 2022] 和 PIX2STRUCT [Lee et al., 2023]。它们的核心优势是统一了文本和视觉建模，同时摆脱了对特定 tokenizer 的依赖。目前还不适合通用大语言模型的工作负载，但在 OCR 相关领域非常有吸引力。

展望 2026 年，byte-level BPE 仍然是生产环境中的默认选择。真正有趣的研究集中在改进合并算法（比如贪心算法 vs 边际似然算法）以及针对每个文档动态选择词汇表。

研究前沿 2024-2026

byte-level BPE 共识之后，接下来会有什么新发展？

学习型分词。 [Yu et al., 2024]（SpaceByte）展示了如何将分词器与语言模型一起端到端训练，让模型自己决定子词单元。效果和 BPE 持平，完全不需要手工设计正则表达式。虽然还没投入生产环境，但已经对 BPE 构成了可信的威胁。

动态词表。 根据文档调整词汇表——比如，分词器知道正在处理 Python 代码时，可以切换到针对代码优化的合并表。早期工作（[Provilkov et al., 2020] BPE-dropout、[Kudo, 2018] subword regularization）在训练阶段探索了随机分词；而 2024-2025 年的研究重点转向推理阶段确定性但上下文感知的分词方法。

多模态分词器。 能同时处理文本、图像、音频和视频的模型需要统一的分词方案。目前主流方法（LLaVA、Qwen-VL、Gemini）是用独立的视觉编码器对图像进行分词，并将视觉 token 插入文本流中。下一代是否采用全模态统一的分词器仍然是一个开放问题。不过，[Team, 2024]（Chameleon）已经证明，早融合的全模态分词方法是可行的。

小结与下一篇

Tokenization 平时看不见，但一旦出问题就很要命：选错 tokenizer，CJK 工作负载成本直接翻倍；跳过 chat template，指令跟随能力就会下降；盲目扩展词表，微调后的模型可能还不如基础模型。挑模型之前，一定要在自己的实际 prompt 上统计 token 数量。务必通过 apply_chat_template 使用模型官方的 chat template。没有几千亿 token 的训练预算，别轻易扩展词表。

下一篇的主题是大规模预训练。我会聊数据混合、去重、200B token 那道悄无声息的悬崖（大多数公开数据集到这里就帮不上忙了），FSDP 和 ZeRO-3 的对比，以及把 LLaMA 风格的训练从 8 张 GPU 扩展到 8000 张时真正会遇到的问题。
