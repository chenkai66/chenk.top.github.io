大模型工程（二）：Tokenization 深挖

BPE、SentencePiece、WordPiece 的差别，byte-level fallback，CJK token 膨胀问题，扩词表的真实代价，以及悄悄塑造每个模型行为的 chat template 特殊 token。

Tokenization 是人人跳过的一层，也是我在生产里 debug 最多 bug 的一层——悄无声息的质量回退、莫名其妙的成本暴涨、模型不听指令只因为有人 chat template 拼错了。这一篇是我希望自己上线多语言产品之前就内化的所有东西。

Tokenizer 到底在做什么

Tokenizer 把字符串映射成整数 ID 列表，反向把 ID 还原成字符串。两个方向都是确定性的，但通常不是双射——tokenizer.decode(tokenizer.encode(s)) 走一圈可能丢空白、规范化 Unicode、合并连续标点，看用什么算法。

主流三家：

  • WordPiece [Schuster & Nakajima, 2012; Wu et al., 2016]——BERT 用。从前往后贪心最长匹配，子词续接用 ## 标。
  • BPE（Byte Pair Encoding） [Sennrich et al., 2016]——GPT-2/3/4、LLaMA、Qwen、DeepSeek 用。反复合并最高频的相邻 pair 直到词表大小到位。原始 BPE 算法来自 Philip Gage 1994 年关于数据压缩的论文，Sennrich 等把它改造给 NMT 用。
  • Unigram（SentencePiece） [Kudo, 2018; Kudo & Richardson, 2018]——T5、mBART、XLM-R 用。先撑出一个超大候选词表，再用 EM 按 unigram 语言模型最大似然剪枝。

2026 年大部分大型生成式 LLM 用的都是 byte-level BPE [Radford et al., 2019] 或 SentencePiece BPE 变体。“byte-level” 是让 tokenizer Unicode 安全的关键：字母表是 256 个字节而不是 15 万个 Unicode 码位，任何编码下的任何字符串都能 tokenize，不会出 <UNK>。

值得钉住的微妙区分：BPE-on-characters（Sennrich 原始版）的字母表是训练里见过的所有不同 Unicode 字符——多语言数据下通常几万个，长尾丢成 <UNK>。BPE-on-bytes（GPT-2 版）的字母表是 256 个字节，就这样。任何 UTF-8 字符串分解成字节就能 tokenize，不会失败。代价是没合并到的单个汉字消耗 3 字节（3 token）而不是 1 字符（1 token）。这就是后面要剖的 CJK token 膨胀的根源。

from collections import Counter

def get_pairs(word):
    return [(word[i], word[i+1]) for i in range(len(word) - 1)]

def merge(words, pair):
    new = []
    for w in words:
        out, i = [], 0
        while i < len(w):
            if i < len(w) - 1 and (w[i], w[i+1]) == pair:
                out.append(w[i] + w[i+1])
                i += 2
            else:
                out.append(w[i])
                i += 1
        new.append(out)
    return new

corpus = ["lower", "lowest", "newer", "wider"]
words  = [list(w) + ["</w>"] for w in corpus]
merges = []
for _ in range(8):
    pair_counts = Counter()
    for w in words:
        pair_counts.update(get_pairs(w))
    best = pair_counts.most_common(1)[0][0]
    merges.append(best)
    words = merge(words, best)
print(merges)
# [('e', 'r'), ('er', '</w>'), ('l', 'o'), ('lo', 'w'), ...]

实际生产 BPE 把 list(w) 换成 w.encode("utf-8") 字节序列，语料从万亿 token 起步，但算法就是这个循环跑几十万次。

byte-level 这一点很重要：GPT-2 把 256 个字节重新映射成可打印的 Unicode 子集只是为了人看着方便，实际编码字母表仍然是 256 个符号。结果是 没有任何输入会 tokenize 失败。OOV 的失败模式不存在。

Tiktoken：OpenAI 真正的实现

OpenAI 的 tiktoken 库是 byte-level BPE 的生产参考实现。它的实现选择跟朴素 BPE 不一样，规模上才不出问题。

预分词正则。 在 BPE 合并之前，tiktoken 用类似下面的正则把文本切成块：

(?i:'s|'t|'re|'ve|'m|'ll|'d)|[^\r\n\p{L}\p{N}]?\p{L}+|\p{N}{1,3}| ?[^\s\p{L}\p{N}]+[\r\n]*|\s*[\r\n]+|\s+(?!\S)|\s+

按空白、数字（截到 3 位避免 “9999999” → 1 token 的怪事）、缩写、连续非字母数字字符切。BPE 合并只在这些块 内部 发生。这避免了跨词边界的合并——不然会产生没意义的多词 token，伤泛化。

Rust 实现，BPE 缓存。 热路径是按预 token 编码：正则切输入字符串，每块用缓存的 merge 表做 BPE 合并。Tiktoken 的 Rust core 把预 token 哈希、缓存合并结果——重复的高频词（如 “the”、“and”）编码 O(1)。单核吞吐英文约 3 MB/s，CJK 约 1.5 MB/s。

词表文件格式。 Tiktoken 把词表存成按 token ID 索引的 base64 编码字节串。要看 cl100k_base 里 token 12345 到底是什么，base64 解码就看到原始字节。很多 token 不可打印（连续空白、不完整 CJK 字符）；词表是字节级不是字符级。

cl100k_base tokenizer（GPT-3.5/4）有 100,256 个 token。o200k_base（GPT-4o、GPT-4o-mini、o1）有 199,997 个。从 10 万到 20 万的扩展是被多语言覆盖驱动的——o200k 合并了多得多的 CJK 字符序列，把英文 token-per-word 从 1.3 降到 0.93。

词表大小：成本-质量旋钮

词表大小 $V$ 是预训练时定死的超参，trade-off 在 嵌入表的 FLOPs 和内存 与 每篇文档的序列长度 之间。

7B 级模型 $d_{\text{model}} = 4096$ 的具体数字：

GPT-250,257tied206 M1.3
LLaMA-232,000untied262 M1.4
Qwen2.5151,936tied622 M0.95
GPT-4o200,000tied819 M0.93
Qwen3152,064tied623 M0.95

词表越大，每篇文档的 token 越少，按 token 计费的推理越便宜，但嵌入和 softmax 越大。当代模型基本都落在 100-200K 之间。Qwen / GPT-4o 这一档在多语言覆盖和嵌入表占比之间取了甜点。

词表变大会发生什么？合并次数变多，更长的常见子词（整个低频词、常见短语）能拿到一个 token。英文里 50K 之后边际收益快速下降，CJK 里到 200K 还在涨。

词表大小和质量的经验 scaling 已经研究过了。[Tao et al., 2024] 发现固定训练预算下，词表大小有个 随模型大小增长 的最优值：小模型偏好小词表（< 32K），大模型偏好大的（> 100K）。直觉是：大模型能摊销嵌入参数代价，从序列长度压缩里得益更多。他们推出 scaling law $V_{\text{opt}} \propto N^{0.65}$，$N$ 是非嵌入参数数。70B 模型最优约 200K，1B 模型最优约 32K。

Llama 3 选 128K 词表的决策

LLaMA-2 用 32K SentencePiece BPE tokenizer，主要在英文上训。LLaMA-3 跳到 128K（具体是 128,256），从 tiktoken 的 cl100k_base 起步，在 Meta 自家预训练混合上重训。Llama 3 论文 [Dubey et al., 2024] 给出理由：

  • 多语言覆盖比 LLaMA-2 tokenizer 多 4 倍
  • 预训练语料平均序列短 15 %
  • 7B 以上规模，序列变短省的算力超过嵌入表代价

70B 模型上嵌入表（128K × 8192 = 1.05B 参数，tied）占总参数的 1.5 %——零头。1B 模型上同样的嵌入表是 100 %，所以小开源模型（Phi-4-mini、Qwen3-0.5B）经常用更小词表（50-64K），把嵌入对非嵌入比保持理智。

LLaMA-3 还在词表末尾加了 28 个保留特殊 token（<|reserved_special_token_0|> 到 <|reserved_special_token_27|>）。这些是给未来微调用的占位符——Meta 的 Llama-3-Instruct 用了几个做 chat template 标记，不必重训嵌入表。这是值得照搬的模式：词表构建时预留几百个 token 给下游特化。

CJK token 膨胀

这事在中文产品上把我坑过一次。GPT-3.5 的 tokenizer（cl100k_base，10 万词表）把英文 “Hello, how are you?” 编成 6 个 token，等价中文 “你好，你今天怎么样？” 编成 17 个。

三个因素叠加：

  1. UTF-8 编码代价。 一个常见汉字占 3 字节 UTF-8，ASCII 占 1。pre-merge 字节级就有 3 倍 token 劣势。
  2. 合并频率。 BPE 合并最高频 pair。中文预训练数据历史上在 OpenAI 语料里占比小，字符级 pair 合并得少。
  3. 没有词边界。 英文 BPE 受益于前导空格的约定（Ġhello），中文没有这种标记。

同一句 “你好，请帮我用 Python 写一个快速排序” 跨 tokenizer 的成本：

LLaMA-2 (32K)39

按 token 计费的话（API 都这么算），同样的中文负载，Qwen2.5 在 token 数上比 GPT-4o 便宜 1.7 倍，还没算单 token 价差。Qwen3 是 13。中文重的产品，要选在中文上训过的 tokenizer，不是热门那家。

实算：一个真实中文产品的成本算术

典型中文客服 bot 负载：用户消息平均 80 个汉字，模型回复平均 200 字，系统 prompt 平均 600 字。每天 100 万次对话。

Tokenizer单对话 token日 token月成本 @ $1/1M
cl100k_base(600 + 80 + 200) × 24/14 ≈ 15091.51 B$45,300
o200k_base(600 + 80 + 200) × 18/14 ≈ 11311.13 B$33,930
Qwen3880 × 13/14 ≈ 8170.82 B$24,510

（换算因子按上面 CJK 单字 token 比例。）同样的负载、同样的单 token 价格，月成本从 $45K 摇到 $24K，纯靠 tokenizer 选型。量大了一年是 7 位数节省。

反向也一样：100 % 英文用户的产品，LLaMA-3 的 128K 词表会比 LLaMA-2 的 32K 给出 更多 token 的 prompt 也更省，因为大词表合并了更多多词短语（如 “in order to” 从 4 token 变 1 token）。英文上的提升（5-15 %）比中文上小得多（2-3 倍），但确实存在。

代码专用 tokenizer

代码是另一种方言，自有它的 tokenization 需求。缩进（4 空格、8 空格、tab）、括号密度、标识符长度分布都跟散文不同。在网页文本上训的通用 tokenizer 对常见代码模式合并不足。

StarCoder [Li et al., 2023] 用 49,152 个 token 的 tokenizer 专在 The Stack（6 TB 代码语料）上训。在典型 Python 文件（1 KB）上比较：

TokenizerTokens
GPT-2384
cl100k_base287
StarCoder218
DeepSeek-Coder215

StarCoder 比 cl100k 节省 24 % token，来自合并常见模式：def 、 return、import 、if __name__ == "__main__":。DeepSeek-Coder 的 tokenizer 走得更远，给缩进显式 token——4 空格是一个 token，8 空格另一个。代码预训练模型几乎都用这类专用 tokenizer，节省成本会复利：代码占预训练数据 8-15 %，代码工作负载推理时 token 重。

FIM（fill-in-the-middle）token。 代码补全模型要在光标处插入，不是只追加。FIM 训练 [Bavarian et al., 2022] 用三个特殊 token——<|fim_prefix|>、<|fim_suffix|>、<|fim_middle|>——教模型在前缀和后缀给定下预测中间。当代每个代码 LLM（GitHub Copilot、Codeium、Cursor 的本地模型）都用 FIM。代码补全的 chat template 跟对话 chat template 完全不同；那是另一套 post-training 体系。

扩词表：通常不要

常见冲动：“我要在 LLaMA-3 上微调中文，它的 tokenizer 中文太烂，我加 5 万个中文 token。” 几乎总是错。

给预训练模型加 token 意味着：

  1. embedding 表多出新行——要么从零随机，要么从子词嵌入平均。
  2. LM head 多出新行（如果 untied）——同样随机。
  3. 模型必须靠微调学会用这些新 token。

新嵌入要追上原 token 的水平，需要数量级更多的训练数据。Chinese-LLaMA [Cui et al., 2023]、CroissantLLM [Faysse et al., 2024] 等论文报告需要 50-200B token 的 continued pretraining 才能完全吸收扩展词表。低于这个量，你拿到的是中文 tokenize 高效但中文生成质量比原版还低的模型。

务实的替代：从头就用目标语言训过 tokenizer 的 base model。中文用 Qwen3，东南亚语言用 Sea-LION 或 Sailor，非洲语言用 BLOOM [Scao et al., 2022]。扩词表是没有这种模型可用时才考虑的。

不可避免要扩词表时（比如给一个完全不重叠的领域 base model 加医学术语），几个做法有用：

新嵌入初始化为子词分解的均值。 新 token “tachycardia” 的嵌入初始化为现有 BPE 片段（“tachy”、“card”、“ia”）嵌入的均值。这给个像样的起点。[Hewitt, 2021] 显示这比随机初始化少训几百 step 就好 3-5 个 perplexity 点。

warmup 阶段冻结模型其余部分。 只训新嵌入几百 step，再解冻全模型。避免模型其他部分为补偿随机新嵌入而扭曲，破坏原有知识。

嵌入层也用 LoRA。 PEFT 支持对 embed_tokens 做 LoRA。配 warmup 技巧，能在数据代价的零头下拿到大部分扩词表收益。

Tokenizer-free 路线：ByT5、MEGABYTE、MAMBA-byte

两条值得知道但还没成主流的路。

ByT5 [Xue et al., 2022] 直接在 UTF-8 字节上训 T5——完全无 tokenizer。词表是 256（字节）+ 3 个特殊 token。ByT5-large 在多语言 benchmark 上跟 mT5-large 持平，尽管序列长得多。代价是序列长度：100 BPE token 的中文段落变 600 字节。每 step 训得慢，但每字节样本效率更高；推理每个回复也更慢。ByT5 在多语言 NER 和 tokenizer mismatch 主导质量的噪声文本任务上找到了利基。

MEGABYTE [Yu et al., 2023] 用层次化处理长字节序列：一个 “patch” 模型把字节按 8-16 大小分组、给每个 patch 跑小 Transformer，再一个 “global” Transformer 在 patch 表征上跑。算力近似 patch 上的 O(n²) 而不是字节上的，让字节级在 1M 序列内可行。

MAMBA-byte [Wang et al., 2024] 把 Mamba 状态空间架构直接套到字节流上。线性时间递推让字节级训练在规模上可行。MAMBA-byte 350M 据报告在 ARC-Easy 和 PIQA 上跟 BPE tokenizer baseline 持平，但需精确词级推理的任务（MMLU、HumanEval）上下降。

2026 年生产的默认还是 byte-level BPE。有意思的工作在更好的合并算法（贪心 vs 边际似然）和文档级动态词表选择上。

Chat template 是 tokenizer 状态

Chat template 是一段 Jinja，把消息列表渲染成可 tokenize 的字符串。LLaMA-3：

3
4

{% for m in messages %}
<|start_header_id|>{{ m.role }}<|end_header_id|>

{{ m.content }}<|eot_id|>
{% endfor %}

里面 <|start_header_id|>、<|end_header_id|>、<|eot_id|>，加上 <|begin_of_text|>，每一个都是 单个特殊 token，编码后只占一个 ID。它们在 tokenizer 的 added_tokens.json 里，也是模型在 post-training 阶段被训练成识别为对话边界的 token。

我在生产里见过两种典型翻车。

翻车一：把 prompt 当普通字符串拼。 有人写：

1
2

prompt = f"User: {user_msg}\nAssistant:"
out = model.generate(tokenizer.encode(prompt))

跑得动，勉强。但模型在 post-training 里没见过这个格式。它能生成，但指令遵循和安全行为都退化了，因为门控这些训练行为的对话 token 不在。永远走 tokenizer.apply_chat_template(messages)。

翻车二：渲染完 template 之后又 add_special_tokens=True。 这会重复加 BOS。模型看到 <|begin_of_text|><|begin_of_text|>，行为会怪。用 tokenizer.apply_chat_template(messages, tokenize=True, add_generation_prompt=True)，它知道要不要再加。

Chat template 也是 function calling 能跑起来的原因。Qwen3 的 chat template 把 JSON 工具调用包在 <tool_call>...</tool_call> 里，模型被训练成会发这些标记。Mistral 用 [TOOL_CALLS]。OpenAI 自家格式不用任何特殊 token，靠消息载荷里的 JSON 字段实现。第 7 篇专门讲 function calling。

常见踩坑

我个人在 tokenization 上烧时间最多的 5 个坑：

1. 训练和推理用不同 tokenizer。 训练用 HuggingFace 的 tokenizer.json，部署为了 “更快” 用 tiktoken。词表不完全一致——即便 BPE 合并规则相同，特殊 token 处理也有差。结果：模型发出特殊 token 推理层不剥离，或推理层请求模型从未见过的 token。永远训练和推理用 完全相同 的 tokenizer 文件。

2. 输入空白规范化。 几个 tokenizer（SentencePiece 系）把前导空格替换成 ▁（U+2581）。tokenizer.decode(tokenizer.encode(s)) 走一圈结果按变体不同会多空格或少空格。链式 pipeline 里这会冒出来：模型 A 的输出再传给模型 B 时空白偏移，质量降。

3. 把 token 当字符。 常见错：设 max_tokens=200 当成 “200 个字符”。Qwen3 中文 200 token ≈ 200-260 字；英文 200 token ≈ 800-1200 字。长度预算必须按模型的 token 想，不是字符或字节。

4. 在特殊 token 中间截断。 把 100K token 的 prompt 截到 32K，截断逻辑按字节走的话能从 <|im_end|> 中间切开。模型看到损坏的特殊 token，要么忽略要么瞎说。永远在 token 边界截断，不要在已 tokenize 流的字节/字符边界截断。

5. Stop token 不匹配。 OpenAI 兼容 API 接受 stop=["</s>"]，但 </s> 可能在模型词表里、也可能不在作为单 token。如果被 BPE 切成多 token，stop-string 检测每生成一步要扫一次解码后文本——又慢又在流式边界 race。当代 serving 框架（vLLM、SGLang）用 EOS-token-ID 匹配 + 滚动字符串检测处理这个，自定义服务器经常错。

生产真相：serving 栈里的 tokenizer

Serving 框架把 tokenizer 当成每个请求的第一跳和最后一跳。热路径是：

  1. 收到请求 → tokenize prompt → token ID 推到调度器
  2. 生成下一 token ID
  3. detokenize 成文本 → 流回客户端

3 个生产约束塑造实现：

Tokenization 吞吐。 vLLM 和 SGLang 都用 Rust tokenizer（HuggingFace 的 tokenizers 库，Rust 写的），比 Python transformers AutoTokenizer 快 10 倍左右。1000 RPS、500-token prompt 下，Python tokenize 能吃满一个 CPU core；Rust 用一小部分。

流式 detokenize。 一次生成一个 token、每次都 detokenize，需要缓冲，因为 BPE token 可能是 UTF-8 部分序列（CJK 尤其）。vLLM 的 detokenizer 维护近期 token 的缓冲，尝试解码，只发到上一个完整 UTF-8 边界为止的文本。这给流式输出加 1-2 token 延迟。

Tokenizer 缓存。 很多 serving 系统缓存不变系统 prompt 的 tokenize 结果。Anthropic 的 prompt caching 一部分就是这个——每个请求前约 1000 token 预 tokenize，相同前缀的请求间复用。配 KV-cache 前缀共享，有长系统 prompt 的聊天应用首 token 时间能降 50-70 %。

字符级和视觉 tokenizer 呢

两条值得知道但还没成主流的路。

字符级 / 纯 byte-level：ByT5、MEGABYTE、MAMBA-byte。完全跳过 BPE merge，直接训字节流。简单，没 tokenization bug，没 CJK 膨胀。代价是序列长度：100 个 BPE token 的中文段落变 600 字节。即使有亚二次注意力也很贵。MEGABYTE [Yu et al., 2023] 用层次化 chunk 处理这个问题。有前景，但通用规模上还没竞争力。

视觉 tokenizer：把文字渲成图，按图 tokenize。Donut [Kim et al., 2022]、PIX2STRUCT [Lee et al., 2023]。卖点是统一的 text-vision 建模，从 tokenizer 选型里解放。通用 LLM 还没就绪，但 OCR 重的领域有意思。

2026 年生产的默认还是 byte-level BPE。有意思的工作在更好的合并算法（贪心 vs 边际似然）和文档级动态词表选择上。

研究前沿 2024-2026

byte-level BPE 共识之后还会发生什么：

学习的 tokenization。 [Yu et al., 2024]（SpaceByte）证明可以端到端跟语言模型一起训 tokenizer，让模型自己决定子词单位。质量持平 BPE，没有手工正则。还没生产用，但是 BPE 的可信威胁。

动态词表。 按文档调整词表——比如知道在处理 Python 代码就切到代码 tuned merge 表的 tokenizer。早期工作（[Provilkov et al., 2020] BPE-dropout、[Kudo, 2018] subword regularization）在训练时探索过随机 tokenization；2024-2025 这一波探索推理时确定但上下文感知的 tokenization。

多模态 tokenizer。 处理文本、图像、音频、视频的模型需要统一 tokenization 方案。当前主流（LLaVA、Qwen-VL、Gemini）用单独的视觉编码器 tokenize 图像、把视觉 token 插入文本流。下一代是否走全模态统一 tokenizer 是开放问题。[Team, 2024]（Chameleon）证明早融合全模态 tokenization 能跑。

小结与下一篇

Tokenization 看不见，咬上来很疼：tokenizer 选错，CJK 负载贵 2 倍；不走 chat template，指令遵循退化；天真扩词表，微调出来比 base 还差。挑模型前永远在自己真实 prompt 上量 token。永远走模型官方的 apply_chat_template。除非你有几千亿 token 训练预算，别扩词表。

下一篇：预训练规模化。数据混合、去重、200B token 那道悄悄的悬崖、FSDP vs ZeRO-3，以及把一次 LLaMA 风格训练从 8 卡扩到 8000 卡时真正会出事的环节。
