陈锴 / Chen Kai

一句话

陈锴。工程师、写作者，也给长跑 Agent 当系统设计师。

我做的事

主线在 AI4Marketing。一个跨境电商团队的运营人员只跟控制器对话，剩下的——邮件、广告创意、视频、TTS——交给各自的子 Agent 自治完成。一句话变成一场战役，这是平台的设计前提。

副线是 AI4Science 与一个全自动科研系统：从一个问题出发，读论文、跑实验、写报告，整条管线长跑数小时甚至数天。它运行的前提是共享记忆与基于 harness 的技能进化——失败的尝试会被压缩成 lesson，下一次起点更高。

业余时间打磨 chenk.top。这个站本身也是工程：Hugo + 手写主题，约 400 篇双语长文，~1500 张 matplotlib 图通过自建管线推送到 OSS。我希望它读起来像杂志，而不是博客模板。

我关心的问题

  • Agent 的令牌经济：长跑系统在多个 Provider 之间该如何分配预算？哪些步骤值得用最贵的模型？
  • 失败如何压缩成可复用的技能，哪些只能丢弃。
  • 多 Agent 共享同一份记忆时，类型系统该有多严格。
  • 从 prompt demo 到产品的最后一公里——可观测性、回放、限流、用户信任。
  • 那些没人愿意做、放着会让整个系统慢慢崩坏的事。

我相信的几件事

  • 工具会被替换，工程素养不会。
  • 文档比代码更值得花时间。
  • 抽象付费过早是工程师最贵的本能。
  • 一个能稳跑 30 天的小系统，比一个跑 30 分钟的演示更有价值。
  • 把 Agent 当作有成本、会出故障的真实系统，而不是一个 prompt。

写作

chenk.top 上的中文与英文不是互译。同一个概念两种语言独立写——读者不同、上下文不同、节奏也不同。中文我学着收，英文我学着展。两边都不放水。

主题主要在机器学习、数学、云基础设施，以及承载这些系统所需要的那一层工程实践。每一篇尽量讲完整、讲诚实——读完应该能去看论文、去设计系统、能分辨教科书的图与生产环境的差距。
