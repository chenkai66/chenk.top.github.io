陈锴 / Chen Kai

工程师，写作者。数学和机器学习出身，这几年都在做云计算和长跑 Agent 系统。任务往往没有边界，所以也就慢慢学会了在没有边界的地方收边。

最常被问到的问题是"你具体在做什么"。摊开讲，主要是几摊事在并行：一摊是 AI4Marketing，跨境电商团队的运营人员只跟一个控制器对话，下游所有渠道——邮件、广告创意、视频生成、TTS 配音——交给各自的子 Agent；一摊是 AI4Science 的控制平面和一个跑得比我睡得还久的自动化科研系统；中间穿插着记忆系统、技能进化的 harness、还有这个站本身。

这些事看起来不相关，但我私下里觉得它们都是同一个问题的不同切面：长跑系统如何在出错、迁移、降本、换模型这些扰动下继续把事情做完。这是我现在最关心的问题。具体到落地，会拆成几个更小的：令牌预算怎么在 Provider 之间分；失败的尝试怎么压缩成下一次能复用的 lesson；多个 Agent 共享同一份记忆时类型该收多紧；从"prompt demo 跑通了"到"用户敢真的把业务压上来"中间那段路要补什么。

写作这件事也是顺着这个口子往下挖。chenk.top 上目前有四百来篇双语长文。中英文不互译——同一件事用两种语言重新写一遍，读者不一样、上下文不一样、节奏也不一样。中文我练着收，英文我练着展开，两边都不敷衍。

有几件事我越来越确信。工具会被替换，工程的判断力不会。文档比代码更值得花时间。过早抽象是工程师最贵的本能。一个能稳跑三十天的小系统，比一个跑三十分钟的演示有用得多。把 Agent 当成有真实成本、会真实出故障的系统，而不是一个会说话的 prompt——这是我所有项目共同的前提。

如果你也在折腾类似的东西，或者只是想聊聊上面任何一段话里你不同意的地方，欢迎写邮件。我通常二十四小时内回。
