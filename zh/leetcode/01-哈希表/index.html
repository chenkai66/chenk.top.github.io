一、哈希表究竟是什么

哈希表的核心是一个底层数组，用来存储 (key, value) 键值对。它的关键机制在于 哈希函数：这个函数能够将任意类型的 key 映射成一个整数索引，从而确定该键值对应存储的具体位置。

基于这样的设计，哈希表具备以下三个重要特性：

  • 确定性：相同的 key 总会被映射到同一个槽位，这是哈希表能够高效查找的基础。
  • 平均时间复杂度 $O(1)$：只要哈希函数设计得当，插入、查找和删除操作的时间复杂度不会随着数据量 $n$ 的增长而增加。
  • 最坏时间复杂度 $O(n)$：如果大量 key 被映射到同一个槽位，性能会退化为线性扫描。这种情况通常只会在恶意输入或哈希函数设计不佳时发生，实际工程中很少遇到。

冲突在所难免

不同的键可能会被哈希到同一个槽位，这种情况被称为 冲突（collision）。为了解决这个问题，哈希表通常采用两种策略之一：

  • 链式法（Separate Chaining）：每个槽位维护一个小型链表（或者平衡树），当发生冲突时，新元素直接追加到链表中。Java 的 HashMap 和 C++ 的 std::unordered_map 都采用了这种方式。
  • 开放寻址法（Open Addressing）：每个槽位只存放一个元素，如果发生冲突，则按照一定的规则向后探测（比如尝试 slot+1、slot+2……），直到找到一个空槽为止。Python 的 dict 和 Go 的 map 就是基于这种策略实现的。

虽然日常开发中你几乎不会亲自去实现冲突处理逻辑，但了解这些机制能够帮助你理解两个关键问题：为什么哈希表在某些情况下会比 $O(1)$ 慢，以及为什么设计一个高效的哈希函数如此重要。

哈希表 vs 其他选择

在实际开发中，我们为什么倾向于使用哈希表，而不是排序数组或者普通的数组呢？

随着数据规模 $n$ 的增长，不同方法之间的性能差距会变得非常显著。举个例子，当 $n = 1000$ 时，普通数组的线性扫描每次查找需要进行大约 1000 次比较，二分查找则只需要约 10 次，而哈希表的查找操作通常仅需约 1 次。当然，这种高效的查找速度是有代价的——内存开销。以 Python 的 dict 为例，它的内存占用通常是裸数组的 8 到 16 倍。这是因为哈希表不仅要存储键和值，还需要额外保存哈希值，同时为了减少冲突，内部会保持一定的稀疏性（即预留空槽）。

选择合适的数据结构

在动手写代码之前，先明确你的实际需求是什么：

一个简单的原则是：数组 > set > map（内存占用从低到高）。尽量选择能满足需求的最轻量级结构。具体来说：

需求推荐使用
用 [0, N) 范围内的整数索引访问数据普通 list / 数组
“这个元素是否出现过？”set
“这个 key 对应的值是什么？”dict
统计元素出现次数collections.Counter
自动初始化空容器collections.defaultdict(list)

在 Python 中，set 和 dict 共享同一套哈希机制，因此选择时主要看代码的可读性——能清晰表达意图的代码，往往也更容易调试。

二、两数之和：补数匹配模式

LeetCode 1. 给定一个数组 nums 和一个目标值 target，找出数组中和为目标值的两个数的下标。题目保证有且仅有一个解，且同一个元素不能重复使用。

输入：nums = [2, 7, 11, 15], target = 9
输出：[0, 1]      # 因为 2 + 7 == 9

先从暴力解入手

最直接的思路就是用两层循环逐一检查所有可能的数对，时间复杂度是 $O(n^2)$，空间复杂度为 $O(1)$：

def two_sum_brute(nums, target):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []

当 $n = 10^4$ 时，这种做法大约需要进行 5000 万次比较。显然，我们有优化的空间。

哈希表的巧妙思路

当我们从左到右遍历数组时，每次遇到一个数字 num，真正需要问的问题不是“num 是否在数组中”，而是 “我之前有没有见过 target − num？” 如果答案是肯定的，那么我们就找到了目标配对。为了实现这一点，我们可以用一个哈希表来维护一个映射 value → index，记录所有已经遍历过的数字及其对应的索引。

from typing import List

def two_sum(nums: List[int], target: int) -> List[int]:
    """基于哈希表的单次遍历解法。时间复杂度 O(n)，空间复杂度 O(n)。"""
    seen: dict[int, int] = {}          # value -> 最早出现的下标

    for i, num in enumerate(nums):
        complement = target - num
        # 先检查再插入：避免将 num 和自身配对。
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i

    return []   # 根据题目描述，此行实际上不会被执行

这里有两点需要注意：

  • 先查后存。如果先执行 seen[num] = i 再进行查找，可能会导致某些特殊情况出错。例如，对于输入 nums = [3, 0, 0] 且 target = 6，程序可能会错误地将同一个元素与自身配对。
  • 返回顺序。seen[complement] 对应的是更早出现的索引，因此在结果中它必须排在前面，即 [seen[complement], i]。

为什么时间复杂度是 $O(n)$

遍历 nums 数组时，总共需要进行 $n$ 次迭代。每次迭代中，我们只会执行一次字典查找操作，外加可能的一次插入操作——这两者的平均时间复杂度都是 $O(1)$。因此，总的时间复杂度为 $O(n)$，而空间复杂度也是 $O(n)$（因为字典最多存储 $n$ 个键值对）。

对于未排序的数组来说，同时优化时间和空间复杂度是不可能的。如果数组已经排好序，可以使用双指针技巧（将在第二部分详细介绍），达到 $O(n)$ 的时间复杂度和 $O(1)$ 的空间复杂度——但前提是先对数组进行排序，而排序本身的时间复杂度是 $O(n \log n)$。

三、字母异位词分组：规范化键模式

LeetCode 49. 给定一个字符串数组，请将所有字母异位词归类到同一组中。

输入：["eat", "tea", "tan", "ate", "nat", "bat"]
输出：[["eat", "tea", "ate"], ["tan", "nat"], ["bat"]]

字母异位词的特点是它们由相同的字符组成，但排列顺序不同。解决这个问题的核心在于设计一个 “键”，使得所有互为异位词的字符串都能映射到同一个分组中。

方案一：将排序后的字符串作为键

最直观的标准化方法就是对字符进行排序。

from collections import defaultdict
from typing import List

def group_anagrams(strs: List[str]) -> List[List[str]]:
    """时间复杂度为 O(n · k log k)，其中 n = len(strs)，k 为字符串的最大长度。"""
    groups: dict[str, list[str]] = defaultdict(list)
    for s in strs:
        key = "".join(sorted(s))   # 例如 "eat"、"tea"、"ate" 都会变成 "aet"
        groups[key].append(s)
    return list(groups.values())

这种方法简单易懂，且适用于任意字符集。不过，代价在于排序操作，每个字符串的排序时间复杂度为 $O(k \log k)$。

方法二：以字符计数作为键值

from collections import defaultdict
from typing import List

def group_anagrams(strs: List[str]) -> List[List[str]]:
    """使用长度为 26 的计数数组作为键，时间复杂度为 O(n · k)。"""
    groups: dict[tuple[int, ...], list[str]] = defaultdict(list)
    for s in strs:
        count = [0] * 26
        for c in s:
            count[ord(c) - ord("a")] += 1
        # 注意：list 类型不可哈希，必须转换为 tuple 才能用作字典的键。
        groups[tuple(count)].append(s)
    return list(groups.values())

这里需要特别注意一个细节：字典的键必须是 不可变且可哈希 的对象。虽然 Python 中的 list 不能直接作为键，但将其转化为 tuple 后即可满足要求。

四、无重复字符的最长子串：滑动窗口 + 哈希表

LeetCode 3. 给定一个字符串 s，找出其中不包含重复字符的最长子串的长度。

输入：s = "abcabcbb"
输出：3            # 子串为 "abc"

这是哈希表与滑动窗口技术完美结合的经典案例。我们维护一个窗口 [left, right]，让右边界 right 逐步向右扩展；如果新加入的字符会导致窗口内出现重复字符，则从左边界开始收缩窗口，直到窗口重新合法为止。

哈希表的作用是记录每个字符 最近一次出现的位置下标。这样，当发现重复字符时，可以直接将左边界 left 跳到重复字符的下一个位置，而无需一步步移动，从而实现 $O(1)$ 的跳跃效率。

def length_of_longest_substring(s: str) -> int:
    """滑动窗口 + 最近出现位置记录。时间复杂度 O(n)，空间复杂度 O(min(n, |Σ|))。"""
    last_seen: dict[str, int] = {}
    left = 0
    best = 0

    for right, ch in enumerate(s):
        # 如果当前字符在窗口内已经出现过，则需要收缩左边界。
        # 使用 max 是为了避免误处理窗口外的历史记录。
        if ch in last_seen and last_seen[ch] >= left:
            left = last_seen[ch] + 1

        last_seen[ch] = right
        best = max(best, right - left + 1)

    return best

五、前 K 个高频元素：频率统计 + 桶排序

LeetCode 347. 给定一个整数数组 nums 和一个整数 k，返回出现频率最高的 k 个元素。

输入：nums = [1, 1, 1, 2, 2, 3], k = 2
输出：[1, 2]

第一步很直观，用哈希表（或者 Python 的 Counter）统计每个元素的出现频率。第二步是关键的设计点：如何高效地选出频率最高的 k 个元素？

方案一：堆（$O(n \log k)$）

我们可以借助一个大小为 k 的小顶堆来处理每个 (count, value) 对：

import heapq
from collections import Counter
from typing import List

def top_k_frequent(nums: List[int], k: int) -> List[int]:
    """时间复杂度 O(n log k)，空间复杂度 O(n)。"""
    freq = Counter(nums)
    # most_common 方法内部已经用堆实现了排序。
    return [val for val, _ in freq.most_common(k)]

实际上，Counter.most_common 底层正是通过堆来完成的。这种方式不仅简洁易懂，而且在大多数场景下性能表现也相当不错。

方法二：桶排序（$O(n)$）

from collections import Counter
from typing import List

def top_k_frequent(nums: List[int], k: int) -> List[int]:
    """按频率进行桶排序。时间复杂度 O(n)，空间复杂度 O(n)。"""
    freq = Counter(nums)

    # buckets[f] 存储出现次数恰好为 f 的所有值。
    # 一个值最多出现 n 次，因此需要 n+1 个桶（索引范围 0 到 n）。
    buckets: list[list[int]] = [[] for _ in range(len(nums) + 1)]
    for value, count in freq.items():
        buckets[count].append(value)

    # 从高频率桶开始向下遍历。
    result: list[int] = []
    for f in range(len(buckets) - 1, 0, -1):
        for value in buckets[f]:
            result.append(value)
            if len(result) == k:
                return result
    return result

这是经典的“频率有界，直接用频率当索引”的技巧。算法的时间复杂度是严格的 $O(n)$，没有任何 $\log$ 因子，这种思路在类似题目如“按字符频率排序”（LeetCode 451）中也会频繁出现。

六、常见坑点

面试中哈希表相关的 bug 几乎都逃不出这几条：

  • 先插入再检查。比如两数之和，必须先查后插，否则可能把一个元素和自己配对。
  • 键不可哈希。list、set 和 dict 都不能直接当字典键，需要转成 tuple(...) 或 frozenset(...)。
  • 插入后再修改键。如果用作键的对象内容被修改了，它的哈希值会过期，原条目可能再也找不到了。记住：键要当作不可变对象对待。
  • 只需要 set 却用了 dict。seen[x] = True 这种写法比 seen.add(x) 多浪费 30%–40% 的内存。
  • 遍历时修改字典。在迭代过程中增删键可能会抛出 RuntimeError。如果非得改，先拷贝一份再操作，比如 list(d.items())。
  • 依赖键的顺序。虽然 Python 3.7+ 保留了插入顺序，但不要把算法正确性建立在键的顺序上。

七、面试前的自检清单

当你觉得某道题适合用哈希表时，动笔之前可以快速过一遍这些问题：

  • 键是什么？ 是原始值、排序后的形式、计数元组，还是前缀和？
  • 值是什么？ 是下标、计数、下标列表，还是布尔值？
  • 用 set 还是 map？ 如果只关心是否存在，set 更轻量。
  • 答案从哪来？ 是在扫描过程中撞上的，还是最后再扫一遍哈希表？
  • 边界情况有哪些？ 空输入、单个元素、全重复、全唯一、含负数？

口头表达时，可以说成这样：

“暴力解是 $O(n^2)$，因为每个元素都要和其他所有元素逐一比较。我可以用哈希表一边扫描一边维护 <key> → <value> 的映射，把每次查找压到 $O(1)$。总时间复杂度 $O(n)$，空间复杂度 $O(n)$。”

八、接下来练什么

当你对这几种模式已经滚瓜烂熟时，可以按以下梯度继续练习：

  • 简单题

    • 存在重复元素（217）
    • 有效的字母异位词（242）
    • 两个数组的交集（349）

  • 中等题

    • 和为 K 的子数组（560）
    • 最长连续序列（128）
    • 四数相加 II（454）
    • 常数时间插入删除取随机（380）

  • 困难题

    • 缺失的第一个正数（41——巧妙地把输入数组本身当作哈希表）
    • 串联所有单词的子串（30）
    • 至多含 K 个不同字符的最长子串（340）

下一篇 双指针 我们会看到另一种思路：当输入有序（或可以接受排序），双指针能在 $O(n)$ 时间内做到只用 $O(1)$ 额外空间，和哈希表打个平手。真正有趣的问题是 什么时候该选哪种方法——下一篇见。

九、为什么这套题型总在面试里出现

原因很简单，也很实在：

  1. 哈希表能把 $O(n^2)$ 压成 $O(n)$。这是“选对数据结构，算法就 trivial 了”最干净的一次演示，而这正是算法面试的核心考点之一。当你能看出“内层循环在找的东西，外层循环已经知道了”，你就该想到哈希表。
  2. 真实系统里到处都在用。请求处理器里的 users[user_id] 是哈希查表；流处理去重用的是 seen.add(...)；数据库前面那层缓存也是哈希表。面试官喜欢考这个题型是因为它迁移性最高。

题面里要识别的信号词：“找出所有满足……的对/三元组/组合”、“之前见过吗”、“统计……出现次数”，这些都是哈希表在等着你写出来。

十、生产环境中的实际应用

我自己写过的代码里，和本文模式一一对应的场景包括：

  • 两数之和 / 配对查找
    对账两边导出的文件：A 文件里某行应该匹配 B 文件里某行的互补 key。先用小文件建一遍 dict，再扫一遍大文件。O(a + b) 而不是 O(a * b)。同一个套路从几千行能扛到几亿行。

  • 按排序字符串分组
    商品 SKU 去重，“red,large,cotton” 和 “large,cotton,red” 是同一个 SKU。规范化形式是 tuple(sorted(...))，作为 dict key。

  • 前缀和哈希做“子数组和等于 K”
    时序分析里很常见：“找出所有 5 分钟窗口，营收正好等于 $X$”。这套技巧可以推广到任何可逆的聚合（求和、异或、计数）。不可逆的（max、min）就得换结构。

面试题是沙盒；生产版通常再叠上流式、过期淘汰、并发，但算法内核完全一样。

十一、用边界用例自查

以 Two Sum 为例——这是哈希表入门的标志题，下面这几条几乎能筛掉一半的提交：

1
2
3
4
5
6

two_sum([], 0)              # 空输入 —— 不能崩
two_sum([3], 3)             # 单元素，无法配对 —— 返回 []
two_sum([3, 3], 6)          # 两个相同值正好凑成答案
two_sum([1, 2, 3], 7)       # 无解
two_sum([-1, -2, -3], -5)   # 负数
two_sum([0, 0], 0)          # 零 —— `if num` 这种写法会在这里翻车

[3, 3] 那条最经典。一边 enumerate 一边 seen[num] = i，顺序写反就会把下标 0 和它自己配成一对。修正办法只在顺序：先查，再存。

1
2
3
4
5
6
7

def two_sum(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        if target - x in seen:        # 先查
            return [seen[target - x], i]
        seen[x] = i                   # 再存
    return []

两行代码，顺序对了就过。这道题被问的全部理由就在这两行的顺序上。
