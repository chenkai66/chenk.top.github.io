LeetCode（二）：双指针技巧

对撞、快慢、滑动窗口、分区四种模式，用循环不变量讲清 Two Sum II、3Sum、盛水容器、环形链表、移动零五题。

Read in English →

哈希表靠内存换速度，双指针则是反过来：用一点结构特性——数组有序、链表可能有环、答案藏在某个连续窗口里——就能做到 $O(n)$ 时间复杂度，同时只用 $O(1)$ 额外空间。代码表面上很简单，无非两个下标加一个 while 循环，但实际操作时新手最容易出问题：下标越界、死循环、漏掉重复值、平局时移错指针，各种坑层出不穷。要彻底规避这些问题，不能仅依赖死记硬背的指针移动规则，而应以 不变量 为基石，进行严谨分析。

本文将双指针归纳为四种典型模式：对撞指针、快慢指针、滑动窗口 和 分区指针，每种模式都对应一道高频面试题，如 Two Sum II、盛最多水的容器、3Sum、环形链表（LeetCode 141）和移动零。

双指针为什么有效

暴力枚举长度为 $n$ 的数组中所有数对，复杂度是 $\Theta(n^2)$，因为每对都得检查一遍。双指针能将复杂度降到 $\Theta(n)$，关键在于每次比较都能排除一大片候选，而不仅仅是单个候选：

利用的结构性质模式每次比较排除的内容
数组有序对撞指针被淘汰端点相关的所有数对
快指针能在有限环中追上慢指针快慢指针所有不在环上的节点
合法性随窗口大小单调变化滑动窗口同一右边界下更小的窗口
每个值属于 $k$ 个分区之一分区指针已处理的前缀和后缀

四种模式的核心思想是一致的：每次比较砍掉的是一个区域，而不是单个元素。如果这个性质不成立（比如数组无序、没有环、合法性不单调、无法干净分区），双指针就会退化到 $O(n^2)$，甚至完全失效。

模式一 —— 对撞指针（有序数组）

不变量

left 和 right 两个指针分别从数组的两端开始，朝着彼此移动。核心不变量是：

如果答案存在，它一定在 [left, right] 这个闭区间内。

每次比较有两种可能：要么当前这一对就是答案，要么可以排除掉某一端的元素。被排除的那一侧指针移动，另一侧保持不动。

LeetCode 167 —— Two Sum II

def two_sum_sorted(numbers: list[int], target: int) -> list[int]:
    """在升序数组中找到唯一一对和为目标值的元素，返回其 1-based 下标。"""
    left, right = 0, len(numbers) - 1
    while left < right:
        s = numbers[left] + numbers[right]
        if s == target:
            return [left + 1, right + 1]
        if s < target:
            left += 1          # 和偏小，丢掉当前最小值
        else:
            right -= 1         # 和偏大，丢掉当前最大值
    return []                  # 题目保证有解，理论上不会到这里

为什么移动左指针不会漏解？ 假设当前 s = numbers[left] + numbers[right] < target。那么 numbers[left] 和 [left+1, right] 中的任何数配对，结果都不会超过 s，也就不可能达到目标值。换句话说，所有以 left 为起点的组合都可以直接排除。同理，当 s > target 时，移动右指针也不会漏解。

复杂度分析。 时间复杂度 $O(n)$，空间复杂度 $O(1)$。每次循环至少移动一个指针，直到两个指针相遇为止。

LeetCode 11 —— 盛最多水的容器

给定一组高度 $h_0, h_1, \dots, h_{n-1}$，找出两个下标 $i < j$，使得矩形面积 $(j - i) \cdot \min(h_i, h_j)$ 最大。

def max_area(height: list[int]) -> int:
    """两条竖线和 x 轴围成的最大矩形面积。"""
    left, right = 0, len(height) - 1
    best = 0
    while left < right:
        h = min(height[left], height[right])
        best = max(best, (right - left) * h)
        # 移动较短的那一边：这是唯一可能让面积变大的方向
        if height[left] < height[right]:
            left += 1
        else:
            right -= 1
    return best

贪心策略为什么有效？ 假设当前左右指针分别指向 $i$ 和 $j$，且 $h_i \le h_j$。考虑所有以 $i$ 为左端点的其他组合 $(i, k)$（其中 $k < j$）：宽度 $(k - i)$ 比当前宽度 $(j - i)$ 小，而高度 $\min(h_i, h_k)$ 不会超过 $h_i$。因此，这些组合的面积都不可能超过当前值 $(j - i) \cdot h_i$。换句话说，以 $i$ 为左端点的所有组合都可以直接排除，放心移动左指针即可。右边的情况是对称的。

关键点在于：移动短边并不保证下一步面积更大，但能确保被丢弃的那一端不会再有更优解。这正是贪心策略正确的核心原因。

def three_sum(nums: list[int]) -> list[list[int]]:
    """返回所有和为 0 的不重复三元组。"""
    nums.sort()                                # O(n log n)，为对撞指针做准备
    n = len(nums)
    result: list[list[int]] = []

    for i in range(n - 2):
        # 剪枝：当前最小值大于 0，直接退出
        if nums[i] > 0:
            break
        # 第一位去重
        if i > 0 and nums[i] == nums[i - 1]:
            continue

        left, right = i + 1, n - 1
        target = -nums[i]
        while left < right:
            total = nums[left] + nums[right]
            if total == target:
                result.append([nums[i], nums[left], nums[right]])
                # 记录答案后，第二、三位去重
                left += 1
                right -= 1
                while left < right and nums[left] == nums[left - 1]:
                    left += 1
                while left < right and nums[right] == nums[right + 1]:
                    right -= 1
            elif total < target:
                left += 1
            else:
                right -= 1
    return result

三个位置都要去重，缺一不可。 排序把相同的数字聚在一起，算法在每个槽位选值时，必须跳过重复的值：

  • 第一位。 如果当前值和前一个值相同，直接跳过：if i > 0 and nums[i] == nums[i - 1]: continue。
  • 第二位。 找到一组解后，移动 left 跳过所有等于刚用过的值。
  • 第三位。 同理，移动 right 跳过右边的重复值。

漏掉任何一个去重步骤，像 [-2, 0, 0, 2, 2] 这样的输入就会产生重复结果。当然，也可以用 set 来兜底去重，但每次插入需要额外 $O(\log n)$ 时间，还会掩盖算法本身的逻辑。

复杂度分析。 排序需要 $O(n \log n)$，外层循环 $O(n)$，内层对撞指针扫描 $O(n)$，总时间复杂度 $O(n^2)$，空间复杂度 $O(\log n)$（排序递归栈）。

模式二 —— 快慢指针（Floyd 龟兔赛跑）

不变量

两个指针以不同速度沿同一条序列移动（经典设定是慢指针走一步，快指针走两步）。核心不变量如下：

如果存在环，快指针会在环内追上慢指针；如果没有环，快指针会先走到终点。

背后的原理很简单：一旦进入环，快慢指针之间的距离每轮都会减少 1。因此，最多经过 $\lambda$ 轮（$\lambda$ 是环的长度），两者必然相遇。

LeetCode 141 —— 环形链表

class ListNode:
    def __init__(self, val: int = 0, next: "ListNode | None" = None) -> None:
        self.val = val
        self.next = next

def has_cycle(head: ListNode | None) -> bool:
    """判断链表是否有环。O(1) 额外空间。"""
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            return True
    return False

为什么快慢指针一定会相遇？ 假设链表的非环部分长度为 $\mu$，环的长度为 $\lambda$。当两个指针都进入环后，fast 每次移动两步，slow 每次移动一步，因此 fast 相对于 slow 在环上的距离每轮减少 1。无论初始偏移是多少（比如 $d \in \{1, 2, \dots, \lambda - 1\}$），最多经过 $d$ 轮，两者就会相遇。整个过程最多需要 $\mu + \lambda$ 步，时间复杂度是 $O(n)$，空间复杂度是 $O(1)$。

为什么速度差必须是 1？ 快慢指针的速度差为 1 的设计是为了避免数论上的陷阱。假设快指针和慢指针的速度分别是 $a$ 和 $b$（满足 $a < b$），只有当 $\gcd(b - a, \lambda) = 1$ 时才能保证必然相遇。而 $(1, 2)$ 这种速度组合的优势在于 $b - a = 1$，它永远能整除环长 $\lambda$，因此无论环的长度是多少，算法都能正确运行，完全不需要担心特殊场景。

找环入口（LeetCode 142）

当 slow 和 fast 在环内的某个节点 m 相遇后，把其中一个指针重置到链表头 head，然后两个指针都以每次一步的速度前进，再次相遇的点就是环的入口。

def detect_cycle(head: ListNode | None) -> ListNode | None:
    """返回环的入口节点，若无环则返回 None。"""
    slow = fast = head
    while fast is not None and fast.next is not None:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            # 第二阶段：从头节点派一个新指针，和 slow 同速前进
            entry = head
            while entry is not slow:
                entry = entry.next
                slow = slow.next
            return entry
    return None

一句话解释入口结论：
慢指针走了 $\mu + k$ 步时，快指针正好走了 $2(\mu + k)$ 步，二者在环内某点相遇 ⇒ 差值 $\mu + k$ 必然是环长度 $\lambda$ 的整数倍。因此，从头节点再走 $\mu$ 步，以及从相遇点再走 $\mu$ 步，都会到达环入口——这就是它们第二次相遇的位置。

模式三 —— 滑动窗口

不变量

滑动窗口的核心是一个区间 $[l, r]$，它在数组或字符串上滑动。窗口是否合法取决于某些约束条件，比如无重复字符、区间和 $\ge k$、或者包含所有目标字符。滑动窗口之所以高效，是因为它的合法性具有单调性：

如果 $[l, r]$ 不合法，那么任何更大的窗口 $[l, r']$（$r' \ge r$）也必然不合法（反之亦然）。

这种单调性保证了 left 指针只需单向移动，不会回头。每个元素最多被 right 和 left 各访问一次，因此总复杂度是 $O(n)$。

def length_of_longest_substring(s: str) -> int:
    """返回字符串 s 中无重复字符的最长子串长度。"""
    last_seen: dict[str, int] = {}  # 字符 -> 它在 s 中最近出现的位置
    left = 0
    best = 0
    for right, ch in enumerate(s):
        # 如果当前字符在窗口内已经出现过，就把左边界移动到它的下一个位置。
        # max(...) 很关键：之前的重复可能已经把 left 推得更靠右了。
        if ch in last_seen and last_seen[ch] >= left:
            left = last_seen[ch] + 1
        last_seen[ch] = right
        best = max(best, right - left + 1)
    return best

为什么 "abba" 能说明 >= left 的必要性？
当 right = 2（第二个 b）时，left 被推到 2。接着 right = 3（第二个 a），此时 last_seen['a'] = 0 已经在窗口 [2, 3] 外面了。如果少了 >= left 这层判断，left 就会错误地跳回 1，破坏“左指针永不回头”的规则，最终导致结果出错。

复杂度分析。
时间复杂度 $O(n)$，每个字符最多被访问两次（一次加入窗口，一次移出窗口）。空间复杂度 $O(\min(n, |\Sigma|))$，取决于字符集大小和字符串长度。

模式四 —— 分区指针（k 路）

不变量

用三个指针 lo、mid 和 hi 把数组分成三块：[0, lo) 放的是小值，[lo, mid) 放的是中值，[mid, hi] 是还没处理的部分，(hi, n) 放的是大值。每次处理 arr[mid] 时，根据它的值决定把它移到哪一块，同时保持这个划分不变。

给定一个仅包含 0、1、2（分别代表红、白、蓝）的数组，原地排序使相同颜色相邻，顺序为红、白、蓝。要求一趟扫描完成。

def sort_colors(nums: list[int]) -> None:
    """原地排序仅含 0/1/2 的数组，时间 O(n)，空间 O(1)。"""
    lo, mid, hi = 0, 0, len(nums) - 1
    while mid <= hi:
        v = nums[mid]
        if v == 0:
            nums[lo], nums[mid] = nums[mid], nums[lo]
            lo += 1
            mid += 1                  # 换上来的值来自 [lo, mid)，按不变量已经是 1，直接跳过
        elif v == 2:
            nums[mid], nums[hi] = nums[hi], nums[mid]
            hi -= 1                   # 注意：mid 不动，换上来的值还没处理
        else:                         # v == 1
            mid += 1

0 分支和 2 分支的处理逻辑完全不同，这是最容易出错的地方。 把元素换到低区时，被换出来的值原本在 [lo, mid)，按不变量它必然是 1，所以可以直接跳过；但把元素换到高区时，被换出来的值来自 [mid+1, hi]，这部分还没分类，必须留到下一轮再处理，因此 mid 不前进。这个细节记不住，代码一跑就崩。

def move_zeroes(nums: list[int]) -> None:
    """把所有 0 移到数组末尾，保持非零元素的相对顺序。"""
    write = 0
    for read in range(len(nums)):
        if nums[read] != 0:
            nums[write], nums[read] = nums[read], nums[write]
            write += 1

这里选择交换而不是直接赋值，是为了确保第二段始终填满 0。每次将一个非零值前移时，留下的位置必然为 0——因为 [write, read) 区间内的位置通过归纳法可以证明全是 0。

哈希表 vs 双指针：到底要不要排序？

直接上大家最关心的对比表：

问题哈希表双指针
找一对和为 $T$（不要求原下标）$O(n)$ 时间，$O(n)$ 空间$O(n \log n) + O(n)$，$O(1)$ 额外空间
找一对，必须返回原下标哈希表更方便排序时带上原下标数组，多 $O(n)$ 空间
找所有对/三元组/四元组去重麻烦$O(n^{k-1})$，跳过重复段去重干净利落
检测链表环$O(n)$ 额外空间$O(1)$ —— Floyd
最长合法连续区间不一定需要滑动窗口是标准解法

怎么选？ 如果需要原下标、或者数组无序且不能排序，用哈希表；如果数据已经有序（或可以排序）、答案要求连续、或者必须节省空间到 $O(1)$，双指针更适合。

五种常见翻车（以及怎么躲开）

用双指针写代码，模式本身不难认，但把循环写对才是真正的挑战。以下是五个高频踩坑点，每个都可能直接毁掉你的面试表现：

1. 平局时移错指针。 比如在“盛最多水的容器”问题中，如果移动较高的那条边，高度上限不变，宽度却缩小了，结果只会更差。正确的做法是移动较短的边；如果两边相等，随便选一个移动即可。

2. 命中后忘记推进。 在 3Sum 中找到一组解后，必须同时移动 left 和 right，否则会陷入死循环，反复测试同一对数。记住，推进操作要和去重的 while 循环配合使用。

3. 滑动窗口里 left 回退。 如果直接写 left = last_seen[ch] + 1 而不加 max(left, ...) 保护，在类似 "abba" 的输入上就会出错。要么用 max 确保 left 不回退，要么在 left 推进时显式删除字符。

4. 忘记检查 fast.next 是否为空。 使用快慢指针时，访问 fast.next.next 必须保证 fast 和 fast.next 都非空。只检查 fast 会在两节点链表上崩掉。务必双重检查。

5. 荷兰国旗问题中 mid 在高分支误推进。 这是最常见的分区指针错误。牢记一句话：低分支推进 mid，高分支不动。

面试沟通

双指针题的解法其实有套路可循，下面这套话术能帮你轻松拿下 80% 的分数：

“这道题数组是有序的 / 要找的是连续窗口 / 明显是链表环问题，所以我会用 <具体模式>。暴力解法复杂度是 $O(n^2)$，而双指针能做到 $O(n)$，因为每次比较都能排除一整行、一整列或者某个前缀，而不是逐个排查。我打算维护的不变量是 <描述清楚不变量>。边界情况我会先处理：比如空输入、单元素数组、或者所有元素相等的情况。”

在写代码之前，先把不变量解释清楚，这是向面试官传递“我懂这个技术”的最强信号。背模板和真正理解之间的区别，往往就体现在这一点上。面试官听到你主动讲不变量，基本都会给你加分。

下一篇预告

快慢指针的思路会在第三篇：链表操作中以另一种形式登场——找中点、切分链表用于归并排序、按 k 个一组反转链表，背后都离不开它。对撞指针的思想还能直接用在链表回文判断上。而让这些技巧真正融会贯通的关键，则是引入一个虚拟头节点——具体怎么玩，咱们下一篇见分晓。
