Read in English →

哈希表通过牺牲内存来换取速度，而双指针则是反过来：利用一些结构上的假设（比如数组有序、链表可能存在环、答案位于某个连续区间），用 $O(n)$ 的时间复杂度和 $O(1)$ 的额外空间解决问题。代码看似简单——两个指针加一个 while 循环——但却是初学者最容易出错的技巧之一：下标越界、死循环、重复元素处理不当、错误移动指针等。要避免这些问题，关键不是死记硬背指针移动的规则，而是从循环不变量的角度去思考。

本文将双指针技巧分为四种常见模式——对撞指针、快慢指针、滑动窗口 和 分区指针——并为每种模式搭配一道高频面试题：Two Sum II、3Sum、盛最多水的容器、环形链表 和 移动零。

双指针为什么高效

暴力枚举长度为 $n$ 的数组中的所有数对需要 $\Theta(n^2)$ 的时间复杂度，因为每个数对都要检查一次。而双指针之所以能将复杂度降到 $\Theta(n)$，是因为它利用了一些结构性质，使得每次比较都能直接排除一大片候选解，而不是逐个排除：

利用的结构性质模式每次比较排除的候选范围
数组已排序对撞指针经过被淘汰指针的所有数对
快指针能在有限环中套圈慢指针快慢指针所有非环节点
合法性关于窗口大小单调滑动窗口同一右边界下的更小窗口
每个值属于 $k$ 个区域之一分区指针已分类的前缀和后缀

这四种模式的核心思想是一致的：每次比较都能排除一片候选解，而不仅仅是一个。如果这些结构性质不存在（比如数组无序、链表不成环、合法性不单调、无法清晰分区），双指针就会退化到 $O(n^2)$ 或者直接失效。

模式一 —— 对撞指针（有序数组）

不变量

两个指针 left 和 right 分别从有序数组的两端开始，逐步向中间靠拢。这里的关键不变量是：

如果答案存在，那么它一定位于闭区间 [left, right] 中。

每次比较的结果要么确认当前的指针对应的值是解，要么排除掉其中一个边界元素。被排除元素对应的指针会移动，而另一个指针保持不动。

def two_sum_sorted(numbers: list[int], target: int) -> list[int]:
    """在升序数组中找到唯一一对和为目标值的元素，返回其 1-based 下标。"""
    left, right = 0, len(numbers) - 1
    while left < right:
        s = numbers[left] + numbers[right]
        if s == target:
            return [left + 1, right + 1]
        if s < target:
            left += 1          # 当前和偏小，尝试更大的数
        else:
            right -= 1         # 当前和偏大，尝试更小的数
    return []                  # 题目保证有解，因此不会执行到这里

为什么 s < target 时移动左指针是安全的？ 假设当前的和 s = numbers[left] + numbers[right] < target。那么，numbers[left] 和 [left+1, right] 中任意一个数配对的结果都必然 $\le s < \text{target}$。换句话说，所有以 numbers[left] 为左端点的组合都不可能满足条件——这一整行的候选都被排除了，而不仅仅是 (left, right) 这一对。同理，当 s > target 时，移动右指针也是安全的。

复杂度分析。 时间复杂度 $O(n)$，空间复杂度 $O(1)$。每次迭代至少有一个指针向中间移动一步，直到两指针相遇为止。

LeetCode 11 —— 盛最多水的容器

题目描述。 给定一组高度 $h_0, h_1, \dots, h_{n-1}$，从中选出两个下标 $i < j$，使得矩形面积 $(j - i) \cdot \min(h_i, h_j)$ 最大。

def max_area(height: list[int]) -> int:
    """计算两条竖线与 x 轴围成的最大矩形面积。"""
    left, right = 0, len(height) - 1
    best = 0
    while left < right:
        h = min(height[left], height[right])
        best = max(best, (right - left) * h)
        # 移动较矮的一侧：这是唯一可能让面积变大的策略
        if height[left] < height[right]:
            left += 1
        else:
            right -= 1
    return best

贪心策略的正确性——为什么要移动短的那一边？ 假设当前选中的两个下标是 $i$ 和 $j$，且 $h_i \le h_j$。如果我们固定 $i$ 不动，尝试将 $j$ 向左移动（即考虑其他对 $(i, k)$，其中 $k < j$），可以发现以下两点：

  1. 宽度 $(k - i)$ 比原来的 $(j - i)$ 更小；
  2. 高度 $\min(h_i, h_k)$ 不会超过 $h_i$，因为 $h_i$ 是当前的瓶颈。

因此，所有以 $i$ 为左端点的矩形面积都不会超过我们刚刚计算出的值 $(j - i) \cdot h_i$。换句话说，$i$ 已经没有潜力成为更优解的左端点了，可以直接舍弃。同理，右边的情况是对称的。

关键点在于：并不是说下一步一定能找到更大的面积，而是说被舍弃的那一侧已经不可能再产生更优解了。这种局部最优的选择足以保证全局最优解的正确性。

def three_sum(nums: list[int]) -> list[list[int]]:
    """返回所有和为零的唯一三元组。"""
    nums.sort()                                # O(n log n)，排序是为了方便使用对撞指针
    n = len(nums)
    result: list[list[int]] = []

    for i in range(n - 2):
        # 剪枝优化：如果当前值大于零，后续的值都更大，不可能再凑出和为零
        if nums[i] > 0:
            break
        # 第一个数去重：跳过重复的外层值
        if i > 0 and nums[i] == nums[i - 1]:
            continue

        left, right = i + 1, n - 1
        target = -nums[i]
        while left < right:
            current_sum = nums[left] + nums[right]
            if current_sum == target:
                result.append([nums[i], nums[left], nums[right]])
                # 记录答案后，对第二、三个数去重
                left += 1
                right -= 1
                while left < right and nums[left] == nums[left - 1]:
                    left += 1
                while left < right and nums[right] == nums[right + 1]:
                    right -= 1
            elif current_sum < target:
                left += 1
            else:
                right -= 1
    return result

三处去重逻辑缺一不可。 排序会把相同的数字聚集在一起，算法在每个槽位选择值时，都需要跳过这些重复值：

  • 第一槽位。 if i > 0 and nums[i] == nums[i - 1]: continue —— 避免在外层循环中处理相同的起始值。
  • 第二槽位。 当找到一个解后，将 left 指针右移，直到跳过所有与刚刚使用的值相同的位置。
  • 第三槽位。 同理，将 right 指针左移，跳过右侧重复的值。

如果漏掉任何一处去重逻辑，在类似 [-2, 0, 0, 2, 2] 的输入上都会产生重复的三元组。当然，也可以用 set 来兜底去重，但这样每次插入操作会额外增加 $O(\log n)$ 的开销，而且掩盖了算法本身的清晰结构。

时间复杂度分析。 排序需要 $O(n \log n)$，外层循环遍历每个元素是 $O(n)$，内层双指针扫描也是 $O(n)$，总时间复杂度为 $O(n^2)$。空间复杂度主要来自排序的栈空间，为 $O(\log n)$。

模式二 —— 快慢指针（Floyd 判圈算法）

核心不变量

两个指针以不同的速度沿同一条序列移动（通常慢指针每次走 1 步，快指针每次走 2 步）。这里的核心性质是：

如果存在环，快指针一定会在环内追上慢指针；如果不存在环，快指针会率先走到序列末尾。

背后的原理非常直观：一旦进入环，快指针和慢指针之间的距离每轮都会缩短恰好 $1$。因此，在最多 $\lambda$ 轮内（$\lambda$ 是环的长度），两者必然会相遇。

class ListNode:
    def __init__(self, val: int = 0, next: "ListNode | None" = None) -> None:
        self.val = val
        self.next = next

def has_cycle(head: ListNode | None) -> bool:
    """判断链表是否存在环，仅使用 O(1) 额外空间。"""
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            return True
    return False

为什么快慢指针一定会相遇？
假设从链表头到环入口的距离为 $\mu$，环的长度为 $\lambda$。当两个指针都进入环后，fast 指针相对于 slow 指针的位置（取模 $\lambda$）每轮都会减少 1。换句话说，如果初始时两者的相对偏移是 $d \in \{1, 2, \dots, \lambda - 1\}$，那么经过 $d$ 轮后，偏移量会归零，两者必然相遇。总的迭代次数不会超过 $\mu + \lambda$，因此算法的时间复杂度为 $O(n)$，空间复杂度为 $O(1)$。

为什么选择 1 vs 2，而不是 1 vs 3 或 1 vs 5？
实际上，只要速度对 $(a, b)$ 满足 $a < b$ 且 $\gcd(b - a, \lambda) = 1$，就能保证快慢指针最终相遇。而 $(1, 2)$ 的特殊之处在于，$b - a = 1$ 始终能整除任何环长 $\lambda$。这意味着无论环的长度是多少，算法都能正常工作，完全避免了因数论性质导致的特殊情况问题。

找到环的入口（LeetCode 142）

def detect_cycle(head: ListNode | None) -> ListNode | None:
    """返回环的入口节点，若无环则返回 None。"""
    slow = fast = head
    while fast is not None and fast.next is not None:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            # 第二阶段：从头节点出发一个新指针，与 slow 指针同步前进
            entry = head
            while entry is not slow:
                entry = entry.next
                slow = slow.next
            return entry
    return None

入口点结论的简洁证明
当 slow 和 fast 相遇时，慢指针已经走了 $\mu + k$ 步（其中 $0 \le k < \lambda$），而快指针走的距离是其两倍，即 $2(\mu + k)$。由于两者最终位于同一节点，因此距离差 $\mu + k$ 必然是环长度 $\lambda$ 的整数倍。这意味着，从 head 再走 $\mu$ 步，以及从相遇点再走 $\mu$ 步，都会到达环的入口——这就是它们第二次相遇的地方。

模式三 —— 滑动窗口

不变量

在一个数组或字符串上，窗口 $[l, r]$ 被认为是合法的，当且仅当它满足某些特定条件（例如：没有重复字符、总和 $\ge k$、包含所有必需字符等）。滑动窗口算法的核心在于，窗口的合法性具有单调性，即窗口内容的变化会以某种单调的方式影响其合法性：

如果窗口 $[l, r]$ 是非法的，那么任何包含它的更大窗口 $[l, r']$（其中 $r' \ge r$）也必然是非法的（反之亦然）。

正是这种单调性保证了指针 left 可以一路向前推进而无需回退。每个索引最多只会被 right 和 left 各访问一次，因此算法的总体时间复杂度为 $O(n)$。

def length_of_longest_substring(s: str) -> int:
    """返回字符串 s 中所有字符均不重复的最长子串的长度。"""
    last_seen: dict[str, int] = {}  # 记录每个字符最近一次出现的位置（下标）
    left = 0  # 滑动窗口左边界
    best = 0  # 记录最长子串的长度
    for right, ch in enumerate(s):  # 遍历字符串，right 是当前右边界
        # 如果当前字符在窗口内已经出现过，则需要将左边界移动到该字符上次出现位置的右侧
        # 使用 max(...) 是为了确保左边界不会回退，因为之前的重复可能已经将 left 推得更远
        if ch in last_seen and last_seen[ch] >= left:
            left = last_seen[ch] + 1
        last_seen[ch] = right  # 更新当前字符的最新位置
        best = max(best, right - left + 1)  # 更新最长子串长度
    return best

为什么需要 >= left？以 "abba" 为例说明。
当 right = 2（第二个 b）时，left 被更新为 2。接着，当 right = 3（第二个 a）时，last_seen['a'] = 0 已经落在当前窗口 [2, 3] 之外。如果没有 >= left 的判断条件，left 会被错误地回退到 1，导致滑动窗口的“左边界永不回退”这一关键性质被破坏，最终计算结果也会出错。

复杂度分析。
时间复杂度为 $O(n)$，因为每个字符最多被访问和移除一次；空间复杂度为 $O(\min(n, |\Sigma|))$，其中 $\Sigma$ 是字符集的大小，哈希表的空间占用取决于字符集的种类和字符串长度。

模式四 —— 分区指针（k 路）

不变量

在数组中维护 $k$ 个分区。当 $k = 3$ 时，这就是经典的荷兰国旗问题。通过三个指针 lo、mid 和 hi，可以将数组划分为以下四个部分：

[0, lo) 存放已经确定的“低值”；[lo, mid) 存放已经确定的“中间值”；[mid, hi] 是尚未处理的部分；(hi, n) 存放已经确定的“高值”。

每次迭代时，程序会检查 arr[mid] 的值，并根据情况将其交换到低值区、保留在中间值区，或者移动到高值区——这一操作确保了分区规则始终成立。

def sort_colors(nums: list[int]) -> None:
    """在 O(n) 时间和 O(1) 空间复杂度下，原地对只包含 0、1 和 2 的数组进行排序。"""
    lo, mid, hi = 0, 0, len(nums) - 1
    while mid <= hi:
        v = nums[mid]
        if v == 0:
            nums[lo], nums[mid] = nums[mid], nums[lo]
            lo += 1
            mid += 1                  # 换进来的值来自 [lo, mid)，根据不变量它一定是 1，可以直接跳过
        elif v == 2:
            nums[mid], nums[hi] = nums[hi], nums[mid]
            hi -= 1                   # 注意：mid 不前进，因为换进来的值尚未处理
        else:                         # v == 1
            mid += 1

0 和 2 分支的处理逻辑不对称，这是最容易出错的地方。 当你将元素交换到低位区时，被替换的值原本位于 [lo, mid) 区间内，而根据算法的不变量，这个区间内的值必定是 1，因此可以安全地跳过。然而，当你将元素交换到高位区时，被替换的值来自 [mid+1, hi]，这部分数据还未被分类，必须留待下一轮处理，所以此时 mid 不应前进。

def move_zeroes(nums: list[int]) -> None:
    """将所有 0 移到数组末尾，同时保持非零元素的相对顺序。"""
    write = 0
    for read in range(len(nums)):
        if nums[read] != 0:
            nums[write], nums[read] = nums[read], nums[write]
            write += 1

这里用交换（而不是直接赋值）是为了确保第二段始终是 0：每次把一个非零元素往前挪时，留下的位置必然变成 0——因为 [write, read) 这一段通过归纳法可以证明全是 0。

哈希表 vs 双指针：什么时候该排序？

这是大家想要但很少有人写清楚的对比表：

问题哈希表双指针
找一对和为 $T$ 的元素（不要求原下标）$O(n)$ 时间，$O(n)$ 空间$O(n \log n) + O(n)$，$O(1)$ 额外空间
找一对和为 $T$ 的元素，必须返回原下标哈希表更方便排序时带上原下标数组，多 $O(n)$ 空间
找所有对/三元组/四元组去重麻烦$O(n^{k-1})$，跳过重复段去重更简洁
检测链表环$O(n)$ 额外空间$O(1)$ —— Floyd
最长合法连续区间通常没必要滑动窗口几乎是标准解法

决策规则。 如果需要保留原下标、或者数组无序且不允许排序，用哈希表；如果数据本身有序（或可排序）、答案要求连续、或者必须节省额外空间到 $O(1)$，用双指针。

五种常见翻车点（以及如何避免）

模式识别其实不难，真正难的是写对循环逻辑。以下这些坑足以毁掉一场面试：

1. 平局时移错指针。 在“盛最多水的容器”中，移动高的一边会让高度上限不变但宽度缩小，结果必然变差。要么移动矮的一边，要么平局时随便选一边。

2. 命中后忘记推进指针。 在 3Sum 中，找到一组三元组后必须同时推进 left 和 right；否则同一对会被反复命中导致死循环。记得配合去重的 while 循环一起用。

3. 滑动窗口中 left 回退。 直接写 left = last_seen[ch] + 1 而不加 max(left, ...) 防护，在输入如 "abba" 时会出错。要么用 max 兜底，要么在 left 推进时显式删除字符。

4. 忘记判空 fast.next。 使用 fast.next.next 时，必须保证 fast 和 fast.next 都非空。只检查 fast 会在两节点链表上崩溃。务必同时检查两者。

5. 荷兰国旗算法中 mid 在高分支误推进。 这是最常见的分区指针错误。记住一句话：低分支推进 mid，高分支不推进。

面试沟通技巧

下面这套话术能帮你拿到任意双指针题的 80% 分数：

“数组已经有序 / 答案要求连续窗口 / 这是链表环检测问题，我会选择 <某种模式>。暴力解法复杂度是 $O(n^2)$，而双指针能做到 $O(n)$，因为每次比较都能排除一整行/列/前缀，而不是逐个排查。我维护的不变量是 <明确说出不变量>。边界情况我会先处理：空输入、单元素、所有元素相等。”

写代码之前先把不变量讲清楚，这是区分“理解技术”和“背模板”的最可靠信号——面试官对此反应非常一致。
