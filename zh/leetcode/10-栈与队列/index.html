1. 栈与队列基础

1.1 栈：后进先出的数据结构

栈（Stack）是一种遵循“后进先出”原则的线性数据结构。可以把它想象成一摞盘子：只能从顶部放盘子，也只能从顶部拿走盘子。它的操作接口非常简洁：

  • push(x)：将元素 x 压入栈顶
  • pop()：弹出栈顶元素并返回
  • peek() 或 top()：查看栈顶元素，但不移除
  • empty()、size()：查询栈的状态

这些操作的时间复杂度均为 O(1)。栈的强大之处在于它天然适合解决某些特定问题——比如“最近一次发生的 X 是什么”这类场景。

在 Python 中，通常不会专门实现一个栈类，直接使用 list 就足够了，因为 append 和末尾 pop 的操作都是均摊 O(1)：

1
2
3
4
5
6

stack = []
stack.append(1)        # push
stack.append(2)
top = stack[-1]        # peek -> 2
val = stack.pop()      # pop  -> 2
empty = not stack

在 Java 中，推荐使用 ArrayDeque，它比旧版的 Stack 类性能更好，行为也更规范：

Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);

int val = stack.pop();

1.2 队列：先进先出的数据结构

队列（Queue）则完全相反，遵循“先进先出”的原则。就像排队买咖啡一样，先来的先被服务。标准操作包括：

  • enqueue(x) 或 offer(x)：从队尾加入元素
  • dequeue() 或 poll()：从队首移除元素
  • peek() 或 front()：查看队首元素
  • empty()、size()：查询队列状态

如果用 Python 的 list 来模拟队列，会发现 pop(0) 的时间复杂度是 O(n)，因为所有后续元素都需要向前移动一位。这种操作在 BFS 等场景下很容易导致超时。正确的做法是使用 collections.deque，它的两端操作都是 O(1)：

1
2
3
4
5
6

from collections import deque
q = deque()
q.append(1)            # enqueue
q.append(2)
front = q[0]           # peek -> 1
val = q.popleft()      # dequeue -> 1

在 Java 中，ArrayDeque 同样是一个很好的选择（通过 offer 和 poll 方法来模拟队列），而 C++ 则提供了 std::queue<int>。

一句话总结：栈记录的是函数调用的历史，队列管理的是待办任务的顺序。无论是 BFS、DFS、撤销/重做功能，还是任务调度或括号匹配，只要记住这句话，就能快速判断该用哪种数据结构。

1.1 栈 —— 后进先出

栈是一种线性数据结构，特点是后进先出（LIFO, Last In First Out）。可以把它想象成一摞盘子：你只能从顶部放盘子，也只能从顶部拿走盘子。它的接口非常简单：

  • push(x) —— 把元素 x 压入栈顶
  • pop() —— 弹出并返回栈顶元素
  • peek() 或 top() —— 查看栈顶元素但不移除它
  • empty() 和 size() —— 检查栈是否为空或获取栈的大小

所有这些操作在最坏情况下的时间复杂度都是 O(1)。栈的核心在于其访问模式：它只关心当前的栈顶元素，因此任何需要回答“最近的一个 X 是什么？”的问题，都可以用栈来优雅地解决。

在 Python 中，通常直接使用 list 来实现栈，因为 append 和 pop 在列表尾部的操作是均摊 O(1) 的：

1.2 队列 —— 先进先出

队列可以看作是栈的镜像：最“老”的元素最先离开。想象一下咖啡店门口的排队场景，这就是队列的直观模型。它的标准接口通常包括以下几个操作：

  • enqueue(x) 或 offer(x) —— 在队尾添加元素
  • dequeue() 或 poll() —— 移除并返回队首元素
  • peek() 或 front() —— 查看队首元素
  • empty() 和 size() —— 判断是否为空或获取当前大小

如果用 Python 的 list 来简单实现队列，表面上似乎没问题，但一旦进行性能测试就会发现问题：list.pop(0) 的时间复杂度是 O(n)，因为每次移除队首元素时，后面的元素都需要向前移动。更高效的底层数据结构是 collections.deque，它在两端的操作都能达到 O(1) 的时间复杂度：

1
2
3
4
5
6

from collections import deque
q = deque()
q.append(1)            # 入队
q.append(2)
front = q[0]           # 查看队首 -> 1
val = q.popleft()      # 出队 -> 1

在 Java 中，类似的实现是 ArrayDeque（通过 offer 和 poll 方法作为队列使用）；而在 C++ 中，你可以使用 std::queue<int>。

一个实用的心智模型是：栈 = 函数调用历史，队列 = 待办事项清单。这句话本身就能告诉你，在 BFS、DFS、撤销/重做功能、任务调度器以及括号匹配等场景中，应该选择哪一种数据结构。

二、栈的经典应用

栈是一种非常基础且强大的数据结构，尤其在解决某些特定问题时显得尤为优雅。以下是几个经典场景和技巧：

广度优先搜索（BFS）与队列

虽然栈主要用于深度优先搜索（DFS），但提到广度优先搜索（BFS），我们通常会用到队列。BFS 的核心思想是按层扩展：先访问所有距离起点为 1 的节点，再访问距离为 2 的节点，依此类推。下面是一个标准的 BFS 模板，总共不过八行代码：

from collections import deque

def bfs(start, neighbours_of):
    seen = {start}
    q = deque([start])
    while q:
        node = q.popleft()
        # ... 在这里处理当前节点 ...
        for nb in neighbours_of(node):
            if nb not in seen:
                seen.add(nb)
                q.append(nb)

写 BFS 时有两个关键点需要牢记，几乎适用于所有题目：

  • 入队时立即标记访问状态，不要等到出队时才标记。否则，同一个节点可能会被多次加入队列，导致时间和空间复杂度飙升。
  • 使用 deque 而不是 list。因为 list.pop(0) 的时间复杂度是 O(n)，一不小心就会把原本 O(V+E) 的 BFS 拖累成 O(V·(V+E))。

层序遍历的实现

层序遍历（如 LeetCode 102 题）其实就是在 BFS 的基础上增加了一个小技巧：在每一轮开始之前，先记录当前队列的长度 len(q)，这样就能知道这一层有多少个节点。处理完一层后，将结果存入最终输出。

以下是层序遍历的实现代码：

def levelOrder(root):
    if not root: return []
    out, q = [], deque([root])
    while q:
        level = []
        for _ in range(len(q)):  # 关键：提前固定当前层的节点数量
            node = q.popleft()
            level.append(node.val)
            if node.left:  q.append(node.left)
            if node.right: q.append(node.right)
        out.append(level)
    return out

通过这种方式，我们可以轻松地按层获取树的节点值，逻辑清晰且高效。

2.1 LeetCode 20 —— 有效的括号

给定一个只包含字符 ()[]{} 的字符串 s，判断其中的每一对括号是否按照正确的嵌套顺序闭合。

核心在于理解嵌套规则：当遇到一个右括号时，它必须与最近未闭合的左括号匹配。而“最近未闭合”正是栈（stack）这种数据结构的核心特性。

算法思路
遍历字符串，遇到左括号就压入栈中；遇到右括号时，栈必须非空且栈顶元素是对应的左括号，否则字符串无效。遍历结束后，字符串有效当且仅当栈为空。

def isValid(s: str) -> bool:
    stack = []
    pairs = {")": "(", "}": "{", "]": "["}
    for ch in s:
        if ch in pairs:                       # 右括号
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:                                  # 左括号
            stack.append(ch)
    return not stack

图 2 展示了算法在输入 ({[]}) 上的执行过程。每一列表示一步操作：高亮字符是当前处理的字符，底部显示的是操作完成后的栈状态。注意栈如何增长到深度为 3，然后对称地弹出。

复杂度分析
每个字符最多被压入和弹出栈一次，因此时间复杂度为 O(n)，空间复杂度在最坏情况下也是 O(n)（例如输入 (((( 时，所有字符都会被压入栈）。

常见陷阱

  • 忘记在比较之前检查栈是否为空——从空栈弹出会抛出异常。
  • 在第一次匹配后就直接返回结果，而没有继续检查剩余的字符串。
  • 遍历结束后忘记检查栈是否为空（例如输入 "(((" 会错误地返回 True）。

2.2 LeetCode 155 — 最小栈

设计一个栈，除了支持 push、pop 和 top 操作外，还需要在 O(1) 时间内支持 getMin() 操作。

如果每次通过遍历栈来找最小值，时间复杂度是 O(n)。为了实现 O(1) 的 getMin()，我们需要在每个深度上记录当前的最小值。最简洁的方式是存储 (value, min_so_far) 这样的键值对：

class MinStack:
    def __init__(self) -> None:
        self.stack: list[tuple[int, int]] = []

    def push(self, val: int) -> None:
        cur_min = val if not self.stack else min(val, self.stack[-1][1])
        self.stack.append((val, cur_min))

    def pop(self) -> None:
        self.stack.pop()

    def top(self) -> int:
        return self.stack[-1][0]

    def getMin(self) -> int:
        return self.stack[-1][1]

所有操作的时间复杂度都是 O(1)，空间开销是每个元素额外存储一个整数。还有一种“两个平行栈”的变体也能实现，但在处理最小值重复出现时会稍微麻烦一些（需要在 push 时比较 <= 而不是 <，并且只有当弹出的值等于当前最小值时才更新辅助栈）。

2.3 LeetCode 150 —— 逆波兰表达式求值

后缀表达式通过将操作符放在操作数之后，彻底消除了括号的存在。例如，((2 + 1) * 3) 可以写成 2 1 + 3 *。它的求值规则可以用一句话概括：栈中始终保存等待操作符处理的操作数。

def evalRPN(tokens: list[str]) -> int:
    stack: list[int] = []
    ops = {"+", "-", "*", "/"}
    for tok in tokens:
        if tok in ops:
            b = stack.pop()                # 先弹出的是右操作数！
            a = stack.pop()
            if   tok == "+": stack.append(a + b)
            elif tok == "-": stack.append(a - b)
            elif tok == "*": stack.append(a * b)
            else:            stack.append(int(a / b))   # 向零取整
        else:
            stack.append(int(tok))
    return stack[0]

有两个细节容易让人踩坑：

  • 操作数顺序。 栈是后进先出的结构，因此第一个弹出的是右操作数。a - b 和 b - a 的结果显然不同。
  • 除法行为。 LeetCode 要求除法结果向零取整（比如 -7 / 2 在 C 中是 -3，这里也一样）。但 Python 的 // 操作符会向负无穷取整（-7 // 2 是 -4）。解决方法是用 int(a / b) 来实现向零取整。

3. 单调栈

单调栈 是一种特殊的栈结构，它的核心特点是栈内元素始终保持严格的递增或递减顺序。每当新元素加入时，如果它破坏了栈的单调性，我们会不断弹出栈顶元素，直到满足单调性为止。这个简单的规则将看似需要 $O(n^2)$ 时间复杂度的问题（例如“对于每个 i，找到下一个满足条件的 j”）优化为均摊 $O(n)$ 的线性扫描，因为每个索引最多只会被压入和弹出栈一次。

单调栈的使用场景可以归纳为以下四类问题：

  • 找右侧第一个更大的元素 → 使用递减栈
  • 找右侧第一个更小的元素 → 使用递增栈
  • 找左侧第一个更大/更小的元素 → 逻辑相同，只需从右向左扫描（或者预处理）

通过单调栈，我们能够高效解决许多涉及顺序关系的问题，同时保持代码简洁优雅。

3.1 LeetCode 739 — 每日温度

对于每一天 i，问需要再等多少天才能遇到一个更暖和的日子？如果不存在这样的日子，输出 0。

暴力解法就是教科书式的 O(n²) 双重循环。而单调栈的优化思路是维护一个索引栈，栈中存储的索引对应的温度从栈底到栈顶是递减的。当今天的温度高于栈顶索引对应的温度时，说明找到了答案——弹出栈顶索引，计算天数差 answer[idx] = i - idx，并重复这一过程。

def dailyTemperatures(temperatures: list[int]) -> list[int]:
    n = len(temperatures)
    answer = [0] * n
    stack: list[int] = []                     # 存储索引，温度递减
    for i, t in enumerate(temperatures):
        while stack and t > temperatures[stack[-1]]:
            j = stack.pop()
            answer[j] = i - j
        stack.append(i)
    return answer

图 3 展示了这个过程的直观效果。每一条绿色弧线表示“某个较冷的日子终于被一个更暖和的日子解决”。底部的行展示了每一步之后栈中的索引状态——注意，栈中的温度始终是递减的。

复杂度分析。 每个索引最多进栈一次、出栈一次，因此内层 while 循环的总工作量是 O(n)。空间复杂度也是 O(n)，因为栈最多存储 n 个索引。

易错点： 栈中存储的是索引而不是温度值，因为我们既需要温度值（用于比较），也需要位置信息（用于计算天数差）。

4. 队列与广度优先搜索 (BFS)

队列最经典的应用场景就是实现 广度优先搜索 (BFS)：先探索所有距离为 1 的邻居节点，再探索距离为 2 的邻居节点，依此类推。整个框架可以用短短八行代码概括：

from collections import deque

def bfs(start, neighbours_of):
    seen = {start}
    q = deque([start])
    while q:
        node = q.popleft()
        # ... 处理当前节点 ...
        for nb in neighbours_of(node):
            if nb not in seen:
                seen.add(nb)
                q.append(nb)

在实现 BFS 时，有两个关键点几乎适用于所有问题：

  • 入队时标记，而不是出队时标记。 否则，同一个节点可能在被处理之前多次入队，导致时间和内存的开销大幅增加。
  • 使用 deque，而不是 list。 list.pop(0) 的时间复杂度是 O(n)。如果用 list 实现 BFS，算法的时间复杂度会从 O(V+E) 恶化到 O(V·(V+E))。
