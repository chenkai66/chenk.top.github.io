二叉树基础

二叉树的核心概念和术语

二叉树是一种特殊的树结构，每个节点最多有两个子节点，并且这两个子节点是有顺序的——左子节点和右子节点不能随意对调。值得注意的是，空树也被视为一种合法的二叉树。这种定义看似简单，但正是它让递归逻辑能够优雅地处理边界情况。

在解决二叉树相关问题时，我们经常会用到以下三个关键术语：

  • 根节点（root）：这是整棵树的起点，也是唯一一个没有父节点的节点。
  • 内部节点（internal node）：这类节点至少有一个子节点，是树的“中间层”。
  • 叶子节点（leaf）：这些节点没有子节点，位于树的“最底层”。

此外，还有两个数字用来描述节点在树中的位置关系：

  • 深度（depth）：从根节点到某个节点所经过的边数。根节点的深度为 0。
  • 高度（height）：从某个节点向下走到任意叶子节点的最长路径上的边数。叶子节点的高度为 0，而整棵树的高度就是根节点的高度。

掌握这五个术语，基本就能应对本文以及绝大多数二叉树相关的题目了。图 1 将这些概念直观地标记在一个示例树上：

常见的几种特殊形状

在算法问题中，有时会用某些特定形状作为约束条件。如果能快速识别这些形状，往往能找到复杂度更低的解法。

  • 满二叉树：每个节点要么没有子节点，要么有两个子节点，绝不会出现只有一个子节点的情况。这种结构在表达式树中非常常见。
  • 完全二叉树：除了最后一层，其他层的节点都填满了，而最后一层的节点从左到右依次排列。堆（Heap）就是利用这种特性存储的，因此可以直接用一维数组实现。
  • 平衡二叉树：对于任意节点，其左右子树的高度差最多为 1。AVL 树和红黑树通过维护这一性质，保证了操作的时间复杂度始终为 $O(\log n)$。
  • 二叉搜索树（BST）：对于任意节点，其左子树中的所有值都严格小于该节点的值，而右子树中的所有值都严格大于该节点的值。在验证一棵树是否为 BST 时，我们会直接利用这一特性。

DFS：一个套路，三种遍历

深度优先搜索（DFS）在每个节点上都会执行三个操作——访问当前节点、递归左子树、递归右子树——区别仅仅在于这三个操作的顺序不同。而这个顺序，正是每种遍历方式名字的由来。

遍历方式操作顺序示例树上的访问结果
前序根 → 左 → 右[3, 9, 20, 15, 7]
中序左 → 根 → 右[9, 3, 15, 20, 7]
后序左 → 右 → 根[9, 15, 7, 20, 3]

def preorder(node, out):
    if not node: return
    out.append(node.val)         # 根
    preorder(node.left, out)     # 左
    preorder(node.right, out)    # 右

def inorder(node, out):
    if not node: return
    inorder(node.left, out)      # 左
    out.append(node.val)         # 根
    inorder(node.right, out)     # 右

def postorder(node, out):
    if not node: return
    postorder(node.left, out)    # 左
    postorder(node.right, out)   # 右
    out.append(node.val)         # 根

选择哪种遍历顺序，核心在于什么时候你需要的信息已经准备好了。如果父节点必须在子节点之前处理，用前序遍历：比如复制一棵树、打印目录结构、序列化时先写出根节点。如果需要利用二叉搜索树（BST）的有序性，用中序遍历：比如生成排序结果、验证 BST 性质、找第 $k$ 小的元素。如果父节点的结果依赖于子节点的结果，用后序遍历：比如删除整棵树、计算子树的大小或总和、求树的高度。

LeetCode 94 —— 中序遍历：递归与迭代的对比

在三种二叉树遍历方式中，中序遍历是最适合用来研究递归和迭代实现差异的。它的迭代版本清晰地展示了递归调用栈背后的运作机制。

from typing import List, Optional

class Solution:
    def inorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        result: List[int] = []

        def visit(node: Optional[TreeNode]) -> None:
            if node is None:
                return
            visit(node.left)         # 1. 先一路向左走到尽头
            result.append(node.val)  # 2. 再记录当前节点的值
            visit(node.right)        # 3. 最后处理右子树

        visit(root)
        return result

时间复杂度为 $O(n)$，因为每个节点只访问一次。空间复杂度为 $O(h)$（$h$ 是树的高度），即平衡树时为 $O(\log n)$，退化成链表状时为 $O(n)$。

迭代实现
递归调用栈实际上完成了两件事：一是记住从根到当前节点的路径（祖先链），二是当左子树处理完后回到当前节点继续处理右子树。这两件事可以用一个显式的栈来模拟：

def inorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
    result: List[int] = []
    stack: List[TreeNode] = []
    cur = root

    while cur is not None or stack:
        # 阶段一：沿着左子树一路向下，将沿途节点压入栈中。
        while cur is not None:
            stack.append(cur)
            cur = cur.left

        # 阶段二：弹出栈顶节点（最深的未访问节点），记录其值，
        # 然后转向其右子树继续遍历。
        cur = stack.pop()
        result.append(cur.val)
        cur = cur.right

    return result

这两个阶段分别对应递归调用的两个部分：入栈相当于“进入下一层递归”，出栈相当于“从递归返回”。图 4 将递归调用栈和显式栈的执行过程逐帧对比，可以发现它们本质上是同一个算法的不同实现方式：

迭代实现中最容易犯的两个错误：

  1. 忘记在弹栈后设置 cur = cur.right，这会导致同一个节点被反复压栈和弹栈，陷入死循环。
  2. 把内层的 while 错写成 if，这样每次只会向下走一步，而无法处理左子树的最深节点——必须一次性沿着左链走到尽头，才能开始第一次弹栈操作。

前序和后序的迭代实现，各用一段说明

前序遍历 的迭代写法非常直观。先把根节点压入栈中，然后进入循环：每次从栈中弹出一个节点，记录它的值，接着先压右子节点，再压左子节点（因为栈是后进先出，左子节点后压入就能先被处理）。

def preorderTraversal(self, root):
    if not root: return []
    out, stack = [], [root]
    while stack:
        node = stack.pop()
        out.append(node.val)
        if node.right: stack.append(node.right)  # 先压 → 后弹
        if node.left:  stack.append(node.left)   # 后压 → 先弹
    return out

后序遍历 可以用一个小技巧来简化。如果把前序遍历的模板稍作修改，改为先压左子节点、再压右子节点，那么得到的访问顺序是“根→右→左”，这刚好是后序遍历结果的反转。因此，只需在最后将收集到的结果列表翻转一下即可：

def postorderTraversal(self, root):
    if not root: return []
    out, stack = [], [root]
    while stack:
        node = stack.pop()
        out.append(node.val)
        if node.left:  stack.append(node.left)
        if node.right: stack.append(node.right)
    return out[::-1]

这种方法巧妙地避开了传统迭代后序遍历时需要为每个节点维护“右子树是否已访问”状态的繁琐操作。

BFS：层序遍历

DFS 是一路深入到底，走不通了再回溯；而 BFS 则恰恰相反——它先访问离根节点距离为 0 的所有节点，接着是距离为 1 的节点，依此类推。实现 BFS 的核心数据结构是一个 FIFO 队列，原因很简单：我们希望先进队的节点的孩子也优先被处理。

在实现 BFS 时，有一个关键技巧需要掌握，那就是记录每层节点数量：在每一轮外层循环开始时，先获取当前队列的长度，这个长度就是当前层的节点总数。这些节点属于当前层，而在处理它们的过程中新加入队列的节点则属于下一层。

from collections import deque
from typing import List, Optional

def levelOrder(self, root: Optional[TreeNode]) -> List[List[int]]:
    if root is None:
        return []

    result: List[List[int]] = []
    queue: deque[TreeNode] = deque([root])

    while queue:
        # 在开始处理下一层的节点之前，先记录当前层的节点数量
        level_size = len(queue)
        level: List[int] = []

        for _ in range(level_size):
            node = queue.popleft()
            level.append(node.val)
            if node.left:  queue.append(node.left)
            if node.right: queue.append(node.right)

        result.append(level)

    return result

时间复杂度为 $O(n)$，空间复杂度为 $O(w)$，其中 $w$ 表示树的最大宽度。对于完全二叉树来说，最底层的节点数大约是 $n/2$，因此在最坏情况下空间复杂度为 $O(n)$。

通过记录每层节点数量的方式，这个方法可以作为许多类似问题的基础框架。比如：锯齿形层序遍历（交替改变层的方向）、二叉树的右视图（取每层的最后一个节点值）、最小深度（找到第一个叶子节点就返回）以及无权图中的最短路径问题，都可以直接复用这个模板，只需稍作调整即可解决不同需求。

LeetCode 104 —— 二叉树的最大深度：后序遍历的巧妙应用

一棵二叉树的最大深度，可以理解为左右子树中较深的那一部分深度加 1。这个定义天然适合用递归来实现，几乎可以直接翻译成代码：

def maxDepth(self, root: Optional[TreeNode]) -> int:
    if root is None:
        return 0
    left = self.maxDepth(root.left)
    right = self.maxDepth(root.right)
    return 1 + max(left, right)

这段代码的核心思想是后序遍历：只有先知道了左右子树的深度，才能计算出当前节点的深度。类似的解法框架还能用来解决其他问题，比如求二叉树的直径、判断是否为平衡二叉树（可以用 -1 表示不平衡的情况），以及计算子树节点值的总和——只需要调整最后的“合并”逻辑即可。

LeetCode 98 —— 验证二叉搜索树：中序遍历的妙用

很多人第一反应会写出这样的代码：node.left.val < node.val < node.right.val，然后递归检查左右子树。但这是错误的！ 这种方法只比较了当前节点和其直接子节点的值，而忽略了更深层的约束。在二叉搜索树（BST）中，左子树的所有节点值必须小于当前节点值，右子树的所有节点值必须大于当前节点值。一个深埋在右子树中的节点完全可能违反与远房祖先节点的大小关系。

正确的做法是维护一个动态的“合法区间” (low, high)，确保当前子树中的所有节点值都落在这个范围内：

def isValidBST(self, root: Optional[TreeNode]) -> bool:
    def check(node, low: float, high: float) -> bool:
        if node is None:
            return True
        if not (low < node.val < high):
            return False
        return (check(node.left,  low,        node.val) and
                check(node.right, node.val,   high))

    return check(root, float("-inf"), float("inf"))

还有一种思路更简洁：利用 BST 的中序遍历特性。我们知道，BST 的中序遍历结果必然是严格递增的序列。因此，我们只需模拟一次中序遍历，检查每个节点的值是否比前一个节点的值更大即可。

def isValidBST_inorder(self, root: Optional[TreeNode]) -> bool:
    prev = float("-inf")
    stack, cur = [], root
    while cur or stack:
        while cur:
            stack.append(cur)
            cur = cur.left
        cur = stack.pop()
        if cur.val <= prev:
            return False
        prev = cur.val
        cur = cur.right
    return True

LeetCode 105 —— 根据前序和中序遍历构建二叉树

这是二叉树构造问题中最经典的一道题，同时也是对“每种遍历方式到底提供了什么信息”解释得最清晰的一个例子。

两个关键点：
前序遍历的第一个元素永远是当前子树的根节点；
中序遍历则告诉我们，在知道根节点后，左子树在哪里结束、右子树从哪里开始（根左边的是左子树，右边的是右子树）。

这两条信息结合起来，就足以唯一确定一棵二叉树。图 5 展示了递归过程在示例树上的三层分解：

最简单的实现方式是每次递归都重新扫描中序数组，并对两个数组进行切片操作，但这会导致时间复杂度达到 $O(n^2)$，同时浪费大量内存。工业级的实现通常会先用一个哈希表记录“值 → 中序索引”的映射关系，然后通过索引递归构建树：

from typing import List, Optional

def buildTree(self, preorder: List[int], inorder: List[int]) -> Optional[TreeNode]:
    """LeetCode 105。时间复杂度 O(n)，空间复杂度 O(n)。"""
    inorder_index = {v: i for i, v in enumerate(inorder)}
    pre_iter = iter(range(len(preorder)))  # 每次访问一个节点时迭代一次

    def build(in_left: int, in_right: int) -> Optional[TreeNode]:
        if in_left > in_right:
            return None

        # 根据前序遍历的定义，当前要构建的子树的根节点就是 preorder 的下一个值。
        root_pre_idx = next(pre_iter)
        root_val = preorder[root_pre_idx]
        root = TreeNode(root_val)

        # 在中序遍历中找到根节点的位置，从而确定左右子树的边界。
        mid = inorder_index[root_val]

        # 必须先构建左子树！因为前序遍历的顺序是“根-左-右”，
        # 只有先递归左子树，`next(pre_iter)` 才能正确指向下一个根节点。
        root.left  = build(in_left, mid - 1)
        root.right = build(mid + 1, in_right)
        return root

    return build(0, len(inorder) - 1)

最容易出错的地方在于两个递归调用的顺序。前序遍历的顺序是“根-左-右”，因此左子树的根节点紧跟在当前根节点之后。如果先构建右子树，就会错误地消费前序遍历中的值，导致最终构建出的树结构不正确。

def buildTreePost(self, inorder: List[int], postorder: List[int]) -> Optional[TreeNode]:
    """LeetCode 106。时间复杂度 O(n)，空间复杂度 O(n)。"""
    inorder_index = {v: i for i, v in enumerate(inorder)}
    post = postorder
    self._i = len(post) - 1

    def build(in_left: int, in_right: int) -> Optional[TreeNode]:
        if in_left > in_right:
            return None
        root_val = post[self._i]
        self._i -= 1
        root = TreeNode(root_val)
        mid = inorder_index[root_val]
        root.right = build(mid + 1, in_right)  # 先构建右子树
        root.left  = build(in_left, mid - 1)
        return root

    return build(0, len(inorder) - 1)

为什么前序 + 中序、后序 + 中序都可以，但前序 + 后序却不行？因为只有中序遍历能够明确区分“左子树到哪里结束”。最经典的反例是两棵树：一棵是根节点为 1，左孩子为 2 的树；另一棵是根节点为 1，右孩子为 2 的树。它们的前序遍历都是 [1,2]，后序遍历都是 [2,1]，但中序遍历分别是 [2,1] 和 [1,2]，正是中序遍历帮助我们区分了这两种情况。

递归与迭代的选择

维度递归迭代
实现难度简单，直接反映问题定义复杂，需要手动管理栈
空间复杂度调用栈占用 $O(h)$显式栈占用 $O(h)$
栈溢出风险深度过大时可能溢出（Python 默认约 1000 层）无，完全由开发者控制
代码可读性逻辑清晰，易于理解通常更繁琐，容易引入错误
性能表现基本相当，存在常数级差异基本相当

实战建议：优先考虑递归实现，只有在以下三种情况下才切换到迭代：(a) 面试官明确要求；(b) 树的深度可能非常大，存在栈溢出风险；(c) 需要在遍历中途以复杂条件退出，而递归难以优雅表达。中序遍历的迭代模板尤其值得熟记，因为它在 BST 迭代器、第 $k$ 小元素等问题中几乎可以直接套用，甚至还能扩展到双指针在 BST 上寻找两数之和的场景。

Q&A

为什么 BST 的中序遍历结果一定是有序的？

这是因为 BST 的定义保证了所有左子树的节点值都小于当前节点，而所有右子树的节点值都大于当前节点。中序遍历的顺序是“先访问左子树（较小值），再访问当前节点，最后访问右子树（较大值）”，正好符合从小到大的排序逻辑。通过归纳法可以证明，这种遍历方式会按照递增顺序输出每个节点的值。这也是为什么中序遍历是验证 BST 或查找第 $k$ 小元素时最自然的选择。

是否可以通过任意两种遍历序列唯一重建一棵树？

并非所有组合都可以。前序+中序 和 后序+中序 在节点值不重复的情况下可以唯一确定一棵树；但 前序+后序 则不行。例如，两棵树分别是根节点 1 带一个左孩子 2，以及根节点 1 带一个右孩子 2，它们的前序和后序序列完全相同。原因在于：你需要一种遍历来确定根节点（前序或后序），还需要另一种遍历来区分左右子树（只有中序能做到这一点）。

如果节点值有重复怎么办？

当存在重复值时，即使是前序+中序也无法唯一确定一棵树。例如，preorder = [1, 1] 和 inorder = [1, 1] 可以对应两棵不同的树。LeetCode 通常会在题目约束中保证节点值互不相同，从而避免这个问题。在实际工程中，如果确实存在重复值，可以通过节点的身份标识（如 Python 中的 id(node)）来区分，而不是单纯依赖节点值。

Morris 遍历值得学习吗？

Morris 遍历通过临时修改树结构，将每个左子树的最右节点指向当前节点的中序后继，从而实现 $O(1)$ 的额外空间复杂度，同时保持 $O(n)$ 的时间复杂度。然而，这种方法在遍历过程中会修改树的结构，因此不适合并发场景，代码逻辑也更难理解和维护。在面试中提到 Morris 遍历可以展示你的知识广度，但在实际工程中，使用显式栈通常更为稳妥——牺牲一点常数因子换来更高的可维护性，通常是值得的。

为什么层序遍历的时间复杂度是 $O(n)$？

层序遍历时，每个节点都会恰好入队一次、出队一次，因此总的队列操作次数是 $2n$。队列的空间占用取决于树的最大宽度 $w$，即某一时刻队列中最多可能存储 $O(w)$ 个节点。这就是为什么层序遍历的时间复杂度是 $O(n)$，而空间复杂度由树的宽度决定。

总结

这篇文章的核心思想可以归结为两条通用法则，无论遇到什么样的二叉树问题，它们都能派上用场：

  1. DFS 的本质是一个固定的套路——访问节点、递归左子树、递归右子树——四种遍历方式（前序、中序、后序、层序）的区别仅仅在于“访问”这一步发生在哪个时机。 前序遍历是在处理子节点之前就完成访问，后序遍历则是在子节点处理完之后才访问，而中序遍历正好夹在两者之间；至于层序遍历，则把 DFS 的 LIFO 调用栈换成 FIFO 队列，优先处理距离根节点为 $k$ 的所有节点，然后再处理距离为 $k+1$ 的节点。

  2. 递归和显式栈本质上是同一个算法的不同表现形式。 栈的入栈操作相当于一次递归调用，出栈操作相当于从递归返回。一旦你理解了这一点，“将递归改写为迭代”的问题就不再需要额外的技巧，而是变成了一种机械化的转换过程。

构造类问题（比如 LeetCode 105 和 106）其实是这个套路的具体应用：前序或后序遍历帮你确定根节点，中序遍历帮你找到左右子树的分界点，然后递归地处理左右两部分即可。

面试前的检查清单：递归实现的前序、中序、后序和层序遍历——掌握；使用显式栈模板实现的迭代版中序遍历——掌握；利用“值→中序索引”哈希表优化到 $O(n)$ 的“前序+中序构造”方法——掌握；最大深度和验证 BST 这两个小而精的问题作为代表性热身题——掌握。当你把这些模板练到形成肌肉记忆后，绝大多数二叉树相关的 LeetCode 题目都可以在五分钟内轻松解决。
