提到动态规划（Dynamic Programming，简称 DP），很多人会觉得这是算法领域的一道分水岭，仿佛掌握了它就能从“普通选手”晋升为“高手”。但实际上，这种印象有点过于夸张了。DP 并不是什么神秘的黑魔法，也不是一堆需要死记硬背的技巧，而是一个非常朴素的思想：重复的子问题只算一次就够了，没必要反复折腾。无论是状态转移方程、滚动数组优化，还是区间 DP，本质上都是围绕这个核心思想展开的。

这篇文章的目的就是带你彻底搞明白 DP 的解题思路。其实只要能清晰回答以下三个问题，LeetCode 上绝大多数 DP 题目你都能轻松应对：

  1. dp[i] 究竟代表什么意思？（状态定义）
  2. 如何通过更小的子问题推导出 dp[i]？（状态转移）
  3. 最基础的情况是什么？（初始条件）

接下来，我们会用这套方法逐一解决七道经典 DP 问题：爬楼梯、打家劫舍、零钱兑换、最长递增子序列、0/1 背包、最长公共子序列以及编辑距离。无论你是初学者还是想巩固基础，这些题目都能帮你建立起对 DP 的直观理解。

为什么需要动态规划：重叠子问题

上面这张图是 fib(6) 的递归调用过程。虽然公式 fib(n) = fib(n-1) + fib(n-2) 在数学上完全正确，但实际运行时你会发现，fib(2) 被重复计算了 5 次，fib(3) 被算了 3 次，依此类推。每一层的计算量都会翻倍，最终时间复杂度达到了 $O(2^n)$。然而，真正不同的子问题其实只有 $n+1$ 个。

动态规划（DP）的核心思想非常简单：算过的子问题答案存下来，下次直接复用。根据实现方式的不同，DP 分为两种：

  • 自顶向下（记忆化递归）：保留递归的逻辑，但每次计算完一个子问题后，把结果存到缓存里。这样一来，那些重复的分支就被“剪掉”了。
  • 自底向上（表格法）：完全抛弃递归，从最小的子问题开始一步步往上填表，直到解决整个问题。

无论哪种方式，都能把原本 $O(2^n)$ 的复杂度降到 $O(n)$。不过，一个问题能用 DP 解决的前提是满足以下两个条件：

  • 最优子结构：问题的最优解可以通过组合子问题的最优解得到。比如最短路径、斐波那契数列、背包问题、编辑距离等都符合这个特性。
  • 重叠子问题：在朴素递归中，同一个子问题会被反复求解多次。而 DP 正是通过缓存避免了这种重复计算。

如果一个问题同时具备这两个特性，那么 DP 就是你的不二之选。如果只有最优子结构而没有重叠子问题（例如归并排序），加缓存也没用。如果两者都不具备，那就得考虑其他算法范式了。

三步搞定动态规划

本文中的所有动态规划（DP）问题都遵循同一个解题框架。反复练习，直到形成肌肉记忆。

第一步：明确状态定义

给 dp[i] 或 dp[i][j] 赋予一个精准无误的定义。模糊的状态定义是导致 DP 出错的罪魁祸首。好的定义通常长这样：

  • dp[i] = 到达第 $i$ 阶楼梯的总方法数。
  • dp[i] = 在前 $i$ 间房子中能偷到的最大金额。
  • dp[i][j] = 字符串 s1[0:i] 和 s2[0:j] 的最长公共子序列（LCS）长度。

检验标准：如果我告诉你 dp[i-1] 和 dp[i-2] 的值是正确的，你能不能直接写出 dp[i] 的递推公式？如果可以，说明状态定义是合理的。

第二步：写出状态转移方程

状态转移的核心问题是：「在位置 $i$ 上，我有哪些选择？哪个选择是最优的？」 写成代码可能只有一行，但背后的逻辑才是 DP 的精髓。

以打家劫舍问题为例：对于第 $i$ 间房子，你可以选择偷（获得 nums[i] 的收益，但只能从前两间房子的状态 dp[i-2] 推导），或者不偷（继承前一间房子的状态 dp[i-1]）。最终的状态转移公式就是取这两种选择的最大值。

第三步：确定初始条件

初始条件是那些不需要依赖其他状态就能直接得出答案的最小子问题。这一步如果出错，整个 DP 表都会崩塌；但如果定义正确，后续的状态会自然填充。

以爬楼梯问题为例：dp[0] = 1（站在地面算一种方法），dp[1] = 1（迈一步到达第一阶）。验证一下：dp[2] 应该是 2，而根据公式 dp[1] + dp[0] = 2，结果一致，说明初始条件没问题。

一维动态规划

一维动态规划（1D DP）是学习动态规划的入门级内容。它的状态通常由单个下标表示，状态转移也只依赖于前面固定数量的状态。这种简洁的结构非常适合用来熟悉动态规划的基本套路。

上图展示了一维 DP 的核心思想：每个状态从前面有限的几个状态推导而来。理解了这一点，就不难明白为什么我们可以将存储空间压缩到少数几个变量——只需要记住状态转移所需的最近几步即可。

LeetCode 70 · 爬楼梯

一个楼梯共有 $n$ 阶，每次可以爬 1 阶或 2 阶。问有多少种不同的方法可以爬到顶端？

解题思路：

  • 状态定义：dp[i] 表示到达第 $i$ 阶的方法总数。
  • 状态转移：dp[i] = dp[i-1] + dp[i-2]。最后一步要么是从 $i-1$ 阶跨一步上来，要么是从 $i-2$ 阶跨两步上来。
  • 初始条件：dp[0] = dp[1] = 1，即爬到第 0 阶和第 1 阶都只有 1 种方法。

def climbStairs(n: int) -> int:
    if n <= 1:
        return 1
    dp = [0] * (n + 1)
    dp[0] = dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
    return dp[n]

时间复杂度为 O(n)，空间复杂度为 O(n)。由于状态转移只依赖前两个值，可以进一步优化空间复杂度为 O(1)：

    return prev1

LeetCode 198 · 打家劫舍

每间房子里有一定数量的现金，但相邻的房子不能同时偷。问最多能偷到多少钱？

解题思路：

  • 状态定义：dp[i] 表示考虑前 $i$ 间房子时能偷到的最大金额。
  • 状态转移：dp[i] = max(dp[i-1], dp[i-2] + nums[i])。对于第 $i$ 间房子，可以选择不偷（沿用 dp[i-1]），或者偷（加上 nums[i] 和 dp[i-2]）。
  • 初始条件：dp[0] = nums[0]，dp[1] = max(nums[0], nums[1])。

def rob(nums: list[int]) -> int:
    n = len(nums)
    if n == 0:
        return 0
    if n == 1:
        return nums[0]
    prev2, prev1 = nums[0], max(nums[0], nums[1])
    for i in range(2, n):
        prev2, prev1 = prev1, max(prev1, prev2 + nums[i])
    return prev1

优化后空间复杂度为 O(1)。这种「选与不选」的模型非常通用，适用于所有类似「选择某个位置会限制其邻居」的问题。

LeetCode 322 · 零钱兑换

给定若干硬币面额和一个目标金额 amount，返回凑出该金额所需的最少硬币数；如果无法凑出，返回 $-1$。

解题思路：

  • 状态定义：dp[a] 表示凑出金额 $a$ 所需的最少硬币数。
  • 状态转移：dp[a] = min(dp[a - c] + 1 for c in coins if c <= a)。枚举最后一枚硬币的面额 $c$，剩余部分由 dp[a - c] 解决。
  • 初始条件：dp[0] = 0，其他位置初始化为一个不可能达到的值（如 amount + 1）。

def coinChange(coins: list[int], amount: int) -> int:
    INF = amount + 1
    dp = [INF] * (amount + 1)
    dp[0] = 0
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != INF else -1

时间复杂度为 O(amount * len(coins))，空间复杂度为 O(amount)。将不可达状态设为 amount + 1 是一个小技巧，可以让代码更简洁。

LeetCode 300 · 最长递增子序列

给定一个数组，求其中最长严格递增子序列的长度。

这是第一道状态定义为「以某位置结尾」的 DP 问题，而非「考虑前缀」，这个细微差别非常重要。

解题思路：

  • 状态定义：dp[i] 表示以 nums[i] 结尾的最长递增子序列的长度。
  • 状态转移：dp[i] = 1 + max(dp[j] for j < i if nums[j] < nums[i])。如果没有满足条件的 $j$，则 dp[i] = 1。
  • 初始条件：所有 dp[i] 初始值为 $1$，因为单个元素本身就是一个长度为 1 的子序列。
  • 最终答案：max(dp)，因为最长递增子序列可能在任意位置结束。

def lengthOfLIS(nums: list[int]) -> int:
    n = len(nums)
    dp = [1] * n
    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)

时间复杂度为 O(n^2)。虽然可以用 bisect 实现 O(n log n) 的解法，但建议先熟练掌握 DP 版本，因为它更直观地展示了问题的结构。

二维动态规划

当状态需要同时追踪两件独立的事情时，比如两个字符串的位置、物品与容量、网格中的行和列，二维 DP 就派上用场了。核心思路不变，只是从一维的“一行格子”变成了二维的“一张表”。

经典问题：0/1 背包

给定 $n$ 件物品，每件物品的重量为 $w_i$，价值为 $v_i$，以及一个容量为 $W$ 的背包。要求选择一些物品放入背包，在不超过容量的前提下最大化总价值。每件物品最多只能选一次。

图中橙色单元格展示了示例（重量 [2,3,4,5]、价值 [3,4,5,6]、容量 $W=5$）的最优回溯路径。最终答案存储在 dp[n][W] 中，通过回溯这些单元格可以还原出具体选择了哪些物品。

解题套路：

  • 状态定义：dp[i][w] 表示从前 $i$ 件物品中选择，且背包容量为 $w$ 时的最大价值。
  • 状态转移：
    • 如果不选第 $i$ 件物品，则价值为 dp[i-1][w]；
    • 如果选第 $i$ 件物品且能装下，则价值为 dp[i-1][w - w_i] + v_i；
    • 取两者的最大值。
  • 边界条件：dp[0][w] = 0，即没有物品时价值为 0。

def knapsack(weights: list[int], values: list[int], capacity: int) -> int:
    n = len(weights)
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]
    for i in range(1, n + 1):
        wi, vi = weights[i - 1], values[i - 1]
        for w in range(capacity + 1):
            dp[i][w] = dp[i - 1][w]
            if w >= wi:
                dp[i][w] = max(dp[i][w], dp[i - 1][w - wi] + vi)
    return dp[n][capacity]

时间复杂度和空间复杂度均为 O(n * W)。如果进行一维空间优化，需要注意一个关键点：容量必须逆序遍历，否则同一件物品可能会被重复选择。

def knapsack(weights: list[int], values: list[int], capacity: int) -> int:
    dp = [0] * (capacity + 1)
    for wi, vi in zip(weights, values):
        # 逆序遍历：确保 dp[w - wi] 还是「上一轮」的值
        for w in range(capacity, wi - 1, -1):
            dp[w] = max(dp[w], dp[w - wi] + vi)
    return dp[capacity]

如果改成正序遍历，dp[w - wi] 在本轮可能已经被当前物品更新过，这实际上是完全背包（每件物品可选无限次）的递推式。虽然完全背包本身是个有用的问题，但在 0/1 背包中这就是一个 bug。

LeetCode 1143 · 最长公共子序列

求两个字符串的最长公共子序列长度。

解题套路：

  • 状态定义：dp[i][j] 表示 text1[0:i] 和 text2[0:j] 的最长公共子序列长度。
  • 状态转移：
    • 如果 text1[i-1] == text2[j-1]，则 dp[i][j] = dp[i-1][j-1] + 1（沿对角线延伸匹配）；
    • 否则，取两者中的较大值：dp[i][j] = max(dp[i-1][j], dp[i][j-1])（分别尝试丢弃 text1 或 text2 的当前字符）。
  • 边界条件：dp[0][j] = dp[i][0] = 0，空串与任何字符串的公共子序列长度为 0。

def longestCommonSubsequence(text1: str, text2: str) -> int:
    m, n = len(text1), len(text2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if text1[i - 1] == text2[j - 1]:
                dp[i][j] = dp[i - 1][j - 1] + 1
            else:
                dp[i][j] = max(dp[i - 1][j], dp[i][j - 1])
    return dp[m][n]

时间复杂度为 O(m * n)，空间复杂度同样为 O(m * n)。可以通过滚动数组优化到 O(min(m, n))：

def longestCommonSubsequence(text1: str, text2: str) -> int:
    if len(text1) < len(text2):
        text1, text2 = text2, text1
    m, n = len(text1), len(text2)
    prev = [0] * (n + 1)
    curr = [0] * (n + 1)
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if text1[i - 1] == text2[j - 1]:
                curr[j] = prev[j - 1] + 1
            else:
                curr[j] = max(prev[j], curr[j - 1])
        prev, curr = curr, prev
    return prev[n]

LeetCode 72 · 编辑距离

求将字符串 word1 转换为字符串 word2 所需的最少操作次数，允许的操作包括插入、删除和替换。

编辑距离可以看作是 LCS 的扩展版，状态转移多了一些分支，但整体框架依然相似。

解题套路：

  • 状态定义：dp[i][j] 表示将 word1[0:i] 转换为 word2[0:j] 所需的最少操作次数。
  • 状态转移：
    • 如果 word1[i-1] == word2[j-1]，则无需操作，dp[i][j] = dp[i-1][j-1]；
    • 否则，取以下三种操作的最小值：
      • 删除 word1[i-1]：dp[i-1][j] + 1；
      • 插入 word2[j-1]：dp[i][j-1] + 1；
      • 替换：dp[i-1][j-1] + 1。
  • 边界条件：
    • dp[i][0] = i，即将 word1 全部删除；
    • dp[0][j] = j，即将 word2 全部插入。

def minDistance(word1: str, word2: str) -> int:
    m, n = len(word1), len(word2)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(m + 1):
        dp[i][0] = i
    for j in range(n + 1):
        dp[0][j] = j
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if word1[i - 1] == word2[j - 1]:
                dp[i][j] = dp[i - 1][j - 1]
            else:
                dp[i][j] = 1 + min(
                    dp[i - 1][j],      # 删除
                    dp[i][j - 1],      # 插入
                    dp[i - 1][j - 1],  # 替换
                )
    return dp[m][n]

时间复杂度和空间复杂度均为 O(m * n)，同样可以通过滚动数组优化到 O(min(m, n))。

空间优化

空间优化的核心思想并不复杂，但要真正写对却需要格外小心。每次实现时，都要问自己一个关键问题：当前状态转移实际需要用到表格中的哪些部分？

  • 两个变量就够了：如果 dp[i] 仅仅依赖于 dp[i-1] 和 dp[i-2]（比如斐波那契数列、爬楼梯问题、打家劫舍问题），那么完全可以用两个变量滚动更新，无需维护整个数组。
  • 单行数组：当 dp[i][j] 只依赖上一行的数据时（例如背包问题、最长公共子序列 LCS、编辑距离问题），可以只保留一行数组。不过要注意遍历顺序：正序还是倒序会直接影响你读取的值是「上一行」还是「当前行已经被覆盖的新值」。
  • 两行数组：如果既要读取对角线位置的 dp[i-1][j-1]，又要读取当前行的值，那么用两行数组会更稳妥。虽然比单行多占用一点内存，但逻辑更清晰，不容易出错。

经验之谈：先写出完整的 O(m * n) 版本，确保逻辑正确，再进行空间压缩优化。直接跳到最优空间版本，很容易引入难以察觉的 off-by-one 错误或读写顺序问题。

自顶向下与自底向上

上面提到的每道题，都可以用记忆化递归写出一个渐进复杂度完全等价的解法：

from functools import cache

@cache
def fib(n: int) -> int:
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)

那么，什么时候该用哪种方法呢？

  • 自顶向下的记忆化递归更适合那些天然适合用递归思路解决的问题，比如树形 DP、博弈论相关问题，或者状态转移图比较复杂的题目。在 Python 中，functools.cache 的存在让这种写法几乎没有任何额外成本。
  • 自底向上的迭代填表法则更适合需要严格控制遍历顺序的场景，比如为了优化空间复杂度，或者当输入规模较大时，避免因 Python 递归深度限制而导致栈溢出。

在面试中，优先选择自底向上的实现方式，可以向面试官传递一个信号：你对遍历顺序和空间优化有过深入思考。而如果情况允许，灵活切换到自顶向下的递归写法也完全没问题，这表明你对递归的理解很扎实。最终的目标是掌握两种方法，并能够根据需求自如地在它们之间切换。

常见 DP 类型分布图

绝大多数动态规划问题都可以归入以下四类框架中。每类问题都有其典型特征和状态设计方式，理解这些模式能帮助你快速找到解题方向。

  • 线性 DP：状态由单一维度的索引表示，通常是一个数组或序列上的位置。比如爬楼梯、打家劫舍、最长递增子序列（LIS）、最大子数组和等问题。这类问题的核心难点在于如何定义状态——是选择「以 $i$ 结尾」还是「考虑前缀 $0..i$」。

  • 二维 / 网格 DP：状态由两个维度的索引表示，通常是两个字符串的下标，或者网格中的行列坐标。例如最长公共子序列（LCS）、编辑距离、不同路径、最小路径和等经典问题。这类问题的状态转移方程往往需要同时考虑两个维度的变化。

  • 区间 DP：状态由一个区间 $[i, j]$ 表示，填表顺序通常是按照区间长度从小到大逐步推进。比如戳气球、矩阵链乘法、最长回文子序列等问题。这类问题的关键在于如何利用子区间的最优解来构造更大的区间解。

  • 状态机 DP：在位置索引的基础上增加一两个额外维度，用于描述更复杂的状态。例如是否持有股票、剩余交易次数 $k$、奇偶性、bitmask 等。带冷却时间或交易限制的股票买卖问题、经典的 0/1 背包问题，以及状压 DP 都属于这一类。这类问题的状态设计灵活多变，但核心思想仍然是通过扩展维度来记录更多信息。

遇到新问题时，第一步就是判断它属于哪一类。类别本身会为你提供状态设计的灵感和方向。

常见错误合集

以下是新手和老手都容易踩的坑，记得反复检查：

  • 状态定义不清晰：写状态转移时总觉得别扭，原因可能是 dp[i] 的定义模棱两可，只是「大概表示答案」。动手写代码前，先把状态定义得更精确。
  • 初始值不对：递推公式本身没问题，但起点的值错了，比如偏移了一格。快速排查的方法是手动计算前三个或四个值，看看是否符合预期。
  • 搞混「以 $i$ 结尾」和「考虑前缀 $0..i$」：在最长递增子序列（LIS）问题中，这会导致你误用 max(dp) 和 dp[-1]，结果天差地别。
  • 一维背包遍历方向反了：如果用正序更新状态，0/1 背包就悄悄变成了完全背包，逻辑全乱了。
  • 记忆化递归默认参数是可变对象：像 def f(n, memo={}): 这样的写法会让 memo 在多次函数调用间共享，引发难以察觉的 bug。推荐改用 functools.cache。
  • 过早优化空间复杂度：这种做法不仅容易出错，调试起来也很麻烦，而且在白板上通常没必要。建议先把逻辑写清楚，最后再考虑压缩空间。

Q&A

如何判断一个问题适合用 DP 解决？

留意题目中的关键词，比如「最大」「最小」「计数」「最长」「最少」等。此外，观察问题是否可以通过做出某个决策后递归到一个更小的子问题来解决，同时状态空间是否足够小以便进行缓存。如果你的第一反应是写一个分支很多的递归函数，那么很可能需要用动态规划（DP）。

应该选择自顶向下还是自底向上的方法？

如果是树形结构或状态图不规则的问题，优先考虑自顶向下的递归实现；如果需要优化空间复杂度，则更适合用自底向上的迭代实现。在面试中，只要分析正确，两种方法都能拿到满分。

定义了状态，但状态转移总是写得不顺畅？

这通常是因为状态定义过于粗糙，缺少必要的信息。可以尝试增加一维，比如：标记「是否使用了上一个元素」、记录「还剩几次操作机会」、或者用 bitmask 表示「已经覆盖了哪些子集」。记住，状态必须包含所有影响转移的信息。

小数据没问题，大数据出错？

这种情况大概率是以下原因之一：C++ 或 Java 中的整数溢出、Python 中递归深度过大导致栈溢出、或者你在空间优化版本中引入了 bug，而二维版本没有这个问题。建议运行未优化的版本，针对出错的用例进行对比调试。

DP 和贪心算法有什么本质区别？

贪心算法在每一步都直接选择当前最优的局部解，并且一旦做出选择就不再回头调整。而 DP 会在每一步枚举所有可能的合理选择，并通过比较选出全局最优解，甚至可能推翻之前的决策。例如，活动选择问题可以用贪心算法解决，但任意面额的零钱兑换问题就不行。

DP 是否总能进行空间优化？

并非如此。只有当状态转移仅依赖于状态表中有限的一部分时，才能进行空间优化。对于那些需要回溯整个 DP 表来还原最优解的问题（而不仅仅是求最优值），就必须保留完整的状态表。

练习路径

一个循序渐进掌握 DP 的合理顺序：

  1. 从一维问题入手：比如经典的爬楼梯（Climbing Stairs）、打家劫舍（House Robber）和最大子数组（Maximum Subarray）。这些问题简单直观，适合熟悉 DP 的基本思想。
  2. 扩展到二维：接下来可以尝试零钱兑换（Coin Change）、不同路径（Unique Paths）以及 0/1 背包问题。这些题目引入了第二个维度，帮助你理解状态的多维表示。
  3. 双字符串问题：像最长公共子序列（LCS）、编辑距离（Edit Distance）以及不同的子序列（Distinct Subsequences）这类问题，通常涉及两个字符串的匹配或变换，是进一步提升的好选择。
  4. 状态机模型：例如带冷冻期的最佳买卖股票时机（Best Time to Buy and Sell Stock with cooldown），或者限制最多 $k$ 次交易的情况。这类问题的状态转移逻辑更复杂，但非常贴近实际应用场景。
  5. 区间类问题：如最长回文子序列（Longest Palindromic Subsequence）和戳气球（Burst Balloons）。这些问题需要考虑区间的划分和合并，对思维能力要求较高。
  6. 状压 DP：最后挑战旅行商问题（Travelling Salesman）或者划分为 K 个相等子集（Partition to K Equal Subsets）。这类问题通常需要用到位掩码（bitmask）来高效表示状态。

如果遇到一道题完全没有思路，不妨先写出暴力递归解法。递归函数中的状态变量往往就是 DP 的状态定义，而函数参数则对应 DP 的索引。在暴力递归的基础上加上记忆化优化，确保代码能跑通后，再根据需要改写成表格法（tabulation），以节省空间。

总结

这篇文章通篇其实只讲了一个套路，用它解决了七道题：状态定义、状态转移、初始条件。只要把这三点搞清楚了，写代码就变成了一件按部就班的事，复杂度分析也几乎是水到渠成。

动态规划的难点从来不在代码本身，而在于找到一个精准的状态定义，让状态转移变得显而易见。把精力花在这一点上，后续的工作自然会迎刃而解。

文中提到的这七道题——爬楼梯、打家劫舍、零钱兑换、LIS、0/1 背包、LCS、编辑距离——不仅仅是练习题那么简单。它们更像是模板，几乎所有的高难度 DP 问题最终都能归结到这些经典模型上。把这些题目彻底掌握，LeetCode 上很多标着 Hard 的题目就会变成「换了个马甲的老朋友」。
