#二分查找#LeetCode#算法
Read in English →

二分查找，这个算法乍一听谁都懂，但真到了面试时让你现场写，多半会翻车。原理简单得不能再简单——每次把搜索范围缩小一半——可一旦落到代码上，各种边界条件、死循环、返回值错误就会接踵而至，让人怀疑人生。这篇文章的目标不是给你再塞一个标准模板，而是帮你建立一套思维模型，让你真正明白这些模板为什么是现在的样子。同时，我们会提供一个小工具箱（三个模板加上答案空间模式），足以应对 LeetCode 上的绝大部分题目。

我们的所有讨论都基于一个核心不变量：在每一轮迭代中，如果答案存在，它一定位于当前的搜索区间内。只要把这个不变量彻底理解了，那些看似神秘的选择——比如用 < 还是 <=，right = mid 还是 right = mid - 1——就不再需要死记硬背，而是可以自然推导出来的结果。

一、心智模型：维护一个逐步缩小的搜索范围

二分查找的核心思想是逐步缩小搜索区间。我们用两个指针 l 和 r 来划定当前可能包含目标值的数组范围，每次取中间位置 m，检查 nums[m] 的值，并根据这个值直接排除掉一半的候选范围。上图展示了一个典型场景：在一个长度为 16 的有序数组中查找目标值 23，经过四次迭代后，搜索范围从最初的 16 个元素收缩到只剩 1 个。

整个算法的正确性依赖于一个关键的不变量：

不变量：在每一轮迭代开始时，如果目标值存在，它一定位于 [l, r]（或 [l, r)，视具体实现而定）这个区间内。

只要这条不变量始终成立，算法就能保证正确性。但如果某一步操作破坏了这条不变量——例如，在寻找目标值最左侧出现位置时，遇到 nums[m] == target 却错误地将右边界设为 r = m - 1——那么后续的所有搜索都将失去意义。

为什么二分查找的时间复杂度是对数级别的？经过 $k$ 轮迭代后，剩余的搜索范围最多还剩 $\lceil n / 2^k \rceil$ 个元素。要让范围收缩到只剩 1 个元素，需要的迭代次数是 $k = \lceil \log_2 n \rceil$。即便是十亿规模的数据，也只需要大约 30 次比较——这正是数据库索引、文件系统和版本控制工具（如 Git bisect）广泛使用二分查找的原因。

二分查找的适用条件并不仅仅是"数组有序"，而是"判定条件具有单调性"。换句话说，只要你能定义一个布尔函数 feasible(x)，它的返回值随着 x 的增大恰好从 False 变为 True（或者反过来），就可以用二分查找来定位这个变化点。本文后续会详细介绍如何利用这一特性解决"答案空间二分"问题。

二、基础模板：闭区间 [l, r]

在有序数组中查找等于 target 的任意下标，如果找不到就返回 -1。这里我们使用闭区间 [l, r] 并配合循环条件 l <= r 来实现：

def binary_search(nums, target):
    """返回任意一个满足 nums[i] == target 的下标，若不存在则返回 -1。"""
    l, r = 0, len(nums) - 1
    while l <= r:
        m = l + (r - l) // 2     # 避免溢出（C++/Java 中尤其重要）
        if nums[m] == target:
            return m
        if nums[m] < target:
            l = m + 1            # target 一定在右半部分
        else:
            r = m - 1            # target 一定在左半部分
    return -1

要记住的关键点只有三个：

  1. 初始边界：r = len(nums) - 1，因为右边界是包含在内的。
  2. 循环条件：l <= r，只要 l 不超过 r，区间 [l, r] 就非空（即使只剩一个元素）。
  3. 更新规则：当 nums[m] != target 时，m 这个位置已经不可能是答案，所以通过 m + 1 或 m - 1 把它彻底排除。

公式 m = l + (r - l) // 2 和 (l + r) // 2 数学上等价，但前者在 l + r 超过 INT_MAX 时不会溢出。虽然 Python 支持大整数无所谓，但这个习惯在 C++ 和 Java 中至关重要。

class Solution:
    def search(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums) - 1
        while l <= r:
            m = l + (r - l) // 2
            if nums[m] == target:
                return m
            if nums[m] < target:
                l = m + 1
            else:
                r = m - 1
        return -1

边界情况会自动处理：

  • 如果数组为空，r = -1，循环条件直接不成立，返回 -1。
  • 如果数组只有一个元素，l == r == 0，循环恰好执行一次，要么找到目标值，要么退出循环返回 -1。

三、边界模板：寻找第一个或最后一个位置

很多问题不仅仅关心目标值是否存在，还要求找到第一个满足条件的位置，或者最后一个满足条件的位置，甚至是新元素的插入位置。这时，基础二分模板就显得力不从心了，因为当 nums[m] == target 时，我们不能直接返回——可能左边还有更早的答案，或者右边还有更晚的答案。

上图展示了两套模板在数组 nums = [1, 2, 5, 5, 5, 5, 8, 9] 和目标值 target = 5 上的行为。它们的代码几乎完全相同，唯一的区别在于比较符 < 是否取等号。

def left_bound(nums, target):
    """半开区间 [l, r)，返回最左插入位置。"""
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] < target:
            l = m + 1            # m 太小，丢弃
        else:
            r = m                # m 可能是答案，留在 [l, r) 中
    return l                     # 退出时 l == r

有三个关键点必须严格遵守：

  • r = len(nums)，而不是 len(nums) - 1。我们使用半开区间 [l, r)，这样即使答案需要插在数组末尾，也能用 l = len(nums) 表示。
  • l < r，这是半开区间的约定。当 l == r 时，区间为空，循环自然终止。
  • r = m，而不是 m - 1。因为当 nums[m] >= target 时，m 本身可能是答案，绝不能排除。

循环结束后，l 就是第一个满足 nums[i] >= target 的下标。如果需要找到目标值的第一次出现位置，还需要额外验证：l < len(nums) and nums[l] == target。

def right_bound(nums, target):
    """半开区间 [l, r)，返回最后一个匹配位置的下标 + 1。"""
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] <= target:
            l = m + 1            # 继续向右探索
        else:
            r = m
    return l - 1                 # 实际答案需要减 1（若不存在，则通过合法性检查兜底）

代码结构与左边界完全一致，唯一的变化是将 < 替换为 <=。循环结束时，l 是第一个满足 nums[i] > target 的位置，因此 l - 1 才是最后一个满足 nums[i] <= target 的位置。同样需要验证：l - 1 >= 0 and nums[l - 1] == target。

class Solution:
    def searchInsert(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums)
        while l < r:
            m = l + (r - l) // 2
            if nums[m] < target:

                r = m

以 nums = [1, 3, 5, 6] 和 target = 2 为例：

  • 初始时，m = 2, nums[2] = 5 >= 2，所以 r = 2；
  • 接着，m = 1, nums[1] = 3 >= 2，所以 r = 1；
  • 最后，m = 0, nums[0] = 1 < 2，所以 l = 1。 此时 l == r，返回 1。将 2 插入到下标 1，得到 [1, 2, 3, 5, 6]，数组依然有序。

class Solution:
    def searchRange(self, nums: list[int], target: int) -> list[int]:
        def left_bound():
            l, r = 0, len(nums)
            while l < r:
                m = l + (r - l) // 2
                if nums[m] < target:
                    l = m + 1
                else:
                    r = m
            return l

        def right_bound():
            l, r = 0, len(nums)
            while l < r:
                m = l + (r - l) // 2
                if nums[m] <= target:
                    l = m + 1
                else:
                    r = m
            return l - 1

        lo = left_bound()
        if lo == len(nums) or nums[lo] != target:
            return [-1, -1]
        return [lo, right_bound()]

class Solution:
    def firstBadVersion(self, n: int) -> int:
        l, r = 1, n
        while l < r:
            m = l + (r - l) // 2
            if isBadVersion(m):
                r = m            # m 可能就是第一个坏版本
            else:
                l = m + 1        # m 是好的，第一个坏版本在右边
        return l

这是对二分本质的最佳诠释：二分的核心在于单调性，无论是数组还是 API，只要存在单调性，就可以用二分解决。

四、两种区间约定，选一种用到底

在第二节中我们使用了闭区间 [l, r]，而第三节则采用了半开区间 [l, r)。这两种方式本身都没有问题，但如果在同一个函数里混用它们，就会导致无限循环。

上图展示了两种约定在同一输入下的运行对比。每一步下方的区间标记清楚地展现了两者的差异：闭区间 [l, r] 包含右端点，而半开区间 [l, r) 则不包含。具体规则如下：

区间约定初始 r循环条件r 更新方式
闭 [l, r]len(nums)-1l <= rr = m - 1
半开 [l, r)len(nums)l < rr = m

建议每个函数从头到尾坚持使用同一种约定。新手最容易犯的一个致命错误是将 l < r 和 r = m - 1 搭配使用：当区间缩小到 [l, l] 时，计算得到的中间值 m = l，随后更新 r = l - 1，虽然循环退出了，但可能已经跳过了正确答案。反过来更糟糕：如果用 l <= r 配合 r = m，当 l == r == m 时，r = m 不会改变，导致程序陷入死循环。

五、旋转数组：还能不能用二分？

前面讨论的数组都是完全有序的，但旋转数组虽然整体无序，却保留了一个重要的特性：无论从哪个位置切开，至少有一半是有序的。这个性质让我们依然可以用 binary search 来解决问题。

上图展示了在 nums = [4, 5, 6, 7, 0, 1, 2] 中寻找目标值 0 的过程。每一步我们都会先判断哪一边是有序的（通过比较 nums[l] 和 nums[m]），然后检查目标值是否落在有序的那一段范围内：如果是，就继续在这一段里搜索；如果不是，就转向另一段。

class Solution:
    def search(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums) - 1
        while l <= r:
            m = l + (r - l) // 2
            if nums[m] == target:
                return m
            if nums[l] <= nums[m]:
                # 左半部分 [l..m] 是有序的
                if nums[l] <= target < nums[m]:
                    r = m - 1
                else:
                    l = m + 1
            else:
                # 右半部分 [m..r] 是有序的
                if nums[m] < target <= nums[r]:
                    l = m + 1
                else:
                    r = m - 1
        return -1

这里需要注意一个细节：nums[l] <= nums[m] 中的 <=。当 l == m 时，左半部分只剩下一个元素 nums[l]，而单个元素必然是有序的。这个等号正好处理了这种边界情况，避免漏掉解。

六、寻找峰值：在无序数组上应用二分查找

所谓峰值，指的是比左右邻居都大的元素。虽然数组本身是无序的，但我们可以利用一种特殊的单调性——斜率的变化方向，来高效地找到峰值。

class Solution:
    def findPeakElement(self, nums: list[int]) -> int:
        l, r = 0, len(nums) - 1
        while l < r:
            m = l + (r - l) // 2
            if nums[m] < nums[m + 1]:
                l = m + 1        # 当前点在上升段，峰值一定在右侧
            else:
                r = m            # 当前点在下降段或已是峰值，往左侧查找
        return l

当 nums[m] < nums[m + 1] 时，说明当前点处于上升趋势；根据题目约定，数组两端视为负无穷（nums[-1] = -inf 和 nums[n] = -inf），因此继续向右搜索必然会遇到下降，从而右侧必然存在一个峰值。而当 nums[m] >= nums[m + 1] 时，当前点可能已经位于峰值，或者正处于下降趋势；结合左端为负无穷的约定，左侧（包括当前点）必然包含一个峰值。无论是哪种情况，搜索区间都会缩小一半，同时保证了“当前区间内至少有一个峰值”这一不变量始终成立。

这道题堪称“利用局部单调性在无序数组中进行二分查找”的经典案例。一旦理解了局部单调性的核心思想，代码实现就变得非常机械化，几乎可以套用模板完成。

七、答案空间上的二分查找：跳出数组的束缚

二分查找的强大之处在于，它不仅可以用于数组，还能直接在答案的可能取值范围中进行搜索。只要我们能定义一个判定函数来验证某个值是否满足条件，就可以用二分查找快速定位最优解。

这种方法适用的场景需要满足以下三个条件：

  1. 答案是一个整数（或实数），并且我们知道它的取值范围 $[lo, hi]$。
  2. 存在一个判定函数 feasible(x)，可以判断 x 是否是合法解。
  3. feasible 是单调的：存在一个阈值 $x^*$，使得当 x < x^* 时 feasible(x) 返回 False，而当 x \geq x^* 时返回 True（或者反过来）。

通过二分查找，我们可以在 $\log(hi - lo)$ 次迭代内找到这个阈值 $x^*$，每次迭代只需调用一次 feasible 函数。上图展示的是“在 D 天内运送所有包裹”问题：上方的图展示了“所需天数 vs 载重”的阶梯函数（单调递减），下方的图则演示了二分区间 [l, r] 如何逐步收缩到最终答案。

class Solution:
    def shipWithinDays(self, weights: list[int], D: int) -> int:
        def feasible(cap: int) -> bool:
            days, cur = 1, 0
            for w in weights:
                if cur + w > cap:
                    days += 1
                    cur = w
                    if days > D:
                        return False
                else:
                    cur += w
            return True

        l, r = max(weights), sum(weights)
        while l < r:
            m = l + (r - l) // 2
            if feasible(m):
                r = m
            else:
                l = m + 1
        return l

这里的设计有三个关键点：

  • 下界 max(weights)：如果载重连最重的单个包裹都装不下，那显然无法完成任务。
  • 上界 sum(weights)：假设一天内运完所有包裹，这显然是可行的，因此作为上限。
  • 单调性：如果载重 c 可行，那么任何比 c 更大的载重 c' 也一定可行（因为每天能运更多，只会更快完成）。可行解的范围是一个上闭区间 $[x^*, +\infty)$，使用左边界模板正好能找到 $x^*$。

八、四种常见 bug，按出现频率排序

写二分时，基本只会犯这四种错误。

区间条件和更新逻辑不匹配。l < r 要配 r = m - 1，而 l <= r 则要配 r = m。记住这个死规矩：闭区间用 <= 和 m ± 1，半开区间用 < 和 m。其他任何组合都是错的，别乱创新。

返回值搞错了。标准模板返回的是 m（或者 -1），左边界模板返回 l，右边界模板返回 l - 1。最容易出问题的是右边界模板，经常漏掉 - 1。不管用哪个模板，最后一定要做一次合法性检查：0 <= idx < len(nums) and nums[idx] == target，确认目标值确实存在。

mid 取整忘了加 +1。如果更新逻辑是 l = m（而不是 l = m + 1），那么中点必须换成上中位数：m = l + (r - l + 1) // 2。否则当 r = l + 1 时，m 会等于 l，然后 l = m 不会改变区间范围，陷入死循环。LeetCode 第 69 题（sqrt）就是这种错误的经典案例。

(l + r) // 2 整数溢出。Python 没这个问题，但 C++ 或 Java 中，当 l 和 r 接近 $2^{31}$ 时就会爆掉。养成写 l + (r - l) // 2 的习惯，成本为零，但换语言时能救你一命。

简单调试方法：在循环里加一句 print(f"l={l} r={r} m={m} nums[m]={nums[m]}")，盯住三点：(a) 区间每轮都在缩小；(b) 不变量“答案在 [l, r] 内”始终成立；(c) 返回的下标对应的值符合预期。

九、如何选择合适的模板

一个简单的决策树可以帮助我们应对大多数场景：

  • 寻找任意匹配的索引，重复元素无关紧要：使用标准模板，闭区间 [l, r]。
  • 寻找第一个匹配项、插入位置或最小的有效值：采用左边界模板。
  • 寻找最后一个匹配项或最大的有效值：使用右边界模板。
  • 针对黑盒性质的判定（如版本号、容量、速度等）：直接在答案空间上应用左边界模板，完全忽略数组本身。
  • 数组整体无序但局部单调（如峰值问题、旋转数组）：依然使用标准模板，但比较逻辑需要关注结构性质，比如哪一半是有序的、哪一侧在递增，而不是简单地与目标值比较。

如果不确定该选哪种模板，优先考虑半开区间 [l, r)。这种形式更通用——它能自然处理“插入到末尾”的情况，从整数索引切换到浮点数时也无需修改代码，循环条件 l < r 更清晰地表达了终止条件。

十、二分查找在实际工程中的应用

二分查找可不只是面试中的“套路题”。它是少数几个真正支撑现代技术基础设施的核心算法之一。

数据库索引——比如 B 树、B+ 树、LSM-tree 的有序段——本质上就是一种多路二分查找，只不过分支因子远大于 2。在一个包含十亿行数据的表上执行 WHERE id = 12345 查询，其实只需要大约 $\log_B n \approx 4$ 次磁盘读取，其中 $B$ 是页扇出（page fanout）。如果没有二分查找，OLTP 数据库根本无法高效运行。

git bisect 则是将二分查找的思想用在了代码提交历史中。标记一个提交为“正常”（good），另一个稍后的提交为“有问题”（bad），然后自动检出中间的提交让你测试，并递归缩小范围。即使面对上千次提交的历史，也能通过大约 10 次测试精准定位引入问题的那次提交。

各种编程语言的标准库函数，例如 Python 的 bisect_left 和 bisect_right、C++ 的 lower_bound 和 upper_bound、Java 的 Arrays.binarySearch，实际上都是基于第二、三节提到的模板实现的。命名方式也很直观：lower_bound 就是左边界，upper_bound 则是右边界减一。

在工业场景中，类似学习率搜索、超参数调度、限流阈值探测等问题，只要满足“判断一次代价高但结果单调”的特性，都会采用二分查找来快速逼近答案。“在 99% 成功率的前提下，系统最多能承受多大的 QPS？”这类问题，和“珂珂吃香蕉”问题用的是同一个解法模板。

# 基础版：找到任意一个等于 target 的下标，找不到返回 -1
def std(nums, target):
    l, r = 0, len(nums) - 1
    while l <= r:
        m = l + (r - l) // 2
        if nums[m] == target: return m
        if nums[m] < target:  l = m + 1
        else:                  r = m - 1
    return -1

# 左边界：找到第一个大于等于 target 的位置
def lb(nums, target):
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] < target: l = m + 1
        else:                 r = m
    return l

# 右边界：找到最后一个小于等于 target 的位置（不存在则返回 -1）
def rb(nums, target):
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] <= target: l = m + 1
        else:                  r = m
    return l - 1

# 答案空间：在 [lo, hi] 范围内找到最小的 x 满足 feasible(x) == True
def search_answer(lo, hi, feasible):
    l, r = lo, hi
    while l < r:
        m = l + (r - l) // 2
        if feasible(m): r = m
        else:            l = m + 1
    return l
