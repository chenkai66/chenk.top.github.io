LeetCode（三）：链表操作

反转、合并有序链表、Floyd 找环入口、删除倒数第 N、LRU 缓存五题串起所有套路，重点是判空与指针赋值顺序。

#LeetCode#链表#算法
Read in English →

链表是最能逼你用指针思考的数据结构。数组靠下标访问，想跳哪儿就跳哪儿；链表只给你一个头指针，剩下的全靠自己一步步走。这种从「随机访问」到「顺序追指针」的转变，正是链表题在面试里反复出现的原因——题目简单到一句话就能说清，做对却要求你具备扎实的基本功：画图、给指针起名字、绝不乱解引用。

这篇文章通过五道经典题目，把链表的核心套路一网打尽：

  • 反转链表：入门级指针重连，迭代和递归两种写法都得会。
  • 合并两个有序链表：掌握「虚拟头节点」如何干掉第一个节点的特判。
  • 环形链表 II（Floyd 算法）：快慢指针 + 一段简单的代数推导，证明它们一定会在入口相遇。
  • 删除倒数第 N 个节点：固定间距的双指针，一次遍历搞定。
  • LRU 缓存：哈希表 + 双向链表的经典组合，几乎所有真实世界的缓存系统都在用这个结构。

每节内容都按同一个框架展开：题目、思路、代码、复杂度分析、再走一遍例子。下面那些图就是把指针在每一步的变化画出来——这就是你在白板上解这类题时脑子里应该跑的画面。

链表基础

单链表节点很朴素，一个值加一个指针：

1
2
3
4

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

整个数据结构对外暴露的就这些。本文所有算法本质上都是「按正确顺序、配合正确的判空，做一连串 node.next = ... 赋值」。

链表 vs 数组

特性数组链表
随机访问$O(1)$$O(n)$
已知位置插入/删除$O(n)$（需要搬移）$O(1)$（只改指针）
内存布局连续分散
缓存友好高低
动态扩容需要重新分配天然支持

取舍很明显：数组适合「快速访问 + 高缓存命中」，链表擅长「频繁插删 + 不依赖随机访问」。想清楚这点，选型就简单了——“要按索引查第 k 个？” 数组；“要在中间切开重连还不复制数据？” 链表。

插入与删除：只需调整两处指针

将节点 X 插入到 A 和 B 之间，只需要两条赋值语句，顺序必须严格遵守：

1
2

X.next = A.next   # X 先指向 B
A.next = X        # 然后 A 指向 X

如果调换顺序，A.next 会被覆盖，导致丢失对 B 的引用。删除操作更简单，只需一行代码让 A 跳过 B，剩下的交给垃圾回收器处理：

1

A.next = B.next   # A 直接跳过 B

链表操作的核心思想可以归结为一句话：按正确顺序调整指针赋值。本文后续内容的所有题目，都不过是这一基本思想的不同变体。

LeetCode 206 —— 反转链表

给定单链表的头节点 head，将其反转并返回新的头节点。

示例：将 1 → 2 → 3 → 4 → 5 反转为 5 → 4 → 3 → 2 → 1。

进阶：分别用迭代和递归实现。

这是指针操作的经典入门题。难点在于，每次翻转 next 指针时，都会立刻切断后续链表的访问路径，因此必须提前保存「下一个节点」的位置。

迭代法：三个指针，$O(1)$ 空间

def reverseList(head):
    """
    原地反转单链表。

    时间：O(n)，每个节点访问一次。
    空间：O(1)，只用三个指针。
    """
    prev, curr = None, head
    while curr:
        nxt = curr.next     # 1. 先存好下一个节点
        curr.next = prev    # 2. 反转当前节点指针
        prev = curr         # 3. prev 移到当前节点
        curr = nxt          # 4. curr 移到下一个节点
    return prev             # 返回新的头节点（原链表的尾节点）

以 1 → 2 → 3 → None 为例走一遍：

步骤prevcurr翻转后的状态
初始None1—
第一步12None ← 1，剩下 2 → 3 → None
第二步23None ← 1 ← 2，剩下 3 → None
第三步3NoneNone ← 1 ← 2 ← 3

最终返回 prev = 3，它就是反转后的新头节点。

循环自动处理的边界情况：

  • 空链表（head is None）：循环不执行，直接返回 prev = None。
  • 单节点链表：循环执行一次，将 1.next 设为 None，返回 1。

递归法：回溯时调整指针

递归的思路很简单：“先把 head 后面的部分反转好，再让原来的 head.next 指回来。”

def reverseList_recursive(head):
    """
    递归反转链表。

    时间复杂度：O(n)。
    空间复杂度：O(n)，递归栈的开销。
    """
    if not head or not head.next:
        return head

    new_head = reverseList_recursive(head.next)

    # 此时 head.next 是反转后子链表的尾节点
    # 让它指向 head，完成局部反转
    head.next.next = head
    # 断开 head 原来的出边，避免形成环
    head.next = None
    return new_head

重点在 head.next.next = head 这一行，拆开来看：

  • head.next 是原始链表中紧跟在 head 后面的那个节点。
  • 经过递归反转后，这个节点变成了子链表的尾节点。
  • 让它指向 head，就完成了局部的反转。

最后一步 head.next = None 很关键，清掉 head 的出边，确保新链表的尾部干净利落地结束。

迭代还是递归？

写法时间空间适用场景
迭代$O(n)$$O(1)$生产环境、长链表，不怕栈溢出
递归$O(n)$$O(n)$白板上秀递归思维，代码简洁

面试时优先写迭代，顺便提一句「递归也能搞定」，展示你思路灵活。生产环境老老实实用迭代——Python 默认递归深度只有 1000，链表稍微长点就等着崩吧。

LeetCode 21 —— 合并两个有序链表

将两个有序链表合并成一个有序链表，复用原有节点，不创建新节点。

示例：l1 = [1,2,4]、l2 = [1,3,4] → [1,1,2,3,4,4]。

合并本身是标准的双指针比较操作，没什么特别。关键在于如何处理结果链表的第一个节点。如果不引入辅助节点，第一次循环就得特判「头节点来自哪个链表」，后续逻辑也会跟着分叉。而使用一个**虚拟头节点（dummy）**后，这个特判就完全消失了。

def mergeTwoLists(l1, l2):
    """
    通过拼接节点合并两个有序链表（不分配新节点）。

    时间：O(m + n)，每个输入节点访问一次。
    空间：O(1)，只用 dummy 和 cur 两个指针。
    """
    dummy = ListNode()    # 哨兵节点，不返回
    cur = dummy

    while l1 and l2:
        if l1.val <= l2.val:
            cur.next = l1
            l1 = l1.next
        else:
            cur.next = l2
            l2 = l2.next
        cur = cur.next

    # 至多还有一个链表非空，整段挂上去
    cur.next = l1 if l1 else l2

    return dummy.next

为什么用虚拟头节点？ 它的作用是提供一个固定的起点，避免单独处理头节点的特殊情况。没有它的话，第一次迭代必须判断 l1 和 l2 谁是头节点，后续逻辑也会因此分叉。有了虚拟头节点，每次迭代的逻辑完全一致，头节点的特殊性被巧妙隐藏。

举个例子：l1 = [1,2,4]、l2 = [1,3,4]

32 < 3l11 → 1 → 2
43 < 4l21 → 1 → 2 → 3
54 ≤ 4l11 → 1 → 2 → 3 → 4

最后返回 dummy.next，就是合并后的链表头节点。

def mergeTwoLists_recursive(l1, l2):
    """
    递归合并。代码优雅但占 O(m + n) 栈。

    时间：O(m + n)。
    空间：O(m + n)。
    """
    if not l1: return l2
    if not l2: return l1
    if l1.val <= l2.val:
        l1.next = mergeTwoLists_recursive(l1.next, l2)
        return l1
    else:
        l2.next = mergeTwoLists_recursive(l1, l2.next)
        return l2

LeetCode 141 / 142 —— 环形链表与环入口

141：判断链表是否存在环。
142：如果存在环，返回环的起始节点；否则返回 None。
要求：$O(1)$ 额外空间。

暴力解法——比如用哈希表记录访问过的节点，或者直接计步数——虽然能解决问题，但要么需要 $O(n)$ 的额外空间，要么得遍历两次链表。而 Floyd 龟兔赛跑算法 则巧妙地利用两个指针和一点简单的数学，在一次遍历中完成任务。

第一阶段：判断链表是否有环

def hasCycle(head):
    """LeetCode 141：用快慢指针检测链表是否有环。"""
    slow = fast = head
    while fast and fast.next:
        slow = slow.next         # 慢指针走一步
        fast = fast.next.next    # 快指针走两步
        if slow is fast:
            return True
    return False

为什么这个方法有效？如果链表没有环，fast 会先走到 None，循环自然结束。如果有环，两个指针最终都会进入环中。由于 fast 每次比 slow 多走一步，二者之间的距离每次缩短 1，因此最多经过 $L$ 次移动（$L$ 是环的长度），它们一定会相遇。

def detectCycle(head):
    """
    LeetCode 142：返回链表中环的入口节点。

    时间复杂度：O(n)。空间复杂度：O(1)。
    """
    slow = fast = head
    while fast and fast.next:
        slow = slow.next          # 慢指针走一步
        fast = fast.next.next     # 快指针走两步
        if slow is fast:          # 第一次相遇，进入第二阶段
            slow = head           # 把慢指针重置到头节点
            while slow is not fast:  # 两个指针同步前进
                slow = slow.next
                fast = fast.next
            return slow           # 返回环的入口
    return None                   # 没有环，返回 None

背下来的数学推导

假设 $a$ 是从头节点到环入口的距离，$b$ 是从入口到相遇点的距离（沿着环的方向），$c$ 是环剩下的部分长度（因此环的总长度为 $L = b + c$）。

当 slow 和 fast 第一次相遇时：

  • slow 走过的距离是：$a + b$
  • fast 走过的距离是：$a + b + kL$（其中 $k \geq 1$ 是某个整数）

因为 fast 的速度是 slow 的两倍，所以 $\text{fast} = 2 \cdot \text{slow}$，代入公式：

$$ 2(a + b) = a + b + kL \implies a = kL - b = (k-1)L + c $$

这个结果很关键：它说明从头节点出发走 $a$ 步会到达环入口，而从相遇点出发也走 $a$ 步同样会到达环入口（$(k-1)L$ 只是绕了几圈环）。因此，把 slow 拉回头节点，然后让两个指针以相同速度前进，它们必然会在环入口处相遇。

LeetCode 19 —— 删除倒数第 N 个节点

给定链表的头节点，删除倒数第 n 个节点并返回修改后的链表头。要求只扫描一次。

示例：head = [1,2,3,4,5]、n = 2 → [1,2,3,5]。

最直接的做法是分两步走：先遍历一遍算出链表长度，再定位到 length - n 的位置进行删除。但要实现一趟扫描，核心技巧是用「间距为 $n + 1$ 的双指针」，再加上一个 dummy 节点来处理删头的情况。

def removeNthFromEnd(head, n):
    """
    一趟扫描删除倒数第 n 个节点。

    时间复杂度：O(L)，L 是链表长度。
    空间复杂度：O(1)。
    """
    dummy = ListNode(0, head)    # 引入 dummy 统一处理头节点删除
    slow = fast = dummy

    # 先让 fast 走 n+1 步，这样当 fast 到达末尾时
    # slow 恰好停在目标节点的前一个位置
    for _ in range(n + 1):
        fast = fast.next

    while fast:
        fast = fast.next
        slow = slow.next

    slow.next = slow.next.next  # 删除目标节点
    return dummy.next

为什么要用 $n+1$ 而不是 $n$？
因为删除操作需要找到目标节点的前驱节点，才能修改它的 next 指针。如果 fast 只提前走 $n$ 步，那么两者之间的间距是 $n$，当 fast 到达链表末尾时，slow 就会停在目标节点上，而不是它的前驱。这样一来，就无法正确完成删除操作。差这一步是这道题最常见的错误。

dummy 节点的作用是什么？
考虑这种情况：head = [1, 2]，n = 2（即删除头节点）。如果有 dummy：

  • fast 走 3 步后到达 None。
  • slow 还停留在 dummy。
  • 执行 dummy.next = dummy.next.next，直接将头节点跳过。
  • 返回 dummy.next 即可得到 [2]。

如果没有 dummy，就必须额外判断是否删除的是头节点，比如写 if n == length: return head.next。但要知道 length 又得先遍历一遍链表，这就违背了「一趟扫描」的要求。

总结一下：n+1 的间距确保 slow 停在正确的位置，而 dummy 节点则简化了头节点删除的逻辑，避免了繁琐的特判。

LeetCode 146 —— LRU 缓存

设计一个数据结构，get(key) 和 put(key, value) 都要 $O(1)$ 摊还时间。容量满时淘汰最久未使用的键。

这道题在工业界几乎无处不在——操作系统页缓存、CDN 边缘缓存、浏览器缓存、ORM 查询缓存，背后都是这个套路。标准解法是一个看似简单但极其优雅的设计：用哈希表实现 $O(1)$ 查找，用双向链表维护「最近使用」的顺序，两者配合得天衣无缝。

为什么需要两个数据结构？

  • 单用哈希表：查找是 $O(1)$，但没法记录「最近使用」的顺序。
  • 单用链表：能维护顺序，但查找变成 $O(n)$。
  • 组合使用：哈希表快速定位节点（$O(1)$），再通过双向链表调整顺序（摘节点、插到头部，都是 $O(1)$）。

链表的核心不变量是：头部存放最近使用的节点，尾部存放最久未使用的节点。引入两个哨兵节点（head 和 tail）后，无论是插入头部还是删除尾部，都不需要额外处理边界情况。

class Node:
    __slots__ = ("key", "value", "prev", "next")
    def __init__(self, key=0, value=0):
        self.key, self.value = key, value
        self.prev = self.next = None

class LRUCache:
    """
    哈希表 + 双向链表 + 哨兵，实现 O(1) 的 get 和 put 操作。

    不变量：
        head <-> 最近使用 <-> ... <-> 最久未用 <-> tail
    """

    def __init__(self, capacity: int):
        self.cap = capacity
        self.map = {}                       # key -> Node
        self.head, self.tail = Node(), Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    # ----- 链表操作原语 -----

    def _remove(self, node: "Node") -> None:
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node: "Node") -> None:
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    # ----- 对外接口 -----

    def get(self, key: int) -> int:
        node = self.map.get(key)
        if not node:
            return -1
        self._remove(node)
        self._add_to_front(node)            # 将节点标记为最近使用
        return node.value

    def put(self, key: int, value: int) -> None:
        node = self.map.get(key)
        if node:
            node.value = value
            self._remove(node)
            self._add_to_front(node)
            return

        if len(self.map) >= self.cap:
            lru = self.tail.prev            # 找到最久未使用的节点
            self._remove(lru)
            del self.map[lru.key]

        new_node = Node(key, value)
        self.map[key] = new_node
        self._add_to_front(new_node)

有三个关键点值得特别注意：

  1. 哨兵节点必不可少。 它让所有插入和删除操作都变成固定的四指针调整，代码里完全不需要写 if head is None 这种判断。
  2. 节点必须存储 key。 淘汰最久未使用的节点时，需要从哈希表中删除对应的键值对。如果节点不存 key，就得遍历整个哈希表来找，复杂度直接升到 $O(n)$。
  3. get 操作复用现有节点，不重新分配内存。 这是保证 $O(1)$ 摊还复杂度的关键。如果每次 get 都重建节点（比如用 list.pop 或重新分配对象），性能会差一个数量级，生产环境绝对不能接受。

快慢指针寻找中点、检测环、定位倒数第 k 个节点环形链表 II、删除倒数第 N 个节点
哈希表 + 链表需要同时实现 $O(1)$ 查找和顺序维护LRU 缓存
递归操作具有天然的分治结构递归反转链表、递归合并链表

常见坑

  1. 解引用前不判空。 curr.next.val 一旦 curr.next 是 None 就直接崩了。记住永远写成 while curr and curr.next:。
  2. 丢了前进路径。 改 curr.next = something 之前，如果还需要原来的 curr.next，先存到临时变量里（比如迭代反转时的 nxt，就是为了这个）。
  3. 忘记推进指针。 每次循环至少要让一个指针往终止条件靠近，不然就死循环了。
  4. 间距差一。 双指针固定间距问题，核心就是看 fast 先走了几步——是 n 还是 n + 1。
  5. 头节点边界硬刚。 凡是头节点可能被修改的情况，dummy 节点都能把一堆边界判断压缩成零行代码。

链表在生产环境中的实际应用

日常开发中，我们很少直接操作 ListNode —— 标准库已经把它封装好了。但链表的底层思想无处不在：

  • LRU 缓存
    Python 的 functools.lru_cache、Redis 的 allkeys-lru 淘汰策略、浏览器的 HTTP 缓存，背后都是 hash map 加双向链表，和 LeetCode 146 题如出一辙。面试官真正想考的是为什么非得这么设计：O(1) 时间把节点挪到队首，必须有指针直接指向节点，而 Python 的 list 做不到这一点。

  • Linux 内核链表
    内核代码里，include/linux/list.h 定义了一种侵入式双向链表，被成千上万的内核数据结构使用，比如进程链表、文件描述符表、网络队列等。实现侵入式链表的核心技巧是 container_of 宏，这是 Linux 内核中最精妙的 C 语言片段之一。

  • 有序字典
    Java 的 LinkedHashMap 和 Python 的 OrderedDict 都支持按插入顺序迭代，靠的就是 hash map 加链表的组合。换了个语言，但底层原理不变。

  • 跳表
    Redis 的 sorted set 使用了跳表（skip list），一种多层链表加概率性快捷路径的结构。相比平衡树，跳表更容易实现无锁操作，因此在高并发场景下更受欢迎。

总结一句话：链表题不是单纯的“算法练习”，而是生产环境中无处不在的模式的最小化体现。

# 从可迭代对象构造链表（快速生成测试数据）
def build(values):
    dummy = ListNode()
    tail = dummy
    for v in values:
        tail.next = ListNode(v)
        tail = tail.next
    return dummy.next

这三个小工具函数能大幅简化链表题的调试和测试。每次写链表题，我都习惯把它们放在文件开头。

反转链表的递归写法不仅比迭代版本更短，还能帮你更好地理解链表结构。只要栈深度允许，推荐优先使用：

def reverse(head):
    if not head or not head.next:
        return head
    new_head = reverse(head.next)
    head.next.next = head
    head.next = None
    return new_head

核心就在 head.next.next = head 这一行——它的意思是"让下一个节点反过来指向我"。这一行代码几乎就是整个算法的全部了。

一个常见坑：遍历的同时修改链表

写链表的双指针代码时，表面上跟数组操作差不多，但出错的后果可不太一样。最常见的错误长这样：

1
2
3
4
5

# 错误示范 —— 改 curr.next 前忘记保存原来的 next 指针
curr = head
while curr:
    curr.next = something_new   # 这一步直接丢掉了链表后半部分
    curr = curr.next            # 下一步顺着新指针走了，原链表断了

怎么避免？记住一条铁律：动 .next 之前，先把原来的 next 存下来。

正确的写法是这样：

1
2
3
4
5

curr = head
while curr:
    nxt = curr.next             # 第一步：存下原来的 next
    curr.next = something_new   # 第二步：放心改
    curr = nxt                  # 第三步：用存下来的指针推进

所有迭代式的链表算法基本都遵循这个模式。如果你的代码跑着跑着跳过了某些节点、陷入死循环、或者“链表莫名其妙少了一半”，十有八九就是没遵守这条规则——赋值操作抢在了保存指针的前头。
