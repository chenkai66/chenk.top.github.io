链表是一种让人不得不用指针思考的数据结构。数组允许你通过下标在 $O(1)$ 时间内直接访问任意位置，而链表只给你一个头指针，剩下的全靠一步步「追」。这种从「随机访问」到「顺序遍历」的转变，正是链表问题频繁出现在面试中的原因——题目描述往往简短到一句话就能说完，但要写对代码却需要扎实的基本功：画图理清思路、给指针起个好名字、绝不解引用空指针。

本文通过五道经典题目，带你掌握链表相关的所有核心技巧：

  • 反转链表：指针操作的入门练习，迭代和递归两种实现方式都要熟练。
  • 合并两个有序链表：学会「虚拟头节点」如何优雅地解决头部特判问题。
  • 环形链表 II（Floyd 算法）：快慢指针的经典应用，配合简单的数学推导证明它们会在环入口相遇。
  • 删除倒数第 N 个节点：双指针保持固定间距，一次遍历完成任务。
  • LRU 缓存：哈希表 + 双向链表的黄金搭档，几乎所有实际缓存系统都基于这个设计。

每一节的内容都遵循相同的结构：题目背景、解题思路、代码实现、复杂度分析，以及一个完整的例子。下面的图示展示了每一步指针的变化过程——这正是你在白板上解题时应该在脑海中模拟的画面。

链表基础

单链表的节点非常简单，包含一个值和一个指向下一个节点的指针：

1
2
3
4

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

这就是链表的全部接口。本文涉及的所有算法，本质上都是按照正确的顺序和安全检查，执行一系列 node.next = ... 的赋值操作。

链表与数组的对比

特性数组链表
随机访问$O(1)$$O(n)$
在已知位置插入/删除$O(n)$（需要移动元素）$O(1)$（只需修改指针）
内存分布连续存储分散存储
缓存局部性高低
动态扩展需要重新分配内存天然支持

取舍非常明确：如果需要快速随机访问或者对缓存友好，数组是更好的选择；如果需要频繁在中间插入或删除节点，而不需要随机访问，链表则更胜一筹。一旦你理解了这一点，选择数据结构就变得直觉化了——“我是否需要通过下标直接访问第 k 个元素？” 用数组；“我是否需要在中间插入或删除而不复制数据？” 用链表。

插入与删除：只需调整两根指针

要将 X 插入到 A 和 B 之间，只需要两步赋值操作，顺序非常重要：

1
2

X.next = A.next   # X 先指向 B
A.next = X        # 然后让 A 指向 X

如果顺序反过来，A.next 的值会在你保存它之前被覆盖，这样一来，B 就会丢失。删除操作更加简单，只需一行代码让 A 跳过 B，剩下的交给垃圾回收机制处理即可：

1

A.next = B.next   # A 直接跳过 B

这就是链表操作的核心思想：链表算法的本质是按照特定顺序进行指针赋值。本文后续的所有内容，都是围绕这一核心思想展开的不同变体。

LeetCode 206：反转链表

给定一个单链表的头节点 head，将其反转后返回新的头节点。

示例：将 1 → 2 → 3 → 4 → 5 反转为 5 → 4 → 3 → 2 → 1。

进阶：分别用迭代和递归两种方法实现。

这可以说是链表指针操作的「入门第一课」。难点在于，每次翻转 next 指针时，都会切断当前节点与后续节点的连接，因此需要提前保存「下一个该去哪」的信息。

迭代法：三指针实现，$O(1)$ 空间

在反转链表时，一个关键点是要意识到，一旦执行 curr.next = prev，当前节点的下一个节点信息就会丢失。因此，我们需要提前保存它。

def reverseList(head):
    """
    原地反转单链表。

    时间复杂度：O(n) —— 每个节点只遍历一次。
    空间复杂度：O(1) —— 只使用了三个指针，没有额外的数据结构。
    """
    prev, curr = None, head
    while curr:
        nxt = curr.next     # 1. 先保存下一个节点
        curr.next = prev    # 2. 反转当前节点的指针
        prev = curr         # 3. 移动 prev 到当前节点
        curr = nxt          # 4. 移动 curr 到下一个节点
    return prev             # 最终 prev 指向新链表的头节点（原链表的尾节点）

以链表 1 → 2 → 3 → None 为例逐步解析：

步骤prevcurr反转后的状态
初始None1—
第一步12None ← 1，剩余部分为 2 → 3 → None
第二步23None ← 1 ← 2，剩余部分为 3 → None
第三步3NoneNone ← 1 ← 2 ← 3

最终返回 prev = 3，即反转后的新链表头节点。

循环逻辑已经正确处理的边界情况：

  • 空链表（head is None）：循环不会执行，直接返回 prev = None。
  • 单节点链表：循环执行一次，将 1.next 设置为 None，然后返回该节点 1。

递归法：在回溯时重新连接

从递归的角度来看，思路是这样的：“先把 head 后面的部分反转好，然后让原来的 head.next 指向 head。”

def reverseList_recursive(head):
    """
    递归反转链表。

    时间复杂度：O(n)。
    空间复杂度：O(n)，递归调用栈的深度。
    """
    if not head or not head.next:
        return head

    new_head = reverseList_recursive(head.next)

    # 此时，head.next 是已经反转好的子链表的尾节点
    # 让它指向 head，完成反转，并断开 head 的出边
    head.next.next = head
    head.next = None
    return new_head

最让人困惑的一行代码是 head.next.next = head，我们慢慢拆解一下：

  • head.next 是原始链表中紧跟在 head 后面的那个节点。
  • 在递归调用结束后，这个节点已经成为反转后子链表的尾节点。
  • 我们希望这个尾节点指回 head，所以通过 head.next.next = head 来实现。

接着，执行 head.next = None，切断 head 原本的出边，确保新链表的尾部干净利落地终止。

迭代还是递归？

方法时间复杂度空间复杂度适用场景
迭代$O(n)$$O(1)$生产环境代码、处理超长链表，避免栈溢出问题
递归$O(n)$$O(n)$白板上展示优雅解法，体现递归思维

在面试中，优先选择迭代实现，可以顺便提一句「递归也能完成」，以展现你的多维度思考能力。而在生产环境中，务必选择迭代——Python 默认的递归深度限制为 1000，一旦链表过长，程序就会崩溃。

LeetCode 21：合并两个有序链表

将两个有序链表合并成一个有序链表，直接复用原链表节点，无需创建新节点。

示例：l1 = [1,2,4]、l2 = [1,3,4] → [1,1,2,3,4,4]。

这道题的核心逻辑并不复杂，无非是经典的双指针比较大小。真正值得讨论的是如何处理结果链表的第一个节点。如果不引入辅助节点，就需要额外判断头节点应该从哪个链表中选取，后续逻辑也会因此变得繁琐。而通过引入一个虚拟头节点（dummy node），这个问题就迎刃而解，代码逻辑也更加清晰流畅。

def mergeTwoLists(l1, l2):
    """
    合并两个有序链表，通过直接拼接节点实现（无需额外分配内存）。

    时间复杂度：O(m + n)，每个节点只遍历一次。
    空间复杂度：O(1)，仅使用了 dummy 和 cur 两个指针。
    """
    dummy = ListNode()  # 虚拟头节点，作为辅助，最终不会返回
    cur = dummy

    while l1 and l2:
        if l1.val <= l2.val:
            cur.next = l1
            l1 = l1.next
        else:
            cur.next = l2
            l2 = l2.next
        cur = cur.next

    # 至多只有一个链表还有剩余节点，直接将其连接到结果链表末尾
    cur.next = l1 if l1 else l2

    return dummy.next

为什么要用虚拟头节点？
虚拟头节点的作用是简化逻辑。如果没有它，第一次迭代时需要单独判断头节点应该取自 l1 还是 l2，这会让后续代码充满分支判断，增加复杂性。而引入虚拟头节点后，每次迭代的逻辑完全一致——头节点的特殊处理被巧妙地隐藏在虚拟节点中。

以 l1 = [1,2,4]、l2 = [1,3,4] 为例逐步解析：

步骤比较条件选择节点当前合并结果
11 ≤ 1l11
21 < 2l21 → 1
32 < 3l11 → 1 → 2
43 < 4l21 → 1 → 2 → 3
54 ≤ 4l11 → 1 → 2 → 3 → 4
6l1 已空接上 l2 剩余部分1 → 1 → 2 → 3 → 4 → 4

最终返回 dummy.next，即合并后的链表头节点。

def mergeTwoLists_recursive(l1, l2):
    """
    使用递归合并两个链表。代码简洁优美，但会占用 O(m + n) 的栈空间。

    时间复杂度：O(m + n)。
    空间复杂度：O(m + n)，递归深度等于链表总长度。
    """
    if not l1: return l2
    if not l2: return l1
    if l1.val <= l2.val:
        l1.next = mergeTwoLists_recursive(l1.next, l2)
        return l1
    else:
        l2.next = mergeTwoLists_recursive(l1, l2.next)
        return l2

LeetCode 141 / 142：链表环检测与环起点定位

141：判断一个链表是否存在环。

142：如果链表有环，返回环的起始节点；如果没有环，则返回 None。

要求：只能使用 $O(1)$ 的额外空间。

最直接的方法是用哈希表记录访问过的节点，或者通过计数器逐步排查，但这些方法要么需要 $O(n)$ 的额外空间，要么需要遍历链表两次。Floyd 龟兔赛跑算法则巧妙地利用双指针和简单的数学推导，在一次遍历中高效解决问题。

第一阶段：检测环是否存在

def hasCycle(head):
    """LeetCode 141：使用快慢指针判断链表是否有环。"""
    slow = fast = head
    while fast and fast.next:
        slow = slow.next         # 每次移动 1 步
        fast = fast.next.next    # 每次移动 2 步
        if slow is fast:
            return True
    return False

这个方法为什么有效？如果没有环，fast 指针最终会走到 None，循环自然结束。如果存在环，快慢指针迟早都会进入环中。由于 fast 每次比 slow 多走一步，二者之间的距离每轮循环都会缩短 1，因此它们必然会在最多 $L$ 次迭代内相遇（其中 $L$ 是环的长度）。

def detectCycle(head):
    """
    LeetCode 142：返回链表中环的起始节点。

    时间复杂度：O(n)。空间复杂度：O(1)。
    """
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            # 第二阶段：将 slow 重置为头节点，然后两个指针同步移动
            slow = head
            while slow is not fast:
                slow = slow.next
                fast = fast.next
            return slow
    return None

数学推导（值得牢记）

设 $a$ 为从头节点到环入口的距离，$b$ 为从环入口到相遇点的距离，$c$ 为环的剩余部分（因此环的总长度为 $L = b + c$）。

当 slow 和 fast 首次相遇时：

$$ \text{slow 走过的距离：} a + b $$$$ \text{fast 走过的距离：} a + b + kL \quad (\text{其中 } k \geq 1 \text{ 为整数}) $$

由于 fast 的速度是 slow 的两倍，$\text{fast} = 2 \cdot \text{slow}$，因此有：

$$ 2(a + b) = a + b + kL \implies a = kL - b = (k-1)L + c $$

这个结果非常关键。它表明：从头节点走 $a$ 步会到达环入口；从相遇点也走 $a$ 步同样会到达环入口（因为 $(k-1)L$ 只是绕了几圈）。因此，我们将 slow 重置为头节点，然后让两个指针以相同速度移动，它们必然在环入口处相遇。

LeetCode 19：删除倒数第 N 个节点

给定链表的头节点，删除倒数第 n 个节点并返回修改后的链表头。要求一次遍历完成。

示例：head = [1,2,3,4,5]、n = 2 → [1,2,3,5]。

最简单的做法需要两次遍历：第一次统计链表长度，第二次走到 length - n 的位置进行删除。而单次遍历的技巧在于使用「固定间距 $n+1$ 的双指针」，同时引入一个 dummy 节点来简化对头节点删除的处理。

def removeNthFromEnd(head, n):
    """
    单次遍历删除倒数第 n 个节点。

    时间复杂度：O(L)，其中 L 是链表长度。
    空间复杂度：O(1)。
    """
    dummy = ListNode(0, head)  # 使用 dummy 节点统一处理头节点删除
    slow = fast = dummy

    # 让 fast 先走 n+1 步，这样当 fast 到达链表末尾时
    # slow 恰好停在目标节点的前驱节点
    for _ in range(n + 1):
        fast = fast.next

    while fast:
        fast = fast.next
        slow = slow.next

    slow.next = slow.next.next
    return dummy.next

为什么是 $n+1$ 而不是 $n$？ 因为我们需要修改的是目标节点的前驱节点的 next 指针，所以 slow 必须停在目标节点的前一个位置。如果 fast 只先走 $n$ 步，那么两者的间距是 $n$，当 fast 到达链表末尾时，slow 会直接停在目标节点上，无法完成删除操作。这种「差一步」的错误是这道题最常见的陷阱。

为什么要用 dummy 节点？ 考虑一个特殊情况：head = [1, 2]、n = 2（即删除头节点）。如果有 dummy 节点：

  • fast 先走 3 步，最终指向 None。
  • slow 停在 dummy。
  • 修改 dummy.next = dummy.next.next，即将头节点删除。
  • 返回 dummy.next，得到 [2]，问题解决。

如果没有 dummy 节点，你必须额外判断是否要删除头节点，例如写 if n == length: return head.next，但这需要先遍历一遍链表计算长度——这就违背了单次遍历的要求。

LeetCode 146：LRU 缓存

设计一个支持 get(key) 和 put(key, value) 操作的数据结构，要求两种操作的时间复杂度均为 $O(1)$。当缓存容量达到上限时，淘汰最近最少使用的键。

这个问题在工业界应用极为广泛，比如操作系统中的页缓存、CDN 边缘缓存、浏览器缓存以及 ORM 查询缓存等场景。标准解法是一个简单但优雅的设计：结合哈希表和双向链表，前者用于 $O(1)$ 查找，后者用于维护键值的「最近使用」顺序。

具体来说，哈希表存储键到链表节点的映射，双向链表则按访问顺序排列节点。每次访问或插入新键时，将其对应的节点移到链表头部（表示最近使用）。当缓存满时，直接移除链表尾部的节点（表示最久未使用）。

为什么需要两种数据结构？

  • 单用哈希表：查找是 $O(1)$，但完全无法表达「最近使用」的概念。
  • 单用链表：可以维护顺序，但查找复杂度退化到 $O(n)$。
  • 结合两者：先用哈希表定位节点（$O(1)$），再从链表中摘下节点并重新挂到头部（双向链表操作，也是 $O(1)$）。

链表的核心不变量是：头部存放最近使用的节点，尾部存放最久未使用的节点。两个哨兵节点（head 和 tail）彻底消除了头插和尾删时的所有边界条件判断。

class Node:
    __slots__ = ("key", "value", "prev", "next")
    def __init__(self, key=0, value=0):
        self.key, self.value = key, value
        self.prev = self.next = None

class LRUCache:
    """
    哈希表 + 双向链表 + 哨兵节点，实现 O(1) 的 get 和 put 操作。

    不变量：
        head <-> 最近使用 <-> ... <-> 最久未用 <-> tail
    """

    def __init__(self, capacity: int):
        self.cap = capacity
        self.map = {}                       # key -> Node
        self.head, self.tail = Node(), Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    # ----- 链表基础操作 -----

    def _remove(self, node: "Node") -> None:
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node: "Node") -> None:
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    # ----- 公共接口 -----

    def get(self, key: int) -> int:
        node = self.map.get(key)
        if node is None:
            return -1
        self._remove(node)
        self._add_to_front(node)            # 标记为最近使用
        return node.value

    def put(self, key: int, value: int) -> None:
        node = self.map.get(key)
        if node is not None:
            node.value = value
            self._remove(node)
            self._add_to_front(node)
            return

        if len(self.map) >= self.cap:
            lru = self.tail.prev            # 最久未使用的节点
            self._remove(lru)
            del self.map[lru.key]

        new_node = Node(key, value)
        self.map[key] = new_node
        self._add_to_front(new_node)

三个值得注意的细节：

  1. 哨兵节点不可或缺。 它的存在让所有插入和删除操作都变成统一的四指针调整，代码里再也不会出现 if head is None 这种分支。
  2. 节点必须存储 key。 当淘汰最久未使用的节点时，需要根据节点的 key 从哈希表中删除对应的映射。如果节点不存 key，就只能遍历整个哈希表去找，复杂度直接崩掉。
  3. get 操作复用现有节点，而不是重新分配。 这才是真正的 $O(1)$ 摊还成本。生产环境如果用 dict + list.pop 的方式重建节点，每次 get 都会变成 $O(n)$，看着对，跑起来慢一个数量级。

技巧使用场景本文对应题目
三指针（prev / curr / next）原地反转或重连反转链表
虚拟头节点 / 哨兵头节点可能被修改或返回新头合并、删除倒数第 N 个、LRU
快慢指针找中点、检测环、找倒数第 k 个环形链表 II、删除倒数第 N 个
哈希表 + 链表既要 $O(1)$ 查找又要维护顺序LRU 缓存

常见坑点

  1. 解引用前没判空。 curr.next.val 在 curr.next is None 的瞬间就会抛异常。永远写成 while curr and curr.next:。
  2. 丢失前进路径。 写 curr.next = something 之前，如果还需要原来的 curr.next，一定要先存到临时变量里（迭代反转里的 nxt 就是为了这个）。
  3. 忘了推进指针。 每次循环至少要有一个指针向终止条件移动，否则就是死循环。
  4. 间距差一格。 双指针固定间距的题，全看你 fast 先走的是 n 还是 n + 1。
  5. 不用 dummy 就硬写。 凡是头节点可能改变的操作，dummy 都能把 5 行的边界判断压缩到 0 行。

链表在生产中的实际应用

应用层代码很少直接碰 ListNode —— 标准库把它藏起来了。但底层到处都是它：

  • LRU 缓存。 functools.lru_cache、Redis 的 allkeys-lru 淘汰、每个浏览器的 HTTP 缓存，都是 hash map + 双向链表，和 LeetCode 146 一模一样。面试真正在考的是为什么必须这么实现 —— O(1) 把节点挪到队首，必须要有节点指针，Python list 给不了你。
  • Linux 内核链表。 include/linux/list.h 定义了一个侵入式双向链表，被几万个内核结构使用——进程链、文件描述符表、网络队列。让侵入式链表能工作的 container_of 宏，是内核里最优雅的几段 C 之一。
  • Java 的 LinkedHashMap 与 Python 的 OrderedDict。 按插入序迭代靠的就是 hash map + 链表。同一个结构，不同的语言。
  • Redis sorted set 里的跳表。 多层链表加概率性快捷路径。选它而不是平衡树，是因为跳表更容易做无锁。

一句话：链表题不是"练习题"。它是你日后在每一层栈上都会遇到的模式的最小例子。

# 从可迭代对象构造链表（写测试用例的救命稻草）
def build(values):
    dummy = ListNode()
    tail = dummy
    for v in values:
        tail.next = ListNode(v)
        tail = tail.next
    return dummy.next

def reverse(head):
    if not head or not head.next:
        return head
    new_head = reverse(head.next)
    head.next.next = head
    head.next = None
    return new_head

head.next.next = head 这一行就是整个算法 —— 读作"我后面那个节点现在应该指回我"。

一个高频 bug：边遍历边改

链表的双指针写法看着像数组，但写错的代价不一样。最常见的 bug：

1
2
3
4
5

# 错 —— 改 curr.next 之前没保存原本的 next 指针
curr = head
while curr:
    curr.next = something_new   # 这一刻链表后半段已经丢了
    curr = curr.next            # 接下来跳到的是新指针

修法是教条的：任何写 .next 的赋值之前，先把原 next 存到临时变量里：

1
2
3
4
5

curr = head
while curr:
    nxt = curr.next             # 先存
    curr.next = something_new
    curr = nxt                  # 再用存下来的指针推进

所有迭代式链表算法都遵循这个套路。如果代码跳节点、死循环、或"链表少了一半"，十有八九是某次赋值跑在了"先存"前面。
