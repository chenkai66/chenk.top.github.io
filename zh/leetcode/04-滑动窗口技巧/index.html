如果你曾经为了检查每个连续子数组而写过双重 for 循环，那么滑动窗口很可能就是你苦苦寻找的优化方法。它通过“复用已经计算过的结果”，将原本复杂度为 $O(nk)$ 或 $O(n^2)$ 的暴力扫描压缩到一次线性遍历。本文会从基础原理入手，逐步拆解这一技巧的核心思想，然后通过四道经典的 LeetCode 题目带你彻底掌握它的应用套路，最后还会延伸到单调队列这一进阶变种，助你更上一层楼。

1. 一图看懂滑动窗口

滑动窗口其实就是数组或字符串上的一段连续区间 [left, right]。当窗口移动时，我们并不需要重新计算整个区间的值，而是把右边新加入的元素加进来，同时把左边移出的元素减掉。每个元素最多只会被处理两次，因此整体复杂度是 $O(n)$。

根据窗口大小是否固定，滑动窗口可以分为两种：

  • 固定窗口——窗口宽度固定为 k，左右指针同步向前移动。
  • 可变窗口——右指针 right 主动扩展窗口，而左指针 left 在窗口不满足条件时被动收缩。

下面这张图展示了一个固定窗口求解最大子数组和的过程：绿色部分是当前窗口，橙色表示即将进入窗口的元素，粉色表示即将离开窗口的元素。右侧的计算过程表明，每次只需要做一次加法和一次减法，完全避免了重新调用 sum() 的开销。这就是滑动窗口的核心思想。

2. 固定窗口：长度为 k 的最大子数组和

问题描述：给定一个数组 arr 和一个整数 k，请找出所有长度为 k 的连续子数组中，元素和最大的那个。

直观解法：逐一计算每个长度为 k 的子数组的和，时间复杂度为 $O(nk)$。这种方法虽然简单，但效率较低。

滑动窗口优化：先计算第一个长度为 k 的窗口的和，然后每次向右滑动一格时，只需减去窗口左侧移出的元素值，并加上右侧新进入窗口的元素值即可。

def max_sum_subarray(arr, k):
    """返回长度为 k 的连续子数组的最大和。"""
    if len(arr) < k:
        return None

    window_sum = sum(arr[:k])     # 初始化第一个窗口的和
    best = window_sum

    for right in range(k, len(arr)):
        # 新加入窗口的元素是 arr[right]，离开窗口的是 arr[right - k]
        window_sum += arr[right] - arr[right - k]
        best = max(best, window_sum)

    return best

public int maxSumSubarray(int[] arr, int k) {
    if (arr.length < k) return Integer.MIN_VALUE;
    int windowSum = 0;
    for (int i = 0; i < k; i++) windowSum += arr[i]; // 计算初始窗口的和
    int best = windowSum;
    for (int right = k; right < arr.length; right++) {
        windowSum += arr[right] - arr[right - k]; // 滑动窗口更新和
        best = Math.max(best, windowSum);
    }
    return best;
}

复杂度分析：时间复杂度为 $O(n)$，因为每次滑动窗口只需要进行一次加法和一次减法操作；空间复杂度为 $O(1)$，因为我们只使用了常数级别的额外空间。

为什么滑动窗口高效？ 窗口的和本质上是由窗口内元素集合决定的。当窗口向右移动一位时，集合的变化仅涉及一个元素的移除和另一个元素的加入。因此，我们只需要对这两个变化进行增量更新即可。这种增量更新的思想不仅适用于求和问题，还可以推广到频率统计、计数器维护，甚至是单调队列（详见第 6 节）等场景。

3. 可变窗口的两种模式

当窗口大小不固定时，我们通常会用两个指针来动态调整窗口，并根据问题需求制定策略：

  • 寻找最长合法窗口：让 right 不断向右扩展；一旦窗口变得不合法，就逐步移动 left，直到窗口重新满足条件为止，同时记录下当前的最大长度。
  • 寻找最短合法窗口：先让 right 向右扩展，直到窗口符合条件；然后尽量收缩 left，看看能否找到更短的合法窗口，每次成功收缩时更新答案。

下图展示了“无重复字符的最长子串”算法在字符串 "abcabcbb" 上的运行过程。绿色表示窗口合法（正在扩展），粉色表示检测到重复字符，紫色表示通过收缩修复窗口。

另一张图则描绘了窗口长度随时间变化的趋势。绿色部分表示窗口在扩展阶段，紫色部分表示窗口在收缩阶段。

这里有一个关键的不变量：left 指针始终只向右移动，不会回头。因此，尽管外层是一个 for 循环，内层嵌套了一个 while 循环，但内层循环在整个运行过程中总共执行的步数不会超过 $n$。这就是滑动窗口能够达到摊还 $O(n)$ 时间复杂度的核心原因。

4. LeetCode 3：无重复字符的最长子串

题目：给定一个字符串 s，返回其不含重复字符的最长子串的长度。

解题思路：使用滑动窗口技巧，目标是找到最长的无重复字符子串。核心约束是“窗口内的所有字符必须唯一”。当发现 s[right] 已经存在于当前窗口时，将左指针 left 直接移动到该字符上一次出现位置的后一位。

def length_of_longest_substring(s):
    """LeetCode 3：求无重复字符的最长子串。"""
    last_seen = {}        # 记录每个字符最近一次出现的位置
    left = 0              # 窗口左边界
    best = 0              # 最长子串长度

    for right, ch in enumerate(s):
        if ch in last_seen and last_seen[ch] >= left:
            # 如果字符已在窗口中，直接跳过上次出现的位置
            left = last_seen[ch] + 1
        last_seen[ch] = right
        best = max(best, right - left + 1)

    return best

public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int left = 0, best = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (lastSeen.containsKey(c) && lastSeen.get(c) >= left) {
            left = lastSeen.get(c) + 1;
        }
        last_seen.put(c, right);
        best = Math.max(best, right - left + 1);
    }
    return best;
}

复杂度分析：时间复杂度为 $O(n)$，因为每个字符最多被插入和移出哈希表一次；空间复杂度为 $O(\min(n, |\Sigma|))$，其中 $|\Sigma|$ 表示字符集的大小。

两种实现方式：上面的代码采用了“跳跃式收缩”——当发现重复字符时，直接将左指针 left 跳到合适位置。另一种常见写法是“步进式收缩”，即维护一个频次表，并通过循环逐步移动左指针，直到窗口内不再有重复字符：

5. LeetCode 76：最小覆盖子串

题目：给定两个字符串 s 和 t，要求从 s 中找到一个最短的子串，使得这个子串包含 t 中的所有字符（包括重复出现的字符）。如果找不到这样的子串，则返回空字符串 ""。

解题思路：滑动窗口 + 可变长度。核心思想是维护一个"满足条件的窗口"，即窗口内的字符能够完全覆盖 t 的所有字符及其出现次数。先通过右指针扩展窗口直到满足条件，再尝试通过左指针收缩窗口以找到最短的合法子串。

为了高效判断窗口是否满足条件，可以引入一个 valid 计数器，表示当前窗口中已经满足要求的目标字符种类数。当 valid == len(need) 时，说明窗口已经合法。这样避免了每次都要比较两个哈希表的开销。

from collections import Counter

def min_window(s, t):
    """LeetCode 76：找到覆盖 t 所有字符的最短子串。"""
    if not s or not t or len(s) < len(t):
        return ""

    need = Counter(t)  # 目标字符及其所需次数
    have = {}          # 当前窗口中目标字符的实际次数
    valid = 0          # 已满足条件的目标字符种类数
    left = 0
    best_len = float("inf")  # 最短子串长度
    best_start = 0     # 最短子串起始位置

    for right, ch in enumerate(s):
        if ch in need:
            have[ch] = have.get(ch, 0) + 1
            if have[ch] == need[ch]:
                valid += 1

        # 窗口合法时尝试收缩
        while valid == len(need):
            # 更新最优解
            if right - left + 1 < best_len:
                best_len = right - left + 1
                best_start = left

            # 移动左指针，尝试缩小窗口
            drop = s[left]
            left += 1
            if drop in need:
                if have[drop] == need[drop]:
                    valid -= 1  # 即将失去一个达标字符
                have[drop] -= 1

    return "" if best_len == float("inf") else s[best_start:best_start + best_len]

public String minWindow(String s, String t) {
    if (s.length() < t.length()) return "";
    Map<Character, Integer> need = new HashMap<>();
    for (char c : t.toCharArray()) need.merge(c, 1, Integer::sum);

    Map<Character, Integer> have = new HashMap<>();
    int valid = 0, left = 0;
    int bestLen = Integer.MAX_VALUE, bestStart = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (need.containsKey(c)) {
            have.merge(c, 1, Integer::sum);
            if (have.get(c).intValue() == need.get(c).intValue()) valid++;
        }
        // 窗口合法时尝试收缩
        while (valid == need.size()) {
            if (right - left + 1 < bestLen) {
                bestLen = right - left + 1;
                bestStart = left;
            }
            char d = s.charAt(left++);
            if (need.containsKey(d)) {
                if (have.get(d).intValue() == need.get(d).intValue()) valid--;
                have.merge(d, -1, Integer::sum);
            }
        }
    }
    return bestLen == Integer.MAX_VALUE ? "" : s.substring(bestStart, bestStart + bestLen);
}

复杂度分析：
时间复杂度为 $O(|s| + |t|)$，因为左右指针均单调移动，每个字符最多被访问两次。
空间复杂度为 $O(|\Sigma|)$，其中 $\Sigma$ 是字符集大小，取决于 t 中不同字符的数量。

易错点：
代码中有一行关键逻辑：if have[drop] == need[drop]: valid -= 1。这一步的含义是，在实际减少窗口内字符计数之前，先判断当前字符是否刚好处于"达标临界点"。如果顺序写反，valid 的值可能会与窗口状态不一致，导致错误结果。

6. LeetCode 567：字符串中的排列问题

题目描述：给定两个字符串 s1 和 s2，判断 s2 中是否存在一个子串是 s1 的排列。

解题思路：我们可以使用一个固定大小的滑动窗口，窗口的宽度为 len(s1)。两个字符串互为排列的充要条件是它们的字符频率分布完全一致。因此，我们只需要在 s2 上维护一个长度为 26 的频率数组（对应英文字母），并将其与 s1 的频率数组进行比较即可。

def check_inclusion(s1, s2):
    """LeetCode 567：判断 s2 是否包含 s1 的某个排列"""
    n, m = len(s1), len(s2)
    if n > m:
        return False

    # 初始化频率数组
    need = [0] * 26  # s1 的字符频率
    have = [0] * 26  # 当前窗口的字符频率
    for c in s1:
        need[ord(c) - ord('a')] += 1
    for c in s2[:n]:
        have[ord(c) - ord('a')] += 1

    # 统计当前匹配的字符种类数
    matches = sum(1 for i in range(26) if need[i] == have[i])

    # 滑动窗口遍历 s2
    for right in range(n, m):
        if matches == 26:  # 如果所有字符都匹配，返回 True
            return True

        # 新加入窗口的字符（右侧）
        r = ord(s2[right]) - ord('a')
        have[r] += 1
        if have[r] == need[r]:  # 新增匹配
            matches += 1
        elif have[r] == need[r] + 1:  # 超过目标频率，减少匹配数
            matches -= 1

        # 移出窗口的字符（左侧）
        l = ord(s2[right - n]) - ord('a')
        have[l] -= 1
        if have[l] == need[l]:  # 恢复匹配
            matches += 1
        elif have[l] == need[l] - 1:  # 减少导致不匹配
            matches -= 1

    return matches == 26  # 最后检查是否完全匹配

public boolean checkInclusion(String s1, String s2) {
    int n = s1.length(), m = s2.length();
    if (n > m) return false;

    // 初始化频率数组
    int[] need = new int[26];
    int[] have = new int[26];
    for (int i = 0; i < n; i++) {
        need[s1.charAt(i) - 'a']++;
        have[s2.charAt(i) - 'a']++;
    }

    // 统计初始匹配的字符种类数
    int matches = 0;
    for (int i = 0; i < 26; i++) {
        if (need[i] == have[i]) matches++;
    }

    // 滑动窗口遍历 s2
    for (int right = n; right < m; right++) {
        if (matches == 26) return true;

        // 新加入窗口的字符（右侧）
        int r = s2.charAt(right) - 'a';
        have[r]++;
        if (have[r] == need[r]) matches++;
        else if (have[r] == need[r] + 1) matches--;

        // 移出窗口的字符（左侧）
        int l = s2.charAt(right - n) - 'a';
        have[l]--;
        if (have[l] == need[l]) matches++;
        else if (have[l] == need[l] - 1) matches--;
    }

    return matches == 26;  // 最后检查是否完全匹配
}

复杂度分析：时间复杂度为 $O(m)$，其中 $m$ 是 s2 的长度。由于我们通过 matches 变量动态维护匹配状态，避免了每次重新扫描整个频率数组，因此单次窗口滑动的操作是常数时间。空间复杂度为 $O(1)$，因为我们只使用了固定大小的辅助数组。

扩展应用：这道题的解法可以直接套用到 LeetCode 438（找到字符串中所有字母异位词）。只需将"提前返回"改为"记录起始下标"即可解决该问题。

7. 进阶变体：滑动窗口最大值（单调队列）

如果滑动窗口需要查询的是窗口内的最大值或最小值，而不是简单的和或者计数，那么仅靠“加入新元素、移除旧元素”的简单操作就显得力不从心了。特别是当当前窗口的最大值被移除时，我们无法立刻知道下一个最大值是谁。这时就需要引入一种经典的数据结构——单调队列。单调队列的核心思想是维护一个严格递减的队列，队首始终保存当前窗口的最大值。

from collections import deque

def max_sliding_window(nums, k):
    """LeetCode 239：计算每个长度为 k 的滑动窗口的最大值。"""
    dq = deque()  # 存储下标，对应的值在 nums 中严格递减
    result = []

    for i, value in enumerate(nums):
        # 1. 移除队尾所有比当前值小的元素，因为它们不可能再成为窗口的最大值
        while dq and nums[dq[-1]] < value:
            dq.pop()
        dq.append(i)

        # 2. 如果队首元素已经不在窗口范围内，则将其移除
        if dq[0] <= i - k:
            dq.popleft()

        # 3. 当窗口大小达到 k 时，队首即为当前窗口的最大值
        if i >= k - 1:
            result.append(nums[dq[0]])

    return result

由于每个元素最多只会入队和出队一次，因此算法的时间复杂度仍然是 $O(n)$，与基础滑动窗口方法一致。

8. 滑动窗口的适用场景

下面这棵决策树清晰地展示了滑动窗口适合解决哪些类型的问题：

当问题满足以下三个条件时，滑动窗口通常是最佳选择：

  1. 目标答案对应一个连续区间（如子数组或子串）。
  2. 窗口的状态可以随着边界移动增量更新，例如通过累加和、频率表、单调队列等方式维护。
  3. 问题的目标是寻找固定大小（k）的区间，或者满足某种约束的最长 / 最短区间。

如果问题不满足上述条件，比如需要处理非连续元素、依赖全局信息，或者需要撤销任意插入操作，那么滑动窗口可能并不适用。此时，可以考虑使用前缀和、双指针、动态规划（dp）或堆等其他方法来解决问题。

def longest_valid(arr):

    state = empty_state()
    best = 0
    for right, x in enumerate(arr):

        while not is_valid(state):

def shortest_valid(arr):
    left = 0
    state = empty_state()
    best = float("inf")
    for right, x in enumerate(arr):
        state = add(state, x)
        while is_valid(state):
            best = min(best, right - left + 1)
            state = remove(state, arr[left])
            left += 1
    return best if best != float("inf") else 0

10. 常见误区

  • 窗口长度计算错误：窗口的长度应该是 right - left + 1，而不是 right - left。别忘了加 1。
  • valid 的更新时机不对：只有当 have[c] 第一次等于 need[c] 时才增加 valid；而当 have[c] 即将小于 need[c] 时，应该先减少 valid。顺序非常重要，搞反了会导致计数错误。
  • 用 list.pop(0) 实现固定窗口：这个操作的时间复杂度是 $O(k)$，效率太低。建议改用 collections.deque 或者简单的下标运算，确保操作是 $O(1)$。
  • 每次都重新扫描窗口内容：滑动窗口的核心思想是增量更新。如果在内层循环中重新遍历整个窗口，那复杂度优化就白做了，直接退化为暴力解法。
  • “和 ≥ target” 中出现负数的情况：在这种情况下，收缩窗口时总和可能不会减少，因此无法直接套用“最短窗口”模板。这时可以考虑使用前缀和配合单调队列，或者尝试换个思路解决问题。

12. 总结

滑动窗口的核心思想其实是摊还分析：与其每次都从头计算窗口的值，不如维护一个小小的状态，并在窗口移动时进行增量更新。两个指针单调递增，就能把暴力解法的 $O(n^2)$ 优化到 $O(n)$。

遇到连续区间问题时，可以问自己四个关键问题：

  1. 窗口大小是固定的，还是可变的？
  2. 如果窗口可变，目标是找最长的合法窗口，还是最短的合法窗口？
  3. 需要维护什么样的状态——是总和、频次表，还是单调队列？
  4. 窗口的不变量是什么？哪个指针的移动会破坏这个不变量？

只要搞清楚这四点，大多数滑动窗口问题基本就是套模板填空了。

13. 滑动窗口在实际工程中的应用

滑动窗口的应用远不止刷题。以下是我在生产环境中用过的三个场景：

  • 限流：令牌桶和滑动窗口计数器是两种经典实现。比如"每分钟最多 1000 次请求"，可以用一个时间戳队列（deque），从左侧逐出过期的时间戳。像 redis-cell 和主流 API 网关都采用了类似的实现。
  • 流式聚合：时序看板中常见的"过去一小时每分钟错误数"，本质上是对事件流做滑动窗口聚合。Kafka Streams 的 TimeWindow 和 Flink 的 tumbling/sliding window 就是这一思想的框架化实现。
  • 网络拥塞控制：TCP 的 cwnd（拥塞窗口）是对飞行字节的滑动窗口管理，而接收端通告的窗口则限制了发送端左边界能推进的距离。

面试题是滑动窗口的简化版，而生产环境中的实现则需要将数组索引换成时间戳，指针移动替换为基于 TTL 的过期逻辑。

虽然 C 风格的 while right < n 循环也能用，但用 for right, x in enumerate(arr) 会让代码更简洁，还能减少差一错误的风险。对比一下：

right = 0
while right < len(arr):

        remove(arr[left]); left += 1

第二种写法省掉了 right += 1（面试中最容易漏的一行），也不需要在循环里反复通过 arr[right] 取值。算法逻辑完全一致，但减少了出错的可能性。

对于频率表类的窗口问题，推荐直接使用 collections.Counter，而不是手写字典，因为它的 += 和 -= 操作已经经过充分测试：

from collections import Counter

def length_of_longest_substring_k_distinct(s: str, k: int) -> int:
    cnt = Counter()
    left = best = 0
    for right, c in enumerate(s):
        cnt[c] += 1
        while len(cnt) > k:
            cnt[s[left]] -= 1
            if cnt[s[left]] == 0:
                del cnt[s[left]]
            left += 1
        best = max(best, right - left + 1)
    return best

这里减完计数后调用 del 是关键，它确保 len(cnt) 的值始终准确。如果不删掉计数为 0 的键，“不同字符数"会错误地包含这些无效键。

15. 边界用例自查清单

提交代码前，建议在脑子里快速跑一遍以下用例，绝大多数"Wrong Answer"问题都能提前发现：

情况为什么会挂
空字符串 / 空数组初始值 best = 0 还是 best = float('inf') 容易搞错
“至多 K 个不同"中 k = 0合法窗口为空，但如果先无脑右移 right，可能会误报长度为 1
单个元素和 best 比较时用 > 还是 >= 在这里会暴露问题
元素全相同“求最长"通常没问题；“求最短"可能瞬间塌缩到长度 1 后无法扩展
target 大于 sum(arr)“和 ≥ target 的最短子数组"应返回 0，而非 float('inf')
包含负数大多数滑动窗口假设 add 是单调递增的，负数会破坏这一点，建议改用前缀和 + 单调队列
窗口比数组还大固定窗口需要显式判断 len(arr) < k
