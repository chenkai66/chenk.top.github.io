疑问 1 —— 宽带"缩水"。 运营商卖的是"100 Mbps"，浏览器下载峰值却只有 12 MB/s。被坑了吗？

第一部分 —— 数据单位：计算机世界的度量衡

位与字节

位（bit） 是二进制中最小的单位，要么是 0，要么是 1。你可以把它想象成一个开关，关就是 0，开就是 1。如果把 8 个这样的“开关”组合起来，就能表示 $2^8 = 256$ 种不同的状态，刚好足够用来编码最初的 ASCII 字符集，包括英文字母、数字和标点符号。这就是为什么 字节（byte） 被定义为 8 位。

关于“Mbps 和 MB/s”的坑：运营商卖宽带时用的是 每秒多少比特（Mbps），而浏览器显示下载速度时用的是 每秒多少字节（MB/s）。两者之间的换算很简单，只需将 Mbps 的数值除以 8 就能得到 MB/s。

你看到的实际含义
100 Mbps每秒传输 1 亿个 比特
12.5 MB/s（理论最大值）每秒传输 1250 万个 字节
10–11 MB/s（实际典型值）扣除 TCP 协议开销和网络拥堵后

所以，标称“100M”的宽带，实际使用时能达到 12 MB/s 左右的速度，完全合理。没人骗你，只是单位不同而已。

1024 与 1000 的差异从何而来？

计算机以 2 的幂次为基准进行计数，而最接近 1000 的 2 的幂是 $2^{10} = 1024$。因此，在计算机内部，1 KB 等于 1024 字节，1 MB 等于 1024 KB，依此类推。然而，硬盘厂商采用国际单位制（SI）标准，其中 1 KB 被定义为 1000 字节。正是这种二进制和十进制单位之间的差异，导致了“标称 1 TB 的硬盘在系统中只显示为 931 GB”的现象。

相比之下，内存（RAM）不会出现类似的“缩水”问题，因为内存模块从设计之初就按照二进制单位来定义容量。例如，一根标称为 8 GB 的内存条，其实际容量就是 $8 \times 2^{30}$ 字节，与系统计算方式完全一致。

第二部分 —— 深入 CPU 核心

CPU 到底在干什么

CPU 的任务说起来很简单，但实现起来却复杂得令人发指：取指令、解码、执行、写回结果，每秒重复几十亿次。至于其他那些复杂的部件——流水线、缓存、分支预测器——它们的存在只是为了加速这个循环，同时确保程序的语义不会发生任何变化。

一颗现代 CPU 核心内部可以分为四大块：

  • 前端（控制逻辑）：负责取指令、解码以及分支预测。
  • 计算单元：包括整数运算单元（ALU）、浮点运算单元、SIMD 单元，以及负责加载和存储数据的 Load/Store 单元。
  • 寄存器文件：一个极小但速度极快的临时存储区域（x86-64 架构下有 16 个通用寄存器，外加 AVX-512 向量寄存器）。
  • 缓存体系：L1 指令缓存 + L1 数据缓存（每个核心独享）、L2 缓存（每个核心独享）、L3 缓存（多个核心共享）。

简单来说，前端负责“准备工作”，计算单元负责“干活”，寄存器文件保存着计算单元需要用到的操作数，而缓存则确保工作数据能够以最快速度送到流水线上，让整个系统满负荷运转。

五级流水线

经典的处理器流水线设计通常分为五个阶段：

  1. IF —— 取指令（从 L1 指令缓存中读取下一条指令）
  2. ID —— 指令解码（解析指令含义，同时读取寄存器操作数）
  3. EX —— 执行（通过算术逻辑单元 ALU 进行运算）
  4. MEM —— 内存访问（执行数据加载或存储操作）
  5. WB —— 写回（将运算结果写入目标寄存器）

如果这些阶段不重叠运行，每条指令需要消耗 5 个时钟周期，5 条指令则总共需要 25 个周期。而通过流水线技术，每个时钟周期都会有一条新指令进入 IF 阶段，同时前一条指令推进到 ID 阶段，依此类推。经过 4 个周期的“预热”后，流水线被完全填满，此后每个周期都能完成一条指令的执行 —— 理想情况下，IPC（每周期指令数）达到 1.0。

现代 x86 处理器核心，例如 Intel 的 Golden Cove 或 AMD 的 Zen 4，设计要复杂得多 —— 每个周期可以解码 6 到 8 条指令，流水线深度超过 10 级 —— 但核心理念始终未变：让流水线的每一级在每个周期都保持忙碌。任何导致流水线中出现“气泡”（即某一级空闲）的情况都会降低吞吐量。接下来的四节内容，重点就在于如何避免或隐藏这些“气泡”。

内存层级结构

如果主存的速度能和寄存器一样快，那么所有的缓存机制就根本没有存在的必要了。但现实并非如此。从寄存器访问到机械硬盘寻道，延迟差距大约横跨八个数量级。

层级延迟容量范围技术实现
寄存器~0.3 ns几十个字节核心中的触发器
L1 缓存~1 ns32–64 KBSRAM，每比特使用 6 个晶体管
L2 缓存~4 ns256 KB–1 MBSRAM（密度更高，速度稍慢）
L3 缓存~15 ns8–64 MBSRAM（多核共享）
DRAM（DDR4/5）~90 ns8–128 GB每比特由 1 个晶体管 + 1 个电容组成
SSD（NVMe）~100 µs0.5–8 TBNAND 闪存
HDD~10 ms1–20 TB传统旋转磁盘

一个直观的类比模型。 假设我们将 CPU 的时钟周期放大到“人类思考 1 秒”的时间尺度（大约 1 纳秒对应 1 秒）：

  • L1 命中：瞬间完成，就像你立刻想起了某件事。
  • L3 命中：需要 15 秒，相当于走到旁边的工位。
  • DRAM：耗时 90 秒，相当于走到隔壁房间。
  • SSD：需要一天半的时间。
  • HDD：足足需要等待 4 个月。

这正是为什么一次 L3 缓存未命中会让 CPU 浪费数十个周期，也是为什么一段对缓存友好的内层循环，即使算术逻辑与另一段代码完全相同，性能却可以高出 10 倍的原因所在。

缓存未命中的三种类型

缓存并不是万能的，它会因为三种不同的原因而失效。了解具体是哪种未命中情况，才能有针对性地解决问题。

  • 强制性未命中（冷启动未命中）：当某个数据块第一次被访问时，缓存中自然不可能有它的记录。这种未命中在首次访问时是不可避免的，但可以通过预取（prefetching）来隐藏延迟。
  • 容量未命中：工作集的大小超过了缓存的容量。即使是一个完全相联且管理完美的缓存，仍然会出现未命中。解决方法包括缩小工作集（例如通过分块或平铺技术），或者使用更大容量的缓存层级。
  • 冲突未命中：在低相联度的缓存中，多个热点数据块可能会竞争同一个缓存组。例如，两个数组的地址恰好相差一个缓存组大小的倍数时，它们会反复互相驱逐。解决方法可以是调整数据布局、对数组进行填充（padding），或者依赖更高相联度的缓存。

现代处理器的 L1 缓存通常采用 8 路相联设计，而 L2 和 L3 缓存的相联度更高，因此纯粹的冲突未命中相比早期处理器已经较少见了。然而，在以 2 的幂为步长遍历数组的紧凑循环中，这种问题仍然可能出现。

分支预测

条件分支是流水线设计中的一大难题：当 if (x > 0) 在执行阶段（EX）真正确定结果时，后续的三四条指令可能已经按照某条路径被取出来了。如果预测错了，所有已经开始执行的工作都会被丢弃，重新取指 —— 现代处理器核心中，一次错误预测的代价高达 15–20 个时钟周期。

如果你在 C++20 中见过 [[likely]] 或 [[unlikely]] 标注，或者在 GCC 中用过 __builtin_expect，那其实是在帮助编译器优化分支预测的准确性。

乱序执行

即使分支预测做得再好，流水线依然会因为数据依赖问题而卡住。比如，从 DRAM 中加载数据（load）需要大约 90 纳秒（在 3 GHz 的频率下相当于 300 个时钟周期），而任何依赖这个值的指令都只能干等着。对于顺序执行的核心来说，它也只能跟着一起等，后续的所有指令都被堵在后面。

相比之下，乱序执行（OoO） 核心的做法就聪明得多：它会将解码后的几十甚至上百条指令暂存在一个叫做重排序缓冲区（Reorder Buffer, ROB）的地方，只要某条指令的操作数已经准备就绪，就可以立刻被调度执行，完全不用管它在程序中的原始顺序。当然，这一切的前提是，最终对外可见的结果（比如架构寄存器状态和内存内容）必须和顺序执行的结果保持一致。至于硬件内部怎么安排执行顺序，那就随便它发挥了。

以图中的例子来看，指令 I3 和 I4 并不依赖那次 load 操作。如果是顺序执行的核心，它们会被卡在 I2 后面等待；而乱序执行的核心则会利用 load 操作尚未完成的空闲周期，提前执行 I3 和 I4。结果呢？整个过程只用了 6 个周期，而不是 8 个 —— 这还只是个简单的例子。在实际工作负载中，配合大容量的重排序窗口（据说苹果 M3 的 ROB 容量能达到约 600 条 micro-op），性能提升不仅显著，而且完全是自动的，程序员完全不需要操心。

这也是为什么微基准测试的结果常常不靠谱的原因之一：如果你的循环体里包含彼此独立的指令，CPU 会自动并行执行它们，你测得的结果其实是瓶颈单元的吞吐量，而不是单条指令的延迟。

多核与 SMT（超线程）

如果不想通过提升单个核心的性能来增加并行能力，有两种方法可以实现：

  1. 多核设计。 把所有组件都复制一份：前端、ALU、寄存器文件、L1 缓存、L2 缓存，只有 L3 缓存和内存控制器是共享的。
  2. SMT（Intel 称之为 Hyperthreading）。 在一个物理核心内部，维护两套架构寄存器状态，让两个软件线程共享同一组执行单元。

理解 SMT 的关键点： 当某个线程因为缓存未命中或分支预测错误而停滞时，它的执行单元就会闲置。SMT 的作用就是让另一个线程立即接管这些空闲的资源。在实际混合负载中，这种技术通常能带来 20%–30% 的吞吐量提升，但不是翻倍 —— 因为两个线程仍然共享同一套 ALU、同一个 L1 缓存以及同一组执行端口。对于那些已经将执行单元完全占满的计算密集型任务（例如紧凑的矩阵乘法），SMT 可能反而会拖累性能，因为它会增加缓存的压力。这正是为什么许多 HPC 工作负载会选择关闭超线程。

这也正好解答了开篇提到的第三个问题。CS2 在 450 FPS 下的瓶颈并不是核心数量 —— 在这个帧率下，引擎的主线程才是主要限制因素。i5-13600K 的单线程性能优于 R9 7950X（更新的 P-core 架构、更高的睿频频率、每核心更大的 L2 缓存）。而 R9 7950X 那些额外的核心则处于闲置状态。只有当你的工作负载能够分解成更多并行线程时，更多的核心才有意义。

第三部分 —— 如何选择 CPU

当下的 Intel 与 AMD 对比

目前这一代处理器的主要取舍如下：

维度Intel（13/14 代酷睿）AMD（Ryzen 7000 / 9000 系列）
单线程性能峰值极高（P 核可睿频至约 6 GHz）极高（Zen 4/5 的 IPC 表现优异）
多线程吞吐能力E 核扩展多线程宽度核心数量多，且对称设计
制造工艺Intel 7 / Intel 4TSMC 5 nm / 4 nm
平台延续性LGA 1700（支持 12/13/14 代）AM5（承诺支持至 2027 年以后）
能效表现高端型号稍逊整体更优
最佳适用场景游戏、混合型负载内容创作、代码编译

根据不同工作负载选择：

  • 游戏：单线程性能是关键。中端的 Intel i5/i7 或者高主频的 Ryzen 7 是理想选择。
  • 视频剪辑、3D 渲染、大型代码编译：核心数量越多越好。推荐 Ryzen 9 或 Threadripper。
  • 日常办公、网页浏览、轻量开发：现代 4–6 核 CPU 完全够用，甚至性能过剩。
  • 服务器：AMD EPYC（支持 96+ 核心、12 通道 DDR5、更多 PCIe 通道）在密度和性价比上占据优势；而 Intel Xeon 在某些需要硬件加速的工作负载（如 AMX、QAT）中依然表现强劲。

32 位与 64 位：4 GB 的限制从何而来？

32 位地址总线理论上可以访问 $2^{32} = 4\ \text{GiB}$ 的内存空间。然而，问题在于，32 位操作系统不仅需要用这 4 GiB 来管理物理内存（RAM），还需要为内存映射 I/O（MMIO）预留一部分空间，比如显存、BIOS 和 PCIe 设备的寄存器等。通常情况下，系统会将顶部大约 0.5–1 GiB 的地址空间分配给这些设备，导致实际可用的 RAM 只剩下 3.0–3.5 GiB 左右。

这就是为什么很多人遇到过这样的情况：明明安装了 4 GB 内存，但 Windows 却只显示 3.25 GB 左右。这种现象的根本原因就在于 32 位系统的地址空间被“挤占”了。而要彻底解决这个问题，唯一的办法就是升级到 64 位操作系统。64 位系统拥有 $2^{64} \approx 16\ \text{EiB}$ 的地址空间，足以让这类限制成为历史。

ECC、多路与服务器主频为何更低

服务器 CPU 的设计选择与其工作负载密切相关，和桌面级 CPU 有着本质区别：

  • ECC 内存能够检测并修复单比特错误。在服务器场景中（数千根 DIMM 全天候运行），比特翻转的发生频率高到不可忽视；在金融、科学计算以及任何对安全性要求极高的领域，ECC 是标配。
  • 多路架构允许两颗或四颗 CPU 共享一个一致的内存视图，这对于超大规模数据库和虚拟化平台来说至关重要。
  • 更多 PCIe 通道（64–128 条，而桌面端通常只有约 20 条）可以连接更多的 GPU、NVMe 固态硬盘和 100 GbE 网卡。
  • 更低的主频（全核约 3 GHz，而非单核冲刺到 5.8 GHz）通过牺牲峰值性能换取了在持续高负载下的能效和可靠性。桌面 CPU 像短跑选手，追求爆发力；服务器 CPU 则是马拉松健将，注重耐力。

快速参考清单

  • 带宽按 bit 计算，文件大小按 byte —— 别忘了除以 8。
  • 硬盘厂商用 1000 进制，操作系统用 1024 进制 —— 标称容量乘以 0.931 就是实际可用空间。
  • 流水线的理想状态是每周期执行一条指令；缓存、分支预测、乱序执行等机制的存在，都是为了尽可能接近这一目标。
  • 内存延迟跨越八个数量级。优化缓存局部性通常是提升性能的关键。
  • 分支预测失败会浪费约 15–20 个周期，DRAM 缺失则可能耗费约 300 个周期。预测器和缓存的作用就是尽量隐藏这些开销。
  • 同时多线程（SMT）通常只能带来 20–30% 的性能提升，而不是翻倍。
  • 更多核心只对可并行化的任务有帮助，而游戏通常不属于此类。

  • 内存故障排查：黑屏、蓝屏、MemTest86。

思考题： 如果 CPU 已经配备了 L1/L2/L3 缓存，为什么还需要 DRAM？（提示：单位成本容量、SRAM 的制造成本。）

推荐阅读

  • Hennessy & Patterson，《计算机体系结构：量化研究方法》—— 领域经典著作。
  • Bryant & O’Hallaron，《计算机系统：程序员视角》（CSAPP）—— 本科阶段必读教材。
  • Agner Fog 的优化手册 —— 实用的 x86 微架构细节指南。
  • Intel ® 64 和 IA-32 架构软件开发者手册。
  • AMD64 架构程序员手册。
  • WikiChip —— 提供现代微架构的简洁且准确的芯片布局图。

附录 A —— 现代流水线的真实面貌

教科书中常见的五级 RISC 流水线（取指 -> 译码 -> 执行 -> 访存 -> 写回）其实是 1990 年代的技术产物。2025 年的 Zen 5 或 Lion Cove 内核要复杂得多：

  • 译码宽度 4–8。 每个周期最多可以译码八条 x86 指令。瓶颈通常在于前端取指，而非后端执行。
  • 微操作缓存（μop cache）。 译码后的微操作会被缓存起来，热循环可以直接跳过译码器。如果循环能完全装入 μop 缓存，IPC 可以稳定在约 6；一旦溢出，同样的代码 IPC 会跌至约 2。
  • 600 条在飞指令。 重排序缓冲区（ROB）非常庞大 —— Zen 5 约 448 项，Lion Cove 约 576 项。CPU 一直在投机执行你代码中接下来约 150 纳秒的部分。
  • 两条整数重命名队列 × 4–6 个 ALU。 你以为的“一个核心”，内部其实是一组分布式调度器。
  • load 队列 200+、store 队列 100+。 写入 L1 的数据需要等到 store 操作退役后才会正式提交，但 load 操作可以通过 store-to-load forwarding 提前获取值。

教科书上的流水线图依然能解释为什么分支预测失败代价高昂，但它没有告诉你的是，现代核心上的真实代价约为 16–20 个周期，因为 rename 单元、调度器、ROB 等结构都需要被清空并重新填充。五级流水线图让分支预测失败看起来很便宜，实际情况并非如此。

附录 B —— 在你的机器上试试这些命令

每位工程师都应该掌握的三条命令。请在你真正关心的实际负载上运行它们，而不是在合成基准测试中。

Linux —— perf stat：

1
2

perf stat -e cycles,instructions,branches,branch-misses,\
cache-references,cache-misses ./your_program

需要关注的几个指标：

  • IPC（instructions / cycles）—— 低于 1.0 表示 CPU 被饿着了；高于 2.0 表示前后端都处于健康状态。
  • 分支错误率超过 2% 说明某个热点分支难以预测。
  • 缓存缺失率超过 5% 通常意味着工作集大于 L2 缓存。

macOS —— Instruments -> Time Profiler / Counters 模板。 Counters 模板暴露了 Apple Silicon 的性能计数器：INST_ALL、CYCLES、L1D_CACHE_MISS_LD、BRANCH_MISPRED_NONSPEC。指标相同，名称不同。

通用方法 —— taskset + time。 将单线程基准测试绑定到一个核心上，以消除调度器噪声：

1

taskset -c 3 perf stat -r 5 ./bench

-r 5 参数表示运行五次并输出均值 ± 标准差。如果标准差超过 5%，说明有其他进程在争抢该核心。

附录 C —— GPU 的独特之处

GPU 的 SM（流式多处理器）并不是缩小版的 CPU 核心，而是一种完全不同的计算单元：

  • 没有分支预测。 GPU 通过遮罩执行分支 —— 两边都会运行，错误路径的结果会被丢弃。一个 warp 中的发散分支会直接将吞吐量减半。CPU 的分支预测器在这里毫无意义，因为 GPU 在这个粒度上没有“每线程的程序计数器”。
  • 没有乱序执行。 延迟通过切换到另一个 warp 来隐藏，而不是通过重排当前指令流。因此，一个 SM 可以同时驻留 32–64 个 warp，而 CPU 一次只处理一个线程的指令流。
  • 合并访存（coalesced access）。 一个 warp 的 32 个线程发出 32 个 load 请求，硬件只有在地址连续且对齐时才会将它们合并成一次 128 字节传输。CPU 因为有缓存，散乱访问几乎无感；同样的访问模式在 GPU 上可能导致 32 倍的带宽浪费。
  • 寄存器文件巨大。 H100 的一个 SM 拥有 256 KB 的寄存器文件 —— 比其 L1 缓存还大。相比之下，CPU 的寄存器文件只有几 KB。优化策略完全不同。

一句话总结：在 CPU 上表现良好的代码，在 GPU 上可能因为完全不同的原因表现糟糕。不要简单地将调优经验从 CPU 照搬到 GPU。
