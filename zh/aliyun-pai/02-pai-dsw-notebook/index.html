阿里云 PAI 实战（二）：PAI-DSW——不会吃掉你权重的 Notebook

PAI-DSW 实战：选对 GPU 镜像、把 OSS 挂好不丢权重、把官方 Quick Start 的 MNIST 完整跑通。再附上一些只在淘宝场景里踩过才知道的坑。

每次新同学进 PAI，第一天都长一个样。开 DSW 实例，pip install 一通，训练一小时，因为某种原因重启了 kernel，然后跑来问我模型文件去哪了。诚实的答案是——“在 /root 里，但那台机器已经不存在了”——这个教训学一次就够。这篇文章就是那种你应该在踩坑前就读到的版本。

DSW 到底是什么

按官方"DSW Overview"，DSW 是一个云上 AI 开发 IDE，集成 JupyterLab、VSCode、终端，预置 PyTorch / TensorFlow 镜像，支持异构算力（CPU / GPU / Lingjun），能挂载 OSS / NAS / CPFS 数据集。落到实操就是：点"打开"，一分钟内你拿到一个真实的 Jupyter，跑在真实的 GPU 上，nvidia-smi 能用，PyTorch 能 import。

文档没明说的是盒子里没有什么。DSW 容器有一块系统盘，但只跟实例同生共死。pip install 装的东西能扛住 kernel 重启，但扛不住实例重启——除非你把 conda 环境持久化到 OSS，或者用快照保存到 ACR。

选实例规格

按文档，DSW 资源类型分两种：公共资源（按量付费）和专有资源（基于通用算力或灵骏的包年包月）。日常工作公共资源就够——按 GPU 分钟计费，秒级精度，开 10 分钟做个实验也不肉疼。

我自己的实操选择：

  • 小实验 / 调试 — ecs.gn7i-c8g1.2xlarge（1 × A10，24G）。便宜，跑个 4-bit 量化的 7B 微调或者 512×512 扩散都够。
  • 正经训练个小模型 — ecs.gn7i-c16g1.4xlarge 或 ecs.gn7e-c12g1.3xlarge（A10 / A100 40G）。CIFAR-10 ResNet、ImageNet-tiny、QLoRA 7B SFT 都从容。
  • LLM 开发 — ecs.gn7e-c12g1.6xlarge 起步（A100 80G）。要 BF16 加载 13-30B 不卸载，必须从这一档起。

真实经验： 控制台显示"无库存"时，换个可用区试。库存是按 AZ 分的，不是按 region。我见过同一分钟 cn-shanghai-h 的 A100 80G 没货，cn-shanghai-l 却有空。

镜像目录

DSW 镜像都是官方、有版本、有标签的。Quick Start 里用的 modelscope:1.26.0-pytorch2.6.0-gpu-py311-cu124-ubuntu22.04 一眼就能读懂：从左到右，ModelScope SDK 1.26、PyTorch 2.6、GPU 构建、Python 3.11、CUDA 12.4、Ubuntu 22.04。

我基本都选 pytorch 或 modelscope 系。TensorFlow 镜像没问题，但通常要滞后一个大版本。还有 dsw-stable 系列，故意滞后——上线临时训练任务最好挑这个，免得 CUDA 中途升级。

也可以自己 baking 镜像推到 ACR。我对那种依赖很重的项目（vllm、flash-attn、自研 CUDA 算子）会这么做——每次新实例少 4 分钟 pip install。

数据到底放哪

整个 DSW 文档里最重要的图：

我到处用的挂载布局：

/mnt/data/
├── datasets/      # OSS 只读挂载（桶永远在）
├── checkpoints/   # OSS 可写前缀（每 N 步保存）
└── code/          # git repo，也放 OSS，新实例一挂即用

挂载 OSS 在创建实例时配置，文档里叫"配置存储"。选好桶和前缀，挂载点 /mnt/data/，访问模式默认 FUSE。开起来后在终端跑 oss ls oss://your-bucket/ 应该能列出文件——这就是你的 PAI ↔ OSS RAM 角色健康检查。

import torch
import torch.nn as nn
import torch.nn.functional as F
from torch.utils.data import DataLoader
from torchvision import datasets, transforms

device = "cuda" if torch.cuda.is_available() else "cpu"
print("Device:", device)

tx = transforms.Compose([transforms.ToTensor(),
                          transforms.Normalize((0.1307,), (0.3081,))])
train = datasets.MNIST("/mnt/data/datasets", train=True,  download=True, transform=tx)
val   = datasets.MNIST("/mnt/data/datasets", train=False, download=True, transform=tx)

train_loader = DataLoader(train, batch_size=128, shuffle=True,  num_workers=2)
val_loader   = DataLoader(val,   batch_size=512, shuffle=False, num_workers=2)

class Net(nn.Module):
    def __init__(self):
        super().__init__()
        self.c1, self.c2 = nn.Conv2d(1, 32, 3, 1), nn.Conv2d(32, 64, 3, 1)
        self.fc1, self.fc2 = nn.Linear(9216, 128), nn.Linear(128, 10)
    def forward(self, x):
        x = F.relu(self.c1(x)); x = F.max_pool2d(F.relu(self.c2(x)), 2)
        x = torch.flatten(x, 1)
        x = F.relu(self.fc1(x))
        return self.fc2(x)

model = Net().to(device)
opt = torch.optim.AdamW(model.parameters(), lr=1e-3)

for epoch in range(3):
    model.train()
    for xb, yb in train_loader:
        xb, yb = xb.to(device), yb.to(device)
        opt.zero_grad(); F.cross_entropy(model(xb), yb).backward(); opt.step()
    # 写到 OSS，不写 /root
    torch.save(model.state_dict(), f"/mnt/data/checkpoints/mnist_e{epoch}.pt")
    print(f"epoch {epoch} done")

Quick Start 期望单 A10 跑 3 epoch 后验证集准确率约 98%。明显低很多大概率是 OSS 挂错了，读到了错误目录——不是模型 bug。

跨会话保存环境

DSW 有两套机制，都值得知道：

  1. 实例镜像快照 — 把当前容器状态（装的包、系统改动）打到 ACR。下次开新实例选这个镜像就能复活。慢（几分钟）但精确。
  2. OSS 上的 conda 环境 — 把 pip 全装到 /mnt/data/envs/myenv/，激活它来用。能扛住实例死亡，无需重新打镜像。快但抓不到系统级改动（apt install 之类）。

项目级工作我用 conda-on-OSS，“半年后还要演示的固化 demo"我用快照。

镜像目录，怎么选不靠猜

每开一次 DSW 实例都要选一个镜像，多数团队第一个季度都至少选错过一次。目录里大致四个家族：

家族标签模式适合不适合
modelscope:*modelscope:1.28.0-pytorch2.6.0-gpu-py311-cu124-ubuntu22.04LLM 开发、ModelScope hub 下载、自带 transformers 和 vllm 的活CUDA 版本要比 ModelScope 出厂的更新
pytorch:*pytorch:2.6.0-gpu-py311-cu124-ubuntu22.04纯 PyTorch、自定义训练循环、讨厌依赖泥潭想要全套 LLM 装备
tensorflow:*tensorflow:2.16.1-gpu-py311-cu123-ubuntu22.04TF / Keras 老项目、TFRecord 流水线2026 年新开项目（别开）
dsw-stable:*dsw-stable:1.10-pytorch2.4-cu121长期跑的 notebook，不希望 CUDA 中途升级要最新特性

命名规则从左到右一致：SDK 版本 → 框架版本 → CPU/GPU → Python 版本 → CUDA 版本 → 操作系统。当菜谱读。我见过最常的两种翻车：

  • 想要纯 PyTorch 却选了 ModelScope。 镜像 ~14 GB，第一次启动要拉 600+ Python 包。如果用不到 modelscope SDK 或 vllm，省下 90 秒冷启动和磁盘压力。
  • 想要 ModelScope 却选了 PyTorch。 然后每开新实例就 pip install vllm flash-attn modelscope。能用，就是每次多 4 分钟，偶尔 flash-attn 决定按错的 nvcc 编译时还得对 CUDA 版本。

vllm 重的 Qwen3 工作流，modelscope:1.28.0-pytorch2.6.0-gpu-py311-cu124-ubuntu22.04 阻力最小——vllm、flash-attn、xformers、transformers 全部预装在兼容版本上。纯自研 CUDA 研究，从 pytorch:2.6.0-gpu-py311-cu124-ubuntu22.04 起步往上加。任何 6 个月后还要回来跑的项目，把镜像 tag 钉在项目 README 里——modelscope:1.28.0-... 半年后即使 tag 一样，构件也不一定一样，因为阿里云会偶尔重建并打依赖补丁。

真实经验： 启动前先看一眼镜像构建日期。超过 60 天大概率漏了至少一个 CVE 补丁；类生产环境就在同一家族选最新 tag。

OSS-FUSE 挂载：延迟剖面，以及什么时候改成拷贝

OSS-FUSE 是默认挂载机制，90% 的活够用，但翻车方式很微妙。心智模型：挂载路径上每个 read() 都是一次 OSS HTTP 请求，每次 write() 在本地缓冲，close() 或者每几 MB 刷一次。这意味着：

操作本地 SSDOSS-FUSENASCPFS
顺序读 1 GB~250 ms600-1200 ms400-700 ms80-120 ms
随机读 4 KB（冷）~0.1 ms30-80 ms5-15 ms1-3 ms
os.listdir(10000 文件)~10 ms800-2000 ms200-500 ms50-100 ms
追加 10 MB 日志<5 ms200-400 ms50-100 ms10-30 ms

数字会随区域和桶类型浮动，但相对关系稳定。我落到肌肉记忆的几条结论：

  • 不要对 1 万+ 文件的 OSS 挂载目录跑 os.listdir()。那是一次 ListObjects HTTP 来回，Python 还会懒迭代——你的训练脚本会在每个 epoch 看起来卡 1-2 秒。改用预先写好的 manifest 文件。
  • 训练时不要不开 enable_easyckpt 直接往 OSS-FUSE 写 checkpoint。一个 7B state_dict ~14 GB；FUSE 会让训练进程阻塞 10-30 秒每次保存，GPU 在那干等。要么先存本地 SSD 再异步上传，要么走 EasyCKPT 路径（见第三篇）。
  • 训练数据 >100 GB 时不要直接从 OSS-FUSE 读。每个 FUSE 挂载点的带宽上限大约 200-400 MB/s，会卡在数据加载上。启动时用 ossutil cp -r --jobs 8 拷到本地 SSD，从本地训练。本地 SSD 装不下的就用 NAS 或 CPFS。
  • 代码、配置、输出目录用 OSS-FUSE 完全没问题。偶尔读延迟可以接受，写也很便宜——只要不在热循环里写。

mount_modes = {
    "oss-fuse":   "默认 — 类 POSIX，懒加载，代码/配置都行",
    "oss-direct": "跳过 FUSE，自己用 ossutil/oss SDK",
    "nas":        "真 POSIX，按 GB-月计费，共享暂存区好用",
    "cpfs":       "HPC 吞吐，贵，>50 GB/s 聚合带宽时才用",
}

我自己默认全 OSS-FUSE，配一个 local_copy_step() 在每次训练开头把热数据拉到 /root/data/（系统盘，快 SSD，临时——但没关系，OSS 上有真源）。50 GB 图像数据集流式读 ~15 分钟，先拷再训 ~3 分钟，从此告别加载瓶颈。

空闲自动停止、自动调规格、GPU 共享

三个生命周期特性，第一次忘记设它们的时候就会知道值钱。

空闲自动停止。 每实例可配。平台监控 CPU + GPU + 网络活动；三项都低于阈值持续 N 分钟，实例停掉。我的默认：开发实例 30 分钟，跟同事共享的实例 15 分钟，长跑训练 notebook 不开（用定时停止兜底）。一台被忘掉的 A10 ~5 RMB/h，过个长周末就是 360 RMB；30 分钟空闲停只花 2.5 RMB，剩下都省。陷阱：你一边屏幕共享、一边盯着每分钟刷新一次的 TensorBoard，会被检测器误判成"活动”。真要保活就在终端跑 nvidia-smi -l 1。

自动调规格。 不太被提到。可以停掉实例、换一个 GPU 规格再启动，/mnt/data/ 挂载和 OSS 上的 conda 环境都不丢。我自己经常临时升到 A100 80 GB 跑半小时压测，第二天上午再降回 A10 写代码。这是手动停 / 改 / 启，不是热切——但状态都持久化了，2 分钟操作完事，不是重新装一遍。

GPU 共享（cGPU）。 最新的旋钮，容易被忽略。PAI 用 cGPU 虚拟化把一张物理 GPU 切给两个 DSW 实例用，按比例分配显存 + 算力（比如把 A10 的 24 GB 切成 16 GB / 8 GB 给两个用户）。一个初级同学做推理只要 <8 GB 时，不必给他单独开一张 A10。代价：cGPU 多 5-15% 开销，租户间隔离是"尽力而为"——别和不可信用户共享。在工作空间的资源共享里配，然后在 DSW 选规格里多出一个新型号。

这三个旋钮配齐，能把开发环境 GPU 账单大致砍一半。新工作空间第一天就值得设好。

快照 vs 自定义镜像 vs git pull——挑对那个

三种让"新开 DSW 实例"感觉像"我刚回来"的办法。各有甜蜜区。

/mnt/data/code/ git pull 永远是源码的正确答案。代码放 OSS，第一次 clone 进来，之后每次启动 git pull。扛得住实例死，是唯一能跟 code review 衔接的机制，零成本。如果你的工作流里没 git，先修这个再继续。

OSS 上的 conda 环境 是 Python 依赖的正确答案。在 /mnt/data/envs/myenv/ 下建 env，激活，pip install。扛得住实例死，因为 env 文件在 OSS 上。比本地 env 启动慢 ~30 秒（conda 激活时扫 FUSE 有开销），但不用重建，无缓存失效。约束：apt install 装的系统包不能这样持久，会跟实例同生共死。

实例镜像快照 是有非 Python 状态时的正确答案。系统包（libnuma、自定义 cuda 库、奇怪的 C++ 依赖）、ACR 管理的内核模块、/etc/ 里的任何东西。快照把整个容器文件系统冻结到 ACR；下次开实例选这个镜像，跟上次一模一样。慢（baking 3-8 分钟，多 1-2 分钟拉镜像）但精确。我只在两种窄场景下用快照：（a）半年后还要演示的固化 demo，（b）我花了一天才调好兼容性的 CUDA 栈，一辈子不想再调一次。

带 Dockerfile 的自定义 ACR 镜像 是团队级可复现性的正确答案。CI 里构建，按日期 tag，推到 ACR，每个团队成员的 DSW 拉同一个镜像。任何 >2 人贡献的项目我都默认这样——快照路径让"到底装了什么"变成黑盒，Dockerfile 可 review。代价：每次改动多 5 分钟 CI，你得维护 Dockerfile。值。

决策树：

  • 只改了代码？git pull。
  • 改了代码 + 自己加几个 pip install？OSS 上的 conda。
  • 重做这套环境会让我难过？快照。
  • 不止一个人用这套环境？自定义 ACR 镜像。

最常见的错：什么都用快照，包括本来该进 git 的东西。然后你拿到一个冻结镜像，里面写死了 /root/notebooks/foo.ipynb，去年改了什么完全 diff 不出来。
