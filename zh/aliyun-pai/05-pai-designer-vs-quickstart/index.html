阿里云 PAI 实战（五）：Designer vs Model Gallery——GUI 什么时候真值钱

PAI-Designer 处理表格 ML 流水线，Model Gallery 一键部署/微调开源模型。一份诚实的决策矩阵：什么时候该跳过 SDK、让 GUI 帮你交付。

CK
Chen Kai March 9, 2026 · 9 分钟 · 4505 字
#阿里云 PAI#PAI-Designer#Model Gallery#低代码
Read in English →

前四篇讲的是底层原语——DSW、DLC、EAS——你用 Python 把它们编排起来。这一篇讲两个把这些原语包起来、给"不想写 Python"的用户用的 GUI 产品：PAI-Designer 处理拖拽式表格流水线，Model Gallery 零代码部署/微调开源模型。它们不是严肃工程师第一反应的工具，但有两个具体场景下，它们就是正确答案。

Designer——拖拽式流水线编辑器

按文档原话，Designer “通过工作流的方式来实现建模与模型调试，用户可以根据建模需求在工作流中通过拖拉拽的方式排布不同组件，像搭积木一样构建 AI 开发流程”。卖点数据：140+ 内置算法组件、可导出为 JSON、可在 DataWorks 调度、节点可用自定义 SQL / Python / PyAlink 脚本。

它的强项：

  • MaxCompute 规模的表格 ML。 Designer 和 MaxCompute 绑得很紧。如果你的训练数据是 2 亿行的 MaxCompute 分区表，Designer 内置的 source / split / encode / train 组件直接在 MaxCompute 里跑，不是隔着网走 Python pod。你付的是 MaxCompute 的算力钱，不是闲着等数据的 GPU pod 钱。
  • 交给非开发同学。 推荐、流失、风控团队里常有"懂业务但不写 Python"的领域专家。Designer 画布是他们能看懂、能改、能 own 的。
  • 内置案例。 文档里列了商品推荐、新闻分类、金融风控、雾霾天气预测、心脏病预测、农业贷款、人口普查。直接当起步模板用，哪怕拆掉一半再换。
  • 离线调度。 工作流导出 JSON，扔给 DataWorks，配上每日/每小时定时和重试。

它的弱项：

  • 任何 LLM 形态的事。Designer 强在特征工程 + 经典 ML，不是你写自定义 PyTorch 训练循环的地方。
  • 自研 CUDA、新损失函数、“算法本身就是产品"的事。

我自己的分工：表格类工作量用 Designer 出，定制训练用 DLC 出，分得很清楚。

Model Gallery——零代码 MaaS 捷径

Model Gallery 是把 DLC + EAS 包起来给非 MLOps 用户用的工具，让他们能 6 次点击微调和部署一个开源模型。文档原话：“Model Gallery encapsulates Platform for AI (PAI)-DLC and PAI-EAS, providing a zero-code solution to efficiently deploy and train open-source large language models”。

Quick Start 里 Qwen3-0.6B 的端到端流程：

  1. Model Gallery 搜 “Qwen3-0.6B” → 点 部署。
  2. 默认 GPU、默认 vLLM 镜像，所有都默认 → 确定。
  3. ~5 分钟后状态翻到 Running。
  4. 查看调用信息 → 拿 Internet Endpoint 和 token。
  5. 接到 Cherry Studio（或 Claude Code MCP，或 Python SDK 用 OpenAI-compatible base URL），开始聊。

微调那块，Quick Start 走了一个物流信息抽取的例子：喂 JSON 数据集，下拉框选 LoRA 超参，它就替你提一个 DLC 任务。Quick Start 还特意点出了蒸馏模式——大教师（Qwen3-235B）打标，小学生（Qwen3-0.6B）学。这个模式值得记住，是我所知性价比最高的微调路径。

Gallery 的强项：

  • 10 分钟评估新模型。 DeepSeek-V3 出的那天，我团队从听说到部署到聊上天，刚好咖啡续完。如果还要自己配 OSS 桶、安全组、SSL，绝无可能。
  • 给非工程同学演示。 点 → 端点 → Cherry Studio 聊天 → 董事会汇报。
  • 一键 LoRA 微调。 多数领域适配，Gallery 选的默认值（LR、epochs、LoRA rank）和最优值差不到 5%。

Gallery 的弱项：

  • 自定义架构。改了模型代码，必须 DSW + DLC。
  • 严苛延迟目标。Gallery 给你的部署默认值是合理的，不是优化过的。要 <100ms p99 得自己写 EAS 部署，把 batching 配好。
  • 离网或跨区部署。Gallery 假设"在哪个区就部署在哪个区”。

什么时候选什么

到目前还能站住的决策矩阵：

启发式：从需求允许的最高层往下选。多数团队第一天过度工程——本来该是 Model Gallery 部署的事，搞成自定义 DLC + EAS 流水线。先优化首字时延，等真有流量真有数据再下沉重构。

一个 Designer 干掉自定义代码的真实例子

真实工单：营销部要每周对 6000 万行的 MaxCompute 表跑用户分群。算法同学的第一反应是 PySpark + scikit-learn 的 DLC 任务，代码放 OSS，SLS 回调到 EventBridge 调度。三天工作量。

Designer 版本：source 节点 → 抽样 → 编码 → KMeans → 写回 MaxCompute。导出 JSON 在 DataWorks 调度。两小时，包括跟营销 PM 解释清楚这玩意。同样的输出表，成本一半（无 GPU pod），维护量十分之一。

一个 Model Gallery 省了一周的真实例子

我们要测 Qwen3-Coder 是不是能替代某个内部基于 qwen-plus 的代码评审 bot。Gallery 之前：读 vLLM 文档、搭 EAS 部署、写 OpenAI-compatible 桥、交付给团队。Gallery 之后：搜 → 部署 → 端点接进现有客户端 → 午饭前完工。我们能集中精力在真正的问题（这模型有没有更好）上，而不是耗在管线上。

一棵具体的决策树

上面的矩阵是启发式，下面是我自己被同事问"这事我该用 Designer / Gallery / DLC / EAS 哪个"时跑的真决策树：

开始：你在做什么？
│
├─ 表格 ML（回归 / 分类 / 聚类 / 推荐）
│   ├─ 数据在 MaxCompute？ ──── 是 ──→ DESIGNER
│   ├─ 数据在 OSS / 外部？ ── DLC + 你的代码
│   └─ 实时打分（>10 QPS、<100 ms p99）？ ── DLC 训 + EAS 推
│
├─ 开源 LLM（Qwen / DeepSeek / Llama 等）
│   ├─ 快速评估（"这模型对 X 够好吗？"） ───── MODEL GALLERY
│   ├─ JSON 数据集上的默认 LoRA 微调 ──── MODEL GALLERY
│   ├─ 自定义训练循环 / 新损失 / RL ────── DLC + 你的代码
│   ├─ 自定义推理 batching / 量化 ──── EAS + 你的容器
│   └─ 要 <100 ms p99 ───────────────── EAS + 你的容器（Gallery 默认太松）
│
├─ 自定义模型（自研架构）
│   ├─ 任何场景 ──────────────────────── DSW（开发）→ DLC（训练）→ EAS（推理）
│
├─ 生成式（图 / 视频 / 音频）
│   ├─ 用托管的万相 / Qwen-TTS API ───── 百炼（另一个系列）
│   ├─ 自托管开源 diffuser / TTS ───── MODEL GALLERY（评估）→ EAS（生产）
│
└─ 数据标注 / human-in-the-loop
    └─ iTAG（本系列没覆盖）

几个不那么明显的分支：

  • 数据在 MaxCompute 是 Designer 最强的信号。 不是数据量、不是算法——是数据放在哪。从 MaxCompute 用 pyodps 拉 5000 万行到 Python pod 慢且贵；同样的模型在 Designer 组件里跑、留在 MaxCompute 里又快又便宜。任何设计文档里你写出"先把表导出来"——你已经输掉了"为什么不用 Designer"那场争论。
  • “快速评估"是 Gallery 的强项，哪怕你最后要自托管。 在 Gallery 花 10 分钟确认 Qwen3-Coder 对你的场景够好，再花 2 天写优化的 EAS 部署，严格快于直接花 2 天写 EAS 部署再发现模型不够好。
  • EAS 那一支永远不走 Gallery。 Gallery 自动部署的 serving 配置还行（中位延迟、没意外），但任何生产级服务你都要手写 EAS spec——固定 vLLM batching 参数、选 GPU 型号、配扩缩容。Gallery 是 dev，手写 EAS 是 prod。

我见过最常的错：本来"先在 Gallery 上线快"是合理的，但拖过了那个合理点，三个月后账单出来或者延迟开始重要了，再慌忙重部。计划好真有用户的那一刻就从 Gallery 切到手写 EAS。

各场景的成本对比

下面是我跑过的真实工作量，归一到月 RMB。具体价格随区域和季节波动，重要的是比例。

场景 A：6000 万行 MaxCompute 表的每周用户分群。

方案搭建时间单次成本月成本（4 次）维护
Designer2 小时~6 RMB（MaxCompute 抢占）~24 RMB可忽略
DLC + PySpark3 天~38 RMB（4 节点 40 分钟）~152 RMB每季度 1 工程师日升镜像
手写 EMR 任务1 周~45 RMB~180 RMB多个工程师日

Designer 在每条轴上都赢。别争。

场景 B：新开源 LLM 的 10 分钟评估（刚发布的 Qwen3-VL-7B）。

方案评估时间评估期成本放弃成本上线成本
Model Gallery15 分钟（部署）+ 实际评估时间~5 RMB/h × 评估时长0（删服务）走 EAS 重新部署上生产
手撸 EAS1-2 天（搞清 vLLM、挂权重、debug）~5 RMB/h × 评估 + 搭建~50 RMB 沉没已经在那
DSW notebook1 小时（下模型、跑推理循环）~5 RMB/h × 评估时长0不行，DSW 不能做 serving

Gallery 在评估时间上赢。放弃就没后续工作；上线也要走 EAS 重部，因为 Gallery 默认值不是生产级。手撸 EAS 只在你确定要上线、且不介意前置成本时才赢。

场景 C：生产 LLM 聊天端点，平均 5 QPS、峰值 30 QPS。

方案搭建月成本p99 延迟备注
Model Gallery（默认）5 分钟~14,000 RMB~2.5 s默认 min_replicas=2，无 batching 调优
EAS 手写（优化过）1-2 天~10,500 RMB~1.2 svLLM 调过、定时扩缩、权重 baked
百炼托管 Qwen-Plus0（API 而已）浮动——通常 ~3-8 RMB / 100 万 token~1.5 s别人的 GPU、别人的事

每次"我们要不要自托管"规划会都该有这个对话。每月 5 万-10 万请求，百炼在成本和运维负担上都赢。超过每月 100 万请求或者有数据驻留要求，自托管 EAS 反超。

场景 D：5000 条 JSON 上微调 Qwen3-7B。

方案搭建跑一次成本质量
Model Gallery 默认 LoRA10 分钟~80 RMB（单 A100，~3 小时）多数任务和最优值差 5% 以内
DLC 自定义 Megatron + LoRA2-3 天~60-100 RMB可调到最优，5 万条以上才值
DSW 手动跑半天~80 RMB跟 Gallery 同水平，可观察性更好

Gallery 这里也赢——除非你有具体理由要调训练循环。Gallery 的"默认 LoRA 超参"出乎意料地好——我对比过手调配置，差距一直很小。

四个场景共同模式：Designer 和 Gallery 在"做出来时间"上赢，在"成本极致优化或大规模质量"上略输。 用它们做前 80%；只有有证据时才把后 20% 切到手写 DLC / EAS。

跳出来：从 Designer/Gallery 切到原始 PAI 资源

某个具体服务总有一天会从 GUI 长出来。我真跑过的迁移路径：

从 Designer 画布到原始 DLC + DataWorks。 痛苦的路是"用 PySpark 重写”；便宜的路是"导出 JSON，把每个 Python 节点替换成对等的 DLC 提交，MaxCompute 步骤保留为 DataWorks SQL 节点"。步骤：

  1. 从 Designer 导出工作流 JSON（右上角菜单）。
  2. 找出纯 MaxCompute 子图；把那些节点搬进 DataWorks 工作流当 ODPS SQL 节点（基本 1:1 转换）。
  3. 找出需要自定义逻辑的 Python 节点；每个写一份 train.py 和 TrainingJob SDK 提交（第三篇）。
  4. 在 DataWorks 工作流里用 “Shell 节点 + pai SDK 调用"触发 DLC 任务。
  5. DataWorks 版本干净跑 2 周后再删 Designer 工作流。

典型 Designer 画布的迁移要 1-2 天。收益是可观察性——DataWorks 给你每阶段日志、重试、合规告警，Designer 画布给不了。

从 Gallery 部署到手写 EAS。 这个我做得最多，“对的"顺序是：

  1. 在 Gallery 服务详情页点查看配置——它会显示底层 EAS 服务 spec（镜像、模型路径、命令、资源、扩缩容）。
  2. 把 spec 拷进仓库的 YAML。这就是一个普通 EAS 部署了，能版本控制。
  3. 用 YAML 起一个新 EAS 服务（独立名字）。验证它和 Gallery 的服务输出一致。
  4. 调优：自定义镜像把权重 baked 进去、指标换 concurrent_requests、加定时扩缩、服务组配镜像。每个改动一个 PR。
  5. 用服务组（第四篇套路）把流量从 Gallery 切到手写。
  6. 删 Gallery 部署。

模式：不要重写，导出后重构。Designer 和 Gallery 发出的配置都是合法 PAI 原语——它们不是魔法。当起步模板用，不要当成你被困在里面的终点。

反向迁移（手写 → Gallery）罕见但可能。 如果你给一个本质上是开源模型 + 标准配置的东西搭了自定义 EAS 部署，可以折回 Gallery 部署来缩减运维面。我做过一次：某个 DeepSeek 自定义部署 YAML 长到 600 行，简化回 Gallery 服务 + EAS 侧一个小代理（Gallery 表达不了的部分）。团队没人怀念那份 YAML。

系列总结

到这里五篇就结束了。回顾一下：

  • 第一篇 — PAI 是什么、各部分如何拼起来。
  • 第二篇 — DSW 做研发。
  • 第三篇 — DLC 做训练。
  • 第四篇 — EAS 做生产推理。
  • 第五篇 — Designer / Model Gallery 在 GUI 是对的场景。

姐妹系列阿里云百炼讲 DashScope、Qwen、万相、Qwen-TTS——也是托管 MaaS 层，下面坐的还是 PAI-EAS 的基础设施。多数团队两个都用：要自己跑模型、自己控 GPU 时用 PAI；要拿别人模型、用 API key 调时用百炼。按你需要控什么选，不是按哪个写在简历上更显眼。
