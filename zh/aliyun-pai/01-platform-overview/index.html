阿里云 PAI 实战（一）：平台总览与产品家族地图

基于官方文档梳理 2026 年阿里云 PAI 的四层服务架构、你真正会用到的五个子产品（DSW、DLC、EAS、Designer、Model Gallery），它们和 ECS、OSS 的关系，以及一套干净的账号/区域/工作空间初始化流程。

只要你的团队在阿里云上训练或部署模型，迟早要进 PAI 控制台。PAI 是个伞形产品，下面挂着真正干活的几个子产品——一�� Notebook 服务、一个分布式训练服务、一个模型推理服务，再加上几层 GUI/快速部署的封装。在某 AI Marketing 平台跑了一年半真实 LLM 业务之后，这个系列就是我希望第一次上线 EAS 之前有人塞给我的那本书。

第一篇是地理课，刻意不写代码，第二到第五篇才是深水区。这篇的目标是：当后面我说"DLC 任务"或者"EAS 端点"时，你脑子里立刻知道它属于哪一格。

PAI 是什么，又不是什么

按官方文档的原话，人工智能平台 PAI 是"阿里云面向 AI 全生命周期的开发平台，覆盖数据标注、模型开发、训练、部署"。pai.console.aliyun.com 是它的入口之一，但 PAI 本质上是一家族相关产品的集合，共享同一套账号体系、OSS 存储底座和同一个 Python SDK。

我用得最顺手的心智模型是这样：

  • PAI 是商场。
  • DSW、DLC、EAS、Designer、Model Gallery 是商场里的工坊。
  • ECS、OSS、NAS、CPFS 才是真正放硅片和字节的地方。PAI 帮你把它们编排起来。

官方"服务架构"主题把这套架构画成四层：

自下而上看。基础设施层是硅片——CPU、GPU、RDMA 高速网络，底下兜着 ACK Kubernetes。在它之上，灵骏（Lingjun）提供高密度 AI 算力，通用计算提供日常的 ECS 系 GPU 池。平台与工具层是你每天打交道的地方：PyTorch / Megatron / DeepSpeed，加上 PAI 自家的优化套件（TorchAcc / BladeLLM / EasyCkpt / AIMaster），再加上你看得见的产品（DSW / DLC / EAS / Designer / FeatureStore / iTAG）。应用层是 PAI 和阿里 MaaS 生态的接口（ModelScope、百炼/DashScope、Model Studio）。最上面的业务层是 PPT 用的行业方案。

为什么不直接租 ECS？因为 PAI 帮你预装好了 CUDA / PyTorch 镜像、挂好了 OSS、配好了监控面板，并且按秒计费。

你真正会碰的五个子产品

在生产环境跑了一年半，我只为这五个子产品掏过钱（直接对应官方"核心组件"表）：

组件文档原话什么时候伸手
DSW（交互式建模）云上 IDE，集成 Jupyter / VSCode / 终端，预置 PyTorch 和 TensorFlow 镜像，支持 GPU交互式开发、调试、小规模训练
DLC（深度学习容器）基于 K8s 的训练，支持 Megatron、DeepSpeed、PyTorch、TF、Slurm、Ray、MPI、XGBoost多卡 / 多机 SFT、预训练、大规模评测
EAS（弹性算法服务）在线推理，自动扩缩容、灰度发布、流量切分、镜像生产推理端点
Designer（可视化建模）140+ 内置算法组件、拖拽式工作流、可导出 JSON、可在 DataWorks 调度给非开发同学用的 ETL → 训练 → 评估流水线
Model Gallery（快速开始）封装了 DLC + EAS，对开源模型零代码部署 / 微调10 分钟评估一个 Qwen / DeepSpeed / Llama

另外还有 iTAG（数据标注）、PAI-Lingjun（超大集群）、PAI-Blade / BladeLLM（推理优化）、FeatureStore，但除非你做 >1000 卡预训练或者推荐系统，第一天可以先忽略。

PAI 和 ECS、OSS 的关系

只要你是从纯云主机背景过来的，这一块一定会被绊一下。三条铁律：

  1. PAI 永远不持有你的数据。 数据集、checkpoint、模型权重全部都在 OSS（要 POSIX 语义就用 NAS，要 HPC 吞吐就用 CPFS）。DSW 或 DLC 实例一死，没写到 OSS 的东西就没了。系统盘当 /tmp 看就行。
  2. PAI 持有计算。 你不自己开 GPU ECS。PAI 维护资源池，你说"给我一台 ecs.gn7i-c8g1.2xlarge"，按秒计费。
  3. PAI 共用你的账号但用自己的 RAM 角色。 给 PAI 授权访问 OSS，本质是给它挂上服务关联角色 AliyunPAIAccessingOSSRole，让 PAI 的算力不用长效 AK 就能读你的桶。这一步千万别跳——不然 DLC 任务会在 data_loader 阶段 403。

真实经验： 最常见的"PAI 坏了"工单全是 PAI 和 OSS 之间的权限问题。在调代码之前，先在 DSW 终端跑 oss ls oss://your-bucket/。失败就先修角色，不是改代码。

账号、区域、工作空间

按这个顺序准备三件事：

  1. 一个 aliyun.com 账号，做完实名认证——任何 GPU 资源都需要。国际站账号大部分区域可用，但杭州、上海、北京的 GPU 库存最稳。
  2. 一个区域。 选定一个就别换。PAI 资源、OSS、GPU ECS 都是区域内的，跨区流量贵又慢。国内生产我默认 cn-shanghai，国际默认 ap-southeast-1（新加坡）。
  3. 一个工作空间。 文档原话：工作空间是 PAI 的租户原语——配额、数据集、模型注册、IAM 都在它里面。一般至少建两个：dev 工作空间给人开 DSW 玩，prod 工作空间放 DLC 任务和 EAS 端点。跨工作空间授权略麻烦，但第一次有实习生不小心重启了生产端点的时候，这层隔离就值回票价了。

跟百炼一样，PAI 一切都有两条路。控制台适合查状态和一次性操作；SDK 才是你写到 CI 里的东西。

计费模型一段话讲完

文档列了五种计费方式：按量付费、包年包月、资源包（DSW 预付配额）、节省计划（承诺消费换折扣）、按推理时长付费（EAS Serverless，无空闲副本费）。DSW 按实例秒级计费，DLC 也是秒级、还有 spot/抢占式队列大约便宜 30-50%（前提你能 checkpoint），EAS 按副本秒级 + 极低的请求百万次费——自动扩缩配置里的"最小副本数"才是大头。Designer 和 Model Gallery 自己不收钱——它们调起的 DLC/EAS 资源正常计费。新账号有几百块的免费额度，跟完整跑完本系列绰绰有余。

有三件事值得画线。第一，每个 node 都显式带 compute.engine——SQL 形态的组件走 MaxCompute，Python / PyAlink 走 DLC，要流式就走 Flink。Designer 不会魔法地挑最便宜的，它按组件类型挑；任何自定义 Python 节点默认会拉一个 DLC pod，不管上游 MaxCompute 步骤跑了 30 秒还是 30 分钟，都按 pod 时长计费。第二，这份 JSON 是幂等和可 diff 的——我把它跟它依赖的 SQL 一起放 git，像 review 一份 Terraform plan 那样去 review 每次改动。第三，同一份 JSON 可以用 SDK 提交（pai.designer.submit_workflow(json_path)），这就是我在 CI 里跑流水线、不开控制台的方式。

藏在背后的升级路径：一旦你拿到了 JSON，“我想把这个迁到代码"就不再是重写。你可以从一份 Python DAG 生成它、给它打版本、做 lint、甚至 mock 输入做单元测试。画布只是这份文件的 UI，不是另一个工具。

Designer 比写代码快的场景，以及反过来的场景

我见过团队花一周写一套自定义 DLC + PySpark 流水线，做的事其实是 90 分钟的 Designer 画布搞定的。我也见过反过来——一位资深工程师非要在 Designer 里做实时特征工程，跟平台搏斗了两周。这条分界线比市场宣传清晰得多：

信号Designer 赢自定义代码赢
数据源在 MaxCompute？是（省掉跨系统拉数）否（Designer 最强的绑定就废了）
流水线主体是表格 ETL + 经典 ML？是否（LLM、RL、自研 CUDA 都该进 DLC）
责任人是数据分析或 BI 工程师？是否（工程师会嫌画布烦）
定时 + 重试够用？是否（自定义触发、多级审批 → 代码化的 Airflow / DataWorks）
单节点自定义 Python 超过 100 行？否（画布只是 .py 的壳）是（直接写脚本）
需要流式或亚秒级 SLA？否（Designer 是批形态）是

一个真实数字：我把一条推荐特征流水线迁到 Designer 上，跑 22 分钟，MaxCompute 抢占式，每天一次约 6 RMB；之前用 PySpark + DLC 4 节点版本要 41 分钟，每天约 38 RMB。便宜 6 倍的原因只有一个——数据从未离开 MaxCompute。反过来的迁移也发生过：曾经想用 Designer 跑 Qwen LoRA，但需要自定义训练循环，最后变成一个 Python 节点里塞 600 行胶水代码，调试痛苦至极，因为画布把堆栈藏在一个超小的 查看日志 链接后面。一个下午把它改写成普通 DLC 提交，团队幸福很多。

陷阱是把 Designer 看成永远对（PM 常这么干）或者永远错（资深工程师常这么干）。它是一个为特定问题类设计的工具——MaxCompute 规模的表格 ML 加上非开发人员负责——在那一类里它真的就是最优解。

Designer 背后那个被藏起来的 DLC

这件事我搞明白花的时间比应该多。在 Designer 画布上点运行、含一个 Python 节点，平台会在背后拉一个 DLC 任务——和你在第三篇用 SDK 提的那种一模一样。画布不会清楚地告诉你这件事，要从运维 → 最近运行 → 查看后端任务往下挖才能看到。这意味着：

  • Python 节点你按 DLC 价格付费，哪怕你的画布看上去"全 MaxCompute”。一个孤零零的 PyAlink 步骤跑在 1 节点 ecs.gn6i-c8g1.2xlarge 上，画布开着的时候差不多就是 4-5 RMB/h。
  • 容器镜像版本很重要。 Designer 把 Python 节点固定到一个具体镜像（通常是 pai-designer:latest 或者工作空间设置里的某个 tag）。如果你的自定义 Python import vllm==0.8.2，但镜像出厂带的是 0.6.0，运行会在 import 阶段挂掉，而那条堆栈不在画布里——只在底层 DLC 日志里。我曾经在这上面浪费过半天。
  • 配额是跨产品共享的。 DLC 配额本来就紧的话，Designer 跑起来需要 Python pod 时会饿到真的 DLC 训练任务，反过来也一样。它们共享同一个工作空间配额池。修法：在工作空间里给 Designer 单独开一个资源配额，封顶大概 4 CPU + 16 GB，免得一个跑飞的画布吃掉训练预算。

一旦你想清楚 Designer 只是一个发 MaxCompute SQL + DLC pod 提交的 UI，整个产品的神秘感就消散了——需要的时候你也能干净地绕开它。
