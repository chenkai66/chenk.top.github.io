阿里云 PAI 实战（三）：PAI-DLC——不用通宵刨坑的分布式训练

在 PAI-DLC 上提交真实多卡训练任务、看懂三种资源池（灵骏、通用、抢占）、用好 AIMaster + EasyCKPT，让一台抽风节点不会让你白干一天。

Read in English →

DSW Notebook 是给一个工程师 + 一张卡用的。一旦你需要 8 卡跨 2 机，或者训练时间超过你愿意守着浏览器的 8 小时，就该切到 DLC。DLC 是 PAI 在托管 K8s 集群之上做的任务提交前端：你描述要什么（镜像、命令、资源、数据挂载），DLC 调度 pod、跑完、留下日志、告诉你结果。文档叫它 Deep Learning Containers，我们口头就叫"DLC 任务"。

文档真正承诺了什么

DLC 概览里有四个特性我想画重点，因为它们都很重要：

  • 多样算力 — 灵骏 AI 智算、ECS、ECI、神龙裸金属、灵骏裸金属。支持异构混合调度。
  • 多种分布式任务 — 内置支持 Megatron、DeepSpeed、PyTorch DDP、TensorFlow PS/Worker、Slurm、Ray、MPI、XGBoost。不需要自建集群。
  • 容错 — AIMaster（看门狗）、EasyCKPT（异步 checkpoint）、SanityCheck（节点预检）、节点自愈。
  • 训练加速 — 内置加速框架，覆盖数据并行、流水线并行、算子拆分、自动并行策略探索、拓扑感知调度、优化通信库。

第一条和第三条，是 DLC 比自己租 GPU ECS 强的根本理由。

任务生命周期

DLC 任务从提交到完成走六个阶段：

其中两个阶段——调度落 pod 和 挂载 OSS / NAS——是几乎所有"我的任务卡 PENDING"工单的发源地。卡调度通常是资源池满了；卡挂载通常是存储 RAM 角色错了。和 DSW 一样，排查动作就是开个小 DSW，用同样的 OSS 挂载，跑一下 oss ls 看通不通。

选资源池

你提交到三种资源池中的一个。文档讲的是配额和账单，实际决策依据是"任务能容忍多少抖动"。

大多数团队的答案就是通用算力 + 按量付费。灵骏适合卡数 ≥ 8 且需要节点间 RDMA 的场景——文档说 RDMA 在灵骏上可配，能"加速节点间通信"，翻译成实操就是 NCCL AllReduce 比标准以太网快 5-10 倍。抢占式适合能干净 checkpoint 的任务，靠 EasyCKPT 大多数任务都能扛。

from pai.job import TrainingJob

job = TrainingJob(
    name="mnist-ddp",
    image_uri="dsw-registry-vpc.cn-shanghai.cr.aliyuncs.com/pai/"
              "modelscope:1.28.0-pytorch2.3.1tensorflow2.16.1-gpu-py311-cu121-ubuntu22.04",
    command=(
        "torchrun --nproc_per_node=2 --nnodes=$WORLD_SIZE "
        "--node_rank=$RANK --master_addr=$MASTER_ADDR "
        "--master_port=$MASTER_PORT /mnt/data/code/train_ddp.py"
    ),
    job_type="PyTorchJob",
    instance_count=4,                # 4 个 worker 节点
    instance_type="ecs.gn7i-c16g1.4xlarge",  # 每节点 2 × A10
    datasets={"train": "oss://your-bucket/datasets/mnist"},
    code_uri="oss://your-bucket/code/mnist-ddp.zip",
    output_uri="oss://your-bucket/runs/mnist-ddp/",
    fault_tolerance=True,            # 开 AIMaster
    enable_easyckpt=True,            # 异步 checkpoint
)
job.submit(wait=False)
print(job.id, job.status)

几个文档里要翻翻才能找到的细节：

  • $WORLD_SIZE、$RANK、$MASTER_ADDR、$MASTER_PORT 是 DLC 注入的。你不用自己做对端发现——DLC 会帮你处理对端发现，并在容器启动前写好这些环境变量。（参见 User Guide 的"内置环境变量"。）
  • fault_tolerance=True 会启动 AIMaster sidecar 看每个 worker。worker pod 死了，AIMaster 会标记它、申请替补，其他 worker 等而不是整任务挂掉。这是任务超过几小时时最重要的一个开关。
  • enable_easyckpt=True 把 torch.save 换成异步路径，写 OSS 不阻塞训练步。70B 模型上，checkpoint 从 3 分钟卡顿变成 ~10 秒重叠。
  • 镜像 URL 是分区域的。 dsw-registry-vpc.cn-shanghai.cr.aliyuncs.com 前缀只在上海 VPC 里能拉，跨区域要用对应前缀，否则拉镜像会超时。

常见失败和它们真正的原因

现象真实原因
任务 Pending >5 分钟资源池满或配额耗尽。换池或减小 instance_count。
启动时 cannot mount oss://...RAM 角色没挂 AliyunPAIAccessingOSSRole。在工作空间设置里重新挂。

成本现实

典型 7B SFT（4 × A10，6 小时）按量付费走通用算力，差不多上海一顿不错的晚饭钱。70B QLoRA（8 × A100 80G，12 小时）大概是杭州一个长周末的开销。抢占式如果任务能扛住每几小时被杀，能省 30-50%——配上 EasyCKPT 后基本都扛得住。

AIMaster 容错：到底重启了什么

文档说 AIMaster 做"容错"，但没说在 pod 层面到底意味着什么——而那才是真问题。下面是我跑了大约 200 个任务之后逆向出来的细节。

AIMaster 是一个 sidecar pod，挂在你训练 pod 旁边，有自己的 ServiceAccount，K8s 权限够它去标记、驱逐、重建 worker pod。它做三件事：

  1. 存活探测。 每 5-10 秒检查每个 worker 的健康端点（AIMaster 通过 side-mount 注入到容器里的小 HTTP server）。检查很浅——进程在、GPU 可见、/proc/[pid]/stack 没有 NCCL 死锁特征。它不深查训练进度；loss 变 NaN 它不会注意。
  2. 重启策略。 worker 连续三次没过存活检查，AIMaster 把 pod 标失败、向调度器申请在另一台健康节点上替补（避开刚故障的那台）、等新 pod 起来。其他 worker 在下一个集合通信操作（all_reduce 或 barrier）那里阻塞——它们不挂，它们等。替补就位后 AIMaster 发"恢复"信号，workers 重建进程组。
  3. Checkpoint 协调。 这一步多数人会漏。AIMaster 不帮你恢复模型状态——你的训练脚本来做。AIMaster 只保证新 worker 能找到最新 checkpoint。如果你开了 enable_easyckpt，它知道哪个是最新的可恢复 checkpoint，通过环境变量 EASYCKPT_RESUME_PATH 暴露。脚本里要 if os.path.exists(EASYCKPT_RESUME_PATH): load_from(EASYCKPT_RESUME_PATH) 起手。

哪些情况不会自动重启：

  • NaN loss / 训练发散。 AIMaster 不看你的 loss 曲线。脚本要自己识别并非零退出，AIMaster 才看见 worker 失败。
  • 训练中途 OOM。 单个 worker OOM，AIMaster 会重启它，但根本原因（一个超大 batch）会继续重现，除非你把 forward 包到 try/except 里跳过坏 batch。否则就是 AIMaster 重启 → OOM → 重启 → OOM 死循环，直到 max retry 用完。
  • 挂死但存活检查没踩到的 NCCL 集合通信。 有些 NCCL bug 让进程技术上还活着，但永远卡在 cudaStreamSynchronize。AIMaster 通过 /proc 栈检查能抓到一些，但抓不全。缓解办法：NCCL_TIMEOUT=1800（30 分钟），真挂的最终崩，AIMaster 才看见失败。
  • AIMaster sidecar 自己挂。 罕见，但我见过一次——某工作空间凭证过期。挂的时候没有容错。DLC 调度器会重启 AIMaster，但中间 worker 处于无人监管状态。

落地建议：每个 worker 设 max_retries=3，外加任务级 max_runtime 上限。AIMaster + EasyCKPT 能处理大部分抽风节点故障；预算上限保护你不被无限循环吃垮——那种应该人来查。

多机 NCCL：RDMA vs TCP，ring vs tree

多机训练里最大的性能杠杆，PAI 自己文档里几乎没讲。默认能用；调过之后通信重的任务每步耗时能减半。

RDMA vs TCP。 灵骏节点默认走 RDMA over Converged Ethernet（RoCE）；通用 GPU 节点走 TCP。一个真实的 8 节点 × 8 张 A100、跑 70B 全数据并行的任务，AllReduce 时差异：

通信方式AllReduce 1 GB每步 bubbletokens/秒/GPU
TCP（通用节点）380-450 ms~30%~85
RoCE（灵骏）35-50 ms~4%~160

灵骏比通用大约贵 1.5 倍（同 GPU 型号），但通信重的任务吞吐近乎 2 倍。8 卡以上的数据并行，灵骏物有所值。流水线并行那种跨 stage 通信稀疏的，性价比就接近持平。

PAI 上开 RDMA 是任务提交时的一个勾选（SDK 里 enable_rdma=True，控制台里"RDMA 加速"）。在容器里还要：

# 训练命令里、torchrun 之前：
export NCCL_IB_HCA=mlx5_0          # mellanox HCA
export NCCL_IB_GID_INDEX=3         # RoCE v2 默认
export NCCL_SOCKET_IFNAME=eth0     # 数据面网卡
export NCCL_DEBUG=INFO             # 一次性 debug，生产去掉

nccl_debug=INFO 显示 via NET/IB 就是真走 RDMA；via NET/Socket 意味着回落到 TCP——付了全价，性能减半。我见过这种情况：一台节点没分到 RDMA fabric——开工单换节点。

Ring vs tree allreduce。 NCCL 自动选，但能强制：

1

export NCCL_ALGO=Ring   # 或 Tree、Auto

中等 buffer（10 MB - 1 GB）ring 带宽最优，小 buffer（<1 MB，比如小模型梯度同步）tree 延迟最优。70B 级数据并行 ring 赢（也是默认）。8 节点高频小 all_reduce 的任务（比如 Megatron 流水线频繁微批同步），tree 能快 30%。诚实的答案：两个都试，各跑 100 步计时，留赢的。想孤立测，跑 nccl-tests。

拓扑感知。 灵骏节点知道自己的 NVSwitch 拓扑，PAI 调度器会尽量把 worker pod 放在相邻节点上（同 rack、同 spine）。调度器做不到的时候（资源紧张），就有跨 spine 流量，AllReduce 延迟上去。任务级 topology_constraint=spine 强制同 spine 落点，代价是可能调度延迟——4 节点以上的任务值。

抢占式 / Spot 的细节：节奏和通知窗口

抢占式省 30-50%，但运维细节市场材料里没有。跑了大约 40 个抢占式任务之后，下面是真正要知道的。

抢占预通知窗口。 PAI 在强杀前给 worker pod 发 SIGTERM，提前 30 秒。够你在 checkpoint 边界做一次紧急 torch.save，不够干净地中途 checkpoint。信号处理函数大致这样：

import signal, os, threading

_preempt = threading.Event()

def _on_term(signum, frame):
    _preempt.set()
    print("[preempt] 收到 SIGTERM，下一个步边界做 checkpoint")

signal.signal(signal.SIGTERM, _on_term)

# 训练循环里：
for step, batch in enumerate(loader):
    train_step(model, batch)
    if _preempt.is_set() or step % CKPT_EVERY == 0:
        save_easyckpt(model, optimizer, step)
        if _preempt.is_set():
            sys.exit(0)   # 干净退出；30 秒内 AIMaster 不会重试

Checkpoint 节奏。 没抢占时规则是"每 N 步存一次，N 取重做 N 步比 checkpoint 自身代价小"。有抢占就变成"每 M 步存一次，M 分钟接近平均抢占间隔"。2026 Q1 在 cn-shanghai 跑 70B SFT，平均抢占间隔 ~6 小时；满序列长度一步 ~12 秒；EasyCKPT 一次 ~10 秒。所以每 100 步存（20 分钟工作量）意味着抢占最多丢 20 分钟——可接受。不开 EasyCKPT，同样 100 步会让 GPU 阻塞 ~3 分钟存一次，有效吞吐掉 15%。

Spot 驱逐模式。 抢占不是随机的——和时段相关（工作时段更频繁，因为按需需求高）也和 GPU 型号相关（A100 80 GB 比 A10 抢得更狠）。我观察的范围：工作时段 A100 抢占率 15-30%/天，A10 接近 5%/天。A100 spot 上 >24 小时的长任务，预计被杀重启 3-5 次。

Retry 陷阱。 任务级 max_retries 一定要设（我用 5）。不设的话，一个表现不好的 spot 任务会反复在 pod 之间弹跳吃光预算，因为 instance_count × 抢占率 × 重启代价 > 原本省的。算一下：4 张 A100 spot 半价，盈亏平衡线大概是每 pod 每天 3 次抢占。再多就是花得比按需更多，换更少的真实训练时间。

分布式训练的数据切片模式

默认是 DistributedSampler，按 (rank, world_size) 分片，内存数据集没问题。一旦数据在 OSS 上或者比内存大，模式开始重要。

模式 1：基于 manifest 文件的索引切片。 一次性写一份排序的纯文本，每行一个路径，存 OSS。每个 worker 读自己的范围：

class ShardedManifestDataset(torch.utils.data.IterableDataset):
    def __init__(self, manifest_path, rank, world_size):
        self.manifest_path = manifest_path
        self.rank = rank
        self.world_size = world_size

    def __iter__(self):
        with open(self.manifest_path) as f:
            for i, line in enumerate(f):
                if i % self.world_size == self.rank:
                    yield self._load_one(line.strip())

便宜，无需协调，扩到几十亿样本也行。切片粒度按行，worker 之间互不重叠。

模式 2：OSS 上的 WebDataset shards。 把数据集预先切成 1-2 GB .tar（比如 images-{00000..00099}.tar），让 WebDataset 处理切片：

1
2
3
4

import webdataset as wds
url = "oss://your-bucket/datasets/coco-shards/images-{00000..00099}.tar"
ds = (wds.WebDataset(url, shardshuffle=True, nodesplitter=wds.split_by_node)
        .decode("pil").to_tuple("jpg", "json"))

split_by_node 做 worker 级 shard 分配。吞吐：OSS-FUSE 上每 worker ~600 MB/s，单文件读才 ~80 MB/s。任何要迭代 10 次以上的数据集，预切这一步都值。

模式 3：海量小文件用 CPFS。 数据是几百万小文件（基因组、时间序列）的话，manifest 和 WebDataset 都救不了——listing 和小读支配开销。CPFS $0.5-2/GB/月 贵，但 listdir 和小文件随机读比 OSS-FUSE 快一个数量级。<500 GB 的数据集通常划得来。

模式 4：恢复时动态重洗牌。 一个隐晦的 bug：worker 在 step 10000 死了重启，DistributedSampler(seed=42) 给它一样的洗牌顺序，但你忘了这个 epoch 跑到哪了。修法是有状态的 sampler，把 (epoch, sample_idx) 写进 checkpoint：

class ResumableSampler:
    def __init__(self, dataset, rank, world_size, seed=42):
        self.dataset = dataset
        self.rank = rank
        self.world_size = world_size
        self.seed = seed
        self.epoch = 0
        self.start_idx = 0   # 恢复时设

    def state_dict(self):
        return {"epoch": self.epoch, "start_idx": self.start_idx}

不做这个的话，每次重启都浪费 worker 已经完成的那部分 epoch。100 万样本数据集上的 7B SFT，每次重启大概浪费 5% 算力。
