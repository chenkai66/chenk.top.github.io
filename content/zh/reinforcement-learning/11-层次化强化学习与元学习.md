标准强化学习将每个问题视为一系列原子级别的决策：观察状态、选择动作、接收奖励，然后重复。这种方法在任务时间跨度较短、奖励密集时效果不错，但面对人类能轻松完成的任务时就会失效。“做早餐”显然不是单次决策，而是一棵由多个子任务组成的树——*煮咖啡*、*煎蛋*、*烤面包*、*装盘上桌*——每个子任务本身都是一套小型策略。**层次化强化学习（HRL）** 的核心思想是将宏动作视为一等公民，让智能体能在多个时间尺度上进行推理和行动。

标准强化学习的另一个短板是：每次遇到新任务都得从零开始学。现实中，一个会骑自行车的人只需一下午就能学会骑摩托车，而不是花上千万步环境交互。**元强化学习（Meta-RL）** 正是为解决这一问题而生：它在一个*任务分布*上进行训练，使得智能体面对未见过的新任务时，仅需几个回合（甚至只需一次循环神经网络的前向传播）就能快速适应。

本文将这两种思路统一起来：层次化带来时间维度的抽象，元学习带来任务维度的抽象。二者都能有效降低学习问题的维度，而当前前沿方法（如 FuN、HIRO、MAML、RL²）正积极地将它们结合起来。

## 一、层次化：Options 框架

### 为什么时序抽象至关重要

扁平策略在每个环境步都做一次决策，因此在长度为 $T$ 的任务中，信用分配路径长达 $T$，探索树的叶节点数量高达 $|\mathcal{A}|^T$，两者均呈指数增长。若引入平均长度为 $\bar k$ 的宏动作，决策次数可降至 $T/\bar k$，探索树规模也缩减至 $|\mathcal{O}|^{T/\bar k}$，其中 $\mathcal{O}$ 是一个通常很小的选项集合。下图对比了两种视角。

![Options 框架：时间线视图与三元组 (I, π, β)](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig1_options_framework.png)

除了渐近复杂度优势，层次化还带来三大实际好处：

1. **模块化** —— 在一个任务上学到的选项可迁移到其他任务（例如 *走到门口* 可泛化到各类导航问题）。
2. **可解释性** —— 高层策略足够小，便于检查；决策点也落在语义上有意义的关键位置。
3. **奖励塑形** —— 即使外部奖励稀疏，子目标也能提供密集的内在奖励。

### Option 三元组

根据 Sutton、Precup 与 Singh（1999）的定义，一个 **Option** 是一种时序扩展动作：
$$o = \langle \mathcal{I}, \pi_o, \beta \rangle,$$
其中 $\mathcal{I} \subseteq \mathcal{S}$ 是该选项可被启动的状态集合，$\pi_o(a \mid s)$ 是其内部策略，$\beta(s) \in [0, 1]$ 是终止概率。一旦引入选项，底层 MDP 就变成了**半马尔可夫决策过程（semi-MDP）**：高层策略 $\mu(o \mid s)$ 选择一个选项，该选项持续执行直至 $\beta$ 触发，此时才进行下一次高层决策。

### Intra-option Q-learning

最朴素的学习方式是等待选项终止后，用累积奖励进行更新。但这很浪费：一个选项可能运行数十步，中间所有转移都被丢弃。**Intra-option Q-learning**（Sutton 等，1999）则在*每一步*都更新 $Q(s, o)$，其关键洞察在于：同一个转移 $(s, a, r, s')$ 对所有“在状态 $s$ 下会选择动作 $a$”的选项都是有效的：
$$Q(s, o) \leftarrow Q(s, o) + \alpha \big[r + \gamma U(s', o) - Q(s, o)\big],$$
其中**延续值**为：
$$U(s', o) = (1 - \beta(s'))\, Q(s', o) + \beta(s')\, \max_{o'} Q(s', o').$$
延续值的设计十分巧妙：若选项继续执行，则保留其 Q 值；若在 $s'$ 终止，则交还控制权给高层策略，并取最优替代选项。

```python
import numpy as np
from collections import defaultdict

class IntraOptionQLearning:
    """离散选项集合上的 intra-option Q-learning。

    每一条原始转移都会更新：
    (1) 当前选项的高层 Q(s, o)；
    (2) 该选项的内部 Q 表，从而隐式修正其内部策略。
    """

    def __init__(self, env, options, alpha=0.5, gamma=0.99, epsilon=0.1):
        self.env, self.options = env, options
        self.alpha, self.gamma, self.epsilon = alpha, gamma, epsilon
        self.Q = defaultdict(lambda: np.zeros(len(options)))     # Q(s, o)
        self.Q_prim = defaultdict(lambda: np.zeros(env.n_actions))

    def _select_option(self, state):
        available = [i for i, o in enumerate(self.options)
                     if o.can_initiate(state)]
        if not available:
            return None
        if np.random.rand() < self.epsilon:
            return int(np.random.choice(available))
        q = self.Q[state][available]
        return int(available[np.argmax(q)])

    def train_episode(self, max_steps=1000):
        state = self.env.reset()
        total_reward = 0.0
        for _ in range(max_steps):
            oid = self._select_option(state)
            if oid is None:
                break
            option = self.options[oid]

            while not option.should_terminate(state):
                action = option.act(state, self.epsilon)
                next_state, reward, done = self.env.step(action)
                total_reward += reward

                # 计算延续值 U(s', o)
                if option.should_terminate(next_state):
                    U = np.max(self.Q[next_state])
                else:
                    U = self.Q[next_state][oid]

                # 更新高层 Q 值
                self.Q[state][oid] += self.alpha * (
                    reward + self.gamma * U - self.Q[state][oid])

                # 更新内部策略（基于原始动作的扁平 Q 表）
                td = (reward + self.gamma
                      * np.max(self.Q_prim[next_state])
                      - self.Q_prim[state][action])
                self.Q_prim[state][action] += self.alpha * td
                option.policy[state] = int(np.argmax(self.Q_prim[state]))

                state = next_state
                if done:
                    return total_reward
        return total_reward
```

在经典的 Four Rooms 基准测试中，使用四个手工设计的“穿过门口”选项后，intra-option Q-learning 的收敛速度比扁平 Q-learning 快 3–5 倍，且学到的选项能直接迁移到新目标位置。

### MAXQ：沿任务树分解值函数

Options 将层次结构隐含在选项集合中，而 **MAXQ**（Dietterich, 2000）则显式地将其表达出来。智能体被赋予一个子任务的有向无环图；对每个复合任务 $i$ 及其子任务 $a$，值函数分解为：
$$Q_i(s, a) = V_a(s) + C_i(s, a),$$
其中 $V_a(s)$ 是完成*子任务* $a$ 的价值，$C_i(s, a)$ 是**完成函数**——即子任务结束后继续完成父任务 $i$ 所带来的额外价值。由于 $V_a$ 仅依赖于 $a$，所有调用 $a$ 的父任务均可复用同一份 $V_a$，这正是样本效率提升的来源。

![MAXQ 任务层次：原语在多个父任务间共用](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig6_task_decomposition.png)

但 MAXQ 的代价是只保证**递归最优性**而非全局最优性：其策略是在给定任务分解下的最优解。若任务图无法表达真正的最优策略，MAXQ 便无法找到它。

## 二、Feudal RL：用连续子目标替代离散选项

![Feudal RL:manager 输出隐式目标,worker 基于该目标行动。](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/en/reinforcement-learning/11-层次化强化学习与元学习/fig11_hierarchy.png)

离散选项难以扩展：在连续控制或像素输入场景中，我们无法手工枚举出有用的选项集合。**Feudal Networks**（FuN，Vezhnevets 等，2017）和 **HIRO**（Nachum 等，2018）提出用高层 Manager 输出的*连续目标向量*取代离散选项。

![Feudal RL：Manager 每 c 步设一次目标，Worker 每步执行](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig2_feudal_architecture.png)

Manager 每隔 $c$ 步（FuN 中 $c = 10$）根据状态 $s_t$ 输出一个目标 $g_t \in \mathbb{R}^d$。Worker 是一个目标条件策略 $\pi_\phi(a \mid s, g)$，通过朝目标方向移动获得**内在奖励**：
$$r^{\text{int}}_t = \cos\!\big(s_{t+c} - s_t,\, g_t\big),$$
而 Manager 则基于环境的*外部奖励*进行训练。这种解耦设计正是 Feudal 架构的优势所在：Worker 在稠密、几何化的内在奖励上学习运动控制，Manager 则专注于长时程信用分配，有效时序长度缩短为 $T/c$。

### HIRO 的子目标重标注技巧

Feudal 训练面临“鸡生蛋”问题：Worker 尚在学习，行为非平稳，导致回放缓冲区中存储的旧目标已无法反映其当前能力。HIRO 通过**子目标重标注**解决此问题：采样转移 $(s_t, g_t, a_{t:t+c}, s_{t+c})$ 时，将 $g_t$ 替换为最能解释实际动作序列的目标：
$$\tilde g_t = \arg\max_{g} \log \pi_\phi(a_{t:t+c} \mid s_{t:t+c},\, g).$$
此举确保 Worker 的训练数据始终与其当前参数保持一致，显著提升了 Manager 的离线学习稳定性。

## 三、目标条件 RL 与 HER

目标条件策略 $\pi(a \mid s, g)$ 值得单独讨论，因为它是连接层次化与元学习的桥梁：同一网络可追求多种目标，而目标 $g$ 仅是一个额外输入。

![目标条件策略与 4 条来自同一网络的轨迹](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig3_goal_conditioned.png)

经典形式是 **通用值函数逼近器**（UVFA，Schaul 等，2015）：学习 $V(s, g)$ 或 $Q(s, a, g)$ 而非传统的 $V(s)$、$Q(s, a)$。但若无额外技巧，UVFA 在稀疏奖励下表现极差——大多数目标在探索中从未达成，奖励信号几乎为零。

**事后经验回放**（HER，Andrychowicz 等，2017）是标准解决方案：当一个回合未能达成目标 $g$ 时，将轨迹重新标记为它*实际达成*的目标（通常是最终状态），从而将失败转化为另一任务的成功示范：
$$(s_t, a_t, r, s_{t+1}, g) \;\longrightarrow\; (s_t, a_t, r', s_{t+1},\, g' = s_T).$$
结合离线策略算法（DDPG、SAC），HER 将稀疏奖励下的目标达成任务从“几乎不可能”变为“常规操作”。

## 四、Meta-RL：学会学习

![强化学习（十一）：层次化强化学习与元学习 — visual](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/en/reinforcement-learning/11-hierarchical-and-meta-rl/illustration_2.png)

Meta-RL 不再假设单一 MDP，而是考虑一个*任务分布* $p(\mathcal{T})$。元训练阶段，智能体接触大量任务 $\mathcal{T}_i \sim p(\mathcal{T})$；元测试阶段，面对新任务时需以最少交互完成适应。

![任务分布与新任务上的适应曲线](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig5_meta_rl_distribution.png)

主流方法分为两类：

- **基于优化**（MAML、Reptile、ANIL）：适应 = 在测试任务上执行几步梯度更新。
- **基于上下文 / 循环**（RL²、PEARL）：适应 = 数据流入时更新 RNN 隐藏状态，测试时无需梯度计算。

### MAML —— 学习一个良好的初始化

**模型无关元学习**（MAML，Finn 等，2017）的目标是找到参数 $\theta$，使得在任意任务 $\mathcal{T}_i$ 上执行几步内层 SGD 后即可获得高性能：
$$
\theta_i' = \theta - \alpha \nabla_\theta \mathcal{L}_{\mathcal{T}_i}(\theta),
\qquad
\theta \leftarrow \theta - \beta \nabla_\theta \sum_{i} \mathcal{L}_{\mathcal{T}_i}(\theta_i').
$$
外层梯度需穿过内层更新反向传播，因此包含 Hessian 项 $\nabla^2 \mathcal{L}$，这也是 MAML 计算昂贵的原因。**FOMAML** 直接忽略二阶项，实证表明其每次外层更新快约 $10\times$，最终回报损失不到 5%。

![MAML 在参数空间中的几何意义与两层循环算法](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig4_maml_inner_outer_loop.png)

左图提供了最直观的理解：元训练并非寻找对单个任务最优的初始化，而是寻找一个“甜区”——从这里出发，一步梯度即可接近各任务的特定最优解。

```python
import torch
import torch.nn as nn

class GaussianPolicy(nn.Module):
    """连续控制下的对角高斯策略"""

    def __init__(self, state_dim=2, action_dim=2, hidden=64):
        super().__init__()
        self.trunk = nn.Sequential(
            nn.Linear(state_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, hidden), nn.ReLU(),
        )
        self.mean = nn.Linear(hidden, action_dim)
        self.log_std = nn.Parameter(torch.zeros(action_dim))

    def sample(self, state):
        h = self.trunk(state)
        mean = torch.tanh(self.mean(h))
        std = self.log_std.exp().clamp(1e-3, 1.0)
        dist = torch.distributions.Normal(mean, std)
        a = dist.rsample()
        return a, dist.log_prob(a).sum(-1)

class MAML:
    """REINFORCE 风格策略梯度的一阶或二阶 MAML"""

    def __init__(self, policy, inner_lr=0.1, outer_lr=1e-3,
                 first_order=False):
        self.policy = policy
        self.inner_lr = inner_lr
        self.first_order = first_order
        self.opt = torch.optim.Adam(policy.parameters(), lr=outer_lr)

    def _rollout(self, env, max_steps=50):
        state = env.reset()
        rewards, log_probs = [], []
        for _ in range(max_steps):
            s = torch.as_tensor(state, dtype=torch.float32).unsqueeze(0)
            a, lp = self.policy.sample(s)
            state, r, done = env.step(a.detach().numpy()[0])
            rewards.append(r)
            log_probs.append(lp)
            if done:
                break
        return rewards, log_probs

    @staticmethod
    def _returns(rewards, gamma=0.99):
        G, out = 0.0, []
        for r in reversed(rewards):
            G = r + gamma * G
            out.insert(0, G)
        out = torch.as_tensor(out, dtype=torch.float32)
        return (out - out.mean()) / (out.std() + 1e-8)

    def _loss(self, rewards, log_probs):
        returns = self._returns(rewards)
        return -(torch.stack(log_probs) * returns).mean()

    def adapt(self, env):
        """在 env 上做一步内层更新，返回适应后的参数列表"""
        rewards, log_probs = self._rollout(env)
        loss = self._loss(rewards, log_probs)
        grads = torch.autograd.grad(
            loss, self.policy.parameters(),
            create_graph=not self.first_order,
        )
        return [p - self.inner_lr * g
                for p, g in zip(self.policy.parameters(), grads)]

    def meta_step(self, task_envs):
        meta_loss = 0.0
        for env in task_envs:
            adapted = self.adapt(env)
            # 暂存原参数，将适应后的参数装入网络进行评估前向
            backup = [p.detach().clone() for p in self.policy.parameters()]
            for p, a in zip(self.policy.parameters(), adapted):
                p.data = a.data
            rewards, log_probs = self._rollout(env)
            meta_loss += self._loss(rewards, log_probs)
            for p, b in zip(self.policy.parameters(), backup):
                p.data = b
        meta_loss /= len(task_envs)

        self.opt.zero_grad()
        meta_loss.backward()
        self.opt.step()
        return meta_loss.item()
```

上述实现通过替换权重后执行前向传播完成元评估，而非使用函数式前向传播。这是最简洁易读的版本，对小型网络已足够。若用于研究级 MAML，建议使用 `higher` 或 `torch.func.functional_call` 等函数式 API。

### RL² —— 将算法折叠进 RNN

**RL²**（Duan 等，2016；Wang 等，2016）另辟蹊径：元测试时冻结参数，所有适应均由 RNN 隐藏状态完成。循环策略接收增强输入：
$$x_t = (s_t,\, a_{t-1},\, r_{t-1},\, d_{t-1}),$$
即状态加上*前一步的动作、奖励和终止标志*。在元试验的多个回合中，隐藏状态 $h_t$ 不断累积当前任务信息——本质上隐式执行贝叶斯信念更新。元训练时，外层优化器（PPO 或 A2C）调整 RNN 权重，使这套“内置 RL 算法”在 $p(\mathcal{T})$ 上具备高效样本利用率。

![RL² 在跨多个回合的元试验上的展开](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/reinforcement-learning/11-层次化强化学习与元学习/fig7_rl_squared.png)

RL² 有两大优势：(i) 测试时无需梯度计算，每步适应仅需一次前向传播；(ii) 适应过程是学出来的，原则上可在训练分布上超越任何手工设计的优化器。代价是元训练时信用分配困难——梯度需流经横跨多回合的数百个循环步骤。

## 五、常见问题

**Options 框架为何能加速学习？**  
三重效应叠加：(i) 有效时序长度从 $T$ 缩至约 $T/\bar k$；(ii) 高层分支因子 $|\mathcal{O}|$ 通常远小于 $|\mathcal{A}|$；(iii) intra-option 学习使每条原始转移都能贡献给所有兼容选项的价值，而非仅当前控制的那个。

**层次化最优与全局最优有何区别？**  
层次化最优策略是在给定分解（选项集或任务图）下的最优解；全局最优策略是底层扁平 MDP 上的最优解。只要分解无法表达真正最优策略，二者就会偏离——例如，一个至少运行 10 步的选项无法实现每 3 步切换行为的策略。

**MAML 为何需要二阶梯度？FOMAML 损失有多大？**  
元损失为 $\mathcal{L}_{\mathcal{T}_i}(\theta_i')$，其中 $\theta_i' = \theta - \alpha \nabla_\theta \mathcal{L}_{\mathcal{T}_i}(\theta)$。对 $\theta$ 求导得 $(I - \alpha \nabla^2_\theta \mathcal{L}_{\mathcal{T}_i})\, \nabla_{\theta_i'} \mathcal{L}_{\mathcal{T}_i}(\theta_i')$，包含 Hessian。FOMAML 忽略 $-\alpha \nabla^2 \mathcal{L}$ 项，将 $\theta_i'$ 视为 stop-gradient。原论文报告，在标准小样本基准上性能损失 <5%。

**何时用 MAML，何时用 RL²？**  
若适应时可计算梯度且任务分布较广（固定策略无法覆盖所有任务），选 MAML；若任务族较窄、偏向利用（如 arms 动态变化的多臂老虎机），或测试时无法计算梯度，选 RL²。PEARL 通过独立编码器推断任务嵌入，常是不错的折中。

**HER 为何仅适用于离线策略算法？**  
HER 修改了转移对应的目标，违反了在线策略算法的假设：生成数据的动作分布依赖于原始目标，而非重标注后的目标。离线策略算法（DQN、DDPG、SAC）只需在更新时知道 $\pi(a \mid s, g')$，而该值可计算，因此重标注无害。

## The Options Framework (Sutton, Precup & Singh)

1999 年 Sutton、Precup 和 Singh 的那篇论文，我反复读了好几遍。它把“先执行一阵子，再做决策”这个直觉，形式化得最干净。Option 是个三元组 $\langle I, \pi, \beta \rangle$：启动集 $I \subseteq \mathcal{S}$ 规定 option 能在哪些状态触发；内部策略 $\pi(a \mid s)$ 负责在 option 运行期间挑选底层动作；终止函数 $\beta(s) \in [0,1]$ 给出每一步把控制权交还给高层策略的概率。

这套框架能立住脚，靠的是一个结构性结论：只要 agent 锁定了一组固定的 option，它在 option 层面看到的世界依然是马尔可夫奖励过程，只是动作变成了扩展动作。严格来说，这是个 *semi-MDP*。状态转移会消耗随机数量的底层步数 $\tau$，累积奖励的折扣因子变成 $\gamma^\tau$，而不是 $\gamma$。这意味着底层那些我信得过的算法（Q-learning, SARSA, actor-critic）可以直接平移到 option 层面，唯一要改的就是处理可变时长的折扣。

得停下来想想 semi-MDP 到底带来了什么，又没带来什么。它带来了收敛性证明：SMDP Q-learning 的收敛条件和普通 Q-learning 的 Robbins-Monro 条件一样，只是作用在 option 层的转移核 $P(s', \tau \mid s, o)$ 上。它*不*保证全局最优。给定一组 option，能表达出的最优策略通常比纯底层的最优策略差，有时差很多。Sutton 的原论文特意区分了这一点，管它叫 *hierarchically optimal*。我见过不少工程师把两者混为一谈，然后花好几周死磕“为什么 option agent 的收敛平台比 DQN 低”。

我实际落地的 Bellman 方程长这样：
$$Q^\Omega(s, o) = r(s,o) + \gamma \sum_{s'} P(s'\mid s,o)\big[(1-\beta(s'))\, Q^\Omega(s',o) + \beta(s')\, V^\Omega(s')\big].$$
混合项才是核心。概率 $1-\beta(s')$ 下，当前 option 继续执行，用 $Q^\Omega(s', o)$ 做 bootstrap；概率 $\beta(s')$ 下，重新选择 option，用 $V^\Omega(s') = \max_{o'} Q^\Omega(s', o')$ 做 bootstrap。我第一次写的时候踩了坑，无条件用了 $V^\Omega$ 做 bootstrap，结果 agent 学会了一步就终止所有 option。

```python
import torch
import torch.nn as nn

class OptionAgent(nn.Module):
    """离散 option 共享主干网络：每个 option 独立维护 pi, beta 和 Q^Omega。"""

    def __init__(self, state_dim, n_actions, n_options, hidden=64):
        super().__init__()
        self.n_options = n_options
        self.trunk = nn.Sequential(
            nn.Linear(state_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, hidden), nn.ReLU(),
        )
        self.pi = nn.Linear(hidden, n_options * n_actions)
        self.beta = nn.Linear(hidden, n_options)
        self.q_omega = nn.Linear(hidden, n_options)
        self.n_actions = n_actions

    def forward(self, s):
        h = self.trunk(s)
        pi_logits = self.pi(h).view(-1, self.n_options, self.n_actions)
        beta = torch.sigmoid(self.beta(h))
        q = self.q_omega(h)
        return pi_logits, beta, q

    def act(self, s, current_option):
        pi_logits, beta, q = self.forward(s)
        # 按终止概率采样是否结束当前 option
        terminate = torch.bernoulli(beta[:, current_option]).bool()
        # 终止则按 Q 值重选，否则保持原 option
        option = q.argmax(-1) if terminate else torch.tensor([current_option])
        action = torch.distributions.Categorical(
            logits=pi_logits[:, option.item()]
        ).sample()
        return action.item(), option.item(), terminate.item()
```

我在经典的 Four Rooms 网格上跑过这套代码。预设了四个手写 hallway option，每个负责导航到一个门口。Option agent 大概 30K 步环境交互就能拿到 90% 的最优回报。同样的网格，纯表格 Q-learning 得跑 200K 步以上才能追平。优势来自分支因子：option 层的搜索树宽度没变，但深度直接降了一个数量级。

实现上有两个坑，各坑了我一天。第一个是 $\epsilon$ 衰减策略。我一开始按底层 agent 的节奏衰减高层策略的 $\epsilon$。但高层策略每个 episode 做的决策少十倍，底层都收敛了，高层还在随机乱选。解决办法很简单：按高层*决策次数*排 exploration 进度表，别按环境步数算。第二个坑在终止函数 $\beta$。Option-critic 变体（Bacon 2017）里 $\beta$ 是可学习的。我试过一开始就让它学，结果极不稳定。agent 会迅速收敛到退化解：所有 option 一步就停，直接退化成普通 Q-learning。前 10K 步冻结 $\beta$，或者给终止动作加一点“思考成本”，就能稳住。

## Feudal RL: Goal-Conditioned Hierarchies (FuN, HIRO)

离散 option 逼着人提前枚举技能。网格世界还能凑合，连续控制直接抓瞎。Feudal 路线换了个思路：Manager 网络输出一个 *goal vector*，Worker 网络学着去达成它。FuN（Vezhnevets 2017）和 HIRO（Nachum 2018）细节不同，骨架一致：Manager 跑得慢，Worker 跑得快，隐空间 goal 是两者之间唯一的契约。

具体跑起来是这样：每隔 $c$ 步（FuN 用 $c=10$，HIRO 看任务用 $c=10$ 到 $50$），Manager 观察 $s_t$，吐出 $g_t \in \mathbb{R}^d$。Worker 在 $\tau \in [t, t+c)$ 期间以 $(s_\tau, g_t)$ 为条件，只要把状态往 $g_t$ 的方向推，就能拿奖励。Manager 只看 *extrinsic* reward，Worker 只看 *intrinsic* reward。解耦是这套架构能训出来的关键。Worker 拿到密集奖励，Manager 的规划视野直接缩短 $c$ 倍。

让 off-policy Manager 训练能跑通的核心技巧，是 HIRO 的 goal relabelling。Worker 是非平稳的。采集数据时合理的 $g_t$，放到现在的 Worker 身上可能根本够不着。从 replay buffer 采样时，把 $g_t$ 替换成最能解释 Worker 实际动作序列的那个 goal：$\tilde g_t = \arg\max_g \log \pi_\phi(a_{t:t+c} \mid s_{t:t+c}, g)$。实操时，我只在一个小候选集里做 argmax（原始 $g_t$、经验状态差 $s_{t+c} - s_t$，再加上周围采的八个高斯样本）。计算开销几乎为零，但在 AntMaze 上样本效率直接翻了 3 到 5 倍。

```python
class FeudalAgent(nn.Module):
    """Manager-Worker 架构：在学出的隐空间中使用连续 goal。"""

    def __init__(self, state_dim, action_dim, goal_dim=16, c=10, hidden=128):
        super().__init__()
        self.c = c
        self.goal_dim = goal_dim
        # Manager：低频运行，输出 goal
        self.manager = nn.Sequential(
            nn.Linear(state_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, goal_dim),
        )
        # 状态到隐空间的映射（与 manager target 共享）
        self.phi = nn.Sequential(
            nn.Linear(state_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, goal_dim),
        )
        # Worker：高频运行，以 goal 为条件
        self.worker = nn.Sequential(
            nn.Linear(state_dim + goal_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, action_dim),
        )

    def manager_step(self, s):
        return self.manager(s)

    def worker_step(self, s, g):
        return self.worker(torch.cat([s, g], dim=-1))

    def intrinsic_reward(self, s_t, s_tpc, g_t):
        # 隐空间内的余弦相似度奖励
        z_diff = self.phi(s_tpc) - self.phi(s_t)
        return torch.nn.functional.cosine_similarity(z_diff, g_t, dim=-1)
```

在稀疏终点奖励的 AntMaze-Medium 上，带 HIRO relabelling 的 FeudalAgent 跑 1M 步，成功率突破 80%。同任务的纯 SAC baseline 卡在 5% 左右。蚂蚁连第一个 U 型弯都过不去，终点奖励信号根本传不回来。隐空间方向 $g_t$ 提供的 intrinsic reward，硬是把长程 credit assignment 的问题扛下来了，纯 TD-bootstrapping 根本做不到。

有个细节我一开始没留意。如果隐空间映射 $\phi$ 和策略一起端到端训练，Manager 和 encoder 会互相迎合，goal space 直接坍缩。FuN 的解法是给 $\phi$ 挂一个 *target* network，用 Polyak averaging 慢速更新。这样 Manager 指向的 goal space 基本保持平稳。HIRO 更干脆，直接挑一部分状态坐标（通常是 agent 的 xy 位置），让 $\phi$ 变成恒等映射。两条路都能走通。先验知道相关子空间时，恒等映射样本效率更高；不知道的时候，学出来的隐空间更稳。

总结

## MAML for RL: Meta-Gradient Through Trajectories

监督学习里的 MAML 已经够绕了，放到 RL 里更头疼。内环 loss 本身就是期望回报的随机估计量。目标函数倒是很紧凑：
$$\theta^* = \arg\min_\theta\, \mathbb{E}_{\mathcal{T} \sim p(\mathcal{T})}\big[ \mathcal{L}_\mathcal{T}(\theta'_\mathcal{T}) \big], \qquad \theta'_\mathcal{T} = \theta - \alpha \nabla_\theta \mathcal{L}_\mathcal{T}(\theta).$$
内环更新 $\theta'_\mathcal{T}$ 就是标准的 policy-gradient 步骤（REINFORCE, PPO 都行）。外环更新要对*适配前*的参数 $\theta$ 求导，计算适配后的 loss 梯度。这要求梯度必须穿过内环步骤。也就是说，内环梯度 $\nabla_\theta \mathcal{L}_\mathcal{T}(\theta)$ 必须留在 autograd 计算图里，不能 detach。

PyTorch 里实现就靠一个参数：`torch.autograd.grad(..., create_graph=True)`。漏掉它，代码会静默退化成 FOMAML。有时候你确实想要 FOMAML，但那已经不是论文里的 MAML 了。

```python
import torch
import torch.nn as nn

class BanditPolicy(nn.Module):
    def __init__(self, n_arms=5, hidden=32):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(1, hidden), nn.ReLU(),
            nn.Linear(hidden, n_arms),
        )

    def logits(self, params=None):
        x = torch.zeros(1, 1)
        if params is None:
            return self.net(x).squeeze(0)
        # 使用外部传入参数进行函数式前向传播
        h = torch.relu(torch.nn.functional.linear(x, params[0], params[1]))
        return torch.nn.functional.linear(h, params[2], params[3]).squeeze(0)

def reinforce_loss(logits, action, reward):
    logp = torch.distributions.Categorical(logits=logits).log_prob(action)
    return -(logp * reward)

def maml_step(policy, tasks, inner_lr=0.1, outer_lr=1e-2, K=5):
    outer_loss = 0.0
    for task in tasks:                       # task 即 bandit 奖励向量
        # --- 内环：K-shot REINFORCE，保留二阶计算图
        params = list(policy.parameters())
        for _ in range(K):
            logits = policy.logits()
            a = torch.distributions.Categorical(logits=logits).sample()
            r = task[a.item()]
            loss = reinforce_loss(logits, a, torch.tensor(r))
            grads = torch.autograd.grad(loss, params, create_graph=True)
            params = [p - inner_lr * g for p, g in zip(params, grads)]
        # --- 在相同任务上评估适配后的策略
        logits = policy.logits(params)
        a = torch.distributions.Categorical(logits=logits).sample()
        r = task[a.item()]
        outer_loss = outer_loss + reinforce_loss(logits, a, torch.tensor(r))
    outer_loss = outer_loss / len(tasks)
    outer_loss.backward()
    with torch.no_grad():
        for p in policy.parameters():
            p -= outer_lr * p.grad
            p.grad.zero_()
    return outer_loss.item()
```

我在一个 5-task bandit 套件上测过。每个任务对应 5 个臂的不同奖励向量（从 $\mathcal{U}[0,1]^5$ 采样）。Meta-training 跑 2000 个外环步骤后，在未见过的任务上只做一步内环 SGD，agent 就能拿到最优臂 80% 的期望奖励。同样只走一步内环，从零开始的 REINFORCE 只能摸到 30%，勉强比均匀随机 20% 的 baseline 强一点。Meta-learned 初始化确实在干活，不是单纯停在任务平均均值附近摆烂。

`create_graph=True` 触发的 Hessian-vector product，就是 MAML 出了名吃算力的根源。在这个 bandit 实验里，二阶导传递让每个外环步骤的耗时比 FOMAML 多了 2.5 倍。换到真实的 Mujoco MAML 实验（locomotion，内环展开 20 步），耗时直接拉到 8 到 10 倍。值不值看任务族。Bandit 套件上 FOMAML 最终回报一样。Locomotion 上 Finn 的原论文报告 MAML 领先 5% 到 15%。

总结

## RL² and PEARL: Context-Conditioned Meta-RL

MAML 在测试期依赖梯度。梯度便宜时很爽，贵的时候直接卡脖子。Meta-RL 的另一条分支干脆丢掉测试期梯度，把适配过程塞进 RNN 的前向传播里。RL²（Duan 2016）是标准做法：拿一个循环策略，每一步喂给它 $(s_t, a_{t-1}, r_{t-1}, d_{t-1})$，并且*在同一个 meta-trial 内的不同 episode 之间，绝对不重置 hidden state*。Hidden state 最终会编码出 agent 对当前任务的后验分布。外层优化器（PPO 或 A2C）负责调整循环权重，让这种隐式后验更新在 $p(\mathcal{T})$ 上尽量省样本。

PEARL（Rakelly 2019）把后验分布显式化了。单独搞一个 encoder $q_\phi(z \mid \tau_{1:k})$，吃进当前任务的几条轨迹，输出任务隐变量 $z$ 的高斯分布。策略变成 $\pi(a \mid s, z)$。训练时 encoder 走 VAE 风格的 ELBO，策略走 off-policy SAC。优势是样本效率。Mujoco meta-benchmark 上，PEARL 拿到同等回报，环境交互步数比 RL² 少 20 到 100 倍。代价是泛化边界。PEARL 只能适配和训练分布相似的任务，encoder 推不出真正没见过的新结构。

Encoder 设计比预想中更关键。朴素的 permutation-invariant encoder（对单步 embedding 做 mean-pool）在简单 meta-bandit 上能跑，放到 locomotion 就崩。动作序列的顺序本身带信息。PEARL 原论文用了高斯乘积分解，天生置换不变。我在对顺序敏感的任务上换过小型 Transformer encoder，效果更好。代价是参数量变大，ELBO 也没那么干净。

```python
class RL2Policy(nn.Module):
    """GRU 策略：输入 (当前状态, 上一步动作, 上一步奖励, 终止标志)。"""

    def __init__(self, state_dim, n_actions, hidden=64):
        super().__init__()
        self.n_actions = n_actions
        in_dim = state_dim + n_actions + 1 + 1
        self.gru = nn.GRUCell(in_dim, hidden)
        self.pi = nn.Linear(hidden, n_actions)
        self.v = nn.Linear(hidden, 1)
        self.hidden = hidden

    def init_hidden(self, batch=1):
        return torch.zeros(batch, self.hidden)

    def step(self, s, prev_a, prev_r, prev_d, h):
        a_oh = torch.nn.functional.one_hot(
            prev_a, num_classes=self.n_actions
        ).float()
        # 拼接状态、历史动作、奖励和 done 标志
        x = torch.cat([s, a_oh, prev_r.unsqueeze(-1), prev_d.unsqueeze(-1)], -1)
        h = self.gru(x, h)
        return self.pi(h), self.v(h).squeeze(-1), h
```

选型时的权衡很明确。MAML 能适配*新*任务，只要任务族里存在内环更新路径就行。RL² 和 PEARL 只能适配训练分布内的任务。任务族窄，预算只够“一次 rollout 摸清情况”，直接上 RL²。任务族宽，预算够跑几步 SGD，选 MAML。

训 RL² 有个实操痛点。跨多个 episode、几百步的循环反向传播极其暴力。我的做法是 BPTT 截断到 200 步，把每个 meta-trial 当成一条 PPO 轨迹处理。长程 credit assignment 会丢一点，但梯度范数能稳住。PEARL 完全避开了这个坑。Encoder 只在短 context window（通常 100 步转移）上训，策略走 off-policy SAC。这也是它样本效率碾压的原因之一，根本不用反向传播穿过上千步的循环网络。

## 终身/持续强化学习：跳出静态任务集

Meta-RL 其实还在作弊。它假设任务分布 $p(\mathcal{T})$ 固定不变，训练集和测试集也切得干干净净。Lifelong RL 直接抛弃了这套假设。智能体必须一直跑下去，任务随时可能切换，也没有 oracle 跳出来提示“任务结束，开始重新适应”。这些挑战直接继承自监督持续学习，最核心的就是灾难性遗忘。RL 还有自己的专属痛点：数据分布强依赖策略本身。任务一旦漂移，数据分布跟着变，非平稳性直接叠加。

我实际落地过的缓解手段不多。用 Elastic Weight Consolidation 惩罚旧任务关键参数的偏移。用 replay buffer 分层留存历史经验。还有 PackNet 风格的 adapter 模块，按任务动态扩容。最接地气的是 CLEAR (Rolnick 2019)。它把历史轨迹的 behaviour-cloning loss 和当前数据的 V-trace off-policy correction 绑在一起。不需要任务边界信号也能稳定收敛。

```python
def clear_update(policy, current_batch, replay_batch,
                 distill_coef=0.5, opt=None):
    # 当前数据：标准策略梯度（REINFORCE 风格，简写）
    s, a, R = current_batch
    logp = policy.log_prob(s, a)
    pg_loss = -(logp * R).mean()

    # 回放数据：对存储的旧策略 logits 做行为克隆
    s_r, old_logits = replay_batch
    new_logits = policy.logits(s_r)
    distill = torch.nn.functional.kl_div(
        torch.log_softmax(new_logits, -1),
        torch.softmax(old_logits, -1),
        reduction="batchmean",
    )

    loss = pg_loss + distill_coef * distill
    opt.zero_grad(); loss.backward(); opt.step()
    return pg_loss.item(), distill.item()
```

真正的开放难题是检测。现在公开的 continual-RL benchmark 大多直接下发 task ID 或任务边界。智能体的“能力”无非是把流量路由到对应的 adapter。撤掉 oracle，性能直接崩盘。原因很直接：在依赖策略的 reward 序列上做 change-point detection，难度极高。我目前没有好解法，学术界也一样。这个领域下一篇值得读的工作，必须能在无预警任务切换下稳住性能，且不回退到简单的 oracle 监督设定。

第二个老毛病：replay buffer 无限膨胀。CLEAR 原论文在 Atari 上塞了 5000 万条 transition。研究集群跑起来很轻松，放到实体机器人上根本跑不动。Reservoir sampling 能锁死内存上限，但分布会严重偏向早期经验。Importance-weighted sampling 能修偏，方差又压不住。我现在的折中方案是按 skill 划分 reservoir，每个限存 10 万条 transition。整体 buffer 卡在 4GB 以内。这纯属工程 hack。理论上更干净的做法是 generative replay model (Shin 2017)。把旧经验压缩进网络权重，不存原始 transition。但我还没遇到生成器开销低到划算的落地场景。

总结来看，持续 RL 仍在工程妥协与理论空白之间摸索，检测机制与内存管理是下一阶段的突破口。接下来
