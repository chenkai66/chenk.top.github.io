二分查找看似简单，实则容易出错。核心思想就是每次把搜索范围缩小一半。但在实际编写时，尤其是在面试这种高压环境下，各种边界条件、死循环和返回值错误等问题就会暴露出来。本文不重复罗列模板代码，而是聚焦于解释模板设计背后的逻辑——即核心不变量。只要抓住了背后的不变量，那些让人纠结的细节——比如用 `<` 还是 `<=`，`right = mid` 还是 `right = mid - 1`——就不再是靠记忆硬背的东西，而是可以自然推导出来的结果。
## 一、心智模型：维护一个不断缩小的区间

![二分搜索：lo、mid、hi 逐步缩小搜索范围](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/gifs/leetcode/binary-search-step.gif)

二分查找的核心思想是**逐步缩小搜索范围**。我们用两个指针 `l` 和 `r` 标记当前可能包含答案的区间，每次取中点 `m` 检查 `nums[m]` 的值，然后根据结果排除掉一半的元素。上面这张图展示了一个典型例子：长度为 16 的有序数组，目标值是 `23`，经过四次迭代后，搜索范围从 16 缩小到 1。

整个算法正确性的关键，在于始终维护以下不变量：

> **不变量**：在每次迭代开始时，如果目标值存在，它一定位于当前区间 `[l, r]`（或 `[l, r)`）内。

只要这条不变量始终成立，算法就是正确的。一旦某一步破坏了它——比如在寻找第一个等于目标值的位置时，遇到 `nums[m] == target` 却直接把右边界设为 `r = m - 1`——后面的搜索就完全失去了意义。

为什么二分查找的时间复杂度是 $O(\log n)$？经过 $k$ 轮迭代后，剩下的区间最多还有 $\lceil n / 2^k \rceil$ 个元素。要让区间缩小到只剩一个元素，需要的迭代次数是 $k = \lceil \log_2 n \rceil$。即使有十亿个元素，也只需要大约 30 次比较——这就是为什么数据库索引、文件系统和 Git bisect 这些工具都依赖二分查找的原因。

**二分查找的适用条件其实不是"数组有序"，而是"判定条件单调"**：只要你能写出一个布尔函数 `feasible(x)`，它的返回值随着 `x` 增大恰好从 `False` 翻转到 `True`（或者反过来），就可以用二分查找定位翻转点。本文后面提到的"答案空间二分"正是基于这个思路展开的。
## 二、基础模板：闭区间 `[l, r]`

最简单的场景：在有序数组中找一个等于 `target` 的下标，找不到就返回 `-1`。这里我们用**闭区间** `[l, r]` 和循环条件 `l <= r`：

```python
def binary_search(nums, target):
    """返回任意一个 nums[i] == target 的下标，找不到返回 -1。"""
    l, r = 0, len(nums) - 1
    while l <= r:
        m = l + (r - l) // 2     # 防止 l + r 溢出（C++/Java 必备习惯）
        if nums[m] == target:
            return m
        if nums[m] < target:
            l = m + 1            # target 肯定在右半边
        else:
            r = m - 1            # target 肯定在左半边
    return -1
```

只需掌握以下三个要点：

1. **初始边界**：`r = len(nums) - 1`，因为 `r` 是闭区间的右端点，必须包含在内。
2. **循环条件**：`l <= r`，只要 `l` 不超过 `r`，区间 `[l, r]` 就非空。
3. **更新规则**：如果 `nums[m] != target`，`m` 这个位置就没用了，直接用 `m + 1` 或 `m - 1` 把它踢出搜索范围。

`m = l + (r - l) // 2` 和 `(l + r) // 2` 数学上是一回事，但前者不会溢出，尤其是在 `l + r` 很大的时候。 Python 不怕大整数，但养成这个习惯，写 C++ 或 Java 时能省不少麻烦。

### LeetCode 704 —— 二分查找

> 给定一个按升序排列的整数数组 `nums` 和一个目标值 `target`，在数组中搜索 `target`，存在则返回下标，否则返回 `-1`。

```python
class Solution:
    def search(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums) - 1
        while l <= r:
            m = l + (r - l) // 2
            if nums[m] == target:
                return m
            if nums[m] < target:
                l = m + 1
            else:
                r = m - 1
        return -1
```

边界情况完全不用操心：  
- 空数组时，`r = -1`，循环直接跳过，返回 `-1`。  
- 单元素数组时，`l == r == 0`，循环执行一次，要么找到目标，要么返回 `-1`。  

该模板简洁可靠，建议熟练掌握。
## 三、边界模板：找第一个或最后一个位置

很多题目不仅要求判断目标值是否存在，还要求找到**第一个满足条件的位置**或者**最后一个满足条件的位置**，甚至插入位置。基础模板应付不了这种情况，因为当 `nums[m] == target` 时不能直接返回——左边可能还有更早的匹配，右边可能还有更晚的匹配。

![左边界与右边界模板在重复元素上的对比](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/05-二分查找/fig2_left_right_bound.png)

上图展示了两套模板在 `nums = [1, 2, 5, 5, 5, 5, 8, 9]` 和 `target = 5` 上的运行过程。代码几乎完全一样，唯一的区别在于比较符 `<` 是否取等。

### 左边界：找到第一个 `nums[i] >= target` 的位置

```python
def left_bound(nums, target):
    """半开区间 [l, r)，返回最左插入位置。"""
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] < target:
            l = m + 1            # m 太小，丢掉
        else:
            r = m                # m 可能是答案，保留
    return l                     # 循环结束时 l == r
```

有三点必须严格遵守：

- **`r = len(nums)`**，而不是 `len(nums) - 1`。我们用的是半开区间 `[l, r)`，这样才能表示“插到最后”的情况。
- **`l < r`**，配合半开区间的定义。`[l, r)` 在 `l == r` 时为空，循环自然终止。
- **`r = m`**，而不是 `m - 1`。当 `nums[m] >= target` 时，`m` 本身可能是答案，不能排除。

退出循环时，`l` 是第一个满足 `nums[i] >= target` 的下标。如果需要转化为“第一次出现 `target`”，再加一步检查：`l < len(nums) and nums[l] == target`。

### 右边界：找到最后一个 `nums[i] <= target` 的位置

```python
def right_bound(nums, target):
    """半开区间 [l, r)，返回最后一个匹配的下标 + 1。"""
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] <= target:
            l = m + 1            # 继续往右找
        else:
            r = m
    return l - 1                 # 实际答案要减 1（不存在时通过合法性检查兜底）
```

代码骨架和左边界几乎一样，唯一区别是把 `<` 换成了 `<=`。退出时，`l` 是第一个满足 `nums[i] > target` 的位置，因此 `l - 1` 才是最后一个 `nums[i] <= target` 的位置。同样要做合法性检查：`l - 1 >= 0 and nums[l - 1] == target`。

### LeetCode 35 —— 搜索插入位置

> 给定一个排序数组 `nums` 和一个目标值 `target`，在数组中找到目标值并返回其索引。如果不存在，返回它将会被按顺序插入的位置。要求 `O(log n)`。

这题就是左边界模板，直接套用即可：

```python
class Solution:
    def searchInsert(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums)
        while l < r:
            m = l + (r - l) // 2
            if nums[m] < target:
                l = m + 1
            else:
                r = m
        return l
```

以 `nums = [1, 3, 5, 6]`、`target = 2` 为例：  
- 第一次迭代：`m = 2, nums[2] = 5 >= 2`，更新 `r = 2`；  
- 第二次迭代：`m = 1, nums[1] = 3 >= 2`，更新 `r = 1`；  
- 第三次迭代：`m = 0, nums[0] = 1 < 2`，更新 `l = 1`。  

此时 `l == r`，返回 `1`。将 `2` 插入到下标 `1` 后，数组变为 `[1, 2, 3, 5, 6]`，依然有序。

### LeetCode 34 —— 在排序数组中查找元素的第一个和最后一个位置

> 给定一个按升序排列的整数数组 `nums` 和一个目标值 `target`，找出目标值在数组中出现的第一个和最后一个位置。如果不存在，返回 `[-1, -1]`。要求 `O(log n)`。

结合左右边界模板，再加上合法性检查即可：

```python
class Solution:
    def searchRange(self, nums: list[int], target: int) -> list[int]:
        def left_bound():
            l, r = 0, len(nums)
            while l < r:
                m = l + (r - l) // 2
                if nums[m] < target:
                    l = m + 1
                else:
                    r = m
            return l

        def right_bound():
            l, r = 0, len(nums)
            while l < r:
                m = l + (r - l) // 2
                if nums[m] <= target:
                    l = m + 1
                else:
                    r = m
            return l - 1

        lo = left_bound()
        if lo == len(nums) or nums[lo] != target:
            return [-1, -1]
        return [lo, right_bound()]
```

### LeetCode 278 —— 第一个错误的版本

> 版本号从 1 到 `n`，其中某个版本开始出错，之后所有版本都是错的。给定 API `isBadVersion(version)`，找到第一个错误版本，尽量减少 API 调用次数。

同样是左边界模板，只不过这次操作的是黑盒判定函数 `isBadVersion(v)`，它从 `False` 单调翻转为 `True`。我们查找的是判定函数，而不是数组：

```python
class Solution:
    def firstBadVersion(self, n: int) -> int:
        l, r = 1, n
        while l < r:
            m = l + (r - l) // 2
            if isBadVersion(m):
                r = m            # m 可能是第一个坏版本
            else:
                l = m + 1        # m 是好的，继续往右找
        return l
```

这是最简洁的例子：二分查找本质上服务于**单调性**，至于数据是数组还是 API 并不重要。
## 四、两种区间约定，选一种用到底

第二节用了闭区间 `[l, r]`，第三节用了半开区间 `[l, r)`。两种写法都没问题，但**混用就会出大问题，直接死循环**。

![闭区间与半开区间在同题上的对照](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/05-二分查找/fig5_template_comparison.png)

上图把两种约定放在一起跑了一遍，每步下面用方括号标出了当前区间。区别一目了然：闭区间 `[l, r]` 包含右端点，半开区间 `[l, r)` 不包含。配对规则很简单：

| 区间约定         | 初始 `r`        | 循环条件   | `r` 更新方式      |
| :--------------- | :-------------- | :--------- | :---------------- |
| 闭 `[l, r]`      | `len(nums)-1`   | `l <= r`   | `r = m - 1`       |
| 半开 `[l, r)`    | `len(nums)`     | `l < r`    | `r = m`           |

一个函数里选定一种约定，坚持到底。新手最容易踩的坑是 `l < r` 配 `r = m - 1`：区间缩到 `[l, l]` 时，`m = l`，再更新 `r = l - 1`，看似退出了循环，但可能已经跳过了答案。反过来更惨：`l <= r` 配 `r = m`，当 `l == r == m` 时，`r = m` 没变，直接卡死在无限循环里。
## 五、旋转数组：还能不能二分？

前面讨论的数组都是有序的，但旋转数组有点特殊。虽然整体无序，但它有一个关键性质：**从任意位置切开，至少有一半是有序的**。这个性质让我们依然可以用二分查找。

![旋转数组搜索 target = 0](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/05-二分查找/fig3_rotated_array.png)

上图展示了 `nums = [4, 5, 6, 7, 0, 1, 2]` 中找 `0` 的过程。每一步我们先判断哪边是有序的（通过比较 `nums[l]` 和 `nums[m]`），然后看目标值是否落在那个有序区间里。如果在，就继续在这个区间里搜；如果不在，就去另一边。

### LeetCode 33 —— 搜索旋转排序数组

> 一个升序数组在某个未知的旋转点处被旋转。给定 `target`，返回其下标；不存在则返回 `-1`。要求 `O(log n)` 时间复杂度。

```python
class Solution:
    def search(self, nums: list[int], target: int) -> int:
        l, r = 0, len(nums) - 1
        while l <= r:
            m = l + (r - l) // 2
            if nums[m] == target:
                return m
            if nums[l] <= nums[m]:
                # 左半 [l..m] 有序
                if nums[l] <= target < nums[m]:
                    r = m - 1
                else:
                    l = m + 1
            else:
                # 右半 [m..r] 有序
                if nums[m] < target <= nums[r]:
                    l = m + 1
                else:
                    r = m - 1
        return -1
```

这里有个细节要注意：`nums[l] <= nums[m]` 用了 `<=` 而不是 `<`。当 `l == m` 时，左半部分只剩一个元素 `nums[l]`，单个元素显然有序，`<=` 正好覆盖了这种情况。少了这个等号，边界条件可能会出问题。
## 六、寻找峰值：在无序数组上二分

峰值的定义是比左右邻居都大的元素。虽然数组本身无序，但只要抓住**斜率变化的方向**，依然可以用二分法高效找到峰值。

### LeetCode 162 —— 寻找峰值

> 峰值元素严格大于其相邻元素。给定整数数组 `nums`（约定 `nums[-1] = nums[n] = -∞`），找到任意一个峰值并返回其下标。要求 `O(log n)` 时间复杂度。

```python
class Solution:
    def findPeakElement(self, nums: list[int]) -> int:
        l, r = 0, len(nums) - 1
        while l < r:
            m = l + (r - l) // 2
            if nums[m] < nums[m + 1]:
                l = m + 1        # m 在上升段，峰一定在右边
            else:
                r = m            # m 在下降段或就是峰，往左搜（含 m）
        return l
```

代码逻辑很简单：  
- 如果 `nums[m] < nums[m + 1]`，说明当前位置处于上升趋势，结合约定 `nums[n] = -inf`，右侧必然存在一个峰值，因此可以直接排除左半边。  
- 如果 `nums[m] >= nums[m + 1]`，说明当前位置处于下降趋势或者本身就是峰值，结合约定 `nums[-1] = -inf`，左侧（包括 `m`）必然存在一个峰值，因此可以排除右半边。  

每次迭代都能将搜索范围缩小一半，同时保证"当前区间内至少存在一个峰值"这一不变量始终成立。

这类问题的核心在于识别出局部单调性——即使数组整体无序，只要某个局部性质满足单调变化，就可以套用二分模板解决。
## 七、答案空间二分：直接扔掉数组

二分查找最强的扩展，是完全不依赖数组，而是直接在**答案可能取值的范围**里搜索。只要能设计一个判定函数 `feasible(x)`，就能用二分快速定位最优解。

![答案空间二分：运送包裹的最小载重](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/05-二分查找/fig4_answer_space.png)

这种思路适用的场景有三个条件：

1. 答案是一个整数（或实数），并且我们知道它的范围 $[lo, hi]$。
2. 存在一个判定函数 `feasible(x)`，可以判断某个值 `x` 是否满足要求。
3. 判定函数 `feasible(x)` 是**单调**的：存在一个阈值 $x^*$，使得 `feasible(x)` 在它的一侧全为 `False`，另一侧全为 `True`。

在这种情况下，二分查找可以在 $\log(hi - lo)$ 次迭代内找到 $x^*$，每次只需要调用一次 `feasible` 函数。上图展示了"在 D 天内运送所有包裹"问题的求解过程：上半部分是"所需天数 vs 载重"的关系曲线（单调递减的阶梯函数），下半部分则演示了二分区间 `[l, r]` 如何逐步收缩到最优解。

### LeetCode 1011 —— 在 D 天内送达包裹的能力

> 传送带上的包裹必须按顺序在 `days` 天内运完，每天载重固定。求能在 `days` 天内运完所有包裹的最低运载能力。

我们需要按顺序把 `weights` 中的所有包裹在 `D` 天内运完，每天的载重固定，求满足条件的**最小载重**。

```python
class Solution:
    def shipWithinDays(self, weights: list[int], D: int) -> int:
        def feasible(cap: int) -> bool:
            days, cur = 1, 0
            for w in weights:
                if cur + w > cap:
                    days += 1
                    cur = w
                    if days > D:
                        return False
                else:
                    cur += w
            return True

        l, r = max(weights), sum(weights)
        while l < r:
            m = l + (r - l) // 2
            if feasible(m):
                r = m
            else:
                l = m + 1
        return l
```

代码中有三个关键设计点：

- **下界 `max(weights)`**：如果载重连最重的单个包裹都装不下，那肯定无解。
- **上界 `sum(weights)`**：如果一天把所有包裹都运完，那必然可行。
- **单调性**：如果载重 `c` 可行，那么任何更大的载重 `c' > c` 也一定可行（每天能运更多只会更快完成）。因此可行解集是"上闭区间" $[x^*, +\infty)$，用左边界模板正好能找到 $x^*$。

### LeetCode 875 —— 爱吃香蕉的珂珂

> 珂珂每小时以速度 `k` 吃一堆香蕉（每小时最多吃一堆）。求在 `h` 小时内吃完所有香蕉的最小速度 `k`。

同样的套路，代码骨架几乎一模一样，唯一的变化是判定函数的实现：

```python
class Solution:
    def minEatingSpeed(self, piles: list[int], h: int) -> int:
        def feasible(k: int) -> bool:
            hours = sum((p + k - 1) // k for p in piles)
            return hours <= h

        l, r = 1, max(piles)
        while l < r:
            m = l + (r - l) // 2
            if feasible(m):
                r = m
            else:
                l = m + 1
        return l
```

写完一道题，剩下的基本都是复制粘贴。 LeetCode 410 （分割数组的最大值）、 1482 （制作 m 束花所需的最少天数）、 719 （找到第 K 小的距离对）……这些题的代码框架完全一致，唯一的创造性工作就是根据题目需求设计那个 `feasible` 函数。
## 八、四种常见 bug，按出错频率排序

写二分时容易犯的错误基本就这四种。

**区间和更新方式不匹配**。`l < r` 配了 `r = m - 1`，或者 `l <= r` 配了 `r = m`。记住闭区间用 `<=` 和 `m ± 1`，半开区间用 `<` 和 `m`。其他组合一律不对。

**返回值搞错**。基础模板返回 `m` 或 `-1`；左边界返回 `l`；右边界返回 `l - 1`，最容易漏掉这个 `- 1`。不管用哪个模板，最后都得检查一下 `0 <= idx < len(nums) and nums[idx] == target`，确认答案合法再返回。

**`mid` 取整忘了 `+1`**。如果更新写成 `l = m`（而不是 `l = m + 1`），就要改用**上中位数**：`m = l + (r - l + 1) // 2`。否则当 `r = l + 1` 时，`m = l`，`l = m` 不会前进，直接死循环。 LeetCode 69 （`sqrt`）就是这种问题的经典例子。

**`(l + r) // 2` 整数溢出**。 Python 的大整数无所谓，但在 C++ 或 Java 里，`l, r` 接近 $2^{31}$ 时会爆。养成写 `l + (r - l) // 2` 的习惯，零成本，换语言时能救命。

简单调试方法：在循环里加一句 `print(f"l={l} r={r} m={m} nums[m]={nums[m]}")`，盯住三点：(a) 区间每轮都在缩小，(b) 不变量“答案在 `[l, r]` 内”始终成立，(c) 返回下标对应的值是否符合预期。
## 九、如何选择合适的模板

简单总结一下，大部分问题都可以归到以下几类：

- **随便找个匹配的下标，重复元素无所谓**：用基础模板，闭区间 `[l, r]`。
- **找第一个匹配、插入位置或最小可行值**：用左边界模板。
- **找最后一个匹配或最大可行值**：用右边界模板。
- **判定条件是个黑盒（版本号、容量、速度等）**：直接在答案空间上跑左边界模板，完全不用管数组本身。
- **数组无序但局部有单调性（比如峰值、旋转数组）**：还是用基础模板，不过比较逻辑要换成检查结构性质，比如哪半边有序、哪边是上升趋势，而不是直接和目标值比大小。

如果不确定选哪种，推荐优先考虑半开区间 `[l, r)`。它的好处很明显：
- 能自然处理“插入到最后”的情况。
- 切换到浮点数时也不用改代码。
- 循环条件 `l < r` 更直观地表达了终止条件。

总之，选对模板能省不少事，尤其是面试时，别让细节拖了后腿。
## 十、二分查找在工程中的实际应用

二分查找绝不仅仅是面试中的考点，它是少数几个**真正支撑现代基础设施**的核心算法之一。

数据库索引结构——比如 B 树、 B+ 树、 LSM-tree 的有序段——本质上就是分支因子大于 2 的多路二分查找。在一个十亿行的表上执行 `WHERE id = 12345`，核心操作不过是对数级别的 $\log_B n \approx 4$ 次磁盘读取，其中 $B$ 是页扇出系数。如果没有二分查找， OLTP 数据库根本无法高效运行。

`git bisect` 是二分思想在版本控制中的经典应用：标记一个 commit 为正常（good），另一个稍后的 commit 为异常（bad），然后自动检出中间的 commit 进行测试，并根据结果递归缩小范围。一千个 commit 的回归问题，最多只需要测试 10 次就能定位问题。

编程语言的标准库中也有二分的身影。 Python 的 `bisect_left` 和 `bisect_right`、 C++ 的 `lower_bound` 和 `upper_bound`、 Java 的 `Arrays.binarySearch`，这些函数的实现逻辑直接对应本文第二、三节中的模板。命名也很直观：`lower_bound` 就是左边界，`upper_bound` 就是右边界减一。

学习率搜索、超参数调优、限流阈值探测等场景中，二分同样大显身手。只要判定条件满足“单次验证成本高但单调性成立”，工业系统都会用到答案空间二分。比如，“99% 成功率下系统能扛住的最大 QPS”这类问题，和 LeetCode 上的“珂珂吃香蕉”问题，本质上用的是同一套代码逻辑。
## 十一、随手翻的小抄

面试前一晚过一遍：

```python
# 基础：找任意一个等于 target 的下标，找不到返回 -1
def std(nums, target):
    l, r = 0, len(nums) - 1
    while l <= r:
        m = l + (r - l) // 2
        if nums[m] == target: return m
        if nums[m] < target:  l = m + 1
        else:                  r = m - 1
    return -1

# 左边界：最小的 i 满足 nums[i] >= target
def lb(nums, target):
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] < target: l = m + 1
        else:                 r = m
    return l

# 右边界：最大的 i 满足 nums[i] <= target（不存在返回 -1）
def rb(nums, target):
    l, r = 0, len(nums)
    while l < r:
        m = l + (r - l) // 2
        if nums[m] <= target: l = m + 1
        else:                  r = m
    return l - 1

# 答案空间：[lo, hi] 内最小的 x 满足 feasible(x) == True
def search_answer(lo, hi, feasible):
    l, r = lo, hi
    while l < r:
        m = l + (r - l) // 2
        if feasible(m): r = m
        else:            l = m + 1
    return l
```
