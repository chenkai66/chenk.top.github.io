如果写过两层 `for` 循环去枚举所有连续子数组，那**滑动窗口**很可能就是你缺的优化点。它能把 $O(nk)$ 或 $O(n^2)$ 的暴力扫描压缩成一次线性遍历，核心在于“复用之前算好的结果”。本文从最基础的思路讲起，通过四道经典的 LeetCode 题目详细解析方法论，并补充了单调队列的高级玩法。
## 一图看懂滑动窗口

滑动窗口的核心思想很简单：在数组或字符串上维护一个连续区间 `[left, right]`。窗口移动时，不需要重新计算整个区间的值，而是**加入右边新进来的元素，移除左边出去的元素**。每个元素最多被访问和更新各一次，整体复杂度是 $O(n)$。

根据窗口大小是否固定，可以分为两种情况：

- **固定窗口**——窗口宽度固定为 `k`，左右指针同步移动。
- **可变窗口**——`right` 指针负责扩张窗口，`left` 指针在窗口不满足条件时收缩。

下图展示了固定窗口求最大子数组和的过程：绿色表示当前窗口，橙色表示新进入的元素，粉色表示即将移出的元素。右侧的算式表明，每次更新只需一次加法和一次减法，这是滑动窗口实现线性时间复杂度的关键。

![固定窗口：求大小为 k 的最大子数组和](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/04-滑动窗口技巧/fig1_fixed_window.png)
## 固定窗口：大小为 k 的最大子数组和

**问题描述**：给定一个数组 `arr` 和一个整数 `k`，找到所有长度为 `k` 的连续子数组中，和最大的那个。

**朴素解法**：暴力枚举每个长度为 `k` 的子数组并计算其和，复杂度为 $O(nk)$。

**滑动窗口优化**：先计算第一个窗口的和，然后每次滑动时减去窗口左侧移出的元素，加上右侧新进入的元素。这样每一步只需两次操作即可完成更新。

```python
def max_sum_subarray(arr, k):
    """求长度为 k 的连续子数组的最大和。"""
    if len(arr) < k:
        return None

    window_sum = sum(arr[:k])     # 初始化第一个窗口的和
    best = window_sum

    for right in range(k, len(arr)):
        # 新进入窗口的元素是 arr[right]，离开的是 arr[right - k]
        window_sum += arr[right] - arr[right - k]
        best = max(best, window_sum)

    return best
```

**Java 实现**

```java
public int maxSumSubarray(int[] arr, int k) {
    if (arr.length < k) return Integer.MIN_VALUE;
    int windowSum = 0;
    for (int i = 0; i < k; i++) windowSum += arr[i];  // 初始化窗口和
    int best = windowSum;
    for (int right = k; right < arr.length; right++) {
        windowSum += arr[right] - arr[right - k];      // 滑动窗口更新
        best = Math.max(best, windowSum);
    }
    return best;
}
```

**复杂度分析**：时间复杂度 $O(n)$，每次滑动只涉及一次加法和一次减法；空间复杂度 $O(1)$。

**为什么高效？** 窗口的和仅取决于其内部元素。当窗口右移时，集合的变化仅限于一个元素的加入和一个元素的移除。因此，只需对这两个变化量进行增量调整即可完成更新。这种增量更新思想不仅适用于求和，还可推广至频率统计、单调队列等场景（详见第 6 节）。
## 可变窗口的两种思路

窗口大小不固定时，我们需要用两个指针配合，根据问题特点调整策略：

- **找最长合法窗口**：`right` 不断向右扩展；一旦窗口变得不合法，就逐步移动 `left`，直到窗口重新合法，同时记录最大长度。
- **找最短合法窗口**：`right` 扩展到窗口刚好满足条件后，尽量压缩 `left`，看看能不能让窗口更短，每次收缩时更新答案。

![可变窗口：无重复字符的最长子串](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/04-滑动窗口技巧/fig2_variable_window.png)

再来看窗口长度随时间变化的情况。绿色区域是扩张阶段，紫色区域是收缩阶段。

![窗口长度随时间变化](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/04-滑动窗口技巧/fig3_expand_contract.png)

这里最关键的一点是：**`left` 永远只往右移动，不会回退**。虽然外层是 `for` 循环，内层是 `while` 循环，但整个过程中 `left` 的总移动次数不会超过 $n$。这就是滑动窗口“摊还 $O(n)$”的核心依据。
### LeetCode 3 —— 无重复字符的最长子串

![无重复字符的最长子串：窗口扩展与收缩](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/gifs/leetcode/sliding-window-substr.gif)


> 给定一个字符串 `s`，找出其中不含重复字符的最长子串的长度。

**解法思路**：用可变窗口找最长子串。核心约束是“窗口内的字符必须互不相同”。当遇到重复字符时，将左边界直接移至该字符上一次出现位置的右侧。

```python
def length_of_longest_substring(s):
    """LeetCode 3：无重复字符的最长子串。"""
    last_seen = {}        # 记录每个字符最近一次出现的位置
    left = 0              # 窗口左边界
    best = 0              # 最长子串长度

    for right, ch in enumerate(s):
        if ch in last_seen and last_seen[ch] >= left:
            # 如果当前字符在窗口内已存在，直接移动左边界
            left = last_seen[ch] + 1
        last_seen[ch] = right
        best = max(best, right - left + 1)

    return best
```

**Java 实现**

```java
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int left = 0, best = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (last_seen.containsKey(c) && last_seen.get(c) >= left) {
            // 跳过重复字符的位置
            left = last_seen.get(c) + 1;
        }
        last_seen.put(c, right);
        best = Math.max(best, right - left + 1);
    }
    return best;
}
```

**复杂度分析**：时间复杂度 $O(n)$，每个字符最多被插入和移出哈希表各一次；空间复杂度 $O(\min(n, |\Sigma|))$，其中 $|\Sigma|$ 是字符集大小。

**两种实现风格**：  
- **跳跃式收缩**：上面的代码中，`left` 直接跳到重复字符的下一个位置，简洁高效，适合处理基础问题。  
- **步进式收缩**：维护一个频次表，通过 `while count[s[right]] > 1: drop(s[left]); left += 1` 逐步收缩窗口。虽然稍显冗长，但更灵活，容易扩展到类似“最多包含 k 种字符”的问题。  

两种方法的时间复杂度都是 $O(n)$，选择哪种取决于具体需求和扩展性考虑。
### LeetCode 76 —— 最小覆盖子串

> 给定两个字符串 `s` 和 `t`，要求从 `s` 中找出一个最短的子串，能够包含 `t` 的所有字符（包括重复字符）。如果不存在这样的子串，返回空字符串 `""`。

**解题思路**：用滑动窗口解决，目标是找到满足条件的最短窗口。这里的约束是“窗口内必须包含 `t` 中所有字符，并且每个字符的出现次数不能少于 `t` 中的要求”。先扩展右边界直到窗口合法，然后尝试收缩左边界以获取最短长度。

判断窗口是否合法时，直接比对两个哈希表会很耗时。更好的办法是维护一个 `valid` 计数器，表示当前有多少种字符已经满足了数量要求。当 `valid == len(need)` 时，说明整个窗口符合条件。

```python
from collections import Counter

def min_window(s, t):
    """LeetCode 76：覆盖 t 所有字符的最短子串。"""
    if not s or not t or len(s) < len(t):
        return ""

    need = Counter(t)  # 目标字符及其数量
    have = {}          # 当前窗口中目标字符的数量
    valid = 0          # 已满足条件的字符种类数
    left = 0           # 窗口左边界
    best_len = float("inf")  # 最短窗口长度
    best_start = 0     # 最短窗口起始位置

    for right, ch in enumerate(s):
        # 如果当前字符是目标字符，更新其计数
        if ch in need:
            have[ch] = have.get(ch, 0) + 1
            # 如果该字符的数量刚好达标，增加 valid
            if have[ch] == need[ch]:
                valid += 1

        # 当窗口合法时，尝试收缩左边界
        while valid == len(need):
            # 更新最短窗口信息
            if right - left + 1 < best_len:
                best_len = right - left + 1
                best_start = left

            # 移除左边界字符
            drop = s[left]
            left += 1
            if drop in need:
                # 如果移除后即将失衡，提前减少 valid
                if have[drop] == need[drop]:
                    valid -= 1
                have[drop] -= 1

    # 返回结果，若未找到合法窗口则返回空字符串
    return "" if best_len == float("inf") else s[best_start:best_start + best_len]
```

**Java 版本**

```java
public String minWindow(String s, String t) {
    if (s.length() < t.length()) return "";
    Map<Character, Integer> need = new HashMap<>();
    for (char c : t.toCharArray()) need.merge(c, 1, Integer::sum);

    Map<Character, Integer> have = new HashMap<>();
    int valid = 0, left = 0;
    int bestLen = Integer.MAX_VALUE, bestStart = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        if (need.containsKey(c)) {
            have.merge(c, 1, Integer::sum);
            if (have.get(c).intValue() == need.get(c).intValue()) valid++;
        }
        // 窗口合法时尝试收缩
        while (valid == need.size()) {
            if (right - left + 1 < bestLen) {
                bestLen = right - left + 1;
                bestStart = left;
            }
            char d = s.charAt(left++);
            if (need.containsKey(d)) {
                if (have.get(d).intValue() == need.get(d).intValue()) valid--;
                have.merge(d, -1, Integer::sum);
            }
        }
    }
    return bestLen == Integer.MAX_VALUE ? "" : s.substring(bestStart, bestStart + bestLen);
}
```

**复杂度分析**：时间复杂度为 $O(|s| + |t|)$，因为左右指针都只遍历一次字符串；空间复杂度为 $O(|\Sigma|)$，其中 $|\Sigma|$ 是字符集大小。

**易错点**：`if have[drop] == need[drop]: valid -= 1` 这一行特别关键。我们需要在实际减少字符计数之前判断“当前刚好达标，移走一个就会失衡”，所以这一步必须放在更新 `have[drop]` 之前。顺序写反会导致 `valid` 和实际窗口状态不同步。
### LeetCode 567 —— 字符串的排列

> 判断 `s2` 中是否存在某个子串是 `s1` 的一个排列。

**解法思路**：固定窗口，长度为 `len(s1)`。两个字符串互为排列的充要条件是它们的字符频率分布完全一致。因此，我们可以用一个长度为 26 的数组分别记录 `s1` 和 `s2` 当前窗口的字符频率，并通过比较两者的匹配情况来判断是否满足条件。

```python
def check_inclusion(s1, s2):
    """LeetCode 567：s2 是否包含 s1 的某个排列？"""
    n, m = len(s1), len(s2)
    if n > m:
        return False

    need = [0] * 26
    have = [0] * 26
    for c in s1:
        need[ord(c) - ord('a')] += 1
    for c in s2[:n]:
        have[ord(c) - ord('a')] += 1

    # 统计当前有多少个字符的频率匹配上了
    matches = sum(1 for i in range(26) if need[i] == have[i])

    for right in range(n, m):
        if matches == 26:
            return True

        r = ord(s2[right]) - ord('a')
        l = ord(s2[right - n]) - ord('a')

        # 新元素从右边进入窗口
        have[r] += 1
        if have[r] == need[r]:
            matches += 1
        elif have[r] == need[r] + 1:
            matches -= 1

        # 旧元素从左边移出窗口
        have[l] -= 1
        if have[l] == need[l]:
            matches += 1
        elif have[l] == need[l] - 1:
            matches -= 1

    return matches == 26
```

**Java 实现**

```java
public boolean checkInclusion(String s1, String s2) {
    int n = s1.length(), m = s2.length();
    if (n > m) return false;
    int[] need = new int[26], have = new int[26];
    for (int i = 0; i < n; i++) {
        need[s1.charAt(i) - 'a']++;
        have[s2.charAt(i) - 'a']++;
    }
    int matches = 0;
    for (int i = 0; i < 26; i++) if (need[i] == have[i]) matches++;
    for (int right = n; right < m; right++) {
        if (matches == 26) return true;
        int r = s2.charAt(right) - 'a';
        int l = s2.charAt(right - n) - 'a';
        have[r]++;
        if (have[r] == need[r]) matches++;
        else if (have[r] == need[r] + 1) matches--;
        have[l]--;
        if (have[l] == need[l]) matches++;
        else if (have[l] == need[l] - 1) matches--;
    }
    return matches == 26;
}
```

**复杂度分析**：时间复杂度 $O(m)$，每次滑动窗口时只需更新 `matches` 计数器，避免了重新扫描整个频率数组；空间复杂度 $O(1)$，因为只用了固定大小的数组。

这个方法同样适用于 **LeetCode 438 （找到所有字母异位词）**，只需将“提前返回”改为“收集起始下标”即可。
### 进阶变体：滑动窗口最大值（单调队列）

如果窗口需要查询的不是和或计数，而是**窗口内的最大值或最小值**，那么简单的“加一个、减一个”就搞不定了——移除当前最大值后，下一个最大值的位置我们无从得知。这时就需要用到**单调队列**：队列里的元素按值严格递减，队首始终是当前窗口的最大值。

![单调队列求滑动窗口最大值](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/04-滑动窗口技巧/fig4_deque_max.png)

```python
from collections import deque

def max_sliding_window(nums, k):
    """LeetCode 239：每个长度为 k 的窗口的最大值。"""
    dq = deque()      # 存储下标；对应的值在 nums 中严格递减
    out = []

    for i, x in enumerate(nums):
        # 1. 踢掉队尾所有比 x 小的元素——它们不可能再成为最大值
        while dq and nums[dq[-1]] < x:
            dq.pop()
        dq.append(i)

        # 2. 如果队首已经滑出窗口，直接移除
        if dq[0] <= i - k:
            dq.popleft()

        # 3. 窗口形成后，队首就是当前窗口的最大值
        if i >= k - 1:
            out.append(nums[dq[0]])

    return out
```

每个下标最多入队和出队一次，整体复杂度仍然是 $O(n)$。
## 滑动窗口的适用场景

下面这张决策树帮你快速判断“什么问题适合用滑动窗口，该用哪种形式”：

![滑动窗口适用场景决策树](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/leetcode/04-滑动窗口技巧/fig5_decision_tree.png)

滑动窗口的核心在于**连续区间+增量更新**。遇到问题时，如果同时满足以下三点，基本就可以锁定滑动窗口解法：

1. 答案是一段**连续区间**（比如子数组、子串）。
2. 区间的属性支持**增量维护**——窗口移动时，只需常数次操作就能更新状态（比如累加和、频率表、单调队列等）。
3. 目标是找**固定大小的最优解**（`k`），或者**最长/最短的合法区间**。

如果问题不满足这些条件，比如需要非连续选择、全局视角，或者频繁撤销任意操作，那就别硬套滑动窗口了。试试前缀和、双指针、动态规划或者堆，可能更合适。
## 三套值得背下来的模板

**固定窗口**

```python
def fixed_window(arr, k):
    state = init_state(arr[:k])           # 例如和、频次表
    best = report(state)
    for right in range(k, len(arr)):
        state = add(state, arr[right])
        state = remove(state, arr[right - k])
        best = better(best, report(state))
    return best
```

**可变窗口——求最长合法**

```python
def longest_valid(arr):
    left = 0
    state = empty_state()
    best = 0
    for right, x in enumerate(arr):
        state = add(state, x)
        while not is_valid(state):
            state = remove(state, arr[left])
            left += 1
        best = max(best, right - left + 1)
    return best
```

**可变窗口——求最短合法**

```python
def shortest_valid(arr):
    left = 0
    state = empty_state()
    best = float("inf")
    for right, x in enumerate(arr):
        state = add(state, x)
        while is_valid(state):
            best = min(best, right - left + 1)
            state = remove(state, arr[left])
            left += 1
    return best if best != float("inf") else 0
```

## 常见坑点

- **窗口长度少算一个**：记住公式是 `right - left + 1`，别漏了加一。
- **`valid` 更新顺序搞错**：`have[c]` 刚等于 `need[c]` 时才加一，快跌破时要先减。顺序反了计数器就对不上。
- **用 `list.pop(0)` 维护固定窗口**：这操作复杂度是 $O(k)$，太慢了。换成 `collections.deque` 或者直接用下标算，保证 $O(1)$。
- **每次都重新算窗口值**：滑动窗口的核心就是增量更新。如果还在扫整个窗口，复杂度直接退化成暴力解法。
- **“和 ≥ target” 碰上负数**：窗口收缩时和可能不降反升，套模板会出问题。这种情况得换思路，比如前缀和加单调队列。

| 中等 | LC 438 找到字符串中所有字母异位词 | 固定窗口 |
| 中等 | LC 713 乘积小于 K 的子数组 | 求最长 |
| 中等 | LC 1004 最大连续 1 的个数 III | 求最长 |
| 困难 | LC 76 最小覆盖子串 | 求最短 |
| 困难 | LC 239 滑动窗口最大值 | 单调队列 |
| 困难 | LC 30 串联所有单词的子串 | 固定窗口 + 哈希 |
## 总结

滑动窗口的核心思想是**摊还优化**：不用每次都从零计算窗口的值，而是维护一个状态，在窗口移动时做增量更新。两个指针单调右移，直接把暴力解法的 $O(n^2)$ 压缩到 $O(n)$。

遇到连续区间问题时，记住这四个关键问题：

1. 窗口大小是**固定**的还是**可变**的？
2. 如果是可变窗口，目标是最**长**合法窗口，还是最**短**合法窗口？
3. 我需要维护什么**状态**——是和、频次表，还是单调队列？
4. 窗口的**约束条件**是什么？哪个指针的移动会破坏它？

厘清这四点后，绝大多数滑动窗口题目均可通过模板化方式解决。
## 滑动窗口在真实场景中的应用

滑动窗口的用途远不止刷题，它在实际工程中也有很多经典落地场景。以下是几个我亲身体验过的案例：

- **限流算法。** 常见的限流实现有两种：令牌桶和滑动窗口计数器。比如“每分钟最多 1000 次请求”的功能，可以用一个存储时间戳的双端队列（deque）来实现，左侧逐出超时的记录即可。像 `redis-cell` 和主流 API 网关的限流模块，基本都采用了类似的思路。
- **实时数据聚合。** 如果你用过监控系统里的“过去一小时错误率”图表，那背后通常就是一个滑动窗口。 Kafka Streams 提供的 `TimeWindow` 或者 Flink 的滚动/滑动窗口，本质上也是对这一模式的封装，只不过加了更高级的框架支持。
- **网络协议优化。** TCP 协议里的拥塞控制窗口（`cwnd`），实际上就是针对未确认字节的一个滑动窗口。而接收方的通告窗口则限制了发送方左边界能推进到的位置。

面试题只是滑动窗口的简化版，真实生产环境中的实现会更复杂。主要区别在于数组下标被替换成了时间戳，指针移动变成了基于 TTL 的自动过期机制。但核心思想完全一致，仍然是维护一个动态窗口并高效更新状态。
## 更地道的 Python 写法

C 风格的 `while right < n` 循环能用，但换成 `for right, x in enumerate(arr)` 不仅更简洁，还能避免常见的差一错误。对比一下：

```python
# C 风格
right = 0
while right < len(arr):
    add(arr[right])
    while bad(): remove(arr[left]); left += 1
    best = max(best, right - left + 1)
    right += 1

# Pythonic
left = 0
for right, x in enumerate(arr):
    add(x)
    while bad():
        remove(arr[left]); left += 1
    best = max(best, right - left + 1)
```

第二种写法省掉了 `right += 1`（面试时最容易漏写的一步），也不用反复通过 `arr[right]` 取值。**逻辑完全一致，且显著降低出错概率**。

如果需要维护频率表窗口，直接用 `collections.Counter`，别手写字典，`+=` 和 `-=` 的操作都经过充分测试，简单可靠：

```python
from collections import Counter

def length_of_longest_substring_k_distinct(s: str, k: int) -> int:
    cnt = Counter()
    left = best = 0
    for right, c in enumerate(s):
        cnt[c] += 1
        while len(cnt) > k:
            cnt[s[left]] -= 1
            if cnt[s[left]] == 0:
                del cnt[s[left]]
            left += 1
        best = max(best, right - left + 1)
    return best
```

注意减完计数后那行 `del`，这是保证 `len(cnt)` 准确的关键 —— 如果不删掉计数为零的键，“不同字符数”会偷偷包含无效的 key。
## 边界用例自查表

提交代码前，花点时间过一遍这张表，滑动窗口问题的“Wrong Answer”基本逃不出这些坑：

| 情况 | 为什么会挂 |
|---|---|
| 空字符串 / 空数组 | 初始化 `best` 时选 `0` 还是 `float('inf')` 很关键，差一格就错 |
| “至多 K 个不同”中 `k = 0` | 合法窗口应该是空的，但如果代码一开始就无脑扩展右边界，会误报长度为 1 |
| 单个元素 | 判断条件写 `>` 还是 `>=` 对比 `best`，在这里很容易暴露问题 |
| 元素全相同 | “求最长”的模板通常没问题；但“求最短”的模板可能直接塌缩到长度 1，再也无法扩展 |
| `target` 大于 `sum(arr)` | “和 ≥ target 的最短子数组”应该返回 0，而不是 `float('inf')` |
| 包含负数 | 滑动窗口的不变量通常假设 `add` 是单调递增的，负数会破坏这个前提，得改用前缀和 + 单调队列 |
| 窗口比数组还大 | 固定窗口模板必须显式检查 `len(arr) < k`，否则会越界 |
