2025 年，每个做软件的团队都会遇到一个老问题：买服务器还是租服务器？答案却完全反过来了——20 年前，我把机器塞进机房；现在，只需用 YAML 写好配置，全球供应商几秒钟就能准备好资源，按秒计费，不付钱就销毁。云计算不只是“别人的电脑”，而是一套覆盖计算、存储和网络的抽象，具备可编程、可计量和支持多租户的特点，彻底改变了企业的运作方式和工程师每天的工作内容。

## 1. 云计算到底是什么

最常被引用的定义来自美国国家标准与技术研究院（NIST），它列出了五个**关键特性**，这些特性之所以重要，是因为它们能清楚地将“看起来像云但其实不是”的东西排除在外。

- **按需自助服务** -- 开发者通过 API 或控制台直接获取资源，不需要提交工单或打电话。
- **广泛的网络访问** -- 资源可以通过标准网络从各种客户端访问。
- **资源池化** -- 物理基础设施是多租户共享的，用户看不到也不需要关心自己的工作负载具体运行在哪台机器上。
- **快速弹性** -- 容量可以在几分钟内扩展或收缩，通常是自动的，对用户来说感觉“几乎无限”。
- **可计量服务** -- 使用情况以非常细的粒度进行计量和计费（按秒、按请求、按 GB-月）。

传统数据中心租给你一个机柜，至少漏掉三条；而你用浏览器登录的一款 SaaS 应用，五条全中——这个定义让边界变得清晰。

### 简史 -- 让今天的设计决策更容易理解

| 时代            | 关键事件                                                                            | 对今天的意义                                                       |
| --------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------------------ |

| 2002-2006       | Salesforce 推出 SaaS； Amazon 发布 S3 （2006-03）和 EC2 （2006-08）                    | 第一次让外部开发者可以按小时租用计算资源                           |
| 2008-2014       | Google App Engine （PaaS）、 Heroku、 OpenStack、 Docker （2013）、 Kubernetes （2014）    | 更高层次的抽象出现，容器革命拉开序幕                               |
| 2015-2020       | Lambda （2014）、 Serverless 成为主流，超大规模厂商年营收突破 500 亿美元              | 按调用计费，运维层彻底“消失”                                       |
| 2021-至今       | 全球年化运行率超过 3000 亿美元， AI 推动针对 GPU 的第二轮扩张                        | 容量规划再次回归 -- 这次围绕的是加速器                             |
## 2. 服务模型 -- 管理边界在哪里

可以把云服务看成一个金字塔：层级越高，你需要管理的内容越少，控制权也越小。常见的四层分别是 IaaS、 PaaS、 FaaS 和 SaaS。

![云服务模型金字塔（IaaS / PaaS / FaaS / SaaS）](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig1_service_model_pyramid.png)

### 基础设施即服务（IaaS）

你租用虚拟机、虚拟磁盘和虚拟网络。从操作系统往上，所有东西都归你管：内核补丁、运行时环境、应用程序、数据。

- **AWS EC2** -- 提供 700 多种实例类型，覆盖通用型、内存优化型、计算优化型、存储优化型和 GPU 优化型
- **Google Compute Engine** -- 支持持续使用折扣、抢占式实例和自定义规格的虚拟机
- **Azure Virtual Machines** -- 深度集成 Active Directory、.NET 和 Windows 授权
- **阿里云 ECS** -- 国内市场占有率领先，提供针对通义推理任务优化的实例族

**什么时候选 IaaS？** 如果你要迁移现有应用且不改代码、需要特定的操作系统或内核模块，或者你的工作负载对性能要求极高（比如 HPC、 GPU 训练、低延迟交易），任何 hypervisor 之上的抽象层都会带来无法接受的性能损失，那就选 IaaS。

**成本参考：** 一台 4 vCPU、 16 GB 的实例按需价格约为 0.20 美元/小时（约 150 美元/月）。如果选择 1 年预留实例，成本可降低 30%-50%；如果使用 Spot 实例且能容忍中断，成本可降低 60%-80%。

### 平台即服务（PaaS）

你只需要提交代码和配置，厂商负责运行操作系统、运行时环境、负载均衡和自动扩展。你看不到服务器，只能看到日志和监控指标。

- **Heroku** -- 最早实现 `git push` 部署体验的平台
- **Google App Engine** -- 能从零自动扩展到每秒数千 QPS，无需人工干预
- **AWS Elastic Beanstalk** -- 上传代码后， AWS 在后台自动创建 EC2、 ELB、 ASG 和 CloudWatch
- **Vercel / Netlify** -- 偏向前端的 PaaS，专为 Next.js 等框架优化

**什么时候选 PaaS？** 如果上线速度是关键、团队规模较小、应用本身比较标准（比如 Web 应用、 API 或后台任务），并且你愿意接受运行时限制以换取不用操心操作系统的便利，那就选 PaaS。

### 函数即服务（FaaS）/ 无服务器

你提交一个函数，它按需运行，按调用次数和执行时间（毫秒级）计费。请求之间没有需要保活的实例。

- **AWS Lambda** -- 经典代表，单次执行最长 15 分钟
- **Azure Functions**、**Google Cloud Functions / Cloud Run** -- 类似的服务
- **阿里云函数计算（FC）**

**什么时候选 FaaS？** 如果你的工作负载是突发型或事件驱动型（比如图片上传触发缩略图生成、 Webhook 触发工作流）、流量模式难以预测，或者用于连接托管服务的胶水代码，那就选 FaaS。**避免使用 FaaS 的场景：** 需要持久连接、冷启动时间必须小于 50 毫秒但又不想购买预置并发（Provisioned Concurrency），或者需要长时间运行的计算任务 -- 当稳态负载超过某个阈值时，单位经济性会变得不利。

### 软件即服务（SaaS）

你直接使用应用程序，厂商负责从芯片到软件的所有事情。

- **典型例子：** Gmail、 Salesforce、 Slack、 Zoom、 Microsoft 365、 Notion、 Figma、 Datadog

**什么时候选 SaaS？** 如果这个应用对你的业务来说是标准化需求（比如邮件、 CRM、视频会议、可观测性工具），那就选 SaaS。**什么时候不该选 SaaS？** 如果这个应用是你的核心竞争力 -- 是你应该自己打造的东西，那就别选 SaaS。

### 一张图：管理边界

| 维度          | IaaS                | PaaS                | FaaS                  | SaaS                |
| ------------- | ------------------- | ------------------- | --------------------- | ------------------- |
| 控制力        | 高                  | 中                  | 低（按函数）          | 仅配置              |
| 运维成本      | 高                  | 中                  | 低                    | 无                  |
| 上线速度      | 较慢                | 较快                | 事件类极快            | 最快                |
| 计费形态      | 按小时              | 按小时 / 按请求     | 按请求 + 按毫秒       | 按席位              |
| 厂商绑定      | 低                  | 中                  | 中-高                 | 高                  |
| 典型代表      | EC2、 ECS            | Heroku、 App Engine  | Lambda、 Cloud Run     | Gmail、 Salesforce   |

最简单的判断方法：**沿着金字塔往上走，管理边界不断上移；边界以下是厂商的责任，边界以上是你的责任。**
## 3. 部署模型 -- 工作负载放在哪里

服务模型决定租什么，部署模型决定用谁的机房、网络怎么连。

![公有云、私有云、混合云、多云对比](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig2_deployment_models_comparison.png)

### 公有云

超大规模厂商负责运行数据中心，我和陌生租户共享底层硬件，靠加密和 hypervisor 实现隔离。适合大多数长尾工作负载： Web 应用、批处理任务、开发测试环境，以及创业公司几乎所有的需求。

### 私有云

我（或者托管服务商）为单一租户运营专用基础设施。适合监管严格、延迟敏感或数据驻留要求高的场景：国防、特定医疗和银行业务，或者对控制器响应时间要求在毫秒级以内的工厂系统。

### 混合云

公有云和私有云通过 VPN 或专线（AWS Direct Connect、 Azure ExpressRoute、 GCP Interconnect、阿里云高速通道）连接，形成一个逻辑整体。经典模式是：敏感客户数据留在本地，批处理任务和突发流量溢出到公有云。**坑点**在于：我现在要维护两套工具链、两套安全策略、两套 oncall 排班。

### 多云

生产环境中同时使用两家或以上的公有云。主要动机包括：避免厂商绑定、利用各家最强的服务（比如 GCP 的 BigQuery、 AWS 的 Bedrock），或者满足客户的硬性要求。代价也很明显：身份管理、网络架构、可观测性、 CI/CD 都得做到云无关。

| 维度       | 公有云          | 私有云          | 混合云          | 多云            |
| ---------- | --------------- | --------------- | --------------- | --------------- |
| 成本形态   | OpEx，低        | CapEx，高       | 混合            | 混合，更高      |
| 扩展性     | 几乎无限        | 有上限          | 灵活            | 非常灵活        |
| 控制力     | 低              | 高              | 中              | 中-低           |
| 复杂度     | 低              | 中              | 高              | 非常高          |
| 绑定风险   | 单厂商          | 无（都是我的）  | 降低            | 最低            |

> **经验法则：** 默认选择单地域公有云。当停机开始真正烧钱时，再加第二个地域。只有出现战略性原因（比如监管要求、客户强制规定、主云没有的关键服务）时，才考虑引入第二个云。
## 4. 市场和它的集中化原因

云服务是现代基础设施中最集中的市场之一。 AWS、 Microsoft Azure 和 Google Cloud 这三家巨头占据了全球 IaaS+PaaS 市场大约三分之二的份额；阿里云在中国大陆独占鳌头；剩下的市场份额则由一些区域性和垂直领域的厂商瓜分。

![全球云基础设施市场份额 -- AWS / Azure / GCP / 阿里云](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig3_market_share.png)

这种集中化不是偶然的。云基础设施是一个资本支出（CapEx）密集型行业，规模效应非常明显：每新增一个区域或可用区（AZ），就能吸引更多的客户；每增加一项服务，都会与现有服务产生协同效应（比如 S3 让 EC2 更具粘性， IAM 则让两者都更难替代）。小厂商只能靠低价、特定垂直领域（如游戏、科研计算）或者区域性优势来竞争。

| 厂商         | 大致份额 | 优势                                              | 最适合场景                                              |
| ------------ | -------- | ------------------------------------------------- | ------------------------------------------------------- |
| **AWS**      | ~32%     | 产品线最全、历史最久、生态最强大                  | 企业用户、初创公司、全球化业务、多样化选择              |
| **Azure**    | ~23%     | 微软技术栈集成（AD / .NET）、混合云支持           | 使用微软技术栈的企业、受监管行业、政府部门              |
| **GCP**      | ~10%     | 数据与 ML 领先、 K8s 原生支持、网络性能优异        | 数据分析、机器学习 / AI、容器原生架构                   |
| **阿里云**   | ~4%      | 深耕中国市场、亚太地区覆盖广、价格有竞争力        | 中国本土业务、亚太地区部署、对成本敏感的场景            |
| **其他**     | ~30%     | OCI、 IBM、腾讯云、华为云及区域性专家              | 细分市场、区域性或垂直领域特定的工作负载                |
## 5. 经济学 -- CapEx 和 OpEx 的时间对比

云的最大商业价值在于成本曲线的形态。本地部署需要提前按块购买容量，而云则是按秒实时购买。

![CapEx vs OpEx -- 36 个月内的容量与需求曲线](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig4_capex_vs_opex.png)

两种效应叠加在一起：

1. **闲置容量。** 本地硬件按峰值需求采购，但大部分时间都处于闲置状态。企业数据中心的平均利用率通常在 15% 到 30% 之间。
2. **容量不足。** 当需求突然激增（比如产品发布、营销活动或病毒式传播），本地环境无法快速扩展；而云可以在几分钟内吸收这些需求。

对于**波动型工作负载**（大多数工作负载都属于这一类），无论硬件折扣多大，云在原始成本上几乎总是占优。但对于**稳态高利用率**的工作负载（比如 7×24 小时运行在 80% 以上的视频转码集群，或者稳定的内部数据库集群），三年预留实例或私有云建设可能比按需云便宜 30%-50%。

### 隐性成本建模

官网标出的按需价格从来不是全部成本。我总会把这些因素加进去：

- **数据出云费用（Egress）** -- 数据从云中传出要收费，跨区域传输也一样。
- **跨可用区流量** -- 大多数云厂商都会收费；如果微服务之间频繁通信，这部分费用可能会占账单的大头。
- **托管服务溢价** -- RDS 比自己在 EC2 上运行 PostgreSQL 贵大约 30%。
- **支持计划** -- Business 或 Enterprise 级别的支持计划通常占月度支出的 3%-10%。
- **人力成本** -- 平台工程师、 FinOps 分析师、安全工程师；“云团队”是实打实的人力投入。

一份可靠的 TCO 模型至少要看三年，涵盖本地硬件的更新周期和云的预留实例折扣，同时把人力成本分摊到两边。
## 6. 架构构建模块

云不仅仅是虚拟机。以下这些基础组件在任何成熟的架构中都会反复出现。

### 虚拟化 -- 底层基础

**Hypervisor** 让多台虚拟机共享一台物理服务器。主要有两种类型：

- **Type 1 （裸金属型）** -- VMware ESXi、 KVM、 Hyper-V、 Xen。直接运行在硬件上，所有公有云都用这种。
- **Type 2 （宿主型）** -- VirtualBox、 VMware Workstation。运行在主机操作系统之上，适合开发和测试。

容器并不是底层虚拟化的替代品，而是运行在虚拟机之上，共享主机 VM 的内核。一个典型的 EKS 或 AKS Pod 是这样工作的：容器 -> 运行在 Linux VM 中 -> 运行在 hypervisor 上 -> 运行在物理服务器上。这部分内容会在第 2 部分（虚拟化）和第 7 部分（云原生）深入探讨。

| 特性     | 虚拟机                          | 容器                                       |
| -------- | ------------------------------- | ------------------------------------------ |
| 隔离性   | 完整客户操作系统                | 共享内核；通过 namespace 和 cgroup 实现隔离 |
| 启动时间 | 通常 30-90 秒                   | 通常 100 毫秒到 2 秒                       |
| 资源开销 | 每实例 GB 级 RAM                | 每容器几十 MB                              |
| 使用场景 | 混合操作系统、遗留应用          | 微服务、 CI 工作节点、机器学习训练          |

### 存储 -- 按访问模式选择，而不是按名称选择

| 类型     | 示例                                  | 访问方式         | 最佳用途                                      | 关键特性                                |
| -------- | ------------------------------------- | ---------------- | --------------------------------------------- | --------------------------------------- |
| 对象存储 | S3、 GCS、 Azure Blob、 OSS              | HTTP/REST        | 备份、媒体文件、静态网站、数据湖落地          | 11 个 9 的持久性（99.999999999%）       |
| 块存储   | EBS、 Azure Managed Disks、 PD         | iSCSI/virtio 块 | 数据库、高 IOPS 应用、启动盘                  | io2/Hyperdisk 支持高达 80k+ IOPS        |
| 文件存储 | EFS、 Azure Files、 Filestore、 NAS      | NFS/SMB          | 共享内容、迁移遗留应用                        | 支持跨实例挂载                          |
| 归档存储 | Glacier、 Archive Blob、 Coldline      | 异步取回         | 合规归档、长期备份                            | 每 TB 每月几美分，取回需数小时          |

S3 风格的**存储分级**让你用取回延迟换取成本优化。常见策略是：热数据放在 Standard， 30 天后转到 Standard-IA， 180 天后转到 Glacier，一年后进入深度归档 -- 全程由生命周期策略驱动，无需手动操作。

### 网络 -- 最容易出问题的部分

- **VPC （虚拟私有云）** -- 你在云中的隔离网络，每个可用区一个子网，配有路由表、安全组和 NACL。
- **负载均衡器** -- L4 （NLB、 TCP）追求吞吐量， L7 （ALB、 HTTP）支持按路径或 Header 路由。
- **CDN** -- 在靠近用户的边缘节点缓存静态内容（以及越来越多的动态内容）；例如 CloudFront、 Cloud CDN、 Azure CDN、 Cloudflare。
- **VPN/专线** -- IPSec 隧道走低成本路径，专线走低延迟路径（通常 < 2 ms，不经过公网）。

### 自动伸缩 -- 弹性的核心引擎

![弹性伸缩生命周期](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig_autoscaling_zh.png)

两个维度：

- **横向（扩/缩）** -- 增加或减少实例数量，这是云原生的默认方式。
- **纵向（升/降配）** -- 调整实例规格，通常需要重启，因此主要用于数据库等有状态工作负载。

### 地域与可用区 -- 故障域设计

**地域（Region）** 是地理区域（如 us-east-1、 eu-west-1、 ap-southeast-1、 cn-hangzhou）。地域之间相互隔离，一个地域的故障不会影响其他地域。每个地域包含多个**可用区（AZ）**，这些 AZ 是相距数公里的独立数据中心，拥有独立的电源、冷却和网络，但通过亚毫秒级光纤互联。

![地域、可用区与跨地域复制](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig6_regions_and_azs.png)

设计思路：

- **同地域多 AZ** 简单又便宜：副本分布在不同 AZ，负载均衡器自动切换，应用代码几乎不用改动。
- **跨地域** 复杂得多：数据需要通过长距离链路复制，最终一致性问题浮现， DNS 或全局负载均衡需要在故障时重定向流量。
- **边缘/POP** 用于内容分发和抗 DDoS，不适合有状态的应用逻辑。

务实建议：**从第一天起就实现多 AZ 架构**（几乎没有额外成本）；**只有当宕机一分钟的损失大于一个月的复制带宽时，才考虑多地域架构**。
## 7. 责任共担模型

过去十年里，几乎每一起公有云安全事故都源于客户的配置错误 -- 比如公开的 S3 存储桶、泄露的 IAM 密钥、权限过宽的安全组。责任共担模型正是为了明确背后的原因。

![IaaS / PaaS / SaaS 三种模型下的责任共担](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig5_shared_responsibility.png)

云服务提供商始终负责底层基础设施的安全 -- 包括物理访问、 hypervisor、数据中心之间的网络平面。不同服务模型的区别在于**责任边界划在哪里**：

- **IaaS** -- 客户负责操作系统、中间件、应用程序、身份管理以及数据；提供商负责虚拟机（VM）及以下的所有内容。
- **PaaS** -- 操作系统和中间件的管理权交给提供商；客户只需关注应用代码、身份管理和数据。
- **SaaS** -- 几乎所有责任都在提供商这边，客户只需管理身份（谁能登录）和数据（他们能做什么）。

从实际操作的角度来看，有两个关键点需要注意：

1. **身份管理永远是你的责任。** 多因素认证、最小权限原则、密钥轮换、 JIT 访问 -- 这些功能都不会自动启用，需要你主动配置。
2. **数据安全永远是你的责任。** 静态数据加密通常默认开启，但传输中的数据加密需要你自己强制实施；备份策略、数据保留周期以及合规性要求完全由你设计。

关于这部分内容，我会在第 5 章（安全与隐私）中详细展开。
## 8. 服务目录 -- 如何读懂这份菜单

每家大厂都提供数百种服务，但归根结底可以分成几个家族。下面这张图能帮你快速摸清任何一家厂商的服务布局。

![云服务目录速查](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/cloud-computing/fundamentals/fig7_service_catalogue.png)

在浏览服务目录时，我总结了几条实用经验：

- **名字会变，本质不变。** 厂商经常改名或者调整分类，但底层的核心功能（比如计算、对象存储、托管 SQL）一直很稳定。
- **三层结构看清楚。** 最内层是“核心”服务，包括计算、存储、网络、 IAM 和监控，这些功能已经非常成熟，各家差异不大。中间层是“托管”服务，比如数据库、消息队列、容器编排，主要比拼易用性和价格。最外层是“前沿”服务，像 AI、数据仓库、自研芯片，这是厂商争夺用户心智的主战场，也是最容易形成绑定的地方。
- **先看配额，再谈架构。** 每个账号都有软硬配额限制，比如 vCPU 数量、 S3 存储桶数量、 API 请求的 QPS。设计系统之前，务必先搞清楚这些限制。
## 9. 真实案例 -- 架构在压力下的表现

### Netflix -- 全球规模的流媒体服务

2016 年， Netflix 关闭了最后一个数据中心，全面迁移到 AWS。同时，他们自建了 Open Connect CDN，部署在 ISP 内部，专门用来传输视频数据。系统由数百个微服务组成，配备了自研的混沌工程工具链（比如 Chaos Monkey 和 Simian Army），并且实现了多 AZ （可用区）和多区域的容灾机制。**结果：** 支撑超过 2 亿订阅用户，服务可用性达到 99.99% 的水平，还能轻松应对爆款剧集上线时流量翻倍的周末高峰。

### Airbnb -- 从初创公司到全球化平台

2009 年， Airbnb 还是一个运行在 AWS 单一区域的单体应用。到了 2018 年，已经演进为跨多个区域的数百个服务。数据层使用 Aurora 和 Vitess，东西向流量通过 Service Mesh 处理，读路径则依赖 ElastiCache （Redis）和 CloudFront。整个迁移过程是渐进式的，每次拆分一个有界上下文，历时多年，而不是一次性重写。**结果：** 支撑数百万房源，能够平稳应对不可预测的季节性流量高峰。

### Capital One -- 受监管企业的云转型

Capital One 是美国最早全面拥抱公有云的银行之一（2015 年宣布， 2020 年完成迁移）。技术不是最难的部分，真正的挑战在于治理——需要向 SOC、 PCI-DSS 和银行业专属监管机构证明，代码化的控制手段（如 Terraform、 Cloud Custodian 和 CloudFormation 的规则约束）与传统物理控制一样具备可审计性。**结果：** 到 2020 年，关闭了全部 8 个遗留数据中心，基础设施成本降低了 30%-40%，软件交付周期从季度级缩短到天级。（2019 年因 WAF 配置错误导致 SSRF 漏洞的事故，正是客户侧责任共担模型重要性的经典案例。）

### Spotify -- 主动设计的多云架构

Spotify 的流媒体服务主要运行在 GCP 上，而数据和机器学习平台则深度依赖 BigQuery、 Dataflow 和 Pub/Sub。 AWS 在存储和部分计算任务中仍然发挥作用。**结果：** 每天处理 PB 级的用户收听数据，为 Discover Weekly 和 Wrapped 提供支持，同时有意识地管理供应商集中度风险。
## 10. 一套能经得起年终复盘的决策框架

云上最难的不是选技术，而是一年后还能清楚解释当初为什么这么选。下面这套七维框架，是我反复使用、觉得最靠谱的一个。

| 维度                | 要回答的问题                                                  | 选错了会在哪里吃亏                                                |
| ------------------- | ------------------------------------------------------------- | ----------------------------------------------------------------- |
| **负载形态**        | 是稳态还是突发？是 CPU、内存、 IO 还是 GPU 密集型？            | 实例族选错，账单直接翻倍                                          |
| **数据重力**        | 数据目前在哪？规模有多大？                                    | Egress 费用和迁移时间可能远超计算成本                             |
| **延迟预算**        | 用户感知的端到端延迟目标是多少？                              | 决定区域数量、是否用边缘节点、多 AZ 还是多区域架构                |
| **合规要求**        | 是否需要满足 HIPAA / GDPR / PCI 或特定区域的数据主权要求？   | 限制可选区域，有时甚至限制可用厂商                                |
| **团队能力**        | 团队已经熟悉哪些技术栈？                                      | 团队搞不定的“最佳”服务，不如他们能驾驭的“够用”服务               |
| **三年 TCO**        | 包括出站流量、支持费用和人力成本， 3 年总成本是多少？          | 18 个月时可能会被账单吓一跳                                       |
| **绑定容忍度**      | 3 年后想换掉这个服务，迁移成本有多高？                        | 前沿服务很好用，直到你想离开                                      |

### 经得起时间考验的最佳实践

- **从小起步，逐步扩展。** 先在一个账号、一个区域部署非核心服务，这是不错的起点。
- **为故障设计系统。** 假设任何组件，包括托管服务，随时可能挂掉；从第一天起就采用多 AZ 架构。
- **优先使用托管服务处理通用任务。** 没有特别理由，别自己运维 PostgreSQL。
- **合理分配资源并定期回顾。** 大多数集群资源超配 20%-40%；每周检查一次，成本低且收益明显。
- **全面启用加密。** 静态数据默认加密已是标配；传输层必须支持 TLS 1.2+。
- **基础设施即代码。** 使用 Terraform、 Pulumi 或 CloudFormation，生产环境绝不手动操作。
- **第一天就要实现可观测性。** 在流量到来之前，先配置好日志、指标、追踪和告警。
## 11. 常见问题

**云一定比本地便宜吗？**  
不一定。对于波动型工作负载和大多数企业，云更有优势。但如果是超大规模、利用率超过 70% 的稳态负载，比如 Dropbox 曾经从 AWS 迁出存储层，节省了数亿美元，这种情况下自建可能更划算。一定要算清楚三年的 TCO，包括硬件更新周期、出云流量费用和人力成本。

**云安全吗？**  
底层基础设施比绝大多数企业自己搭建的都要安全。至于应用和配置的安全性，完全取决于你的操作水平。责任共担模型是分界线，大部分安全事故都发生在用户负责的部分。

**厂商宕机怎么办？**  
宕机会发生，必须提前设计应对方案。多 AZ 能以极低成本解决常见问题（单个可用区或机房故障）。多地域部署可以覆盖罕见情况（整个区域故障），但成本较高。多云架构能应对极罕见的情况（整个厂商宕机），但代价非常大。投入要匹配你的真实风险承受能力，而不是被新闻头条牵着走。

**该先学哪些技能？**  
先考一个基础认证，比如 AWS Cloud Practitioner 或 Solutions Architect Associate、 Azure Fundamentals、 GCP ACE，这是快速入门的好方法。接着学 Linux 基础、网络知识、基础设施即代码（Terraform）以及容器技术（Docker、 Kubernetes）。厂商特定的知识迁移起来比想象中容易。

**学 AWS、 Azure 还是 GCP？**  
看你的雇主或目标雇主用什么。这三家在概念上差别不大，切换起来几周就能上手，不用花几年时间。 AWS 的就业市场最大； Azure 在企业 IT 领域占主导地位； GCP 在数据和机器学习领域有明显优势。
## 总结

| 概念              | 核心要点                                                                              |
| ----------------- | ------------------------------------------------------------------------------------- |
| 服务模型          | IaaS （控制权） -> PaaS （托管运行时） -> FaaS （按调用计费） -> SaaS （直接使用）         |
| 部署模型          | 默认公有云；监管需求选私有云；遗留系统过渡用混合云；真正有战略意义才考虑多云          |
| 经济学            | OpEx 灵活匹配需求； CapEx 闲置浪费、高峰期又不足； TCO 要看三年总成本                   |
| 责任共担          | 厂商负责底层基础设施安全，我负责身份、数据和配置安全                                  |
| 地域与 AZ         | 一开始就用多 AZ；宕机损失大于复制成本时再扩展到多地域                                 |
| 厂商              | AWS （覆盖广）、 Azure （企业级 / 微软生态）、 GCP （数据 / ML）、阿里云（中国 / APAC）    |
| 选型              | 先看负载形态、数据重力、延迟要求、合规性、团队能力，再算 TCO 和锁定风险 -- 按这个顺序 |
