## 你将学到

- **两条路径**：实时推荐系统其实是两条管道拼起来的——异步的*写路径*（事件 → 状态 → 模型）和同步的*读路径*（请求 → 召回 → 排序 → 返回）。
- **毫秒花在哪了**：延迟不是看平均值，用户感受到的是 p99。我会按阶段、按分位拆解 100 ms 的预算。

- **流式架构**：一个具体的 Kafka + Flink + KV 存储方案，带 checkpoint 和 exactly-once 语义。
- **概念漂移**：怎么检测（ADWIN、 DDM、 Page-Hinkley），检测到之后又该怎么办。

## 1. 为什么需要实时，以及“实时”到底是什么

有三个现实让我不得不考虑实时：

1. **用户会话很短。** Feed 类应用的中位会话时长只有 3-7 分钟。如果模型每天才更新一次，大多数会话在结束前根本不会被模型捕捉到。
2. **热点趋势稍纵即逝。** 一个爆款视频可能在发布后的前 6 小时就完成了 80% 的互动量。昨天训练的离线模型对此完全无能为力。
3. **反馈闭环是模型的核心。** 推荐出去的内容，用户点击与否，直接成为下一轮训练的数据。将反馈闭环从“天级”缩短到“秒级”，是区分学习系统和停滞系统的关键。

但“实时”不是一个绝对概念，而是一个范围：
| 层级 | 更新频率 | 典型用途 |
|------|---------|---------|
| 实时 | < 1 秒 | 会话意图、Feed 内去重、风控信号 |
| 准实时 | 1 秒 – 1 小时 | 近期点击序列、单作者 CTR |
| 小时级 | 1 – 24 小时 | 热门话题、物品热度衰减 |
| 离线 | 1 天以上 | 用户人口属性、物品 Embedding、召回索引 |

最常见的误区是试图让所有东西都实时化。后面图 4 会提到，像人口属性和物品元数据这类特征，走实时管道几乎没有任何 AUC 提升；但近期点击序列却能带来 2-3 个百分点的收益。**实时是一种资源，应花在真正能提升指标的地方。**

## 2. 双路径架构

![实时推荐系统：异步写路径不停刷新状态，同步读路径在硬性 SLO 下提供服务](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig1_pipeline.png)

实时推荐系统可以清晰地分为两条路径：

**写路径（异步，吞吐优先）。** 用户行为数据从客户端发出，进入按 `user_id` 分区的 Kafka topic。 Flink 任务对这些数据进行聚合，生成滚动窗口特征（比如最近 N 次点击、过去 10 分钟的 CTR 等）。处理后的数据会写入两个地方：一个是供读路径使用的*特征存储*（Redis 或 RocksDB），另一个是用于更新模型权重的*在线学习器*。每隔几分钟，新的模型快照会被推送到模型注册中心。

**读路径（同步，延迟敏感）。** 请求到达后，先通过召回模块（基于 Embedding 的 ANN 搜索，结合倒排索引来覆盖新鲜物品），然后从特征存储中一次性拉取特征，接着进入排序阶段，再根据多样性或业务规则进行重排，最后返回结果。整个过程必须控制在 100 毫秒以内。

工程上的关键在于保持这两条路径的解耦。读路径**绝不**参与模型更新、训练或流式计算的阻塞操作。即使流式处理出现延迟，读路径仍然可以用稍微过时的特征继续工作——性能可能下降，但系统不会崩溃。
## 3. 延迟预算 —— 每一毫秒花在哪

![延迟预算按阶段、按分位拆解，展示召回、特征拉取、排序、重排各自在 p50/p95/p99 上的占比](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig2_latency_budget.png)

Feed 类产品的端到端延迟预算通常是 100 ms （人因研究表明，视觉更新“感觉瞬时”的阈值大约在 100-200 ms）。在这 100 ms 的预算里，时间分配大致如下：

| 阶段 | p50 | p95 | p99 | 备注 |
|------|-----|-----|-----|------|
| 入向网络 | 4 | 7 | 12 | 长连接摊薄了 TLS 握手开销 |
| 召回（ANN） | 10 | 18 | 28 | HNSW 或 ScaNN 在亿级物品上的查询 |
| 特征拉取 | 6 | 14 | 30 | Redis pipeline；尾部延迟来自 GC 或网络抖动 |
| 排序（DNN） | 18 | 32 | 55 | 对约 500 个候选进行批量打分 |
| 重排 + 日志 | 4 | 9 | 18 | 多样性调整、业务规则过滤、异步日志记录 |
| 出向网络 | 3 | 6 | 11 |  |
| **端到端** | **45** | **86** | **154** | p99 超过预算，这是常见现象 |

两条实用经验：

1. **均值会骗人。** p50 是 45 ms，看似有余量，但 p99 达到 154 ms，意味着 1% 的请求超预算。如果平台每天处理 10 亿请求，就会有数百万次超时。
2. **排序是大头。** 候选批处理、模型蒸馏、 TensorRT/ONNX-Runtime 量化，这些优化带来的收益远超其他环节。 Pinterest 曾公开分享，将 6 层 DNN 蒸馏成 2 层学生模型并加入特征交叉后， p99 延迟降低了 30%。

## 4. 流式架构：生产环境中的 Kafka + Flink

![流式参考架构：客户端写入 Kafka 主题，Flink 执行有状态聚合与在线学习，输出分发到特征存储、模型注册中心和指标系统](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig5_streaming_arch.png)

### 4.1 Kafka —— 可靠的传输层

Kafka 的定位很明确：一个持久化、分区化、可重放的日志系统。它有三个核心特性：

- **按 `user_id` 分区**：确保同一个用户的事件始终落在同一个分区，保留因果顺序——这对「点击是否发生在曝光之后」这类有状态关联至关重要。
- **副本机制**（通常 `replication-factor=3`）：即使某个 broker 挂了，数据也不会丢失。
- **数据保留**：支持回放最近 7 天的数据，方便新模型的回填——线上和恢复路径共用同一套代码。

Kafka 不负责的事情：聚合、 join、机器学习。它的角色更像是一个邮局。

```python
from kafka import KafkaProducer
import json, time

producer = KafkaProducer(
    bootstrap_servers=["kafka-1:9092", "kafka-2:9092", "kafka-3:9092"],
    value_serializer=lambda v: json.dumps(v).encode(),
    key_serializer=lambda k: k.encode(),
    acks="all",        # 等待所有 ISR 副本确认
    enable_idempotence=True,  # 生产者端 exactly-once 语义
    compression_type="lz4",
    linger_ms=5,       # 5 毫秒微批处理，用 5ms 延迟换取 10 倍吞吐
)

def emit_click(user_id: str, item_id: str, position: int) -> None:
    """发送一次点击事件，同一用户始终落同一个分区。"""
    producer.send(
        topic="clicks",
        key=user_id,                # 分区键
        value={
            "user_id": user_id,
            "item_id": item_id,
            "position": position,
            "ts": int(time.time() * 1000),
        },
    )
```

### 4.2 Flink —— 有状态的流处理引擎

如果说 Kafka 是传输层， Flink 就是有状态的流处理器。它的杀手锏是**故障下的 exactly-once 处理**，基于 Chandy-Lamport 风格的分布式快照实现：每隔 60 秒（默认 checkpoint 周期）， Flink 会原子性地捕获每个算子的状态并持久化到 S3。发生故障时， Flink 会将 Kafka offset 回退到上一个 checkpoint 并重新处理——对外表现就像没出过问题一样。

一个典型的 Flink 点击归因任务如下：

```python
# Flink SQL —— 每用户最近 10 分钟点击数，写入特征存储
from pyflink.table import EnvironmentSettings, StreamTableEnvironment

t_env = StreamTableEnvironment.create(
    environment_settings=EnvironmentSettings.in_streaming_mode()
)

t_env.execute_sql("""
CREATE TABLE clicks (
    user_id STRING,
    item_id STRING,
    ts BIGINT,
    event_time AS TO_TIMESTAMP_LTZ(ts, 3),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'clicks',
    'properties.bootstrap.servers' = 'kafka-1:9092',
    'format' = 'json',
    'scan.startup.mode' = 'latest-offset'
)
""")

t_env.execute_sql("""
CREATE TABLE user_features (
    user_id STRING,
    window_end TIMESTAMP(3),
    clicks_10m BIGINT,
    PRIMARY KEY (user_id) NOT ENFORCED
) WITH (
    'connector' = 'redis',
    'host' = 'redis.prod',
    'ttl-sec' = '900'
)
""")

# 10 分钟滑动窗口，步长 1 分钟 —— 每分钟输出一次最新的 CTR
t_env.execute_sql("""
INSERT INTO user_features
SELECT
    user_id,
    HOP_END(event_time, INTERVAL '1' MINUTE, INTERVAL '10' MINUTE) AS window_end,
    COUNT(*) AS clicks_10m
FROM clicks
GROUP BY
    user_id,
    HOP(event_time, INTERVAL '1' MINUTE, INTERVAL '10' MINUTE)
""")
```

watermark 是最容易踩坑的地方。它的作用是告诉 Flink：「我不会再收到 `event_time < watermark` 的事件了。」这授权 Flink 关闭窗口并输出结果。设置得太紧，晚到的事件会被丢弃；设置得太松，特征更新就会延迟。
## 5. 在线学习：从 SGD 到 FTRL

![推荐系统（十五）—— 实时推荐与在线学习 — visual](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/illustration_2.png)

![在线学习 vs 离线重训。左：稳态任务的收敛过程；右：概念漂移下的表现](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig3_online_vs_batch.png)

### 5.1 核心更新公式

假设有一个逻辑回归排序器，参数是 $\theta$，单条样本是 $(x_t, y_t)$。它的更新公式如下：

$$\theta_{t+1} = \theta_t - \eta_t \, \nabla_\theta \mathcal{L}(\sigma(\theta_t^\top x_t), y_t)$$
这就是普通的 SGD。虽然能用，但在 Web 规模的 CTR 数据上——特征数以百万计，每个特征只在极小比例的样本中出现——它有两个明显问题：

1. **没有按特征的学习率**：一个特征每 1 万条数据才出现一次，显然需要更大的步长，而高频特征则需要更谨慎。
2. **缺乏稀疏性**：权重会在噪声中漂移，无法真正归零。一个有 10⁹ 参数且永不归零的模型，根本没法上线服务。

### 5.2 AdaGrad —— 按特征自适应步长

AdaGrad 解决了第一个问题，通过累积每个特征的梯度平方来调整步长：
$$\theta_{t+1, i} = \theta_{t, i} - \frac{\eta}{\sqrt{G_{t, i} + \varepsilon}} g_{t, i}, \quad G_{t, i} = \sum_{s=1}^{t} g_{s, i}^2$$
罕见特征的 $G$ 小，因此在少数几次出现时可以迈大步；高频特征的 $G$ 大，更新会更加保守。

### 5.3 FTRL-Proximal —— 生产环境的主力算法

FTRL-Proximal （McMahan 等，《Ad Click Prediction: a View from the Trenches》， KDD 2013——Google 公开其广告系统核心算法的论文）结合了 AdaGrad 的按特征自适应学习率和 $L_1$ 正则化，后者能够生成*真正的零值*，而不仅仅是小权重。逐坐标更新规则如下：
$$
z_{t,i} \leftarrow z_{t-1,i} + g_{t,i} - \frac{\sigma_{t,i}}{\eta} \theta_{t-1,i},
\qquad
\theta_{t,i} =
\begin{cases}
0 & \text{若 } |z_{t,i}| \le \lambda_1 \\
-\frac{1}{\eta_{t,i}} \big(z_{t,i} - \mathrm{sign}(z_{t,i}) \lambda_1\big) & \text{否则}
\end{cases}
$$
其中 $\sigma_{t,i} = \frac{1}{\eta_{t,i}} - \frac{1}{\eta_{t-1,i}}$，$\eta_{t,i} = \alpha / (\beta + \sqrt{\sum_s g_{s,i}^2})$。

关键特性：模型是真正稀疏的。 Google 报告称，相比朴素 SGD 加 $L_2$ 正则化， FTRL-Proximal 在相同 AUC 下将模型体积压缩了一个数量级。

```python
import numpy as np

class FTRLProximal:
    """FTRL-Proximal 在线逻辑回归。

    参考文献：McMahan 等，"Ad Click Prediction: a View from the
    Trenches"，KDD 2013。Google 当时广告系统真正运行的算法。
    """
    def __init__(self, n_features: int, alpha=0.1, beta=1.0,
                 l1=1.0, l2=1.0):
        self.n = n_features
        self.alpha, self.beta, self.l1, self.l2 = alpha, beta, l1, l2
        # 只持久化 z 和 n，权重 w 按需懒计算
        self.z = np.zeros(n_features)
        self.n_sq = np.zeros(n_features)

    def _weight(self, i: int) -> float:
        """从 (z_i, n_sq_i) 懒计算 w_i。"""
        z, n_sq = self.z[i], self.n_sq[i]
        if abs(z) <= self.l1:
            return 0.0
        sign = 1.0 if z > 0 else -1.0
        return -(z - sign * self.l1) / (
            (self.beta + np.sqrt(n_sq)) / self.alpha + self.l2
        )

    def predict_proba(self, x_indices: np.ndarray, x_values: np.ndarray) -> float:
        """稀疏预测，只在非零特征上累加。"""
        s = sum(self._weight(i) * v for i, v in zip(x_indices, x_values))
        return 1.0 / (1.0 + np.exp(-max(min(s, 35.0), -35.0)))

    def update(self, x_indices: np.ndarray, x_values: np.ndarray, y: int) -> float:
        """单样本更新，返回更新前的预测概率。"""
        p = self.predict_proba(x_indices, x_values)
        for i, v in zip(x_indices, x_values):
            g = (p - y) * v                       # 该特征上的梯度
            sigma = (np.sqrt(self.n_sq[i] + g * g) - np.sqrt(self.n_sq[i])) / self.alpha
            self.z[i]   += g - sigma * self._weight(i)
            self.n_sq[i] += g * g
        return p
```

### 5.4 在线 vs 离线 —— 全景对比

图 3 左侧展示了稳态任务的表现：在线学习平滑收敛，离线重训则是阶梯式——每 200 条事件生成一个新快照，中间停滞不前。在静态任务上，两者的差距最终会缩小，但从时间积分来看，在线学习占优。

右侧才是关键：突发分布漂移（爆款事件、新品上线、节日效应）。离线模型对训练窗口外的变化一无所知，需要一个完整窗口才能恢复；在线学习从第一条新样本就开始调整，大约 100 条事件后就能接近峰值 AUC。**生产环境中，漂移是常态，不是例外**——这才是在线学习成为工程利器的原因，而不仅仅是一个学术偏好。

## 6. 特征新鲜度 —— 真的那么重要吗？

![特征陈旧度 vs AUC：AUC 在 log 陈旧度上近似线性下降，几乎所有损失都集中在行为类特征](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig4_freshness_auc.png)

特征新鲜度是每个认真做推荐系统的团队都会争论的问题。原因很简单，把特征做成实时太贵了。从公开资料来看（比如 Meta 的深度学习推荐模型、 Pinterest 的 PinnerSAGE、字节的 Monolith），经验规律非常一致：

- **AUC 随 log 陈旧度大致线性下降。** 从 1 秒延迟到 1 分钟，几乎没什么影响。从 1 分钟延迟到 1 小时， AUC 损失约 0.005，已经能感受到差距。从 1 小时延迟到 1 天，损失显著增加，大约在 0.015 到 0.020 AUC。
- **损失主要来自行为类特征。** 近期点击序列和会话意图占了新鲜度溢价的 80% 左右。人口属性和物品元数据基本没变化，即使每周更新一次也看不出明显影响。

这直接影响了系统架构设计：不要为那些无法带来收益的特征支付高昂的流式处理成本。一个典型的 Feed 系统通常会并行运行三条流水线——行为特征走实时更新，热度类聚合按小时级更新， Embedding 和人口属性则按天级更新。
## 7. Bandit —— 探索与利用的理论故事

### 7.1 一句话讲清楚问题

你有 $K$ 个物品可选，每个物品的点击概率 $\mu_i$ 是未知的。每轮选一个，观察点击结果并更新。经过 $T$ 轮后，**遗憾**是你比不上一直选最优物品所损失的收益：
$$R_T = T \mu^* - \mathbb{E}\!\left[\sum_{t=1}^T \mu_{a_t}\right]$$
好的算法要做到*次线性遗憾*：$R_T = o(T)$，也就是平均每轮的损失趋近于 0。

### 7.2 UCB1 —— 不确定时保持乐观

UCB1 （Auer 等， 2002）选择上置信界最高的臂：
$$a_t = \arg\max_i \left( \hat\mu_i + \sqrt{\frac{2 \ln t}{n_i}} \right)$$
其中 $n_i$ 是臂 $i$ 被选择的次数。随着 $n_i$ 增加，探索奖励逐渐缩小——不确定时多探索，确定后转向利用。 UCB1 的遗憾是 $O(\log T)$，对稳态 Bandit 来说这是理论上最优的结果（Lai-Robbins 下界）。

### 7.3 Thompson Sampling —— 贝叶斯方法

为每个臂的成功率维护一个后验分布。对于伯努利奖励，共轭先验是 Beta 分布：
$$\theta_i \sim \text{Beta}(\alpha_i, \beta_i), \quad \text{更新规则：} (\alpha_i, \beta_i) \leftarrow (\alpha_i + r, \beta_i + 1 - r)$$
每轮从每个臂的后验分布中采样一个 $\theta_i$，然后选择采样值最大的臂。后验分布宽的臂会被更多探索（因为样本噪声大，偶尔会高出其他臂）；后验分布窄的臂则会被更多利用。 Thompson Sampling 在渐近意义上达到和 UCB1 相同的 $O(\log T)$ 遗憾（Agrawal & Goyal， 2012），并且在大多数基准测试中表现更好，同时更容易扩展到延迟反馈、批量更新和复杂奖励模型。

```python
class ThompsonSampling:
    """伯努利 Thompson Sampling。生产环境实现基本就是这样——
    唯一需要加的是衰减（每天 alpha, beta *= 0.99），
    用来应对非稳态的物品池。"""

    def __init__(self, n_arms: int):
        self.alpha = np.ones(n_arms)   # 成功次数 + 1
        self.beta  = np.ones(n_arms)   # 失败次数 + 1

    def select(self) -> int:
        return int(np.argmax(np.random.beta(self.alpha, self.beta)))

    def update(self, arm: int, reward: int) -> None:
        if reward:
            self.alpha[arm] += 1
        else:
            self.beta[arm] += 1
```

### 7.4 LinUCB —— 上下文驱动的个性化推荐

普通 Bandit 学习的是全局排序，但推荐系统的核心价值在于*个性化*。 LinUCB （Li 等，《A Contextual-Bandit Approach to Personalized News Article Recommendation》， WWW 2010 —— 雅虎当年用于首页新闻个性化的算法）假设期望奖励对上下文向量 $x_t$ 是线性的：
$$\mathbb{E}[r_a \mid x_t] = x_t^\top \theta_a$$
每个臂维护一个岭回归模型 $(A_a, b_a)$，其中 $A_a = I + \sum X X^\top$、$b_a = \sum r X$。选择规则如下：
$$a_t = \arg\max_a \left( x_t^\top \hat\theta_a + \alpha \sqrt{x_t^\top A_a^{-1} x_t} \right), \quad \hat\theta_a = A_a^{-1} b_a$$

奖金项 $\sqrt{x_t^\top A_a^{-1} x_t}$ 是岭回归在当前上下文下的预测标准差——当上下文与该臂的历史样本差异较大时，奖金会更高。遗憾界是 $\tilde O(\sqrt{d T})$，其中 $d$ 是上下文维度。

```python
class LinUCB:
    """LinUCB —— disjoint 模型版本。
    参考文献：Li, Chu, Langford, Schapire，WWW 2010。"""

    def __init__(self, n_arms: int, n_features: int, alpha: float = 1.0):
        self.alpha = alpha
        self.A = [np.eye(n_features) for _ in range(n_arms)]
        self.b = [np.zeros(n_features) for _ in range(n_arms)]

    def select(self, x: np.ndarray) -> int:
        ucbs = []
        for A_a, b_a in zip(self.A, self.b):
            A_inv = np.linalg.inv(A_a)
            theta = A_inv @ b_a
            ucbs.append(x @ theta + self.alpha * np.sqrt(x @ A_inv @ x))
        return int(np.argmax(ucbs))

    def update(self, arm: int, x: np.ndarray, reward: float) -> None:
        self.A[arm] += np.outer(x, x)
        self.b[arm] += reward * x
```

实际应用中几乎不会直接用「纯」 LinUCB。线上服务跑的是深度排序模型， Bandit 在其上一层，负责决定给某次请求分配哪种策略（比如创作者助推、新物品助推或纯利用）。这里的臂是策略，而不是具体的物品。
## 8. 概念漂移 —— 检测它，别装看不见

![概念漂移检测：上图是观测 CTR 加滚动均值与参考窗口；下图是在渐进与突发漂移上触发的 z-score 检测器](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig6_drift_detection.png)

在线学习只有在学习器*能*适应时才有效。如果学习率已经衰减到接近零，模型就无法应对漂移。生产环境会主动检测漂移并采取措施。

### 8.1 三种经典检测方法

- **Page-Hinkley 检验**：累积偏离均值的偏差，当 CUSUM 超过阈值时报警。适合单调性漂移。
- **DDM （Drift Detection Method， Gama 等 2004）**：跟踪二项分布错误率 $p_t$ 和其标准差 $s_t$。当 $p_t + s_t \ge p_{\min} + 2 s_{\min}$ 时预警，$\ge p_{\min} + 3 s_{\min}$ 时报警。
- **ADWIN （Bifet & Gavaldà 2007）**：维护一个自适应窗口，当两个子窗口的均值差异超过 Hoeffding 界时，丢弃较旧的一半。误报率有理论保证。

图 6 展示的是简化版 z-score 检测器，也是实际部署 v1 的首选方案：保留一个「已知良好」的参考窗口，计算近期滚动均值 CTR 相对参考窗口的 z-score，当 $|z| > 3$ 时触发报警。

### 8.2 应对漂移的策略

光检测不行动只是发个告警。实际应对策略按成本从低到高排序：

1. **提高学习率**（低成本、可逆）。将 $\eta$ 在固定窗口内放大 2-5 倍。
2. **重置 Bandit 后验**（中等成本）。把受影响臂的 $\alpha, \beta$ 减半——保留先验形状，但增加不确定性。
3. **强制回滚 checkpoint**（高成本）。回退到上一个已知良好的快照，从 Kafka 重放数据，在漂移后的窗口上重新训练。

```python
class DriftAdaptiveLearner:
    def __init__(self, base_learner, base_lr=0.1, window=200, z_thresh=3.0):
        self.learner = base_learner
        self.base_lr = base_lr
        self.window = window
        self.z_thresh = z_thresh
        self.recent = []        # 滚动误差
        self.ref_mean = None
        self.ref_std = None

    def update(self, x, y):
        pred = self.learner.predict(x)
        err = abs(pred - y)
        self.recent.append(err)
        if len(self.recent) > self.window:
            self.recent.pop(0)

        # 第一个窗口稳定后建立参考
        if self.ref_mean is None and len(self.recent) == self.window:
            self.ref_mean = np.mean(self.recent)
            self.ref_std = np.std(self.recent) + 1e-6

        # 检测漂移
        if self.ref_mean is not None:
            z = (np.mean(self.recent[-50:]) - self.ref_mean) / self.ref_std
            if abs(z) > self.z_thresh:
                self.learner.learning_rate = self.base_lr * 4   # 提高学习率
                # 处理后重置参考，避免长期处于过激状态
                self.recent = []
                self.ref_mean = None

        self.learner.update(x, y)
```
## 9. 缓存与计算 —— 真正需要调优的权衡

![缓存 vs 计算：左图是延迟、新鲜度、成本随缓存 TTL 的变化；右图是各策略的 Pareto 前沿，混合策略胜出](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/15-实时推荐与在线学习/fig7_cache_vs_compute.png)

读路径上的每个决策，最终都会落在缓存和计算的权衡曲线上：

- **总是重算**：结果最准确，但代价高昂。每次请求都得打模型，每个特征都实时计算。
- **激进缓存**：速度快、成本低，但数据陈旧。用户意图刚变，拿到的还是几分钟前的特征。
- **混合模式**：生产系统基本都采用这种方案：
  - **热路径**（头部用户 / 活跃会话 / 新物品）：实时计算，不走缓存。
  - **冷路径**（其他流量）：从缓存返回， TTL 设置为 30-60 秒。
  - **负缓存**：显式缓存「无变化」的响应，避免对静默用户的重复计算。

图 7 右侧说明了原因： 60 秒的 TTL 能获得大部分延迟优化收益，新鲜度损失却很小；纯重算花了 5 倍计算资源，只提升了约 0.005 AUC。「总是重算」虽然在 Pareto 前沿上，但很少值得采用。

## 10. 拼到一起 —— 一个最小但接近真实的系统

```python
class RealtimeRanker:
    """端到端实时排序器。整合：
      - 特征存储（Flink 提供的新鲜特征）
      - FTRL-Proximal 在线学习器（按事件更新权重）
      - LinUCB 探索层（基于学习器得分）
      - 漂移检测器（在分布变化时提升学习率）
      - 快照回滚机制（保障生产安全）
    """

    def __init__(self, n_features: int, n_strategies: int = 4):
        self.scorer = FTRLProximal(n_features=n_features, alpha=0.1,
                                    beta=1.0, l1=1.0, l2=1.0)
        # Bandit 选择的是排序策略，不是物品
        self.bandit = LinUCB(n_arms=n_strategies, n_features=n_features, alpha=0.5)
        self.drift  = DriftDetector(z_thresh=3.0)
        self.snapshots = []  # 环形缓冲区，存 (时间戳, 学习器状态)

    def rank(self, ctx: np.ndarray, candidates: list[dict]) -> list[str]:
        # 1. Bandit 选择策略
        strategy = self.bandit.select(ctx)
        # 2. FTRL 给每个候选打分——稀疏向量
        scored = [
            (c["id"], self.scorer.predict_proba(c["idx"], c["val"]), c)
            for c in candidates
        ]
        # 3. 策略调整得分（比如新品助推、创作者助推）
        scored = apply_strategy(strategy, scored)
        scored.sort(key=lambda r: -r[1])
        return [r[0] for r in scored[:20]]

    def observe(self, ctx, x_idx, x_val, y, strategy):
        """一次反馈循环。"""
        # 更新在线学习器
        p = self.scorer.update(x_idx, x_val, y)
        # 更新 Bandit，针对实际使用的策略
        self.bandit.update(strategy, ctx, reward=float(y))
        # 将校准误差喂给漂移检测器
        if self.drift.update(abs(p - y)):
            self.scorer.alpha *= 4  # 提升学习率
        # 周期性保存快照
        if len(self.snapshots) == 0 or time.time() - self.snapshots[-1][0] > 600:
            self.snapshots.append((time.time(), copy.deepcopy(self.scorer)))
            self.snapshots = self.snapshots[-12:]  # 保留最近 2 小时
```

几个看似细节、但在生产环境中非常关键的点：

- Bandit 运行在策略层面，而不是物品层面。「从一亿个物品中选一个」的动作空间太大，任何上下文 Bandit 都无法在合理时间内收敛；但「从 4 个排序策略中选一个」正好是它的强项。
- 快照保存的是在线学习器的状态，而不是预测结果。如果一批坏数据污染了模型权重，可以回滚状态，再从 Kafka 重放干净的事件流来恢复。
- `DriftDetector` 监控的是校准误差 `|p - y|`，而不是原始 CTR。相比单纯看点击率，校准漂移能捕捉到更多失效模式。

## Q & A

**Q：如何对在线学习系统进行 A/B 测试？**  
标准 A/B 测试假设观测值相互独立，但在线学习系统会从实验组中不断学习。解决方法有两种：一是使用 **interleaved comparisons**（Chapelle 等， 2012），二是将一个实验组固定为「冻结模型」，另一个实验组使用实时在线模型，比较的是*累积奖励*，而不是单次请求的 CTR。

**Q： Flink 和 Spark Streaming 的本质区别是什么？**  
Flink 是逐事件处理，延迟在毫秒级； Spark Streaming 是微批处理，延迟在秒级。对于推荐系统来说， Flink 的低延迟和更成熟的 exactly-once 状态管理都很重要——几乎所有近年大规模构建的 Feed 系统都选择了 Flink （或者类似 Twitter 的 Heron、字节跳动的 Aiops 这样的内部解决方案）。

**Q：在线学习是否不稳定？**  
确实可能不稳定，而这就是需要通过工程手段解决的失效模式。我通常用三种防护措施：(1) 限制学习率；(2) 梯度裁剪；(3) 保留快照并设置回滚路径。 Google 的 FTRL 论文专门有一节讲「trade tricks」——饱和保护、按坐标的学习率下限、校准损失监控——这些技术的核心目标就是让在线学习在生产环境中表现稳定。

**Q： Bandit 如何处理延迟反馈？**  
两种方式。**批量更新**：收集 $k$ 条观测后一次性更新，然后重复这个过程。 UCB 和 Thompson Sampling 都有理论证明，批量更新最多会让遗憾增加 $\log k$ 倍。**乐观计数**：每次拉取一个臂时，立即增加 $n_i$ 并假设奖励等于历史均值（或者为了安全起见设为 0），等真实奖励到达后再修正。生产环境普遍采用乐观计数，因为它能避免在延迟窗口内过度采样某个单一臂。

**Q：什么时候不应该使用实时管道？**  
特征变化缓慢时（如人口属性、长期口味 Embedding），陈旧推荐的成本较低时（如仓库搜索、分类浏览），或者收益难以衡量时（如低流量场景中从 1 小时缩短到 1 秒的新鲜度提升淹没在噪声中）。一条实时管道的成本大约是离线日批的 10-100 倍——确保收益足够再投入。
## 总结

- 实时推荐系统是 **两条管道通过特征存储和模型注册中心连接起来**：异步写路径持续更新状态，同步读路径在严格的 SLO 下提供服务。
- **延迟是个长尾问题。** 优化 p99，而不是 p50——排序模块占主导地位，蒸馏和量化在这里最有价值。
- **生产环境中的在线学习用的是 FTRL-Proximal。** 这是 Google 提出的广告 CTR 算法， SGD 和 AdaGrad 只是铺垫。
- **流处理 = Kafka + Flink。** Kafka 负责可靠传输， Flink 借助分布式快照实现有状态计算和精准一次语义。
- **新鲜度有成本曲线。** 行为特征值得投入实时成本，人口统计和物品元数据不值得。
- **Bandit 在排序器之上工作，而不是针对单个物品。** 它的动作空间是「选择哪种策略」，而不是「推荐哪个物品」。
- **漂移不可避免。** 检测它（z-score、 ADWIN、 DDM），低成本应对（调高 $\eta$），并保留回滚方案。
- **缓存和计算不是非此即彼。** 生产环境的答案是混合模式——热路径实时计算，冷路径缓存结果， 30-60 秒 TTL 以极低成本覆盖 95% 的流量。

我在每个见过的大规模系统中都验证过一条经验：*只把能提升 AUC 的部分做成实时，其余按实际变化频率分配预算*。
