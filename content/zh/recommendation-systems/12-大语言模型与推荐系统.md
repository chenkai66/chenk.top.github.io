用户打开电影 App，输入：“想看类似《盗梦空间》的，但别太压抑。”传统推荐系统如协同过滤、双塔 DNN 甚至 DIN 无法从这句话中提取任何可用信号——没有点赞按钮可参考，没有共看关系图可查，也没有用户 ID 关联的历史记录——系统必须先将这句话转换为 ID 才能继续处理。

大语言模型的情况正好相反：它拥有海量的世界知识，却缺乏当前用户的上下文。它清楚《盗梦空间》是诺兰导演的作品，采用非线性叙事，结尾充满希望但模棱两可；它明白电影中“压抑”的含义；它能轻松列出二十部符合描述的影片，却无法判断这些影片中哪些是用户已看过、打过低分，或中途放弃的。

2023 到 2026 年，推荐系统领域真正值得关注的问题不是 LLM 还是传统模型，而是如何将它们结合起来。本文详细介绍了三种已在工业界大规模落地的组合模式，并附上代码和实际权衡。

## LLM 在推荐系统中的三种角色

![LLM 在推荐中的三种角色：增强器、预测器、智能体](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig1_three_roles.png)

相关文献看似纷繁——P5、M6-Rec、KAR、TallRec、BIGRec、GenRec、LlamaRec、ChatREC——实则均可归入三类角色。角色决定了延迟、成本、训练开销，以及 LLM 在系统中的具体作用。

### 角色一：增强器（离线特征生成）
LLM 不参与在线请求。离线时，它读取物品的标题、描述和评论，生成更丰富的特征：结构化属性、稠密语义向量、扩写后的描述或伪标签，这些特征随后被送入一个完全传统的 CTR/CF 系统。

这是最早一批「LLM + 推荐」工作的核心思路：**P5**（Geng et al., RecSys 2022）用统一的文本到文本 T5 模型整合了五种推荐任务；**M6-Rec**（Cui et al., 2022）用阿里 M6 生成物品描述和行为提示；**KAR**（Xi et al., 2024）将 LLM 的推理结果蒸馏成可复用的特征向量。在线服务延迟可稳定控制在 50ms 以内，因为 LLM 计算开销在百万级请求中被均摊。

**适用场景**：我已经有可用的排序模型，但冷启动物品、稀疏文本或难以打标的类别正在拖累召回。

### 角色二：预测器（直接排序或生成）

LLM 就是排序器。将用户历史和候选列表（甚至不给候选）交给它，它会输出 yes/no 判断、分数，或者直接生成下一个物品。

两个流派：**判别式**（TallRec、 BIGRec——用 LoRA 在 `(历史, 物品) → yes/no` 上微调 Llama-2-7B）和**生成式**（GenRec——直接生成下一部电影的标题）。**TallRec**（Bao et al., RecSys 2023）有一个惊艳的结果：用 LoRA 微调的 7B 模型，在只有几百条训练样本的情况下，击败了大得多的零样本 LLM，并追平强序列推荐基线。**LlamaRec**（Yue et al., 2023）更进一步，加了一个 verbalizer 头，让 LLM 一次前向就能对整个候选集排序，而不是 K 次单独调用。

**适用场景**：物品文本信息丰富、数据量小（冷启动领域、新业务线），或者需要排序结果自带解释。

### 角色三：智能体（带工具的编排者）

LLM 不仅做排序，还负责规划。它维护对话状态，决定何时调用检索、何时反问澄清、何时按年份或价格过滤、何时把任务交给下游模型。**ChatREC**（Gao et al., 2023）是这个范式的典型代表；**RecAgent**（Wang et al., 2023）和 **InteRecAgent** 进一步引入模拟用户进行评测。

**适用场景**：我在构建对话式产品（聊天机器人、语音助手、带追问的搜索），或者用户确实没法一句话说清需求。

成本排序大致是：增强器 ≪ 预测器 ≪ 智能体。灵活性的排序也是这样。多数生产系统从增强器起步，升级为预测器（接在传统排序后面做重排），只在特定产品形态上才用智能体。

## 基于 Prompt 的排序

![LLM-as-recommender 的 Prompt 模板解剖图](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig2_prompt_template.png)

基于 Prompt 的推荐系统是入门首选：不用训练，只需一个 API key 和精心设计的 Prompt。模板分为五个部分，顺序至关重要。

```python
def build_recommendation_prompt(history, target, examples=None):
    """TallRec / P5 风格的排序 Prompt。"""
    # 1. 角色 + 任务：限定 LLM 的行为范围
    role = (
        "You are a movie recommender. Given the user's viewing history, "
        "predict whether they will like the target movie."
    )

    # 2. 用户历史：按时间顺序排列，带喜好信号
    hist_lines = "\n".join(
        f"  - {item['title']} ({'liked' if item['liked'] else 'disliked'})"
        for item in history[-20:]   # 截断以适应上下文长度
    )

    # 3. 目标物品：附带属性信息
    target_line = (
        f'Target movie: "{target["title"]}" '
        f'({target["genre"]}, {target["year"]}, dir. {target["director"]})'
    )

    # 4. 输出格式：强约束，尽量单 token 输出
    output_spec = 'Answer with exactly one token: "Yes" or "No".'

    # 5. Few-shot 示例：可选，但能显著提升精度
    fewshot = ""
    if examples:
        fewshot = "\n".join(
            f"[Example {i+1}]\nHistory: {ex['history']}\n"
            f"Target: {ex['target']}\nAnswer: {ex['answer']}\n"
            for i, ex in enumerate(examples)
        )

    return f"""{role}

{fewshot}
History:
{hist_lines}

{target_line}

{output_spec}"""
```

有三条工程规则看似简单，却每一条都至关重要：

1. **逐 token 约束输出**。`Yes` 或 `No` 这种单 token 输出可以解析；像「我觉得用户可能会喜欢，因为……」这种自由文本完全不可用。对 K 个物品排序时，直接要求模型输出 `[3, 1, 5, 2, 4]`，避免自由文本。如果 SDK 支持，使用 logit bias 或语法约束解码。
2. **温度控制在 0.3 以下**。推荐不是创意写作，高温度会导致相同输入产生不同排序，调试起来让人崩溃。
3. **截断历史记录**。 LLM 处理超过 50 条历史记录时性能会下降，且成本随 prompt token 数线性增长。优先保留最近的记录，旧记录可以通过摘要方式压缩。

### 零样本 vs 少样本 vs 微调

| 方案 | 启动成本 | 单次调用成本 | 冷启动精度 | 热启动精度 |
|---|---|---|---|---|
| 零样本（GPT-4o） | 数小时 | 高（$0.01+/次） | 强 | 一般 |
| 少样本（3-5 例） | 数小时 | 更高（更多 token） | 强 | 较好 |
| 微调（TallRec， 7B+LoRA） | 数天 + GPU | 低（$0.0005/次） | 强 | 强 |

TallRec 的结果令人印象深刻：仅用 **256 条微调样本**，经过 LoRA 微调的 Llama-7B 就能在推荐任务上追平甚至超越零样本 GPT-4，而单次调用成本仅为零头。如果你手头有一些带标签的交互数据，微调绝对是值得尝试的方向。

## LLM 作为增强器：更精准的 Embedding，更优的冷启动

![LLM 描述增强前后的 embedding 聚类对比](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig3_embedding_quality.png)

将 LLM 引入生产环境最经济、风险最低的方式是：离线使用它扩展物品的简短文本为丰富描述，再用现有的句向量编码器重新生成嵌入。

电影标题 `"Tenet"` 对句向量编码器来说几乎没有信息量。让 GPT-4 将其扩写为：

> 「2020 年由克里斯托弗·诺兰执导的科幻动作惊悚片。以熵的逆转为核心剧情装置，探讨自由意志与宿命论的主题。基调冷峻且氛围感强，包含精心设计的实拍动作场面。适合喜欢《盗梦空间》《星际穿越》或其他烧脑叙事影片的观众。节奏紧凑，情感核心围绕友谊与牺牲。」

——生成的嵌入会显著靠近其真实的语义邻居。在 MovieLens 类型数据上的离线测试中，按类型聚类的轮廓系数（silhouette）通常从 ~0.2 （原始标题）提升到 ~0.65 （LLM 增强后）， P5 和 KAR 的论文也报告了类似的效果。

```python
from openai import OpenAI

client = OpenAI()

def enhance_item(item):
    """离线：将稀疏的物品文本扩展为丰富的语义描述。"""
    prompt = f"""Write an 80-word description of this item for a
recommendation system. Cover: themes, tone, mood, target audience,
similar items, what kind of user would love it.

Title: {item['title']}
Existing tags: {', '.join(item.get('tags', []))}
Year: {item.get('year', 'unknown')}

Description:"""
    return client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.4,
        max_tokens=200,
    ).choices[0].message.content

def build_enhanced_index(items, encoder):
    """每个物品处理一次；缓存永久有效（直到物品变更）。"""
    enhanced_texts = [enhance_item(it) for it in items]
    return encoder.encode(enhanced_texts, batch_size=64)
```

这种方式之所以经济，是因为**增强过程是离线的、一次性完成的**。一个百万物品的目录，每次调用成本 $0.0002，总计 $200，仅需支付一次。如果每次在线请求都调用 LLM，单次成本相同，但每百万次查询需要 $200，**持续累积**。

### 离线约束的痛点在哪里

LLM 增强只能生成稳定的**物品侧特征**，无法做到以下几点：

- 响应**新物品**上线后第一小时的交互反馈  
- 捕捉用户专属信号（如口味、当前情绪、本次会话意图）  
- 反映突发趋势（如「最近大家都在看 X，因为 Y」）

这些场景需要在线调用 LLM，或者更实际地说，需要混合管道来解决。

## 混合管线（最终都会采用的架构）

![推荐系统（十二）—— 大语言模型与推荐系统 — visual](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/illustration_2.png)

![混合管线：ANN 检索 + DNN 排序 + LLM 重排](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig4_hybrid_pipeline.png)

每个将 LLM 推向生产环境的团队，最终都会收敛到同一个架构：

1. **物品池**（10⁶–10⁸）
2. **ANN 检索** 使用 Faiss/HNSW，耗时 ~10ms，成本 ~$0，筛选到约 1000 个候选
3. **CF 或深度排序**（DIN、 DCN-V2、双塔），耗时 ~30ms，成本 ~$0，筛选到约 50 个
4. **LLM 重排**（GPT-4o 或微调的 7B），耗时 ~1-2s，成本 ~$0.005-0.01，选出 Top-10
5. **用户** 看到 Top-10 和 LLM 生成的解释

漏斗比例是关键：在调用昂贵的 LLM 之前，已经完成了 **5-7 个数量级** 的剪枝。 LLM 只需要处理约 50 个物品，而不是百万级别的目录。正是这个架构选择，让 LLM 在推荐场景中变得可行。

```python
class HybridRecommender:
    """ANN -> DNN -> LLM。所有团队最终都会采用的架构。"""

    def __init__(self, ann_index, dnn_ranker, llm_reranker):
        self.ann = ann_index            # Faiss / HNSW
        self.ranker = dnn_ranker        # DIN / DCN-V2 / 双塔
        self.llm = llm_reranker         # GPT-4o 或微调 7B

    def recommend(self, user_id, query_embedding, k=10):
        # 第一阶段：ANN —— 百万级别缩小到约 1000，耗时约 10ms
        cand_ids = self.ann.search(query_embedding, top_k=1000)

        # 第二阶段：DNN 排序 —— 从 1000 缩小到约 50，耗时约 30ms
        scored = self.ranker.score_batch(user_id, cand_ids)
        top50 = sorted(scored, key=lambda x: -x['score'])[:50]

        # 第三阶段：LLM 重排 —— 从 50 中选出 Top-10，耗时约 1.5s
        # 关键点：prompt 能装下，成本可控
        return self.llm.rerank(user_id, top50, k=k)
```

中间的 DNN 排序阶段，往往是刚接触 LLM 的人最容易忽视的。不能直接从 ANN 跳到 LLM 重排： ANN 筛出的 1000 个候选仍然包含大量噪声（热门但不相关的物品占主导），把这些塞进 prompt 要么超出上下文限制，要么把有效信号淹没。 DNN 提供的是一个干净的 Top-50，并且带有 LLM 不具备的**学到的交互模式**。
## 冷启动： LLM 真正的优势场景

![冷启动：随用户交互数变化，LLM 零样本 vs CF](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig5_cold_start.png)

画一张精度对用户历史长度的曲线，就能清楚回答「LLM 还是 CF」这个问题：

- **0-5 次交互**（冷启动）： LLM 零样本胜出，优势达到 4-7 倍。 CF 没有数据可用，而 LLM 的世界知识成了唯一的信号。
- **10-50 次交互**（升温阶段）：差距迅速缩小。 CF 开始捕捉到真正的协同信号。
- **100+ 次交互**（热用户）： CF 在纯排序精度上反超。此时， LLM 的世界知识反而成了噪声，用户的实际偏好才是关键。

结论很明确：**按用户状态分流**。

```python
def route_recommend(user_id, query, cf_model, llm_model):
    """根据用户的数据密度选择合适的模型。"""
    n_interactions = cf_model.interaction_count(user_id)

    if n_interactions < 5:
        # 冷启动：LLM 直接利用 query 调用世界知识
        return llm_model.recommend(query)
    elif n_interactions < 50:
        # 升温阶段：混合策略
        cf_recs = cf_model.recommend(user_id, k=50)
        return llm_model.rerank(query, cf_recs, k=10)
    else:
        # 热用户：CF 表现更优，LLM 仅用于生成解释
        return cf_model.recommend(user_id, k=10)
```

这也说明了一个现象：很多团队反馈「试过 LLM，效果不如 DCN」，其实是因为测试集中在热用户场景。在热用户场景下， CF 本来就占据绝对优势，这种对比结果并不公平。

## 对话式推荐： ChatREC 风格

![对话式推荐流程：对话状态与工具调用](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig6_chat_flow.png)

对话式推荐系统完全不同，它是一个**规划器**，而不是一次性排序器。 LLM 负责维护对话状态，决定什么时候检索、什么时候澄清用户意图、什么时候给出推荐。推荐这一步通常会交给检索 + DNN 排序的组合来完成。

```python
class ChatREC:
    """LLM 是规划器；检索和排序是它调用的工具。"""

    def __init__(self, llm, retriever, ranker):
        self.llm = llm
        self.retriever = retriever      # ANN 检索
        self.ranker = ranker            # DNN
        self.state = {
            "history": [],              # 消息记录
            "preferences": {},          # 用户偏好积累
            "candidates": [],           # 当前候选列表
        }

    def turn(self, user_message):
        self.state["history"].append({"role": "user", "content": user_message})

        # LLM 决定下一步动作：澄清、检索、推荐还是解释
        plan = self.llm.plan(
            history=self.state["history"],
            current_prefs=self.state["preferences"],
            current_candidates=self.state["candidates"],
        )

        if plan.action == "extract_preference":
            self.state["preferences"].update(plan.extracted_prefs)
            return self.llm.acknowledge(plan.extracted_prefs)

        if plan.action == "retrieve":
            query = self.llm.build_query(self.state["preferences"])
            cand = self.retriever.search(query, top_k=200)
            self.state["candidates"] = self.ranker.rank(cand, top_k=20)
            return self.llm.present(self.state["candidates"][:5])

        if plan.action == "filter":
            self.state["candidates"] = [
                c for c in self.state["candidates"]
                if plan.filter_fn(c)
            ]
            return self.llm.present(self.state["candidates"][:5])

        if plan.action == "explain":
            return self.llm.explain(plan.target_item, self.state["preferences"])

        return self.llm.respond(self.state["history"])  # 默认兜底逻辑
```

这段代码只是冰山一角，真正的难点在别处。首先是规划器的 prompt 设计：暴露哪些动作、如何避免 LLM 推荐不在 `candidates` 列表中的物品。其次是状态管理：用户改变主意时，什么时候丢弃旧偏好；过滤后，什么时候刷新候选列表。这些细节决定了系统的实际效果。
## 成本与质量的帕累托前沿

![LLM 推荐方法在成本/质量平面上的帕累托前沿](https://blog-pic-ck.oss-cn-beijing.aliyuncs.com/posts/zh/recommendation-systems/12-大语言模型与推荐系统/fig7_cost_quality.png)

把所有已发表的方法映射到（每千次请求成本，相对 CF 基线的 NDCG 提升）坐标系上，能看出三点：

1. **每千请求超过 1 美元后，收益急剧下降**。从微调 7B （~$0.4）升级到 GPT-4 生成式（~$25）， NDCG 只提升 ~2.4 点。而从 CF 基线升级到微调 7B， NDCG 能提升 ~8.6 点，延迟成本却不到 100 倍。
2. **微调 7B 是生产环境的最佳选择**。 TallRec 和 LlamaRec 的精度接近 GPT-4，但单次调用成本只有 GPT-4 的 1%。如果能一次性投入 4 张 A100 训练 LoRA，全年推理成本都能省下来。
3. **对话式智能体是另一种产品，不是曲线上的另一个点**。它贵的原因不是模型本身，而是每个会话都是多轮交互——同样的推荐结果，你得多付 5 到 10 倍的请求量。要为它买单，得看对话体验本身的价值，而不是排序指标的提升。

### 实用的成本优化手段

| 手段 | 典型节省 | 注意事项 |
|---|---|---|
| 缓存相同 prompt | 30%-60% | 只有查询重复时才有效 |
| 历史截断到 20 条 | 20%-40% | 可能丢失长尾偏好 |
| 重排 Top-20 而非 Top-50 | 50% | 精度损失小（~1 NDCG 点） |
| 微调 7B 替代 GPT-4 | 90%+ | 需要标注数据和 GPU 时间 |
| 热用户直接路由到 CF | 40%-80% | 取决于用户分布 |

最大的优化手段，毫无疑问，是按用户状态分流（上一节提到）。在一个典型平台上， 70% 以上的请求来自热用户，这些场景下 CF 明显更优。把他们从 LLM 路径中分流出去，等于免费省下一大笔钱。
## 评测：到底要测什么

标准排序指标依然适用（NDCG@K、 Recall@K、 MRR），但 LLM 系统需要增加一些新的维度：

```python
import numpy as np

def evaluate_llm_recommender(predictions, ground_truth, llm_calls, latencies):
    """标准指标 + LLM 专属的运营指标。"""
    # --- 质量 ---
    k = 10
    hits = [set(p[:k]) & set(gt) for p, gt in zip(predictions, ground_truth)]
    precision = np.mean([len(h) / k for h in hits])
    recall = np.mean([len(h) / max(len(gt), 1) for h, gt in zip(hits, ground_truth)])

    def ndcg(pred, gt, k=10):
        gt_set = set(gt)
        dcg = sum(1 / np.log2(i + 2) for i, p in enumerate(pred[:k]) if p in gt_set)
        idcg = sum(1 / np.log2(i + 2) for i in range(min(k, len(gt_set))))
        return dcg / idcg if idcg > 0 else 0.0

    ndcg_score = np.mean([ndcg(p, gt) for p, gt in zip(predictions, ground_truth)])

    # --- 运营 ---
    p50_latency = float(np.percentile(latencies, 50))
    p99_latency = float(np.percentile(latencies, 99))
    cost_per_request = float(np.mean(llm_calls)) * 0.005   # 单次调用成本粗略估算

    return {
        "precision@10": precision,
        "recall@10": recall,
        "ndcg@10": ndcg_score,
        "p50_latency_s": p50_latency,
        "p99_latency_s": p99_latency,
        "cost_per_request_usd": cost_per_request,
    }
```

运营指标的重要性远超预期。一个 NDCG 提升 2 分，但 p50 延迟增加 1.5 秒的模型，对大多数 C 端产品来说都是得不偿失——用户流失的速度远远快于排序质量提升带来的收益。

对话式系统还需要额外关注以下指标：

- **达成满意的轮数**：用户接受推荐前需要几轮交互
- **每个 session 的推荐多样性**：是否每次对话都集中在相同的五个物品上
- **幻觉率**： LLM 推荐不在候选集中的物品的比例

## 常见问题

### 我到底什么时候该用 LLM？

满足以下任意一条就该考虑：(a) 物品有丰富的文本内容，(b) 冷启动问题严重，(c) 需要给用户展示解释，(d) 在构建对话式推荐系统。如果这些都不沾边，调优后的 DCN-V2 或 DIN 能以更低的成本胜过任何 LLM。

### 用 GPT-4 还是微调 7B？

几乎总是选择微调。**TallRec** 和 **LlamaRec** 的研究表明， LoRA 微调的 7B 模型在推荐任务上能追平甚至超越 GPT-4 的零样本性能，推理成本只有 GPT-4 的 1%。继续用 GPT-4 的唯一理由是：完全没有训练数据，或者领域变化太快，来不及重新微调。

### 怎么避免 LLM 输出不存在的物品？

两层防护机制：(1) 限制 LLM 输出为候选集的索引（如 `[3, 1, 5, 2, 4]`，而不是标题），(2) 返回前严格校验输出是否在候选列表中。如果 LLM 输出了未知索引，直接回退到 DNN 排序的结果。

### 传给 LLM 的候选集应该多大？

经验值是 20 到 50。低于 20， LLM 没有足够的选项进行有意义的重排；高于 50， prompt 变得又长又慢又贵，模型还容易忽略中间部分的物品（即“lost in the middle”效应）。

### 推荐需要 RAG 吗？

视情况而定。当 LLM 需要超出 prompt 的**事实性**信息时（如发售日期、规格参数、当前库存）， RAG 会派上用场。但如果只是对已知目录排序，把目录本身放进 prompt 就相当于检索，不需要额外的向量数据库。

### 实时 feed 场景延迟怎么解决？

实时 feed （如 TikTok 式无限下滑、新闻流）通常无法承受每次曝光都同步调用 LLM。可行的做法包括：(a) 在低峰时段预计算 LLM 重排结果并缓存，(b) 只在第一页使用 LLM，后续页用 DNN 排序，(c) 把 LLM 放到离线批处理路径中，用于生成次日推荐。

### 多语言目录怎么处理？

LLM 天然支持多语言——翻译和跨语言推理能力内置其中。但要注意：如果只用英文交互数据微调，多语言能力会大幅下降。解决方法有两种：要么在微调数据中保留一定比例的非英文样本，要么直接微调一个多语言基座模型（如 Qwen、 Llama-3 多语言版、 mT5）。
## 总结

LLM 并不是要取代推荐系统，而是填补上一代技术无法解决的特定空白。三种角色正好对应三个缺口：

- **增强器**解决**内容理解**的问题。离线运行，永久缓存，生成更干净的嵌入向量，冷启动能力更强。
- **预测器**解决**冷启动和小数据**的问题。微调一个 7B 模型，用它对传统流水线输出的 Top 20–50 进行重排。
- **智能体**解决**交互**的问题。只在真正需要对话的场景中使用，并且明确接受延迟带来的成本。

所有生产团队最终都会采用的架构是混合流水线： ANN 召回 → DNN 排序 → LLM 重排 → 可选的解释模块。这种方式可行，因为 LLM 只处理 Top 50 的候选集，而不是全量商品库。成本可控，同时质量还能借助 LLM 的语义提升。

从这个架构入手。第一天就同时测量运营指标和排序指标。把活跃用户从 LLM 路径中分流出来，避免账单失控。一旦积累了几百条标注数据，立刻开始微调模型。

## 10K QPS 推理： LLM-as-ranker 梦碎的地方

LLM4Rec 的论文通常对延迟问题避而不谈。但在实际生产环境中，一个典型的排序模块需要满足以下要求：

- 峰值 QPS 达到 10K 到 100K  
- 端到端 p99 延迟小于 200 ms （包括召回、排序和业务规则）  
- 每次请求处理 50 到 500 个候选物品  

这些数字直接决定了 LLM 作为排序模型的可行性。

**直接用 LLM 排序完全不可行**。以 GPT-4o-mini 为例，单次调用对 50 个物品进行排序， p50 延迟约为 800 ms， p99 延迟高达 2.5 秒。如果要支持 10K QPS，意味着需要同时处理 25,000 个并发请求，每次调用成本 $0.0007 → 每秒 $7 → 每天 $60 万。这种方案根本无法落地。

**蒸馏后的小型 LLM 排序勉强可用**。一个 3B 参数量的蒸馏模型，加上 4-bit 量化，在 A10 显卡上对 50 个物品排序的 p50 延迟约为 80 ms。要支持 10K QPS，大约需要 150 张 GPU （假设每张卡通过批处理达到 100 QPS）。云服务成本为每天 $1.5K，包年实例每月 $5 万。对于漏斗顶端的场景还能接受，但对于长尾流量来说依然昂贵。

**LLM 作为特征提供方才是真正的生产答案**。把 LLM 放在离线环境中运行，将其输出（如物品摘要、语义标签、 embedding 增量）存储到特征库中。在线排序时，使用一个轻量级的 DCN 或 DIN 模型，在 5 到 10 ms 内消费这些特征。实际上，所有号称“LLM4Rec”的生产系统都采用这种方式——即使有些博客文章让人误以为 LLM 是实时参与请求路径的。

以下是某个真实场景的延迟预算分解：

| 组件 | 延迟预算 | 实现方式 |
|------|----------|----------|
| 召回 | 30 ms | 对 1000 万物品进行 ANN 搜索，返回 top 1000 |
| 预排序 | 20 ms | 使用小 MLP 对 1000 个物品筛选出 top 200 |
| 排序 | 60 ms | 使用 DIN 模型结合 LLM 衍生特征，选出 top 50 |
| 重排 / 业务规则 | 30 ms | 考虑多样性、新鲜度和业务规则 |
| 网络 + 序列化 | 60 ms | 预留缓冲时间 |
| **总计** | **200 ms** | |

LLM 的真正贡献体现在“LLM 衍生物品特征”这一环节——这些特征提前计算好，线上从特征库中获取，耗时不到 2 ms。
## 为什么生产团队用 LLM 增强特征替换了基于 prompt 的排序

我观察到多家公司在两年内经历了类似的演变。

**阶段 1 （2023 年中）：用物品列表 prompt LLM，碰运气**  
"GPT-4 能排序！"的博客文章刷屏了。在 demo 中处理 10 个物品时效果不错，但规模一大就撑不住了。

**阶段 2 （2023 年底到 2024 年中）：蒸馏后上线**  
把 GPT-4 的输出当作标签，训练一个 7B 参数的模型来模仿它，然后上线。延迟降低了 5 倍，但对长尾 QPS 来说还是太贵。质量比 teacher 模型下降了约 3 个 NDCG 点。

**阶段 3 （2024 年底至今）： LLM 成为离线特征工厂**  
LLM 不再直接参与请求路径，而是专注于以下任务：  
1. 生成物品摘要或抽取属性（每个物品运行一次，永久缓存）  
2. 从搜索 query 或浏览 session 中提取用户意图标签（每小时为活跃用户运行一次）  
3. 在物品入库时检测内容质量和安全问题  

线上模型仍然是 DIN 或 DCNv2——一个 5000 万参数的经典排序模型——但多了 30 多个 LLM 衍生特征。这一阶段的效果提升显著： CTR 提高了 1% 到 3%，转化率提升了 2% 到 5%，而且线上成本没有增加。

教训很明确：不要把 LLM 放进请求路径，而是要把它的输出放进模型里。
