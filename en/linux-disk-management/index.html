1. Storage basics: HDD vs SSD

1.1 Random vs sequential I/O

  • HDD performance is dominated by mechanical latency (seek + rotation). Random I/O can be orders of magnitude slower than sequential I/O.
  • SSD has much lower latency; random I/O is closer to sequential performance. SSDs also have write amplification and garbage collection behaviors that matter under sustained write load.

1.2 Sectors, filesystem blocks, and ‚Äúsmall files still consume space‚Äù

  • Disks store data in sectors (historically 512B; many modern devices are 4K).
  • Filesystems allocate space in blocks (allocation units). A 1-byte file still consumes at least one block plus inode/metadata.

1.3 Object storage (when a single disk is not the right abstraction)

For large amounts of cold blobs (images, archives, backups), object storage such as S3 / OSS is often a better operational choice than growing a single mounted filesystem.

3. RAID (software RAID with mdadm)

3.1 RAID levels (quick reference)

  • RAID 0: throughput, no redundancy.
  • RAID 1: mirroring, capacity ~50%.
  • RAID 5: parity, survives 1 disk failure.
  • RAID 6: double parity, survives 2 disk failures.
  • RAID 10: mirrored stripes, strong performance + redundancy.

6. Inodes, links, and ‚Äúwhy space doesn‚Äôt come back‚Äù

6.1 Inodes vs data blocks

A filename is a directory entry that points to an inode. The inode stores metadata and pointers to data blocks.

6.2 Hard links vs symlinks

  • Hard link: another directory entry to the same inode (cannot cross filesystems; typically not for directories).
  • Symlink: its own inode; content is a path to the target (can dangle; can cross filesystems).

6.3 Deleted files still consuming space
