Architecture Design: From Monolith to Autonomous Agents

How my architectural thinking evolved from a single Next.js app to distributed autonomous agent systems — and the patterns that emerged along the way.

The Shape of a System#

Every architecture is a frozen argument. It records what you believed about the problem at the time you committed the code. Looking back across four systems I built over eighteen months — a marketing content platform, a zero-dependency skill routing engine, an autonomous research agent, and a multi-model coding orchestrator — I can trace how my architectural instincts shifted. Not always forward. Sometimes sideways. But there is a clear progression: from “keep it in one process” to “let the agents govern themselves.”

This is not a tutorial on distributed systems. It is a reflection on why I chose the structures I did, what broke, and what principles survived contact with production.

Act I: The Monolith That Worked (AI4Marketing)#

AI4Marketing started as a weekend prototype: a Next.js app that called Qwen to generate marketing copy. Eighteen months later it has 121 API routes, 21 database tables, a video production pipeline, a short-drama generation engine, a GEO optimization system, and a payment flow with four provider integrations (Alipay, WeChat Pay, Stripe, PayPal). It is still a single Node.js process managed by PM2 in fork mode — not even cluster mode. One process, one thread for JavaScript, one event loop.

I did not plan a monolith. I planned to ship fast. The architecture is the residue of that velocity.

Why it stayed monolithic#

The critical insight is that AI4Marketing is fundamentally a request-response application with long-running side effects. A user submits a content generation request; the system validates, deducts quota, calls an LLM, and returns results. The video pipeline takes minutes, but it is still triggered by a single HTTP request and writes its progress to the same PostgreSQL database. There is no inter-service communication because there are no services — just function calls within a process.

There was never a compelling reason to split. The “microservices would be cleaner” argument kept losing to “I can grep the entire codebase in one terminal.” When the video pipeline hangs, I check lib/video-pipeline-v2.ts. When quota logic is wrong, I look at lib/quota-checker.ts. When the payment webhook misbehaves, the handler is right there in app/api/webhook/alipay/route.ts. There is no service mesh, no message queue, no container orchestration, no Kubernetes, no Docker Compose. Just one process, one database, one deploy target, and pm2 restart ai4m.

The database schema is a star model with User at the center. Posts, VideoProjects, DramaProjects, Orders, and Subscriptions radiate outward. Twenty-one tables total. Every relationship is a direct foreign key — no event sourcing, no CQRS, no eventual consistency. When I need to know a user’s quota, I read one row. When I need their video projects, I join two tables. This is boring technology, and it works.

The patterns that made the monolith survivable#

What kept 121 routes from becoming unmaintainable spaghetti was discipline at the handler level. Every single route follows an identical layered pattern:

Auth -> Validation (Zod) -> Rate Limit -> Quota Check -> Business Logic -> Failure Refund

This is not a framework feature. It is a convention enforced by reading my own code until the pattern became muscle memory. The withMetrics higher-order function wraps every route handler, recording http_request_duration_seconds and http_requests_total per route without requiring opt-in. Route labels are explicitly specified — never auto-derived from dynamic path parameters — to prevent cardinality explosion in the metrics store.

The quota system deserves special attention because it solves a real concurrency problem. The naive approach — “read balance, check if sufficient, deduct” — has a textbook TOCTOU (time-of-check-to-time-of-use) race: two concurrent requests both read “balance = 1”, both pass the check, both deduct. The fix is a conditional update that makes the check and the deduction atomic:

const claim = await tx.user.updateMany({
  where: { id: user.id, quotaUsed: { lte: effectiveLimit - pointCost } },
  data: { quotaUsed: { increment: pointCost } },
})
if (claim.count === 0) return { allowed: false }

This single WHERE clause eliminates the race condition. Two concurrent requests cannot both succeed because only one can match the condition at the row level. PostgreSQL’s row-level locking guarantees this. It is a 3-line fix to a problem that has cost production systems millions of dollars.

The rate limiter uses nine presets: GENERATE (10/min), VIDEO (5/hour), LOGIN (5/min), REGISTER (3/5min), QUERY (60/min), STATUS_POLL (120/min), SEARCH (30/min), ADMIN (30/min), WEBHOOK (20/min). These are in-memory sliding windows — no Redis dependency. The memory is bounded by periodic cleanup (every 5 minutes, cap at 10,000 entries). This is good enough for a single-process application because the memory never migrates.

The graceful shutdown handler registers on SIGTERM, sets a shuttingDown flag to reject new requests, then polls the activePipelines set every two seconds. If all pipelines drain within 10 minutes, it exits cleanly. If not, it force-exits. PM2 respects the 10-minute kill_timeout. This means I can deploy without killing in-flight video renders — they finish, then the process recycles.

The API key manager: distributed reliability in a monolith#

One piece of hidden complexity is the API key manager (lib/api-key-manager.ts). AI4Marketing calls DashScope for every generation. With 16 API keys spread across CN and INTL regions, the system needs intelligent rotation:

  • On 429 (rate limited): immediately switch to the next key, no backoff
  • On other errors: exponential backoff retry
  • Per-key circuit breaker: N consecutive non-rate-limit failures parks the key
  • Global circuit breaker: all keys failed simultaneously stops all retries
  • Regional routing: CN keys go to dashscope.aliyuncs.com, INTL keys to dashscope-intl.aliyuncs.com

This is a distributed-systems pattern (circuit breaking, key rotation, regional failover) compressed into a single-process library. It handles the most common production failure mode — API rate limits — without any infrastructure beyond a JavaScript object that tracks key state.

What the monolith cannot do#

The limitation appeared when I added the Spark Drama pipeline — an agentic short-film production system where an LLM writes scripts, directs scenes, reviews dailies against character reference images, and iterates. This pipeline is fundamentally multi-step, multi-model, and state-heavy. A single drama project spawns a Python sidecar via child_process.spawn that orchestrates multiple LLM calls across screenwriting, directing, and reviewing phases, streaming NDJSON status lines back to Node.

It works. It is ugly. It is the point where the monolith’s seams became visible — because the drama pipeline needs its own lifecycle, its own error recovery, its own resource budget, independent of the content generation pipeline sharing the same event loop. When the drama sidecar leaks memory, it drags the entire application with it. When I need to restart drama processing, I have to restart everything.

This is the moment that made me understand the real boundary: not code size, but lifecycle independence.

Act II: Zero Dependencies as Architecture (DaaS)#

DaaS (Documentation-as-a-Skill) was built with a conscious reaction to AI4Marketing’s accidental complexity. The premise: Python’s standard library is enough. No FastAPI, no SQLAlchemy, no Redis, no Docker, no pip install in production.

The entire HTTP server is ThreadingHTTPServer from http.server. One class, DaaSHandler, overrides do_GET and do_POST with pattern-matching dispatch. 96 route patterns, matched by string operations on the URL path. The frontend is 22 static HTML/CSS/JS files served with in-memory gzip caching and mtime-based invalidation. No build step, no webpack, no node_modules, no CDN at runtime.

The philosophy behind zero-dependency#

Every dependency is a liability with a TTL. It will break on upgrade, go unmaintained, or introduce a supply-chain vulnerability. The left-pad incident. The colors.js sabotage. The event-stream backdoor. These are not hypotheticals — they are historical facts of the npm ecosystem. For a system I intend to run for years with minimal maintenance, the only dependencies I trust are the ones that ship with CPython and are maintained by the CPython core team.

DaaS runs on a single Guangzhou ECS instance behind a Cloudflare Tunnel. The deployment procedure is rsync -a . server:/var/www/DaaS/ && systemctl restart daas-api. If the Python version changes from 3.10 to 3.12, http.server and threading and json and os still work identically. If I need to deploy to a fresh box, I copy the directory and run python3 -m src.api.server --host 0.0.0.0 --port 80. There is nothing to install.

What zero-dependency forces you to design#

Without an ORM, I use atomic file writes for all state persistence. Every mutation writes to a temporary file, then calls os.replace() to atomically swap it into place. This means a crash mid-write never corrupts state — you either get the old version or the new version, never a partial write. This pattern runs through the entire codebase: keys.json, usage.json, billing.json, geo_measurements.json — all 14 state files in /var/lib/daas/ use it.

Without a message queue, background jobs run as daemon threads with timeout guards. The scheduler spawns seven jobs (drift_check, graph_rebuild, recall_snapshot, data_gc, billing_cycle, edge_explore, geo_remeasure), each in a separate thread with a hard timeout (60-300 seconds depending on the job). If a job hangs, the timeout kills it. The scheduler retries next cadence. No dead-letter queues, no retry policies, no visibility timeouts. Just a thread, a timer, and a restart.

Without a framework’s middleware stack, auth is a function called at the top of each handler that returns the identity or raises a 401. The three-layer classification (public, authenticated, paid) is documented in comments and enforced by convention. Payment webhooks are public (Alipay/Stripe need to reach them). Route queries are public (AI crawlers need to reach them). Ingestion is paid. Workspace creation is paid. This is simple enough to hold in your head.

The caching architecture is worth examining because it emerged from the zero-dependency constraint rather than being designed top-down:

CacheScopeTTLInvalidation
Static filesIn-memory bytes + gzipped variantmtime-basedFile modification detected

All caches use inflight deduplication via threading.Event. When two requests arrive simultaneously for the same uncached data, one computes while the other blocks on the Event. This prevents thundering-herd recomputation — a problem that cache-miss storms cause in real production systems.

The god-file problem and its resolution#

The original server.py was 4,750 lines. This is the natural terminus of “no framework” — all logic converges into the HTTP handler because there is no router, no middleware chain, no dependency injection container to separate concerns. The fix was mechanical: identify cohesive groups of functions, extract them into modules (routing.py, analytics.py, geo.py, geo_loop.py, geo_routes.py, billing.py, payment.py, keys.py, auth.py, ingest.py, edge_explore.py, improve.py, gap_radar.py, skill_graph.py), and leave the server file as pure HTTP dispatch + orchestration.

The result is 17 extracted modules, each with clear responsibilities and no circular imports. server.py dropped to ~1,200 lines of routing dispatch. The lesson: zero-dependency does not mean zero-structure. The structure just has to be self-imposed rather than framework-imposed. And the discipline to extract modules must be applied proactively — before the god-file becomes too tangled to safely decompose.

Act III: The State Machine as Universal Controller (Research Agent)#

The Research Agent is where my architecture made its biggest conceptual leap. This is a fully autonomous system that reads papers from arXiv, builds a knowledge graph (65,000+ nodes, 197,000+ edges in SQLite), generates research ideas, debates them through a three-round adversarial process, designs experiments, dispatches execution to worker nodes, runs statistical analysis, and writes papers. It operates 24/7 without human intervention on a 4-vCPU, 7.5 GB main server with two 128 GB worker nodes for computation.

The core insight that made this possible: explicit finite state machines for every entity lifecycle, enforced at the data layer.

Why FSM#

Before the FSM, the coordinator used a 15-level priority if/elif chain to decide what to do next. The logic checked flags: “if there’s an idea with status approved and no protocol.json, dispatch experiment_designer.” It worked for three states. At twelve states with failure paths, retries, and infrastructure errors, it became a combinatorial explosion. Real bugs emerged: an experiment would sit at executed status forever because the conditional for “dispatch statistician” checked for analysis.json non-existence — but a crash during analysis left a partial file that satisfied the existence check without containing valid results.

The FSM (framework/pipeline_state.py) eliminated this entire class of bugs by making transitions explicit and exhaustive:

Idea lifecycle:
  proposed -> approved -> experiment_designed -> experiment_completed
      |          |              |                        |
      |          |              +-> negative_result      |
      |          |              +-> paper_written ------>+
      |          |                                      |
      |          +-> killed                             +-> paper_reviewed
      |          +-> failed
      +-> merged
      +-> subsumed
      +-> killed

Protocol lifecycle:
  designed -> running -> executed -> analyzed
      |          |          |           |
      |          |          +-> failed  +-> pending_failure_analysis
      +-> needs_redesign               +-> needs_redesign (back to designed)
      +-> abandoned                    +-> needs_rerun (back to running)
      +-> dataset_failed

Every transition(entity, from_state, to_state) call validates that the transition is legal, records it to .history.jsonl with a timestamp, and raises InvalidTransition on illegal moves. You cannot move an idea from proposed directly to paper_written — the FSM enforces that it must pass through design, execution, and analysis first. This is a compile-time guarantee at the data layer.

The dispatch decision layer#

The Boss Agent is a ReAct LLM orchestrator that decides what to do next. But it does not decide from scratch — it evaluates nine structured predicates from framework/dispatch_decision.py:

@dataclass
class Decision:
    dispatch: bool
    reason: str
    artifact_required: str  # path the agent MUST produce
    priority: int           # higher = dispatched first
    target: str             # idea/experiment ID

Each predicate (e.g., should_dispatch_experimenter(experiment_id)) reads the FSM state, checks for required preconditions (protocol exists, no active lock, slot available), and returns a structured decision. The Boss reads all nine predicates’ outputs, sorts by priority, and dispatches the top one.

Why this matters: when something fails to dispatch, the reason field tells you exactly why. “Protocol not yet designed” or “All experimenter slots occupied” or “Target locked by PID 12847.” There is no mystery. There is no “the system just didn’t do it.” Every non-action has an auditable explanation.

The priorities create an implicit scheduling policy: statistician at 200 (because analysis unblocks writer + reviewer + formatter), failure_analyst at 150 (because failure analysis triggers redesign which unblocks experimenter), merger at 85 (pre-debate check). These numbers live in a JSON file that hot-reloads every 30 seconds — I can reprioritize the entire system’s behavior without restarting any process.

Fleet: distributed execution via HTTP polling#

The main server has 7.5 GB of RAM. Running LLM-driven experiments (which can require loading large datasets and making hundreds of API calls) would OOM it immediately. The fleet architecture solves this with radical simplicity:

Workers poll the main server every 5 seconds: “Any commands for me?” The main server responds with queued work or empty. Workers execute, then POST the result back. Workers also POST heartbeats every 60 seconds with system metrics (load, memory, disk, running agent count). A separate sync loop pushes generated artifacts (papers, KG patches, experiment data) every 60 seconds.

This is HTTP polling. It is the simplest possible distributed execution model. There is no WebSocket, no gRPC stream, no message broker. A worker can go offline and come back — it simply resumes polling. The main server tracks worker health by heartbeat freshness. If a heartbeat goes stale (>600s), the worker is marked unselectable and its dispatched tasks are eventually reaped by the orphan dispatch reaper.

Why not Celery/Temporal/Bull? Because those systems are themselves complex distributed systems that need monitoring, Redis/RabbitMQ backing stores, and failure modes I would have to learn and debug. My fleet protocol is four HTTP endpoints. I can debug it with curl. I can replay any dispatch by re-posting the command JSON. The entire fleet server is ~400 lines of Python.

Self-healing: forty rules of automatic recovery#

On a 7.5 GB machine running six systemd services (coordinator, pipeline, dashboard, supervisor, kg-merger, dingtalk-listener), OOM kills are not theoretical — they happen weekly. The architecture does not try to prevent OOM. It assumes OOM will happen and designs for recovery:

  1. OOM priority triage: The pipeline gets OOMScoreAdjust=+500 (sacrificed first). The coordinator gets -500 (protected). When memory pressure hits, Linux kills the pipeline before the coordinator. The pipeline’s state is all on disk — it restarts and resumes from where it was.

  2. Internal memory watchdogs: Both processes have background threads that check RSS+Swap every 2-3 minutes. The coordinator triggers aggressive GC at 1500 MB swap and force-exits at 2400 MB RSS. The pipeline force-exits at 1900 MB. Force-exit is deliberate: it is cheaper to restart from a known state than to limp along with degraded memory.

  3. Supervisor reconciliation: supervisor_lib/heal_rules.py contains 40+ rules that fire every minute. Rule 1: if analysis.json exists but status is still executed, force transition to analyzed. Rule 2: if failure_analysis.json exists with a decision, apply the decision. Rule 3: if redesign count exceeds 5, kill the idea (backstop against infinite loops). Rule 4: if needs_redesign has been stuck for 24 hours with 5+ dispatch attempts, set abandoned.

  4. Infrastructure error detection: If a failure_analysis finds that the root cause contains “timeout” or “429” or “404”, it resets the protocol to designed for retry rather than triggering a redesign. Transient infrastructure failures should not waste a redesign cycle.

  5. Stale lock reaping: Locks use fcntl.flock() with PID + timestamp written inside. The supervisor checks whether the PID is still alive. Dead PIDs get their locks removed. Locks older than 4 hours (configurable TTL) are force-removed regardless.

The invariant checker (framework/invariant_checker.py) runs after every boss cycle and enforces global consistency: status/artifact alignment, no duplicate dispatches, no orphaned locks. Violations are logged at ERROR level and persisted to data/invariant_violations.json. This is the final safety net — even if a bug introduces an inconsistency, the next boss cycle will detect and flag it.

Act IV: DAG Decomposition for Complex Tasks (Elevator)#

Elevator is an autonomous coding system that orchestrates multiple Chinese LLMs (Qwen, DeepSeek, Kimi, GLM, MiniMax) through a structured pipeline: goal decomposition into a subgoal DAG, parallel batch execution, cross-model verification, and experience learning. All models are accessed via DashScope’s unified OpenAI-compatible API, which means model switching is a string change.

The DAG as execution plan#

When a user submits a goal — “Build a WeChat mini-game idle RPG” — the planner (planner.py, 474 lines) converts it into a structured JSON DAG. The planner’s context injection chain is critical: it reads global lessons (past failures to avoid), skills (successful plan templates), workspace state (current files), and project memory (cross-milestone decisions) before generating the plan.

The resulting DAG for a real project looks like:

m1:  Canvas framework + game loop        (no deps)
m2:  Character system + stats            (no deps)
m3:  Configuration-driven architecture   (depends: m1, m2)
m4:  Combat system                       (depends: m3)
m5:  Inventory + equipment               (depends: m3)
m6:  Cultivation progression             (depends: m4)
m7:  NPC + dialogue system               (depends: m3)
m8:  Map + exploration                   (depends: m7)
m9:  UI polish                           (depends: m5, m6)
m10: Art assets                          (depends: m9)
m11: Sound effects                       (depends: m10)
m12: Save/load system                    (depends: m8)
m13: Performance optimization            (depends: m11, m12)
m14: Final QA + packaging                (depends: m13)

The runtime processes this via topological batch execution: m1 and m2 execute in parallel (no deps), then m3 waits for both, then m4/m5/m7 execute in parallel (all depend only on m3), and so on. A ThreadPoolExecutor manages the parallel execution. The wall-clock time equals the critical path, not the sum of all milestones.

DAG validation is non-trivial because LLMs sometimes produce invalid plans. The planner performs cycle detection, dangling reference removal (references to non-existent subgoal IDs), and ID uniqueness enforcement before accepting a plan. An optional critique step (critique_plan()) sends the plan to a cross-family model for review — if critical issues are found, revise_plan() rewrites it (bounded by MAX_CRITIQUE_ROUNDS to prevent infinite revision).

Cross-family verification: the anti-sycophancy pattern#

Elevator’s most consequential architectural decision is that no model evaluates its own output. The execution pipeline enforces this structurally:

Executor (qwen3.6-plus) produces code
    |
    v
Reviewer (deepseek-v4-pro) examines for bugs, edge cases, test gaps
    |
    +--> issues found: inject feedback, loop back to executor (max 2 rounds)
    +--> no issues: proceed to verification
    |
    v
Verifier (qwen3.6-plus, different instance) checks each criterion independently
    |
    +--> low confidence: escalate to ensemble panel (3 models, majority vote)
    +--> all criteria pass: git commit, mark done
    +--> criteria fail: escalate model tier, retry

Why cross-family? Because models within the same family share training data, architecture biases, and failure modes. When Qwen writes buggy code, another Qwen instance often fails to catch the bug — it has the same blind spots. DeepSeek’s MoE architecture processes information differently enough to catch errors that dense transformers miss. I discovered this empirically: within-family review had a ~15% miss rate on bugs that cross-family review caught.

Progressive escalation: cost optimization as reliability#

The three-tier escalation path is designed so that 80% of tasks complete on the cheapest model:

Attempt 0: qwen3.6-plus       (4 CNY/M in, 12 CNY/M out)  — handles most tasks
Attempt 1: qwen3.6-max-preview (20 CNY/M in, 60 CNY/M out) — harder problems
Attempt 2: deepseek-v4-pro    (4 CNY/M in, 16 CNY/M out)  — cross-family fallback

The cost difference between attempt 0 and attempt 1 is 5x. The insight is that model capability follows a power law for task difficulty: most coding tasks are straightforward, and the cheap model handles them fine. Only the tail of complex tasks — architectural decisions, subtle concurrency bugs, complex algorithm implementations — needs the expensive model. And when the expensive model from the same family also fails, switching families (attempt 2) often succeeds because the failure mode was family-specific.

Complexity is estimated before execution: keywords like “fix”, “typo”, “rename” classify as light (12 turns max), while “architecture”, “refactor”, “full-stack” classify as heavy (35 turns max). This pre-classification prevents wasting expensive model budget on trivial tasks.

Tripwires: behavioral anomaly detection#

The executor runs an agent loop — an LLM repeatedly choosing and invoking tools until it declares the task complete. This loop can go wrong in characteristic ways: infinite read loops (reading files without writing), repeat tool calls (calling the same bash command 10 times), or premature submission (declaring done without running tests).

The tripwire system (tripwires.py, 309 lines) monitors for these patterns in real-time:

  • shallow_no_test: Code written but no pytest/bash ever run? Block submission.
  • repeat_loop: Same tool call repeated N times? Warn, then force a different action.
  • no_progress_streak: Multiple turns with no file writes? Inject a reflection prompt.
  • explore_overload: Extended reading without producing code? Prompt to start writing.
  • consecutive_read_only: N+ consecutive read-only turns? Force action.

The escalation path is: warn -> force_action -> auto-submit (after 3 forced actions). This prevents the most expensive failure mode: an agent loop that burns 35 turns of expensive model calls while accomplishing nothing.

Skill evolution: accumulated institutional knowledge#

After every successful task, Elevator distills a “skill” — a plan template abstracted from the specific task. The concrete file paths, variable names, and API endpoints are stripped; the structural pattern is retained. Skills follow a lifecycle:

candidate (score 0.5) -> validation gate -> active (score grows on reuse)
    |                                           |
    | rejected                                  | fail_streak >= 3: needs_review
    v                                           | score < threshold: archived
  archived                                      | unused 180+ days: archived

Retrieval uses dual strategies: semantic (embedding cosine similarity) and keyword (overlap between query keywords and skill context keywords). Higher-scoring skills rank above lower-scoring ones. Only active skills are retrieved — candidates and archived skills are invisible to the planner.

When the skill store exceeds a threshold, similar skills (Jaccard coefficient >= 0.3 on keyword sets) are clustered and merged into aggregate meta-skills. This is unsupervised knowledge compression: the system retains what works in general and forgets what worked only once.

The Principles That Emerged#

Looking across these four systems, I can extract patterns that recur regardless of language, framework, or problem domain:

1. State machines are not just for network protocols#

Every system that manages multi-step workflows benefits from explicit FSM. AI4Marketing’s video pipeline tracks six phases. Research Agent’s idea lifecycle has 12 possible states with audited transitions. Elevator’s task/subgoal/milestone hierarchy is a nested three-level state machine. DaaS tracks ingest job states through its pipeline phases.

The alternative — checking boolean flags and timestamps to infer state — is the single largest source of bugs I have encountered across all four systems. “Is this experiment done?” should be answered by protocol.status == 'analyzed', not by inferring from file existence and modification times.

2. The monolith earns its keep until lifecycle boundaries emerge#

AI4Marketing at 121 routes is still a monolith and still the right choice for its problem domain. The moment I needed multi-step, multi-agent orchestration with independent failure domains and independent resource budgets (Research Agent), the monolith became untenable. The inflection point is not lines of code, team size, or request throughput — it is whether different parts of the system need independent lifecycle management (startup, shutdown, crash recovery, resource allocation, deployment cadence).

3. Zero-dependency is a form of reliability engineering#

DaaS has been running for months with zero dependency-related incidents. No npm audit alerts, no breaking upstream API changes, no “the new version of library X changed its return type.” The cost is more boilerplate code. The benefit is that every failure mode is my own code — debuggable, fixable, and predictable. I will take a known bug in my own threading code over a mystery crash in someone else’s event loop any day.

4. Dispatch predicates beat priority queues#

Both Research Agent and Elevator use declarative predicates (“should this happen? why or why not?”) rather than imperative scheduling (“do this next”). The predicate approach is composable (add new predicates without modifying existing ones), testable (unit test each predicate in isolation), and debuggable (read the reason field when something fails to dispatch). Priority queues tell you what ran. Predicates tell you why something did not run.

5. Self-healing is not optional for autonomous systems#

The Research Agent runs without human intervention for weeks at a time. This is only possible because it assumes failure as a normal operating condition: memory watchdogs, stale lock cleanup, state/artifact reconciliation, invariant checking after every cycle, and 40+ auto-repair rules. An autonomous system without self-healing is just a system that breaks silently and accumulates invisible state corruption.

6. Cross-family verification prevents model hallucination laundering#

If the same model (or model family) both produces and evaluates output, errors compound rather than cancel. The model has systematic blind spots that it cannot detect in its own output. Elevator’s insistence on cross-family verification caught real bugs that within-family review missed. The cost is double the API spend on review. The benefit is catching bugs before they compound through dependent subgoals.

The Meta-Pattern#

If I had to compress everything above into a single architectural principle, it would be this:

Make the system’s state explicit, its transitions auditable, and its recovery automatic. Everything else is optimization.

AI4Marketing makes quota state explicit with atomic conditional updates. DaaS makes routing state explicit with mtime-invalidated caches and atomic file persistence. Research Agent makes research progress explicit with FSM transitions audited to history files. Elevator makes task execution explicit with DAG batches and structured verification checkpoints.

The systems that cause me the least operational pain are the ones where I can answer “what is happening right now?” by reading a single file or querying a single endpoint. The systems that wake me up at 3 AM are the ones where state is implicit — scattered across log files, inferred from timestamps, or worst of all, encoded in the absence of other state (“if this file does not exist, then we must be in phase 3”).

Architecture is not about choosing between monolith and microservices, SQL and NoSQL, sync and async. Those are implementation details that follow from a more fundamental choice: where does state live, how does it change, and who is allowed to change it? Answer those three questions with clarity and discipline, and the rest of the system designs itself.
